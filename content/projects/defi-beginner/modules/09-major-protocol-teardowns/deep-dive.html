Module 09: Major Protocol Teardowns (Beginner Deep Dive)

Mode: bridge | Apply one repeatable method across major protocols

1. Why This Topic Exists

You now know components. This module teaches how to evaluate full systems consistently. Without a teardown method, analysis becomes narrative-driven and inconsistent.

2. Teardown Terms

  • State map: key balances, debts, liquidity, open interest.
  • Action map: user/keeper/governance operations.
  • Invariant map: non-negotiable safety conditions.
  • PnL map: who earns and who absorbs losses.
  • Trust map: offchain dependencies and governance controls.

3. Running Example: One Template, Three Domains

Apply same template to:

  • Uniswap v4 (AMM + hooks)
  • Perp protocol (Hyperliquid/dYdX/GMX)
  • Prediction market (Polymarket/Augur-style)

If template cannot explain a protocol clearly, your model is incomplete.

4. Step-by-Step Teardown Workflow

  1. List state variables.
  2. List primary state transitions.
  3. Write invariants with break conditions.
  4. Trace normal-case cash flows.
  5. Run stress scenario and observe first break.
  6. Document governance intervention paths.

5. Worked Example A (Uniswap v4 Lens)

State: pool reserves, LP positions, fee config, hook state.

Break lens: can hook logic alter expected accounting or create exploitable sequencing?

Cashflow lens: traders pay fee -> LPs + possible hook-directed splits.

6. Worked Example B (Perp Lens)

State: open interest, collateral balances, insurance fund, funding index.

Break lens: sharp move + congestion -> liquidation backlog -> insurance drawdown.

Cashflow lens: funding transfers, trading fees, liquidation penalties.

7. Misconception Correction

Misconception: “Best protocol is the one with highest volume.”

Correction: volume without risk quality can be fragile. Robustness under stress matters more than peak activity.

8. Failure Modes to Compare Across Protocols

  • Oracle dependency failures
  • Execution congestion / delayed liquidations
  • Governance override abuse
  • Incentive misalignment under volatility

9. Capstone Deliverables

  1. 2-page memo per protocol family
  2. State transition diagram
  3. Failure table (severity + likelihood + mitigation)
  4. One improvement proposal with tradeoff discussion

10. Check Your Understanding (With Answers)

  1. Why use same teardown template across domains?
    Consistency prevents narrative bias and exposes real design differences.
  2. What is often the first thing to break in stress?
    Execution capacity and risk-off throughput, not headline formulas.
  3. What makes an improvement proposal credible?
    Clear mechanism change + explicit tradeoffs + measurable expected impact.
