Unit 8

Protocol Stack & Packet Delivery

Modulation encodes bits onto radio waves. But how do those bits become delivered data packets -- or lost ones? This unit bridges the physical layer (BER, modulation) and the transport layer (packets, retransmissions, goodput). It covers the sharp cliff from BER to packet error rate, the ARQ retransmission process, goodput calculation, latency sources, jitter, and the adaptive modulation system that ties everything together.

Learning Objectives

  • Derive packet error rate (PER) from bit error rate and packet size
  • Explain the ARQ retransmission protocol and compute drop probability
  • Calculate goodput as a function of data rate, drop probability, and average retransmissions
  • Enumerate latency components and explain how each varies during a pass
  • Identify the three sources of jitter in satellite packet delivery
  • Describe how adaptive modulation with hysteresis selects ModCods during a pass
  • Explain the per-packet Monte Carlo simulation and the causal chain concept

1. From BER to Packet Error Rate

The bit error rate (BER) from Unit 7 tells us how likely any single bit is to be corrupted. But users send and receive packets, not individual bits. A packet is corrupted if any one of its bits is flipped. This creates a dramatic amplification effect.

The PER Formula

PER = 1 - (1 - BER)N

Where N is the packet size in bits. For a standard 1500-byte Ethernet frame: N = 1500 * 8 = 12,000 bits.

This formula assumes independent bit errors (valid for AWGN channels after interleaving). The key insight: even a tiny BER becomes devastating at packet scale.

The Sharp Cliff

Packet size: 1500 bytes = 12,000 bits

BER = 10^-8:   PER = 1-(1-10^-8)^12000  = 0.00012  = 0.012%   OK
BER = 10^-7:   PER = 1-(1-10^-7)^12000  = 0.0012   = 0.12%    OK
BER = 10^-6:   PER = 1-(1-10^-6)^12000  = 0.0119   = 1.2%     Marginal
BER = 10^-5:   PER = 1-(1-10^-5)^12000  = 0.1131   = 11.3%    Bad
BER = 10^-4:   PER = 1-(1-10^-4)^12000  = 0.6988   = 69.9%    Catastrophic
BER = 10^-3:   PER = 1-(1-10^-3)^12000  = 0.99999  = ~100%    Total loss

Notice how the transition from "fine" to "catastrophic" happens in just two orders of magnitude of BER (10-6 to 10-4). This is the cliff.

This cliff is why satellite links use FEC coding and adaptive modulation. The system must keep BER below ~10-6 for acceptable packet delivery. Above that threshold, performance collapses within a very narrow SNR range.

2. ARQ -- Automatic Repeat Request

When a packet arrives corrupted (detected by CRC checksum failure), the receiver requests retransmission. The simulator uses a simple Stop-and-Wait ARQ with a maximum of 4 retries (5 total attempts).

Drop Probability

A packet is permanently dropped only if all 5 transmission attempts fail:

Pdrop = PER5

Drop Probability Examples

PER = 1% (0.01):
  P_drop = 0.01^5 = 10^-10  (one in 10 billion -- essentially zero)

PER = 10% (0.1):
  P_drop = 0.1^5 = 10^-5  (one in 100,000 -- very rare)

PER = 50% (0.5):
  P_drop = 0.5^5 = 0.03125 = 3.1%  (starting to hurt)

PER = 70% (0.7):
  P_drop = 0.7^5 = 0.168 = 16.8%  (severe packet loss)

PER = 90% (0.9):
  P_drop = 0.9^5 = 0.590 = 59%  (link is effectively down)

3. Goodput Calculation

Goodput is the useful data rate actually delivered to the application -- data rate minus overhead from retransmissions and dropped packets. It is the metric that matters to users.

Goodput = DataRate × (1 - Pdrop) / avg_transmissions

Where:

  • DataRate = raw throughput from the selected ModCod (spectral efficiency * bandwidth)
  • P_drop = PER5 (probability a packet is permanently lost)
  • avg_transmissions = average number of transmissions per delivered packet (including retries)

Goodput Example

ModCod: 16APSK 3/4  ->  Data rate: 742 Mbps
BER: 10^-6  ->  PER: 1.2%

P_drop = 0.012^5 = 2.5 * 10^-10  (negligible)
avg_tx = (1 - 0.012^5) / (1 - 0.012) = 1 / 0.988 = 1.012

Goodput = 742 * (1 - 0) / 1.012 = 733 Mbps
Overhead: only 1.2% from retransmissions. Excellent.

Now with BER = 10^-4 (SNR dropped 4 dB):
PER = 69.9%
P_drop = 0.699^5 = 0.168
avg_tx = (1 - 0.168) / (1 - 0.699) = 0.832 / 0.301 = 2.76

Goodput = 742 * (1 - 0.168) / 2.76 = 742 * 0.301 = 224 Mbps
Loss: 70% of throughput! This is the catastrophic regime.

The goodput equation captures two distinct losses: wasted bandwidth from retransmissions (each retry occupies channel time that could carry new data) and permanent drops (packets that never get through despite all retries). In the healthy regime (PER < 5%), retransmission overhead is small and drops are negligible. In the degraded regime (PER > 30%), both effects compound.

4. Latency Components

End-to-end packet latency in a LEO satellite link has several components, each varying differently over the course of a pass:

ComponentTypical ValueVariation
Propagation delay1.8 - 6 msVaries with slant range (elevation angle)
Processing delay~1 msNearly constant (onboard + ground)
Encoding/modulation~0.5 msConstant for a given ModCod
Retransmission delay0 - 25 ms0 if first attempt succeeds; adds one RTT per retry
Total (first attempt)3.3 - 7.5 msDominated by propagation

Propagation Delay

The speed of light gives the minimum latency floor. At zenith (550 km), the one-way delay is:

tprop = d / c = 550,000 m / 3 × 108 m/s = 1.83 ms

At low elevation (10 degrees), the slant range stretches to ~1800 km, giving tprop = 6.0 ms. The round-trip time (RTT) is double these values: 3.7 ms at zenith to 12.0 ms at low elevation.

Retransmission Delay

Each retry adds one full RTT to the packet's latency. If a packet requires 3 attempts (2 retries), its total latency is the one-way propagation plus 2 additional RTTs for the NACK-retransmit cycles. This is the dominant source of latency variability.

5. Jitter Sources

Jitter is the variation in packet arrival times. Even if the average latency is acceptable, high jitter degrades real-time applications (voice, video, gaming). The simulator models three distinct jitter sources:

Retransmission Jitter

The most significant source. Packets that arrive on the first attempt have latency ~3 ms. Packets requiring 2 retries have latency ~9 ms. This creates bimodal (or multimodal) arrival time distributions. When PER is moderate (~5-10%), most packets arrive quickly, but a fraction arrive significantly later, creating jitter spikes.

Compute Scheduling Jitter

The onboard processor handles multiple tasks (routing, beamforming, ISL management, housekeeping). When the CPU is heavily loaded, packet processing can be delayed by 0.1-1.0 ms. In power Mode 1 (compute reduced 30%), this jitter increases because the same workload runs on fewer resources.

Doppler Tracking Transients

The phased-locked loop (PLL) that tracks Doppler shift is not instantaneous. When the Doppler rate changes rapidly (near TCA -- time of closest approach), the tracking loop experiences transient errors. These transients can briefly degrade demodulation performance, causing isolated packet errors and variable processing delays.

6. Adaptive Modulation with Hysteresis

The satellite continuously measures channel SNR and selects the highest ModCod that the channel can support, with a 1.5 dB hysteresis margin. This margin prevents rapid oscillation when the SNR hovers near a threshold.

How It Works During a Pass

As a satellite rises from the horizon:

  1. Low elevation (10-20 deg): Long slant range, high path loss, atmospheric loss, ground noise. SNR might be only 5-8 dB. System uses QPSK 3/4 or 8PSK 2/3.
  2. Mid elevation (30-50 deg): Shorter range, less atmospheric path. SNR improves to 12-15 dB. System steps up to 16APSK 3/4 or 5/6.
  3. Near zenith (60-90 deg): Minimum range, maximum antenna gain. SNR peaks at 16-20 dB. System uses 32APSK or 64APSK for maximum throughput.
  4. Descending: Mirror of the ascent. SNR drops, ModCod steps down.

Notice that the ModCod step-up and step-down points are asymmetric: the system steps up to a higher ModCod only when SNR exceeds the threshold by 1.5 dB, but steps down as soon as SNR drops below the threshold. This prevents the rapid "ModCod flapping" that would occur without hysteresis -- analogous to the power mode hysteresis in Unit 6.

7. Per-Packet Monte Carlo Simulation

The simulator does not use average statistics. Instead, it simulates every individual packet using Monte Carlo methods with a seeded pseudo-random number generator (PRNG).

Per-Packet Process

  1. Compute BER from the current SNR and selected ModCod
  2. Compute PER from BER and packet size
  3. Roll the dice: generate a random number in [0,1]. If random < PER, the packet is corrupted
  4. If corrupted: increment retry counter. If retries < 4, go to step 3 with a new random draw
  5. If max retries exceeded: mark packet as dropped
  6. If delivered: record latency (propagation + processing + retry delays)
  7. Record the causal chain: snapshot of all hardware state at transmission time

Why Seeded PRNG?

The PRNG seed ensures reproducibility. Running the same simulation with the same seed produces identical packet-level results. This is essential for debugging: if a user reports throughput collapse at T=240s, the developer can replay the exact same simulation and inspect every packet decision at that time. Without deterministic randomness, debugging stochastic behavior would be nearly impossible.

// Simplified per-packet simulation
function simulatePacket(state: SimState, prng: PRNG): PacketRecord {
  const ber = computeBER(state.snr_linear, state.modcod);
  const per = 1 - Math.pow(1 - ber, PACKET_BITS);

  let attempts = 0;
  let delivered = false;
  while (attempts < MAX_RETRIES + 1) {
    attempts++;
    if (prng.next() >= per) {
      delivered = true;
      break;
    }
  }

  return {
    delivered,
    attempts,
    latency: computeLatency(state, attempts),
    causalChain: snapshotState(state),
  };
}

8. The Causal Chain

Every PacketRecord in the simulator carries a causalChain object -- a frozen snapshot of the entire hardware state at the moment the packet was transmitted. This is the most powerful debugging tool in the system.

The causal chain captures:

  • Orbital state: elevation, azimuth, range, Doppler shift
  • Antenna state: scan angle, realized gain, mutual coupling loss
  • RF chain: PA power, backoff, phase noise, EVM
  • Thermal: PA junction temperature, radiator temperature
  • Power: SoC, power mode, PA allowed power
  • Link budget: EIRP, FSPL, atmospheric loss, C/N, effective SNR
  • Modulation: selected ModCod, BER, PER
  • Protocol: attempt number, retry count, latency breakdown
Why this matters: When you see a dropped packet, you can inspect its causal chain and trace exactly why it was dropped. For example: "This packet was dropped because BER=10-4, which was caused by SNR=8 dB, which was caused by EIRP being 3 dB low because the PA was in Mode 2 (8W cap) because SoC=27% because the satellite had been in eclipse for 30 minutes with a degraded battery." Every metric is a downstream consequence of hardware state.

This is the culmination of the "hardware-first" philosophy: the simulator does not generate random packet losses. Every packet outcome is a deterministic consequence of the physical state of the spacecraft and the channel. Nothing is independent; everything is causal.

9. Code Connection

The protocol layer lives in:

  • src/simulation/protocol/packet-layer.ts -- The per-packet Monte Carlo simulation. Implements the ARQ retry loop, PER computation, latency calculation, jitter modeling, and causal chain snapshots. The main function simulatePackets(state, count, prng) generates a batch of packet records for each simulation tick.
  • src/simulation/modulation/modcod.ts -- The selectModCod(snr_dB) function that implements adaptive modulation with the 1.5 dB hysteresis margin. Called at the beginning of each tick to determine the current ModCod before packet simulation.

Key integration point:

// In the main simulation tick loop:
function tick(state: SimState, prng: PRNG): TickResult {
  // 1. Update orbital position (Unit 1)
  updateOrbit(state);

  // 2. Update power state (Unit 6)
  stepPower(state);

  // 3. Update thermal state (Unit 4)
  stepThermal(state);

  // 4. Compute antenna gain (Unit 2)
  updateAntennaGain(state);

  // 5. Compute RF chain (PA, phase noise) (Unit 3)
  updateRFChain(state);

  // 6. Compute link budget (Unit 5)
  computeLinkBudget(state);

  // 7. Select ModCod (Unit 7)
  state.modcod = selectModCod(state.effectiveSNR);

  // 8. Simulate packets (Unit 8 -- this unit)
  const packets = simulatePackets(state, packetsPerTick, prng);

  // 9. Aggregate metrics
  return aggregateResults(packets);
}

The tick loop shows the full causal chain in code: orbital position drives antenna pointing drives RF chain drives link budget drives modulation selection drives packet simulation. Each step depends on the outputs of previous steps. Nothing is computed in isolation.
