Unit 7

Fault Injection & Cross-Layer Causality

Real satellites degrade. Components fail. Temperatures spike. Power budgets shrink. This unit covers how faults propagate through the physical layers of a spacecraft and why understanding these causal chains is the difference between a textbook exercise and real satellite engineering.

Learning Objectives

  • Identify the 10 fault categories modeled in the simulator and their severity ranges
  • Trace cascade effects from a hardware fault to application-layer goodput
  • Explain why throughput = f(elevation) is fundamentally wrong
  • Describe the 10-step causal simulation tick loop
  • Perform root-cause analysis: given a symptom, work backward to identify the fault
  • Compare clean vs degraded satellite metrics at the same elevation angle

1. Why Fault Analysis Matters

Satellites operate in one of the harshest environments known to engineering: vacuum, extreme thermal cycling (-150 C to +150 C), radiation, micrometeorite bombardment, and zero accessibility for repair. Components degrade over time. Solar panels lose efficiency. Amplifiers drift. Oscillators age.

The cost of not understanding failure modes is catastrophic: a $200M satellite becomes a piece of space debris. Even for LEO constellations like Starlink, where individual satellites are cheaper and replaceable, understanding degradation patterns is essential for:

  • Capacity planning -- knowing how many satellites are operating below spec
  • Anomaly detection -- distinguishing normal variation from incipient failure
  • Graceful degradation -- keeping the satellite useful even with partial failures
  • Design improvement -- feeding failure data back into the next generation

4. The Key Insight: Independent Curves Are Wrong

The Fundamental Problem with Simple Models

Most satellite simulators model throughput as a function of elevation angle: throughput = f(elevation). You look up the elevation, you get the throughput. This is fundamentally wrong.

Why? Because it treats the satellite as a black box. It assumes a fixed relationship between geometry and performance. In reality, throughput emerges from hardware state. The same elevation angle can produce wildly different throughput depending on:

  • PA temperature (affects EIRP by 0-6 dB)
  • Element health (affects antenna gain by 0-4 dB)
  • Battery state (affects available power by 0-3 dB)
  • Oscillator stability (affects tracking penalty by 0-3 dB)
  • Thermal state (affects noise figure by 0-2 dB)

Combined, hardware state can swing throughput by 10-15 dB at the same elevation. That is the difference between 64QAM and QPSK -- a 3x difference in data rate.

This is the core reason the simulator exists: to demonstrate that throughput is an emergent property of physics and hardware state, not a lookup table indexed by elevation.

Think about it this way: two Starlink satellites cross overhead at the same elevation. One has fresh solar panels, a cool PA, all elements working, and a stable oscillator. The other has 20% degraded panels, is coming out of eclipse with a warm PA, has 5% failed elements, and a drifting oscillator. The first delivers 900 Mbps. The second delivers 200 Mbps. Same geometry. Completely different outcome.

5. The Simulation Tick Loop

Every simulation tick (default: 1 second), the engine executes a 10-step causal chain. Each step produces outputs that feed the next. The order is critical -- it reflects the actual physics of the system.

The order matters. You cannot compute the link budget (step 8) without knowing the PA output power (step 6), which depends on thermal state (step 4), which depends on power budget (step 3) and fault state (step 2), which depends on orbital position (step 1).

6. Root Cause Analysis

Root cause analysis (RCA) is the inverse of the simulation loop: you start with a symptom at the output and trace backward to find the cause at the input. This is the core skill the simulator teaches.

Example: Goodput Dropped at T+247s

Observation: goodput dropped from 900 Mbps to 350 Mbps at T+247s. Elevation was 52 degrees (still high). What happened?

Step backward through the causal chain:

Step 10: Goodput dropped
  -> because ModCod dropped from 64QAM 3/4 to QPSK 3/4

Step 9: ModCod dropped
  -> because effective_SNR fell from 28 dB to 8 dB (-20 dB!)

Step 8: SNR fell
  -> because EIRP dropped -6 dB AND Doppler penalty spiked +2 dB

Step 6: EIRP dropped
  -> because PA backed off due to thermal protection

Step 4: Thermal protection triggered
  -> PA temperature hit 85 C (threshold)
  -> Radiator efficiency at 45% (fault injected at T+200s)

Step 7: Also, scan angle was 63 degrees (additional -3 dB)

ROOT CAUSE: radiator degradation fault (T+200s) combined
with high scan angle created a compound failure.

At 52 deg elevation, a healthy satellite would have
SNR ~28 dB. With the fault, SNR = ~8 dB.
Same elevation, 20 dB difference.

The analysis pattern:

  1. Start at the symptom (goodput, ModCod, BER)
  2. Look at the immediate upstream metric (SNR for ModCod, EIRP + G/T for SNR)
  3. For each degraded upstream metric, look at its upstream metric
  4. Continue until you reach a fault or a geometric cause (elevation, range)
  5. Identify which faults and which geometric factors combined to produce the symptom

The element failures (-4 dB antenna gain) plus power limitation (-5 dB PA output) plus compute overload (+1.6 dB penalty) combine to push SNR below the 64QAM threshold, forcing a fall to QPSK 1/2.

9. Code Connection

The fault injection and simulation engine live in:

  • src/simulation/faults/fault-scheduler.ts -- Manages the fault timeline. Stores fault definitions (type, severity, start time, duration). At each tick, determines which faults are active and passes them to the applicator.
  • src/simulation/faults/fault-applicator.ts -- Takes active faults and applies their effects to the appropriate simulation state. Element failures modify the antenna model, PA degradation modifies the RF chain, thermal faults modify the thermal model, etc.
  • src/simulation/engine.ts -- The main tick loop that implements the 10-step causal chain. Orchestrates the calls to orbit, faults, power, thermal, oscillator, PA, antenna, link budget, modulation, and protocol modules in the correct order.

The engine.ts tick function structure:

function simulateTick(state: SimState, dt: number): SimState {
  // 1. Update orbit (position, velocity, elevation, range)
  const orbit = updateOrbit(state.orbit, dt);

  // 2. Check and apply faults
  const faults = scheduleFaults(state.faults, state.time);
  const faultState = applyFaults(faults, state);

  // 3. Power bus (solar generation, battery, bus voltage)
  const power = updatePower(orbit, faultState.power, dt);

  // 4. Thermal (PA temp, radiator, sun/eclipse)
  const thermal = updateThermal(orbit, power, faultState.thermal, dt);

  // 5. Oscillator (frequency stability, phase noise)
  const osc = updateOscillator(thermal, faultState.oscillator);

  // 6. PA / RF chain (output power, backoff)
  const pa = updatePA(power, thermal, faultState.pa);

  // 7. Antenna (gain, scan loss, element pattern)
  const antenna = updateAntenna(orbit, faultState.antenna);

  // 8. Link budget (EIRP, FSPL, C/N, effective SNR)
  const link = computeLinkBudget(pa, antenna, orbit, osc, thermal);

  // 9. Modulation (ModCod selection, BER)
  const mod = selectModulation(link, state.modulation);

  // 10. Protocol (PER, ARQ, goodput, jitter)
  const proto = computeProtocol(mod, link);

  return { orbit, faults, power, thermal, osc, pa, antenna, link, mod, proto };
}

Each step function is a pure computation: it takes its inputs and produces outputs without side effects. This makes the simulation deterministic and debuggable -- you can snapshot any tick and inspect every intermediate value.

Questions

  1. Cascade tracing. A solar panel suffers 30% damage at T+60s. Trace the cascade through the 10-step simulation loop. Which steps are affected? At which step does the throughput impact first become visible? Estimate the approximate dB loss at each affected step.
  2. Independent curves critique. A colleague proposes modeling Starlink throughput as throughput = 1000 * sin(elevation)^1.5 Mbps. Give three specific scenarios where this model would produce wrong results, and estimate how wrong it would be in each case.
  3. Fault compounding. Which two faults from the table, if they occur simultaneously, would produce the worst throughput degradation? Explain the cascade that makes them interact (i.e., the compound effect is worse than the sum of individual effects).
  4. Root-cause tracing exercise. You observe the following at T+312s (elevation 38 degrees): goodput = 50 Mbps (expected ~700 Mbps), ModCod = QPSK 1/4, BER = 2e-4, effective_SNR = -0.2 dB. The PA output power is nominal. The antenna gain is nominal. What fault or combination of faults could cause this? Work backward through the causal chain.
  5. Tick ordering. Explain why the thermal step (step 4) must come after the power step (step 3) but before the PA step (step 6). What would go wrong if you swapped the order of steps 4 and 6?
  6. Design exercise. You are designing a fault-tolerant Starlink satellite. You have budget for one redundancy measure: (a) dual-redundant PA, (b) oversized radiators (2x area), or (c) extra battery capacity (50% more). Using the cascade analysis from this unit, argue which single measure provides the most protection against throughput degradation. Consider which fault categories each measure mitigates.
