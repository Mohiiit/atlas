Unit 11 -- Capstone

Cross-Layer Causality & Root Cause Analysis

Why independent throughput curves are wrong and how to debug from hardware up.

Learning Objectives

  • Articulate why throughput = f(elevation) is fundamentally incomplete for satellite communications
  • Enumerate the 10-step simulation tick loop and explain each step's data dependencies
  • Explain why steps 3-4 use the previous tick's state to break circular dependencies
  • Perform a root cause analysis: starting from a goodput anomaly, trace backward through the causal chain to identify hardware root causes
  • Describe the causalChain field in PacketRecord and its role in per-packet attribution
  • Predict emergent behaviors from interacting faults that no single-layer model would capture

11.1 The Fundamental Insight

Most satellite communication simulators model throughput as a function of elevation angle: as the satellite rises above the horizon, throughput climbs; at TCA (Time of Closest Approach) it peaks; then it falls as the satellite sets. The shape is smooth, predictable, and wrong.

This model treats the satellite as a black box. It assumes that given the same geometry, the satellite will always produce the same throughput. But reality is different: two passes at the same elevation can produce wildly different throughput depending on:

  • PA temperature: If the PA has been transmitting for 3 minutes, it is 30°C hotter than at pass start. Output power is lower, EVM is higher (Unit 3, Unit 4).
  • Battery state: During eclipse or after sustained high-power operation, the battery SoC may trigger power-saving modes that reduce PA drive (Unit 4).
  • Element health: Degraded array elements reduce gain and increase sidelobes regardless of elevation (Unit 2).
  • Oscillator stability: Frequency drift and phase noise depend on thermal history, not geometry (Unit 9).
  • Fault state: Active faults inject hardware degradation that has nothing to do with the pass geometry (Unit 10).
The core problem with elevation-based models: They conflate correlation with causation. Throughput correlates with elevation because elevation determines path loss -- but throughput also depends on a dozen hardware states that are invisible to geometry. The simulator exists to make these hidden variables visible.
Figure 11.1 -- The dashed gray curve is what an elevation-only model predicts: smooth, symmetric, deterministic. The cyan curve is the simulator output with hardware coupling and faults: jagged, asymmetric, with dips and recoveries caused by thermal throttling, attitude glitches, and oscillator effects. Same orbital pass, same geometry -- different hardware states produce different throughput.

11.2 The 10-Step Simulation Tick Loop

Every second of simulated time, the simulation engine executes 10 steps in a specific order. The order is not arbitrary -- it is determined by the data dependencies between subsystems. Each step consumes outputs from previous steps and produces inputs for subsequent ones.

1

Orbit Geometry

SGP4 propagation computes satellite position. From position + ground station location, derive: elevation, azimuth, slant range, Doppler shift, and the required beam steering angle.

Inputs: TLE, current time, ground station coordinates | Outputs: elevation, azimuth, range, steeringAngle

2

Fault Engine

Check the fault schedule against current time. Activate, maintain, or deactivate faults. Apply fault effects to component states: disable elements, reduce P1dB, drop emissivity, set PLL unlock flag, etc.

Inputs: fault schedule, current time | Outputs: modified component states (antenna, PA, oscillator, thermal, power)

3

Power Bus

Compute solar panel output (based on sun angle and panel health), battery charge/discharge, and available bus power. Determine if power-saving modes are needed. Uses previous tick's thermal state to compute panel efficiency degradation from temperature.

Inputs: sun angle, battery SoC, panel faults, prev thermal state | Outputs: busVoltage, availablePower, batteryState, powerMode

4

Thermal Model

Lumped thermal model: compute heat generation from each component, radiative cooling (Stefan-Boltzmann), conductive coupling between nodes. Uses previous tick's PA heat dissipation to break the PA-temperature circular dependency.

Inputs: radiator state, prev PA heat, sun flux, eclipse state | Outputs: paTemp, oscTemp, batteryTemp, busTemp

5

Oscillator

Compute frequency offset from thermal drift model (Unit 9). Determine PLL lock state. Calculate phase noise including temperature degradation and unlock penalties.

Inputs: oscTemp (from step 4), fault flags, PLL state | Outputs: freqOffset, phaseNoise, pllLocked, oscEvm

6

Power Amplifier

Rapp model: compute output power at current operating point, accounting for thermal derating (paTemp from step 4), available bus power (step 3), and PA faults (step 2). Calculate AM/AM compression, AM/PM distortion, and PA contribution to EVM.

Inputs: paTemp, availablePower, PA faults, backoff setting | Outputs: outputPower, paEvm, heatDissipated

7

Phased Array

Compute antenna gain at the current steering angle (from step 1). Apply scan loss, mutual coupling degradation, and element failure effects (from step 2). Calculate effective beam pattern.

Inputs: steeringAngle, active elements, element faults | Outputs: antennaGain, scanLoss, effectivePattern

8

Link Budget

The central calculation (Unit 5). EIRP = PA output + antenna gain. Path loss from slant range. Atmospheric attenuation from elevation angle. Receiver G/T. Compute C/N, then SNR including all penalties: EVM from PA, EVM from oscillator, pointing loss.

Inputs: outputPower, antennaGain, range, elevation, paEvm, oscEvm | Outputs: eirp, pathLoss, snr, effectiveSNR

9

Modulation Selection

DVB-S2X adaptive modulation: select the highest ModCod whose SNR threshold is met with the required margin (Unit 6). Compute spectral efficiency and theoretical BER.

Inputs: effectiveSNR, channel bandwidth | Outputs: modCod, spectralEfficiency, theoreticalBER, bitsPerSymbol

10

Protocol Layer

Packet formation, FEC encoding, transmission. Apply BER to compute packet error rate. ARQ retransmissions. Compute goodput = gross bitrate × (1 - overhead) × (1 - packetLossRate). Build causalChain record.

Inputs: modCod, spectralEfficiency, BER, scheduling jitter | Outputs: goodput, packetsSent, packetsLost, causalChain

Why order matters: If you computed the PA output before knowing the temperature (swapping steps 4 and 6), the PA model would use stale thermal data and miss thermal throttling. If you computed the link budget before the antenna gain (swapping steps 7 and 8), you would have no gain value to use. The simulation's causal fidelity depends on this ordering.

11.3 Breaking Circular Dependencies

There is a fundamental circular dependency in the simulation: the PA generates heat that affects temperature, but temperature determines how much power the PA can output, which determines how much heat it generates. In continuous time, this is a differential equation. In discrete simulation ticks, we must break the cycle.

The solution: steps 3 (Power) and 4 (Thermal) use the previous tick's PA heat dissipation to compute the current tick's temperature. Then steps 5-6 use the current tick's temperature to compute the new PA output and heat. This introduces a one-tick (1 second) delay in the thermal feedback loop.

T[t] = f(T[t-1], QPA[t-1], Qradiator[t-1])
PPA[t] = g(T[t], Vbus[t], faults[t])
QPA[t] = PPA[t] × (1 - η)

This is the standard approach in real-time simulation (Euler integration with explicit thermal coupling). The 1-second delay is physically justified: the thermal mass of the PA heat sink and the thermal bus means that heat generated at time t does not instantaneously change the temperature -- it takes seconds to propagate. The simulation's 1-second tick rate is well within the thermal time constant (~30-60 seconds).

Stability criterion: This explicit integration scheme is stable as long as the tick rate is much shorter than the system's fastest thermal time constant. With a 30-second PA thermal time constant and a 1-second tick, we have a 30x margin. If you tried to run the simulation at 1 tick per 60 seconds, the thermal feedback could become unstable.

11.5 A Debugging Walkthrough

This is the core skill of hardware-first satellite communications engineering: given an anomaly in the output, trace backward through the causal chain to find the root cause. Let us work a concrete example.

The Symptom

At T+247s into a pass, goodput drops from 120 Mbps to 40 Mbps -- a 67% collapse. The pass elevation at T+247s is still 58° (well above the 10° minimum), so geometry alone cannot explain this.

The Investigation

Step 1: What did the ModCod do?
The modulation dropped from 16QAM 3/4 (4.44 bits/symbol) to QPSK 1/2 (1.0 bits/symbol) at T+245s. That is a 4.4x spectral efficiency reduction, which accounts for the ~3x throughput drop (the remaining factor comes from increased retransmissions). The ModCod selection was responding to an SNR threshold crossing.
Step 2: What happened to SNR?
Effective SNR dropped from 18.2 dB to 10.1 dB -- an 8.1 dB collapse. 16QAM 3/4 requires 16.5 dB; QPSK 1/2 requires 6.7 dB. The SNR fell through the 16QAM threshold and landed with margin only for QPSK. Now we need to find the 8.1 dB.
Step 3: Where did the 8.1 dB go?
Breaking down the SNR components at T+247s vs T+240s (before the drop):
-- EIRP dropped by 3.2 dB (from 44.1 to 40.9 dBW)
-- Free-space path loss increased by 0.3 dB (elevation changed slightly)
-- EVM penalty increased by 4.6 dB (from 1.1 to 5.7 dB effective SNR penalty)
Total: 3.2 + 0.3 + 4.6 = 8.1 dB. Accounted for.
Step 4: Why did EIRP drop 3.2 dB?
PA output power fell from 9.8 W to 4.7 W. The PA temperature hit 85°C, triggering the thermal throttle derating curve. Additionally, antenna gain dropped 0.5 dB because the steering angle increased as the satellite moved past TCA. Root cause #1: PA thermal throttling.
Step 5: Why did EVM spike 4.6 dB?
Two contributing factors:
-- PA EVM increased from 3.2% to 5.8% because the PA operating point shifted closer to compression as output power was throttled (the backoff decreased).
-- Oscillator EVM increased from 0.5% to 4.1% because the oscillator temperature reached 62°C, causing 180 kHz of frequency drift. The PLL tracking error grew to 15% of the loop bandwidth.
RSS total EVM: sqrt(5.8² + 4.1²) = 7.1%, translating to the 5.7 dB SNR penalty. Root cause #2: Oscillator thermal drift (coupled to PA heating).

Root Cause Summary

The 120 → 40 Mbps collapse was caused by two coupled hardware effects:

  1. PA thermal throttling at 85°C reduced EIRP by 3.2 dB (direct power loss).
  2. Oscillator thermal drift from PA heating increased EVM by 4.6 dB (signal quality degradation).

Both root causes trace to the same physical origin: accumulated PA heat after 4 minutes of high-power transmission. An elevation-only model would have predicted 120 Mbps at 58° elevation. The hardware-coupled simulation correctly predicts 40 Mbps because it tracks the thermal history.

Figure 11.3 -- Root cause tree for the T+247s goodput collapse. The investigation branches at the SNR level (EIRP vs EVM), then converges on a common root cause: PA accumulated heat over 4+ minutes of operation.

11.6 Per-Packet Causality: The causalChain Record

Every packet generated by the simulator carries a causalChain field -- a snapshot of the complete hardware state at the moment of transmission. This enables per-packet root cause analysis: you can select any packet and see exactly what every subsystem was doing when that packet was created. 

interface CausalChain {
  // Geometry (Step 1)
  elevation: number;         // degrees
  steeringAngle: number;    // degrees from boresight
  slantRange: number;       // km

  // Active faults (Step 2)
  activeFaults: string[];   // list of active fault types

  // Power (Step 3)
  busVoltage: number;       // V
  batterySoC: number;       // 0-1
  powerMode: string;        // 'nominal' | 'reduced' | 'survival'

  // Thermal (Step 4)
  paTemp: number;           // °C
  oscTemp: number;          // °C

  // Oscillator (Step 5)
  freqOffset: number;       // Hz
  pllLocked: boolean;
  oscEvm: number;           // % from oscillator

  // PA (Step 6)
  paOutputPower: number;    // W
  paEvm: number;            // % from PA compression

  // Antenna (Step 7)
  antennaGain: number;      // dBi
  scanLoss: number;         // dB
  activeElements: number;

  // Link Budget (Step 8)
  eirp: number;             // dBW
  pathLoss: number;         // dB
  effectiveSNR: number;     // dB (after all penalties)

  // Modulation (Step 9)
  modCod: string;           // e.g., '16QAM 3/4'
  spectralEfficiency: number; // bits/symbol
}

This data structure is the key to the simulator's educational value. Instead of just seeing "throughput dropped," a student can inspect the causalChain and see exactly which hardware parameters contributed to the drop. The CausalityView component in the UI visualizes this data as an interactive waterfall.

Every field corresponds to a tick-loop step. The causalChain is assembled at step 10 by collecting outputs from all previous steps. It is a complete cross-section of the system state at one instant in time. Comparing causalChain records from two adjacent seconds reveals exactly what changed and by how much.

11.7 Emergent Behavior from Interacting Faults

The most valuable insight from hardware-coupled simulation is that interacting faults produce behavior that no single-fault analysis would predict. Here are three examples:

Triple Degradation: Thermal + Power + Geometry

Consider a satellite late in its pass (elevation falling, 250s after AOS) with:

  • Radiator degradation (severity 0.3): all temperatures are 15°C above nominal
  • Battery degradation (severity 0.3): power mode has shifted to "reduced," cutting PA drive by 20%
  • High steering angle (50°): scan loss is 4 dB from the phased array

Each factor independently might cost 2-4 dB of SNR. But together, they interact:

  • The PA is thermally derated AND power-limited simultaneously, so output power drops more than either effect alone.
  • The reduced PA output means less heat generation, which slightly helps thermal -- but the reduced radiator emissivity means the thermal benefit is smaller than expected.
  • The high scan angle means the antenna gain is already low, so the EIRP drop from PA derating has less "headroom" to absorb before crossing the ModCod threshold.

Result: each factor alone might cause a 1 ModCod step down. Together, they cause a 3-step drop, producing less than 25% of the expected throughput. This is not additive -- it is multiplicative degradation in the dB domain.

Recovery Oscillation

When a PA thermally throttles, it reduces output power, which reduces heat generation, which allows the temperature to drop, which allows the PA to increase power, which increases heat generation again. This can produce oscillation in throughput with a period of 30-60 seconds (the thermal time constant).

An elevation-only model shows a smooth curve. The hardware-coupled simulation shows throughput oscillating by 20-30% as the PA hunts between thermal throttle and full power. This is a real phenomenon observed in satellite communications systems.

Eclipse Transition Cliff

When the satellite enters eclipse, solar panel output drops to zero. The battery takes over, but if the battery is degraded (fault #7), the bus voltage sags. This causes:

  1. PA drive is reduced (power-limited operation)
  2. Oscillator compensation circuit voltage changes, causing a frequency jump
  3. PLL must track the sudden frequency change while simultaneously losing 3-4 dB of EIRP
  4. If the PLL bandwidth is marginal, it can lose lock entirely

The result is a "cliff" in throughput at eclipse entry that is much steeper than the gradual EIRP reduction from power limitation alone. Without the oscillator coupling, you would predict a smooth ~3 dB degradation. With it, you get a potential complete link loss for several hundred milliseconds.

This is why the simulator exists. No amount of spreadsheet link-budget analysis would predict the eclipse transition cliff, the PA thermal oscillation, or the triple degradation interaction. These are emergent properties of a coupled system that only appear when all subsystems are simulated together at sufficient time resolution.

11.8 Code Connection

src/simulation/engine.ts -- The Tick Loop

The simulation engine implements the 10-step tick loop. The core structure: 

function simulateTick(
  prevState: SimulationState,
  time: number,
  faults: FaultConfig[]
): TickResult {
  // Step 1: Geometry
  const geo = computeGeometry(time, prevState.tle, prevState.groundStation);

  // Step 2: Faults
  const faulted = applyFaults(prevState, faults, time);

  // Step 3: Power (uses PREV thermal)
  const power = computePower(faulted.power, prevState.thermal);

  // Step 4: Thermal (uses PREV PA heat)
  const thermal = computeThermal(
    faulted.thermal,
    prevState.pa.heatDissipated,  // <-- previous tick!
    geo.inEclipse
  );

  // Step 5: Oscillator
  const osc = computeOscillator(faulted.osc, thermal.oscTemp);

  // Step 6: Power Amplifier
  const pa = computePA(faulted.pa, thermal.paTemp, power.available);

  // Step 7: Phased Array
  const ant = computeAntenna(faulted.antenna, geo.steeringAngle);

  // Step 8: Link Budget
  const link = computeLinkBudget(pa, ant, osc, geo);

  // Step 9: Modulation
  const mod = selectModCod(link.effectiveSNR);

  // Step 10: Protocol + causalChain
  const proto = computeProtocol(mod, faulted.compute);
  const chain = buildCausalChain(geo, power, thermal, osc, pa, ant, link, mod);

  return { geo, power, thermal, osc, pa, ant, link, mod, proto, chain };
}

Note lines 3-4: computePower receives prevState.thermal and computeThermal receives prevState.pa.heatDissipated. This is the explicit one-tick delay that breaks the circular dependency discussed in Section 11.3.

src/components/causality/CausalityView.tsx

The UI component that visualizes the causal chain. It renders an interactive waterfall chart showing each contribution to the final SNR, and highlights which factors changed most between selected time steps: 

// Simplified structure of the causality visualization
function CausalityView({ tickData }: Props) {
  const { chain } = tickData;

  // Build waterfall: EIRP contribution breakdown
  const waterfall = [
    { label: 'PA Output',    value: toDbw(chain.paOutputPower) },
    { label: 'Antenna Gain',  value: chain.antennaGain },
    { label: 'Scan Loss',     value: -chain.scanLoss },
    { label: '= EIRP',        value: chain.eirp, isTotal: true },
    { label: 'Path Loss',     value: -chain.pathLoss },
    { label: 'Atm Loss',      value: -chain.atmosphericLoss },
    { label: 'Rx G/T',        value: chain.rxGT },
    { label: 'EVM Penalty',   value: -chain.evmPenalty },
    { label: '= Eff SNR',     value: chain.effectiveSNR, isTotal: true },
  ];

  return <WaterfallChart data={waterfall} />;
}
