Unit 6

Digital Modulation & the Protocol Stack

The link budget from Unit 5 gives us a signal-to-noise ratio. This unit answers: what do we do with that SNR? We encode bits onto carrier waves, protect them with error correction, adapt the scheme to channel conditions, and push packets through a protocol stack that converts raw bits into useful throughput.

Learning Objectives

  • Explain how QPSK, 8PSK, 16QAM, and 64QAM map bits to symbols
  • Read a constellation diagram and understand why higher-order modulation needs more SNR
  • Calculate BER from Eb/N0 using the erfc function and interpret BER curves
  • Describe how FEC (LDPC codes) creates a coding gain waterfall
  • Explain Adaptive Coding and Modulation (ACM) and hysteresis
  • Compute PER from BER and understand the packet cliff effect
  • Trace the path from raw bit rate to goodput through overhead and retransmissions

1. Why Digital Modulation?

A satellite downlink transmits a carrier wave -- a sinusoid at 12 GHz. To carry information, we must vary some property of that wave. Digital modulation encodes discrete bit patterns by changing the carrier's:

  • Amplitude (ASK) -- vary the height of the wave
  • Phase (PSK) -- rotate the wave's starting angle
  • Both (QAM) -- combine amplitude and phase changes

Each distinct combination of amplitude and phase is a symbol. With M symbols, each symbol carries log2(M) bits. The fundamental tradeoff: more symbols = more bits per symbol = higher data rate, but the symbols are closer together and more vulnerable to noise.

2. QPSK -- Quadrature Phase Shift Keying

QPSK uses 4 symbols, each separated by 90 degrees in phase. Each symbol carries 2 bits. It is the workhorse of satellite communications because of its robustness to noise.

Why is QPSK so robust? The symbols are maximally separated. Noise must push a received symbol more than 70% of the way to a neighbor before an error occurs. At the same power, QPSK has the same BER performance as BPSK (because they have the same minimum distance per bit of energy).

At a symbol rate of 240 Msym/s, QPSK delivers: 240 × 2 = 480 Mbps raw.

3. Higher-Order Modulation

To increase data rate without more bandwidth, we pack more bits per symbol. But the symbols crowd together, requiring higher SNR to distinguish them.

4. BER and the erfc Function

The Bit Error Rate (BER) is the probability that a received bit is wrong. For QPSK and M-QAM, BER is computed from Eb/N0 (energy per bit to noise spectral density ratio) using the complementary error function:

BERQPSK = 0.5 · erfc( sqrt(Eb/N0) )

BERM-QAM ≈ (2/log2M) · (1 - 1/sqrt(M)) · erfc( sqrt( 3·log2M / (2(M-1)) · Eb/N0 ) )

What is Eb/N0? It is the SNR normalized per bit. If C/N is the total SNR in the bandwidth:

Eb/N0 = C/N + 10·log10(BW / Rbit)   [dB]

Where BW is noise bandwidth and Rbit is the bit rate. Higher modulation orders require more Eb/N0 to achieve the same BER.

Key observation: the BER curves are steep. A 1-2 dB change in Eb/N0 can shift BER by an order of magnitude. This is why the link budget must be accurate -- small errors in the loss calculation have dramatic effects on error rate.

5. Forward Error Correction (FEC)

Forward Error Correction adds redundant bits that let the receiver detect and correct errors without retransmission. The DVB-S2X standard used by Starlink employs LDPC (Low-Density Parity-Check) codes.

How FEC Helps

A code rate of r = 3/4 means 3 out of every 4 transmitted bits carry information; 1 in 4 is redundancy. The receiver uses this redundancy to correct errors.

The coding gain is the reduction in required Eb/N0 to achieve a target BER. Modern LDPC codes achieve 1-2 dB from the Shannon limit:

Code RateRedundancyCoding Gain (approx)Throughput Penalty
1/475%~9 dB4x overhead
1/250%~7 dB2x overhead
2/333%~5 dB1.5x overhead
3/425%~4 dB1.33x overhead

The FEC "waterfall" effect: below a threshold Eb/N0, virtually every block has errors. Above that threshold, errors drop precipitously -- a cliff from BER ~ 10-2 to 10-10 in just 1-2 dB. This is the hallmark of modern iterative codes.

DVB-S2X Code Rates

DVB-S2X defines code rates from 1/4 to 9/10. In LEO satellite systems like Starlink, the typical operating range is 1/4 to 3/4. Higher code rates (5/6, 9/10) are rarely used because the channel conditions change too fast -- by the time you adapt, the channel has moved.

6. Adaptive Modulation and Coding (ACM)

ACM dynamically selects the modulation and code rate (the "ModCod") based on the current SNR. When the link is strong (satellite overhead), use 64QAM 3/4 for maximum throughput. When the link weakens (satellite near horizon), fall back to QPSK 1/4 for robustness.

Hysteresis

Without hysteresis, the system would oscillate between ModCods when the SNR is near a threshold. Hysteresis adds an asymmetric dead band: step up requires SNR > threshold + 1 dB, step down triggers at SNR < threshold - 0.5 dB. This prevents rapid toggling.

7. Packet Layer

The physical layer delivers bits. The packet layer organizes them into 1500-byte (12,000-bit) packets (standard MTU). Even a tiny BER creates a significant Packet Error Rate (PER):

PER = 1 - (1 - BER)12000

This is the "packet cliff" effect: once BER exceeds a threshold, PER shoots to nearly 100%.

BERPERMeaning
10-80.0012%Essentially error-free
10-61.2%Occasional packet loss
10-511.3%Noticeable degradation
10-469.9%Severe -- most packets corrupt
10-399.99%Link effectively down

Notice the sharp transition between 10-6 and 10-4: the link goes from "mostly works" to "mostly broken" in just 20 dB of BER change, which corresponds to only about 3-4 dB of Eb/N0 change. This is why the FEC waterfall and ACM thresholds are so critical.

8. ARQ Retransmissions

Automatic Repeat Request (ARQ) handles the packets that FEC cannot correct. When a packet fails its CRC check, the receiver requests retransmission.

  • Maximum retries: 4 (configurable in the simulator)
  • Drop probability: The probability of losing a packet after all retries = PER5 (original + 4 retries)

Example

At PER = 5%:

P(drop) = 0.05^5 = 3.125e-7  -- virtually no drops
Avg retransmissions = PER/(1-PER) = 0.053
Effective throughput = raw_rate * (1 - overhead) / (1 + avg_retx)

At PER = 30%:

P(drop) = 0.30^5 = 0.00243  -- 1 in 411 packets dropped
Avg retransmissions = 0.30/0.70 = 0.43
Throughput reduced by ~30% from retransmissions alone

Each retransmission adds latency: one round-trip time to the satellite (~4 ms for LEO). A packet needing 3 retries adds 12 ms of delay, which feeds into jitter.

9. Goodput vs Throughput

Three distinct measures of data rate:

  • Raw bit rate: symbol_rate × bits_per_symbol = what the modulator outputs
  • Throughput: raw_rate × code_rate × (1 - frame_overhead) = useful bits after FEC overhead
  • Goodput: throughput × 1/(1 + avg_retransmissions) × (1 - Pdrop) = actual useful data delivered to the application

At high SNR, the overhead is dominated by FEC (25% at rate 3/4). At low SNR, retransmissions dominate and goodput can drop below 50% of raw rate. The simulator tracks all three metrics separately so you can see where the losses occur.

10. Jitter Sources

Jitter is variation in packet delivery time. In a LEO satellite link, four sources contribute:

SourceMagnitudeCause
Retransmission delay4-16 msEach retry adds one RTT (~4 ms). Up to 4 retries.
Compute scheduling0.5-2 msOnboard processor contention. Worse under compute overload faults.
Doppler rate changes0.1-0.5 msTracking loop adjustments cause momentary phase transients.
ACM switching1-3 msModCod transitions cause brief buffer accumulation as rates change.

Total jitter budget: typically 5-20 ms during a healthy pass. Under fault conditions (compute overload + high retransmission rate), jitter can spike to 50+ ms, causing problems for real-time applications like VoIP and gaming.

11. Code Connection

The modulation and protocol computations live in:

  • src/simulation/modulation/modcod.ts -- The ModCod table and ACM selection logic. Contains the SNR thresholds, hysteresis parameters, and spectral efficiency lookup. Called every tick to select the current ModCod based on effective SNR.
  • src/simulation/modulation/ber.ts -- BER calculation from Eb/N0 using the erfc approximation. Implements the BER formulas for QPSK, 8PSK, 16QAM, and 64QAM. Also computes PER from BER.
  • src/simulation/protocol/packet-layer.ts -- The ARQ retransmission model, drop probability calculation, goodput computation, and jitter accumulation. Converts BER/PER into application-level metrics.

Data flow through the code:

// 1. modcod.ts: select ModCod
const modcod = selectModCod(effectiveSNR, currentModCod); // with hysteresis

// 2. ber.ts: compute error rates
const ber = computeBER(ebN0, modcod.modulation);
const per = 1 - Math.pow(1 - ber, PACKET_BITS);

// 3. packet-layer.ts: compute goodput
const dropProb = Math.pow(per, MAX_RETRIES + 1);
const avgRetx = per / (1 - per);
const goodput = throughput / (1 + avgRetx) * (1 - dropProb);
