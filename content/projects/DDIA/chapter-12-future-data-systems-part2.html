Unbundling Databases

Kleppmann's central thesis for Part III: a database is a bundle of features — storage, indexes, query language, transactions, replication. The trend in modern data engineering is to unbundle these features: use separate, specialized systems for each, connected via event logs.

"If you imagine the database decomposed into separate components — the storage, the indexing, the query planning, the coordination — you can see how each of these could be replaced by specialized systems. The log is the glue that holds them together."

Dataflow Systems

Kleppmann's vision: rather than thinking of data systems as request-response (client sends query, database returns result), think of them as dataflow pipelines. Data flows from producers, through transformations, to consumers. State is maintained in local stores, updated by streaming joins.

Combining Batch and Stream Processing

The unified dataflow model:
1. All writes → event log (Kafka)
2. Stream processor maintains derived state (RocksDB local state)
3. Batch rebuild: replay log from offset 0 → same code builds same state
4. Output: update search indexes, caches, warehouses, push notifications

Key property: the same logic handles both ongoing streaming and historical reprocessing. No separate batch and streaming codebases.

Stable vs. Mutable State

Mutable state problem
If state can be modified in-place (UPDATE, DELETE), it's hard to replicate, audit, or reprocess. What was the value before the update? Lost.
Immutable events + derived state
Events are immutable (append-only log). Current state = derived by processing the log. To "undo" an action: add a compensating event. Full history preserved.

Correctness in Dataflow Systems

How do you ensure correctness in a distributed dataflow? Traditional databases use ACID transactions. In a pipeline of loosely coupled services, you need different techniques.

End-to-End Argument for Correctness

The end-to-end argument: a feature (like delivery guarantees) must be implemented end-to-end, not just at one layer. The network layer can try to deliver packets reliably, but if the application doesn't handle duplicate delivery, correctness is still broken.

Example: idempotent message processing. Even if Kafka guarantees at-least-once delivery, the consumer must handle duplicates. The correct approach: include a unique event ID in each message. The consumer stores processed IDs (in its local DB) and skips duplicates. This makes processing idempotent end-to-end, even if the message is delivered twice.

Exactly-Once via Deterministic Functions

If a stream processing function is deterministic (same input → same output), replaying events produces the same result. Combined with idempotent writes (same write applied twice = same effect), you get exactly-once semantics without distributed transactions.

Conditions for exactly-once processing:
1. Deterministic processing: same input events → same output (no random numbers, no current-time calls)
2. Idempotent writes: writing the same output twice has no extra effect (use upsert with event ID as key)
3. Offset tracking: record the input offset atomically with the output write

If all three: you can safely replay events on failure without duplicating effects.

Temporal Integrity (Timeliness and Integrity)

Kleppmann distinguishes between two correctness properties:

PropertyMeaningCan be sacrificed?
Timeliness Users see up-to-date data. Derived views are current. Eventually consistent systems may be stale for seconds or minutes. Yes — eventual consistency is often acceptable. Staleness for analytics is fine.
Integrity No corruption: no lost data, no contradictions, no violations of invariants. A deleted record truly gone. A constraint truly enforced. No — integrity violations are permanent damage. Must be preserved even at the cost of timeliness.
The key insight: eventual consistency sacrifices timeliness, not integrity. As long as the system will converge to a correct state (even if not immediately), eventual consistency is fine for many use cases. The problem is systems that violate integrity — losing writes, allowing double-spends, or corrupting data permanently.

Auditing and Immutability

An event log provides a complete audit trail by design. Every state change is recorded. This is valuable for:

  • Debugging: trace how a record reached its current state
  • Compliance: prove to auditors what happened and when
  • Fraud detection: analyze patterns in historical events
  • Recovery: restore any point-in-time state by replaying events
Traditional mutable databases make auditing hard: once a record is updated, the old value is gone. You need to add audit tables, triggers, or external audit systems. With an event log, auditing is free — the log is the audit trail.
