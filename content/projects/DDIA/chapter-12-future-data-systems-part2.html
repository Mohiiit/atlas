Chapter 12: The Future of Data Systems

Part 2 -- Correctness, Integrity, and Ethics — Designing applications that are not only reliable and performant, but also correct and responsible

Aiming for Correctness

Stateless services are forgiving: fix a bug, restart, and everything returns to normal. Stateful systems like databases remember things forever, so errors can have permanent consequences. For roughly four decades, ACID transactions have been the tool of choice, but their foundations are weaker than they appear -- witness the confusion around weak isolation levels, and Kyle Kingsbury's Jepsen experiments exposing gaps between claimed and actual safety guarantees.

The Core Problem

If your application can tolerate occasionally corrupting or losing data, life is simple. If you need stronger assurances, serializability and atomic commit are established approaches -- but they typically work only in a single datacenter, limiting scale and fault tolerance. We need new abstractions for application-specific end-to-end correctness in distributed environments.

The End-to-End Argument for Databases

Just because an application uses a data system with strong safety properties (e.g., serializable transactions) does not guarantee freedom from data loss or corruption. If the application has a bug that writes incorrect data, transactions will not save you. Immutability helps recovery, but it is not a cure-all.

Exactly-Once Execution of an Operation

When something goes wrong while processing a message, you can drop it (data loss) or retry it (risk of double processing). Processing twice is a form of data corruption -- charging a customer twice or double-incrementing a counter. Exactly-once semantics means arranging computation so the final effect is the same as if no faults occurred, even if the operation was retried.

The most effective approach is to make operations idempotent: ensuring an operation has the same effect whether executed once or multiple times. This requires maintaining additional metadata (e.g., the set of operation IDs that have updated a value) and ensuring proper fencing during failover.

Duplicate Suppression

TCP uses sequence numbers to suppress duplicate packets -- but only within a single TCP connection. Consider a money transfer: if the client's connection drops after sending COMMIT but before hearing back, it does not know whether the transaction committed. Reconnecting and retrying creates a new TCP connection, outside the scope of TCP's dedup. The transfer could execute twice ($22 instead of $11).

Even two-phase commit does not fully solve this. And the problem extends beyond the database: if the end user is on a weak cellular connection and their HTTP POST succeeds but the response is lost, the browser shows "Are you sure you want to submit this form again?" -- and the user says yes.

Operation Identifiers

To make an operation idempotent through several hops of network communication, you need to consider the end-to-end flow. Generate a unique operation ID (e.g., a UUID) at the client, include it as a hidden form field, and pass it all the way through to the database.

The Idempotent Transfer Pattern

  • ALTER TABLE requests ADD UNIQUE (request_id);
  • Within the transaction: INSERT INTO requests (request_id, from_account, to_account, amount) VALUES (...)
  • If the UUID already exists, the INSERT fails and the transaction aborts
  • The requests table also serves as an event log, hinting toward event sourcing
  • Balance updates can be derived from the request event in a downstream consumer

The End-to-End Argument

"The function in question can completely and correctly be implemented only with the knowledge and help of the application standing at the endpoints of the communication system. Therefore, providing that questioned function as a feature of the communication system itself is not possible." -- Saltzer, Reed, and Clark, 1984

The end-to-end argument applies beyond duplicate suppression: checksums in Ethernet, TCP, and TLS detect network corruption but not bugs in software or corruption on disk. WiFi encryption protects against local snooping but not attackers elsewhere. Only end-to-end mechanisms (checksums, encryption, dedup IDs) can address all sources of failure. Low-level mechanisms remain useful as performance optimizations, reducing problems that reach higher levels.

Uniqueness Constraints and the Log

Enforcing a uniqueness constraint in a distributed setting requires consensus. The most common approach is single-leader coordination, with partitioning by the unique value for scale-out. Asynchronous multi-master replication is ruled out because different masters may concurrently accept conflicting writes.

Uniqueness in Log-Based Messaging

In the unbundled database approach, a log ensures total order broadcast (equivalent to consensus). A stream processor consumes messages sequentially on a single thread per partition, deterministically deciding which of several conflicting operations came first.

Multi-Partition Request Processing

A money transfer touches potentially three partitions: request ID, payee account, and payer account. Instead of an atomic commit across all three, we can use partitioned logs with end-to-end request IDs:

Three-Step Coordination-Free Transfer

  • Step 1: Client assigns a unique request ID. Request is appended to a log partition (by request ID).
  • Step 2: A stream processor reads the request log and emits: a debit instruction (partitioned by payer A) and a credit instruction (partitioned by payee B), both tagged with the request ID.
  • Step 3: Downstream processors consume debit/credit streams, deduplicate by request ID, and apply balance changes.

If the step-2 processor crashes and replays, it produces the same deterministic instructions. Step-3 deduplicates. No distributed transaction needed.

Timeliness vs. Integrity

Kleppmann argues that the term "consistency" conflates two distinct requirements that should be considered separately. ACID transactions bundle both, but event-based dataflow systems can decouple them.

Correctness of Dataflow Systems

Reliable stream processing can preserve integrity without distributed transactions through four mechanisms:

1. Atomic Single Message

Represent the write operation as a single message that can be written atomically (fits event sourcing).

2. Deterministic Derivation

Derive all state updates from that single message using deterministic functions (like stored procedures).

3. End-to-End Request ID

Pass a client-generated request ID through all processing levels for duplicate suppression and idempotence.

4. Immutable + Reprocessable

Keep messages immutable and allow derived data to be reprocessed from time to time to recover from bugs.

Coordination-Avoiding Data Systems

Many real applications can get away with loosely interpreted constraints. If two people book the same seat, you apologize and offer alternatives. If stock is oversold, you order more and offer a discount. Banks charge overdraft fees rather than blocking all withdrawals with linearizable locks. Airlines deliberately overbook flights.

The Apology Workflow

Whether the cost of an apology (refund, rebooking, discount) is acceptable is a business decision. If it is acceptable, the traditional model of checking all constraints before writing data is unnecessarily restrictive. You can write optimistically and check constraints after the fact -- ensuring validation before expensive irreversible actions, but not before every single write.

Coordination reduces apologies for inconsistencies but may increase apologies for outages. The sweet spot depends on business requirements.

Two key observations combine powerfully: (1) Dataflow systems can maintain integrity without atomic commit, linearizability, or synchronous cross-partition coordination. (2) Many applications are fine with loose constraints that may be temporarily violated and fixed later, as long as integrity is preserved. Together, these mean coordination-avoiding data systems can achieve better performance and fault tolerance while maintaining strong integrity guarantees -- even across multiple datacenters with asynchronous replication.

Trust, but Verify

Our system models assume certain things work correctly: fsync persists data, memory is not corrupted, CPU multiplication returns correct results. These assumptions hold most of the time, but not always. Data can corrupt silently on disk. Network corruption can evade TCP checksums. Random bit-flips from hardware faults or even the rowhammer attack can flip bits in memory.

Maintaining Integrity in the Face of Software Bugs

Even widely-used databases have bugs: MySQL failing to maintain uniqueness constraints, PostgreSQL exhibiting write skew under serializable isolation. Application code has far more bugs. The ACID notion of "consistency" assumes transactions are bug-free -- but they are not.

Don't Just Blindly Trust

Checking data integrity is known as auditing. HDFS and Amazon S3 run background processes that continually read back files, compare them to replicas, and move data between disks. If you want to be sure your data is still there, you have to actually read it and check. You must periodically test restoring from backups -- otherwise you may discover your backup is broken only when you have already lost data.

Designing for Auditability

Traditional transaction logs (insertions, updates, deletions across tables) do not clearly capture why mutations were performed. Event-based systems do better: user input is represented as a single immutable event, and all resulting state is derived deterministically. This makes provenance clear and integrity checking feasible.

Tools for Auditable Data Systems

Cryptographic auditing relies on Merkle trees -- trees of hashes that efficiently prove a record appears in a dataset. Certificate transparency uses Merkle trees to validate TLS/SSL certificates. Blockchain and distributed ledger technologies explore this space: distributed databases where mutually untrusting replicas continually check each other's integrity using consensus protocols. While Kleppmann is skeptical of proof-of-work and Byzantine fault tolerance, the integrity-checking aspects are interesting and may become more widely used in data systems.

Doing the Right Thing

Every system is built for a purpose, and every action has both intended and unintended consequences. Many datasets are about people -- their behavior, interests, and identity. We must treat such data with humanity and respect. A technology is not good or bad in itself; what matters is how it is used and how it affects people.

The Engineer's Responsibility

Software development increasingly involves important ethical choices. The ACM's Software Engineering Code of Ethics exists but is rarely discussed or enforced. It is not sufficient for engineers to focus exclusively on technology and ignore its consequences. Reasoning about ethics is difficult, but it is too important to ignore.

Predictive Analytics and Privacy

Using data to predict weather or disease spread is one thing. Predicting whether a convict will reoffend, a loan applicant will default, or an insurance customer will make expensive claims directly affects individual lives. Organizations naturally want to be cautious and say "no" when in doubt. But when algorithmic decision-making becomes widespread, someone labeled as "risky" faces a cascade of rejections.

Algorithmic Prisons

Systematically being excluded from jobs, air travel, insurance, housing, and financial services -- without proof of guilt and with little chance of appeal -- has been called an "algorithmic prison." In countries that respect human rights, the criminal justice system presumes innocence until proven guilty. Automated systems can bypass this principle entirely.

Bias and Discrimination

If there is systematic bias in the input data, the system will learn and amplify that bias in its output. In racially segregated neighborhoods, a postal code or IP address is a strong predictor of race. Anti-discrimination laws prohibit using protected traits, but correlated features serve as proxies.

"Machine learning is like money laundering for bias." -- Maciej Ceglowski

Predictive analytics extrapolate from the past: if the past is discriminatory, they codify that discrimination. A credit score summarizes "how did you behave?" while predictive analytics asks "who is similar to you, and how did they behave?" -- implying stereotyping. If we want the future to be better than the past, moral imagination is required, and only humans can provide it. Data and models should be our tools, not our masters.

Feedback Loops and Accountability

When services predict what content users want to see, they may show only opinions users already agree with, creating echo chambers that breed stereotypes, misinformation, and polarization. When predictive analytics affect people's lives, self-reinforcing feedback loops emerge: bad credit leads to joblessness, which worsens credit further.

Automated decision-making raises questions of responsibility and accountability. If a human makes a mistake, they can be held accountable and decisions can be appealed. But who is accountable when an algorithm goes wrong? If a self-driving car causes an accident? If a scoring algorithm systematically discriminates? Can you explain to a judge how the algorithm made its decision?

Systems Thinking

We cannot always predict feedback loops, but many consequences can be predicted by analyzing the entire system -- not just the computerized parts, but also the people interacting with it. Does the system reinforce existing differences (making the rich richer or poor poorer), or does it combat injustice?

Privacy, Surveillance, and Consent

When a user's activity is tracked as a side effect of other actions, the service takes on interests of its own. If the service is funded by advertising, users' interests take second place. The tracking can be appropriately described as surveillance.

"In our surveillance-driven organization we collect real-time surveillance streams and store them in our surveillance warehouse. Our surveillance scientists use advanced analytics and surveillance processing in order to derive new insights." -- Try replacing "data" with "surveillance" and see if it still sounds good.

Privacy does not mean keeping everything secret. It means having the freedom to choose which things to reveal to whom. It is a decision right, an aspect of autonomy. When data is extracted through surveillance, this right is transferred from individuals to corporations. Companies keep the surveillance outcomes secret because revealing them would be perceived as creepy.

Consent is problematic: users have little knowledge of what data they feed into systems or how it is processed. Many privacy policies obscure rather than illuminate. When a service becomes so popular that it is "regarded by most people as essential for basic social participation," opting out is not a realistic choice -- surveillance becomes inescapable.

Data is not just an asset -- it is a "toxic asset." It attracts thieves, governments, and unscrupulous future management.

Data as a Pollution Problem

"Data is the pollution problem of the information age, and protecting privacy is the environmental challenge. Almost all computers produce information. It stays around, festering. How we deal with it -- how we contain it and how we dispose of it -- is central to the health of our information economy." -- Bruce Schneier

Just as the Industrial Revolution brought prosperity alongside pollution, child labor, and exploitation -- requiring decades of regulation to remedy -- the information age requires us to confront the harms of unchecked data collection. We need a culture shift: stop regarding users as metrics to be optimized, self-regulate data practices, educate end users, and allow individuals to maintain control over their own data.

Chapter and Book Summary

Data Integration

No single tool serves all use cases. Applications compose multiple systems, using batch processing and event streams for data integration. Systems of record feed derived data (indexes, materialized views, ML models) through asynchronous, loosely coupled transformations. Problems in one area don't spread to unrelated parts.

Correctness Without Coordination

Strong integrity guarantees can be implemented scalably with asynchronous event processing, end-to-end operation identifiers for idempotence, and asynchronous constraint checking. Clients can wait for validation or proceed optimistically. This is much more scalable and robust than distributed transactions.

Trust and Auditing

Dataflow architectures enable auditability: immutable events, deterministic derivation, Merkle trees for cryptographic integrity. We should build self-verifying systems that continually check their own integrity rather than relying on blind trust.

Ethics and Responsibility

Data can be used for good but also causes significant harm: unjust automated decisions, discrimination, surveillance, exploitation, and data breaches. Engineers carry a responsibility to consider consequences, build systems that treat people with humanity and respect, and work toward a world that preserves human dignity and autonomy.

"As software and data are having such a large impact on the world, we engineers must remember that we carry a responsibility to work toward the kind of world that we want to live in: a world that treats people with humanity and respect. I hope that we can work together toward that goal."

Review Questions

1. Why is TCP's duplicate suppression insufficient for preventing double-execution of a database transaction? What is the end-to-end solution?
Hover to reveal answer
TCP dedup only works within a single connection. If the client's connection drops after sending COMMIT but before receiving the response, a retry creates a new TCP connection -- outside TCP's dedup scope. The end-to-end solution is a client-generated unique operation ID (e.g., UUID) that is passed all the way from the user's device through the application server to the database, where a uniqueness constraint on the ID prevents double execution.
2. What is the difference between timeliness and integrity? Which is more important and why?
Hover to reveal answer
Timeliness means users see an up-to-date state (violation = "eventual consistency," temporary, self-resolving). Integrity means no data loss or corruption (violation = "perpetual inconsistency," permanent, requires explicit repair). Integrity is generally more important: a credit card statement showing a 24-hour lag is annoying but tolerable; a statement where the balance does not equal the sum of transactions is catastrophic.
3. How can a coordination-avoiding data system enforce a money transfer across multiple partitions without using distributed transactions?
Hover to reveal answer
Step 1: Append the transfer request (with unique ID) to a log partition based on request ID. Step 2: A stream processor reads the request and emits a debit instruction (partitioned by payer) and a credit instruction (partitioned by payee), both tagged with the request ID. Step 3: Downstream processors consume these streams, deduplicate by request ID, and apply balance changes. If step 2 crashes and replays, it deterministically produces the same instructions, and step 3 deduplicates them. No atomic commit protocol needed.
4. What is the "apology workflow" and when is it an acceptable alternative to linearizable constraints?
Hover to reveal answer
The apology workflow handles constraint violations after the fact (e.g., refunding a double charge, rebooking an overbooked passenger, offering discounts for oversold stock). It is acceptable when the business cost of the apology is lower than the performance/availability cost of enforcing strict linearizable constraints. Many real businesses (airlines overbooking, banks allowing overdrafts) already operate this way. The key requirement is integrity (no lost data) even if timeliness is relaxed.
5. What does "trust, but verify" mean in the context of data systems? What tools support this approach?
Hover to reveal answer
It means we should not blindly assume that hardware and software are always correct. Instead, we should continuously audit data integrity. Tools include: end-to-end checksums (beyond just TCP/Ethernet), periodic backup restoration tests, background processes that read back and compare replicas (as HDFS and S3 do), Merkle trees for cryptographic proof of record existence, deterministic event log derivations that can be replayed to verify derived state, and certificate transparency-style integrity checking.
6. How do self-reinforcing feedback loops in predictive analytics cause harm? Give an example and explain what "systems thinking" means in this context.
Hover to reveal answer
Example: A person suffers a financial misfortune, misses payments, their credit score drops, employers using credit scores reject them, unemployment deepens poverty, which further worsens credit -- a downward spiral hidden behind mathematical rigor. "Systems thinking" means analyzing the entire system including people, not just the software. It asks: does the system reinforce existing differences (rich get richer, poor get poorer) or combat injustice? Many consequences can be predicted this way, though not all.
