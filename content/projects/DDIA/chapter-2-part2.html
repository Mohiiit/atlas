1. Relational vs Document: Full Comparison

Neither model is universally superior. The right choice depends on your data's structure and your access patterns. The main dimensions: locality, schema flexibility, join support, and relationship complexity.

Diagram 1 — When to Use Document vs Relational

Document DB is better when…

  • Your data has a document-like structure: tree of one-to-many relationships, usually loaded all at once
  • The data structure varies record to record (different document schemas)
  • You don't need joins — data isn't highly interconnected
  • You read entire documents frequently (locality)
  • Schema evolution is frequent and needs to be agile
User profiles Product catalogs CMS articles

Relational DB is better when…

  • Data is highly interconnected — many-to-many relationships are common
  • You need to join data from multiple entities frequently
  • You need strong consistency guarantees (ACID transactions)
  • You have a well-defined, stable schema
  • You need to query the data in many different, ad-hoc ways
E-commerce orders Financial ledgers Social graphs

Schema-on-Read vs Schema-on-Write

Document databases are sometimes called schemaless, but that's misleading — there is still a schema, it's just not enforced by the database. Better terms:

Diagram 3 — Schema-on-Write vs Schema-on-Read

Schema-on-Write (Relational)

Application writes data
▼
DB enforces schema
Wrong type? Rejects write.
Missing column? Error.
▼
Data stored, guaranteed valid

Analogous to static typing: type checking at compile time (write time).

Schema-on-Read (Document)

Application writes data
▼
DB accepts anything
No enforcement. Data stored as-is.
▼
Application interprets schema at read time

Analogous to dynamic typing: type checking at runtime (read time).

When Schema-on-Read Shines

  • There are many different types of objects, impractical to put each in its own table
  • The structure of the data is determined by external systems that you don't control
  • You're performing a migration: with schema-on-read, just start writing new documents with new fields; old code reads old docs, new code reads new docs. With relational, you'd need an ALTER TABLE ADD COLUMN

Data Locality for Queries

A document is usually stored as a single continuous string (JSON or BSON). If your application often needs to access the entire document, there is a storage locality advantage — the database only needs one disk seek. In a relational database, the same profile data is spread across tables, requiring multiple index lookups and disk seeks to assemble.

The locality advantage only applies if you need large parts of the document at once. If you only need a small part (one field), loading the entire document is wasteful. Updates to a document usually require rewriting the whole document — so keep documents small and avoid writes that cause documents to grow.

2. Query Languages for Data

When the relational model was introduced, SQL was a radical idea: a declarative query language. Before SQL, databases had imperative query APIs — you told the database how to retrieve the data step by step. SQL lets you say what you want, leaving the how to the optimizer.

Diagram 4 — Declarative vs Imperative: Finding Animals

Imperative (old databases, most code)

// JavaScript: how to find sharks
const sharks = [];
for (const animal of db.animals) {
  if (animal.family === "Sharks") {
    sharks.push(animal);
  }
}
// Must specify loop order, filtering logic
// Optimizer can't help — you're doing it yourself

You specify the exact execution steps. Hard to parallelize or optimize.

Declarative (SQL)

-- SQL: what you want
SELECT * FROM animals
WHERE family = 'Sharks';

-- That's it. The DB decides:
-- which index to use,
-- whether to parallelize,
-- whether to use a hash join,
-- how to order the scan.

You specify the result, not the path. The optimizer picks the best execution plan.

Why Declarative Wins in the Long Run

  • Conciseness: SQL is far shorter than imperative traversal code for the same query
  • Optimizer freedom: the database can improve performance without you changing your queries. Add an index → all queries that could benefit get faster automatically
  • Parallelism: declarative queries are easier to parallelize across CPU cores or machines, since you haven't specified a sequential execution order
  • CSS analogy: CSS is declarative (say what you want things to look like). Imagine styling web pages with imperative JavaScript — change every element one by one. CSS lets you say "all li in .nav should be blue" and the browser handles it
Diagram 5 — CSS as a Declarative Query Language (Analogy)

Imperative DOM styling

// Find every li in .selected, style it
const items = document.querySelectorAll(
  '.selected > li'
);
for (const item of items) {
  item.style.background = 'blue';
}
// Fragile: need to re-run on DOM changes

Declarative CSS

li.selected > p {
  background-color: blue;
}
/* Browser applies this rule to all
   matching elements, now and future,
   even after DOM updates */

The CSS rule specifies the condition and outcome. The browser's CSS engine handles when and how to apply it. Just like SQL specifies the condition and outcome; the query optimizer handles execution.

3. MapReduce Querying

MapReduce is a programming model for processing large datasets across many machines, popularized by Google's 2004 paper. MongoDB's MapReduce is a limited form for querying a single database. It sits between declarative and fully imperative: you write two JavaScript functions (map and reduce), which are called by the database framework.

Diagram 6 — MapReduce Pattern: Count Sharks per Month

Task: count how many sharks were observed per month.

Step 1: Map function — runs once per document, emits key-value pairs

function map() {
  if (this.family === "Sharks") {
    emit(
      this.observationMonth,  // key: "2014-12"
      this.numAnimals         // value: 3
    );
  }
}
▼ framework groups all emitted values by key ▼

Step 2: Reduce function — runs once per key, aggregates values

function reduce(key, values) {
  return Array.sum(values);
  // key="2014-12", values=[3, 5, 2]  → 10
  // key="2015-01", values=[7, 1]     → 8
}
Map output
"2014-12" → 3
"2014-12" → 5
"2015-01" → 7
Group by key
"2014-12" → [3,5,2]
"2015-01" → [7,1]
Reduce output
"2014-12" → 10
"2015-01" → 8
Diagram 7 — Same Query in SQL vs MongoDB MapReduce vs Aggregation Pipeline

SQL (most readable)

SELECT date_trunc('month', observation_date) AS observation_month,
       sum(num_animals) AS total_animals
FROM observations
WHERE family = 'Sharks'
GROUP BY observation_month;

MongoDB Aggregation Pipeline (declarative, added later)

db.observations.aggregate([
  { $match: { family: "Sharks" } },
  { $group: {
      _id: { $month: "$observationDate" },
      totalAnimals: { $sum: "$numAnimals" }
  }}
]);
SQL: most expressive, easiest to read Aggregation Pipeline: declarative, composable MapReduce: flexible but verbose and imperative

MongoDB added the Aggregation Pipeline specifically because MapReduce is too verbose. This mirrors the broader lesson: declarative is almost always better for queries than imperative traversal code.

MapReduce Limitations

  • The map and reduce functions must be pure — no side effects, no database queries, no external state. The framework calls them many times in parallel and must be able to re-execute them
  • Verbose: a 1-line SQL query becomes 10+ lines of JavaScript
  • Difficult to optimize: the database can't inspect the JavaScript to find a better execution plan
  • MongoDB's aggregation pipeline is now preferred — it's declarative and the optimizer can reason about it
