When Neither Relational Nor Document Fits

Relational databases handle simple many-to-many relationships. Document databases shine when data arrives in nested, self-contained units. But what happens when everything is a many-to-many relationship, and the connections between items are as important as the items themselves?

Classic examples where graph databases win:

  • Social graphs — people and their friendships/follows
  • Web graphs — pages and hyperlinks (PageRank)
  • Road/rail networks — cities and routes
  • Recommendation engines — users, products, and purchase history
  • Knowledge graphs — facts and their relationships (DBpedia, Wikidata)
A graph doesn't need to be homogeneous. Facebook's graph contains people, locations, events, check-ins, and comments — all wired together.

Key Properties of the Property Graph Model

  • Any vertex can have an edge to any other vertex — no schema restrictions
  • Given a vertex, you can efficiently traverse both incoming and outgoing edges
  • Different labels on edges allow multiple relationship types in one graph
  • Properties on both vertices and edges store arbitrary key-value metadata
This flexibility enables a single graph to represent: people, locations, cities, countries, continents — all linked with typed, labeled relationships.

The Cypher Query Language

Cypher is Neo4j's declarative query language for property graphs. Its syntax is designed to be readable as ASCII art.

Creating Data

CREATE
  (NAmerica:Location {name:'North America', type:'continent'}),
  (USA:Location {name:'United States', type:'country'}),
  (Idaho:Location {name:'Idaho', type:'state'}),
  (Lucy:Person {name:'Lucy'}),
  (Idaho) -[:WITHIN]-> (USA) -[:WITHIN]-> (NAmerica),
  (Lucy) -[:BORN_IN]-> (Idaho),
  (Lucy) -[:LIVES_IN]-> (Idaho)
The pattern (Lucy) -[:BORN_IN]-> (Idaho) reads naturally: Lucy was born in Idaho. Arrow direction = relationship direction.

Querying: People Who Emigrated from the US to Europe

MATCH
  (person) -[:BORN_IN]->  () -[:WITHIN*0..]-> (us:Location {name:'United States'}),
  (person) -[:LIVES_IN]-> () -[:WITHIN*0..]-> (eu:Location {name:'Europe'})
RETURN person.name
What this query does, step by step:
Find any vertex (person)
↓
that has a BORN_IN edge to some vertex
↓
that has WITHIN chain leading to "United States"
↓
AND has a LIVES_IN edge to some vertex
↓
that has WITHIN chain leading to "Europe"
↓
Return person.name
-[:WITHIN*0..]->* means zero-or-more hops: Idaho → USA → North America. The query engine handles arbitrary depth — you never write the loop.

This same query in SQL would require recursive CTEs or repeated self-joins — verbose and awkward for variable-depth traversal.

Graph Queries in SQL

You can model a property graph in a relational database using two tables: vertices and edges. The problem? Variable-depth traversal requires recursive queries.

WITH RECURSIVE
  -- Find all locations within USA (any depth)
  in_usa(vertex_id) AS (
    SELECT vertex_id FROM vertices WHERE properties->>'name' = 'United States'
    UNION
    SELECT edges.tail_vertex FROM edges
      JOIN in_usa ON edges.head_vertex = in_usa.vertex_id
      WHERE edges.label = 'within'
  ),
  -- Find all people born in those locations
  born_in_usa(vertex_id) AS (
    SELECT edges.tail_vertex FROM edges
      JOIN in_usa ON edges.head_vertex = in_usa.vertex_id
      WHERE edges.label = 'born_in'
  )
  -- ... repeat for Europe ...
SELECT vertices.properties->>'name'
FROM vertices JOIN born_in_usa ON vertices.vertex_id = born_in_usa.vertex_id
             JOIN lives_in_europe ON vertices.vertex_id = lives_in_europe.vertex_id;
Same query: 4 lines of Cypher vs 30+ lines of SQL. The relational model was not designed for graph traversal — it can do it, but it's painful.

_:idaho a :Location; :name "Idaho"; :type "state"; :within _:usa.
_:usa   a :Location; :name "United States"; :type "country"; :within _:namerica.
_:namerica a :Location; :name "North America"; :type "continent".

SPARQL Query Language

SPARQL (SPARQL Protocol and RDF Query Language) is to triple-stores what Cypher is to property graphs. It follows the same pattern-matching philosophy.

PREFIX : <urn:example:>
SELECT ?personName WHERE {
  ?person :name ?personName.
  ?person :bornIn / :within* / :name "United States".
  ?person :livesIn / :within* / :name "Europe".
}
SPARQL is notably concise. The path expression :bornIn / :within* / traverses the chain automatically. This influenced Cypher's design.

Datalog: The Grandfather of Query Languages

Datalog predates both SQL and Cypher. It's a logic programming query language derived from Prolog, used in systems like Datomic.

Datalog Syntax

Facts are written as predicates: predicate(subject, object).

within(namerica, 'North America', continent).
within(usa, 'United States', country).
within(idaho, 'Idaho', state).
born_in(lucy, idaho).
lives_in(lucy, idaho).
within(france, 'France', country).
born_in(alain, beaune).
lives_in(alain, paris).

Defining Rules (the power of Datalog)

/* Rule: x is within_recursive of z if x is within y AND y is within z */
within_recursive(Location, Name, Type) :-
  within(Location, Name, Type).
within_recursive(Location, Name, Type) :-
  within(Location, Via, _),
  within_recursive(Via, Name, Type).

/* Rule: person migrated from Source to Dest */
migrated(Name, Source, Dest) :-
  person(PID, Name),
  born_in(PID, BornLoc),
  within_recursive(BornLoc, Source, country),
  lives_in(PID, LivingLoc),
  within_recursive(LivingLoc, Dest, country).

/* Query */
?- migrated(Who, 'United States', 'Europe').
Datalog defines rules that build up from base facts. Rules can reference themselves (recursive rules). The query engine figures out how to execute them. This approach is foundational — SQL's WITH RECURSIVE is essentially Datalog.

Chapter 2 Summary

This chapter showed how the data model you choose shapes everything downstream: how you write queries, how your schema evolves, and how well different access patterns perform.

Relational Model

  • Tables, rows, foreign keys
  • Normalization eliminates duplication
  • Joins connect related data
  • SQL: declarative, optimizer-driven
  • Best: business/OLTP data

Document Model

  • JSON/BSON nested records
  • Locality: load full document
  • Schema-on-read flexibility
  • Weak many-to-many support
  • Best: self-contained records

Graph Model

  • Vertices + edges + properties
  • Traversal-first design
  • Variable-depth paths trivial
  • Cypher/SPARQL/Datalog
  • Best: highly connected data

Historical Context

  • IMS (1968): hierarchical model
  • CODASYL: network model
  • Codd's relational: 1970
  • NoSQL wave: 2000s scalability
  • Polyglot persistence: 2010s+

Review Q&A

1
What is the core problem the property graph model solves that relational databases struggle with?
›
Variable-depth traversal of highly connected data. In relational databases, finding "all locations within the United States (at any depth of nesting)" requires recursive CTEs — verbose and unnatural. In a property graph with Cypher, you write -[:WITHIN*0..]-> and the engine handles any depth. The graph model makes connections first-class citizens rather than an afterthought.
2
In the property graph model, what information does each vertex and each edge contain?
›
Vertex: unique ID, set of outgoing edges, set of incoming edges, map of key-value properties. Edge: unique ID, tail vertex (where the edge starts), head vertex (where the edge ends), a label describing the relationship type, map of key-value properties. The label is crucial — it distinguishes BORN_IN from LIVES_IN from MARRIED_TO while all being edges in the same graph.
3
How does Cypher's syntax visually represent graph relationships?
›
It uses ASCII art: (vertex) -[:LABEL]-> (vertex). Parentheses enclose vertices, square brackets with a colon enclose edge labels, and arrows show direction. For example, (lucy) -[:BORN_IN]-> (idaho) reads as "Lucy was born in Idaho." The *0.. quantifier means zero-or-more hops: -[:WITHIN*0..]-> traverses any chain of WITHIN edges.
4
What is a triple-store and how does it represent data differently from property graphs?
›
A triple-store stores all data as (subject, predicate, object) triples. The subject is always a vertex. The predicate is either an edge label (if object is another vertex) or a property key (if object is a primitive like a string). For example: (lucy, age, 33) stores a property, while (lucy, BORN_IN, idaho) stores a relationship. Triple-stores use the RDF data model; SPARQL is the standard query language.
5
Why did the Semantic Web vision largely fail to materialize?
›
The Semantic Web aimed for every website to publish structured RDF data, allowing machines to reason across the entire web. It failed primarily due to: (1) high complexity for publishers — IRIs, namespaces, OWL ontologies; (2) lack of incentive — web publishers gain nothing from making their data machine-readable for others; (3) decentralized coordination failures — no central authority to enforce standards. Triple-stores succeeded inside enterprises (internal knowledge graphs, data integration), just not on the public web.
6
How does Datalog differ from Cypher and SPARQL in its approach to querying?
›
Cypher and SPARQL are pattern-based: you write the shape of what you want and the engine matches it. Datalog is rule-based: you define reusable rules (like functions) that build up from base facts. Rules can be recursive and can call each other. This makes Datalog more powerful for complex multi-step reasoning but harder to read for simple queries. Datomic (Clojure) uses Datalog; it predates SQL and influenced later declarative languages.
7
What is "polyglot persistence" and why is it the emerging standard?
›
Polyglot persistence means using multiple different database types within a single application, each chosen for what it does best. A social app might use: PostgreSQL for user accounts and billing (relational), MongoDB for user-generated posts (document), Neo4j for the friend graph (graph), Redis for session caching (key-value), Elasticsearch for full-text search. No single database excels at all workloads. This approach maximizes fit between data model and access patterns at the cost of operational complexity.
8
What advantage does a graph database have over a relational database for modeling Facebook's data?
›
Facebook's graph includes: people, locations, events, check-ins, pages, comments, likes, photos — all interconnected. In a relational model, each entity type is a separate table; queries joining many tables become complex. In a property graph, all vertices and edges coexist in one structure regardless of type. The labeled edges let you distinguish "likes," "follows," "tagged_in," and "friends_with" relationships without separate join tables. Traversal queries (e.g., "mutual friends of friends within 2 hops") are natural in Cypher.
9
Why is it important that edges in a property graph can have properties?
›
Edge properties store metadata about the relationship itself, not just its existence. Examples: a WORKED_AT edge might have start_date and end_date properties; a REVIEWED edge might have a rating and review_text; a TRANSACTION edge might have amount and timestamp. Without edge properties, you'd need to introduce intermediate "reification" nodes, making simple queries unnecessarily complex.
10
If the relational model was designed to solve the problems of CODASYL, what problems did CODASYL have?
›
CODASYL (the network model) used a hierarchical tree of records connected by pointers. Problems: (1) Access path dependency — queries had to traverse specific pointer paths; changing the access path broke all existing queries; (2) No independence — application code was tightly coupled to the database's physical layout; (3) Complex updates — inserting a record required maintaining multiple pointer lists. Codd's relational model introduced independence: store data in normalized tables, let a query optimizer figure out the access path.
