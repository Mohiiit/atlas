Log-Based Message Brokers

The key insight behind Kafka: instead of a queue (where messages are deleted after acknowledgment), use an append-only log. Consumers track their position (offset) in the log. Rewinding is trivially possible.

Append-only log structure
offset: 0 | event: {user:1, action:"login"}
offset: 1 | event: {user:2, action:"purchase", item:"A"}
offset: 2 | event: {user:1, action:"logout"}
offset: 3 | event: {user:3, action:"login"}
offset: 4 | event: {user:2, action:"refund", item:"A"}
Consumer A (fraud detection): at offset 4 — up to date
Consumer B (analytics): at offset 2 — processing at its own pace
Consumer C (audit): can rewind to offset 0 any time

Apache Kafka Architecture

Kafka implements the log-based message broker model at scale. Events are organized into topics → partitions. Partitions are the unit of parallelism and ordering.

Topics and Partitions

Topic "orders" with 3 partitions, each replicated across 3 brokers
Partition 0
Leader: Broker 1
Replica: Broker 2
Replica: Broker 3
Offset: 0..1053
Partition 1
Leader: Broker 2
Replica: Broker 3
Replica: Broker 1
Offset: 0..2107
Partition 2
Leader: Broker 3
Replica: Broker 1
Replica: Broker 2
Offset: 0..891
Ordering guaranteed within a partition. No ordering guarantee across partitions. Producers choose partition by key hash (all events for user_id=123 → same partition = ordered per user).

Producer Key Routing

Producer: event has key="user_id:123"
partition = hash("user_id:123") % num_partitions
All events for user 123 → same partition → same consumer → ordered per user

No key: round-robin across partitions → maximum throughput, no ordering
Custom partitioner: e.g., VIP users → partition 0 (dedicated consumer)

Consumer Groups in Kafka

Topic: 4 partitions
Consumer Group A (stream processor)
Consumer A1 → P0, P1
Consumer A2 → P2, P3
2 consumers share 4 partitions
Consumer Group B (analytics)
Consumer B1 → P0, P1, P2, P3
1 consumer reads all partitions independently
Each group maintains its own offset. Groups don't interfere.
Kafka's partition model: max parallelism = number of partitions. You can have more consumers than partitions in a group, but the extra consumers are idle. To increase parallelism, increase partitions (but this is a one-way operation — you can't easily decrease).

Kafka Guarantees

  • Messages in a partition are totally ordered — consumer sees them in the order they were written
  • A message is durable once the leader writes it and acks=all replicas acknowledge
  • Consumer offset is stored in a special Kafka topic (__consumer_offsets), enabling crash recovery
  • Retention: default 7 days, configurable; compacted topics keep only the latest value per key

Change Data Capture (CDC)

Many systems have a central database (the "system of record") and want to propagate changes to derived systems: caches, search indexes, data warehouses, analytics. Change Data Capture extracts every write to the database as an event stream.

The CDC Problem

Without CDC — dual-write problem:
App: write to DB AND write to Elasticsearch
If DB write succeeds but Elasticsearch write fails → inconsistent
If app crashes between the two writes → inconsistent
No atomic guarantee across two different systems.
With CDC — single write, derived streams:
App: write to DB only
CDC reads DB's write-ahead log → emits change events to Kafka
Consumers: Elasticsearch indexer, data warehouse ETL, cache invalidator
DB is the single source of truth. All derivatives are eventually consistent with it.

How CDC Works

PostgreSQL WAL / MySQL binlog / MongoDB oplog
Every write (insert, update, delete) is recorded in the DB's replication log as a structured event.
↓
CDC connector (Debezium, Maxwell)
Reads the WAL as a replication slave. Converts WAL entries to structured change events (JSON/Avro).
↓
Kafka topic per table
Each DB table's changes → dedicated Kafka topic. Change events are ordered per row (keyed by primary key).
↓
Consumers
Elasticsearch indexer, data warehouse loader, cache updater, stream processor.
Debezium (open source) supports CDC for PostgreSQL, MySQL, MongoDB, Oracle, SQL Server. It handles the tricky details: initial snapshot (reading existing data), WAL position tracking, schema changes, tombstone records for deletes.

Event Sourcing

Event sourcing takes the CDC idea further: instead of storing the current state of a record, store the entire sequence of events that led to it. The current state is derived by replaying the event log.

Traditional (CRUD)
Table: accounts
Row: {user_id:1, balance:$250}

Update: balance = $250 → $150
Previous state: gone. You only know current balance.
Event sourcing
Events log:
{type:"deposit", amount:$300}
{type:"withdrawal", amount:$50}
{type:"withdrawal", amount:$100}

Current balance: replay all events → $150. Full history preserved.

Benefits and Challenges of Event Sourcing

BenefitsChallenges
Full audit trail — every state change recorded Current state requires replaying all events (mitigated by snapshots)
Debug time-travel: see exact state at any point in history Event schema evolution: old events must be readable with new schema
Derived views: build new read models by replaying the event log Queries on current state require a read model (CQRS pattern)
Natural fit for event-driven architectures and stream processing Eventual consistency between write model (events) and read model (projections)
Event sourcing + CQRS (Command-Query Responsibility Segregation): commands produce events → events update the event store → projectors replay events to build read models (e.g., current account balance, user profile). Read models are derived, eventually consistent. This is the architecture of systems like Axon, EventStoreDB, and many DDD (domain-driven design) systems.
