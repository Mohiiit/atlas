Doing the Right Thing

Chapter 12 ends DDIA with an important reminder: data systems are built and operated by humans, for humans. The technical decisions we make have real-world consequences that go beyond performance metrics.

Predictive Analytics and Discrimination

Machine learning models trained on historical data can encode historical biases. A credit scoring model trained on biased historical decisions will perpetuate that bias. A hiring algorithm trained on past hires may discriminate against underrepresented groups.

The feedback loop problem: A predictive policing algorithm flags neighborhoods for increased patrols → more crimes detected in those neighborhoods → algorithm trained on this data → neighborhoods flagged more → self-fulfilling prophecy. The algorithm appears accurate (it correctly predicts where crime is recorded) but the prediction is circular.

Proxy Discrimination

Even if a model explicitly excludes protected attributes (race, gender, religion), other correlated features can serve as proxies. ZIP code correlates with race. Browser type correlates with income. Name can signal ethnicity. Removing the protected attribute doesn't eliminate discrimination; it just makes it less visible.

The legal standard in many jurisdictions is "disparate impact": if a decision-making system produces statistically different outcomes for protected groups, it may be illegal regardless of intent. Data systems that automate decisions must be audited for disparate impact — the automated nature doesn't remove legal or ethical responsibility.

Privacy and Surveillance

The same event log infrastructure that enables powerful analytics also enables surveillance. Location data, browsing history, purchase records, communications metadata — when combined, they reveal intimate details about individuals' lives.

Data as a Liability

Data you collect
Useful for analytics, personalization, fraud detection. But also: a target for hackers, subject to breach disclosure laws, potential liability in litigation, subject to government requests.
Data minimization
Collect only what you need. Delete when no longer needed. Anonymize where possible. Don't aggregate personal data "just in case" — the cost (privacy risk, security risk) may outweigh the benefit.

Consent and Transparency

Users should understand what data is collected about them, how it's used, and who it's shared with. The GDPR (EU) and similar regulations formalize this: right to access, right to erasure ("right to be forgotten"), right to portability. Building these features into data systems requires careful design:

  • Right to erasure: difficult with event logs (events are immutable). Solution: encrypt user data with a per-user key; "forget" the user by deleting the key (cryptographic erasure).
  • Data portability: systems need to export all data for a user in a standard format.
  • Consent tracking: which data processing activities has a user consented to? This is itself a data management problem.

Accountability and Explainability

When an algorithm makes a consequential decision (loan denied, job application rejected, bail denied), the person affected has a right to understand why. "The model predicted X" is not an explanation — it's a black box.

The GDPR includes a right to explanation for automated decisions. Interpretable models (logistic regression, decision trees) are easier to explain than opaque models (deep neural networks). There's a real tension between model accuracy (complex models win) and explainability (simple models win). Data systems must sometimes choose the simpler model for accountability reasons.

Designing for Humans

Every data system is ultimately built for humans — operated by humans, producing output consumed by humans. Good system design considers the humans in the loop:

Minimize errors
Design data pipelines to catch and surface errors early. A silently wrong number is worse than a loud failure. Alert on data quality issues.
Human oversight
Automated systems should enable human review, not bypass it. Critical decisions should have a human in the loop for verification.
Avoid lock-in
Use open formats (Parquet, Avro, ORC). Prefer standard protocols. Make it possible to migrate data. Don't trap users in your system.
Resilience
Systems fail. Design for graceful degradation. Operators should be able to understand what's happening, diagnose issues, and recover. Complexity is the enemy of resilience.

Review Q&A

1
Why is dual-writing to multiple data stores an anti-pattern, and what's the correct alternative?
›
Dual-writing means an application writes the same data to multiple stores simultaneously (e.g., PostgreSQL AND Elasticsearch AND Redis). The problems: (1) Non-atomicity: the two writes are not atomic. If the app crashes between them, one store has the update and the other doesn't — they're inconsistent. There's no way to roll back the first write or guarantee the second succeeds. (2) Race conditions: two concurrent clients may write to the stores in different orders, leaving them inconsistent with each other. (3) Error handling complexity: what do you do if the Elasticsearch write fails? Retry? Roll back the DB write? Complex and error-prone. The correct alternative: write to one system (the system of record) and use CDC or event streaming to propagate changes to derived systems. The database's WAL/change stream is the authoritative, ordered log of changes. All derived systems (Elasticsearch, Redis, warehouse) consume it and apply updates. Failures in derived systems don't affect the primary write — they retry from the log. All consumers see the same ordered sequence of changes.
2
What is the Lambda Architecture and why has it fallen out of favor?
›
Lambda Architecture (Nathan Marz, 2012) combines a batch layer (processes all historical data periodically, produces accurate "batch views") and a speed layer (processes recent data in real-time, produces approximate "real-time views"). The serving layer merges both views to answer queries. The motivation: at the time, batch systems (Hadoop) were accurate but slow, and stream systems were fast but approximate. Lambda combined them to get accuracy from the batch layer and freshness from the speed layer. Why it's fallen out of favor: (1) Dual codebases: you must implement the same logic twice — once for batch (e.g., Spark), once for streaming (e.g., Storm). They diverge over time with subtle differences in behavior and bugs. (2) Operational complexity: running and debugging two systems is much harder than one. (3) Modern stream processors (Flink, Kafka Streams, Spark Streaming) handle both batch and streaming in a single system. Historical reprocessing = replay the event log from the beginning. The Kappa Architecture (single stream processor) largely replaces Lambda for new architectures.
3
What's the difference between timeliness and integrity as correctness properties, and which is more important?
›
Timeliness: does the system show up-to-date data? Is derived state current? A system that's eventually consistent but temporarily stale violates timeliness. Timeliness is a liveness property — it says something good (up-to-date data) eventually happens. Integrity: is the data correct and uncorrupted? No lost writes, no constraint violations, no contradictions, no duplicate payments. Integrity is a safety property — a violation (corruption, lost data) is a permanent error that can't be undone. Integrity is more important. A temporarily stale cache is inconvenient. A corrupted database or double-charged payment is catastrophic. Many systems acceptably sacrifice timeliness (eventual consistency) as long as they preserve integrity. The "eventually consistent" model is really about timeliness: "the system will eventually become consistent (integrity preserved, timeliness improved)." The problematic systems are those that sacrifice integrity — losing writes, allowing double-spends, corrupting data — which is never acceptable.
4
How can you implement the "right to be forgotten" in an event-sourced system where events are immutable?
›
Event sourcing stores events as immutable history — you can't simply delete records. The GDPR "right to be forgotten" requires that personal data be deleted. The tension: immutability vs. erasure. Solution: cryptographic erasure. Encrypt all personal data in events with a per-user encryption key. Store the key separately in a key management system (KMS). To "forget" a user: delete their encryption key from the KMS. The events still exist in the log, but the personal data is now ciphertext with no key — effectively erased (with the computational hardness of the encryption as the guarantee). Event log entries become unreadable for that user's data. The aggregate or derived view of the user can then be rebuilt without any personal data. This approach: (1) preserves the immutability of the event log; (2) satisfies erasure requirements; (3) is operationally simple (delete one key = erase all events for that user); (4) works even with replicated/backed-up logs (backups still exist but are now encrypted with a destroyed key). The residual risk: if the encryption is broken in the future, old events could be read. Use strong modern encryption (AES-256) to minimize this risk.
5
What is the "end-to-end argument" and how does it apply to exactly-once processing?
›
The end-to-end argument (Saltzer, Reed, Clark 1984) states: a function (like reliable delivery) can only be correctly implemented at the application level, not just at lower layers. Implementing it only at a lower layer doesn't eliminate the need for the application to implement it too. Applied to exactly-once processing: even if the message broker (Kafka) guarantees at-least-once delivery, the consumer can still receive a duplicate if it crashes and restarts. The broker-level guarantee is not enough for the application. The application must also: (1) assign unique IDs to events; (2) track which event IDs have been processed (in a durable store); (3) skip events with already-seen IDs. With this application-level deduplication, the system achieves end-to-end exactly-once semantics — regardless of how many times the broker delivers the message. The lesson: don't rely on a single layer for correctness guarantees. Implement the guarantee at the level where correctness actually matters — the application's visible output — not just at the infrastructure layer.
6
Why is data minimization important from both an ethical and a technical standpoint?
›
Ethically: collecting data about people has real consequences. Data can be breached, shared with governments, used in litigation, or repurposed in ways users didn't consent to. Location data reveals whether someone visited a medical clinic or a religious organization. Purchase data reveals political affiliations or health conditions. Data collected "just in case it's useful" may cause real harm — people can be discriminated against, targeted, or harmed based on data about them. Technically: excess data is a liability. More data means: larger storage costs, longer processing times, bigger attack surface for breaches, more complex compliance requirements (GDPR, CCPA), more breach disclosure requirements if compromised. The principle: collect only data that you have a specific, current use for. Delete data when the use case is satisfied. Aggregate or anonymize where individual-level data isn't needed. Consider the data subjects: if you wouldn't be comfortable with the data subjects knowing exactly what you collect and how you use it, you probably shouldn't be collecting it. Data minimization is both an ethical obligation and good technical hygiene.
