DDIA — Part III: Derived Data

Chapter 11: Stream Processing

Batch processing operates on a bounded dataset with a known finite size. Stream processing deals with data that is incrementally made available over time -- unbounded, never-ending. This chapter explores how to transmit, store, and process event streams, and how the boundary between databases and message systems is blurring.

Transmitting Event Streams

In stream processing, a record is called an event -- a small, self-contained, immutable object containing the details of something that happened at some point in time. An event is generated once by a producer (publisher/sender) and potentially processed by multiple consumers (subscribers/recipients). Related events are grouped into a topic or stream.

Polling a database for new events is expensive: the more often you poll, the higher the overhead of empty requests. Instead, consumers should be notified when new events appear. This is the job of messaging systems.

Messaging Systems

A messaging system allows producers to send messages (events) that are then pushed to consumers. Two critical design questions differentiate messaging systems:

1. What if producers outpace consumers?

  • Drop messages -- acceptable for metrics where data is approximate anyway
  • Buffer in a queue -- but what happens when queue grows? Memory? Disk?
  • Backpressure (flow control) -- block the producer. Unix pipes and TCP use this.

2. What if nodes crash?

  • Durability requires writing to disk and/or replication
  • If you can afford to lose messages, you get higher throughput and lower latency
  • For counters/billing, every lost message = incorrect data

Direct Messaging from Producers to Consumers

Some systems skip the intermediary entirely:

UDP Multicast

Used in financial industry for stock market feeds. Low latency. Application-level protocols recover lost packets by having the producer remember and retransmit on demand.

Brokerless Libraries (ZeroMQ, nanomsg)

Implement pub/sub messaging over TCP or IP multicast. No central broker -- producers and consumers connect directly.

StatsD / Brubeck

Use unreliable UDP for collecting metrics from all machines. Counter metrics are only correct if all messages are received -- making metrics approximate at best.

Webhooks

Consumer exposes a callback URL. Producer makes an HTTP/RPC request to push events. Used widely in APIs (Stripe, GitHub, etc.).

Limitation of Direct Messaging

These systems generally require the application to handle message loss. They assume producers and consumers are constantly online. If a consumer is offline, it misses messages. If a producer crashes, it loses its retry buffer.

Message Brokers

A message broker (message queue) is a server that acts as an intermediary. Producers write messages to the broker; consumers read from it. By centralizing data in the broker, systems can tolerate clients that come and go.

Acknowledgments and Redelivery

The Reordering Problem

Brokers use acknowledgments: a consumer must explicitly confirm it processed a message before the broker deletes it. If a consumer crashes, the message is redelivered to another consumer.

But this causes reordering: if Consumer 2 crashes while processing message m3, while Consumer 1 is processing m4, then m3 is redelivered to Consumer 1 -- which processes them in order m4, m3, m5. Messages m3 and m4 are delivered out of order.

Fix: Use a separate queue per consumer (no load balancing), or accept reordering if messages are independent.

Partitioned Logs

Traditional message brokers delete messages after delivery -- they are transient. You cannot replay them. But databases keep data permanently. Can we combine the durable storage of databases with the low-latency notification of messaging? Yes: log-based message brokers.

Log-Based Message Brokers (Kafka)

Core Idea

A log is an append-only sequence of records on disk. A producer appends messages to the end; a consumer reads the log sequentially. The log is partitioned for scalability: different partitions on different machines. Each message within a partition gets a monotonically increasing offset.

Consumer Offsets

Simple Bookkeeping

Since messages are consumed sequentially, tracking progress is simple: all messages with offset < current offset have been processed. The broker only needs to periodically record each consumer's offset -- no per-message acknowledgment tracking.

This is very similar to the log sequence number in database replication. The broker is like a leader database; the consumer is like a follower.

If a consumer fails, another node takes over its partitions and resumes from the last recorded offset. Messages processed but not yet checkpointed may be processed twice.

Disk Space Usage & Log Compaction

Circular Buffer on Disk

The log is divided into segments; old segments are deleted or archived. This is effectively a bounded-size buffer (ring buffer) on disk. A 6 TB drive at 150 MB/s sequential write can buffer ~11 hours of messages at maximum throughput. In practice, logs retain days or weeks.

Unlike in-memory brokers, throughput remains constant regardless of retention -- all messages are written to disk anyway.

Log Compaction

Log compaction keeps only the most recent value for each key. The storage engine periodically scans for duplicate keys and discards older entries. A tombstone (null value) marks a deleted key.

With compaction, you can rebuild a derived data system (e.g., search index) by starting a new consumer from offset 0 and scanning the entire compacted log -- no separate snapshot needed. The compacted log is guaranteed to contain the latest value for every key.

Supported by Apache Kafka. Allows the message broker to serve as durable storage, not just transient messaging.

Use Log-Based When...

  • Message ordering is important
  • High throughput, fast per-message processing
  • You need replay / reprocessing capability
  • Building derived data systems (CDC, event sourcing)
  • Experimental consumers on production data

Use Traditional Brokers When...

  • Messages are expensive to process individually
  • You need fine-grained message-level parallelism
  • Message ordering is not critical
  • Task queue / RPC-style async processing
  • Number of consumers needs to exceed partitions

Databases and Streams

A write to a database is an event. A replication log is a stream of database write events. The connection between databases and streams is fundamental -- we can apply ideas from streams to databases, and vice versa.

Keeping Systems in Sync

Most applications need multiple storage systems: OLTP database, cache, search index, data warehouse. They all need to stay in sync. One approach is dual writes: the application explicitly writes to each system. But this has serious problems.

The Dual Writes Problem -- Race Condition
Client 1
→
DB: X=A
Client 1
→
Index: X=A
Client 2
→
DB: X=B
Client 2
→
Index: X=B

Result: DB has X=B (Client 2 wrote last), but Index has X=A (Client 1 wrote last due to unlucky interleaving). Permanently inconsistent!

Problems with Dual Writes

  • Race condition: No single leader determines write order. DB and index may see writes in different orders, leading to permanent inconsistency.
  • Partial failure: One write may succeed while the other fails. Ensuring both succeed or both fail is the atomic commit problem (expensive to solve).
  • No concurrency detection: Without version vectors, one value silently overwrites another.

Solution: make one system the leader (the database), and make all others followers of its change stream.

Change Data Capture (CDC)

Change Data Capture is the process of observing all data changes written to a database and extracting them as a stream that can be replicated to other systems. It makes one database the leader and turns all derived systems into followers.

Implementing CDC

  • Database triggers: Register triggers that observe changes and write to a changelog table. Fragile, significant performance overhead.
  • Parsing the replication log: More robust. LinkedIn's Databus, Facebook's Wormhole, Debezium (MySQL binlog), Bottled Water (PostgreSQL WAL), GoldenGate (Oracle).
  • CDC is usually asynchronous: The source DB doesn't wait for consumers. Adding a slow consumer doesn't affect the source. But replication lag issues apply.

Initial Snapshot

Keeping the entire change history forever is impractical. To bootstrap a new derived system, you need a consistent snapshot corresponding to a known offset in the change log, then apply changes from that point forward.

API Support for Change Streams

Databases increasingly support change streams as first-class features: RethinkDB (query subscriptions), Firebase/CouchDB (change feeds), MongoDB oplog, Kafka Connect (integrates CDC for many DBs with Kafka).

Event Sourcing

Event sourcing stores all changes to application state as a log of immutable events, designed at the application level to reflect user intentions, not low-level state mutations.

Commands vs. Events

A user request arrives as a command -- it may still fail validation. Once accepted, it becomes an event: durable, immutable, a fact. Even if a customer later cancels a reservation, the fact that they formerly held it remains true. Consumers cannot reject events; all validation must happen before the command becomes an event.

State, Streams, and Immutability

"Transaction logs record all the changes made to the database. High-speed appends are the only way to change the log. From this perspective, the contents of the database hold a caching of the latest record values in the logs. The truth is the log. The database is a cache of a subset of the log." -- Pat Helland

State = integral of event stream over time. Change stream = derivative of state over time. Two sides of the same coin.

Advantages of Immutable Events

  • Auditability: Like accounting ledgers, mistakes are corrected with compensating entries, not by erasing. The full history is preserved.
  • Bug recovery: If buggy code writes bad data, an immutable log makes it easy to diagnose and recover (vs. destructive overwrites).
  • Richer information: A user adding then removing a cart item -- the database loses the "considered but rejected" signal, but the event log keeps it. Valuable for analytics.
  • Multiple views: Separate the write format from the read format. Derive different read-optimized representations from the same event log (CQRS -- Command Query Responsibility Segregation).
  • Simplified concurrency: Appending events is a single write, easy to make atomic. Single-threaded log consumers need no concurrency control.

Limitations of Immutability

  • High-churn datasets: Workloads with frequent updates/deletes on a small dataset cause the immutable history to grow prohibitively large. Compaction and garbage collection performance becomes crucial.
  • Privacy & regulation: GDPR and similar laws may require truly deleting user data -- not just appending a "deleted" event, but rewriting history (Datomic's "excision", Fossil's "shunning").
  • True deletion is hard: Copies live in storage engines, filesystems, SSDs (write to new locations), and backups. Deletion is more "making it harder to retrieve" than making it impossible.

Processing Streams

Once you have a stream, what can you do with it? Three options: (1) write to a database/cache/index for querying, (2) push to users (email, dashboard, notifications), or (3) process input streams to produce output streams. A piece of code that does this is called an operator or job.

The critical difference from batch: a stream never ends. Sorting is impossible. Sort-merge joins cannot be used. Fault tolerance can't "restart from the beginning" after years of running.

Uses of Stream Processing

Complex Event Processing (CEP)

Search for patterns across event streams using declarative rules (like regex for events). Queries are stored long-term; events flow past them. The relationship between queries and data is reversed compared to normal databases. Implementations: Esper, IBM InfoSphere Streams.

Stream Analytics

Aggregations and statistical metrics over time windows: rates, rolling averages, percentiles. Uses probabilistic algorithms (HyperLogLog, Bloom filters) for memory efficiency. Frameworks: Storm, Spark Streaming, Flink, Samza, Kafka Streams.

Materialized View Maintenance

Deriving alternative views of data and keeping them up-to-date as underlying data changes. Unlike analytics, may require all events ever (not just a time window). Samza and Kafka Streams support this via log compaction.

Search on Streams

Store queries, let documents flow past them (inverse of normal search). Media monitoring, real estate alerts. Elasticsearch's percolator feature. Can index queries to optimize matching.

Message Passing and RPC vs. Stream Processing

  • Actor frameworks: manage concurrency, ephemeral one-to-one communication, arbitrary patterns (including cycles)
  • Stream processors: data management, durable multi-subscriber logs, acyclic pipelines
  • Crossover: Apache Storm's "distributed RPC" farms out queries to stream processing nodes

Reasoning About Time

Stream processors often use time windows ("average over the last 5 minutes"). But whose time? When? The notion is surprisingly tricky.

Event Time

The timestamp when the event actually occurred. Used in batch processing. Makes processing deterministic: same input = same result. But events may arrive out of order or very late.

Processing Time

The wall-clock time on the processing machine. Simple but breaks down with lag. If processor restarts and processes a backlog, it creates a false spike in metrics. Star Wars analogy: episodes released out of narrative order.

Whose Clock Are You Using?

Mobile apps may buffer events offline for hours/days. Device clocks can't be trusted. Solution: log three timestamps:

  • Time event occurred (device clock)
  • Time event was sent to server (device clock)
  • Time event was received by server (server clock)

Offset = (server receive time) - (device send time). Apply this offset to estimate the true event time.

Handling Straggler Events

When windowing by event time, you can never be sure all events for a window have arrived. Two options for late-arriving stragglers:

  • Ignore them: Track the number of dropped events as a metric. Alert if significant.
  • Publish a correction: Update the window's result to include the straggler. May need to retract the previous output.

Stream Joins

Joins are as important in stream processing as in batch. But streams are unbounded and events arrive at unpredictable times, making joins more challenging. Three types:

Three Types of Stream Joins
Stream-Stream Join
(window join)

Both inputs are activity event streams. Join related events within a time window (e.g., search query + click within 1 hour, matched by session ID). Processor maintains state: indexed events in the window.

Example: click-through rate = join search events with click events.

Stream-Table Join
(stream enrichment)

Enrich activity events with database info (e.g., add user profile to activity event by user ID). Processor keeps a local copy of the table, updated via CDC. Like a hash join with a "beginning of time" window for the table.

Remote DB lookups are too slow; use local copy updated via change stream.

Table-Table Join
(materialized view)

Both inputs are database changelogs. Maintains a materialized view of a join between two tables. Twitter timeline: join tweets stream with follows stream to maintain per-user inbox caches.

Mathematically: (u·v)' = u'v + uv' -- any change in tweets joins with current followers, and vice versa.

Time-Dependence of Joins

All three join types require maintaining state from one input and querying it from the other. When state changes over time, which version do you join with?

  • Tax rates change. When joining sales to tax rates, use the rate at the time of sale.
  • If ordering across streams is undetermined, the join becomes nondeterministic -- rerunning on the same input may give different results.
  • In data warehousing, this is the slowly changing dimension (SCD) problem. Addressed by giving each version a unique ID. Trade-off: prevents log compaction since all versions must be retained.

Fault Tolerance

Batch processing achieves exactly-once semantics (or more precisely, "effectively-once"): failed tasks are restarted, partial output is discarded. In stream processing, the stream never ends, so you can't "wait until done" to make output visible. How do we handle faults?

Microbatching (Spark Streaming)

Break the stream into small blocks (~1 second), treat each as a mini batch job. Provides exactly-once within the framework. Implicitly creates a tumbling window equal to batch size. Trade-off: smaller batches = more overhead; larger batches = more latency.

Checkpointing (Apache Flink)

Periodically snapshot operator state to durable storage (e.g., HDFS). On crash, restart from the last checkpoint. Triggered by barriers in the message stream. Doesn't force a particular window size. More flexible than microbatching.

Atomic Commit / Transactions

Ensure all outputs and side effects of processing an event happen atomically: messages to downstream, DB writes, state changes, offset advancement. Used by Google Cloud Dataflow, VoltDB. Not XA -- kept internal to the framework. Overhead amortized by batching messages per transaction.

Idempotence

Make operations safe to perform multiple times. Store the Kafka offset alongside the written value so you can detect duplicates. Requires: deterministic processing, same message order on replay, no concurrent writes to same value. Effective exactly-once with small overhead.

Fault Tolerance Approaches Compared
Microbatching
Simple model. Fixed window granularity. ~1 second latency floor. Spark Streaming.
Checkpointing
Flexible windows. Lower latency possible. More complex recovery. Apache Flink.
Transactions
Strongest guarantees. Higher overhead. Kept internal to framework (not XA). Cloud Dataflow, VoltDB.
Idempotence
Low overhead. Requires determinism and message ordering. Works with log-based brokers. Storm Trident.

Caveat: All these approaches only guarantee exactly-once within the framework. Side effects that leave the framework (emails, external DB writes) may happen twice on restart.

Rebuilding State After a Failure

  • Remote datastore: Keep state in a replicated database. Simple but slow (network round-trip per message).
  • Local state + replication: Flink snapshots operator state to HDFS. Samza/Kafka Streams replicate state changes to a dedicated Kafka topic with log compaction.
  • Rebuild from input: If state is over a short window, simply replay input events. If it's a local DB replica via CDC, rebuild from the log-compacted change stream.
  • Redundant processing: VoltDB processes each input on multiple nodes in parallel.

First-Principles Questions

Test your understanding. Click to reveal the answer.

1Why is polling a database for new events a bad approach, and what is the alternative?▼
Polling becomes expensive as you increase frequency: most requests return no new events, so you waste resources on empty checks. The more often you poll, the higher the overhead-to-useful-work ratio. The alternative is a push-based notification system (messaging system) where consumers are notified when new events appear, eliminating wasteful polling.
2Why does combining load balancing with acknowledgment-based redelivery break message ordering?▼
When a consumer crashes mid-processing, the unacknowledged message is redelivered to a different consumer. But that other consumer may have already moved ahead and processed later messages. So the redelivered message arrives after messages that were originally sent after it. Example: Consumer 2 crashes on m3 while Consumer 1 processes m4. m3 is redelivered to Consumer 1, which processes m4, then m3, then m5 -- out of the original order. The fix is either separate queues per consumer, or accepting that reordering can happen.
3What is the fundamental insight that makes log-based message brokers (like Kafka) different from traditional brokers (like RabbitMQ)?▼
Traditional brokers treat messages as transient -- they delete messages after delivery. Log-based brokers treat messages as durable data -- they append messages to a persistent, partitioned log on disk. Reading does not delete the message. This combines the durable storage approach of databases with the low-latency notification of messaging. Consumers simply track their offset, enabling replay, multiple independent consumers, and experimentation without affecting production.
4Why are dual writes fundamentally flawed for keeping multiple systems in sync?▼
Two problems: (1) Race condition: With no single leader determining write order, two concurrent clients may write to the database and search index in different orders, causing permanent inconsistency (DB says X=B, index says X=A). (2) Partial failure: One write may succeed while the other fails, and ensuring atomic commit across heterogeneous systems is expensive. The solution is to designate one system as leader and use its change stream (CDC) to update all other systems, guaranteeing a single source of truth and consistent ordering.
5How does Change Data Capture (CDC) solve the problems that dual writes cannot?▼
CDC makes one database the single leader and extracts its change stream (by parsing the replication log or write-ahead log). All derived systems (search indexes, caches, warehouses) become followers that consume this stream in order. Since there is a single source of truth with a total order of writes, race conditions are eliminated. A log-based message broker (like Kafka) preserves this order during transport. If a derived system needs to be rebuilt, you can replay the change stream from the beginning (or from a snapshot).
6What is the key difference between event sourcing and change data capture?▼
The level of abstraction. CDC operates at the database level: it extracts low-level row changes (inserts, updates, deletes) from the replication log. The application writes to the database normally and doesn't need to know CDC exists. Event sourcing operates at the application level: events represent meaningful user actions ("student cancelled enrollment") and are explicitly designed as immutable, append-only records. This has implications for log compaction -- CDC can compact (latest value per key suffices), but event sourcing generally cannot (later events don't override prior ones; full history is needed to reconstruct state).
7What does "the database is a cache of the log" mean, and why is this a powerful mental model?▼
Pat Helland's insight: the append-only log of all changes is the truth (the system of record). The current database state is just a materialized view -- a cached subset showing the latest value of each record. This mental model is powerful because it means you can derive any number of different read-optimized views from the same log (CQRS), evolve your application by adding new views without modifying existing systems, and recover from bugs by replaying the log. State = integral of the event stream over time; the change stream = derivative of state.
8Why is confusing event time with processing time dangerous in stream processing?▼
If you use the processing machine's clock to assign events to time windows, any processing delay corrupts your measurements. Example: if the stream processor restarts and processes a backlog, all the buffered events land in the same processing-time window, creating a false spike in metrics. The actual event rate may have been perfectly steady. Event time (the timestamp of when things actually happened) produces correct, deterministic results. But it comes with its own challenges: events arrive out of order, and you can never be sure all events for a window have arrived (the "straggler" problem).
9What are the four types of time windows, and when would you use each?▼
Tumbling: Fixed-size, non-overlapping (e.g., 1-min buckets). Simplest. Every event belongs to exactly one window. Good for basic counting. Hopping: Fixed-size, overlapping (e.g., 5-min window with 1-min hop). Provides smoothing. Events belong to multiple windows. Sliding: Contains all events within some interval of each other. No fixed boundaries. Good when you care about relative proximity of events (e.g., "all events within 5 min of each other"). Session: No fixed duration. Groups events for the same user that occur close together; ends after inactivity timeout. Used for website analytics and user behavior tracking.
10Explain the three types of stream joins and how they relate to each other.▼
All three require maintaining state from one input and querying it on the other. Stream-stream join: Both inputs are event streams. Join events that occur within a time window (e.g., search + click within 1 hour). State = indexed events in the window. Stream-table join: One input is events, the other is a database changelog. Enrich events with current DB data. State = local copy of table (updated via CDC). Like stream-stream but with a "beginning of time" window for the table. Table-table join: Both inputs are changelogs. Maintains a materialized view of a join between two tables (e.g., Twitter timeline = join of tweets and follows). Any change on one side joins with the latest state of the other.
11Why can't microbatching and checkpointing alone guarantee exactly-once processing end-to-end?▼
Microbatching and checkpointing provide exactly-once semantics within the stream processing framework. But as soon as output leaves the framework -- writing to an external database, sending an email, making an HTTP call -- the framework cannot discard or roll back that side effect. If a failure occurs after the external write but before the checkpoint, restarting will process the batch again and perform the side effect a second time. To handle this, you need either atomic commit (transactions that span the framework and external system) or idempotence (making the external operation safe to repeat).
12How does idempotence enable exactly-once semantics, and what assumptions does it require?▼
An idempotent operation has the same effect whether performed once or multiple times (e.g., "set key to value X" is idempotent; "increment counter" is not). Even non-idempotent operations can be made idempotent by storing metadata: include the Kafka offset of the message that triggered a write alongside the value. On retry, check if the offset matches; if so, the update was already applied -- skip it. This requires three assumptions: (1) replay delivers the same messages in the same order (log-based brokers guarantee this), (2) processing is deterministic, and (3) no other node concurrently writes to the same value. With these guarantees, idempotence achieves effectively-once semantics with minimal overhead.
