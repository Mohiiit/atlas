DDIA — Part I: Foundations of Data Systems

Chapter 4: Encoding and Evolution

"Everything changes and nothing stands still." — Heraclitus. Applications change over time. Old and new code, old and new data formats, must coexist. This chapter is about how to make that work.

The Core Problem: Compatibility

During rolling upgrades, different nodes run different code versions simultaneously. Old and new data formats coexist. We need two kinds of compatibility:

Backward Compatibility

Newer code can read data written by older code.

Usually easy — you wrote the new code, you know the old format.

Forward Compatibility

Older code can read data written by newer code.

Trickier — requires old code to ignore additions made by newer versions.

Formats for Encoding Data

Programs work with data in two representations: in-memory (objects, structs, trees with pointers) and byte sequences (for files/network). Translation = encoding (serialization/marshalling). Reverse = decoding (parsing/deserialization).

Language-Specific Formats: Avoid Them

Java Serializable, Python pickle, Ruby Marshal

  • Tied to one language — can't read in other languages
  • Security risk: decoding can instantiate arbitrary classes → remote code execution
  • Versioning is an afterthought (poor compatibility)
  • Often terrible performance (Java serialization is notorious)

Rule: Never use language-specific encoding for anything beyond transient purposes.

JSON, XML, CSV: Widespread but Flawed

Subtle Problems

  • Number ambiguity: JSON can't distinguish int vs float; integers > 253 lose precision in JavaScript. Twitter includes tweet IDs twice (number + string) as a workaround.
  • No binary strings: Must Base64-encode binary data (+33% size).
  • Schema optional: Without schema, interpretation of numbers/binary depends on app-level convention.
  • CSV: No schema at all, vague format (comma in value? newline?), not all parsers handle escaping correctly.

Good enough for interchange between organizations. For internal use, binary formats are better.

Binary Schema-Driven Formats

The key insight: if you have a schema, you don't need to include field names in every record. Use field tags (numbers) instead → much more compact.

Thrift (Facebook)

Schema with numbered field tags. BinaryProtocol (verbose) and CompactProtocol (packs type + tag into one byte, variable-length ints). Dedicated list datatype.

Protocol Buffers (Google)

Very similar to Thrift. One binary format. Uses repeated instead of list type — so optional can evolve to repeated. Variable-length ints.

Avro (Hadoop)

No tag numbers at all. Fields matched by name between writer's schema and reader's schema. Most compact. Great for dynamically generated schemas.

Schema Evolution: The Rules

Thrift & Protocol Buffers

Field Tags Are Sacred

  • You can change field names freely (encoded data uses tag numbers, not names)
  • You can never change a field's tag number — would invalidate all existing data
  • Adding fields: give new tag number. Old code ignores unknown tags (forward compatible). New fields must be optional or have defaults (backward compatible)
  • Removing fields: can only remove optional fields. Never reuse the tag number
  • Changing datatypes: possible but risky (e.g., 32-bit → 64-bit: old code truncates large values)

Avro

Writer's Schema vs. Reader's Schema

Key idea: the writer's and reader's schemas don't have to be the same — they just need to be compatible. The Avro library resolves differences by matching fields by name.

  • Field in writer but not reader → ignored
  • Field in reader but not writer → filled with default value
  • You may only add/remove fields that have a default value
  • No tag numbers → great for dynamically generated schemas (e.g., dump a relational DB: column names become field names, generate new schema on every export)

How Does the Reader Know the Writer's Schema?

  • Large file (Hadoop): writer's schema included once at the beginning of the file
  • Database records: version number per record → look up schema in a schema registry
  • Network connection: negotiate schema on connection setup

The Merits of Schemas (Over Schemaless JSON)

  • Much more compact — omit field names from encoded data
  • Schema = living documentation (required for decoding, so always up to date)
  • Schema registry enables compatibility checks before deployment
  • Code generation for statically typed languages → compile-time type checking
  • Schema evolution gives the same flexibility as schemaless, with better guarantees

Modes of Dataflow

Three main ways data flows between processes, each with different compatibility implications:

Three Modes of Dataflow
1. Databases

Writer encodes → Reader decodes. "Sending a message to your future self." Need both backward + forward compatibility. Beware: old code reading new data, updating, writing back → may lose unknown fields.

"Data outlives code" — 5-year-old data still in original encoding unless migrated.

2. Services (REST/RPC)

Client encodes request → Server decodes → Server encodes response → Client decodes. Servers updated first, clients second. Need backward compat on requests, forward compat on responses.

3. Message Passing

Sender → Message Broker → Receiver. Asynchronous, one-way. Broker buffers if recipient unavailable, redelivers on crash. Sender doesn't know/care who consumes. Loosely coupled.

Why RPC Is Fundamentally Flawed (as a Concept)

A network request is NOT like a local function call:

  • Network is unpredictable — requests/responses can be lost
  • Timeouts: you don't know if request got through or not
  • Retries may cause duplicate execution (need idempotence)
  • Latency is wildly variable (ms to seconds)
  • Parameters must be serialized (can't pass pointers)
  • Client and server may use different languages (type mismatch)

Modern RPC frameworks (gRPC, Finagle) acknowledge this: they use futures/promises, streams, and don't pretend remote = local.

Message Brokers & Actors

Message Broker Advantages

  • Buffer if recipient unavailable
  • Auto-redeliver on crash
  • Sender doesn't need recipient's address
  • One message → many recipients
  • Decouples sender from receiver

Examples: RabbitMQ, ActiveMQ, NATS, Apache Kafka

Actor Model

  • Concurrency model: logic in actors, not threads
  • Each actor has local state, communicates via async messages
  • Distributed actors: same model across nodes (Akka, Orleans, Erlang OTP)
  • Location transparency works better here than RPC (actors already assume messages may be lost)
  • Still need forward/backward compatibility for rolling upgrades

First-Principles Questions

1Why do we need both backward and forward compatibility?▼
During rolling upgrades, some nodes run new code while others still run old code. New code must read old data (backward compatibility). Old code must be able to handle data from new code without breaking (forward compatibility). Without both, you can't do rolling upgrades — you'd have to update all nodes simultaneously, which means downtime.
2Why are language-specific serialization formats (Java Serializable, Python pickle) dangerous?▼
Four problems: (1) Tied to one language — can't read in other languages. (2) Security: decoding can instantiate arbitrary classes, enabling remote code execution attacks. (3) Versioning/compatibility is an afterthought. (4) Often terrible performance and bloated encoding (Java serialization is notorious). Never use for anything beyond transient in-process purposes.
3How do Thrift/Protocol Buffers achieve forward compatibility?▼
Fields are identified by tag numbers (not names) in the encoded data. When old code encounters a field with a tag number it doesn't recognize, the datatype annotation tells the parser how many bytes to skip. The unknown field is simply ignored. This means old code can read records written by new code without breaking.
4Why can you never change a field's tag number in Thrift/Protobuf, and never reuse a removed field's tag?▼
The tag number is the identity of the field in encoded data. Changing it would make all existing encoded data unreadable (the parser would interpret the old tag as a different field). Reusing a removed tag is equally dangerous: old data in the wild may still contain records with that tag number, and the parser would misinterpret them as the new field.
5How is Avro's approach to schema evolution fundamentally different from Thrift/Protobuf?▼
Avro has no tag numbers. Fields are matched by name between the writer's schema and reader's schema. The Avro library resolves differences: fields in writer but not reader are ignored; fields expected by reader but missing from writer are filled with defaults. This makes Avro ideal for dynamically generated schemas (e.g., auto-generating from a database schema where column names map directly to field names).
6What does "data outlives code" mean, and what problem does it create?▼
You can replace all running code with a new version in minutes (rolling upgrade). But database contents stay in their original encoding unless explicitly migrated — data written 5 years ago is still there in the old format. This means a database may contain a mix of data encoded with many different historical schema versions, and schema evolution must handle all of them. The database needs to appear as if all data conforms to one schema.
7Why is a network request fundamentally different from a local function call?▼
Six key differences from the book: (1) Network is unpredictable — requests/responses can be lost. (2) Timeouts: no way to know if request succeeded. (3) Retries may execute the action multiple times (need idempotence). (4) Latency is wildly variable. (5) Parameters must be serialized to bytes (can't pass pointers). (6) Client and server may use different languages with different type systems. This is why the RPC model of "location transparency" is fundamentally flawed.
8What are the advantages of a message broker over direct RPC?▼
Five advantages: (1) Acts as a buffer if recipient is unavailable/overloaded. (2) Auto-redelivery to crashed processes. (3) Sender doesn't need to know recipient's IP/port (useful in cloud with dynamic VMs). (4) One message can be sent to multiple recipients. (5) Logical decoupling — sender publishes and doesn't care who consumes. The trade-off: communication is one-way and asynchronous (sender doesn't get a direct response).
9Why does the book say JSON's handling of numbers is problematic?▼
JSON distinguishes strings from numbers but doesn't distinguish integers from floats and doesn't specify precision. Integers greater than 253 cannot be exactly represented in IEEE 754 double-precision floating point (which JavaScript uses). Twitter hit this: tweet IDs are 64-bit numbers that get corrupted when parsed as JavaScript numbers. Their API includes each tweet ID twice — once as a JSON number and once as a string.
10Why does the actor model handle distribution better than traditional RPC?▼
The actor model already assumes that messages may be lost, even within a single process. So when you extend it across nodes (distributed actors), there is less of a fundamental mismatch between local and remote communication. RPC tries to pretend remote calls are like local calls (location transparency), which fails because network introduces unpredictability that local calls don't have. The actor model embraces this reality from the start.
