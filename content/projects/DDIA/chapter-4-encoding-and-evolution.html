DDIA — Part I: Foundations of Data Systems

Chapter 4: Encoding and Evolution

"Everything changes and nothing stands still." — Heraclitus of Ephesus, as quoted by Plato in Cratylus (360 BCE). Applications change over time. Old and new code, old and new data formats, must coexist. This chapter is about the formats and mechanisms that make that work — encoding data for storage and transmission, evolving schemas without breaking things, and the many ways data flows between processes.

The Core Problem: Compatibility During Change

Applications inevitably change over time. Features are added or modified as new products are launched, user requirements become better understood, or business circumstances change. A change to application features usually requires a change to the data it stores — perhaps a new field or record type, or existing data presented in a new way.

However, in a large application, code changes cannot happen instantaneously. With server-side applications, you perform a rolling upgrade (staged rollout), deploying new versions to a few nodes at a time. With client-side applications, users may not install the update for weeks or months. This means old and new versions of code, and old and new data formats, coexist simultaneously.

Backward Compatibility

Newer code can read data written by older code.

Usually not hard to achieve: as the author of the newer code, you know the format of data written by older code, and you can explicitly handle it.

Forward Compatibility

Older code can read data written by newer code.

Trickier — requires old code to gracefully ignore additions made by newer versions of the code it doesn't know about.

Formats for Encoding Data

Programs work with data in two representations: in-memory (objects, structs, hash tables, trees with pointers — optimized for CPU access and manipulation) and byte sequences (for files or network — a self-contained sequence since pointers are meaningless to other processes). The translation from in-memory to bytes is called encoding (also serialization or marshalling). The reverse is decoding (parsing, deserialization, unmarshalling).

"Serialization" is unfortunately also used in the context of transactions (Chapter 7) with a completely different meaning. To avoid overloading the word, the book sticks with "encoding."

Language-Specific Formats: Avoid Them

Many languages have built-in encoding: Java has java.io.Serializable, Ruby has Marshal, Python has pickle, plus third-party libraries like Kryo for Java. They are convenient but deeply problematic.

JSON, XML, CSV: Widespread but Flawed

JSON and XML are the obvious contenders for standardized, language-independent encoding. JSON's popularity stems from its built-in support in web browsers (being a subset of JavaScript) and simplicity relative to XML. CSV is another popular format, albeit less powerful. All three are textual and somewhat human-readable, but have subtle problems.

Subtle Problems with Textual Formats

  • Number ambiguity: XML and CSV cannot distinguish between a number and a string of digits (without an external schema). JSON distinguishes strings from numbers but doesn't distinguish integers from floats and doesn't specify precision. Integers > 253 cannot be exactly represented in IEEE 754 double-precision floating point (which JavaScript uses). Twitter includes tweet IDs twice — as a JSON number and as a decimal string — as a workaround.
  • No binary strings: JSON and XML support Unicode text but not binary data. Must Base64-encode binary data, increasing size by 33%. Schema indicates the value should be interpreted as Base64.
  • Schema optional: XML Schema and JSON Schema are powerful but complicated. Many JSON tools don't bother using schemas, so correct interpretation of numbers and binary depends on hardcoded application logic.
  • CSV: No schema at all. Vague format (what if a value contains a comma or newline?). Escaping rules are formally specified but not all parsers implement them correctly.

Despite these flaws, JSON/XML/CSV are good enough for many purposes — especially as interchange formats between organizations. The difficulty of getting different organizations to agree on anything outweighs format concerns.

Binary Encoding Formats

For data used only internally within your organization, you can choose more compact and faster formats. JSON is less verbose than XML, but both use far more space than binary formats. This led to binary encodings for JSON (MessagePack, BSON, BJSON, UBJSON, BISON, Smile) and XML (WBXML, Fast Infoset). These keep the JSON/XML data model unchanged and since they don't prescribe a schema, they include all field names in the encoded data.

MessagePack: Binary JSON (Not Much Better)

MessagePack is a binary encoding for JSON. The first byte 0x83 indicates an object (top 4 bits = 0x80) with 3 fields (bottom 4 bits = 0x03). Next byte 0xa8 indicates a string (top 4 bits = 0xa0) that is 8 bytes long (bottom 4 bits = 0x08). Then 8 bytes for "userName" in ASCII. The result: 66 bytes — only a little less than the 81-byte textual JSON. Not worth the loss of human-readability.

Thrift (Facebook) and Protocol Buffers (Google)

Both Thrift (originally Facebook, open-sourced 2007) and Protocol Buffers (originally Google, open-sourced 2008) are binary encoding libraries based on the same principle: define a schema with numbered field tags, generate code for your programming language, encode data using tag numbers instead of field names.

Thrift IDL Schema

struct Person {
  1: required string userName,
  2: optional i64    favoriteNumber,
  3: optional list<string> interests
}

Protocol Buffers Schema

message Person {
  required string user_name       = 1;
  optional int64  favorite_number = 2;
  repeated string interests       = 3;
}

Both come with code generation tools that produce classes implementing the schema in various programming languages. An important note: required vs optional makes no difference to encoding — nothing in the binary data indicates which it is. The difference is purely a runtime check that fails if a required field is not set.

Thrift vs Protobuf: Key Difference in List Handling

Thrift: Dedicated list datatype, parameterized with element type. Supports nested lists. But cannot evolve single-valued to multi-valued.

Protobuf: Uses repeated marker — same tag appears multiple times. Can evolve optional → repeated. Old code reading new multi-valued field sees only last element. But no nested lists.

Schema Evolution: The Rules

Schemas inevitably change over time. We call this schema evolution. The question is: how do Thrift, Protocol Buffers, and Avro handle schema changes while keeping backward and forward compatibility?

Thrift & Protocol Buffers: Field Tags Are Sacred

Must never reuse the tag number — old data with tag 3 would be misinterpreted.

The Cardinal Rules:

Change field names freely (not in encoded data) NEVER change a field's tag number NEVER reuse a removed tag number New fields must be optional or have defaults Datatype changes: possible but risky (truncation)

Datatypes and Schema Evolution

Changing a field's datatype is possible but risky. Example: change 32-bit integer to 64-bit. New code easily reads old data (fills missing bits with zeros). But old code reading new data truncates — a 64-bit value won't fit in a 32-bit variable, and large values are silently corrupted.

Avro: A Fundamentally Different Approach

Apache Avro was started in 2009 as a subproject of Hadoop, because Thrift wasn't a good fit for Hadoop's use cases. Avro is interestingly different from Protocol Buffers and Thrift in a fundamental way: there are no tag numbers in the schema.

Avro IDL (Human-Editable)

record Person {
  string              userName;
  union {null, long}  favoriteNumber = null;
  array<string>       interests;
}

Avro JSON Schema (Machine-Readable)

{
  "type": "record",
  "name": "Person",
  "fields": [
    {"name":"userName","type":"string"},
    {"name":"favoriteNumber",
     "type":["null","long"],
     "default":null},
    {"name":"interests",
     "type":{"type":"array",
             "items":"string"}}
  ]
}

Notice: no tag numbers. The Avro binary encoding is just 32 bytes — the most compact of all. The encoding consists of values concatenated together with nothing to identify fields or their datatypes. A string is just a length prefix followed by UTF-8 bytes. An integer uses variable-length encoding. To parse the binary data, you go through fields in schema order and use the schema to determine each field's datatype. The binary can only be decoded correctly if the reader uses the exact same schema as the writer — or a compatible one.

Diagram 9: Avro Schema Resolution — Writer's Schema vs Reader's Schema

The key idea: the writer's and reader's schemas don't have to be identical — they just need to be compatible. The Avro library resolves differences by matching fields by name.

Diagram 10: Avro Schema Evolution Rules

Allowed

  • Add a field with a default value
  • Remove a field that has a default value
  • Change datatype (if Avro can convert)
  • Change field name using aliases (backward compat only)
  • Reorder fields (matched by name, not position)

Not Allowed / Dangerous

  • Add field without default → breaks backward compat
  • Remove field without default → breaks forward compat
  • Change field name without alias → breaks backward compat
  • Adding a union branch → backward compat only (not forward)

Avro's Null Handling (Different from Protobuf/Thrift)

Avro doesn't have optional/required markers. Instead, use union types. To allow null, declare union {null, long}. You can only use null as a default if null is one of the union branches. This is more verbose but prevents bugs by being explicit about nullability.

Dynamically Generated Schemas: Avro's Killer Feature

Why does it matter that Avro has no tag numbers? Because Avro is friendlier to dynamically generated schemas. Example: you have a relational database whose contents you want to dump to a binary file. With Avro, generate a record schema for each table — each column becomes a field, column names map directly to field names. If the database schema changes (column added/removed), just generate a new Avro schema. The data export process doesn't need to care about schema changes.

With Thrift or Protocol Buffers, field tags would have to be assigned by hand. Every time the database schema changes, an administrator would have to manually update the mapping from column names to field tags. This kind of dynamically generated schema simply wasn't a design goal of Thrift or Protobuf, whereas it was for Avro.

Code Generation and Dynamically Typed Languages

Thrift and Protocol Buffers rely on code generation: define a schema, generate classes for your language. This is useful in statically typed languages (Java, C++, C#) for efficient in-memory structures, type checking, and IDE autocompletion. In dynamically typed languages (JavaScript, Ruby, Python), code generation is less useful since there's no compile-time type checker.

Avro provides optional code generation but can work without it. An Avro object container file embeds the writer's schema and is self-describing — you can open it and inspect the data like a JSON file. This is especially useful with data processing tools like Apache Pig, which can open Avro files and start analyzing them without thinking about schemas.

Why Schema-Driven Formats Win for Internal Use

  • Much more compact — omit field names from encoded data (32 bytes vs 81 bytes for the same record)
  • Schema is a valuable form of documentation — required for decoding, so always up to date (unlike manually maintained docs that diverge from reality)
  • Schema registry enables compatibility checks before deployment — catch breaking changes before they hit production
  • Code generation for statically typed languages enables compile-time type checking and IDE autocompletion
  • Schema evolution provides the same flexibility as schemaless/schema-on-read JSON databases, with better guarantees about your data and better tooling

Historical note: these ideas are not new. ASN.1, standardized in 1984, uses tag numbers similar to Protobuf/Thrift. Its binary encoding (DER) is still used for SSL certificates (X.509).

Dataflow Through Databases

In a database, the process that writes encodes the data, and the process that reads decodes it. This may be a single process (the reader is a later version of yourself) or different processes accessing the database concurrently. Either way, it's like "sending a message to your future self."

Solution: Application code must preserve unknown fields when reading and re-writing a record. Encoding libraries like Avro can handle this automatically.

"Data Outlives Code"

You can replace all running code with a new version in minutes (rolling upgrade). But database contents stay in their original encoding unless explicitly migrated — data written 5 years ago is still there in the old format. A single database may contain a mix of data encoded with many different historical schema versions.

Schema evolution allows the entire database to appear as if encoded with a single schema (the latest), even though underlying records use many different historical versions. Schema migrations (e.g., ALTER TABLE) are possible but expensive on large tables. Most relational databases allow simple additions (adding a nullable column) without rewriting existing data — old rows read with null for the new column.

Dataflow Through Services: REST and RPC

When processes communicate over a network, the most common arrangement is clients and servers. The server exposes an API (a service), and the client connects and makes requests. The API consists of a standardized set of protocols and data formats. The web works this way: browsers (clients) make requests to web servers.

A server can itself be a client to another service. This is the service-oriented architecture (SOA), more recently rebranded as microservices architecture. The key idea: make services independently deployable and evolvable. Expect old and new versions of servers and clients to be running simultaneously, so encoding must be compatible across versions.

Why "Location Transparency" in RPC Is Fundamentally Flawed

RPC frameworks try to make a remote call look like a local function call. This is a leaky abstraction. A network request is NOT like a local function call:

  • Unpredictable network: requests and responses can be lost, delayed, or duplicated
  • Timeouts: a local call either returns or throws an exception. A network call might just hang — you don't know if the request got through
  • Retries may duplicate: if a timeout occurs, the request may have been executed. Retrying may execute it again. Need idempotence
  • Wildly variable latency: local call is nanoseconds. Network call is milliseconds to seconds. Fundamentally different performance model
  • Parameters must be serialized: can't pass pointers, references, or complex in-memory objects. Everything must be encoded to bytes
  • Cross-language type mismatch: client and server may use different languages with different type systems. A JavaScript number, a Java long, and a Go int64 are not the same thing

Modern RPC frameworks (gRPC, Finagle, Rest.li) acknowledge this reality: they use futures/promises, streams, and service discovery. They don't pretend that remote = local. The distinction between local and remote is an important part of the API design.

RPC Compatibility: Custom Approaches

RPC compatibility depends on the encoding format used:

  • Thrift/gRPC/Avro RPC: inherit schema evolution rules from their respective encoding formats
  • SOAP: requests and responses are XML, schema managed through WSDL
  • RESTful APIs: use JSON, commonly managed through API versioning (URL paths like /api/v1/, or HTTP headers)

For RESTful APIs, it is reasonable to expect all servers to be updated first, then clients. Thus you only need backward compatibility on requests (new server handles old client requests) and forward compatibility on responses (old client handles new server responses).

Message-Passing Dataflow

Asynchronous message passing is between direct RPC (one process sends request, expects response) and databases (one process writes, another reads later). A message broker (message queue) is an intermediary: the sender sends a message to a named queue or topic, the broker delivers it to one or more consumers/subscribers.

Message Broker Advantages Over Direct RPC

  • Buffer if recipient unavailable: acts as a queue, improving system reliability. If the consumer crashes, messages wait
  • Auto-redeliver on crash: prevents messages from being lost
  • Sender doesn't need recipient's address: useful in cloud environments with dynamically created/destroyed VMs
  • One message → many recipients: a single message published to a topic can be consumed by multiple subscribers
  • Logical decoupling: sender publishes without knowing or caring who consumes. This makes it easy to add new consumers without modifying the sender

The communication is one-way and asynchronous: the sender fires and forgets. If a response is needed, it's done via a separate message on a different channel (the "reply queue" pattern). Typically the sender doesn't wait for the message to be consumed.

Message brokers typically don't enforce any particular data model — a message is just a sequence of bytes with some metadata. You can use any encoding format. If the encoding is forward and backward compatible, you have the flexibility to independently change publishers and consumers, deploying them in any order.

Distributed Actor Frameworks

The actor model is a programming model for concurrency in a single process. Rather than dealing with threads directly, logic is encapsulated in actors. Each actor has its own local state and communicates with other actors by sending and receiving asynchronous messages. Message delivery is not guaranteed — even within a single process, messages may be lost.

Why Actors Handle Distribution Better Than RPC

The actor model already assumes messages may be lost, even within a single process. When you extend it across nodes (distributed actors), there is less of a fundamental mismatch between local and remote communication. RPC tries to pretend remote calls are like local calls ("location transparency"), which fails because the network introduces unpredictability that local calls don't have. The actor model embraces this reality from the start — it was designed for unreliable message delivery.

However, there's a caveat: if you do a rolling upgrade of your actor system, you still need forward and backward compatibility. Akka's default of using Java serialization is a known pain point — many teams switch to Protocol Buffers or a custom serializer for production use.

First-Principles Questions

1Why do we need BOTH backward and forward compatibility? Why isn't one sufficient?▼
During rolling upgrades, some nodes run new code while others still run old code. New code must read old data (backward compatibility) — this is relatively easy because you wrote the new code and know the old format. Old code must handle data from new code without breaking (forward compatibility) — this is trickier because old code must gracefully ignore additions. Without both, you cannot do rolling upgrades. You'd have to update all nodes simultaneously, which means downtime. For client-side applications, it's even worse: you have no control over when users upgrade, so you may have many versions in the wild simultaneously.
2Why are language-specific serialization formats (Java Serializable, Python pickle) dangerous for anything beyond transient purposes?▼
Four deep problems: (1) Language lock-in: encoding is tied to one language. If you store data this way, you're committed to that language potentially forever, precluding integration with systems in other languages. (2) Security: decoding requires instantiating arbitrary classes. An attacker who can feed your application a crafted byte sequence can instantiate arbitrary classes, leading to remote code execution. (3) No versioning: forward and backward compatibility are afterthoughts, making schema evolution painful or impossible. (4) Performance: Java's built-in serialization is notorious for bad performance and bloated output. These aren't minor issues — they're fundamental architectural risks.
3Why is MessagePack (binary JSON) only marginally smaller than textual JSON, while Protobuf is dramatically smaller?▼
MessagePack keeps the JSON data model unchanged: it still includes all field names ("userName", "favoriteNumber", "interests") as strings in the encoded data, because it has no schema. It just encodes them more efficiently as binary. The field names dominate the payload. Protocol Buffers uses a schema with numbered field tags, so field names never appear in the encoded data — only small tag numbers (1, 2, 3). Additionally, Protobuf uses variable-length integer encoding (small numbers use fewer bytes), while MessagePack must encode field name strings in full. The lesson: the biggest savings come from omitting field names, not from encoding values in binary.
4How do Thrift/Protocol Buffers achieve forward compatibility (old code reads new data)?▼
Fields are identified by tag numbers (not names) in the encoded data. Each field also has a datatype annotation. When old code encounters a field with a tag number it doesn't recognize (a new field added by newer code), the datatype annotation tells the parser how many bytes to skip. The unknown field is simply ignored. This is why old code can safely read records written by new code — it just ignores the fields it doesn't understand. The same mechanism is why new fields must be optional or have defaults: if new code reads old data, the new field simply won't be present, and the code must handle that gracefully.
5Why can you never change a field's tag number in Protobuf/Thrift, and why can you never reuse a removed field's tag?▼
The tag number is the identity of the field in encoded data. Changing it would make all existing encoded data unreadable — the parser would interpret the old tag as a different field, causing data corruption. Reusing a removed tag is equally dangerous: old data in the wild (in databases, log files, message queues) may still contain records with that tag number. The parser would silently misinterpret the old field as the new field, causing subtle data corruption. These rules are absolute because encoded data can persist for years — much longer than any code version.
6How is Avro's approach to schema evolution fundamentally different from Thrift/Protobuf, and what use case does this enable?▼
Avro has no tag numbers at all. Fields are matched by name between the writer's schema and reader's schema. The Avro library resolves differences: fields in writer but not reader are ignored; fields expected by reader but missing from writer are filled with defaults. This enables dynamically generated schemas. Example: you want to dump a relational database to a binary file. With Avro, you auto-generate a schema from the database schema (column names become field names), export the data, and when the database schema changes, just generate a new Avro schema. With Thrift/Protobuf, you'd need to manually assign tag numbers every time a column is added — you'd have to track which tags were previously used. Avro was specifically designed for this Hadoop use case.
7What does "data outlives code" mean, and what specific danger does it create when old code writes to a database?▼
You can replace all running code in minutes (rolling upgrade). But database contents stay in their original encoding unless explicitly migrated — data written 5 years ago is still in the old format. This creates a specific danger: during a rolling upgrade, an old application instance may read a record written by new code (which includes a new field like "email"), update some other field, and write the record back without the unknown "email" field — silently losing data. The solution: application code (or the encoding library) must preserve unknown fields when reading and re-writing records. Not all encoding libraries handle this correctly.
8Why is the RPC concept of "location transparency" fundamentally flawed?▼
Location transparency tries to make remote calls look identical to local function calls. This is a leaky abstraction because of six fundamental differences: (1) Network is unpredictable — requests/responses can be lost. (2) Timeouts: no way to know if the request succeeded, partially succeeded, or never arrived. (3) Retries may execute the action multiple times (need idempotence). (4) Latency is wildly variable (nanoseconds for local vs milliseconds-to-seconds for network). (5) Parameters must be serialized to bytes (can't pass pointers or complex objects). (6) Client and server may use different languages with different type systems. Hiding these differences behind a function call interface leads to brittle, poorly-designed systems. Modern RPC frameworks (gRPC) embrace the difference by exposing futures/streams/error handling explicitly.
9What are the five advantages of a message broker over direct RPC, and what is the fundamental trade-off?▼
Five advantages: (1) Acts as a buffer if recipient is unavailable or overloaded — messages wait in the queue. (2) Auto-redelivery: if a consumer crashes after receiving a message, the broker can redeliver to another consumer. (3) Sender doesn't need to know recipient's IP/port — useful in cloud environments with dynamically created VMs. (4) One message can be delivered to multiple recipients via topics/subscriptions. (5) Logical decoupling: sender publishes without knowing who consumes, making it easy to add new consumers. The fundamental trade-off: communication is one-way and asynchronous. The sender doesn't get a direct response. If request-response is needed, you need a separate reply queue pattern, adding complexity.
10Why does the actor model handle distribution better than traditional RPC?▼
The actor model already assumes that messages may be lost, even within a single process. Communication between actors is always asynchronous and one-way — an actor sends a message and doesn't wait for a response. When you extend this across nodes (distributed actors), there is less of a fundamental mismatch between local and remote communication. RPC tries to pretend remote calls are like local calls, which fails because network introduces unpredictability. The actor model embraces unreliable delivery from the start. That said, there are still differences: network latency is higher and message loss is more common across nodes. And for rolling upgrades, you still need encoding compatibility.
11Why does JSON's number handling cause real-world bugs, and why did Twitter have to work around it?▼
JSON distinguishes strings from numbers but doesn't distinguish integers from floating-point numbers and doesn't specify precision. Most JSON parsers (especially JavaScript's) parse numbers as IEEE 754 double-precision floating point. Integers greater than 253 (about 9 quadrillion) cannot be exactly represented in this format — they lose precision. Twitter hit this: tweet IDs are 64-bit integers, which can exceed 253. When a JavaScript application parses a tweet ID as a JSON number, it gets silently corrupted — the last few digits change. Twitter's workaround: their API includes each tweet ID twice — once as a JSON number (for non-JS consumers) and once as a decimal string (for JavaScript consumers). This is a fundamental limitation of JSON for large numeric identifiers.
12In a microservices architecture, which compatibility direction matters for requests vs responses, and why?▼
In practice, servers are updated before clients. This means: for requests, you need backward compatibility — the new server must understand requests from old clients (which use the old request schema). For responses, you need forward compatibility — old clients must be able to handle responses from the new server (which may include new fields). Think of it this way: the server is always "newer" than the client. The server receives old-format requests (backward compat) and sends new-format responses that old clients must tolerate (forward compat). This is why API versioning (/api/v1/) exists — it lets the server know which request/response format the client expects.
