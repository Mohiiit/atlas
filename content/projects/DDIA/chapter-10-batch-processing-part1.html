Key Insight: Batch as Building Block

MapReduce (published 2004) was called "the algorithm that makes Google so massively scalable." While its importance is declining, it provides a clear mental model for understanding why and how batch processing is useful. Stream processing (Chapter 11) builds upon batch processing concepts.

Batch Processing with Unix Tools

Before diving into distributed batch systems, we examine Unix pipelines -- the original batch processing paradigm. The ideas from Unix carry directly over to large-scale distributed data systems.

Sorting vs. In-Memory Aggregation

  • Ruby/hash-table approach: Keep an in-memory hash table mapping URL → count. Working set depends on number of distinct URLs, not total log entries. Works well if it fits in memory.
  • Unix sort approach: Spills to disk when data exceeds memory. GNU sort automatically handles larger-than-memory datasets and parallelizes across CPU cores. Follows sequential I/O patterns (like SSTables).
  • Tradeoff: Hash table is faster if data fits in RAM; sort approach scales to datasets much larger than memory.

The Unix Philosophy

"We should have some ways of connecting programs like a garden hose -- screw in another segment when it becomes necessary to massage data in another way." -- Doug McIlroy, 1964 (inventor of Unix pipes)
Diagram 3: The Four Principles of Unix Philosophy

1. Do One Thing Well

To do a new job, build afresh rather than complicating old programs by adding new "features".

sort uniq grep awk

2. Expect Composability

Expect the output of every program to become the input to another, as yet unknown, program. Don't clutter output with extraneous information.

3. Build Fast, Iterate

Design and build software to be tried early, ideally within weeks. Don't hesitate to throw away clumsy parts and rebuild.

4. Use Tools Over Manual Labor

Use tools in preference to unskilled help to lighten a programming task, even if you have to detour to build the tools.

This sounds remarkably like Agile and DevOps -- surprisingly little has changed in four decades.

What Makes Unix Composability Work?

Uniform Interface

  • Everything is a file descriptor (ordered sequence of bytes)
  • Files, sockets, pipes, devices -- all share the same interface
  • Convention: ASCII text with \n as record separator
  • Fields separated by whitespace/tabs/commas

Separation of Logic and Wiring

  • Programs read from stdin, write to stdout
  • The shell user wires up I/O, not the program
  • This is a form of loose coupling / inversion of control
  • Limitation: tricky for programs needing multiple inputs/outputs

Transparency and Experimentation

  • Immutable inputs: Commands never modify input files -- run as often as you want without damage
  • Inspect anytime: Pipe intermediate output into less to inspect at any stage
  • Checkpoint stages: Write intermediate output to a file, restart later stages without re-running everything

Biggest limitation: Unix tools run on a single machine only. That is where Hadoop comes in.

HDFS: Hadoop Distributed File System

MapReduce reads and writes files on a distributed filesystem. HDFS is Hadoop's implementation, an open-source reimplementation of Google File System (GFS). It uses the shared-nothing principle: ordinary commodity machines connected by a conventional datacenter network, no special hardware.

Each block is replicated across multiple DataNodes for fault tolerance. Blocks are typically hundreds of MB. The biggest HDFS deployments run on tens of thousands of machines with hundreds of petabytes of storage.

Shared-Nothing vs. Shared-Disk

  • Shared-disk (NAS/SAN): Centralized storage appliance, custom hardware, Fibre Channel
  • Shared-nothing (HDFS): Commodity machines, conventional network, no special hardware

Fault Tolerance

  • Replication: Multiple copies of same data on different machines
  • Erasure coding: Reed-Solomon codes for lower storage overhead than full replication
  • Similar to RAID but across machines over a network

MapReduce Job Execution

MapReduce is a programming framework for processing large datasets in a distributed filesystem. It is comparable to a single Unix process: takes inputs, produces outputs, does not modify input, and has no side effects.

The Shuffle in Detail

  1. Each map task partitions its output by reducer (hash of key)
  2. Each partition is written as a sorted file on the mapper's local disk (using SSTable-like technique)
  3. When a mapper finishes, the scheduler notifies reducers
  4. Reducers fetch (pull) sorted files from all mappers for their partition
  5. Reducer merges sorted files together, preserving sort order
  6. Reducer processes records with same key adjacent in merged stream

Note: Despite the name "shuffle," there is no randomness -- it is a deterministic sort-based redistribution of data.

Reduce-Side Joins: Sort-Merge Join

In batch processing, "join" means resolving all occurrences of an association across an entire dataset simultaneously -- not looking up one record at a time. MapReduce has no indexes; it performs full table scans, which makes sense for analytic queries over large datasets.

Example: User Activity + User Profile Join

You have a log of user activity events (fact table) and a user database (dimension table). You want to correlate activity with user profile info (e.g., which pages are popular with which age groups). Activity events contain only user_id, not full profile info.

Secondary sort ensures the user database record arrives before the activity events for each user_id. The reducer only needs to keep one user record in memory at a time.

Why "Sort-Merge"?

  • Sort: Mapper output is sorted by key
  • Merge: Reducers merge together the sorted lists from both sides of the join
  • All necessary data for a particular user_id is brought to the same place (a single reducer call)
  • No network requests needed during reduce -- all data is local

Mappers as Message Senders

A useful mental model: mappers "send messages" to reducers. The key acts like a destination address. All key-value pairs with the same key are delivered to the same reducer call. This separates physical network communication from application logic -- MapReduce handles all the networking, shielding application code from partial failures.

Handling Skew (Hot Keys)

The "bring all records with the same key to the same place" pattern breaks down when a single key has disproportionate data volume. A celebrity with millions of followers creates a hot key that overloads one reducer, causing the entire job to wait.

Map-Side Joins

Reduce-side joins are general-purpose but expensive due to all the sorting, copying, and merging. If you can make certain assumptions about input data, you can perform joins entirely in the map phase -- no reducers, no sorting.

Broadcast Hash Join

Assumption: One input is small enough to fit in memory.

  • Each mapper loads the entire small dataset into an in-memory hash table
  • Scans the large input, looks up each key in the hash table
  • "Broadcast" = small input sent to all map partitions
  • Alternative: store small input as a local read-only index on disk (page cache makes it fast)
Pig: replicated join Hive: MapJoin

Partitioned Hash Join

Assumption: Both inputs partitioned same way (same key, same hash function, same number of partitions).

  • Each mapper loads only its partition of the small dataset
  • Only records that could join are in the same partition
  • Much less memory than broadcast join
Hive: bucketed map join

Map-Side Merge Join

Assumption: Both inputs partitioned and sorted by the same key.

  • Mapper reads both files incrementally in ascending key order
  • Performs the same merge that a reducer would do
  • Works even if inputs don't fit in memory
  • Often possible when prior MR jobs created the sorted layout

Philosophy of Batch Process Outputs

Batch processing follows the same philosophy as Unix: inputs are immutable, outputs completely replace previous outputs, and there are no side effects.

Common Batch Outputs

  • Search indexes: Mappers partition documents, reducers build indexes (Lucene/Solr)
  • Key-value stores: Build database files in MapReduce, load in bulk (Voldemort, HBase)
  • ML models: Classifiers, recommendation systems
  • Aggregated reports: Analytics summaries

Benefits of Immutable Output

  • Rollback: If buggy code produces bad output, revert and re-run ("human fault tolerance")
  • Retry safety: Failed tasks are automatically re-run on same immutable input
  • Multiple consumers: Same files can be input for different jobs, monitoring, etc.
  • Separation of concerns: Logic separated from wiring (input/output directory config)

Why NOT Write Directly to a Database from MapReduce?

  • Too slow: Network request per record is orders of magnitude slower than batch throughput
  • Database overwhelmed: Many parallel map/reduce tasks writing concurrently can kill the DB
  • Breaks atomicity: MapReduce guarantees all-or-nothing output; external DB writes produce visible side effects from partially completed jobs

Better approach: Build database files as output, then atomically swap them into the serving layer (e.g., Voldemort serves old files while new ones are copied, then atomically switches).

Review Questions

1How does the Unix pipeline approach to log analysis differ from an in-memory hash table approach, and when is each preferable?
▾

The Unix pipeline relies on sorting -- repeated URLs become adjacent, and uniq -c counts them. The hash table approach (e.g., Ruby script) keeps a counter per URL in memory. The hash table is faster when the working set (distinct URLs) fits in memory. The sort approach is better when data exceeds available memory, because GNU sort automatically spills to disk and parallelizes across CPU cores, following sequential I/O patterns similar to SSTables. The sort approach easily scales to datasets much larger than RAM without running out of memory.

2What is the "shuffle" phase in MapReduce, and why is the term somewhat misleading?
▾

The shuffle is the process of partitioning mapper output by reducer (based on hash of key), sorting, and copying data partitions from mappers to reducers. Each map task writes sorted output files partitioned by reducer to local disk. When a mapper finishes, reducers are notified and pull the relevant files. Despite the name "shuffle" (which suggests randomness, like shuffling cards), the process is entirely deterministic -- it is a sort-based redistribution of data.

3Explain how secondary sort enables efficient sort-merge joins. What role does it play?
▾

In a sort-merge join between activity events and a user database, both are mapped with user_id as the key. Secondary sort ensures that within each user_id group, the user database record appears first, followed by activity events (e.g., in timestamp order). This means the reducer can read the user profile into a local variable, then iterate over all activity events for that user, outputting joined records. The reducer only needs to keep one user record in memory at any time and never needs network requests -- all data is local.

4What are the three types of map-side joins, and what assumptions does each require?
▾

1. Broadcast hash join: Requires one input to be small enough to fit in memory (or a local read-only index). Each mapper loads the entire small input into a hash table and scans the large input. 2. Partitioned hash join: Requires both inputs to be partitioned the same way (same key, hash function, partition count). Each mapper only loads its corresponding partition of the smaller input. 3. Map-side merge join: Requires both inputs to be partitioned and sorted by the same key. The mapper reads both files incrementally in ascending key order, performing the merge that a reducer would normally do. Each successive type requires stronger assumptions but uses less memory and can handle larger datasets.

5Why does MapReduce write intermediate results to HDFS between chained jobs instead of streaming them like Unix pipes?
▾

Chained MapReduce jobs write to HDFS (materialized intermediate state) rather than streaming through in-memory buffers because: (1) A batch job's output is only considered valid when the job has completed successfully -- MapReduce discards partial output of failed jobs. (2) The next job can only start when the prior job's output is fully written. (3) This enables fault tolerance -- if a job fails, its output is discarded and it can be re-run. The tradeoff is that this materialization is slower than Unix pipes (which use small in-memory buffers for direct process-to-process communication), but it provides stronger reliability guarantees for distributed execution.

6Why is writing MapReduce output directly to a production database a bad idea? What is the recommended alternative?
▾

Three reasons: (1) Network overhead: Making a network request per record is orders of magnitude slower than normal batch throughput. (2) Database overload: Many parallel map/reduce tasks writing concurrently can overwhelm the database and degrade query performance for other users. (3) Loss of atomicity: MapReduce normally provides clean all-or-nothing job output, but external DB writes create visible side effects from partially completed or failed jobs. The recommended alternative is to build immutable database files as MapReduce output (like Voldemort, ElephantDB, or HBase bulk loading), then atomically swap them into the serving layer. The server continues serving from old files while new ones are copied, then performs an atomic switch.
