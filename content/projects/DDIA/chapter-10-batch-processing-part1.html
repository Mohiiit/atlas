Chapter 10 — Batch Processing

Part 1: Unix Philosophy, MapReduce, Distributed Filesystems

Batch processing is the oldest form of data processing: take a large dataset, run a computation over it, produce output. Chapter 10 covers the tools and patterns that make this work at scale — from Unix pipes to MapReduce to modern dataflow engines.

Three Types of Systems

Services (online)
Wait for requests, respond quickly. Success = low latency. Examples: web servers, APIs, databases.
Batch processing (offline)
Run periodically on large input data sets. Success = high throughput. Examples: Hadoop, Spark jobs. Minutes to hours.
Stream processing (near-real-time)
Process events as they arrive. Between batch and online. Seconds latency. Examples: Kafka Streams, Flink.

The Unix Philosophy

Batch processing in Unix is built around a simple, powerful idea: small programs that do one thing well, composable via stdin/stdout pipes.

Unix philosophy: (1) Make each program do one thing well; (2) Expect output of every program to become input to another; (3) Design and build software to be tried early; (4) Use tools in preference to unskilled help.

Unix as a Batch Processing System

Count most frequent search terms:
cat access.log |
awk '{print $7}' |
sort |
uniq -c |
sort -rn |
head -5
Each command reads from stdin, writes to stdout. They compose perfectly. Replace any step with a different tool — the interface (text streams) stays the same.

Unix Pipe Properties (that make it work)

  • Input is immutable: cat access.log doesn't modify the log file
  • Each process is stateless: output depends only on input
  • Failures are transparent: if a step fails, the pipeline stops — nothing is partially applied
  • Debugging: you can inspect intermediate results at any stage
The limitation of Unix tools: they run on a single machine. The entire dataset must fit in one machine's disk. To scale to petabytes, we need the distributed equivalent.

MapReduce

MapReduce (Google, 2004) is the distributed equivalent of Unix pipes. The key insight: bring computation to the data, not data to the computation. Store data across many machines; send the computation to each machine; collect results.

The MapReduce Model

Input data split across HDFS (distributed filesystem)
↓
Map phase
For each input record, the mapper emits zero or more (key, value) pairs.
Runs in parallel on every machine that holds input data.
Example: input=web log line → output=(URL, 1)
↓
Shuffle & Sort
Framework collects all (key, value) pairs from all mappers.
Groups by key. Sorts within each key.
Delivers to the reducer responsible for that key.
↓
Reduce phase
For each unique key, the reducer gets all values for that key.
Produces output (written to HDFS).
Example: key=URL, values=[1,1,1,...] → sum → (URL, count)
↓
Output written to HDFS

Distributed Filesystems (HDFS)

MapReduce requires a distributed filesystem to store both input data and intermediate output. HDFS (Hadoop Distributed File System) is the standard choice, modeled after Google's GFS.

HDFS Architecture

NameNode
Single master. Stores file system metadata: which blocks belong to which file, which DataNodes hold each block. Does NOT store data.
DataNodes
Many workers. Each stores blocks of data (128MB default). Reports block locations to NameNode. Replicates blocks to 3 nodes for fault tolerance.
Read flow:
Client asks NameNode: "where is file /logs/2024-01-01.log?"
NameNode returns list of DataNodes for each block
Client reads directly from DataNodes (NameNode not in data path)

Write flow:
Client → NameNode: create file
Client → DataNode 1 → DataNode 2 → DataNode 3 (pipeline replication)
All 3 acknowledge → write confirmed

HDFS Design Properties

  • Commodity hardware: designed to run on cheap servers where failures are normal
  • Replication (3×): each block stored on 3 different nodes across different racks
  • Write-once, append-only: files are created, written, and then read. No random writes. Simplifies consistency.
  • Large blocks: 128MB per block reduces NameNode metadata overhead, optimizes for sequential reads
MapReduce exploits data locality: the framework tries to schedule map tasks on the same DataNode that holds the input block. This avoids network transfer of input data — the computation moves to the data. Only the smaller intermediate (key, value) pairs are shuffled across the network.

MapReduce Fault Tolerance

Why MapReduce is fault tolerant:
1. All input is stored durably in HDFS (3 replicas) before job starts
2. Map task fails → framework re-runs it on a different node (input still in HDFS)
3. Reduce task fails → framework re-runs it (map output on disk of mapper node, or re-run map task)
4. All output is written to HDFS before the next stage begins
The cost: materializing intermediate data to disk at every stage is slow. Modern engines (Spark) avoid this by keeping data in memory across stages.
