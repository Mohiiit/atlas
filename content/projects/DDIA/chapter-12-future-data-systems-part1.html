Chapter 12 — Part 1 of 2

The Future of Data Systems: Data Integration & Unbundling

How to combine specialized tools into a coherent whole, why dual writes are dangerous, the limits of total ordering, lambda architecture and its successors, and the vision of unbundled databases composing a "meta-database of everything."

1. The Data Integration Problem

In complex applications, data is used in many different ways. No single piece of software handles every access pattern well. You inevitably end up cobbling together multiple specialized tools: an OLTP database, a full-text search index, an analytics warehouse, a cache, a recommendation engine, and more. The challenge is keeping all of these representations consistent with each other.

Why Integration Is Hard

  • Each tool excels at one access pattern but is poor for others
  • As the number of derived representations grows, synchronization complexity explodes
  • You must be clear about inputs and outputs: where is data written first? Which representations are derived from which sources?
  • The need for data integration only becomes apparent when you zoom out and look at dataflows across an entire organization

Reasoning About Dataflows

When the same data lives in several storage systems, you need to be very clear about the dataflow direction. Data should first be written to a system of record, and all other representations should be derived from it. If change data capture (CDC) is the only way of updating a derived system, you can be confident the derived view is consistent with the source (barring bugs). Writing to the database is the only way of supplying new input into the system.

2. The Dual Writes Problem

The simplest (and most dangerous) approach is to have the application write directly to both the database and the search index. This introduces a dual writes problem: when two clients concurrently send conflicting writes, the two storage systems may process them in different orders, becoming permanently inconsistent.

Neither system is "in charge" of ordering — both make contradictory decisions.

Why Dual Writes Fail

  • No single authority decides the order of writes
  • Network delays cause different arrival orders at different systems
  • One write may succeed while the other fails, leaving partial state
  • Even with retry logic, you cannot guarantee consistent ordering without a central sequencer

3. CDC-Based Integration: The Safe Approach

The solution is to funnel all user input through a single system that decides on an ordering for all writes. Then you derive other representations by processing writes in that same order. This is an application of state machine replication via total order broadcast.

Properties of Log-Based Derivation

  • Deterministic: same input log always produces the same derived state
  • Idempotent: replaying events is safe; you can recover from faults by reprocessing
  • Decoupled: producers and consumers don't need to know about each other
  • Auditable: the event log is an immutable record of every change

4. Derived Data vs. Distributed Transactions

Both derived data systems and distributed transactions aim to keep multiple data stores consistent. They achieve a similar goal by different means:

Diagram 4: Derived Data vs. Distributed Transactions
Aspect Log-Based Derived Data Distributed Transactions (2PC)
Ordering mechanism Append-only event log (total order) Locks for mutual exclusion (2PL)
Atomicity Deterministic retry + idempotence Atomic commit across participants
Consistency model Eventual consistency (async updates) Linearizability (read your own writes)
Fault tolerance Local fault containment; log buffers Any participant failure aborts all; cascading
Performance High throughput; async processing High latency due to lock contention
Heterogeneous? Works across different technologies Requires all participants to support XA
Adoption Widely practical (Kafka, Debezium, etc.) Limited adoption; poor XA support
"In the absence of widespread support for a good distributed transaction protocol, log-based derived data is the most promising approach for integrating different data systems." — Kleppmann

The Middle Ground

Kleppmann argues we should not simply accept "eventual consistency — suck it up." Instead, we can build stronger guarantees on top of asynchronously derived systems (explored in Part 2 of this chapter). The goal is a middle ground between the strictness of distributed transactions and the lax guarantees of pure eventual consistency.

5. The Limits of Total Ordering

Constructing a totally ordered event log works well for small systems (single-leader replication produces exactly such a log). But as scale and geographic distribution grow, total ordering becomes infeasible:

When Total Ordering Breaks Down

  • Throughput exceeds single node: Partitioned logs have ambiguous ordering across partitions
  • Multi-datacenter: Each datacenter has its own leader; cross-DC coordination is too slow
  • Microservices: Each service owns independent state; no defined order between services
  • Offline clients: Client-side state updated optimistically; clients and servers see events in different orders

Total order broadcast is equivalent to consensus, and most consensus algorithms require all events to pass through a single leader. Designing consensus that scales beyond a single node's throughput and works in geo-distributed settings remains an open research problem.

6. Causal Ordering: The Unfriend/Message Problem

When there is no causal link between events, lack of total order is harmless. But causal dependencies can arise in subtle ways across different services, leading to dangerous reorderings.

Diagram 5: The Unfriend/Message Causal Race Condition
Alice
Step 1: Unfriends Bob → writes to Friendship Service
Alice
Step 2: Posts rude message about Bob → writes to Message Service

Notification Service processes events out of causal order:

message-send event
→ processed BEFORE →
unfriend event

Bob receives notification of the rude message!

The causal dependency (unfriend happened-before message) is lost because the two events live in different services with independent ordering.

Approaches to Capturing Causality

  • Logical timestamps: Provide total ordering without coordination, but recipients must handle out-of-order delivery and carry extra metadata
  • Causal event references: Log the system state the user saw before making a decision; later events reference that snapshot's identifier to record the dependency
  • Conflict resolution: Algorithms like CRDTs help maintain state with out-of-order events, but do not help with external side effects (e.g., sending a notification)

7. Maintaining Derived State: The Functional Pipeline

Batch processing has a strong functional flavor: deterministic, pure functions whose output depends only on their input, treating inputs as immutable and outputs as append-only. Stream processing extends this model to allow managed, fault-tolerant state. Together, they form a functional data pipeline.

Same input → same output. Replay-safe. Fault recovery = just re-run the function.

Why Asynchrony Helps

Derived data could in theory be maintained synchronously (like a DB updating secondary indexes within the same transaction). But asynchrony is what makes event-log systems robust: a fault in one consumer is contained locally. The event log buffers messages, and the faulty consumer can catch up when fixed. By contrast, distributed transactions abort everything if any participant fails, amplifying local faults into system-wide failures.

8. Reprocessing for Application Evolution

Stream processing reflects changes with low delay. Batch processing allows reprocessing large amounts of historical data to derive entirely new views. Together, they enable gradual schema migration without downtime.

Diagram 7: Gradual Migration via Derived View Reprocessing
Immutable Event Log (Source of Truth)
↓
Old Schema View (v1)
95% of users
↓
New Schema View (v2)
5% of users (canary)
Phase 1: Both views live side-by-side → Phase 2: Gradually shift traffic → Phase 3: Drop old view

Every stage is reversible. If v2 has bugs, route traffic back to v1 instantly.

"The beauty of gradual migration is that every stage of the process is easily reversible if something goes wrong: you always have a working system to go back to. By reducing the risk of irreversible damage, you can be more confident about going ahead, and thus move faster." — Kleppmann

The Railway Analogy

In 19th-century England, different railway gauges were incompatible. After a standard gauge was chosen (1846), tracks were converted by adding a third rail (mixed gauge), allowing trains of both gauges to run simultaneously. The old gauge was removed only after all trains had been converted. This is exactly how derived view reprocessing works: maintain old and new side by side, migrate gradually, and roll back safely if needed.

9. Lambda Architecture

The lambda architecture attempts to combine batch and stream processing by running two parallel systems: a batch layer for correctness and a speed layer for low latency.

Problems with Lambda Architecture

  • Dual codebase: You must maintain the same logic in two different frameworks (batch + stream) — significant engineering overhead
  • Merging is hard: Combining batch and realtime outputs is easy for simple aggregations but very difficult for joins, sessionization, and non-time-series data
  • Reprocessing cost: Running full batch jobs over all historical data is expensive, forcing incremental batching which adds complexity and defeats the purpose of the "simple batch layer"
  • Operational burden: Debugging, tuning, and monitoring two separate pipelines that must produce compatible results

10. Unifying Batch and Stream Processing

Modern systems eliminate the lambda architecture's downsides by implementing both batch and stream computation in a single unified engine. This requires three key capabilities:

Replay Historical Events

The same engine processes both the stream of recent events and replayed historical data (e.g., from log-based brokers like Kafka or from HDFS).

Exactly-Once Semantics

Output is the same as if no faults occurred, even when faults do occur. Failed tasks' partial output is discarded, just like in batch processing.

Event-Time Windowing

Windowing by event time (not processing time), because processing time is meaningless when replaying historical events. Apache Beam / Flink provide this.

Spark vs. Flink: Two Philosophies

  • Spark: Stream processing on top of a batch engine via microbatches (may struggle with hopping/sliding windows)
  • Flink: Batch processing on top of a stream engine (treats a bounded dataset as a finite stream)
  • In principle, one can emulate the other, but performance characteristics differ

11. Unbundling Databases

Databases, Hadoop, and operating systems all ultimately do the same thing: store data and let you process/query it. Kleppmann draws a deep analogy between the internal features of a database and the external data integration architecture we've been building:

Composing Data Storage Technologies

Diagram 9: Internal DB Features as External Composed Systems

Inside a Traditional Database

  • Secondary indexes (B-tree, hash)
  • Materialized views (precomputed caches)
  • Replication logs (follower sync)
  • Full-text search indexes
  • Triggers and stored procedures
  • Transaction manager (2PL, MVCC)

The Unbundled Equivalent

  • Elasticsearch / Solr (search index)
  • Redis / Memcached (cache layer)
  • Kafka CDC (replication across systems)
  • Dedicated NLP + inverted index
  • Stream processors (Flink, Spark)
  • Event log ordering + idempotent writes

CREATE INDEX = Replay the Replication Log

Think about what happens when you run CREATE INDEX: the database scans a consistent snapshot, builds the index, then processes the backlog of writes made since the snapshot. This is remarkably similar to setting up a new follower replica or bootstrapping CDC. Running CREATE INDEX is essentially reprocessing the existing dataset to derive a new view.

The Insight

Creating an index, setting up a new follower, and bootstrapping a CDC consumer are all the same operation at different levels of abstraction: take a snapshot, build the derived structure, then consume the ongoing change stream to keep it up to date.

12. The Meta-Database of Everything

13. Federation vs. Unbundling

There are two avenues for composing diverse storage and processing tools into a cohesive system:

Diagram 11: Federation vs. Unbundling Comparison

Federated Databases (Unifying Reads)

  • Unified query interface to diverse storage engines
  • Also known as a polystore
  • Example: PostgreSQL foreign data wrappers
  • Follows the relational tradition: high-level query language, elegant semantics, complex implementation
  • Does NOT solve write synchronization

Unbundled Databases (Unifying Writes)

  • Reliable sync of writes across systems via CDC + event logs
  • Like unbundling a DB's index-maintenance features
  • Example: Kafka Connect + Debezium
  • Follows the Unix tradition: small tools that do one thing well, uniform low-level API (pipes), composable via higher-level language
  • The harder engineering problem

What's Missing: The "Unix Shell" for Data Systems

We don't yet have the unbundled-database equivalent of the Unix shell — a high-level language for composing storage and processing systems declaratively. Kleppmann imagines being able to write mysql | elasticsearch by analogy to Unix pipes, which would be the unbundled equivalent of CREATE INDEX: automatically capture all changes from MySQL and apply them to Elasticsearch, without custom application code.

14. Write Path vs. Read Path

The dataflow through a system of derived data can be understood through two complementary concepts:

Diagram 12: Write Path vs. Read Path

Write Path (Eager)

User Write Event
↓
System of Record
↓ CDC stream
Derive & Store Index
↓
Derive & Store Cache

Pre-computes derived data at write time. Work happens in advance of any query.

Read Path (Lazy)

User Query
↓
Query Parser / Planner
↓
Scan Storage & Compute
↓
Return Result

Computes result on demand. Work happens when the query arrives.

Caches and indexes shift work from the read path to the write path. A materialized view is the extreme of the write path (fully pre-computed). A full table scan is the extreme of the read path (nothing pre-computed).

The Tradeoff

  • Write path: Pre-computes derived data eagerly. Reads are fast but writes are more expensive. Good when reads vastly outnumber writes.
  • Read path: Computes results lazily at query time. Writes are cheap but reads pay the computation cost. Good for ad-hoc queries and exploration.
  • Most real systems use a mix: indexes and caches (write path) combined with query processing (read path).
  • A derived data system decides what to pre-compute. A full-text search index is a write-path optimization. An analytics query over raw data is a read-path computation.

Key Takeaways

Data Integration

  • No single tool handles all access patterns; you must combine specialized systems
  • Dual writes lead to permanent inconsistency; use CDC from a single source of truth
  • The event log provides deterministic, idempotent, fault-tolerant derivation

Ordering & Causality

  • Total ordering works for small systems but breaks down at scale, across DCs, and across services
  • Causal dependencies across services can cause subtle bugs (unfriend/message problem)
  • Logical timestamps and causal references are partial solutions

Batch + Stream Unification

  • Lambda architecture is influential but operationally painful
  • Modern engines (Flink, Beam) unify both in a single framework
  • Reprocessing enables gradual, reversible schema evolution

Unbundling Databases

  • An organization's dataflow is like one meta-database; each system is an "index type"
  • Federation unifies reads; unbundling unifies writes
  • CREATE INDEX is the same operation as bootstrapping a CDC consumer

Test Your Understanding

1Why is the dual-writes approach to keeping a database and search index in sync fundamentally broken, and what specifically causes permanent inconsistency? ▾

Dual writes means the application writes directly to both the database and the search index. The fundamental problem is that no single system is "in charge" of determining write order. When two clients send conflicting writes concurrently, the database might process them in order A-then-B while the search index processes them B-then-A (due to different network delays). This results in the database storing one value and the search index storing a different value — a permanent inconsistency that will not resolve itself. Additionally, if one write succeeds and the other fails (partial failure), you're left in an inconsistent state with no automatic recovery mechanism.

2Explain the analogy between running CREATE INDEX in a database and setting up a CDC consumer. What three phases do they share? ▾

Both operations share three identical phases: (1) Take a consistent snapshot of the current data; (2) Build the derived structure (index or external system) from that snapshot; (3) Process the backlog of writes that occurred since the snapshot was taken, then continue consuming the ongoing change stream to keep the derived structure up to date. This is also exactly how setting up a new follower replica works. The insight is that these are all the same fundamental operation at different levels of abstraction: derive a new view from existing data, then keep it synchronized via the change stream.

3What are the four scenarios where constructing a totally ordered event log becomes infeasible, and what is the underlying theoretical reason? ▾

The four scenarios are: (1) Throughput exceeds a single node — partitioned logs have ambiguous cross-partition ordering; (2) Multi-datacenter deployment — each DC needs its own leader, and cross-DC synchronous coordination is too slow; (3) Microservices with independent state — events in different services have no defined mutual order; (4) Offline-capable clients — client-side state is updated optimistically, so clients and servers see events in different orders. The underlying theoretical reason is that total order broadcast is equivalent to consensus, and most consensus algorithms require all events to pass through a single leader node. Scaling consensus beyond a single node's throughput in a geo-distributed setting remains an open research problem.

4What are the three main practical problems with the lambda architecture, and how does unifying batch and stream processing solve them? ▾

The three problems: (1) Dual codebase maintenance — the same logic must be implemented and debugged in two separate frameworks (e.g., MapReduce + Storm); (2) Merging outputs is hard — combining batch and realtime views is simple for tumbling window aggregations but extremely difficult for joins, sessionization, or non-time-series data; (3) Expensive full reprocessing — running batch over all historical data is costly, forcing incremental batching which adds the very complexity the batch layer was supposed to avoid. Unified systems like Flink solve these by allowing the same code to process both historical replays and live streams, with exactly-once semantics and event-time windowing, eliminating the need for separate pipelines and output merging.

5Explain the difference between federation and unbundling. Which problem does each solve, and which follows the "relational tradition" versus the "Unix tradition"? ▾

Federation (polystore) unifies reads by providing a single query interface across diverse storage engines (e.g., PostgreSQL foreign data wrappers). It follows the relational tradition: a high-level query language with elegant semantics and a complex implementation. However, it does not solve write synchronization. Unbundling unifies writes by reliably syncing data changes across systems via CDC and event logs. It follows the Unix tradition: small tools that do one thing well, communicating through a uniform low-level API (like pipes) and composable via higher-level abstractions. Unbundling is the harder engineering problem because keeping writes in sync across heterogeneous systems requires dealing with ordering, idempotence, and fault tolerance — which is essentially the distributed index-maintenance problem that databases solve internally.

6In the unfriend/message example, why do standard conflict resolution algorithms (like CRDTs) fail to prevent the notification from reaching the ex-partner? ▾

The unfriend event and the message-send event live in different services (friendship service vs. message service), so there is no shared ordering between them. When the notification service joins the message stream with the friend list, it may process the message-send event before the unfriend event, sending the notification to the ex-partner. Conflict resolution algorithms like CRDTs help with maintaining state by converging on a consistent value regardless of event order — but they do not help with external side effects. A notification, once sent, cannot be "un-sent." The damage is done before any conflict resolution has a chance to act. This is why the problem requires capturing causal dependencies (the message was sent after the unfriend), not just resolving conflicting state updates. Approaches like logging the state the user saw before acting, or using logical timestamps with causal metadata, are needed but add complexity.
