Chapter 12 — The Future of Data Systems

Part 1: Data Integration, Derived Views, Lambda Architecture

Chapter 12 synthesizes the entire book. Modern data systems are built from many specialized tools, each optimized for a specific task. The challenge: making them work together reliably. Kleppmann's vision: an architecture built around immutable events and derived views.

The Challenge: Data Integration

A typical modern application uses many data systems simultaneously: a relational database for transactions, Elasticsearch for search, Redis for caching, a data warehouse for analytics, Kafka for event streams. Keeping them in sync is the central data engineering challenge.

PostgreSQL
Transactions, constraints
Elasticsearch
Full-text search
Redis
Low-latency cache
Redshift
Analytics warehouse
Kafka
Event streams
ML model
Recommendations
Problem: all need consistent data. Any update to PostgreSQL must propagate to Elasticsearch, Redis, Redshift. If one propagation fails, systems diverge.

Why No Single System Wins

No single database can be optimal at everything simultaneously:

  • OLTP needs low-latency writes and transactions
  • Full-text search needs inverted indexes
  • Analytics needs columnar storage and vectorized execution
  • Caching needs in-memory key-value access
  • Graph queries need adjacency-list traversal

Each storage/query pattern has a different ideal data structure. The polyglot persistence approach: use the right tool for each job, integrate them.

Keeping Systems in Sync

Two approaches to keeping multiple data stores in sync:

1. Dual Writes (Anti-Pattern)

Application-level dual writes (bad)
App: write to database → write to Elasticsearch → write to cache
Problem 1: crash between writes → systems out of sync
Problem 2: two concurrent clients → race condition → different systems in different orders
Problem 3: partial failure → no automatic retry or rollback

2. Change Data Capture (Correct Pattern)

CDC pipeline (good)
App: write to database only (single source of truth)
CDC reads WAL → publishes change events to Kafka
Consumers: Elasticsearch indexer, cache invalidator, warehouse loader
All consumers see the same ordered sequence of changes. No race conditions.

Derived Data and Views

The key architectural insight of Chapter 12: most data stores in your system are derived data — computed from some authoritative source. Recognizing this simplifies the architecture.

System of Record
The authoritative source. Written when events occur. Normalized, consistent. Example: the primary database.
Derived: Elasticsearch index
Computed from the record. Denormalized for fast search. Can be rebuilt from source.
Derived: Redis cache
Materialized view of frequently-read data. Can expire and be recomputed.
Derived: data warehouse
Denormalized fact tables for analytics. Rebuilt via ETL from source.
If a derived system fails, it can always be rebuilt by replaying the source data. Derived systems are expendable; the system of record is precious.

Lambda Architecture

Lambda Architecture (Nathan Marz, 2012) is an early attempt to combine batch and stream processing:

Incoming data
↓
Batch layer
Process all historical data. Accurate. Slow (hours). Produces "batch views".
Speed layer
Process recent data. Fast but approximate. Produces "real-time views".
↓
Serving layer
Merge batch view + real-time view → serve queries

Lambda Architecture: Critique

Lambda architecture forces you to maintain the same logic in two different systems (batch + streaming). They often diverge over time (different behavior, different bugs). Debugging is complex. The serving layer merge is complex. Modern stream processors (Flink, Kafka Streams) can handle both batch and stream in one system — making Lambda obsolete.

Kappa Architecture

Jay Kreps (Kafka co-creator) proposed the Kappa Architecture: use only a stream processor, no separate batch layer. Historical reprocessing = replay the event log from the beginning with a new processor version.

Kappa: one system for all
All events stored in Kafka (long retention or compacted log)
Stream processor handles both "real-time" (current events) and "batch" (replay from beginning)
New version deployment: start new consumer at offset 0 → build new state → switch traffic → delete old
One codebase. One language. One operational concern.
Kappa works well when all data fits in the event log. For truly massive historical datasets (petabytes of S3 data), batch jobs (Spark) are still better for the "from scratch" rebuild. In practice, many systems use a hybrid: Kafka for recent data + Spark/Presto for historical data.
