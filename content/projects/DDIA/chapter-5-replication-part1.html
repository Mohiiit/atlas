Chapter 5: Replication — Leaders, Followers, and Replication Logs

Replication means keeping a copy of the same data on multiple machines connected via a network. This first part covers the foundational architecture of leader-based replication: how leaders and followers coordinate, how data flows between them, what happens when nodes fail, and the different ways replication logs are implemented under the hood.

Why Replicate Data?

If the data that you are replicating does not change over time, replication is trivial: copy the data to every node once and you are done. All of the difficulty in replication lies in handling changes to replicated data. But first, why bother at all?

Diagram 1 — Three Reasons to Replicate
Reduce Latency
Keep data geographically close to users. A replica in Tokyo serves Japanese users faster than a single server in Virginia.
Increase Availability
If some parts of the system fail, the system continues working. Redundancy means no single point of failure.
Scale Read Throughput
Scale out the number of machines that can serve read queries. Distribute read load across many followers.

Key Assumption for This Chapter

The dataset is small enough that each machine can hold a copy of the entire dataset. Chapter 6 relaxes this assumption by introducing partitioning (sharding) for datasets too big for a single machine.

There are three popular algorithms for replicating changes between nodes: single-leader, multi-leader, and leaderless replication. This part focuses on single-leader replication, which is the most common approach used in production databases.

Leaders and Followers

Each node that stores a copy of the database is called a replica. With multiple replicas, the fundamental question is: how do we ensure that all the data ends up on all the replicas? Every write must be processed by every replica, or they will diverge.

The most common solution is leader-based replication (also known as active/passive or master-slave replication). It works in three steps:

How Leader-Based Replication Works

  • Step 1: One replica is designated the leader (master/primary). All client writes go to the leader, which writes new data to its local storage.
  • Step 2: Other replicas are followers (read replicas/slaves/secondaries). The leader sends data changes to all followers via a replication log or change stream. Followers apply writes in the same order as the leader.
  • Step 3: Clients can read from the leader or any follower. However, writes are only accepted on the leader.

Synchronous vs. Asynchronous Replication

An important detail of a replicated system is whether the replication happens synchronously or asynchronously. This is often a configurable option in relational databases.

Synchronous: Leader waits for follower to confirm before acknowledging to client. Guaranteed up-to-date copy, but blocks on follower failure.
Asynchronous: Leader sends the message but does not wait. Fast, but follower may fall behind by seconds or even minutes.

Synchronous Advantages

  • Follower guaranteed to have up-to-date copy consistent with leader
  • If leader suddenly fails, data is still available on the synchronous follower
  • Strong durability guarantees

Synchronous Disadvantages

  • If sync follower does not respond (crash, network fault), the write cannot be processed
  • Leader must block all writes and wait
  • One node outage can halt the entire system

The Semi-Synchronous Compromise

It is impractical for all followers to be synchronous: any one node outage would cause the whole system to grind to a halt. In practice, semi-synchronous replication means:

Guarantee: Up-to-date data on at least two nodes (the leader + one synchronous follower) at all times. If the sync follower becomes slow or unavailable, an async follower is promoted to synchronous.

Fully Asynchronous: The Trade-off

Often, leader-based replication is configured to be completely asynchronous. If the leader fails and is not recoverable, any writes that have not yet been replicated to followers are lost. The write is not guaranteed to be durable, even if it has been confirmed to the client.

However, the advantage is that the leader can continue processing writes even if all of its followers have fallen behind. This is why asynchronous replication is nevertheless widely used, especially with many followers or geographically distributed replicas.

Setting Up New Followers

From time to time you need to set up new followers — to increase replicas or replace failed nodes. Simply copying data files is not sufficient because clients are constantly writing; a standard file copy would see different parts of the database at different points in time. Locking the database would violate availability. Fortunately, it can be done without downtime:

Diagram 5 — Setting Up a New Follower (4 Steps)
1 Take a Consistent Snapshot
Take a consistent snapshot of the leader's database at some point in time, without taking a lock on the entire database. Most databases have this feature (it is also required for backups). For MySQL, tools like innobackupex can be used.
2 Copy Snapshot to New Follower
Copy the snapshot to the new follower node. This can be done over the network or via physical media for very large datasets.
3 Request Changes Since Snapshot
The follower connects to the leader and requests all data changes that happened since the snapshot. This requires the snapshot to be associated with an exact position in the leader's replication log. PostgreSQL calls this the log sequence number; MySQL calls it the binlog coordinates.
4 Caught Up!
When the follower has processed the backlog of data changes since the snapshot, we say it has caught up. It can now continue to process data changes from the leader as they happen in real time.

Handling Node Outages

Any node can go down — unexpectedly due to a fault, or due to planned maintenance like rebooting for a kernel security patch. The goal is to keep the system as a whole running despite individual node failures and keep the impact of a node outage as small as possible.

Follower Failure: Catch-Up Recovery

Each follower keeps a log of the data changes it has received from the leader on its local disk. Recovery is straightforward:

Leader Failure: Failover

Handling a failure of the leader is trickier: one of the followers needs to be promoted to be the new leader, clients need to be reconfigured to send their writes to the new leader, and other followers need to start consuming data changes from the new leader. This process is called failover.

Things That Can Go Wrong During Failover

Failover is fraught with dangers. These are not edge cases — they are real problems that have caused production incidents:

Diagram 8 — Failover Dangers

Danger 1: Lost Writes (Discarded Data)

With asynchronous replication, the new leader may not have received all writes from the old leader before it failed. If the old leader rejoins, what happens to those unreplicated writes? The most common solution: the old leader's unreplicated writes are simply discarded. This violates clients' durability expectations.

Old Leader: tx100-tx105
→
New Leader: tx100-tx103 only
tx104, tx105 LOST!

Danger 2: Coordinated Systems Diverge (The GitHub Incident)

Discarding writes is especially dangerous when other storage systems must stay coordinated with the database. At GitHub, an out-of-date MySQL follower was promoted to leader. The database used an autoincrementing counter for primary keys, but the new leader's counter lagged behind, so it reused primary keys previously assigned by the old leader. These keys were also used in Redis, causing private data to be disclosed to the wrong users.

New MySQL Leader
reuses PK #1234 →
Redis still maps PK #1234 to old user
Data leak!

Danger 3: Split Brain

In certain fault scenarios, two nodes both believe they are the leader. This is called split brain. If both leaders accept writes with no conflict resolution process, data is likely to be lost or corrupted. Some systems have a mechanism to shut down one node if two leaders are detected — but if not carefully designed, you can end up with both nodes being shut down.

Node A
"I am leader!"
⚔
Node B
"I am leader!"
Both accept writes → conflict!

Danger 4: Wrong Timeout

Too long: Longer time to recovery when the leader actually fails. Too short: Unnecessary failovers caused by temporary load spikes or network glitches. An unnecessary failover during high load makes the situation worse, not better.

Some operations teams prefer to perform failovers manually, even if the software supports automatic failover, because of these dangers.

Implementation of Replication Logs

How does leader-based replication work under the hood? Several different methods are used in practice. Each has different trade-offs around coupling, compatibility, flexibility, and performance.

1. Statement-Based Replication

The leader logs every write request (statement) it executes and sends that statement log to followers. For a relational database, this means every INSERT, UPDATE, or DELETE statement is forwarded to followers, which parse and execute the SQL as if received from a client.

Diagram 9 — Statement-Based Replication
Leader
→
INSERT INTO users VALUES(...);
UPDATE users SET name='...';
DELETE FROM sessions WHERE...;
→
Follower
The follower parses and executes each SQL statement independently

Problems with Statement-Based Replication

  • Nondeterministic functions: NOW(), RAND() produce different values on each replica
  • Autoincrementing columns: Statements depending on existing data (e.g., UPDATE ... WHERE <condition>) must execute in exactly the same order on each replica
  • Side effects: Triggers, stored procedures, user-defined functions may produce different results on each replica unless absolutely deterministic

The leader can replace nondeterministic function calls with fixed values when logging, but the edge cases are numerous. MySQL used this before v5.1; VoltDB still uses it but requires transactions to be deterministic.

2. Write-Ahead Log (WAL) Shipping

Every write is appended to a log: in LSM-tree storage engines, this log is the main storage; in B-tree engines, modifications are written to a write-ahead log before the actual data structure. This exact same log can be used to build a replica: the leader sends the log across the network to followers.

Diagram 10 — Write-Ahead Log (WAL) Shipping
Leader
Storage Engine
→
WAL bytes
Block 42, offset 0x1A0: write 64 bytes...
Block 97, offset 0x040: write 128 bytes...
→
Follower
Builds identical data structures
Key drawback: The log describes data at a very low level (which bytes changed in which disk blocks). This makes replication tightly coupled to the storage engine. You typically cannot run different versions of the database software on leader and followers, meaning zero-downtime upgrades become impossible.

Operational Impact

If the replication protocol allows version mismatch (follower on newer version), you can perform a zero-downtime upgrade: upgrade followers first, then failover to make an upgraded node the new leader. WAL shipping often does not allow this. Used by PostgreSQL and Oracle.

3. Logical (Row-Based) Log Replication

An alternative is to use a different log format for replication than for the storage engine, decoupling the replication log from storage internals. This is called a logical log. A logical log for a relational database is a sequence of records describing writes at the granularity of a row:

Diagram 11 — Logical (Row-Based) Log Replication
INSERT
Log contains: new values of all columns
users: {id:42, name:"Alice", email:"a@b.com"}
DELETE
Log contains: enough info to uniquely identify the row (typically the primary key)
users: {id:42} (or all old column values if no PK)
UPDATE
Log contains: row identifier + new values of changed columns
users: {id:42, name:"Bob"}
A transaction modifying several rows generates several log records, followed by a commit record.

Advantages

  • Decoupled from storage engine: leader and follower can run different versions or even different storage engines
  • Easier to keep backward compatible
  • Easier for external applications to parse (enables change data capture — Chapter 11)
  • Can send data to data warehouses, custom indexes, caches

Who Uses This?

MySQL's binlog (when configured for row-based replication) uses this approach. It is the default in modern MySQL (5.1+). The logical log format is the foundation of change data capture (CDC) systems.

4. Trigger-Based Replication

The approaches above are all implemented by the database system itself. But sometimes you need more flexibility: replicate only a subset of data, replicate between different kinds of databases, or apply conflict resolution logic. In these cases, replication moves up to the application layer.

Trade-off

Trigger-based replication has greater overhead than database-native replication methods and is more prone to bugs and limitations. However, it can be useful due to its flexibility — it is the only approach that gives application code control over what is replicated and how.

First-Principles Questions

Test your understanding of the core concepts. Click each question to reveal the answer.

1Why is fully synchronous replication impractical, and what compromise do production systems actually use? ▾

If all followers are synchronous, any single node outage (crash, network fault, slow disk) blocks all writes to the entire system — the leader must wait for every follower to acknowledge before confirming to the client. With N followers, the probability of at least one being unavailable at any moment is high.

The practical compromise is semi-synchronous replication: one follower is synchronous (guaranteeing an up-to-date copy on at least two nodes), and the rest are asynchronous. If the synchronous follower becomes unavailable, an asynchronous follower is automatically promoted to synchronous. This provides a balance between durability (data on 2+ nodes) and availability (one slow follower does not block the system).

2During leader failover, why can't the new leader simply replay the old leader's unreplicated writes when it comes back online? ▾

The new leader may have already accepted conflicting writes in the meantime — writes that were not on the old leader. If the old leader had autoincrementing IDs, the new leader may have reused those same IDs for different data. Simply replaying the old leader's writes could cause primary key collisions, constraint violations, or data corruption.

Furthermore, external systems (Redis caches, search indexes, analytics stores) may have already been updated based on the new leader's data. Replaying old writes would create inconsistencies between the database and these external systems — as demonstrated by the GitHub incident where primary key reuse caused private data disclosure to wrong users.

The most common (if unfortunate) solution is to discard the old leader's unreplicated writes, accepting a violation of durability guarantees.

3Why is WAL shipping considered tightly coupled to the storage engine, while logical log replication is not? ▾

A WAL describes changes at the physical byte level: "write these 64 bytes at offset 0x1A0 in disk block 42." This format is intimately tied to the storage engine's internal data structures — the layout of B-tree pages, the format of LSM-tree segments, etc. If the storage engine changes its format between versions (reorganizing pages, changing compression, etc.), the WAL format changes too, making it impossible for a follower running a different version to interpret the log.

A logical log, by contrast, describes changes at the row level: "in table users, insert row with id=42, name='Alice'." This is a higher-level abstraction that does not depend on how the data is physically stored on disk. The leader could use a B-tree and the follower could use an LSM-tree — the logical log would work for both. This decoupling enables zero-downtime upgrades (upgrade followers first, then failover) and enables change data capture by external systems.

4When setting up a new follower, why must the snapshot be associated with an exact position in the replication log? ▾

The snapshot represents the state of the database at one specific point in time. After the snapshot is taken, the leader continues accepting writes. The new follower needs to "catch up" by requesting all changes that happened after the snapshot. Without an exact log position associated with the snapshot, the follower has no way to know where to start in the replication log.

If it starts too early, it might try to apply writes that are already reflected in the snapshot, causing duplicates or errors. If it starts too late, it misses writes, causing the follower to have gaps in its data. The log position (called log sequence number in PostgreSQL or binlog coordinates in MySQL) is the precise pointer that connects the snapshot to the ongoing stream of changes, ensuring a gapless, duplicate-free transition from snapshot to live replication.
