Chapter 5 — Replication

Part 1: Why Replicate, Single-Leader Replication, Synchronous vs Async, Failover

Replication means keeping copies of the same data on multiple machines. Every write to the data needs to be processed by every replica — how you handle that coordination is where distributed systems get complex.

Why Replicate Data?

Latency
Keep data geographically close to users. A Tokyo user shouldn't wait for transatlantic round-trips.
Availability
System keeps working even if some nodes fail. Other replicas serve requests while one is down.
Read throughput
Scale out reads across many replicas. 10 replicas ≈ 10× read capacity (writes still bottleneck on leader).

Three Replication Approaches

Single-Leader
One node accepts all writes, replicates to followers. Simple, widely used.
Multi-Leader
Multiple nodes accept writes, replicate to each other. Good for multi-datacenter setups.
Leaderless
No leader — clients write to multiple replicas directly. Dynamo, Cassandra, Riak.

Leaders and Followers (Single-Leader)

One replica is the leader. All writes go through it. The leader writes locally, then sends the change to all followers (aka read replicas) via a replication log. Followers apply changes in the same order. Reads can go to any replica.

Most production systems use semi-sync: one sync follower ensures at least 2 nodes always have the latest data. All others async. PostgreSQL's synchronous_standby_names controls this.

Setting Up New Followers

You can't just copy files — the leader is continuously accepting writes. The standard approach:

1. Snapshot leader (without locking — use MVCC consistent snapshot)
↓
2. Copy snapshot to new follower
↓
3. Follower connects to leader, requests all changes since snapshot's log position (LSN in PostgreSQL, binlog coordinates in MySQL)
↓
4. Process replication backlog until caught up
↓
5. Now in sync — ready to serve reads

Handling Node Outages

Follower failure — catch-up recovery: The follower reads its local log to find the last processed transaction, reconnects to the leader, and requests all changes since that point. Straightforward.

Leader failure — failover: Far more complex. Three steps:

1. Detect failure — leader hasn't responded for timeout (~30s). Hard to distinguish crashed vs. slow vs. network partition.
↓
2. Elect new leader — follower with most up-to-date replication log, or by controller node consensus. Remaining nodes must agree.
↓
3. Reconfigure — redirect client writes to new leader. Old leader must recognize new leader when it returns (to avoid split-brain).

What Can Go Wrong During Failover

Lost writes: If the old leader had uncommitted writes not yet replicated, those are lost when new leader is promoted. The old leader may return and try to commit them — conflicting with the new leader's state.
Split brain: Two nodes both believe they are leader. Both accept writes. Data diverges. This is catastrophic. Some systems use STONITH (Shoot The Other Node In The Head) — forcibly shut down the suspected old leader.
Auto-increment ID reuse: If old leader issued IDs 100–150 before crashing, new leader's IDs may overlap with those used by other systems (e.g., Redis cache, application code).
These problems don't have clean solutions — they're inherent to distributed systems. Many teams prefer manual failover (operator triggers it) over automatic, accepting longer downtime for safety.
