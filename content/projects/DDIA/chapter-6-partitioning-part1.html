Chapter 6 — Partitioning

Part 1: Why Partition, Partitioning + Replication, Key-Range vs Hash Partitioning

Partitioning (sharding) means splitting a large dataset across multiple nodes. Each partition is a subset of the data. The goal: spread both data and query load so each node handles only a fraction of the total.

Terminology: MongoDB/Elasticsearch call it "sharding"; HBase calls regions; Bigtable calls tablets; Cassandra/Riak call vnodes. Same concept.

Why Partition? Scalability

Single node
Limited to one machine's disk, RAM, network. Vertical scaling has a ceiling.
→
Partitioned cluster
Each partition on its own node. 10 nodes = 10× storage, 10× throughput (ideally). Add more nodes to scale.

Partitioning and Replication Combined

Partitioning and replication are orthogonal concepts that are always used together. Each partition has multiple replicas for fault tolerance.

Node 1
Partition A (leader)
Partition B (follower)
Partition C (follower)
Node 2
Partition B (leader)
Partition C (follower)
Partition A (follower)
Node 3
Partition C (leader)
Partition A (follower)
Partition B (follower)
3 partitions × 3 replicas each = 9 total copies. Each node is leader for one partition, follower for two others. Writes go to the leader partition, replicated to followers.

Partitioning of Key-Value Data

The fundamental question: given a key, which partition does it belong to? Two main approaches:

Strategy 1: Partitioning by Key Range

Assign each partition a contiguous range of keys. Like an encyclopedia's A-B, C-D, E-F volumes.

Sensor data example: timestamp as key
Partition 1
Jan 1–Mar 31
Partition 2
Apr 1–Jun 30
Partition 3
Jul 1–Sep 30
Partition 4
Oct 1–Dec 31
✓ Range queries are efficient: "all readings for March" → hits only Partition 1
✗ Hot spots: all writes for today go to the current day's partition. One node overwhelmed while others idle.

Hot spot fix: Prefix key with sensor ID → all sensors' data is spread across partitions. But now "all readings for sensor X" requires scanning all partitions.

Used by: Bigtable, HBase, MongoDB (range-based sharding mode).

Strategy 2: Partitioning by Hash of Key

Apply a hash function to each key and assign hash ranges to partitions. Distributes keys uniformly, destroying key ordering.

Hash partitioning: keys distributed uniformly
hash("user_123")
↓
0x7A3F → 31%
↓
Partition 2
(25%–50%)
hash("user_456")
↓
0xC1B2 → 76%
↓
Partition 4
(75%–100%)
✓ Even distribution of load across partitions. No hot spots for random keys.
✗ Lose ability to do efficient range queries. "Users 100–200" must scan all partitions.
Cassandra uses a compound partition key: first part is hashed (for distribution), second part is sorted within the partition (for range queries within a partition). Best of both worlds for specific access patterns.

Skewed Workloads and Hot Spots

Even with hash partitioning, certain keys can become hot. Example: a celebrity's Twitter account — millions of reads/writes all go to the same key's partition, regardless of hashing.

Hot spot problem
Key "user_bieber" hashes to Partition 3. Every request for that user → Partition 3 overloaded.
Mitigation
Add random suffix to hot keys: store writes as "user_bieber_17", "user_bieber_42" across N partitions. Reads must query all N shards and combine results. Application complexity.
There's no automatic solution. Databases can't detect and mitigate hot spots automatically — application developers must handle it. Instagram/Twitter built custom fan-out logic for celebrity accounts.
