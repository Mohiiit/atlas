Chapter 6 — Part 1 of 2

Partitioning: Strategies and Indexes

Breaking data into pieces for scalability — key range vs. hash partitioning, the hot spot problem, and secondary index strategies (document-partitioned vs. term-partitioned).

Why Partition Data?

In Chapter 5 we explored replication — keeping copies of the same data on multiple nodes for fault tolerance. But for very large datasets or very high query throughput, replication alone is not sufficient. We need to break the data into partitions (also called shards), where each piece of data belongs to exactly one partition.

The Goal of Partitioning

  • Scalability: Distribute data across many disks; distribute query load across many processors
  • Parallelism: Each node independently executes queries for its own partitions
  • Throughput: 10 nodes should ideally handle 10x the data and 10x the throughput of a single node

Terminology Across Systems

The same concept goes by many names:

MongoDB: shard HBase: region Bigtable: tablet Cassandra/Riak: vnode Couchbase: vBucket Elasticsearch: shard

Partitioning and Replication

Partitioning is usually combined with replication so that copies of each partition live on multiple nodes for fault tolerance. A node may be the leader for some partitions and a follower for others. The choice of partitioning scheme is mostly independent of the replication scheme.

Partitioning by Key Range

Assign a continuous range of keys to each partition, like the volumes of a paper encyclopedia. If you know the boundaries, you can determine which partition holds a given key and route requests directly to the right node.

Key Properties

  • Ranges are not necessarily evenly spaced — boundaries adapt to the data distribution
  • Within each partition, keys are kept in sorted order, enabling efficient range scans
  • Used by: Bigtable, HBase, RethinkDB, MongoDB (< v2.4)

The Timestamp Hot Spot Problem

If the key is a timestamp, all writes land on the partition for "today" while other partitions sit idle. The fix: prefix each timestamp with another dimension (e.g., sensor name), so writes spread across partitions. Trade-off: range queries now require one query per sensor prefix.

Partitioning by Hash of Key

A good hash function takes skewed data and makes it uniformly distributed. Instead of assigning key ranges to partitions, we assign hash ranges: every key whose hash falls within a partition's range is stored in that partition. This distributes keys fairly regardless of their natural ordering.

Hash Functions in Practice

  • Cassandra & MongoDB: use MD5
  • Voldemort: uses Fowler–Noll–Vo function
  • Built-in language hash functions (Java's Object.hashCode()) are not suitable — same key can produce different hashes in different processes
  • No need for cryptographic strength — just uniform distribution

The Trade-off: No Range Queries

By hashing the key we lose the ability to do efficient range scans. Keys that were once adjacent are now scattered across all partitions. In MongoDB with hash-based sharding, any range query must be sent to all partitions. Riak, Couchbase, and Voldemort do not support range queries on the primary key at all.

Cassandra's Compound Key Compromise

Cassandra achieves a best-of-both-worlds compromise. A table's primary key consists of several columns. Only the first column is hashed to determine the partition; the remaining columns are used as a concatenated index for sorting data within that partition's SSTables.

Skewed Workloads and Hot Spots

Even with hash partitioning, if all reads and writes target the same key, all requests route to a single partition. The hash of identical IDs is still identical. This is the celebrity problem: a viral post or a celebrity user with millions of followers can create a storm of activity on one key.

Partitioning and Secondary Indexes

Primary key partitioning is straightforward — route by key. But secondary indexes (searching by color, make, author, etc.) complicate things because a secondary index value can appear in any partition. There are two main approaches:

Document-Partitioned (Local Index)

  • Each partition maintains its own secondary index
  • Index covers only documents in that partition
  • Writes are simple: only touch one partition
  • Reads require scatter/gather across all partitions
  • Also called a local index

Term-Partitioned (Global Index)

  • A global index covers data in all partitions
  • The index itself is partitioned by term
  • Reads are efficient: query only the relevant index partition
  • Writes are complex: may affect multiple index partitions
  • Updates are often asynchronous

Document-Partitioned Secondary Index (Local Index)

Imagine a used-car website. Each car listing has a unique document ID. The database is partitioned by document ID (IDs 0–499 in P0, 500–999 in P1, etc.). Each partition independently maintains secondary indexes on color and make covering only its own documents.

Term-Partitioned Secondary Index (Global Index)

Instead of each partition maintaining its own local index, we construct a global index that covers data in all partitions. This global index is itself partitioned — but by the index term, not the document ID. For example, colors a–r go to index partition 0, and colors s–z go to index partition 1.

The Write Complexity Trade-off

A single document write may affect multiple index partitions (one per indexed term). In an ideal world, the index would update synchronously, but that requires a distributed transaction. In practice, global secondary indexes are updated asynchronously.

Amazon DynamoDB: global secondary indexes updated "within a fraction of a second" normally, but may lag during infrastructure faults.

Key Takeaways

Partitioning = Scalability
Break data into partitions so each node handles a subset. Combined with replication for fault tolerance.
Key Range: Sorted, Range-Scannable, but Skew-Prone
Great for range queries and time-series data. Risk of hot spots with sequential keys (timestamps).
Hash: Uniform Distribution, but No Range Queries
Eliminates most skew. Keys that were adjacent are now scattered. Cassandra's compound key is the best compromise.
Hot Spots Persist Even with Hashing
Celebrity problem: identical keys hash identically. Mitigation: random suffixes (application-level responsibility).
Secondary Indexes: Local vs. Global Trade-off
Document-partitioned = simple writes, expensive reads. Term-partitioned = efficient reads, complex writes with async updates.

Test Your Understanding

1Why can't you simply assign records to nodes randomly? ▼

Random assignment distributes data evenly, but when reading a particular record, you have no way of knowing which node it is on. You would have to query all nodes in parallel for every single read, which is extremely inefficient. Partitioning schemes (key range or hash) provide a deterministic mapping from key to partition, enabling direct routing.

2A sensor network writes timestamped readings. Why might key-range partitioning cause a hot spot, and how do you fix it? ▼

If the key is purely a timestamp, all current writes go to the partition covering "now" (e.g., today's date range), creating a hot spot. The fix is to prefix the timestamp with the sensor name: the key becomes (sensor_name, timestamp). This distributes writes across partitions (one per sensor group). The trade-off: to query a time range across all sensors, you now need a separate range query per sensor prefix.

3Explain how Cassandra's compound primary key achieves a compromise between key-range and hash partitioning. ▼

Cassandra declares a compound primary key of several columns, e.g., (user_id, update_timestamp). The first column is hashed to determine the partition (providing uniform distribution like hash partitioning). The remaining columns serve as a concatenated sort key within that partition's SSTables (providing range-scan capability like key-range partitioning). You cannot range-scan the first column, but given a fixed first column value, you can efficiently range-scan the rest.

4What is the "scatter/gather" pattern, and why does it cause tail latency amplification? ▼

Scatter/gather is the query pattern for document-partitioned secondary indexes: a query is sent to all partitions in parallel, and results are combined. The overall response time equals the slowest partition's response time. Even if most partitions respond quickly, one slow partition (due to GC pause, disk I/O, network delay) dominates the total latency. With N partitions, the probability of hitting at least one slow response increases — this is tail latency amplification.

5Why are global (term-partitioned) secondary index updates typically asynchronous? ▼

A single document write may need to update index entries on multiple different partitions (one per indexed term). Synchronous updates would require a distributed transaction across all affected partitions, which is expensive and not supported by all databases. Instead, systems like DynamoDB update global indexes asynchronously — the trade-off being that reads may briefly see stale index data after a write.

6You are designing a social media platform. Celebrity users with millions of followers cause hot spots. The random-suffix mitigation splits writes across 100 keys. What are the downsides? ▼

Three main downsides: (1) Reads become expensive — you must query all 100 suffixed keys and merge the results. (2) Bookkeeping overhead — you need to track which keys are "hot" and deserve splitting; applying random suffixes to all keys would add unnecessary overhead to the vast majority of low-traffic keys. (3) Application-level responsibility — the database does not do this automatically, so the application must manage the splitting logic, suffix tracking, and read-time merging.
