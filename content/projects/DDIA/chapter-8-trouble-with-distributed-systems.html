DDIA — Part II: Distributed Data

Chapter 8: The Trouble with Distributed Systems

"A distributed system is one in which the failure of a computer you didn't even know existed can render your own computer unusable." — Leslie Lamport. This chapter is a deeply pessimistic exploration of everything that can go wrong in distributed systems: networks, clocks, and processes. Understanding these problems is the prerequisite to solving them.

Faults and Partial Failures

A program on a single computer is fundamentally deterministic: it either works or it doesn't. Hardware problems (bad RAM, disk corruption) can introduce nondeterminism, but we generally expect well-written software on a single node to be predictable. Distributed systems are different: partial failures are nondeterministic. Some parts may be broken while others work fine, and you may not even know whether an operation succeeded or failed.

"In my limited experience I've dealt with long-lived network partitions in a single data center, PDU failures, switch failures, accidental power cycles of whole racks, whole-DC backbone failures, whole-DC power failures, and a hypoglycemic driver smashing his Ford pickup truck into a DC's HVAC system. And I'm not even an ops guy." — Coda Hale

The Core Difficulty

If you try to do anything involving multiple nodes and the network, it may sometimes work and sometimes unpredictably fail. You may not even know whether something succeeded or not, because the time it takes for a message to travel across a network is itself nondeterministic. Suspicion, pessimism, and paranoia pay off in distributed systems.

Building Reliable Systems from Unreliable Components

This is an old idea in computing. Error-correcting codes transmit data accurately over a noisy channel. TCP provides reliable transport over unreliable IP (retransmitting lost packets, eliminating duplicates, reordering). The higher-level system is more reliable than its parts, though there are always limits.

Unreliable Networks

The distributed systems in this book are shared-nothing systems: machines connected by an asynchronous packet network. The network is the only way they communicate. When you send a request and expect a response, many things can go wrong:

What Can Go Wrong When Sending a Network Request
1

Your request is lost (someone unplugged a cable)

2

Your request is waiting in a queue — will be delivered later

3

The remote node has failed (crashed or powered down)

4

Remote node temporarily stopped responding (GC pause) — will respond later

5

Remote node processed your request but the response was lost

6

Remote node processed your request but the response is delayed

Key insight: If you don't get a response, it is IMPOSSIBLE to tell why. All these failures look the same from the sender's perspective.

The only option is a timeout: after some time, give up waiting and assume the response won't arrive. But even after a timeout, you still don't know if the remote node got your request or not.

Network Faults in Practice

Even Datacenters Have Network Problems

  • One study found ~12 network faults per month in a medium-sized datacenter, half disconnecting a single machine, half disconnecting an entire rack
  • Adding redundant networking gear doesn't help as much as hoped — doesn't guard against human error (misconfigured switches), a major cause of outages
  • EC2 is notorious for frequent transient network glitches
  • A software upgrade for a switch can trigger topology reconfiguration, delaying packets for over a minute
  • Sharks bite undersea cables. A network interface can drop all inbound packets but send outbound successfully.
  • Even if faults are rare, software must be able to handle them — untested error handling can cause permanent deadlocks or even data deletion

Network Partitions (Netsplits)

When part of the network is cut off from the rest, it's called a network partition. Handling network faults doesn't always mean tolerating them — showing error messages to users during problems is valid — but you must know how your software reacts and ensure it can recover. This is the idea behind Chaos Monkey: deliberately trigger problems and test the response.

Detecting Faults

Many systems need to automatically detect faulty nodes (load balancers removing dead nodes, followers being promoted to leader). But the network makes this hard.

Limited Feedback You Might Get
TCP RST/FIN

If you can reach the machine but no process is listening on the port, the OS sends RST/FIN. But if the node crashed while handling your request, you don't know how much was processed.

Process crash notification

If the process crashed but the OS is still running, a script can notify other nodes (HBase does this). Faster than waiting for timeout.

ICMP Unreachable

A router may reply with ICMP Destination Unreachable if the IP is unreachable. But the router doesn't have magic detection either.

No response at all

In general, assume you'll get no response. Retry, wait for timeout, eventually declare dead. Even TCP ACK only confirms delivery, not processing.

Timeouts and Unbounded Delays

If a timeout is the only sure way of detecting a fault, how long should it be? There is no simple answer.

Long Timeout

  • Long wait until node is declared dead
  • Users see errors or hangs during this time
  • Delays recovery and failover

Short Timeout

  • Detects faults faster
  • Higher risk of false positives (declaring alive node dead during temporary slowdown)
  • Premature failover causes duplicate actions (e.g., email sent twice)
  • Transferring load from "dead" node to others can cause cascading failure

The Ideal Timeout (If Networks Were Reliable)

If packets were guaranteed to arrive within time d, and nodes handled requests within time r, then a reasonable timeout would be 2d + r. But asynchronous networks have unbounded delays and most servers can't guarantee a maximum processing time. There is no "correct" timeout value.

Network Congestion and Queueing

The variability of packet delays is most often due to queueing, which can happen at multiple points:

Where Queueing Delays Happen (The 5 Stages of Delay)
1

Network Switch Queue

Multiple nodes sending to same destination → switch queues packets. If queue fills up, packets are dropped and must be resent.

2

OS Incoming Queue

All CPU cores busy → incoming requests queued by the OS until application is ready. Can take arbitrary time under load.

3

VM Monitor Buffer

In virtualized environments, VM is paused for tens of ms while another VM uses a CPU core. Network data queued by the hypervisor.

4

TCP Flow Control (Backpressure)

TCP limits send rate to avoid overloading the network or receiver → additional queueing at the sender before data enters the network.

5

TCP Retransmission

If a packet isn't ACK'd within a timeout, TCP retransmits. The application sees delay (waiting for timeout + retransmit + ACK) but not the packet loss itself.

TCP vs. UDP: A Trade-off

Latency-sensitive apps (VoIP, video calls) use UDP instead of TCP. UDP skips flow control and retransmission, reducing variable delays. The trade-off: lost packets stay lost. For VoIP, a lost packet's time slot is filled with silence — retransmitting would be too late. "Could you repeat that please? The sound just cut out for a moment." The retry happens at the human layer.

In multi-tenant cloud environments, you share network resources with other customers. A noisy neighbor running MapReduce can saturate links. You can only choose timeouts experimentally by measuring round-trip time distributions. Even better: use a Phi Accrual failure detector (used by Akka and Cassandra) that automatically adjusts timeouts based on observed response time variability.

Synchronous vs. Asynchronous Networks

Why can't we just make the network reliable at the hardware level? To answer this, compare computer networks to the traditional telephone network:

Why Packet Switching Wins for Data

Datacenter networks use packet switching because data transfers are bursty. A web request doesn't need a fixed bandwidth — it just wants to complete as fast as possible. Circuit switching would waste capacity. The trade-off: variable delays are the price we pay for better utilization and lower cost.

Variable delays are not a law of nature — they are the result of a cost/benefit trade-off. Bounded delays are achievable (with static resource partitioning, QoS, dedicated hardware) but expensive and lower utilization. Multi-tenancy with dynamic partitioning is cheaper but has variable delays.

Resource Partitioning: Static vs. Dynamic

↔
Dynamic (Packet)
Shared bandwidth on demand.
High utilization. Unbounded delay.

Same trade-off applies to CPUs: dynamic thread scheduling (time-sharing) vs. dedicated CPU cycles.

Unreliable Clocks

Clocks are crucial in distributed systems — for timeouts, performance measurement, cache expiry, timestamps in logs, event ordering, and more. But in a distributed system, time is tricky: communication is not instantaneous, each machine has its own clock (a quartz crystal oscillator), and these clocks drift at different rates.

Monotonic vs. Time-of-Day Clocks

Modern computers have two types of clocks that serve fundamentally different purposes:

Time-of-Day Clocks

  • Returns current date and time (wall-clock time)
  • clock_gettime(CLOCK_REALTIME) on Linux, System.currentTimeMillis() in Java
  • Synchronized with NTP → timestamp from one machine ideally means the same on another
  • Can jump backward if NTP resets the clock
  • May ignore leap seconds
  • Historically coarse-grained (10ms steps on old Windows)
  • Unsuitable for measuring elapsed time

Monotonic Clocks

  • Suitable for measuring durations (intervals)
  • clock_gettime(CLOCK_MONOTONIC) on Linux, System.nanoTime() in Java
  • Guaranteed to always move forward
  • Absolute value is meaningless (e.g., nanoseconds since boot)
  • Makes no sense to compare between two different machines
  • NTP may slew the clock speed (±0.05%) but never jump it
  • Microsecond resolution or better on most systems
  • Good for timeouts and response time measurement

Clock Synchronization and Accuracy

Time-of-day clocks need synchronization via NTP, but this is unreliable:

Why You Can't Trust Clocks

  • Quartz drift: Google assumes 200 ppm (parts per million) clock drift → 6 ms drift in 30 seconds, 17 seconds drift per day
  • NTP reset: If clock differs too much from NTP server, it may be forcibly reset → time jumps backward or forward
  • NTP firewalled: If node is accidentally cut off from NTP servers, the misconfiguration may go unnoticed for a long time
  • Network delay limits NTP accuracy: Minimum 35 ms error over the internet; spikes to ~1 second are common
  • NTP servers can be wrong: Some report time off by hours. Clients query multiple servers and ignore outliers, but it's still risky.
  • Leap seconds: A minute that is 59 or 61 seconds long. Has crashed many large systems. Best workaround: "smearing" the adjustment over a day.
  • VM clock virtualization: VM pauses for tens of ms while another VM runs → clock appears to jump forward
  • User devices: Users may deliberately set wrong time (e.g., to circumvent game time limits)

Relying on Synchronized Clocks: Dangerous!

Incorrect clocks are silent failures: unlike a broken CPU or misconfigured network that fails loudly, a drifting clock keeps working but gradually diverges from reality. If software relies on clock accuracy, the result is subtle, silent data loss.

Danger: Using Timestamps to Order Events (Last-Write-Wins)

Scenario: Two clients write to a multi-leader database. Writes tagged with local wall-clock timestamps.

Client A writes x = 1 on Node 1 at timestamp 42.004

Client B writes x = 2 on Node 3 at timestamp 42.003

B's write is causally later (it increments A's value), but has an earlier timestamp!

Result: Node 2 applies LWW, keeps x=1, drops x=2. Client B's increment is silently lost.

Problems with LWW (Last Write Wins):

  • Writes can mysteriously disappear: a node with a lagging clock can't overwrite values from a node with a fast clock
  • Can't distinguish sequential vs. concurrent writes (need version vectors for causality)
  • Two nodes can generate identical timestamps (millisecond resolution) — tiebreaker needed but still violates causality

Solution: Use logical clocks (incrementing counters) instead of physical clocks for ordering events.

Clock Readings Have a Confidence Interval

A clock reading is not a point in time — it's a range. If you only know the time ±100 ms, the microsecond digits are meaningless.

Google's TrueTime API (Spanner)

Unlike most systems, Google's TrueTime explicitly reports the confidence interval: you get back [earliest, latest]. Spanner uses this for snapshot isolation across datacenters: if two transactions' confidence intervals don't overlap, their order is certain. To ensure this, Spanner deliberately waits for the length of the confidence interval before committing. To keep waits short, Google deploys GPS receivers and atomic clocks in every datacenter, achieving ~7 ms uncertainty.

Process Pauses

Even if you fix clock issues, there's another problem: a thread can be paused for an arbitrary length of time and resume without knowing it was paused. This breaks any assumptions about "checking a condition and then acting on it."

The Lease Example: A Dangerous Pattern

A leader obtains a lease (a lock with a timeout) and checks if (lease.isValid()) before processing a request. But what if the thread pauses for 15 seconds between the check and the processing? The lease expires, another node becomes leader, but the original thread resumes and processes the request anyway — believing it still holds the lease. Two nodes now act as leader simultaneously.

Reasons a Process Can Pause for a Long Time

Garbage Collection (GC)

Stop-the-world GC pauses can last several minutes. Even "concurrent" collectors (CMS, G1) must stop the world occasionally.

VM Suspension

A VM can be suspended and resumed (live migration). Pause length depends on memory write rate. Arbitrary duration.

Context Switches & Steal Time

OS context-switches to another thread, or hypervisor switches to another VM. Under heavy load, a paused thread may wait a long time to run again.

Synchronous Disk I/O

Thread blocks on slow disk I/O. Java classloader lazily loads classes, causing surprise disk access. Network filesystem (EBS) adds network delay variability.

Memory Swapping (Paging)

A memory access triggers a page fault requiring disk I/O. Under memory pressure, system thrashes — spending most time swapping pages.

SIGSTOP / Laptop Lid Close

Unix SIGSTOP (Ctrl-Z) stops a process immediately until SIGCONT. Could be sent accidentally by an ops engineer.

Mitigating GC Pauses

An emerging approach: treat GC pauses like brief planned outages. When GC is needed, stop sending requests to that node, let it drain, GC, then resume. This hides GC pauses from clients. Another approach: use short-lived objects only (restart processes periodically before long-lived objects accumulate), using the runtime more like a short-lived process.

Real-Time Systems: The Expensive Alternative

In safety-critical systems (aircraft, rockets, cars), software must respond within a strict deadline. This requires an RTOS, extensive testing, limited libraries, limited languages, and restricted memory allocation. Very expensive, lower throughput. Not appropriate for server-side data processing. "Real-time" does not mean "high-performance" — it means "guaranteed timely response."

Knowledge, Truth, and Lies

A node in a distributed system cannot know anything for sure. It can only make guesses based on messages received (or not received). If a remote node doesn't respond, you can't distinguish network problems from node problems. This borders on the philosophical: what do we know to be true in our system?

The Truth Is Defined by the Majority

A Node Cannot Trust Its Own Judgment

Consider a node with an asymmetric fault: it can receive all messages but its outgoing messages are dropped. The node works perfectly, but the other nodes declare it dead because they can't hear its responses. "I'm not dead!" the node screams — but nobody hears. Similarly, a node in a GC pause is declared dead, even though from its perspective it was only paused for a moment.

The solution: a node cannot independently declare itself as anything (leader, lock holder, etc.). Decisions require a quorum — a minimum number of votes from nodes. Most commonly, decisions require a majority (more than half). With three nodes, one failure can be tolerated; with five, two. There can only be one majority, so conflicting decisions are prevented.

Quorum Decision Making
Node 1
VOTE: Yes
Node 2
VOTE: Yes
Node 3
No response

Quorum reached (2 out of 3). Decision proceeds. Only ONE majority can exist at any time.

The Leader and the Lock Problem

Systems often require only one of something: one leader per partition, one lock holder per resource, one user per username. But in a distributed system, even if a node believes it's "the chosen one," the majority may have demoted it (due to a GC pause or network issue). If the deposed leader keeps acting as leader, data corruption can result.

→

Lease expires!

→
Writes to file

Fencing Tokens: The Solution

Fencing Tokens Prevent Stale Writes
Lock server grants lease to Client 1 with token 33
→
Client 1 enters GC pause...
Lease expires. Lock server grants Client 2 token 34
→
Client 2 writes with token 34 ✓
Client 1 wakes up, sends write with token 33
→
Storage rejects: 33 < 34. Already seen higher token. SAFE!

The fencing token is a monotonically increasing number. The resource itself (storage server) must check the token and reject writes with older tokens than the highest it has seen. ZooKeeper's zxid or node version cversion can serve as fencing tokens.

Key Insight: Server-Side Validation

The resource must actively check fencing tokens. It's not enough for clients to check their own lock status — a paused client won't know its lock expired. Server-side checking also protects against accidentally abusive clients. This is a general principle: don't trust clients to enforce invariants.

Byzantine Faults

Fencing tokens protect against nodes that are inadvertently acting in error. But what if a node deliberately lies? Sends fake fencing tokens? This is a Byzantine fault: nodes sending arbitrary, faulty, or corrupted responses.

The Byzantine Generals Problem

A generalization of the Two Generals Problem: n generals must agree on a battle plan, but some are traitors who send conflicting messages. A system is Byzantine fault-tolerant if it operates correctly even when some nodes malfunction or malicious attackers interfere. Requires a supermajority of more than two-thirds of nodes to be honest.

When Byzantine Tolerance Matters

  • Aerospace: radiation can corrupt CPU/memory, causing unpredictable responses. System failure = crash. Flight control systems must tolerate Byzantine faults.
  • Multi-organization systems: participants may cheat or defraud (e.g., blockchain / Bitcoin — mutually untrusting parties agreeing on transactions without central authority).

When You Can Ignore It

  • In your own datacenter, all nodes are controlled by your organization
  • Radiation levels are low enough that memory corruption isn't a major problem
  • Byzantine protocols are very complex and impractical for most systems
  • If an attacker compromises one node, they can probably compromise all (same software)
  • Traditional security (auth, encryption, firewalls) remains the main defense

Weak Forms of "Lying": Worth Defending Against

  • Corrupted packets: TCP/UDP checksums catch most, but not all. Add application-level checksums.
  • Malicious user input: Sanitize inputs, check value ranges, limit string sizes. Prevent SQL injection, XSS, etc.
  • Misconfigured NTP servers: Query multiple servers, ignore outliers. More robust than trusting one source.

System Model and Reality

To reason about distributed algorithms, we need to formalize our assumptions about faults into a system model: an abstraction describing what an algorithm may assume.

Timing Models

Synchronous

Bounded network delay, bounded process pauses, bounded clock error. You know the upper bounds. Not realistic for most systems.

Partially Synchronous

Behaves synchronously most of the time, but sometimes exceeds bounds. Most realistic model. Networks usually behave well, but occasionally don't.

Asynchronous

No timing assumptions at all. Not even a clock. Can't use timeouts. Very restrictive. Some algorithms work here, but most don't.

Node Failure Models

Crash-Stop

A node fails by crashing. Once it stops, it's gone forever. It never comes back. Simplest model.

Crash-Recovery

Nodes may crash and come back after unknown time. Stable storage (disk) survives crashes; in-memory state is lost. Most practical model.

Byzantine

Nodes may do absolutely anything, including lying and sending malicious messages. Hardest to tolerate.

The Most Useful Model for Real Systems

Partially synchronous + crash-recovery is generally the most useful system model. Most of the time, the network and processes behave well. But we must handle the cases where they don't.

Correctness of Algorithms: Safety and Liveness

To define what it means for a distributed algorithm to be "correct," we describe its properties. For a fencing token algorithm:

Properties of a Correct Fencing Token Algorithm
Uniqueness
No two requests get same token
SAFETY
Monotonic Sequence
If x completes before y begins, tx < ty
SAFETY
Availability
A non-crashed node eventually gets a response
LIVENESS

Safety Properties

  • Informally: "nothing bad happens"
  • If violated, you can point to a specific moment when it broke
  • Once violated, cannot be undone — the damage is done
  • Must hold always, in all possible situations, even total failures

Liveness Properties

  • Informally: "something good eventually happens"
  • Often includes the word "eventually" (e.g., eventual consistency is a liveness property)
  • May not hold at a point in time, but there's always hope it will be satisfied later
  • Allowed to have caveats (e.g., "only if a majority of nodes are alive and network recovers")

Mapping Models to Reality

Theoretical models are invaluable for reasoning about correctness, but real implementations face messier facts: stable storage can be corrupted, firmware bugs can prevent a server from recognizing its disks on reboot, quorum algorithms break if a node has "amnesia." The difference between theory and practice: printf("Sucks to be you") and exit(666) — i.e., letting a human operator clean up. This is arguably the difference between computer science and software engineering.

Still, proving algorithms correct is a crucial first step: theoretical analysis uncovers bugs that might remain hidden for years in production and only surface when unusual circumstances defeat your timing assumptions.

First-Principles Questions

Test your understanding. Click to reveal the answer.

1Why are partial failures the defining characteristic of distributed systems, and why are they harder than total failures?▼
In a single-node system, hardware either works or it doesn't — failures are typically total and deterministic. In a distributed system, some parts can fail while others work fine, and you may not even know whether an operation succeeded. Partial failures are nondeterministic: the same operation may succeed sometimes and fail others, depending on network delays, queueing, GC pauses, and other unpredictable factors. This nondeterminism makes reasoning about correctness fundamentally harder.
2Why does a supercomputer handle faults by escalating to total failure, while cloud systems cannot?▼
Supercomputers run offline batch jobs (weather simulation, molecular dynamics) that can be checkpointed and restarted. Stopping the entire cluster for repair is acceptable. Cloud/internet services must serve users 24/7 with low latency — stopping the service is not an option. Additionally, cloud systems use commodity hardware with higher failure rates and are often geographically distributed. With thousands of nodes, something is always broken, so the system must tolerate partial failures and keep running.
3If you send a network request and get no response, why is it impossible to know what happened?▼
In an asynchronous network, six different things could have happened and they all look identical (no response): (1) request lost, (2) request queued, (3) remote node crashed, (4) remote node is in a GC pause, (5) remote node processed request but response was lost, (6) response is delayed. The sender has zero information to distinguish these cases. Even a TCP ACK only confirms delivery, not processing. This fundamental ambiguity is why distributed systems are so hard.
4Why is there no "correct" timeout value, and what happens if you set it too short or too long?▼
Too long: slow to detect failures, users wait or see errors, delayed failover. Too short: false positives (declaring alive nodes dead during temporary slowdowns), which triggers unnecessary failovers, duplicate action execution, and extra load on remaining nodes — potentially causing a cascading failure where overloaded nodes are declared dead too. A "correct" timeout would require 2d + r (max network delay + max processing time), but asynchronous networks have unbounded delays and servers can't guarantee bounded processing time. You can only determine timeouts experimentally.
5At which five points in the system can queueing delays occur, and why does this make network delay fundamentally unbounded?▼
(1) Network switch queue: multiple senders to same destination, switch buffers packets, drops if full. (2) OS incoming queue: all CPU cores busy, requests wait until application is ready. (3) VM monitor buffer: hypervisor pauses VM, queues incoming data. (4) TCP flow control: sender-side queueing to avoid overloading receiver (backpressure). (5) TCP retransmission: lost packets are retransmitted after a timeout, adding delay invisible to the application. Since each of these queues can grow without bound (especially under load), total delay is unbounded. Near maximum capacity, delays spike dramatically.
6Why do datacenter networks use packet switching instead of circuit switching, even though circuit switching provides bounded delays?▼
Data transfers are bursty: a web request doesn't need constant bandwidth, it just wants to complete fast. Circuit switching reserves a fixed bandwidth that is wasted when idle. If you guess bandwidth too low, transfers are slow; too high, the circuit can't be established. Packet switching dynamically shares bandwidth, adapting to available capacity via TCP. This maximizes wire utilization, making each byte cheaper. Variable delays are the cost/benefit trade-off for better utilization. The same principle applies to CPU time-sharing vs. dedicated CPU cycles.
7Why should you never use time-of-day clocks to measure elapsed time, and what should you use instead?▼
Time-of-day clocks can jump backward when NTP resets them, ignore leap seconds, and have coarse resolution. Two readings may show negative elapsed time. Monotonic clocks are the correct choice: they are guaranteed to always move forward, offer microsecond resolution, and NTP only adjusts their rate (slewing by up to ±0.05%) without ever causing jumps. The absolute value is meaningless (can't compare between machines), but the difference between two readings on the same machine is a reliable duration measurement.
8Why is using timestamps for "Last Write Wins" ordering dangerous, even with tightly NTP-synchronized clocks?▼
Even with excellent NTP sync (clock skew < 3 ms), timestamps can order events incorrectly. A causally-later write (B increments A's value) can have an earlier timestamp than A's write, causing LWW to silently drop B's update. Problems: (1) Writes from a node with a lagging clock mysteriously disappear. (2) Can't distinguish sequential vs. truly concurrent writes (need version vectors). (3) Identical timestamps at millisecond resolution need tiebreakers that violate causality. NTP accuracy is limited by network round-trip time — you'd need the clock source to be more accurate than what you're measuring. Logical clocks (incrementing counters) are safer for ordering.
9How does Google Spanner use TrueTime's confidence intervals to implement distributed snapshot isolation?▼
TrueTime returns [earliest, latest] instead of a single timestamp. Spanner observes: if two transactions A and B have non-overlapping confidence intervals (Alatest < Bearliest), then B definitely happened after A. To ensure intervals don't overlap for causally related transactions, Spanner deliberately waits for the length of the confidence interval before committing a read-write transaction. To minimize wait times, Google deploys GPS receivers and atomic clocks in every datacenter, achieving ~7 ms uncertainty. This allows snapshot isolation across datacenters using physical timestamps — something no other mainstream database achieves.
10What is a fencing token, and why must the resource (not the client) check it?▼
A fencing token is a monotonically increasing number returned by the lock service every time a lock/lease is granted. Clients include their token with every write. The storage service remembers the highest token it has seen and rejects writes with lower tokens. This prevents a paused client (whose lease expired) from corrupting data after waking up: its old token (e.g., 33) is rejected because the storage already processed a write with a higher token (34). The resource must check because a paused client doesn't know it was paused — it can't check its own lock status reliably. Server-side checking also protects against accidentally abusive clients.
11What is the difference between safety and liveness properties, and why does the distinction matter?▼
Safety = "nothing bad happens." If violated, you can point to a specific moment; the violation is permanent (damage done). Example: uniqueness, monotonic sequence. Liveness = "something good eventually happens." May not hold now but could be satisfied later. Often includes "eventually." Example: availability, eventual consistency. The distinction matters because distributed algorithms require safety to hold always (even if all nodes crash and the network fails), while liveness can have caveats (e.g., "only if a majority of nodes survive and the network eventually recovers"). This separation lets us make practical guarantees in an imperfect world.
12Why is "partially synchronous with crash-recovery" the most useful system model, and what are its alternatives?▼
It's the most realistic model: networks and processes behave well most of the time (synchronous), but occasionally have unbounded delays and crashes. Nodes can crash and recover with data on stable storage preserved. Alternatives: Synchronous (bounded delays, bounded pauses, bounded clock error) — too optimistic, real systems exceed these bounds. Asynchronous (no timing assumptions at all, no clocks, no timeouts) — too restrictive, most practical algorithms can't work without timeouts. Crash-stop (crashed nodes never return) — too pessimistic. Byzantine (nodes may lie) — needed for aerospace/blockchain but too complex and expensive for typical datacenter systems.
