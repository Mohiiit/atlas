Chapter 9 — Consistency and Consensus

Distributed Consensus

Consensus is the problem of getting multiple nodes to agree on a single value, in the face of failures. This is the core problem underlying leader election, atomic commit, and total order broadcast.

Formal definition: A consensus algorithm must satisfy: (1) Uniform agreement — every correct node decides the same value; (2) Integrity — no node decides twice; (3) Validity — the decided value must have been proposed by some node; (4) Termination — every correct node eventually decides.

Termination is a liveness property; the others are safety properties. The FLP impossibility result shows that in a fully asynchronous system, no deterministic algorithm can guarantee all four properties if even one node can crash. Real systems get around this via partial synchrony assumptions and randomization.

Raft Consensus Algorithm

Raft is the most widely-understood modern consensus algorithm (Paxos is older but notoriously difficult to understand). Both solve the same problem; Raft was designed for understandability.

Raft Key Concepts

Leader election
Nodes start as followers. A candidate starts an election by requesting votes. A node wins if it gets a majority. Time is divided into terms — each term has at most one leader.
Log replication
Leader receives client requests, appends to its log, sends AppendEntries RPC to followers. Once a majority acknowledge, entry is committed. Leader notifies followers to apply it.
Safety
Raft guarantees that if a log entry is committed, it will be in the log of any future leader. Elections only succeed if the candidate has the most up-to-date log.
Terms
Monotonically increasing integers. If a node sees a message with a higher term, it immediately steps down as leader. Terms detect stale leaders.

Raft Log Commit Flow

1. Client sends write to leader
Leader appends entry to its log (uncommitted).
↓
2. Leader replicates to followers
AppendEntries RPC sent to all followers. Leader waits for majority acknowledgment.
↓
3. Majority ACK → entry committed
Leader advances commit index, applies to state machine, responds to client.
↓
4. Followers apply committed entries
Next AppendEntries message tells followers the new commit index. They apply and respond.

ZooKeeper and Coordination Services

ZooKeeper is a highly available coordination service built on top of ZooKeeper Atomic Broadcast (ZAB) — a consensus algorithm similar to Raft. Applications use ZooKeeper as infrastructure for distributed coordination.

What ZooKeeper Provides

Linearizable writes
All writes go through the leader and are applied in total order. zxid (transaction ID) monotonically increases.
Watches (notifications)
Clients subscribe to a znode (ZooKeeper node). When it changes, all watchers are notified. Used for leader election triggers, config updates.
Ephemeral nodes
Znodes that are automatically deleted when the client session disconnects. Used for presence detection: if a leader crashes, its ephemeral znode disappears → triggers new election.
Sequential znodes
ZooKeeper assigns monotonically increasing numbers to sequential znodes. Used for fair queuing and fencing tokens.

Leader Election with ZooKeeper

1. Each node attempts to create ephemeral znode /leader
2. First node to create it becomes leader
3. Other nodes watch /leader
4. Leader crashes → session expires → ephemeral znode deleted
5. Watch fires → all watchers try to create /leader → one succeeds → new leader elected
Exactly one leader at all times. Failure detection via session expiry.
ZooKeeper is intentionally not for storing application data. It's for small amounts of frequently-read, infrequently-written coordination data: cluster membership, leader identity, config values, distributed locks. Application data goes in your actual database.

Membership and Coordination Services

A fundamental challenge in distributed systems: maintaining a consistent view of which nodes are alive (membership).

The Problem with Membership

A node thinks another node is dead (timeout) and removes it from membership. But the "dead" node was just slow — it's actually alive and doesn't know it's been expelled. Now you have two nodes that disagree about membership.

Using Consensus for Membership

If membership changes go through a consensus protocol, all nodes agree on the exact membership list atomically. When a node is added or removed, every node learns about the change in the same order, at the same logical time.

This is why ZooKeeper, etcd, and Consul are used for service discovery in systems like Kubernetes: they maintain a consistent registry of which services are running, updated via consensus. Every node sees the same view of which services exist.

Chapter 9 Summary

Linearizability
Strongest consistency. One copy of data. All ops at a single logical point. Expensive: requires coordination, hurts performance under partitions.
Causality
Weaker than linearizability. Preserves cause-effect order. No coordination needed for concurrent events. Practical for multi-region.
Total order broadcast
Deliver messages to all nodes in same order. Equivalent in power to consensus. Basis for replicated state machines.
2PC
Atomic commit across nodes. Coordinator-driven. Blocking: if coordinator fails mid-protocol, participants stuck. Not used alone — needs timeouts/recovery.
Raft / Paxos
Non-blocking consensus. Leader elected by majority. Log replicated to quorum. If leader fails, new election. etcd/ZooKeeper implement this.
ZooKeeper
Coordination service. Linearizable writes, watches, ephemeral nodes, sequential znodes. Use for: leader election, locking, config, membership. Not for app data.

Review Q&A

1
What is linearizability and how does it differ from serializability?
›
Linearizability (also called atomic consistency) is a property of individual operations on a single object: every operation appears to take effect at a single point in time between its start and end. Once a write completes, all subsequent reads must see the new value. It respects real-time ordering. Serializability is a property of transactions (multi-object, multi-step): the result must be equivalent to executing transactions in some serial order. Serializability doesn't require that serial order to match real-time order. Strict serializability combines both: serializable transactions that also respect real-time ordering. Spanner provides this. The practical difference: linearizability is about individual ops on single objects (like a lock); serializability is about multi-step transactions spanning many objects.
2
Why do leaderless (Dynamo-style) systems fail to provide linearizability even with quorum reads and writes?
›
Quorum reads/writes ensure that the read and write sets overlap (w + r > n), but this does NOT guarantee linearizability. Scenarios where quorums fail: (1) Sloppy quorums: writes go to nodes outside the quorum (hinted handoff), so the read quorum may miss them. (2) Concurrent writes: two concurrent writes both achieve quorum but different nodes see them in different orders. Without a single serialization point, nodes can disagree on the final value. (3) Read repair races: during read repair, the repair update may not complete before another read, leaving some nodes with stale values. (4) Clock-based LWW: using wall clocks to break ties can discard the "later" write if clocks are out of sync. True linearizability requires a single point of coordination — a leader or consensus protocol.
3
What are Lamport timestamps and what problem do they NOT solve?
›
Lamport timestamps are logical clocks that produce a total order consistent with causality. Each node has a counter. On any event: increment. On send: include counter in message. On receive: set counter = max(local, received) + 1. If A happened-before B, then ts(A) < ts(B). Ties broken by node ID.

What Lamport timestamps do NOT solve: detecting concurrent events. If ts(A) < ts(B), you can't tell whether A caused B or they were concurrent. Lamport timestamps don't distinguish these cases. For detecting concurrency, you need version vectors (one counter per node). Also: Lamport timestamps don't help with uniqueness problems that require knowing other nodes' current state at the moment of decision — you can order events after the fact, but you can't know, when you generate a username, whether another node concurrently generated the same username.
4
What is total order broadcast and how does it relate to consensus?
›
Total order broadcast (TOB) is a protocol that delivers messages to all nodes such that: (1) every node receives every message (reliable delivery); (2) every node receives messages in the same order (total order). TOB is equivalent in power to consensus: if you can solve consensus, you can implement TOB, and vice versa. TOB → consensus: to agree on a value, broadcast the value via TOB; the first one in the order wins. Consensus → TOB: use consensus to decide each message's position in the sequence. In practice, ZooKeeper and etcd implement TOB (via ZAB/Raft) and expose it to applications. Applications use TOB for: replicated state machines (applying writes in same order on all replicas), distributed lock services (all lock requests processed in total order), and atomic commit (all participants receive commit/abort decisions in same order).
5
Explain the 2PC blocking problem and why it's dangerous.
›
In two-phase commit: Phase 1 (Prepare) asks all participants to vote yes/no. Phase 2 (Commit/Abort) sends the final decision. The blocking problem: after a participant votes YES in Phase 1, it must wait for the coordinator's Phase 2 decision. If the coordinator crashes after Phase 1 but before Phase 2, participants are stuck "in doubt" — they've committed to committing, but don't know whether the coordinator decided commit or abort. They cannot: (a) commit unilaterally (what if coordinator decided abort?); (b) abort unilaterally (what if coordinator decided commit?). So they hold their locks and wait. This blocks the system: any other transaction that wants the same locks is stuck. Recovery requires the coordinator to restart and replay its WAL (write-ahead log) to replay the decision — or if the WAL is lost, manual DBA intervention. This is why 2PC is called a "blocking" protocol and why distributed databases prefer non-blocking consensus (Raft/Paxos) for atomic commit.
6
How does Raft prevent two leaders from being elected simultaneously?
›
Raft uses terms (monotonically increasing integers) and majority votes. Each term has at most one leader. To become leader, a candidate must receive votes from a majority (> N/2) of nodes. Since any two majorities must overlap by at least one node, two candidates cannot simultaneously receive majority votes for the same term — at least one node would have to vote for both, but nodes can only vote for one candidate per term. Additionally: nodes always vote for the candidate with the most up-to-date log (highest term + longest log). If a node sees a message with a higher term than its own, it immediately steps down to follower. This prevents a network-partitioned old leader from interfering — it will see the new term from the new leader and step down. So: at any point in time, there is at most one node that believes itself to be the leader with a current majority.
7
What does ZooKeeper's ephemeral node feature enable, and how does it work?
›
Ephemeral znodes in ZooKeeper are nodes that are automatically deleted when the client session that created them disconnects (due to crash or network failure). This enables: presence detection — a node's existence in ZooKeeper signals that the corresponding process is alive. When the process dies, the znode disappears. Combined with watches (event notifications on znode changes), this enables leader election: the leader holds an ephemeral znode. If it crashes, the znode is deleted, watches fire, and followers trigger a new election. Similarly, service registries use ephemeral nodes: a service registers itself on startup; if it crashes, its registration disappears automatically. This is far more reliable than time-based TTLs (where a crashed node stays "registered" until TTL expires) and doesn't require an explicit deregistration step that might not run on crash.
8
What is the FLP impossibility result and how do real systems work around it?
›
The FLP impossibility result (Fischer, Lynch, Paterson 1985): in a fully asynchronous distributed system, no deterministic consensus algorithm can guarantee termination (liveness) if even a single process can crash. The intuition: in a fully async system, you can't distinguish a crashed node from a very slow one. A slow node might be about to send a vote that changes the outcome — so you can't terminate. Real systems work around FLP via: (1) Partial synchrony assumption: assume the network is usually synchronous (messages arrive within some time). Timeouts are used as failure detectors. Raft and Paxos use this. (2) Randomization: algorithms like Ben-Or's use randomized coin flips to break symmetry and guarantee termination with probability 1. (3) Failure detectors: separate process that provides (possibly inaccurate) information about which processes have crashed. With an "eventually accurate" failure detector, consensus is solvable even in async networks. In practice, Raft and Paxos assume partial synchrony (network is usually fast) and use randomized election timeouts to avoid split votes.
