1. Thinking About Data Systems

Most applications today are data-intensive, not compute-intensive. The bottleneck is rarely raw CPU power — it's the amount of data, the complexity of data, and the speed at which it changes. An application built from a database, a message queue, a cache, a search index, and a batch processing pipeline is not just using individual tools — it is a data system.

The Three Core Concerns of Data Systems

🛡️
Reliability

System works correctly even when things go wrong — hardware fails, software bugs, human mistakes.

📈
Scalability

System handles growth gracefully — more data, more traffic, more complexity without degrading.

🔧
Maintainability

Different people can work on the system productively — fixing bugs, adapting, adding features.

2. Reliability

A reliable system continues to work correctly even when things go wrong. "Correctly" means performing the function the user expected, tolerating mistakes, preventing unauthorized access, and maintaining adequate performance. The things that can go wrong are called faults. A system that anticipates faults and copes with them is called fault-tolerant or resilient.

"Fault-tolerant" is slightly misleading — a system can only tolerate a certain class of faults, not all possible ones. A black hole swallowing your datacenter cannot be tolerated. Kleppmann suggests a more precise framing: which faults does this system tolerate?

Fault vs Failure — A Critical Distinction

Fault A component deviates from its spec. One disk sector is corrupt. One request handler throws an exception. One upstream service returns a wrong value.

Failure The whole system stops providing the required service to the user. Visible outage.

Goal: prevent faults from causing failures. Counterintuitively, Netflix's Chaos Monkey deliberately introduces faults (kills random VMs) to ensure fault tolerance is actually exercised and not just theoretically present.

Diagram 2 — The Three Types of Faults
⚙️
Hardware Faults

Hard disk crashes, RAM errors, power outages, network cable cut. Random, uncorrelated. MTTF ~10-50 years per disk; at 10,000 disks, ~1 failure/day.

🐛
Software Errors

Systematic bugs that affect all nodes simultaneously. Harder to anticipate than hardware faults. A runaway process consuming all CPU/memory/disk/network.

👤
Human Errors

Configuration mistakes, deployment mistakes, bad API design forcing workarounds. Studies show operators are the leading cause of outages, not hardware.

Hardware Faults

Hard disks have a mean time to failure (MTTF) of roughly 10 to 50 years. On a storage cluster with 10,000 disks, you should expect roughly one disk failure per day. The first response was hardware redundancy: RAID for disks, dual power supplies, hot-swappable CPUs, diesel generators. This kept systems running through single-component failures.

Diagram 3 — Evolution of Hardware Fault Tolerance

Era 1: Hardware Redundancy (Single Machine)

RAID disk array Dual power supplies Hot-swap CPUs Diesel generators

Works for single-component failures. Downtime still required for maintenance. Adequate when a machine could run for years without needing a restart.

Era 2: Software Fault Tolerance (Multi-Machine)

Rolling upgrades (no downtime) Tolerate loss of entire machines Cloud VMs (higher single-machine fault rate) Replication across nodes

Required by: cloud VMs (less reliable than physical), need to patch OS without downtime, large clusters where machine failure is routine.

As platforms moved to cloud, single-VM reliability dropped but total-system reliability improved through software-level redundancy. The expectation shifted: machines will fail; design around it.

Software Errors

Hardware faults are random and independent — one disk failing doesn't make another more likely to fail. Software errors are different: a bug in the OS kernel, a runaway process, a downstream service returning wrong results, a cascading failure triggered by unexpected input — these affect every node simultaneously.

Real Examples of Systematic Software Faults

  • A leap second on June 30, 2012 caused hundreds of servers running Linux to go into infinite CPU loops simultaneously — Reddit, Mozilla, LinkedIn, and others were all affected
  • A bad input that causes every instance of a service to crash (e.g., an invalid configuration value propagated to all nodes at deploy time)
  • A cascading failure where a slow downstream service causes upstream request queues to fill up, which eventually takes down the whole system
  • A memory leak that slowly degrades performance until all nodes require restarting

There is no quick fix. Requires: careful thinking about assumptions, thorough testing, process isolation, monitoring, and allowing processes to crash and restart automatically.

Human Errors

Humans design, build, and operate software systems. Humans are also unreliable. One study of internet services found that configuration errors by operators were the leading cause of outages, while hardware faults played a role in only 10–25% of outages. How do you make systems reliable when humans are involved?

Diagram 5 — Six Strategies to Minimize Human Error Impact
1. Design for correctness

APIs and abstractions that make it easy to do the right thing and hard to do the wrong thing. If the easiest path leads to the right behavior, operators will naturally do the right thing.

2. Decouple risk

Provide sandbox environments where people can explore and experiment without affecting real users. A staging environment that mirrors production lets operators test changes safely.

3. Test thoroughly

Unit tests, integration tests, end-to-end tests, manual tests, automated checks. Testing is especially valuable at the edges — corner cases humans wouldn't think to test manually.

4. Enable fast recovery

Make it fast to roll back configuration changes. Roll out new code gradually (canary releases). Provide tools to reprocess/re-derive data in case something went wrong.

5. Monitor carefully

Detailed performance metrics, error rates, SLA compliance. When something goes wrong, telemetry is essential to understand what is happening. Monitoring is your early warning system.

6. Train and manage

Good management practices, training in complex systems, runbooks for common operations. The culture of an organization affects how humans behave in the system.

When Is Reliability Not Worth It?

There are situations where sacrificing reliability is acceptable — a prototype or MVP where correctness doesn't matter yet, or a service where the cost of fault tolerance exceeds the cost of the fault itself. But be deliberate about it. Knowing when you're cutting corners on reliability is different from not knowing they were cut at all.

Unreliability has costs beyond the outage itself: loss of productivity (employees can't work), revenue (e-commerce downtime), regulatory fines (financial systems), and — most insidiously — gradual erosion of user trust that's hard to quantify and hard to rebuild.
