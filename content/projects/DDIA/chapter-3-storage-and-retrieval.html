DDIA -- Part I: Foundations of Data Systems

Chapter 3: Storage and Retrieval

A database needs to do two things: store data when you give it, and give it back when you ask. Understanding how it does this lets you choose the right engine for your workload. This chapter examines the internals of storage engines -- from the simplest append-only log to sophisticated column-oriented warehouses.

"Wer Ordnung halt, ist nur zu faul zum Suchen." (If you keep things tidily ordered, you're just too lazy to go searching.) -- German proverb

The World's Simplest Database

Two bash functions implement a complete key-value store. db_set key value appends a comma-separated pair to a file. db_get key greps for the key and returns the last match (the most recent value). This reveals the core tension in storage systems:

#!/bin/bash
db_set () { echo "$1,$2" >> database; }
db_get () { grep "^$1," database | sed -e "s/^$1,//" | tail -n 1; }

$ db_set 42 '{"name":"San Francisco","attractions":["Golden Gate Bridge"]}'
$ db_set 42 '{"name":"San Francisco","attractions":["Exploratorium"]}'
$ db_get 42
{"name":"San Francisco","attractions":["Exploratorium"]}
Diagram 1: Append-Only Log -- Step by Step
Step 1: db_set 123 "London" Data File (database): +---------------------------------------------+ | 123,London | ← offset 0 +---------------------------------------------+ Step 2: db_set 42 "San Francisco" Data File (database): +---------------------------------------------+ | 123,London | ← offset 0 | 42,San Francisco | ← offset 28 +---------------------------------------------+ Step 3: db_set 42 "Exploratorium" (update!) Data File (database): +---------------------------------------------+ | 123,London | ← offset 0 | 42,San Francisco | ← offset 28 (stale!) | 42,Exploratorium | ← offset 56 (latest!) +---------------------------------------------+ db_get 42 → Must scan ALL lines, take last match Write: O(1) -- just append Read: O(n) -- scan everything

Writes are O(1) because appending to the end of a file is the simplest possible write operation -- no seeking, no rewriting. But reads are O(n) because we must scan the entire file to find the latest value for a key. Real databases use logs (append-only sequences of records) as a fundamental building block, but they add indexes to speed up reads.

The Fundamental Trade-Off of Indexes

An index is an additional structure derived from the primary data. It speeds up reads but slows down writes, because every write must also update the index. Databases don't index everything by default -- the application developer must choose indexes using knowledge of typical query patterns. This is a conscious trade-off: well-chosen indexes speed up reads enormously without slowing writes too much.

Simply appending to a file is the fastest possible write. Any index you add will slow writes down. The goal is to find the sweet spot where read speedup vastly exceeds write overhead.

Two Families of Storage Engines

OLTP storage engines fall into two broad categories with fundamentally different design philosophies. Understanding which family your database belongs to helps predict its performance characteristics.

Hash Indexes (Bitcask)

The simplest indexing strategy: keep an in-memory hash map where every key maps to a byte offset in the data file. On write: append to file, update hash map. On read: look up offset in hash map, seek to that position, read the value. This is exactly what Bitcask (the default storage engine in Riak) does.

Diagram 3: In-Memory Hash Index Pointing to Byte Offsets
IN-MEMORY HASH MAP DATA FILE ON DISK +-------------------+ +---------------------------------------+ | key | offset | | Byte 0: 123456,{"name":"London"...} | |-------|-----------| | Byte 64: 42,{"name":"San Fran"...} | | 123456| 0 ----+---------→ | Byte 128: 99,{"name":"Paris"...} | | 42 | 64 ----+---------→ | Byte 192: 42,{"name":"Explora"...} | | 99 | 128 ----+---------→ | | +-------------------+ +---------------------------------------+ WRITE db_set(42, "new_value"): 1. Append "42,new_value" to end of file at byte 256 2. Update hash map: 42 → 256 (overwrites old offset 64) READ db_get(42): 1. Hash map lookup: 42 → byte 256 2. Seek to byte 256 in file, read value (one disk seek!)

Requirement: All keys must fit in RAM (hash map is entirely in memory). Values can exceed RAM since they're loaded from disk with a single seek. If the data file portion is in the filesystem cache, reads require zero disk I/O.

Bitcask Use Case

Ideal for workloads with many writes per key but a manageable number of distinct keys. Example: URL of a cat video (key) mapped to play count (value). Lots of writes (every play increments the counter), but relatively few distinct URLs. All keys fit comfortably in memory.

Compaction and Segment Merging

Problem: the data file grows forever. Solution: break the log into segments of a certain size. When a segment is full, close it and start a new one. Then run compaction: throw away duplicate keys, keeping only the most recent value. Merge multiple compacted segments into one.

Old segments deleted after merge completes. Reads use old segments until merge is done -- zero downtime.

Implementation Details for Real Systems

  • File format: Binary format (length-prefixed strings), not CSV. Faster and simpler.
  • Deleting records: Append a tombstone -- a special deletion marker. Merging process discards previous values for that key.
  • Crash recovery: In-memory hash maps are lost on restart. Bitcask stores snapshots of each segment's hash map on disk for fast recovery.
  • Partially written records: Checksums detect corrupted records from mid-write crashes.
  • Concurrency: One writer thread (sequential appends). Segment files are immutable after closing, so multiple reader threads are safe.

Limitations of Hash Indexes

  • Hash table must fit in memory: If you have billions of distinct keys, the hash map won't fit in RAM. On-disk hash maps are possible but perform poorly (lots of random I/O, expensive to grow, hash collision handling is complex).
  • Range queries are inefficient: Cannot scan keys between kitty00000 and kitty99999 -- must look up each key individually. No ordering.

These two limitations motivate the next evolution: SSTables.

SSTables and LSM-Trees

A simple but powerful change: require that key-value pairs in each segment are sorted by key. This format is called a Sorted String Table (SSTable). Each key appears only once per merged segment (compaction ensures this).

Three Advantages of SSTables over Hash Indexes

  • Efficient merging: Like the merge step of mergesort. Read input segments side by side, output the lowest key. When the same key appears in multiple segments, keep the value from the most recent segment. Works even if files are larger than available memory.
  • Sparse in-memory index: Don't need every key in memory. If you know offsets for handbag and handsome, then handiwork must be between them -- just scan that small range on disk. One index entry per few kilobytes of segment file is enough.
  • Block compression: Group records into blocks, compress before writing to disk. Each sparse index entry points at the start of a compressed block. Saves disk space and I/O bandwidth.

Constructing SSTables: The Memtable

How do you sort data by key when writes arrive in random order? Maintain a sorted structure in memory using a balanced tree (red-black tree or AVL tree). This in-memory tree is called a memtable. When it exceeds a size threshold (typically a few MB), write it out to disk as a sorted SSTable file.

LSM-Tree Write Path -- Step by Step

1
Write arrives: Insert key-value pair into the memtable (in-memory balanced tree). Also append to the Write-Ahead Log (WAL) for crash recovery. The memtable automatically maintains sorted order.
2
Memtable full: When the memtable exceeds a threshold (e.g., 4 MB), it becomes an immutable memtable. A new empty memtable is created for incoming writes. The immutable memtable is scheduled for flushing.
3
Flush to disk: The immutable memtable is written as a new SSTable file (Level 0). Since the tree was already sorted, this is a sequential write -- very fast. The corresponding WAL segment is discarded.
4
Background compaction: A background thread merges and compacts SSTables. In leveled compaction (LevelDB, RocksDB): Level 0 SSTables are merged into Level 1, Level 1 into Level 2, etc. Each level is 10x larger than the previous. In size-tiered compaction (HBase, Cassandra): newer, smaller SSTables are merged into older, larger ones.
Diagram 7: Write-Ahead Log (WAL) for Crash Recovery
WRITE REQUEST: SET("cat", 7) | | Step 1: Append to WAL FIRST (before memtable) | v +----------------------------------+ | WAL (append-only file on disk) | | ... | | SET cat=7 ← new entry | +----------------------------------+ | | Step 2: Then update memtable v +----------------------------------+ | MEMTABLE (in-memory) | | bird=2, cat=7, dog=3, fox=1 | +----------------------------------+ IF CRASH HAPPENS: +----------------------------------+ | Memtable is LOST (it was in RAM) | +----------------------------------+ | | On restart: replay WAL from disk v +----------------------------------+ | Reconstruct memtable from WAL | | bird=2, cat=7, dog=3, fox=1 | | No data lost! | +----------------------------------+ AFTER FLUSH: When memtable is written to SSTable on disk, the corresponding WAL segment is discarded. WAL is only needed for data not yet in an SSTable.

Crash Recovery in LSM-Trees

The memtable lives in memory -- if the database crashes, its contents are lost. The WAL solves this: every write is appended to the WAL before updating the memtable. On crash recovery, the WAL is replayed to reconstruct the memtable. The WAL is not sorted (entries appear in write order), but that doesn't matter -- its only purpose is crash recovery. Once a memtable is flushed to an SSTable, the corresponding WAL can be deleted.

Compaction Strategies

Size-Tiered Compaction

Used by HBase, Cassandra (default). Newer, smaller SSTables are successively merged into older, larger SSTables. Simple but can temporarily use a lot of disk space (old + new SSTables coexist during compaction).

Leveled Compaction

Used by LevelDB, RocksDB. Key range is split across smaller SSTables at each level. Level N is ~10x larger than Level N-1. More incremental, uses less disk space, but more compaction work overall. Better read performance (fewer files to check at each level).

Diagram 8: Bloom Filter -- Avoiding Unnecessary Disk Reads
PROBLEM: Reading a key that doesn't exist in the database. Without optimization: must check memtable + ALL SSTables at all levels. Could require many disk reads just to conclude "key not found." SOLUTION: Bloom Filter A memory-efficient probabilistic data structure for each SSTable. +-------------------------------------------------------------------+ | Bloom Filter for SSTable-3: [0,1,0,1,1,0,0,1,0,1,0,0,1,0,1,0] | | Bit array (very compact, ~10 bits | | per key in the SSTable) | +-------------------------------------------------------------------+ How it works: 1. Hash the key with k different hash functions 2. Check if ALL corresponding bits are set to 1 Query: "Is key 'zebra' in SSTable-3?" hash1("zebra") = bit 3 → bit[3] = 1 hash2("zebra") = bit 7 → bit[7] = 1 hash3("zebra") = bit 11 → bit[11] = 0 ← Not all bits are 1! Answer: "zebra" is DEFINITELY NOT in this SSTable. Skip it! Key properties: - If Bloom filter says NO → key is definitely not in the SSTable (100% certain) - If Bloom filter says YES → key might be in the SSTable (small false positive rate) - False positive rate tunable: ~1% with 10 bits/key, ~0.1% with 15 bits/key - Saves enormous amounts of disk I/O for nonexistent key lookups

Performance Optimizations in Practice

  • Bloom filters: Each SSTable has a Bloom filter. Before reading an SSTable from disk, check the Bloom filter. If it says "no," skip the SSTable entirely. Essential for LSM-trees because without it, nonexistent key lookups would be extremely slow.
  • Sparse index caching: Keep the sparse index for each SSTable in memory for fast binary search.
  • Block cache: Cache frequently accessed SSTable blocks in memory to avoid repeated disk reads.
  • Fence pointers: Store min/max key for each SSTable to quickly rule out SSTables during range queries.

LSM-Trees in the Real World

The LSM-tree algorithm is used in LevelDB and RocksDB (embeddable key-value stores), Cassandra, HBase (distributed databases inspired by Google Bigtable), and Lucene (full-text search engine used by Elasticsearch and Solr). Lucene's term dictionary maps each word to a postings list (list of document IDs containing that word), stored in SSTable-like sorted files merged in the background.

B-Trees

The most widely used indexing structure, introduced in 1970 and called "ubiquitous" less than 10 years later. B-trees remain the standard index in virtually all relational databases. Like SSTables, B-trees keep key-value pairs sorted by key (enabling lookups and range queries), but the design philosophy is completely different.

B-Tree Design Philosophy

  • Break the database into fixed-size pages (blocks), typically 4 KB, corresponding to disk hardware
  • Each page can be read/written as a unit and can reference other pages via on-disk pointers
  • One page is the root -- all queries start here
  • Internal pages contain keys in sorted order + references to child pages
  • Leaf pages contain actual values (or pointers to where values are stored)
  • The branching factor is typically several hundred (depends on page size and key size)
  • A 4-level tree with branching factor 500 stores up to 256 TB of data

Making B-Trees Reliable: WAL (Redo Log)

B-trees modify pages in place on disk. A page split requires writing multiple pages atomically -- if the database crashes mid-split, the tree is corrupted (orphan pages with no parent). The solution: a Write-Ahead Log (WAL), also called a redo log. Every B-tree modification is written to the WAL before applying it to the tree pages. After a crash, the WAL is replayed to restore the B-tree to a consistent state.

Concurrency control: Multiple threads accessing a B-tree simultaneously can see inconsistent state during page splits. Solution: latches (lightweight locks) protect the tree's data structures. This is more complex than log-structured approaches where background merging doesn't interfere with queries.

B-Tree Optimizations

  • Copy-on-write: Instead of overwriting pages + WAL, write modified pages to new locations. Parent pages point to new locations. Used by LMDB. Also useful for snapshot isolation (MVCC).
  • Key abbreviation: Interior pages only need enough key info to act as boundaries. Shorter keys = higher branching factor = fewer levels = faster lookups.
  • Sequential leaf layout: Try to place leaf pages in sequential order on disk for fast range scans. Hard to maintain as tree grows (LSM-trees are better at keeping sequential keys close on disk).
  • Sibling pointers: Each leaf page has references to its left and right neighbors, enabling efficient range scans without backtracking to parent pages.
  • Fractal trees: B-tree variants that borrow log-structured ideas to reduce disk seeks (used in TokuDB).

Write Amplification Explained

Write amplification = one application write results in multiple writes to disk over the database's lifetime. For B-trees: at minimum 2x (WAL + page), often more due to page splits and full-page writes even for tiny changes. Some engines overwrite pages twice to avoid partial updates on power failure. For LSM-trees: data is written once to WAL, once to memtable flush, then rewritten multiple times through compaction levels. On SSDs, write amplification matters because flash cells wear out after a limited number of write cycles.

Other Indexing Structures

Secondary Indexes

Both B-trees and LSM-trees can serve as secondary indexes. The key difference from primary indexes: keys are not unique (many rows may match). Two approaches: (1) value is a list of matching row IDs (postings list), or (2) make each key unique by appending a row ID.

Storing Values: Heap File vs. Clustered Index

Heap File (Non-Clustered)

Index stores key → pointer to row in a heap file. Actual data lives in the heap file in no particular order. Multiple indexes can reference the same heap location. Extra hop from index to heap on reads.

Clustered Index

Row data stored directly within the index. No extra hop. MySQL InnoDB: primary key is always a clustered index. Secondary indexes reference the primary key (not heap location). Faster reads but more expensive writes and storage.

Covering Index (Index with Included Columns)

A compromise: stores some columns within the index (not all). If a query only needs those columns, it can be answered from the index alone (the index "covers" the query). Speeds up common queries without the full overhead of a clustered index.

Multi-Column Indexes

Concatenated Index & Multi-Dimensional Index

  • Concatenated index: Combines fields into one key (e.g., (lastname, firstname)). Like a phone book: useful for lastname queries or lastname+firstname, but useless for firstname-only queries.
  • Multi-dimensional index (R-trees): For geospatial queries like "all restaurants within this rectangle on a map." A standard B-tree can only give you latitude ranges OR longitude ranges, not both simultaneously. R-trees and space-filling curves solve this. Used by PostGIS, HyperDex.
  • Beyond geography: Color search (red, green, blue dimensions), weather data (date, temperature), etc.

Full-Text Search and Fuzzy Indexes

Standard indexes assume exact key matches or range queries. Full-text search needs synonyms, grammatical variations, typo tolerance (edit distance). Lucene uses an SSTable-like structure for its term dictionary, with an in-memory index that is a finite state automaton (similar to a trie) over characters. This can be transformed into a Levenshtein automaton for efficient fuzzy search within a given edit distance.

In-Memory Databases

Why In-Memory Databases Are Fast

Products like VoltDB, MemSQL, Redis, and RAMCloud keep all data in memory. Counterintuitively, their speed advantage is not because they avoid reading from disk -- a disk-based engine with enough RAM also avoids disk reads thanks to OS caching. Rather, in-memory databases are fast because they avoid the overhead of encoding data structures in a disk-writable format. They can use data structures like priority queues, sets, and sorted sets that would be complex to implement on disk.

Durability is achieved via WAL on disk, periodic snapshots, or replication to other machines. The anti-caching approach (evict cold data to disk, reload on access) extends in-memory databases beyond available RAM while keeping indexes fully in memory.

Transaction Processing (OLTP) vs. Analytics (OLAP)

Databases serve two fundamentally different workloads with very different access patterns. Understanding this distinction is crucial for choosing the right storage engine.

Why Separate OLTP and OLAP?

OLTP databases must be highly available and low-latency -- they're often critical to business operations. Running expensive analytic queries (scanning millions of rows) harms transaction performance. Database administrators are reluctant to let analysts run ad hoc queries on production OLTP databases. The solution: extract data into a separate data warehouse optimized for analytics.

Data Warehousing & ETL

A data warehouse is a separate, read-only database that contains copies of data from all the various OLTP systems in a company, optimized for analytic queries. Data is extracted from OLTP databases, transformed into an analysis-friendly schema, cleaned up, and loaded into the warehouse.

OLTP vs. Data Warehouse Internals

Both use SQL and look similar on the surface. But internally they are very different. OLTP uses row-oriented storage with B-tree/LSM-tree indexes. Warehouses use column-oriented storage with bitmap indexes and compression. Some products (SQL Server, SAP HANA) support both in the same system, but they're increasingly separate engines behind a common SQL interface.

Warehouse vendors: Teradata, Vertica, SAP HANA, Amazon Redshift (based on ParAccel). Open source: Apache Hive, Spark SQL, Presto, Apache Drill (some based on Google Dremel/BigQuery).

Star Schema (Dimensional Modeling)

The most common data model for data warehouses. A central fact table captures individual events (each row = a sale, a page view, a click). Dimension tables describe the who, what, where, when, how, and why of each event.

Fact Tables Are HUGE

Fact tables capture individual events for maximum analysis flexibility. A big enterprise (Apple, Walmart, eBay) may have tens of petabytes of transaction history, mostly in fact tables. Fact tables often have 100+ columns (some are attributes like price/quantity, others are foreign keys to dimension tables). Dimension tables are much smaller (millions of rows) but can be very wide (all metadata relevant for analysis).

Column-Oriented Storage

Analytic queries typically access only 4-5 columns out of 100+. Row-oriented storage loads entire rows even if you only need a few columns -- extremely wasteful when scanning billions of rows. The idea behind column-oriented storage is simple: store all values from each column together, in separate files.

ROW-ORIENTED STORAGE (traditional OLTP) Disk layout: all columns of a row stored together +--------------------------------------------------------------+ | Row 1: | 2024-01-05 | Apple | Store-A | 10 | $1.50 | ... | | Row 2: | 2024-01-05 | Banana | Store-B | 5 | $0.80 | ... | | Row 3: | 2024-01-06 | Apple | Store-A | 12 | $1.50 | ... | | Row 4: | 2024-01-06 | Candy | Store-C | 20 | $3.00 | ... | | Row 5: | 2024-01-07 | Banana | Store-A | 8 | $0.80 | ... | +--------------------------------------------------------------+ Query: SELECT SUM(quantity) WHERE product = 'Apple' Must load ALL 100+ columns per row just to check product and sum quantity! ~98% of loaded data is thrown away. ================================================================ COLUMN-ORIENTED STORAGE (data warehouse) Disk layout: each column stored in a separate file date.col: | 2024-01-05 | 2024-01-05 | 2024-01-06 | 2024-01-06 | 2024-01-07 | product.col: | Apple | Banana | Apple | Candy | Banana | store.col: | Store-A | Store-B | Store-A | Store-C | Store-A | quantity.col: | 10 | 5 | 12 | 20 | 8 | price.col: | $1.50 | $0.80 | $1.50 | $3.00 | $0.80 | Query: SELECT SUM(quantity) WHERE product = 'Apple' Only load product.col and quantity.col -- 2 columns out of 100+! ~98% reduction in data loaded from disk. Why Column Storage Works for Analytics

  • Reduced I/O: Only load columns referenced by the query. For a 100-column table where the query touches 5 columns, this is a 20x reduction in data read from disk.
  • Better compression: Values in a single column have the same type and often repeat heavily (e.g., only 200 distinct countries across billions of rows). This enables aggressive compression.
  • CPU cache efficiency: Compressed column chunks fit in L1 cache. Tight loops without function calls. SIMD instructions operate on column vectors directly.
  • Parquet format: Columnar storage for Hadoop/Spark, supports document data models. Based on Google Dremel.

Compression Details

If a column has n distinct values, we create n bitmaps (one bit per row each). If n is very small (e.g., ~200 countries), bitmaps are stored directly. If n is larger, most bitmaps are sparse (mostly zeros) and benefit from run-length encoding. The sorted first column compresses especially well: long runs of the same value. Even with billions of rows, the first sort column may compress to just a few kilobytes.

Important: Cassandra and HBase have "column families" but are NOT column-oriented. They store all columns from a row together within each column family. The Bigtable model is still mostly row-oriented.

Sort Order in Column Storage

Sorted Column Storage

Rows can be sorted by a chosen column (e.g., date_key). All column files must be sorted in the same order (otherwise the k-th entry invariant breaks). The first sort column compresses extremely well -- long runs of the same value become tiny with run-length encoding. A second sort column can further organize rows with the same first-column value.

C-Store/Vertica trick: Store the same data sorted in different orders on different replicas. Since data must be replicated for fault tolerance anyway, you might as well sort each replica differently. Queries use whichever sort order best fits the query pattern. This is like having multiple secondary indexes, but without pointers -- just differently sorted column data.

Writing to Column-Oriented Storage

Inserting a row means updating every column file. With compressed, sorted columns, update-in-place is impossible. Solution: the LSM-tree approach. Writes first go to an in-memory store (can be row-oriented or column-oriented). When enough writes accumulate, they are merged with the on-disk column files in bulk. This is what Vertica does. Queries examine both on-disk column data and recent in-memory writes, combining the two transparently.

Chapter 3 -- The Complete Big Picture

Storage engines serve two broad categories:

OLTP (transaction processing): User-facing, high volume of small queries. Bottleneck = disk seek time. Two schools: (1) Log-structured -- append-only, sequential writes, LSM-trees (Bitcask, LevelDB, RocksDB, Cassandra, HBase, Lucene). Higher write throughput, compaction overhead. (2) Update-in-place -- B-trees, fixed pages, overwrite (PostgreSQL, MySQL, Oracle). More predictable reads, easier transactions. Each key in exactly one place.

OLAP (analytics): Analyst-facing, low volume of huge queries. Bottleneck = disk bandwidth. Column-oriented storage, bitmap compression, vectorized processing. Star/snowflake schemas. Materialized views and data cubes for precomputed aggregates. Writing uses LSM-tree approach (buffer in memory, bulk merge to column files).

The core insight: OLTP and OLAP have fundamentally different access patterns (seek vs. bandwidth, point vs. scan, current state vs. history). Different problems require different storage engines. Understanding these internals helps you choose the right tool and tune it effectively.

"As an application developer, if you're armed with this knowledge about the internals of storage engines, you are in a much better position to know which tool is best suited for your particular application. If you need to adjust a database's tuning parameters, this understanding allows you to imagine what effect a higher or a lower value may have."

First-Principles Questions

Test your understanding. Click to reveal the answer.

1Why does an append-only log have O(1) writes but O(n) reads?▼
Writing just appends to the end of the file -- always the same cost regardless of database size. No seeking, no rewriting, no index update. But reading requires scanning the file from beginning to end to find the latest value for a key, because later writes with the same key override earlier ones. If you have n records, you must scan up to n entries. This is why we need indexes: they trade a small write overhead for a massive read speedup.
2What is write amplification, and why does it matter differently for SSDs vs HDDs?▼
Write amplification means one application-level write results in multiple physical writes to disk over the database's lifetime. For B-trees: at minimum 2x (WAL + page write), often more during page splits or when writing full 4KB pages for small changes. For LSM-trees: data is rewritten through compaction levels. On HDDs, write amplification mainly affects throughput (disk bandwidth is shared between application writes and compaction). On SSDs, it's more critical because flash cells have a limited number of write-erase cycles before wearing out. Higher write amplification = faster SSD degradation. LSM-trees typically have lower write amplification than B-trees, making them more SSD-friendly.
3How does an SSTable improve over a hash-indexed log, and what is the key trick that makes it possible?▼
SSTables require key-value pairs to be sorted by key within each segment. This enables: (1) efficient merging via mergesort (read segments side-by-side, output lowest key), (2) a sparse in-memory index (don't need every key -- scan between known offsets), (3) efficient range queries, and (4) block compression. The trick that makes sorting possible despite random writes: maintain a memtable (in-memory balanced tree like a red-black tree) that keeps data sorted automatically. When the memtable is full, write it out to disk as a sorted SSTable. The sorting happens in memory, so disk writes are always sequential.
4Why do both B-trees AND LSM-trees need a Write-Ahead Log, but for different reasons?▼
B-trees need a WAL because they modify pages in place. A page split requires writing multiple pages (two split pages + updated parent). If the crash happens mid-split, the tree is left in a corrupted state with orphan pages. The WAL records the intended changes so the tree can be restored to a consistent state. LSM-trees need a WAL because the memtable lives in memory. If the process crashes, the memtable contents (all writes since the last flush) are lost. The WAL allows reconstructing the memtable on restart. In both cases, the WAL is append-only (fast sequential writes) and serves as a recovery mechanism, but the failure modes it protects against are different: structural corruption (B-trees) vs. data loss (LSM-trees).
5What is "compaction runaway" in LSM-trees, and why don't B-trees have this problem?▼
In LSM-trees, background compaction threads share disk bandwidth with foreground writes and reads. If write throughput is very high and compaction can't keep up, the number of unmerged SSTables on disk keeps growing. This causes: (1) reads slow down (must check more SSTables), (2) disk space grows unboundedly, (3) eventually the system degrades catastrophically. Most SSTable engines don't throttle incoming writes even when compaction falls behind, so explicit monitoring is needed. B-trees don't have this problem because they have no background compaction process -- writes go directly to the tree pages. Their write cost is more predictable (and more expensive per write), but there's no risk of runaway accumulation.
6Why is the OLTP bottleneck "disk seek time" while the OLAP bottleneck is "disk bandwidth"?▼
OLTP queries read a small number of specific records by key. The cost is dominated by finding each record -- seeking to its location on disk (or traversing a B-tree). The actual data transfer is tiny. OLAP queries scan millions or billions of rows, reading only a few columns. The cost is dominated by how fast you can stream data from disk to memory. Seeking isn't the issue (you're reading sequentially); the bottleneck is the raw data transfer rate. This is why OLTP uses indexes (minimize seeks) while OLAP uses column storage + compression (maximize bandwidth efficiency by minimizing bytes read).
7Why is column-oriented storage so much better for compression than row-oriented storage?▼
Values in a single column have the same data type and are often highly repetitive. A "country" column with billions of rows might have only ~200 distinct values. A "product_category" column might have ~50 distinct values. Bitmap encoding creates one bitmap per distinct value (one bit per row), and sparse bitmaps compress further with run-length encoding. When the column is sorted, the first sort column has extremely long runs of identical values (e.g., all January dates together), compressing to just kilobytes even with billions of rows. Row-oriented storage mixes different data types (strings, integers, dates, floats) in each row, making compression much less effective because there's no type-level or value-level repetition pattern to exploit.
8How does Vertica solve the write problem for column-oriented storage?▼
Inserting a row into compressed, sorted column storage would require rewriting every column file. Vertica uses the LSM-tree approach: writes go to an in-memory buffer (which can be row-oriented or column-oriented), where they're sorted and accumulated. When enough writes build up, they're bulk-merged with the on-disk column files, producing new column files. Queries transparently combine on-disk column data with recent in-memory writes. This is the same memtable-flush-merge pattern from LSM-trees, adapted for columnar storage. Reads see a consistent, up-to-date view even though the physical storage lags behind.
9What is a Bloom filter, and why is it described as "essential" for LSM-trees but unnecessary for B-trees?▼
A Bloom filter is a memory-efficient probabilistic data structure. It answers "Is key X in this set?" with either "definitely no" (100% certain) or "maybe yes" (small false positive rate). For LSM-trees, looking up a nonexistent key requires checking the memtable + every SSTable at every level. Without Bloom filters, this could mean dozens of disk reads just to conclude "not found." Bloom filters let you skip SSTables that definitely don't contain the key, turning a potentially expensive operation into a few in-memory bit checks + at most 1-2 disk reads. B-trees don't need Bloom filters because each key exists in exactly one place. A B-tree lookup follows a single path from root to leaf -- you either find the key or you don't, in O(log n) page reads regardless.
10Why does the star schema use a fact table with foreign keys rather than denormalizing everything into one giant table?▼
The fact table captures individual events (each sale, click, or page view) with foreign keys to dimension tables. This design provides maximum flexibility: analysts can slice and dice by any combination of dimensions without the data model constraining them. Denormalizing everything into one table would (1) massively duplicate dimension data (store name, address, etc. repeated billions of times), (2) make updates to dimension data (e.g., renaming a store) require updating billions of fact rows, and (3) make the already-wide fact table even wider (hundreds of columns for every dimension's attributes). The star schema separates "what happened" (facts) from "context about what happened" (dimensions), allowing each to be managed and queried independently.
11Why do in-memory databases get their speed advantage from avoiding disk-format encoding overhead, NOT from avoiding disk reads?▼
This is counterintuitive. A disk-based database with sufficient RAM rarely reads from disk either, because the operating system caches recently used disk blocks in memory (page cache). So both in-memory and disk-based databases serve reads from memory. The real advantage of in-memory databases is that they don't need to encode data structures in a format suitable for writing to disk. Disk-based engines must serialize/deserialize data to/from page-aligned, crash-safe formats. In-memory databases can use native data structures (red-black trees, skip lists, hash tables, priority queues) directly, without this encoding overhead. They still write to disk for durability (WAL, snapshots), but reads never touch disk-format data.
12What is the trade-off of a data cube (materialized OLAP cube), and why do most warehouses keep raw data anyway?▼
A data cube precomputes aggregates (SUM, COUNT, AVG) for every combination of dimension values. Queries that match the cube's dimensions are instant (just look up the cell). But the cube loses flexibility: you can only query along dimensions that were included when the cube was built. "What proportion of sales were items over $100?" can't be answered if price isn't a cube dimension. "Show me sales for a specific customer segment" fails if customer segment isn't a dimension. Rebuilding the cube with new dimensions is expensive. So most warehouses keep all raw fact data and use cubes only as a performance optimization for known, frequent queries -- not as a replacement for raw data access.
