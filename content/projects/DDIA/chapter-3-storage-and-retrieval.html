DDIA — Part I: Foundations of Data Systems

Chapter 3: Storage and Retrieval

A database needs to do two things: store data when you give it, and give it back when you ask. Understanding how it does this lets you choose the right engine for your workload.

The World's Simplest Database

Two bash functions: db_set key value appends to a file. db_get key greps for the latest value. Writes are O(1) (just append). Reads are O(n) (scan the whole file). This is the core insight: an append-only log has great write performance but terrible read performance. We need an index to speed up reads.

The Fundamental Trade-Off

An index is an additional structure derived from the primary data. It speeds up reads but slows down writes (every write must also update the index). This is a trade-off: well-chosen indexes speed up reads without slowing writes too much.

Hash Indexes (Bitcask)

Keep an in-memory hash map: key → byte offset in the data file. Every write appends to the file and updates the hash map. Reads: look up offset, seek to position, read value.

Compaction & Segment Merging

Break the log into segments. When a segment reaches a size limit, close it and start a new one. Then run compaction: throw away duplicate keys, keeping only the most recent value. Merge several segments into one. Old segments are deleted after merging completes.

  • Each segment has its own in-memory hash table
  • To find a key: check most recent segment first, then next-oldest, etc.
  • Merging keeps the number of segments small

Limitations of Hash Indexes

  • Hash table must fit in memory — doesn't work if you have very many keys
  • Range queries are not efficient (can't scan keys in order)

SSTables and LSM-Trees

Key improvement: require that key-value pairs in each segment are sorted by key. This format is called a Sorted String Table (SSTable).

Advantages over Hash Indexes

  • Efficient merging: mergesort-like — read side-by-side, output lowest key. Like merge step of mergesort.
  • Sparse in-memory index: don't need every key in memory. Know that key is between two index entries → scan that range on disk.
  • Range queries: scan keys in order.
  • Compression: group records into blocks, compress before writing to disk.
How an LSM-Tree Works
1. Write → Add to in-memory balanced tree (memtable) — e.g. red-black tree
↓
2. Flush → When memtable exceeds threshold (~few MB), write it to disk as an SSTable segment (already sorted!)
↓
3. Read → Check memtable first, then most recent on-disk segment, then next-oldest, etc.
↓
4. Compact → Background process merges and compacts segment files, discarding overwritten/deleted values

Crash Recovery

If the database crashes, the memtable (in-memory) is lost. Solution: keep a separate write-ahead log (WAL) on disk — every write is appended to this log first. Used only to restore the memtable after a crash, then discarded.

Bloom Filters

Problem: reading a key that doesn't exist requires checking memtable + all segments. Solution: a Bloom filter — a memory-efficient probabilistic data structure that tells you if a key definitely does not exist or might exist. Saves many unnecessary disk reads.

B-Trees

The most widely used indexing structure. Like SSTables, they keep key-value pairs sorted by key. But the design philosophy is completely different.

How B-Trees Work

  • Break the database into fixed-size pages (typically 4 KB), corresponding to the underlying disk hardware
  • Each page can be read or written as a unit; pages can reference other pages (like pointers on disk)
  • One page is the root; queries start here
  • Each page contains keys in sorted order, and references to child pages
  • Leaf pages contain the actual values (or references to where values are found)
  • The branching factor is typically several hundred

Branching factor ~500 → 4-level tree stores up to 256 TB. Most DBs fit in 3-4 levels.

B-Tree Updates & WAL

To update: find the leaf page, change the value, write the page back to disk (overwrite in place). To add: find the page, add key; if page full, split into two half-full pages. Parent updated to reference both.

Dangerous: overwriting pages can leave tree in corrupted state if crash mid-write (especially during page splits). Solution: write-ahead log (WAL / redo log). Every modification is written to the WAL before applying to the tree pages.

Data Warehousing & ETL

OLTP databases must be highly available and low-latency. Running expensive analytic queries on them harms performance. Solution: a separate data warehouse — a read-only copy optimized for analytics.

Star Schema (Dimensional Modeling)

Fact Table + Dimension Tables

  • Fact table: center of the star. Each row = an event (e.g., a sale). Can have 100+ columns. Often trillions of rows.
  • Dimension tables: who, what, where, when, how, why. E.g., dim_product, dim_store, dim_date, dim_customer.
  • Fact table columns are either attributes (price, quantity) or foreign keys to dimension tables.
  • Snowflake schema: dimensions further broken into subdimensions (more normalized, but star is simpler for analysts).

Column-Oriented Storage

Analytics queries typically touch only 4-5 columns out of 100+. Row-oriented storage loads entire rows even if you only need a few columns. Wasteful.

Row vs. Column Storage

Row-Oriented

Row 1: [date, product, store, qty, price, ...]
Row 2: [date, product, store, qty, price, ...]
Row 3: [date, product, store, qty, price, ...]
Must load ALL columns to read any.

Column-Oriented

date.col: [d1, d2, d3, d4, d5, ...]
product.col: [p1, p2, p3, p4, p5, ...]
qty.col: [q1, q2, q3, q4, q5, ...]
Only load columns you need!

Key invariant: the k-th entry in every column file belongs to the same row.

Column Compression

Bitmap encoding: n distinct values → n bitmaps, one bit per row. WHERE IN = bitwise OR. WHERE AND = bitwise AND. Sparse bitmaps use run-length encoding.

Vectorized Processing

Compressed column chunks fit in CPU L1 cache. Tight loops with no function calls. Bitwise ops on compressed data directly. SIMD instructions. Much faster than row-at-a-time processing.

Sort Order in Column Storage

Sort rows by a chosen column (e.g., date_key) — all column files sorted in the same order. First sort column compresses extremely well (long runs of same value → run-length encoding). C-Store/Vertica: store same data sorted in different orders on different replicas. Each query uses the best-sorted replica.

Writing to Column Storage

Update-in-place won't work (compressed columns). Solution: LSM-tree approach. Writes go to in-memory store first, then merged with column files on disk in bulk. Vertica does this. Queries combine column data on disk + recent writes in memory.

Materialized Views & Data Cubes

Materialized view: precomputed query results stored on disk (unlike virtual views which expand on-the-fly). Must be updated when underlying data changes → makes writes more expensive.

Data cube (OLAP cube): a grid of aggregates grouped by multiple dimensions. Fast for precomputed aggregates but loses flexibility — can't answer queries about dimensions not in the cube.

Chapter 3 — The Big Picture

OLTP storage: Two schools — (1) Log-structured (append-only, sequential writes, LSM-trees) vs (2) Update-in-place (B-trees, fixed pages, overwrite). Log-structured enables higher write throughput; B-trees have more predictable reads.

OLAP storage: Column-oriented storage, bitmap compression, vectorized processing. Star/snowflake schemas. Materialized views for precomputed aggregates.

The core insight: OLTP bottleneck is disk seeks; OLAP bottleneck is disk bandwidth. Different problems → different storage engines.

First-Principles Questions

1Why does an append-only log have O(1) writes but O(n) reads?▼
Writing just appends to the end of the file — always the same cost regardless of database size. But reading requires scanning the file from beginning to end to find the latest value for a key, because later writes with the same key override earlier ones. The more data you have, the more you scan. This is why we need indexes.
2What is the fundamental trade-off of indexes?▼
An index is an additional structure derived from the primary data. It speeds up reads but slows down writes, because every write must also update the index. The database doesn't index everything by default — the application developer must choose indexes using knowledge of typical query patterns, to get the right balance of read speedup vs. write overhead.
3How does an SSTable improve over a simple hash-indexed log?▼
SSTables require key-value pairs to be sorted by key within each segment. This enables: (1) efficient merging via mergesort-style algorithm, (2) a sparse in-memory index (you don't need every key — just scan between known offsets), (3) efficient range queries (scan keys in order), and (4) better compression (group sorted records into blocks).
4How does an LSM-tree maintain sorted order despite random writes?▼
Writes go to an in-memory balanced tree (a memtable, e.g. red-black tree) which maintains sorted order automatically. When the memtable exceeds a size threshold, it is written out to disk as a sorted SSTable segment. The sort happens in memory — writes to disk are always sequential. Background compaction merges sorted segments together.
5Why do B-trees need a write-ahead log (WAL)?▼
B-trees modify pages in place on disk. If the database crashes mid-write — especially during a page split that requires updating a parent page — the tree can be left in a corrupted state (orphan pages). The WAL is an append-only log where every modification is recorded before applying it to the tree pages. After a crash, the WAL is used to restore the B-tree to a consistent state.
6Why are LSM-trees generally better for write-heavy workloads?▼
LSM-trees turn random writes into sequential writes (append to memtable, then flush to disk in sorted order). Sequential writes are much faster than random writes on both hard drives and SSDs. B-trees must overwrite specific pages at random locations on disk. However, LSM-trees have write amplification from compaction — data is rewritten multiple times through compaction levels.
7What is the OLTP vs OLAP bottleneck difference, and why does it matter for storage?▼
OLTP bottleneck = disk seek time (finding a specific small record). OLAP bottleneck = disk bandwidth (scanning millions of rows). This is why OLTP uses B-trees/LSM-trees (index to avoid scanning) while OLAP uses column-oriented storage (minimize bytes read from disk by only loading needed columns, plus compression).
8Why is column-oriented storage so good for compression?▼
Values in a single column are often very repetitive (e.g., a country column with only ~200 distinct values across billions of rows). Bitmap encoding: n distinct values → n bitmaps with one bit per row. Sparse bitmaps are further compressed with run-length encoding. Sorted columns compress even better: the first sort key has long runs of the same value, compressible to a few kilobytes even with billions of rows.
9What is a Bloom filter and why is it essential for LSM-trees?▼
A Bloom filter is a memory-efficient probabilistic data structure. It can tell you if a key definitely does not exist or might exist in a set. For LSM-trees, looking up a nonexistent key would otherwise require checking the memtable and every SSTable segment on disk. A Bloom filter avoids these unnecessary disk reads by quickly ruling out segments that don't contain the key.
10Why does the star schema have a "fact table" at the center, and what goes in dimension tables?▼
The fact table captures individual events (e.g., each sale) because this allows maximum flexibility of analysis later. Each row has attributes (price, quantity) and foreign keys to dimension tables. Dimension tables represent the who, what, where, when, how, and why of each event (dim_product, dim_store, dim_date, dim_customer). The name "star" comes from the visual: fact table in the middle, dimension tables radiating outward like rays.
11How does Vertica solve the problem of writing to column-oriented storage?▼
Column-oriented storage with compression makes update-in-place impossible (inserting a row means rewriting all column files). Vertica uses the LSM-tree approach: writes first go to an in-memory store, where they are sorted. When enough writes accumulate, they are merged with the column files on disk in bulk. Queries combine both the on-disk column data and recent writes in memory.
