Part 2: Joins in MapReduce, Group-By, Skew, Dataflow Engines

Joins in MapReduce

Joins are common in analytics: enrich log data with user info, combine orders with product details. In MapReduce, joins require careful design — there's no SQL optimizer, and poor join strategies can create bottlenecks.

Sort-Merge Join (Reduce-Side Join)

The most straightforward join in MapReduce. Both datasets are processed by mappers, keyed on the join key, and sent to reducers. The reducer receives all records for a given join key and performs the join.

Join user_events (large) with users (large) on user_id
Mapper A (events)
Input: event records
Output: (user_id, {type:"event", ...})
Mapper B (users)
Input: user records
Output: (user_id, {type:"user", ...})
↓ Shuffle: both outputs grouped by user_id
Reducer receives all records for user_id=123:
[{type:"user", name:"Alice"}, {type:"event", page:"/home"}, {type:"event", page:"/about"}]
→ Output: {user: "Alice", events: ["/home", "/about"]}
Both datasets must be sorted and shuffled. Works even if both datasets are huge. No data stays in memory.

Broadcast Hash Join (Map-Side Join)

If one dataset is small enough to fit in memory, load it into a hash table and broadcast to all mappers. Each mapper does the join in-memory, no shuffle needed.

When to use:
One dataset is small (fits in mapper's RAM — say <1GB). Large dataset is partitioned across mappers.
Benefit: no shuffle phase at all. Much faster.
Limitation: small dataset must fit in memory of every mapper instance.
Example: large table of web logs joined with small table of ~1000 top products.

Partitioned Hash Join

Both datasets are partitioned by the same hash of the join key. Mapper for partition K only needs to load partition K of the smaller dataset into a hash table. No network shuffle needed at all — each mapper independently joins its partition.

Hive calls this a "bucketed map join." Both tables must be bucketed in the same way (same key, same number of buckets). Requires pre-partitioning when writing the data. The reward: very fast joins at query time.

GROUP BY and Aggregations

SQL GROUP BY maps naturally to MapReduce: the map phase emits (group-by key, value), and the reduce phase aggregates all values per key.

SELECT country, COUNT(*) FROM events GROUP BY country
Mapper: for each event → emit (country, 1)
Combiner (local reduce): on each mapper node, pre-aggregate counts before shuffle → fewer bytes shuffled
Shuffle: all (country, count) pairs grouped by country
Reducer: sum all counts → emit (country, total)
Combiners are a critical optimization: they apply the same function as the reducer but locally on each mapper. This can dramatically reduce the shuffle data volume. Only valid for commutative, associative functions (sum, count, max, min — but not average directly).

Handling Skew

Skew = some keys have far more records than others. In a join, a hot key (e.g., user_id=1 for an admin account) might cause one reducer to receive 1000× more data than others, creating a bottleneck.

Problem: reducer skew
Key "US" → 50% of all records → 1 reducer does 50% of work.
Other reducers idle while the "US" reducer is the bottleneck.
Job completes only when the slowest reducer finishes.
Solution: salting
Append random suffix to hot keys during map: "US_1", "US_2", ..., "US_10".
10 reducers share the work for "US".
Combine the 10 partial results in a second pass.
Pig's "skewed join" and Hive's "skewjoin" handle this automatically: they detect hot keys, salt them during the join, and merge the partial results. In Spark, broadcast joins are the alternative for skewed dimensions.

Dataflow Engines: Spark, Flink, Tez

MapReduce has a fundamental inefficiency: every intermediate result is written to HDFS (disk), then read back. A multi-stage job with 5 MapReduce steps does 10 HDFS writes/reads for intermediates. This is very slow.

The MapReduce Pipeline Problem

Step 1: Map → Reduce
↓ write to HDFS (disk)
Step 2: Map → Reduce
↓ write to HDFS (disk)
Step 3: Map → Reduce
↓ write to HDFS (disk)
Final output
Disk I/O dominates. Mostly sequential reads (efficient), but the round-trips to HDFS per stage add up to minutes or hours for complex pipelines.

Dataflow Engines (Spark model)

Spark, Tez, and Flink represent the "dataflow" approach: express the entire job as a directed acyclic graph (DAG) of operators. Intermediates are kept in memory where possible; only shuffle across network when needed.

Key improvements over MapReduce:
1. Operators are general (not just map/reduce). Any function: filter, join, sort, window.
2. Intermediates stay in memory (no HDFS write between stages)
3. Shuffle only when necessary (not after every stage)
4. Lazy evaluation: build the full DAG first, then optimize and execute
5. Result: 10–100× faster for iterative algorithms (ML) and multi-stage pipelines

Spark RDD and Lineage

RDD (Resilient Distributed Dataset)
Immutable collection of records distributed across nodes. Can be in memory or on disk. Operations on RDDs are lazy: they build a DAG (lineage graph) of transformations.
Fault tolerance via lineage
If a partition is lost, Spark re-computes it from the lineage (the chain of transformations). No need to write to disk — just re-run the relevant part of the DAG. Much faster than HDFS-based recovery.
Spark's key advantage for machine learning: iterative algorithms (gradient descent, PageRank) process the same dataset many times. With MapReduce, each iteration reads from and writes to HDFS. With Spark, the dataset stays in memory across iterations — 100× speedup is common.

Comparing MapReduce and Spark

AspectMapReduceSpark (dataflow)
Intermediate data Always written to HDFS (disk) Kept in memory (falls back to disk)
Execution model Always map→shuffle→reduce cycles General DAG of operators
Iterative algorithms Requires new job per iteration. Very slow. Same data stays in memory. 10–100× faster.
Fault tolerance Re-run from HDFS checkpoints Re-compute lost partitions via lineage
Cluster utilization Often underutilized (mapper idle during reduce) Better pipelining; operators run concurrently
