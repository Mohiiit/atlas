DDIA -- Chapter 10: Batch Processing (Part 2 of 2)

Outputs, Hadoop vs MPP, and Beyond MapReduce

We move beyond the mechanics of MapReduce to explore what batch jobs produce, how Hadoop compares with MPP databases, and why dataflow engines like Spark and Flink superseded the original MapReduce model. This part covers materialization problems, operator DAGs, graph processing with Pregel, and the evolution toward declarative high-level APIs.

"Raw data is better." -- The sushi principle. Hadoop embraces dumping data first and worrying about schema later, shifting the burden of interpretation to consumers rather than producers.

1. Output of Batch Workflows

A batch processing job reads input and produces output. But what form does that output take? The two most common outputs are search indexes and key-value stores (or other pre-computed databases).

Building Search Indexes

Google's original use of MapReduce was to build indexes for its search engine. Mappers partition documents, each reducer builds the index for its partition (term dictionary + postings lists), and the resulting index files are written to the distributed filesystem. These indexes are immutable once created, and querying by keyword is a read-only operation. When documents change, you either rerun the entire indexing workflow or build indexes incrementally (as Lucene does with segment merging).

Key-Value Stores as Batch Output

Another common output is a database for serving queries -- for example, a recommendation system that maps user IDs to suggested friends, or a classifier's output. Rather than writing directly to a production database from within the batch job (which causes performance and consistency problems), the best practice is to build a brand-new database file inside the batch job and bulk-load it into read-only serving nodes. Systems like Voldemort, Terrapin, ElephantDB, and HBase bulk loading support this pattern.

Why Not Write Directly to a Database?

  • Performance: Network request per record is orders of magnitude slower than batch throughput
  • Overload: Parallel mappers/reducers can overwhelm the database, degrading query performance
  • Consistency: Partial job output becomes externally visible, breaking MapReduce's clean all-or-nothing guarantee

2. Philosophy of Batch Process Outputs

The Unix philosophy carries directly into MapReduce: a program reads its input, writes its output, leaves the input unchanged, completely replaces any previous output, and produces no other side effects. This makes batch jobs easy to reason about and safe to rerun.

Diagram 2: Four Pillars of Batch Output Philosophy
Immutable Inputs
Input data is never modified. The same files can feed multiple jobs safely.
No Side Effects
Jobs produce only their designated output. No writes to external databases mid-job.
Easy Rollback
If code has a bug, roll back to previous code version and rerun. Old output stays intact in a separate directory.
Automatic Retry
Failed tasks are retried safely because inputs are immutable and partial output is discarded.

Human Fault Tolerance

The ability to recover from buggy code (not just hardware faults) is sometimes called human fault tolerance. Because batch outputs are immutable and replaceable, you can always rerun with corrected code. This minimizes irreversibility and enables faster feature development -- a principle aligned with Agile software development.

Separation of Logic and Wiring

Like Unix tools, MapReduce jobs separate what the job does from where its input comes from and where output goes. One team writes the processing logic; other teams decide when and where to run it. This separation of concerns enables code reuse and loose coupling -- jobs do not need to know who produces their input or consumes their output.

3. Comparing Hadoop to Distributed Databases

When MapReduce was published, the algorithms it used (parallel joins, partitioned aggregation) had already been implemented in Massively Parallel Processing (MPP) databases for over a decade (Gamma, Teradata, Tandem NonStop SQL). The key difference: MPP databases focus on parallel SQL analytics, while Hadoop + HDFS provides a general-purpose distributed operating system for running arbitrary programs.

The "Sushi Principle" and Data Lakes

Hadoop popularized the idea of dumping raw data into HDFS first and figuring out how to process it later. This schema-on-read approach (sometimes called a "data lake") shifts the burden of interpretation from producer to consumer. Multiple teams can create different views of the same raw data. The careful schema design required by MPP databases slows down data collection, while Hadoop's approach speeds it up.

Hadoop is commonly used for ETL: raw transaction data is dumped into HDFS, MapReduce jobs clean and transform it into relational form, and the result is imported into an MPP data warehouse for analytics. Data modeling still happens, but it is decoupled from data collection.

Designing for Frequent Faults

MapReduce was designed at Google where batch jobs run at low priority on shared clusters. Higher-priority production services can preempt batch tasks at any time. At Google, a MapReduce task running for one hour has an approximately 5% risk of being preempted -- more than 10x the rate of actual hardware failures. For a job with 100 tasks each running 10 minutes, there is a greater than 50% chance at least one task gets terminated.

This is why MapReduce eagerly writes to disk and recovers at task granularity -- not because hardware is unreliable, but because the freedom to arbitrarily terminate processes enables better resource utilization in a shared computing cluster. In open-source schedulers (YARN, Mesos), preemption is less common, which is one reason the MapReduce design has been questioned and superseded.

4. Materialization of Intermediate State

In complex workflows (e.g., recommendation systems with 50-100 MapReduce jobs), the output of one job becomes the input to the next. When this output is only used by one downstream job, it is simply intermediate state. Writing this intermediate state to HDFS files is called materialization -- eagerly computing and storing a result rather than streaming it on demand.

Three Problems with Full Materialization

  • Straggler problem: A MapReduce job can only start when all tasks in the preceding job have completed. A few slow "straggler" tasks delay the entire workflow, unlike Unix pipes where processes run concurrently.
  • Redundant mappers: Mappers in the next job often just read back the file a reducer just wrote, to re-partition and re-sort it. This mapper could be folded into the previous reducer -- but MapReduce forces alternating map/reduce stages.
  • Over-replication: Intermediate state files on HDFS are replicated across several nodes (typically 3x), which is overkill for temporary data that will be discarded once the next job finishes.

5. Dataflow Engines: Spark, Tez, and Flink

To fix the problems with MapReduce's materialization, new execution engines were developed. The most well-known are Spark, Tez, and Flink. They handle an entire workflow as one job rather than breaking it into independent sub-jobs, and they model the flow of data through processing stages as a directed acyclic graph (DAG) of operators.

Operators Instead of Map/Reduce

Unlike MapReduce's strict alternation of map and reduce functions, dataflow engines use flexible operators that can be assembled in any order. Three connection strategies between operators:

Repartition + Sort

Like MapReduce shuffle. Enables sort-merge joins and grouping. Used when order matters.

Repartition Only

Partition by key but skip sorting. Saves effort for partitioned hash joins where order is irrelevant.

Broadcast

Send same output to all partitions of the next operator. Used for broadcast hash joins with small inputs.

Advantages Over MapReduce

Six Key Improvements

  • Sort only when needed: Expensive sorting is performed only where required, not by default between every stage.
  • No redundant mappers: Mapper work is incorporated into the preceding operator when partitioning does not change.
  • Locality optimization: All data dependencies are declared, so the scheduler can place tasks that produce and consume data on the same machine.
  • In-memory intermediate state: Data between operators stays in memory or local disk -- no HDFS replication overhead.
  • Pipelined execution: Operators start as soon as input is ready, no need to wait for the entire preceding stage.
  • JVM reuse: Existing JVM processes run new operators, reducing startup overhead (MapReduce launches a new JVM per task).

6. Fault Tolerance in Dataflow Engines

MapReduce's fault tolerance is simple: intermediate state is on HDFS, so a failed task re-reads its input. Dataflow engines keep intermediate state in memory, so they need a different approach: recomputation.

Spark tracks the lineage of each Resilient Distributed Dataset (RDD) -- the chain of transformations that produced it. If a partition is lost, Spark recomputes it by replaying transformations from the nearest available ancestor. Flink takes a different approach: it periodically checkpoints operator state, allowing it to resume from the last checkpoint.

Determinism Matters

When recomputing data, operators must be deterministic -- given the same input, they must produce the same output. If recomputed data differs from the original, downstream operators face contradictions. Sources of accidental nondeterminism include: hash table iteration order, random number generators, system clock reads, and external data lookups. Fix these by using fixed seeds and avoiding external dependencies.

If intermediate data is much smaller than the source or if computation is very CPU-intensive, it may be cheaper to materialize it (like MapReduce) rather than recompute. This is a trade-off, not a universal rule.

7. Graphs and Iterative Processing

Graph processing in a batch context means performing offline analysis on an entire graph -- for example, PageRank (estimating web page popularity) or recommendation engines. Many graph algorithms work by traversing edges one at a time, propagating information between adjacent vertices, and repeating until convergence.

Plain MapReduce cannot express "repeat until done" -- it performs only a single pass. An external scheduler can loop over MapReduce jobs, but this is inefficient because MapReduce reads the entire input and produces a completely new output each iteration, even if only a small part of the graph changed.

The Pregel / BSP Model

The Bulk Synchronous Parallel (BSP) model, popularized by Google's Pregel paper, optimizes graph batch processing. It is implemented by Apache Giraph, Spark GraphX, and Flink Gelly. The core idea: vertices send messages to each other (typically along edges), and computation proceeds in synchronized rounds called supersteps.

Barrier synchronization: All messages from superstep N must be delivered before superstep N+1 begins. Like the actor model, but with fixed round-based timing guarantees.

Key Properties of Pregel

  • Vertices communicate only by message passing (not direct queries)
  • Vertices remember state across iterations (unlike MapReduce which re-reads everything)
  • If no messages are sent to a part of the graph, no work is done there
  • Messages are guaranteed to be processed exactly once at destination

Pregel Fault Tolerance

  • Periodically checkpoints all vertex state to durable storage at end of iteration
  • On failure: roll back entire graph to last checkpoint and restart
  • With deterministic algorithms + message logging: can selectively recover only the lost partition

Practical Consideration: Single Machine May Win

Distributed graph algorithms have high cross-machine communication overhead -- often the messages are bigger than the graph itself. If your graph fits in memory on a single machine, a single-threaded algorithm will likely outperform a distributed batch process. Even for graphs bigger than memory, single-machine frameworks like GraphChi can process graphs that fit on disk. Use Pregel-style distributed processing only when the graph is too big for a single machine.

8. High-Level APIs and Languages

As batch processing infrastructure matured (petabytes on 10,000+ machine clusters), attention shifted from operational challenges to improving the programming model. Writing raw MapReduce jobs by hand is laborious -- you need to implement join algorithms from scratch.

Diagram 9: Evolution of Batch Processing APIs
2004 -- Raw MapReduce
Google publishes the MapReduce paper. Engineers write Java map/reduce functions by hand. Implementing joins requires writing algorithms from scratch. Powerful but extremely laborious.
2007-2009 -- Pig, Hive, Cascading
Higher-level abstractions on top of MapReduce. Hive provides SQL-like interface (HiveQL). Pig offers a dataflow scripting language. Cascading provides a Java API. All compile down to MapReduce jobs but are much easier to use.
2010-2012 -- Tez and Dataflow Engines
Apache Tez emerges as an alternative execution engine. Pig and Hive can switch from MapReduce to Tez with a configuration change -- no code rewrites. Significant performance improvement by eliminating unnecessary materialization.
2012-2014 -- Spark, Flink, High-Level DataFlow APIs
Spark introduces RDDs and a rich functional API (map, filter, reduceByKey, join). Flink offers pipelined execution with operator state checkpointing. Both include their own schedulers, network layers, and high-level APIs inspired by FlumeJava.
2014+ -- Declarative / SQL Convergence
Spark SQL, Hive LLAP, Impala bring cost-based query optimizers to batch processing. Vectorized execution, JVM bytecode generation (Spark), LLVM native code generation (Impala). Batch frameworks begin to look like MPP databases in performance while retaining the flexibility to run arbitrary code.

The Move Toward Declarative Queries

By specifying joins as relational operators rather than hand-coded algorithms, the framework's query optimizer can automatically choose the best join algorithm (sort-merge, broadcast hash, partitioned hash) and even reorder joins to minimize intermediate state. Hive, Spark, and Flink all have cost-based optimizers for this.

Declarative Benefits

  • Automatic join selection: The optimizer picks the best algorithm based on input properties
  • Column-oriented storage: Declarative filters allow reading only required columns from disk (Parquet, ORC)
  • Vectorized execution: Tight inner loops friendly to CPU caches; avoids function call overhead
  • Code generation: Spark generates JVM bytecode; Impala uses LLVM for native inner loop code
  • Interactive exploration: High-level APIs allow writing analysis code incrementally in a shell

9. Chapter Summary

This chapter explored batch processing from Unix tools through MapReduce to modern dataflow engines. The design principles that emerged:

Core Design Principles

  • Inputs are immutable: outputs become the input to another (as yet unknown) program
  • Complex problems are solved by composing small tools that "do one thing well"
  • The uniform interface is a distributed filesystem (HDFS), analogous to Unix files and pipes
  • Dataflow engines add pipe-like transport to avoid materializing intermediate state

Partitioning

Mappers partition by input file blocks. Output is repartitioned, sorted, and merged into reducer partitions to bring related data together. Post-MapReduce engines avoid sorting unless required but take a similar approach to partitioning.

Fault Tolerance

MapReduce writes frequently to disk (easy recovery, slow execution). Dataflow engines keep more in memory (faster execution, must recompute on failure). Deterministic operators reduce the amount of recomputation needed.

Key Takeaways

  • Batch processing jobs are stateless callbacks with no externally visible side effects -- the framework hides distributed systems complexity
  • The framework guarantees output is the same as if no faults occurred, providing much stronger semantics than online services
  • Batch input is bounded (known, fixed size), so jobs eventually complete. The next chapter covers stream processing where input is unbounded and jobs never finish.

Test Your Understanding

1Why is it better to build database files inside a batch job and bulk-load them, rather than writing directly to a production database from mappers/reducers? ▾

Three reasons: (1) Performance -- making a network request per record is orders of magnitude slower than batch throughput. (2) Overload -- parallel mappers/reducers can overwhelm the database, degrading its query performance and causing operational problems. (3) Consistency -- writing to an external database produces externally visible side effects, breaking MapReduce's all-or-nothing guarantee. Partial job output becomes visible, and you must handle speculative execution and task retries. Building immutable files and atomically switching is cleaner and safer.

2What are the three main problems with MapReduce's full materialization of intermediate state, and how do dataflow engines solve them? ▾

(1) Straggler blocking: A job cannot start until all tasks in the preceding job finish. Dataflow engines pipeline execution -- operators start as soon as input is available. (2) Redundant mappers: The next job's mapper just re-reads and re-partitions what the previous reducer wrote. Dataflow engines fold this work into the preceding operator. (3) Over-replication: HDFS replicates intermediate files 3x across nodes, which is overkill for temporary data. Dataflow engines use in-memory buffers or local disk instead.

3How does Spark's RDD lineage provide fault tolerance without writing intermediate state to HDFS? ▾

Spark tracks the lineage (ancestry) of each RDD -- which input partitions it was derived from and which transformations were applied. If a machine fails and an RDD partition is lost, Spark recomputes it by replaying the transformation chain from the nearest available ancestor (a prior intermediate stage, or the original HDFS input). This works because transformations are deterministic: given the same input, they produce the same output. For CPU-intensive computations or when intermediate data is much smaller than the source, it may be cheaper to materialize (checkpoint) the intermediate data rather than recompute it.

4In the Pregel/BSP model, how do vertices communicate, and why is this different from both MapReduce and the actor model? ▾

In Pregel, vertices communicate exclusively through message passing along graph edges in synchronized rounds (supersteps). Unlike MapReduce, vertices remember their state across iterations, so only new incoming messages need processing -- if no messages reach a part of the graph, no work is done there. Unlike the actor model, Pregel has a fixed round-based timing guarantee: all messages sent in superstep N are delivered in superstep N+1 (actors have no such guarantee). Messages are processed exactly once at their destination. Fault tolerance is achieved by periodically checkpointing all vertex state to durable storage.

5Why was MapReduce designed to tolerate frequent task termination, and is this design equally relevant in open-source cluster schedulers? ▾

MapReduce was designed at Google where batch jobs run at low priority on shared clusters with production services. Higher-priority processes can preempt batch tasks at any time to reclaim resources. At Google, a task running for one hour has approximately a 5% chance of being preempted -- 10x more common than hardware failures. This "picking up scraps under the table" approach enables better resource utilization through overcommitting. However, in open-source schedulers (YARN, Mesos, Kubernetes), general priority preemption is less common, so MapReduce's eagerness to write to disk and its per-task recovery overhead are less justified -- which is one reason dataflow engines with lighter-weight fault tolerance have become preferred.

6How are batch processing frameworks and MPP databases converging, and what does each side gain from the other? ▾

Batch frameworks (Spark, Flink) are adopting declarative features from MPP databases: cost-based query optimizers that automatically choose join algorithms, column-oriented storage integration (Parquet/ORC), vectorized execution for CPU-cache-friendly inner loops, and code generation (Spark generates JVM bytecode, Impala uses LLVM). Meanwhile, MPP databases are becoming more programmable, with better UDF support and flexible data format handling. The result is convergence toward unified systems that combine SQL's declarative power and optimization with the flexibility to run arbitrary code on any data format. Both are ultimately "systems for storing and processing data."
