Serializability

Serializable isolation is the strongest isolation level. It guarantees that concurrent transactions have the same effect as if they ran one at a time in some serial order — even if they actually ran concurrently.

Three main approaches to implementing serializability:

Actual Serial
Execution
Literally run one transaction at a time. No concurrency problems possible.
Two-Phase
Locking (2PL)
Acquire locks before reads/writes. Hold them until commit. Blocks concurrent access.
Serializable
Snapshot Isolation
Optimistic approach. Allow concurrent runs, detect conflicts, abort if needed.

Actual Serial Execution

The simplest approach: only one transaction runs at a time, on a single CPU core. This sounds like a step backward, but it became viable around 2007 when:

  • RAM got cheap enough to keep the entire working dataset in memory
  • OLTP transactions are often short and touch few records
  • Many real-world applications have low concurrency requirements per shard
VoltDB, Redis (single-threaded), Datomic, and FoundationDB use actual serial execution. Redis processes all commands in a single thread — yet handles hundreds of thousands of ops/sec because each operation is O(1) or O(log n) and done in memory.

Stored Procedures: The Key Enabler

With serial execution, all transactions must complete quickly. But network round-trips between application and database are slow. Solution: stored procedures — send the entire transaction logic to the database in one shot.

Interactive transaction (slow)
1. App → DB: READ seat availability
← Wait for network (~10ms)
2. User selects seat (seconds!)
3. App → DB: LOCK seat
← Wait
4. App → DB: INSERT booking
← Wait
5. App → DB: COMMIT
Database blocked for the entire user think time!
Stored procedure (fast)
1. User selects seat (application-side)
2. App → DB: EXECUTE book_seat(user, seat)
← Wait (~1ms)
3. Done. Database ran the whole thing atomically in microseconds with no external waits.

Limitations of Actual Serial Execution

  • Single core bottleneck: Throughput limited to one core. Solution: partition data across cores/nodes, route transactions to the right partition.
  • Cross-partition transactions: If a transaction touches multiple partitions, each partition must be locked in order. Expensive and rare in well-designed systems.
  • Stored procedure language: Different databases use different languages (T-SQL, PL/pgSQL, Lua for Redis). Less tooling than application code.

Two-Phase Locking (2PL)

Two-phase locking has been the standard serializable isolation approach in SQL databases since the 1970s. The idea:

The two phases
Phase 1: Growing
Acquire locks as needed.
Never release a lock in this phase.
Phase 2: Shrinking
Release all locks at once at COMMIT/ROLLBACK.
Never acquire new locks in this phase.
Hold locks until commit — this is what makes 2PL different from simple locking

Lock Types in 2PL

Action Lock Required Compatible with Shared? Compatible with Exclusive?
Read object Shared (read) lock Yes — many readers OK No — wait for writer to commit
Write object Exclusive (write) lock No — wait for all readers No — wait for other writer
Read then write (upgrade) Upgrade shared → exclusive Must wait for all other shared locks to release first
Writers block readers AND other writers. Readers block writers (but not other readers). This is very different from MVCC, where "readers never block writers and writers never block readers." 2PL has worse read performance than MVCC.

Deadlocks are inevitable with 2PL. The database detects them (by scanning the wait-for graph) and aborts one transaction. The application must retry.

Predicate Locks and Index-Range Locks

To prevent phantoms, 2PL needs to lock not just existing rows but also the potential rows that a query would return — a predicate lock.

-- Predicate lock: lock all rows matching this condition
SELECT * FROM bookings
WHERE room_id = 123 AND start_time >= '2:00pm' AND end_time <= '3:00pm'
FOR UPDATE;

Predicate locks are expensive to check (every insert must be checked against all active predicate locks). In practice, databases use index-range locks (also called gap locks) — a coarser lock covering an index range. Slightly more locking than needed, but much faster.

Serializable Snapshot Isolation (SSI)

SSI is an optimistic concurrency control mechanism. It provides full serializability with performance closer to snapshot isolation. Instead of blocking transactions, it detects conflicts and aborts when needed.

The core insight from M. J. Cahill's PhD thesis (2008): serial anomalies arise from cycles in the dependency graph between transactions.

How SSI Tracks Dependencies

The database tracks two types of anti-dependencies:

  1. Read by committed transaction, then written by concurrent transaction: If Tx2 writes data that Tx1 already read (from a stale snapshot), Tx1 may have made a decision based on incorrect data → abort one
  2. Premises become false: If transaction reads a set of rows based on a condition, and a concurrent write changes the result of that condition
2PL (pessimistic)
Assumes conflicts will happen.
Blocks transactions preemptively.
High latency, deadlocks possible.
SSI (optimistic)
Assumes conflicts are rare.
Runs transactions freely.
Aborts after detecting conflict.
SSI wins when conflicts are rare. 2PL wins when conflicts are frequent (fewer wasted aborts).
PostgreSQL 9.1+ implements SSI as its "Serializable" isolation level. It offers full serializability with performance close to Snapshot Isolation for low-conflict workloads. This is one of the most practically important advances in database concurrency in decades.

Review Q&A

1
Why is "Consistency" in ACID considered the odd one out?
›
Atomicity, Isolation, and Durability are database properties — the database enforces them mechanically. "Consistency" is an application property — it means the application maintains its own invariants (e.g., account balances are never negative, every order has a valid user). The database can help with consistency (constraints, foreign keys, UNIQUE), but it can't guarantee all application-level invariants. The "C" was added to make the acronym spell out "ACID" — it was retrofitted after the other three were already named.
2
How does PostgreSQL's MVCC prevent dirty reads without blocking reads?
›
When a transaction updates a row, PostgreSQL keeps both the old version and the new version. Each row version is tagged with the transaction ID that created it (xmin) and optionally the transaction ID that deleted/updated it (xmax). Readers look at xmin to determine if the creating transaction was committed at the time of their snapshot. If Tx1 has updated a row but not committed, readers still see the old version (where xmax is Tx1's ID, but Tx1 hasn't committed). Since readers access old versions directly from storage (not from a lock), they never need to block. Only writers acquiring exclusive locks on specific rows block each other.
3
What is the difference between a lost update and write skew?
›
Lost update: Both transactions write to the same object. Tx1 reads value, Tx2 reads same value, Tx1 writes new value, Tx2 overwrites with its own new value (based on the old read). Tx1's write is "lost." Example: two users both incrementing a counter. Write skew: Transactions read some objects, then write to different objects. But the combination of their writes violates an invariant that each individual transaction checked was true. Example: two doctors both check "count > 1 on-call doctors?" → both see true → both go off-call → 0 doctors remain. Each transaction individually is valid; together they violate the invariant.
4
What is a phantom read, and why is it harder to prevent than non-repeatable reads?
›
A non-repeatable read happens when you read the same row twice and get different values (because another transaction updated it between your reads). You can prevent this by locking the specific rows you read. A phantom read happens when you execute a query with a predicate (WHERE condition) and get different result sets on two executions — because another transaction inserted or deleted rows matching that predicate. You can't prevent phantoms by locking existing rows, because the phantom row didn't exist yet when you first checked. You'd need to lock the entire key range, which is expensive — or use predicate locks / index-range locks / SSI.
5
How does Two-Phase Locking prevent phantoms compared to simple row locking?
›
Simple row locking locks specific row IDs. If a transaction reads "all rooms booked at 2pm" and gets an empty set, there are no row IDs to lock — so nothing prevents another transaction from inserting a new booking at 2pm. 2PL with predicate locks locks not specific rows but all rows matching a condition. Any new insert that would match an active predicate lock is blocked. In practice, predicate locks are expensive (every insert must be checked against all predicate locks), so databases use index-range locks (gap locks) — they lock the index range that encompasses the predicate, preventing any insert into that range.
6
Why did actual serial execution become viable around 2007?
›
Two factors converged: (1) RAM became cheap enough to hold entire working datasets in memory — no more slow disk I/O between operations. An in-memory transaction can complete in microseconds rather than milliseconds, so a single CPU core can handle many transactions per second. (2) OLTP transactions are typically short — most user-facing operations touch a small number of rows and complete in under a millisecond. The bottleneck shifted from CPU to I/O, and with in-memory databases, I/O is no longer the bottleneck. VoltDB, H-Store, and similar systems demonstrated that serial execution of stored procedures could outperform concurrent 2PL systems.
7
What is the key difference between 2PL and SSI in their approach to concurrency?
›
2PL is pessimistic: It assumes conflicts will happen and prevents them upfront by acquiring locks. Any transaction that might conflict with another must wait. This is safe but causes high latency and deadlocks. SSI is optimistic: It assumes conflicts are rare and lets transactions proceed without blocking. It tracks what each transaction has read and what has been written. At commit time, it checks if any transaction's reads have been invalidated by concurrent writes. If a conflict is detected, one transaction is aborted. SSI is better when conflict rates are low (most real OLTP workloads); 2PL is better when conflicts are frequent (aborts in SSI would waste work).
8
Why does SELECT FOR UPDATE prevent lost updates in read-modify-write cycles?
›
SELECT FOR UPDATE acquires an exclusive lock on the selected rows immediately at read time. If Transaction 2 tries to SELECT FOR UPDATE the same rows while Transaction 1 holds the lock, Tx2 must wait. Once Tx1 commits, Tx2 re-reads the rows (now seeing Tx1's updated value) and computes its update from that. This serializes the read-modify-write cycle: Tx1 completes its cycle fully before Tx2 starts. The "FOR UPDATE" part is crucial — a regular SELECT doesn't acquire any lock in PostgreSQL and won't prevent the lost update.
9
What does "VACUUM" have to do with MVCC in PostgreSQL?
›
MVCC keeps old versions of rows to serve active snapshots. A row that was updated or deleted isn't immediately removed — older transactions might still be reading the old version. VACUUM is PostgreSQL's garbage collection process: it scans tables for row versions where xmax is committed and older than the oldest active transaction snapshot. Such row versions are no longer needed and can be reclaimed. Without VACUUM, the table would grow indefinitely (called "table bloat"). autovacuum runs in the background automatically. If VACUUM falls behind (e.g., a very long-running transaction holds an old snapshot), old row versions accumulate and performance degrades.
10
How does the "hospital on-call" write skew problem demonstrate that snapshot isolation alone is insufficient?
›
Snapshot isolation prevents dirty reads and read skew by giving each transaction a consistent view of data. But it doesn't prevent write skew because: each transaction reads a consistent snapshot (2 doctors on call), makes a valid decision based on that snapshot (safe to go off-call), and writes to a different row (their own record). Neither transaction's read was "dirty" — both read committed data from their snapshot. The problem is that the two transactions' combined decisions violated an invariant, but each individual decision looked valid in isolation. Preventing this requires either: explicit locking (SELECT FOR UPDATE on the on-call rows), materializing the conflict (create a lock target), or serializable isolation (SSI detects the dependency cycle and aborts one transaction).
