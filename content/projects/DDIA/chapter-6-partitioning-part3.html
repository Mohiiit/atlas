Request Routing

When a client wants to read or write a key, which node do they talk to? The partition-to-node mapping changes as partitions rebalance. Three approaches to routing:

1. Client-side routing
Client knows partition mapping and contacts the right node directly. Requires clients to track routing info.
Used by: Cassandra, Riak (gossip protocol)
2. Routing tier
All requests go to a load balancer / routing layer that knows partition ownership and forwards to correct node.
Used by: CouchBase, Elasticsearch
3. Contact any node
Client contacts any node. If that node owns the partition, handle it. Otherwise, forward to the correct node and proxy the response.
Used by: Cassandra, Riak coordinator

Service Discovery: How to Stay Up-to-Date

All three approaches require some component to know the partition-to-node mapping. How is this kept current as partitions move?

ZooKeeper
Authoritative registry of partition assignment. Nodes register themselves. Clients / routing tier subscribe to changes. Notified when partitions move. Used by: HBase, Kafka, SolrCloud.
Gossip protocol
Nodes gossip cluster state to each other. No single coordinator. Eventually consistent view. Request goes to any node; it forwards. Used by: Cassandra, Riak.
Config server
Dedicated config nodes track routing table. mongos router queries config servers. Used by: MongoDB.
Consistent hashing (the ring model used by Cassandra) allows nodes to be added/removed with minimal remapping. Each node owns a range on the ring; adding a node only steals half of one neighbor's range. Virtual nodes (vnodes) spread each physical node across multiple ring positions for better load balance.

Parallel Query Execution

Massive Parallel Processing (MPP) databases for analytics (Teradata, Redshift, BigQuery) break complex SQL queries into stages and execute them in parallel across all partitions simultaneously.

SQL query: SELECT product, SUM(revenue) FROM sales GROUP BY product
Query optimizer decomposes query into stages
↓
Node 1
partial SUM
from shard 1
Node 2
partial SUM
from shard 2
Node 3
partial SUM
from shard 3
Node 4
partial SUM
from shard 4
↓
Coordinator aggregates partial results → final answer
A query that takes 10 minutes on one node takes ~10 seconds on 60 nodes. Data warehouses achieve petabyte-scale analytics this way.

Review Q&A

1
Why is hash mod N a bad partitioning strategy?
›
With hash(key) % N, the partition number depends on N — the total number of nodes. When you add node N+1, almost every key remaps to a different partition: keys where hash(key) % N ≠ hash(key) % (N+1) must move, which is approximately N/(N+1) of all keys. On a 10-node cluster, adding one node triggers 91% of all data to migrate. This makes rebalancing prohibitively expensive. The correct approach is consistent hashing or fixed-partition assignment where adding a node only moves the partitions assigned to it.
2
What is the scatter-gather problem with local secondary indexes?
›
With document-partitioned (local) secondary indexes, each partition maintains its own index of its own data. A query on the indexed field (e.g., "all red cars") can't route to a single partition — it doesn't know which partitions have red cars. It must broadcast to all partitions, each returns its local results, and the coordinator merges the responses. This "scatter-gather" operation is expensive: latency is determined by the slowest partition, and all partitions incur load. For N partitions, every secondary index query does N parallel reads. Global (term-partitioned) indexes avoid this but complicate writes.
3
How does Cassandra's compound partition key combine hash and range partitioning benefits?
›
Cassandra's primary key has two parts: (1) the partition key — hashed to determine which partition the data lives on, ensuring even distribution; (2) the clustering columns — sorted within the partition, enabling range queries within a specific partition. Example: PRIMARY KEY (user_id, timestamp). All data for user_id is on the same partition (hash-distributed by user_id), and within that partition, rows are sorted by timestamp, enabling efficient "all events for user X between time A and B" queries. The tradeoff: range queries across different users (different partition keys) still require scatter-gather.
4
What is a virtual node (vnode) and why does Cassandra use them?
›
Instead of each physical node owning one contiguous range on the consistent hash ring, a physical node owns many small non-contiguous ranges (virtual nodes). For example, a 3-node cluster with 256 vnodes: each physical node might own 85 vnodes scattered around the ring. Benefits: (1) When a node is added, it takes vnodes from every existing node — the new load is spread across the entire cluster, not just the two neighbors; (2) Heterogeneous hardware: more powerful nodes get more vnodes; (3) A failed node's load distributes evenly across all remaining nodes, not just its two neighbors. The trade-off: more metadata to track and slightly more complex coordination.
5
How does ZooKeeper help with routing in systems like HBase and Kafka?
›
ZooKeeper maintains an authoritative, strongly-consistent directory of cluster state. For HBase: it tracks which RegionServer owns which region (partition). For Kafka: it tracks which broker is leader for which partition. When a partition moves (rebalancing) or a node fails (failover), the new ownership is written to ZooKeeper atomically. Clients and routing components subscribe to ZooKeeper change notifications, so they learn about the new mapping immediately. This provides a centralized source of truth for routing decisions. The trade-off: ZooKeeper itself is a bottleneck and single point of failure (mitigated by its own replication). Kafka has been moving away from ZooKeeper (KRaft protocol).
