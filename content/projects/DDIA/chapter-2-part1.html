1. Data Models Are Layers of Abstraction

Data models are probably the most important part of software development — they affect not just how the software is written, but how we think about the problem. Every application is built on top of a data model, which is itself built on another data model. Each layer hides the complexity of the layer below it via a clean interface.

Each layer provides a clean abstraction to the one above. The choice of data model shapes the entire application design — it determines what queries are easy, what is awkward, and what is impossible.

2. The Relational Model

Edgar Codd proposed the relational model in 1970. Data is organized into relations (tables), each is an unordered collection of tuples (rows). The key insight: the query optimizer decides how to execute the query — the application developer only describes what data they want, not how to retrieve it. This separation allowed databases to improve performance without changing application code.

The Rise of NoSQL

By the 2010s, relational databases had dominated for ~40 years. Then came the "NoSQL" movement — not really a single technology, but a loose collection of motivations for trying something different:

Why NoSQL Emerged (Four Driving Forces)

  • Scale: need for very high write throughput or very large datasets that relational databases struggled with. Companies like Amazon, Google, Facebook built proprietary systems (Dynamo, Bigtable, Cassandra)
  • Open source preference: commercial relational DBs (Oracle) were expensive; engineers wanted FOSS alternatives
  • Specialized queries: some use cases (graph traversal, geospatial, full-text) are awkward in the relational model
  • Restrictive schema: the rigid schema of relational databases felt too constraining for rapidly evolving data models

The name "NoSQL" was a Twitter hashtag for a meetup in 2009. It was never a coherent movement. The term is now commonly reinterpreted as "Not Only SQL" to acknowledge that polyglot persistence (using multiple databases for different purposes) is the reality.

The Object-Relational Impedance Mismatch

Most application development today uses object-oriented programming. If data is stored in relational tables, an impedance mismatch exists between the object model and the relational model. The application needs to translate between the two — a translation layer that generates boilerplate code that feels awkward to write and maintain.

Diagram 3 — Impedance Mismatch: Object vs Table

  id: number
  name: string
  contactInfo: ContactInfo   // nested object
  positions: Position[]      // one-to-many
  education: Education[]     // one-to-many
  skills: string[]           // multi-valued
}

class Position {
  title: string
  company: Company           // reference
  startYear: number
  endYear: number | null
}

Natural to write and reason about. Objects nest. Lists are direct members.

Relational Model (in DB)

TABLE users
  id, name, city, country
TABLE contact_info
  user_id (FK), type, value
TABLE positions
  user_id (FK), title,
  company_id (FK), start, end
TABLE skills
  user_id (FK), skill_name

Spread across 4+ tables. Need JOINs to reassemble. ORM generates boilerplate.

The ORM Gap

3. The Document Model

JSON representation of a LinkedIn-style profile is more natural than the multi-table relational schema — the one-to-many relationships between User and their positions, education history, and contact information are represented as a single document with nested structure. This reduces the impedance mismatch and the need for joins.

All data in one place — no JOIN needed for profile page Schema flexible — add/remove fields without migration Hard to do: "find all users who worked at Microsoft"

The Key Trade-off: IDs vs Human-Readable Strings

Notice region_id: "us:91" and industry_id: 131 rather than storing the string "Greater Seattle Area" and "Computer Software" directly. Why? Because using IDs and normalizing removes duplication. If the region name needs to change (or be translated), it changes in one place — the lookup table — not across every user document.

The document model handles one-to-many (user → positions) well. But it doesn't naturally handle many-to-one (many users in the same region → one region name) or many-to-many relationships. These require either denormalization (with update anomalies) or joins (which most document databases don't support well).

Data Gets More Interconnected Over Time

Even if you start with data that fits naturally into a document, over time features tend to add interconnections. A LinkedIn profile starts as isolated document. Then: users can recommend each other (many-to-many). Schools and organizations become entities shared across profiles. User mentions in a feed create connections across documents. What started as a document becomes a graph.

4. History: How Relational Won Over Hierarchical and Network Models

The problem of representing many-to-many relationships in a database is not new. IBM's IMS (1968) used a hierarchical model — a tree structure. It worked well for one-to-many but struggled with many-to-many (data duplication or denormalization). Two solutions were proposed:

Diagram 7 — Three Solutions to Many-to-Many: Historical Battle

CODASYL Model (Network Model) — 1969

Records linked by pointers. Many-to-many supported via access paths — sequences of pointers to navigate from one record to another. Querying required writing code that traversed the pointer graph step by step. If the access paths changed, all queries had to be rewritten. Powerful but extremely rigid and hard to maintain.

Supports many-to-many Application must track access paths Schema changes break all queries

Relational Model — 1970 (Codd)

Relations (tables) of tuples (rows). No nested structure, no pointers. Any row in any table can be referenced by value (foreign key). The query optimizer chooses the access path automatically — the developer writes what, not how. Adding a new index doesn't require changing queries. The key innovation: data independence — application code is independent of storage layout.

Declarative queries Optimizer chooses access path Schema changes don't break queries

Relational databases won because the query optimizer was a one-time investment — build it once, all queries benefit. In the network model, the "optimizer" was the human developer writing traversal code for every query. The relational model separated the what from the how, which is still its greatest strength.
