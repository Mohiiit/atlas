Chapter 8 — The Trouble with Distributed Systems

Part 2: Unreliable Clocks, Process Pauses, Truth and Fencing Tokens

Unreliable Clocks

Distributed systems need clocks to: measure timeouts, order events, set TTLs. But clocks on different machines are not synchronized — and even on one machine, clocks drift.

Monotonic vs. Time-of-Day Clocks

Monotonic clock
System.nanoTime(), clock_gettime(CLOCK_MONOTONIC)
Always increases. Safe for measuring elapsed time (timeout durations).
The absolute value is meaningless — only differences matter.
✓ Use for: timeouts, latency measurements
Time-of-day clock
System.currentTimeMillis(), gettimeofday()
Returns current wall-clock time. Synchronized via NTP.
Can jump backward when NTP corrects the clock.
✗ Unreliable for: measuring durations, ordering events

Clock Synchronization and Accuracy

NTP (Network Time Protocol) synchronizes clocks to within ~10ms on a well-run network. But:

  • A quartz clock drifts 200ms per day in a warm environment
  • A node that was rebooting or in sleep didn't run NTP for that period
  • NTP synchronization itself has network delay — the measurement is imprecise
  • VMs share a physical clock, and VM pauses (context switches to other VMs) cause the virtual clock to "freeze"

Relying on Synchronized Clocks: Dangerous!

Classic example of timestamp-based ordering failing:

Last-write-wins (LWW) with clocks — why it's broken
Node A (clock: 10:00:00.000) → write x=1 at timestamp 10:00:00.000
Node B (clock: 10:00:00.005, slightly fast) → write x=2 at timestamp 10:00:00.005
LWW says: x=2 wins (later timestamp). Seems right.
But: Node B's clock is 5ms ahead. "10:00:00.005" on B actually happened BEFORE "10:00:00.000" on A in real time.
Result: x=1 is silently discarded. Correct value may be lost.
Google Spanner uses TrueTime: GPS receivers + atomic clocks in every datacenter. TrueTime returns a time interval [earliest, latest], guaranteed to contain the true time. Spanner waits out the uncertainty before committing. This is expensive infrastructure most systems don't have.

Clock Readings Have a Confidence Interval

You can't know the current time exactly — you have an uncertainty range. Safe timestamp usage: compare timestamps only when the uncertainty intervals don't overlap.

Process Pauses

A running process can be paused for a surprisingly long time without knowing it:

GC stop-the-world
JVM GC can pause for 50ms–10 seconds. Thread is frozen, then resumes as if nothing happened.
VM live migration
Hypervisor suspends the VM, copies memory to another host, resumes it. VM's clock doesn't advance during the pause.
OS context switch
Thread preempted by the OS scheduler at any point. May not run again for milliseconds.
Disk I/O pause
Memory-mapped file access can block if the page isn't cached. Can pause for seconds.
Why process pauses break distributed lock assumptions
1. Client A acquires lease (distributed lock) with 30-second expiry
2. Client A reads current time: 12:00:00. Lease expires at 12:00:30.
3. GC pause for 45 seconds
4. Client A resumes. Its local time: 12:00:45. It reads 12:00:45 > 12:00:30 → knows lease expired → does NOT proceed
5. OK this time.
But: What if A doesn't check? What if A checks timestamp, decides it's safe, then gets paused between the check and the write?
Any code that assumes "nothing bad happened while I was running" is broken in a distributed system. A thread can be paused at any point. When it resumes, the world has moved on. The only safe approach: use mechanisms (fencing tokens) that detect stale operations.

Knowledge, Truth, and Lies

A node cannot know anything for certain. It can only make inferences based on messages it has received. The network may deceive it.

The Truth Is Defined by the Majority

In many distributed algorithms, truth is determined by quorum — a majority of nodes. One node's belief doesn't matter; what the majority has agreed on does.

This is why single-node decisions are dangerous. A node may believe it's the leader when the rest of the cluster has moved on and elected a new leader. The node can't trust its own belief without confirmation from others.

The Leader and the Lock Problem

Consider: only one node is allowed to hold a distributed lock at a time. A node acquires the lock, then pauses (GC). The lease expires. Another node acquires the lock. The first node resumes and proceeds — now two nodes hold the lock.

Without fencing — dangerous scenario
Client 1: Acquires lock (token 33). GC pause begins
Client 2: Acquires lock (token 34).
Client 2: Writes to storage with token 34. ✓
Client 1 resumes: Writes to storage. Storage accepts! (doesn't know lock was stolen)
Result: Client 2's write overwritten. Corruption.

Fencing Tokens: The Solution

Every time a client acquires a lock, the server assigns a monotonically increasing fencing token. The storage service only accepts writes with a token higher than the last it processed.

With fencing — safe
Client 1: Acquires lock (token 33). GC pause begins
Client 2: Acquires lock (token 34).
Client 2: Writes to storage with token 34. Storage records: last token = 34. ✓
Client 1 resumes: Tries to write with token 33.
Storage: 33 < 34 → REJECT. Write refused.
The storage service is the authority. It checks the token, not the client. No stale client can corrupt data.
ZooKeeper's zxid and epoch numbers serve as fencing tokens. HBase uses ZooKeeper's epoch to prevent stale writes. The key insight: make the decision about "is this still the current lock holder?" at the resource, not the client.
