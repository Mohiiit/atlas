Chapter 8 Part 2: Process Pauses, Truth, and System Models

The Trouble with Distributed Systems -- beyond unreliable networks and clocks, we confront process pauses, the meaning of truth in a system where no node can trust itself, and the formal models that let us reason about correctness.

1. Process Pauses

Consider a database partition leader that holds a lease (a lock with a timeout). The leader must periodically renew the lease. But what if the leader's thread is paused for longer than the lease duration? The leader thinks it still holds the lease, but in reality another node has already taken over.

The Dangerous Lease Code

while (true) { request = getIncomingRequest(); // Ensure that the lease always has at least 10 seconds remaining if (lease.expiryTimeMillis - System.currentTimeMillis() < 10000) { lease = lease.renew(); } // DANGER: What if this thread pauses for 15 seconds right here? if (lease.isValid()) { process(request); // May run AFTER lease expired! } }

Two Bugs in This Code

  • Synchronized clock dependency: The lease expiry was set by a different machine, but compared to the local system clock. Clock skew breaks the check.
  • Pause vulnerability: Even with a local monotonic clock, a 15-second pause between isValid() and process(request) means the lease may expire mid-processing -- and another leader has taken over.

Six Causes of Process Pauses

A thread can be paused at any arbitrary point and resume later without noticing. Here are six reasons this can happen:

Diagram 2: Six Sources of Process Pauses

1. Garbage Collection

Stop-the-world GC pauses can last several minutes. Even "concurrent" collectors like CMS need occasional full pauses. Allocation patterns and tuning help, but cannot eliminate.

2. VM Suspend/Resume

Virtual machines can be suspended (saved to disk) and resumed at any time. Used for live migration between hosts. Pause length depends on memory write rate.

3. Context Switches

OS context-switches or hypervisor switches to other VMs pause the thread at any point. Under heavy load, steal time can be significant -- long queues before rescheduled.

4. Synchronous Disk I/O

Blocking disk access pauses the thread. Surprisingly, even Java classloader lazily loads classes on first use. If disk is a network filesystem (e.g., EBS), network delays compound.

5. Memory Paging (Swap)

A page fault requires loading from disk. Under memory pressure, this triggers thrashing -- spending most time swapping pages in/out. Servers often disable swap entirely.

6. SIGSTOP Signal

A Unix process paused with SIGSTOP (Ctrl-Z) gets zero CPU cycles until SIGCONT is sent. Can happen accidentally from an operations engineer.

"A node in a distributed system must assume that its execution can be paused for a significant length of time at any point, even in the middle of a function. During the pause, the rest of the world keeps moving and may even declare the paused node dead."

Real-Time vs. Regular Systems

Diagram 3: Real-Time Guarantees Spectrum
Hard Real-Time
Aircraft, airbags, rockets. Missing a deadline = system failure. Requires RTOS, no dynamic allocation, extreme testing.
Soft Real-Time
Video streaming, audio processing. Deadlines are important but occasional misses are tolerable.
Best-Effort (Most Servers)
Web apps, databases, data pipelines. No timing guarantees. Must tolerate pauses and clock instability.

"Real-time" in embedded systems means guaranteed timing -- not the vague "real-time" of web push notifications.

Mitigating GC Impact (Without Real-Time)

  • Treat GC as planned downtime: When GC is needed, stop sending requests to that node, drain outstanding work, run GC, then resume. Hides pauses from clients.
  • Short-lived objects only: Use GC only for short-lived objects (fast to collect). Restart processes periodically before long-lived objects accumulate -- rolling upgrade style.
  • Used in practice: Latency-sensitive financial trading systems use this "GC-aware load balancing" approach.

2. Knowledge, Truth, and Lies

In a distributed system, a node cannot know anything for sure. It can only make guesses based on the messages it receives (or does not receive). If a remote node does not respond, there is no way to distinguish a network problem from a node crash. This leads to a philosophical question: what is truth in a system where perception is unreliable?

The Epistemological Challenge

  • No shared memory -- only messages over an unreliable network
  • A node cannot trust its own judgment (it may have been paused and declared dead)
  • The solution: define a system model (formal assumptions about behavior) and prove algorithms correct within that model
  • Reliable behavior is achievable even with very few guarantees -- but it is not straightforward

The Truth Is Defined by the Majority

A node cannot rely on its own view of the world. Instead, distributed algorithms rely on a quorum -- voting among nodes. Decisions require some minimum number of votes to reduce dependence on any one node. If a quorum declares a node dead, it must be considered dead, even if the node still feels alive.

Diagram 4: Why a Node Cannot Trust Itself
Scenario 1: Asymmetric Fault
Node receives all messages, but outgoing replies are dropped. Node works fine from its own perspective. Others declare it dead after timeout. "I'm not dead!" -- but nobody hears.
Scenario 2: Semi-Disconnected
Node notices its messages are not acknowledged. Realizes there is a network fault. But it is still declared dead by others and cannot do anything about it.
Scenario 3: GC Resurrection
Node enters 1-minute GC pause. Others declare it dead. GC finishes. Node "wakes up" and cheerfully starts chatting, not realizing a minute has passed.

The Leader and the Lock

Many systems require exactly one of something: one leader per partition (to avoid split brain), one lock holder per resource (to prevent corruption), one owner per username. But a node that believes it is "the chosen one" may have already been demoted by a quorum -- and if it continues acting on that false belief, it can corrupt data.

Fencing Tokens

The solution to the expired-lease problem is fencing tokens. Every time the lock service grants a lock or lease, it also returns a monotonically increasing number. The storage service tracks the highest token it has seen and rejects any write with a lower token.

ZooKeeper's zxid or node version cversion can serve as fencing tokens (guaranteed monotonically increasing). The resource itself must enforce the check -- clients cannot be trusted to check their own lock status.

Key Insight: Server-Side Checking is Essential

It is not sufficient for clients to check their own lock status. The resource (storage service) must actively reject writes with stale tokens. This is good practice anyway -- a service should protect itself from accidentally abusive clients whose priorities may differ from those running the service.

3. Byzantine Faults

Fencing tokens protect against nodes that inadvertently act in error. But what if a node deliberately sends a fake fencing token? This is the domain of Byzantine faults -- where nodes may "lie" (send arbitrary faulty or corrupted responses).

Diagram 9: The Byzantine Generals Problem

G4
Loyal
N generals must agree on a battle plan. Some are traitors who send contradictory messages to sow confusion. The loyal generals must reach consensus despite the traitors. Byzantine fault-tolerant algorithms require a supermajority (>2/3) of correct nodes.

When BFT Matters

  • Aerospace (radiation corrupts memory)
  • Blockchain / peer-to-peer (mutually untrusting parties)
  • Multi-organization systems

When BFT is Overkill

  • Single-org datacenters (nodes are trusted)
  • Web apps (use auth + input validation instead)
  • Same software on all nodes (bug affects all)

Weak Forms of Lying

Even without full BFT, it is worth guarding against "weak lies" -- invalid messages from hardware issues, bugs, or misconfiguration:

  • Network corruption: TCP/UDP checksums catch most, but not all. Add application-level checksums.
  • Input sanitization: Validate ranges, limit string sizes to prevent denial-of-service.
  • NTP cross-checking: Use multiple NTP servers. Exclude outliers. Majority agreement makes NTP robust.

4. System Models

To reason about distributed algorithms formally, we define a system model: an abstraction that describes what faults an algorithm may assume can occur. This involves two orthogonal dimensions: timing assumptions and failure assumptions.

Three Timing Models

Synchronous

Bounded network delay, bounded process pauses, bounded clock error. Does not mean zero delay -- just a known upper bound. Not realistic for most systems.

Partially Synchronous

Behaves synchronously most of the time, but occasionally exceeds bounds. The most realistic model for practical systems. Timing assumptions may be "shattered" sometimes.

Asynchronous

No timing assumptions at all. No clock, no timeouts. Very restrictive. Some algorithms can be designed for this, but it is the hardest model.

Three Failure Models

Crash-Stop

A node fails by crashing and never comes back. Simplest model. Once gone, gone forever.

Crash-Recovery

Nodes may crash and later restart. Stable storage survives; in-memory state is lost. The most practical model.

Byzantine (Arbitrary)

Nodes may do anything -- including lying, sending fake messages, colluding. Hardest to handle. Requires >2/3 honest nodes.

Safety vs. Liveness Properties

Algorithm correctness is defined by properties. These come in two flavors with fundamentally different characteristics:

Diagram 11: Safety vs. Liveness Properties

Safety ("Nothing bad happens")

  • If violated, you can point to the exact moment it broke
  • Once violated, cannot be undone -- damage is done
  • Must always hold, even if all nodes crash or network fails
  • Examples: uniqueness, monotonic sequence of fencing tokens

Liveness ("Something good eventually happens")

  • May not hold right now, but there is hope for the future
  • Often includes the word "eventually"
  • Allowed to have caveats (e.g., only if majority alive, network recovers)
  • Examples: availability, eventual consistency
The deal: Safety properties must always hold (no exceptions). Liveness properties must hold only under certain conditions (majority alive, network eventually recovers).

Fencing Token Algorithm: Properties Example

  • Safety Uniqueness: No two requests return the same fencing token value
  • Safety Monotonic sequence: If request x completed before y began, then token_x < token_y
  • Liveness Availability: A node that requests a token and does not crash eventually receives a response

Mapping Models to Reality

"Proving an algorithm correct does not mean its implementation on a real system will necessarily always behave correctly. But it's a very good first step."

Theory Meets Practice: The Gaps

  • Crash-recovery models assume stable storage survives. But disks get corrupted, firmware bugs prevent recognition on reboot.
  • Quorum algorithms require nodes to remember stored data. If a node has "amnesia," quorum conditions break.
  • Real implementations must handle "impossible" cases too -- even if just printf("Sucks to be you") and exit(666).
  • This is "the difference between computer science and software engineering."

5. Chapter 8 Summary: Three Categories of Trouble

Distributed systems face three fundamental categories of problems. All three can happen simultaneously, and a robust system must handle them all.

The only way information flows: messages over the unreliable network. Major decisions require quorum agreement.

The Pragmatic Takeaway

  • If you can solve a problem on a single machine, do it. A single computer can do a lot.
  • But scalability, fault tolerance, and low latency (geographic placement) force us into distributed systems.
  • Fault detection is hard (timeouts are the best we have). Fault tolerance requires quorum protocols.
  • Use partially synchronous model + crash-recovery faults as the default mental model.
  • Theoretical analysis and empirical testing are equally important.

Review Questions

Q1: Why can't a node simply check its own lease expiry time to decide if it is still the leader?
Two reasons. First, the lease expiry was set by a different machine, so comparing it to the local clock depends on clock synchronization -- which may be off. Second, even with a local monotonic clock, the thread may pause (GC, VM suspend, etc.) between checking the lease and processing the request. During that pause, the lease can expire and another node can become leader, but the paused thread has no way to detect this when it resumes. The thread continues executing as if nothing happened.
Q2: What is a fencing token, and why must the check be enforced server-side rather than client-side?
A fencing token is a monotonically increasing number returned by the lock service every time it grants a lock or lease. The storage service tracks the highest token it has seen and rejects writes with lower tokens. This must be enforced server-side because a client cannot be trusted to check its own lock status -- it may be paused, buggy, or have stale state. The resource itself must be the authority, rejecting stale writes regardless of what the client believes. ZooKeeper's zxid or cversion can serve as fencing tokens.
Q3: Name three of the six causes of process pauses and explain why each is dangerous in a distributed system.
(1) Garbage Collection: Stop-the-world GC pauses can last minutes, during which the node sends no heartbeats and is declared dead by others. (2) VM Suspend/Resume: A VM can be frozen to disk for live migration, pausing all processes for arbitrary duration -- the node has no awareness it was paused. (3) Memory Paging: Under memory pressure, a simple memory access triggers a page fault requiring disk I/O. Thrashing can effectively halt a process. All three are dangerous because the thread resumes without knowing time passed, potentially acting on stale state (expired leases, outdated data).
Q4: What is the difference between a Byzantine fault and a crash-recovery fault? When would you choose to handle each?
In the crash-recovery model, nodes are "honest" -- they either respond correctly or not at all (crashed). They may restart, and stable storage survives. This is appropriate for most server-side systems where all nodes are controlled by one organization. In the Byzantine model, nodes may "lie" -- sending arbitrary, corrupted, or deliberately misleading messages. This requires >2/3 of nodes to be honest. Byzantine fault tolerance is needed in aerospace (radiation corrupts memory), blockchain/peer-to-peer systems (untrusting parties), and multi-organization systems where participants may cheat. BFT is impractical for most datacenter systems due to cost and complexity.
Q5: Explain the difference between safety and liveness properties with an example from fencing tokens.
Safety ("nothing bad happens") must hold at all times with no exceptions. If violated, you can identify the exact moment. Example: uniqueness -- no two fencing token requests return the same value. If this is violated, it cannot be undone. Liveness ("something good eventually happens") may not hold right now but should eventually. It is allowed to have caveats (e.g., only if majority of nodes are alive). Example: availability -- a node that requests a fencing token and doesn't crash eventually receives a response. The key design principle: safety must always hold (even during total failures), while liveness may be conditional.
Q6: Why is the partially synchronous model with crash-recovery faults considered the most practical system model?
The synchronous model is too optimistic -- it assumes bounded delays, which real networks violate during congestion or failures. The asynchronous model is too restrictive -- no timeouts at all makes many algorithms impossible (FLP result). Partial synchrony captures reality: the system behaves well most of the time but occasionally exceeds bounds. Similarly, crash-stop is too pessimistic (nodes do restart in practice), while Byzantine is too expensive for trusted environments. Crash-recovery matches real servers: they crash, reboot, and rejoin with data from stable storage. Together, this model enables practical algorithms like Raft and Paxos while being honest about real-world failure modes.
