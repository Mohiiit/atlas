Chapter 3 — Storage & Retrieval

Part 1: How Databases Store Data — Logs, Hash Indexes, SSTables, and LSM-Trees

A database must solve two problems: store data when you write it, and find data when you read it. These goals are surprisingly in tension — optimizing for one often hurts the other.

As an application developer you usually don't choose a storage engine from scratch. But understanding how storage engines work helps you choose the right database and tune it for your access patterns.

The World's Simplest Database

Start from first principles. The simplest possible database is two bash functions:

#!/bin/bash
db_set () {
  echo "$1,$2" >> database   # append key,value to a file
}
db_get () {
  grep "^$1," database | tail -n 1 | sed -e "s/^$1,//"
}

db_set 42 '{"name":"San Francisco","attractions":["Pier 39"]}'
db_get 42   # → {"name":"San Francisco","attractions":["Pier 39"]}

This append-only log has surprisingly good performance for writes. Sequential disk writes are fast — much faster than random writes. But reads are O(n): grep scans the entire file every time.

The append-only log — fast writes, slow reads
42, {SF data}
123, {NY data}
42, {SF updated}
456, {LA data}
...
Write: O(1) Just append to end
Read: O(n) Scan entire file, take last match

Note that db_set doesn't update in place — it appends a new entry. The most recent entry for a key wins. This is how log-structured databases work.

Why not update in place? Sequential writes are 10–100× faster than random writes on spinning disks, and even on SSDs. Immutable append-only logs are also easier to reason about for crash recovery.

Hash Indexes (Bitcask)

The simplest way to speed up reads on a log: keep an in-memory hash map that maps every key to the byte offset of its most recent value in the log file.

Read key 42 → look up hash map → offset 128 → seek to that position in file → return value. One disk seek!

This is exactly how Bitcask (Riak's storage engine) works. All keys must fit in RAM (the hash map), but values can be larger than RAM since they stay on disk.

Compaction and Segment Merging

The log file grows forever — we need to reclaim space. Solution: segment files + compaction.

Step 1: Split log into fixed-size segments

k2=b
k1=c
k3=d
Segment 2 (closed)
k2=e
k4=f
k2=g
Active (current writes)

k3=h

← much smaller, one entry per key

Compaction happens in a background thread. The old segments are deleted once the new merged segment is written. Reads continue against the old segments until the merge is complete.

Practical Details That Make It Work

  • File format: Binary (length-prefixed records) rather than CSV — faster and handles arbitrary values
  • Deletion: A special tombstone record; during compaction, tombstones suppress old values
  • Crash recovery: Each segment file has a snapshot of its hash map on disk; rebuild from snapshot on startup
  • Checksums: Detect partially-written records from crashes
  • Concurrency: One writer thread; many reader threads (immutable segments)

Limitations of Hash Indexes

  • Range queries are slow: Hash maps don't support "find all keys between k100 and k200" — you'd need to scan every key individually
  • Keys must fit in RAM: On-disk hash maps require many random I/Os and are too slow
Bitcask works brilliantly when you have a moderate number of keys but high write throughput — e.g., a key per video URL with a constantly-updated view count. Riak uses it for exactly this pattern.

Constructing SSTables: The Memtable

Crash safety problem: The memtable is in RAM. If the database crashes before flushing, those writes are lost. Solution: maintain a separate write-ahead log (WAL) on disk — append every write there first. Discard the WAL after a successful flush.

From Memtable + SSTables → LSM-Tree

This algorithm — Log-Structured Merge-Tree (LSM-Tree) — was originally described by O'Neil et al. (1996). LevelDB, RocksDB, Cassandra, and HBase all use variants of it.

Compaction Strategies

As more SSTables accumulate on disk, merging and compaction runs in the background. Two major strategies:

Strategy How It Works Good For Downside
Size-tiered New smaller SSTables → merge into larger ones as sizes match. HBase, Cassandra. Write-heavy workloads Temporarily 2× space during compaction
Leveled SSTables split into fixed size, organized into levels (L1, L2...). LevelDB, RocksDB. Read-heavy, less space amplification More I/O during compaction
Leveled compaction — each level ~10× larger than the previous
L0
← recently flushed SSTables (may overlap)
L1 (~10MB)
← no overlapping key ranges per file
L2 (~100MB)
L3 (~1GB)
L4+ (~10GB+)

Performance Characteristics of LSM-Trees

Writes
Excellent. Sequential appends to memtable/WAL. No random I/O.
Reads
Good for recent keys. Slow for old keys (must check many SSTables). Bloom filters help.
Range queries
Excellent. Keys are sorted; sequential scan is fast.
Space
Moderate. Old values persist until compaction. Compaction uses extra space temporarily.
Write amplification: LSM-trees write data multiple times as it moves through levels. One byte written by the application may trigger many bytes of disk I/O during compaction. This matters especially for SSDs which have limited write cycles.
