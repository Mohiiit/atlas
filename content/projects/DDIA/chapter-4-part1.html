1. The Compatibility Problem

Applications change. Features are added, data models evolve, bugs are fixed. But code cannot change everywhere at once. Server-side apps use rolling upgrades: a few nodes at a time, so some nodes run v1 while others run v2. Client-side apps are worse — users may not upgrade for months. This means old and new code must coexist, reading and writing the same data.

Backward Compatibility

Newer code can read data written by older code. Usually easy — you wrote the new code and know the old format.

Forward Compatibility

Older code can read data written by newer code. Trickier — old code must silently ignore fields it doesn't know about.

2. Encoding and Decoding

Programs work with data in two representations. In-memory: objects, structs, hash maps, trees — optimized for CPU access, use pointers. Byte sequences: for files and network — self-contained since pointers are meaningless outside a process. The translation from in-memory → bytes is encoding (serialization / marshalling). The reverse is decoding (deserialization / unmarshalling).

Language-Specific Formats: Never Use Them

Every language has a built-in encoding: Java has java.io.Serializable, Python has pickle, Ruby has Marshal. They are convenient but have four deep problems.

Diagram 3 — Why Language-Specific Encoding Is Dangerous
Language Lock-in

Encoding tied to one language. Reading from Python, Go, or Java means you can never switch. Commits you for years.

▼
Security Hole

Decoding instantiates arbitrary classes. Attacker sends crafted bytes → arbitrary class instantiation → remote code execution.

▼
No Versioning

Forward and backward compatibility are afterthoughts. Schema evolution is poorly supported or ignored entirely.

▼
Bad Performance

Java's built-in serialization is notorious for poor CPU performance and bloated output size.

Rule: Never use language-specific encoding for anything beyond transient in-process purposes.

JSON, XML, CSV: Widespread but Subtly Flawed

JSON is dominant for web APIs (built into browsers, simple vs XML). XML was the 2000s enterprise standard. CSV is ubiquitous for tabular data. All three are textual and somewhat human-readable. But each has subtle problems that matter at scale.

The Four Subtle Problems

  • Number ambiguity: JSON has no integer/float distinction — both are just "numbers". Integers > 253 lose precision in IEEE 754 doubles (JavaScript's type). Twitter includes tweet IDs as both a JSON number and a decimal string to work around this.
  • No binary strings: JSON and XML only support Unicode text. Binary data must be Base64-encoded, increasing size by 33%.
  • Optional schemas: JSON Schema and XML Schema are complex and many tools ignore them. Interpretation depends on hardcoded application logic.
  • CSV has no schema: what if a value contains a comma or newline? Escaping rules exist but aren't consistently implemented.
Diagram 4 — Twitter's 64-bit ID Problem in JSON

Tweet IDs are 64-bit integers. JavaScript's Number type (IEEE 754 double) can only represent integers exactly up to 253 = ~9 quadrillion.

Broken: ID as JSON number

{
  "id": 1234567890987654321
}

Twitter's workaround: both forms

{
  "id": 1234567890987654321,
  "id_str": "1234567890987654321"
}

JS clients use the string. Non-JS clients use the number.

"Despite these flaws, JSON, XML, and CSV are good enough for many purposes, particularly as data interchange formats between organizations. In these situations, the difficulty of agreeing on anything outweighs the advantages of a more efficient format." — Kleppmann, DDIA

3. Binary Encoding Formats

For data used only internally within your organization, you can choose formats optimized for compactness and speed. JSON is less verbose than XML, but both use far more space than binary formats. The key question: how much can we shrink the same record?

Diagram 5 — Encoding Size Comparison for the Same Record

Record: {"userName":"Martin","favoriteNumber":1337,"interests":["daydreaming","hacking"]}

JSON (minified) = 81 bytes. How much can binary formats save?

JSON (text) — 81 bytes  [field names as strings in every record]
MessagePack (binary JSON) — 66 bytes  [still includes field names]
Thrift BinaryProtocol — 59 bytes  [field tags replace names]
Thrift CompactProtocol — 34 bytes  [packed type+tag, varint integers]
Protocol Buffers — 33 bytes  [similar to Thrift Compact]
Avro — 32 bytes  [no field tags or names at all]
Schemaless (JSON/MsgPack): must include field names → large Schema-driven (Thrift/Protobuf/Avro): use tags or positions → small

MessagePack: Binary JSON (Barely Worth It)

MessagePack is a drop-in binary replacement for JSON. It encodes types more compactly in the byte header — 0x83 means "object with 3 fields", 0xa8 means "8-byte string". But field names are still included as strings in every record because there's no schema. Result: 66 bytes vs 81 — only a 19% saving. The loss of human-readability is not worth it.

Thrift and Protocol Buffers: Schema-Driven Binary

Both Apache Thrift (Facebook, open-sourced 2007) and Protocol Buffers (Google, open-sourced 2008) solve the field-name problem with a schema. You define a schema in an IDL (Interface Definition Language), assign each field a tag number (a small integer), and the encoding uses tag numbers instead of field name strings.

Thrift IDL

struct Person {
  1: required string userName,
  2: optional i64    favoriteNumber,
  3: optional list<string> interests
}

Protocol Buffers IDL

message Person {
  required string user_name       = 1;
  optional int64  favorite_number = 2;
  repeated string interests       = 3;
}

Both come with code-generation tools that produce language-specific classes. One important note: required vs optional makes no difference to the encoding — nothing in the binary data indicates which it is. The difference is only a runtime check (which is why Protocol Buffers deprecated required in proto3).

How Variable-Length Integers (varint) Work

Each byte uses 7 bits for the value and 1 bit as a "continuation" flag. Small numbers fit in 1 byte. Numbers up to 16383 need 2 bytes. Only huge numbers need 8 bytes. Since most integers in practice are small, this saves significant space.

The Core Insight: Field Names vs Field Tags

The biggest space saving doesn't come from binary encoding of values — it comes from replacing field name strings with small integer tags. "userName" costs 8 bytes every record. Tag number 1 costs 1 byte. With a schema, the encoder/decoder uses a lookup table to map between names and tags. Without a schema, names must be included in the data itself.
