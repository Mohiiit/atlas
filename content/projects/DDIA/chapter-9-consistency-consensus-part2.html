Ordering and Causality

Ordering is fundamental to distributed systems. When events happen concurrently on different nodes, we need a way to reason about their order — not to determine who was "first" in wall-clock time, but to preserve causality.

The Causal Order

Events have a causal relationship if one event influenced another. If A happened before B in the causal sense, any correct system must reflect this ordering.

Causally ordered events:
Alice posts: "The weather is nice today." [event A]
Bob reads Alice's post. [event B — causally after A]
Bob replies: "I agree!" [event C — causally after A and B]
If a node shows Bob's reply before Alice's post — causality is violated.
Causal consistency: you always see causes before effects.

Causality vs. Total Order

Ordering typeWhat it meansStrength
Causal order If A caused B, then A is ordered before B. Concurrent events (no causal relationship) have no defined order. Weaker. Partial order. Allows concurrency.
Total order Every pair of events is comparable. Every event has a position in a single global sequence. Stronger. No concurrency — all events are ordered.
Linearizability A total order that also matches real-time: if A completed before B started, A comes first. Strongest. Subsumes both.
Causal consistency is the strongest consistency model that does not incur the performance penalty of linearizability. It can be implemented without requiring coordination across datacenters. Many "eventual consistency" systems actually provide causal consistency.

Sequence Numbers and Lamport Timestamps

We need a way to generate timestamps that respect causality — without relying on synchronized wall clocks.

Lamport Timestamps

A simple mechanism for ordering events causally in a distributed system. Each node maintains a counter. The rules:

Rules
1. On any event: increment local counter
2. On send: attach (counter, node_id) to message
3. On receive: set counter = max(local, received) + 1

The result: if A causally happened before B, then timestamp(A) < timestamp(B).
Example
Node 1: event at counter=1 → send (1, N1)
Node 2: receives (1, N1), sets counter = max(0, 1) + 1 = 2
Node 2: event at counter=2 → send (2, N2)
Node 1: receives (2, N2), sets counter = max(1, 2) + 1 = 3
Lamport timestamps provide a total order: compare (counter, node_id). If counters are equal, break ties by node_id.

The Limitation of Lamport Timestamps

Lamport timestamps give total order, but they don't tell you about causality in reverse — you can't determine from timestamps alone whether two events are causally related or concurrent.

Lamport timestamps say: if A happened-before B, then ts(A) < ts(B). But the reverse isn't necessarily true: ts(A) < ts(B) does NOT mean A happened-before B. They could be concurrent. For detecting concurrency, you need version vectors (Ch 5).

Sequence Numbers Are Not Enough for Some Problems

Consider: generating unique usernames. Two nodes each generate a unique username concurrently. With Lamport timestamps, you can order the events after the fact — but you don't know, at the time of the operation, whether the username has been taken on another node. You'd need to wait for all nodes to agree.

Total Order Broadcast

Lamport timestamps give a total order, but they require the entire history to determine ordering. What we really need is a protocol that delivers messages to all nodes in the same order: Total Order Broadcast (also called atomic broadcast).

Total Order Broadcast: a protocol that delivers messages to all nodes such that (1) every node receives every message, and (2) every node receives them in the same order. No message is missed; no message is received out of order.

Properties of Total Order Broadcast

Reliable delivery
No messages are lost. If the message is delivered to one node, it is delivered to all nodes. (Crash-recovery: if a node crashes, it receives the message when it recovers.)
Totally ordered delivery
Messages are delivered to every node in the same order. If node 1 delivers message A before B, then node 2 also delivers A before B.

Using Total Order Broadcast

Total order broadcast is useful for:

  • Database replication: Deliver writes in the same order to all replicas. Every replica applies writes in the same sequence → all replicas stay consistent.
  • Serializable transactions: Apply transactions in the same order on every node → same result everywhere.
  • Lock service: All lock requests processed in the same order → no two nodes grant the same lock.
  • Unique ID generation: Sequence numbers assigned by total order → globally unique IDs.
ZooKeeper and etcd implement total order broadcast. Their zxid (ZooKeeper transaction ID) and etcd's revision numbers are monotonically increasing sequence numbers that serve as total order. Every write gets a new sequence number; reads can specify "read at sequence number X" to get consistent data.

Total Order Broadcast and Linearizability

Total order broadcast and linearizability are closely related but distinct:

Total order broadcast → linearizable writes: append every write to the log. Log order = write order.
Linearizable reads: require an extra step. You must read the latest log position before reading data — a sync read. Otherwise, you might read stale data.
So: total order broadcast gives linearizable writes for free, but linearizable reads require a round-trip to confirm you've seen all messages up to now.

Two-Phase Commit (2PC)

So far we've discussed single-node operations. What about distributed transactions — atomicity across multiple nodes? Two-phase commit is the classic protocol.

The Problem: Atomic Commit Across Nodes

A distributed transaction touches multiple nodes (e.g., debit on node A, credit on node B). Both operations must succeed or both must fail. How do you ensure this when nodes can fail independently?

Without 2PC — the failure scenario:
Node A: debit $100 → committed
Node B: credit $100 → crash before commit
Result: $100 disappears. Atomicity violated.

Two-Phase Commit Protocol

Coordinator orchestrates the commit across all participants
Phase 1: Prepare
Coordinator sends PREPARE to all participants.
Each participant: execute transaction, write to WAL, send VOTE-YES (or VOTE-NO if failed).
Participants are now "in doubt" — they've promised to commit but haven't yet.
↓
Phase 2: Commit or Abort
If all voted YES: coordinator sends COMMIT. Each participant commits, releases locks.
If any voted NO: coordinator sends ABORT. Each participant rolls back.
Decision is final. Coordinator writes decision to disk before sending.

The 2PC Failure Problem

Coordinator crashes after Phase 1 but before Phase 2:
Participants have voted YES and are "in doubt" — they hold locks and can't proceed.
They cannot commit (coordinator might decide abort) or abort (coordinator might decide commit).
They must wait for coordinator to recover. If coordinator disk is lost: manual recovery required.
This is the "blocking" nature of 2PC: participants are stuck until coordinator recovers.
2PC is a "blocking" protocol. If the coordinator fails at the wrong moment, participants are stuck in an uncertain state, holding locks indefinitely. This can block the entire system. Coordinator recovery (from disk WAL) is the only way out — or manual DBA intervention.

Three-Phase Commit (3PC) — Theoretical Fix

3PC adds a "pre-commit" phase to break the blocking nature of 2PC. It's non-blocking in theory but assumes a synchronous network with bounded delay — which real networks don't provide. 3PC is not used in practice.
