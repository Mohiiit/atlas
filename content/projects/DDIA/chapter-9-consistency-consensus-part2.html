DDIA -- Chapter 9: Consistency and Consensus

Part 2: Distributed Transactions and Consensus

Consensus is one of the most important and fundamental problems in distributed computing: getting several nodes to agree on something. This part covers atomic commit (2PC), fault-tolerant consensus algorithms (Raft, Paxos, Zab, VSR), and coordination services like ZooKeeper.

Why Do We Need Consensus?

There are several situations where nodes must agree on something, and getting it wrong leads to data loss, inconsistency, or system failure.

Leader Election

In single-leader replication, all nodes must agree on which node is the leader. A contested leadership (split brain) means two nodes accept writes, leading to divergence and data loss.

Atomic Commit

In a transaction spanning multiple nodes/partitions, all nodes must agree on the outcome: either all commit or all abort. Partial commitment violates atomicity.

The FLP Impossibility Result

Fischer, Lynch, and Paterson (1985) proved that no algorithm can always reach consensus if there is a risk that a node may crash -- in the asynchronous system model (no clocks/timeouts). However, if the algorithm can use timeouts or random numbers, consensus becomes solvable in practice. The FLP result is of great theoretical importance, but distributed systems can usually achieve consensus.

Atomic Commit and Two-Phase Commit (2PC)

From Single-Node to Distributed Atomic Commit

On a single node, atomicity is simple: the storage engine writes data to a WAL, then appends a commit record. The commit point is the moment the disk finishes writing the commit record -- a single device makes the decision atomic.

With multiple nodes, simply sending a commit request to each independently can fail: some nodes may commit while others abort (constraint violation, network loss, crash). Once committed on one node, it cannot be retracted. A node must only commit once it is certain all other nodes will also commit. This requires a protocol: Two-Phase Commit (2PC).

Don't Confuse 2PC and 2PL!

Two-Phase Commit (2PC) provides atomic commit in a distributed database. Two-Phase Locking (2PL) provides serializable isolation. They are entirely separate concepts despite the unfortunate similarity in names.

The Full 2PC Sequence

2PC introduces a coordinator (transaction manager) that orchestrates the commit across all participants. The protocol has two phases -- prepare and commit/abort -- but the full lifecycle has more steps.

A System of Promises -- The 6 Steps

Step 1: Request Transaction ID
Application requests a globally unique transaction ID from the coordinator.
Step 2: Begin Single-Node Transactions
Application begins a transaction on each participant, attaching the global txn ID. All reads and writes happen here. Any failure at this stage allows coordinator or participant to abort.
Step 3: Coordinator Sends Prepare (Phase 1)
When ready to commit, coordinator sends a prepare request to all participants. If any request fails or times out, coordinator sends abort to all.
Step 4: Participants Vote
Each participant ensures it can definitely commit (writes data to disk, checks constraints). By voting "yes," the participant surrenders the right to abort but has not yet committed. This is a point of no return for the participant.
Step 5: Coordinator Decides (Commit Point)
If all voted "yes," coordinator decides to commit; otherwise abort. The coordinator writes this decision to its transaction log on disk. This is the commit point -- the second point of no return.
Step 6: Coordinator Sends Commit/Abort (Phase 2)
Commit or abort request sent to all participants. If this fails, the coordinator retries forever. There is no going back. If a participant crashed, the transaction will be committed when it recovers.
"The protocol contains two crucial 'points of no return': when a participant votes 'yes,' it promises that it will definitely be able to commit later; and once the coordinator decides, that decision is irrevocable."

Coordinator Failure and the In-Doubt Problem

If the coordinator crashes before sending prepare requests, participants can safely abort. But if it crashes after participants have voted "yes" and before sending the commit/abort decision, participants are stuck: they cannot unilaterally commit (another participant may have aborted) or abort (the coordinator may have decided to commit). This state is called in doubt or uncertain.

Diagram 2: Coordinator Crash -- The In-Doubt Problem
Coordinator Database 1 Database 2 | | | | Prepare | | |--------------------------->| | |----------------------------------------------->| | | | | Vote YES | | |<---------------------------| | | Vote YES | |<-----------------------------------------------| | | | | Decision: COMMIT | | | (written to disk) | | | | | | Commit -------X CRASH! | | | | | | | +------>| ??? | | | IN DOUBT | | | Cannot commit | Commit received | | Cannot abort | COMMITTED | | Must WAIT | | | |

Database 1 is stuck in doubt: it voted "yes" but never received the coordinator's decision. It cannot abort (Database 2 already committed) or commit (what if coordinator had decided abort?). It must wait for the coordinator to recover.

Holding Locks While In Doubt

An in-doubt participant holds all its database locks (row-level exclusive locks, shared locks for 2PL). These locks cannot be released until the transaction commits or aborts. If the coordinator takes 20 minutes to restart, locks are held for 20 minutes. If the coordinator's log is lost, locks are held forever -- or until an administrator manually resolves the situation. Other transactions that need those rows are blocked, potentially causing large parts of the application to become unavailable.

Recovery: The Only Way Out

  • The coordinator must write its decision to a transaction log on disk before sending commit/abort to participants
  • On recovery, the coordinator reads its log and resolves in-doubt transactions
  • Transactions without a commit record are aborted
  • Orphaned in-doubt transactions (corrupted log, bugs) require manual administrator intervention
  • The "heuristic decision" escape hatch lets participants unilaterally decide -- but this probably breaks atomicity

Three-Phase Commit (3PC)

2PC is a blocking atomic commit protocol -- it can get stuck waiting for the coordinator. Three-phase commit (3PC) was proposed as a nonblocking alternative.

Why 3PC Doesn't Work in Practice

  • 3PC assumes a network with bounded delay and nodes with bounded response times
  • In most practical systems with unbounded network delay and process pauses, 3PC cannot guarantee atomicity
  • Nonblocking atomic commit requires a perfect failure detector -- a reliable mechanism for telling whether a node has crashed (timeouts are not reliable in unbounded-delay networks)
  • For this reason, 2PC continues to be used despite the known coordinator failure problem

XA is not a network protocol -- it is a C API. The coordinator lives inside the application process. If the application crashes, the coordinator goes with it, leaving participants in doubt until the app is restarted and the coordinator recovers from its local disk log.

Two Types of Distributed Transactions

Database-Internal

  • All participants run the same database software
  • Can use any internal protocol with optimizations
  • VoltDB, MySQL Cluster NDB
  • Can work quite well (e.g., distributed SSI possible)

Heterogeneous (XA)

  • Participants are different technologies (different DBs, message brokers)
  • Must use lowest common denominator protocol
  • Cannot detect cross-system deadlocks
  • Cannot work with SSI across systems
  • Much more challenging operationally

Limitations of Distributed Transactions

Major Operational Problems

  • Coordinator is a single point of failure -- if not replicated, its failure blocks the entire system
  • Breaks stateless application model -- coordinator logs become critical durable state that must survive crashes
  • Lowest common denominator -- no cross-system deadlock detection, no SSI across systems
  • All participants must respond -- any broken part fails the entire transaction
  • Amplifies failures -- runs counter to fault-tolerance goals
  • Heavy performance penalty -- MySQL distributed transactions are 10x slower than single-node due to extra fsync and network round-trips

Fault-Tolerant Consensus

Consensus means getting several nodes to agree on something. One or more nodes may propose values, and the consensus algorithm decides on one of those values.

Diagram 5: The Four Properties of Fault-Tolerant Consensus

1. Uniform Agreement

No two nodes decide differently.

Safety Property

2. Integrity

No node decides twice.

Safety Property

3. Validity

If a node decides value v, then v was proposed by some node.

Safety Property

4. Termination

Every node that does not crash eventually decides some value.

Liveness Property

Agreement + Integrity = core idea of consensus (everyone agrees, decisions are final). Validity rules out trivial solutions (always deciding null). Termination = fault tolerance (the algorithm must make progress). Safety properties are always met, even during majority failure. Termination requires that fewer than half of nodes are crashed.

"If you don't care about fault tolerance, then satisfying the first three properties is easy: just hardcode one node to be the 'dictator.' This is, in fact, what happens in 2PC: if the coordinator fails, in-doubt participants cannot decide whether to commit or abort."

Consensus Algorithms: Raft, Paxos, Zab, VSR

Consensus = Total Order Broadcast

Total order broadcast requires messages to be delivered exactly once, in the same order, to all nodes. This is equivalent to performing repeated rounds of consensus: in each round, nodes propose the next message to deliver, and decide on the order.

Mapping Consensus Properties to Total Order Broadcast

  • Agreement -- all nodes deliver the same messages in the same order
  • Integrity -- messages are not duplicated
  • Validity -- messages are not corrupted or fabricated out of thin air
  • Termination -- messages are not lost

Epoch Numbering and Quorum Voting

All consensus protocols use a leader internally, but they don't guarantee the leader is unique globally. Instead, they define an epoch number (ballot number in Paxos, view number in VSR, term number in Raft) and guarantee that within each epoch, the leader is unique.

"The voting process looks superficially similar to two-phase commit. The biggest differences are that in 2PC the coordinator is not elected, and that fault-tolerant consensus algorithms only require votes from a majority of nodes, whereas 2PC requires a 'yes' vote from every participant."

Limitations of Consensus

Consensus algorithms are a huge breakthrough -- concrete safety properties in uncertain environments that remain fault-tolerant. But they come at a cost.

Diagram 9: Five Limitations of Consensus
1
Synchronous replication overhead
Voting before decisions = synchronous replication. Many databases prefer async replication for performance, accepting data loss on failover.
2
Strict majority required
Need 3 nodes to tolerate 1 failure, 5 nodes for 2 failures. Network partitions block the minority side completely.
3
Fixed membership assumption
Most algorithms assume a fixed set of voting nodes. Dynamic membership extensions exist but are much less well understood.
4
Timeout-based failure detection
In geo-distributed systems with variable delays, nodes may falsely believe the leader is dead. Frequent spurious elections destroy performance.
5
Network sensitivity
Raft can thrash leadership between two nodes on a flaky link, effectively never making progress. Designing algorithms robust to unreliable networks remains an open research problem.

Membership and Coordination Services: ZooKeeper

ZooKeeper, etcd, and Consul look like key-value stores but implement consensus internally. They hold small amounts of data (fits in memory, written to disk for durability) replicated via fault-tolerant total order broadcast. ZooKeeper is modeled after Google's Chubby lock service.

Diagram 10: ZooKeeper -- Four Key Features
1. Linearizable Atomic Operations
Atomic compare-and-set for distributed locks. Implemented as leases with expiry times (released if client fails). Only feature that truly requires consensus.
2. Total Ordering of Operations
Monotonically increasing zxid (transaction ID) and cversion (version number) for fencing tokens. Prevents conflicts from process pauses.
3. Failure Detection
Long-lived sessions with periodic heartbeats. Session survives temporary disconnects. After timeout, session declared dead. Locks auto-released via ephemeral nodes.
4. Change Notifications
Clients watch keys/locks for changes. Get notified when another client joins (writes to ZK), fails (ephemeral nodes disappear), etc. No polling required.

It is the combination of these features that makes ZooKeeper so useful. Individually, only linearizable operations require consensus. Together, they enable leader election, work allocation, partition assignment, and automatic fault recovery.

How ZooKeeper Is Used

Leader Election / Work Allocation

Multiple instances of a service need to choose a leader. Using atomic operations + ephemeral nodes + notifications: when the leader fails, its ephemeral node disappears, other nodes are notified, and a new leader is elected automatically. Used by HBase, Hadoop YARN, OpenStack Nova, Kafka.

Partition Assignment

When partitioned resources (DB, message streams, files) need to be assigned to nodes: as nodes join/leave/fail, ZooKeeper coordinates rebalancing. The application can automatically recover from faults without human intervention.

Service Discovery

Service Discovery vs Consensus

ZooKeeper, etcd, and Consul are often used for service discovery -- finding which IP address to connect to for a service. Services register their endpoints when they start up. However, service discovery does not strictly require consensus -- DNS works fine with caching and stale reads. What does require consensus is leader election. Since the consensus system already knows who the leader is, it makes sense to also serve that information for discovery. Some consensus systems support read-only caching replicas that asynchronously receive the log but don't vote, enabling scalable non-linearizable reads.

Membership Services

What Is a Membership Service?

A membership service determines which nodes are currently active and live members of a cluster. Due to unbounded network delays, you cannot reliably detect node failure. But coupling failure detection with consensus lets nodes agree on which nodes should be considered alive or dead. A node may be incorrectly declared dead, but the system has agreement -- and that agreement enables choosing a leader (e.g., lowest-numbered living member) consistently across all nodes.

The Grand Unification: Consistency, Algorithms, and Implementations

A remarkable insight from this chapter: linearizable compare-and-set, atomic commit, total order broadcast, locks/leases, membership services, and uniqueness constraints are all reducible to consensus. If you can solve one, you can solve them all.

Not Everything Needs Consensus

When Consensus Is Not Required

Leaderless and multi-leader replication systems typically do not use global consensus. The conflicts that occur in these systems are a consequence of not having consensus, but that may be acceptable. Sometimes we simply need to cope without linearizability and learn to work with data that has branching and merging version histories. Alternative methods (explored in Chapters 11 and 12) can achieve cross-system consistency without the pain of heterogeneous distributed transactions.

Test Your Understanding

1Why does 2PC block when the coordinator crashes, while Raft/Paxos do not? ▼

In 2PC, the coordinator is a fixed, non-elected role, and the protocol requires a "yes" vote from every participant. If the coordinator crashes after participants vote "yes" but before sending the decision, participants are stuck in doubt because they cannot unilaterally decide (aborting might conflict with a committed participant, committing might conflict with an aborted one). They must wait for the coordinator to recover and read its transaction log.

In Raft/Paxos, the leader is elected and can be replaced. Decisions require only a majority quorum, not unanimity. If the leader fails, a new leader is elected via a higher epoch number, and the protocol has a defined recovery process to bring nodes into a consistent state. This satisfies the termination property that 2PC lacks.

2What is the "commit point" in 2PC, and why is it a single-node atomic operation? ▼

The commit point is the moment when the coordinator writes its commit (or abort) decision to its transaction log on disk. Before this moment, the transaction can still be aborted. After this moment, the decision is irrevocable and must be enforced even if retries are needed.

This is a single-node atomic operation because it comes down to writing a single commit record on one disk on one machine -- exactly like a single-node transaction commit. The entire complexity of 2PC ultimately reduces to this single-device atomic write on the coordinator. This is also why coordinator failure is so problematic: if that one disk's log is lost, the in-doubt transactions cannot be resolved.

3Why does 3PC not work in practice, and what would it need to work? ▼

3PC was designed to be a nonblocking alternative to 2PC by adding an intermediate "pre-commit" phase. However, it assumes a network with bounded delay and nodes with bounded response times. In real-world systems, network delays and process pauses are unbounded (as discussed in Chapter 8), so 3PC cannot guarantee atomicity.

For nonblocking atomic commit to work, you would need a perfect failure detector -- a reliable mechanism for determining whether a node has truly crashed. In networks with unbounded delay, a timeout is not a reliable failure detector because a request might time out due to network congestion, not a crash. This is why 2PC continues to be used despite its blocking problem.

4Explain the "chicken-and-egg" problem of consensus and how epoch numbering resolves it. ▼

Consensus algorithms implement total order broadcast, which is equivalent to single-leader replication. But single-leader replication requires a leader. Electing a leader requires consensus. So to solve consensus, we seemingly need to first solve consensus -- a circular dependency.

The resolution is epoch numbering. The protocols don't guarantee a globally unique leader -- they guarantee a unique leader within each epoch. When a leader appears to fail, a new election with a higher epoch number occurs. If two leaders exist in different epochs, the one with the higher epoch prevails. Before making any decision, a leader must collect votes from a quorum. The key insight is that the quorum for leader election and the quorum for proposal voting must overlap: at least one node participated in both, ensuring that no higher-epoch leader was elected without the current leader detecting it.

5Why does ZooKeeper run on a small fixed number of nodes even when the application has thousands of nodes? ▼

Performing majority votes over thousands of nodes would be terribly inefficient. Instead, ZooKeeper runs on a fixed number of nodes (usually 3 or 5) and performs majority votes among those few nodes. It then supports a potentially large number of clients that connect to the ZooKeeper cluster. This "outsources" consensus, operation ordering, and failure detection to the small ZooKeeper ensemble, while the application nodes just use the results.

The data managed by ZooKeeper is slow-changing (which node is the leader for a partition, cluster membership) -- on the timescale of minutes or hours. Runtime application state changing thousands of times per second should use other tools (like Apache BookKeeper).

6What are the six problems that are all "equivalent to consensus," and what does equivalence mean here? ▼

The six equivalent problems are:

  1. Linearizable compare-and-set registers -- atomically deciding whether to set a value based on its current value
  2. Atomic transaction commit -- all nodes agreeing to commit or abort a distributed transaction
  3. Total order broadcast -- all nodes agreeing on the order of message delivery
  4. Locks and leases -- deciding which client successfully acquired a lock
  5. Membership/coordination service -- agreeing which nodes are alive or dead
  6. Uniqueness constraints -- deciding which conflicting record to allow when multiple transactions try to create records with the same key

"Equivalent" means that if you have a solution for any one of these problems, you can transform it into a solution for any of the others. For example, total order broadcast and linearizable compare-and-set are both equivalent to consensus -- as proved by Herlihy (1991). This is a profound insight: these seemingly different problems are all the same problem in disguise.
