Part 2: Snapshot Isolation, Preventing Lost Updates, Write Skew & Phantoms

Snapshot Isolation and Repeatable Read

Snapshot isolation is the solution to read skew. The idea: each transaction reads from a consistent snapshot of the database — the committed data at the point the transaction started. Subsequent commits by other transactions don't affect what this transaction sees.

Snapshot isolation solves Alice's bank account problem
Time
Alice (snapshot txn)
Transfer transaction
t1
BEGIN (snapshot = txn 12)
t2
READ account1 = $500
t3
Transfer $100. COMMIT (txn 13)
t4
READ account2 = $200 (sees snapshot 12, not 13!)
Alice consistently sees: account1=$500, account2=$200. Total=$700... wait, that's wrong! No — she sees both accounts as they were at her snapshot (before transfer): account1=$500, account2=$100. Total=$600 ✓

Implementing Snapshot Isolation: MVCC

Multi-Version Concurrency Control (MVCC) — the standard implementation. Instead of updating data in place, the database keeps multiple versions of each row, each tagged with the transaction ID that created it.

Multiple versions of a row in PostgreSQL's MVCC
created_by_txn deleted_by_txn account2.balance
txn 1 txn 13 $100 (original value)
txn 13 ∞ $200 (after transfer)
Alice (snapshot txn 12) reads: finds the version created_by_txn=1 (visible at txn 12), deleted_by_txn=13 (not yet deleted at txn 12). Returns $100.
Carol (snapshot txn 14) reads: version created=1 is deleted at 13 (before her snapshot). Returns $200 (created=13, not yet deleted).

Visibility Rules

A transaction sees a row version if:

  1. The created_by_txn is already committed and the transaction ID ≤ this transaction's snapshot ID
  2. The deleted_by_txn is either not set, or the deleting transaction had not yet committed at snapshot time
Reads never block writes, and writes never block reads. This is why PostgreSQL and MySQL (InnoDB) can have high read concurrency — readers and writers don't conflict. The downside: old row versions accumulate and must be garbage collected (VACUUM in PostgreSQL).

Indexes and Snapshot Isolation

Two approaches for indexes with MVCC:

  • Point to all versions: Index entries point to all versions of a row; at query time, filter by visibility rules
  • Copy-on-write B-trees: Each write creates a new B-tree root, so each snapshot has its own root. Old roots serve old snapshots. (Used by CouchDB, Datomic)

Preventing Lost Updates

Read Committed and Snapshot Isolation are about what a reading transaction sees. Now consider two concurrent write transactions. The lost update problem:

Lost update — read-modify-write cycle
Time
Transaction 1 (counter++)
Transaction 2 (counter++)
t1
READ counter = 42
t2
READ counter = 42
t3
WRITE counter = 43
t4
WRITE counter = 43 (overwrites Tx1's write!)
Expected: counter = 44. Actual: counter = 43. One increment was lost.

Common patterns where lost updates occur:

  • Incrementing a counter or updating an account balance
  • Editing a wiki page (read → modify → write back)
  • Adding an item to a list stored as JSON blob
  • Two users editing their shopping carts simultaneously

Solutions to Lost Updates

Solution How Notes
Atomic operations UPDATE counter SET val = val + 1 — the database atomically reads and increments Best option when applicable. No application-level read needed.
Explicit locking SELECT ... FOR UPDATE — lock the row before reading; other transactions must wait Forces serial execution. Works but reduces concurrency.
Compare-and-set UPDATE ... WHERE val = old_val — only write if value hasn't changed since we read it May not work if database allows reading from old snapshot (check docs)
Automatic detection Database detects lost update (e.g., PostgreSQL's repeatable read) and aborts one transaction Transparent to application. PostgreSQL, MySQL, SQL Server, Oracle all support this.
Conflict resolution (multi-master) In multi-master setups, allow concurrent writes, merge conflicts later Hard to get right. Last-write-wins loses updates. Custom merge logic needed.

Write Skew and Phantoms

Lost updates happen when two transactions write the same object. Write skew is more subtle: two transactions each read some objects, then write to different objects — but together their writes violate an invariant.

Classic write skew: hospital on-call doctors
Invariant: At least one doctor must always be on call.
Currently: Alice and Bob are both on call.
Time
Alice's transaction
Bob's transaction
t1
SELECT count(*) from oncall → 2
t2
SELECT count(*) from oncall → 2
t3
count ≥ 2, safe to leave
count ≥ 2, safe to leave
t4
UPDATE alice: oncall = false
UPDATE bob: oncall = false
t5
COMMIT
COMMIT → 0 doctors on call! ⚠️
Each transaction read the same rows, made a decision based on that read, then wrote to different rows. The writes together violated the invariant. This is NOT a lost update — both writes committed successfully.

More Write Skew Examples

Meeting room booking
Two people both check "is room free at 2pm?" → both see no conflicts → both book. Double-booked.
Multiplayer game
Two players both try to move to a square simultaneously. Both check "is square empty?" → both move. Two pieces on same square.
Username uniqueness
Both transactions check "is username taken?" → both see no → both insert. Duplicate usernames. (Solved with UNIQUE constraint.)
Spending limit
Two payments both check "is total spending < $1000?" → both see total=$900 → both approve $200 charges. Total = $1100.

The Phantom Problem

Write skew is often caused by phantoms — a transaction reads a set of objects based on a condition, another transaction inserts a new object matching that condition, and the first transaction's read result is now stale.

Tx1: SELECT * FROM rooms WHERE time = '2pm' AND booked = false
Returns 0 rows (no booking conflict)
↓
Tx2: INSERT INTO bookings (room, time, user) VALUES (101, '2pm', bob)
Creates a "phantom" row that Tx1 didn't see
↓
Tx1: INSERT INTO bookings (room, time, user) VALUES (101, '2pm', alice)
Double booking! Tx1's check was valid but is now stale

The phantom is the new row that "appeared" between Tx1's check and Tx1's write. Since it didn't exist when Tx1 checked, there's nothing to lock. Regular snapshot isolation doesn't prevent this.

Materializing Conflicts

One approach to phantoms: create artificial lock targets. For the room booking problem, pre-populate a table with time slots (even if not yet booked) and acquire row locks on those time slot rows. Now Tx2's insert conflicts with Tx1's lock on the time slot.

Materializing conflicts is fragile and ugly — it leaks concurrency control into the data model. The better solution is serializable isolation (covered in Part 3).
