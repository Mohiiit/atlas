Leaderless Replication (Dynamo-Style)

Amazon's Dynamo paper (2007) popularized an approach where there is no designated leader. Clients send writes to multiple replicas directly (or via a coordinator). Any replica can accept any write. Cassandra, Riak, and Voldemort use this model.

Client
↙
↓
↘
Node 1
✓ ack
Node 2
✓ ack
Node 3
✗ down
2 out of 3 acks = write succeeds. Node 3 will catch up later.

Quorums for Reading and Writing

With n replicas, a write that requires w nodes to acknowledge, and a read that queries r nodes:

w + r > n → reads will see the most recent write
n=3 replicas
Typical: w=2, r=2
w+r=4 > 3 ✓ guaranteed to overlap
n=5 replicas
Typical: w=3, r=3
w+r=6 > 5 ✓
Fast reads
w=n, r=1
Write all replicas, read one. Very fast reads, slow writes.
Common choice: w=quorum(n), r=quorum(n) e.g., n=3, w=r=2 (majority quorum)

When a read returns values from multiple nodes, the client takes the most recent version (by timestamp or version number). The client (or coordinator) then does read repair — writes the up-to-date value back to any lagging replicas.

Sloppy Quorums and Hinted Handoff

What if a network partition means the client can only reach nodes outside the "home" set? Should the write fail, or be accepted by any available node?

Strict quorum
Write must go to w of n "home" nodes. Partition → write fails. Higher consistency.
Sloppy quorum
Write accepted by any w reachable nodes — even if they're not the "home" n nodes. When network heals: hinted handoff sends data to proper home nodes.
Sloppy quorums improve availability (writes succeed during partitions) but weaken consistency (reads of home nodes may not see latest value). Cassandra uses sloppy quorums by default.

Detecting Concurrent Writes

Leaderless databases allow concurrent writes to the same key. The system must detect when writes conflict and decide which wins.

Last Write Wins (LWW)

Each write is tagged with a timestamp. On conflict, the write with the higher timestamp wins; others are discarded. Simple but lossy — concurrent writes are silently dropped. Cassandra uses LWW as its default. DynamoDB does too.

LWW achieves eventual consistency by discarding data. If two clients concurrently write to the same key, one write is silently lost. The only safe use: if each write is intended to replace the previous one (not to be merged), and data loss is acceptable.

The "Happens-Before" Relationship

Operation A happens-before B if B knows about A or depends on A. If neither happens-before the other, they are concurrent — the system cannot tell which came first.

Shopping cart example: tracking causality with version numbers
Client 1: GET cart → v1:[milk]. Adds eggs. PUT cart [milk,eggs] v1 → server returns v2:[milk,eggs]
Client 2 (concurrently): GET cart → v1:[milk]. Adds flour. PUT cart [milk,flour] v1 → server has CONFLICT with v2
Server stores both versions: v2:[milk,eggs] and v3:[milk,flour]. Client must merge: [milk,eggs,flour].

Version Vectors

In a multi-replica leaderless system, each replica tracks versions independently. A version vector (or vector clock) maps each replica ID to its counter. This allows detecting which writes are concurrent vs. causally related.

Replica 1 sees: {R1:3, R2:1}
← knows about 3 writes on R1, 1 on R2
Replica 2 sees: {R1:2, R2:2}
← knows about 2 writes on each
Client collects version vectors from all replicas. If one dominates (all counters ≥), it's a causal successor. If neither dominates, they're concurrent → both kept → client merges.
Riak uses dotted version vectors. CRDTs (Conflict-free Replicated Data Types) like counters and sets merge automatically. For text documents, operational transformation or CRDTs are needed (Google Docs uses OT).

Review Q&A

1
Why is async replication faster but potentially data-losing?
›
Async replication means the leader acknowledges the write to the client before it replicates to followers. The write is in the leader's memory/WAL and confirmed to the client, but only on one node. If the leader crashes before replicating, those writes are gone — the new leader doesn't have them. Sync replication waits for at least one follower to confirm before acking the client, ensuring 2+ copies exist before the client is told "success".
2
What is split brain and how is it dangerous?
›
Split brain occurs when two nodes both believe they are the leader and both accept writes. This typically happens during a network partition: one side can't communicate with the other, so each promotes its own leader. Both leaders accept client writes. When the partition heals, the two leaders have diverged data with no way to automatically reconcile. Depending on the application, this could mean duplicate orders, duplicate payments, corrupted data, or inconsistent state. STONITH (forcibly shutting down the old leader) prevents split brain but risks unnecessary shutdowns during false alarms.
3
What is read-your-own-writes consistency and when does it matter?
›
Read-your-own-writes (aka read-after-write consistency) guarantees that after you write something, you can immediately read it back and see your own write. Without this guarantee, if your write goes to the leader and you then read from a lagging follower, you see the old value — as if your write was lost. This matters everywhere users expect to see their own changes immediately: social media posts, profile edits, form submissions, etc. Implementations route the user's own reads to the leader, or track a "write timestamp" and only read from followers that are caught up past that timestamp.
4
Explain the quorum condition w + r > n and why it guarantees seeing the latest write.
›
With n replicas, a write goes to w nodes and a read goes to r nodes. If w + r > n, then the sets of w nodes that received the write and r nodes queried for the read must overlap by at least one node (pigeonhole principle). That overlapping node has the latest write. So among the r values returned, at least one will be up-to-date. The client picks the highest-version value. Classic setup: n=3, w=2, r=2. Write goes to 2 of 3 nodes; read queries 2 of 3 — they must share at least 1 node. Note: this is only guaranteed for strict quorums, not sloppy ones.
5
What is the problem with Last Write Wins (LWW) in practice?
›
LWW silently discards data. If two clients concurrently write to the same key with different values, the one with the higher timestamp "wins" and the other is dropped — the losing client's write is lost with no error. The problem is amplified because timestamps are typically wall-clock times, and clocks across nodes can't be perfectly synchronized (Chapter 8 covers this). So "which timestamp is higher" is unreliable for concurrent writes. The only safe usage: each key is only written by one client at a time (so true concurrency never occurs), and the data is safe to lose (e.g., caching). For anything else, LWW is risky.
6
What is hinted handoff and when is it used?
›
Hinted handoff is used with sloppy quorums. When a write can't reach its "home" replicas (due to network partition), it's accepted by other available nodes instead. These nodes tag the write with a "hint" indicating which home node it belongs to. When the network heals and the home node becomes reachable, the node holding the hint forwards the write to the correct home node. This allows writes to succeed during partial failures, at the cost of temporarily having data in the wrong place. After hinted handoff completes, the home node has the data and strict quorum reads can see it.
7
What is monotonic reads and how do you implement it in a replicated system?
›
Monotonic reads guarantees that if you make multiple reads, you never see an older state than you've already seen — data doesn't appear to go "back in time." Without it, you might read from a fast follower (seeing a new comment), then read from a slow follower (the comment disappeared). Implementation: assign each user a consistent replica to read from — e.g., hash their user_id to a specific replica. They always read from that replica, so their view only moves forward. If that replica fails, they're re-routed to another, but start from the latest known position.
