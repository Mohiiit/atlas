Leaderless Replication (Dynamo-Style)

Some datastores abandon the concept of a leader entirely. Instead of funneling all writes through a single node, any replica can directly accept writes from clients. Amazon's in-house Dynamo system popularized this approach, inspiring open-source databases like Riak, Cassandra, and Voldemort.

In some implementations the client sends writes directly to multiple replicas. In others, a coordinator node forwards the write on the client's behalf — but crucially, that coordinator does not enforce any ordering. There is no concept of failover because there is no leader to fail.

Read Repair and Anti-Entropy

After an unavailable node comes back online, it needs to catch up. Dynamo-style datastores use two mechanisms:

Read Repair

When a client reads from multiple nodes and detects a stale response, it writes the newer value back to the stale replica.

  • Triggered on every read
  • Works well for frequently-read values
  • Infrequently-read data may stay stale forever

Anti-Entropy Process

A background process continuously scans for differences between replicas and copies missing data.

  • Runs independently of reads
  • Does NOT copy writes in any particular order
  • May have significant delay
  • Not implemented by all systems (e.g., Voldemort lacks it)

Warning: Reduced Durability

Without an anti-entropy process, values that are rarely read may be missing from some replicas indefinitely, because read repair only runs when the application actually reads the value.

Quorums for Reading and Writing

If there are n replicas, every write must be confirmed by w nodes, and every read must query at least r nodes. As long as w + r > n, at least one node in the read set must have the latest write. These are called quorum reads and quorum writes.

Key Insight

Reads and writes are always sent to all n replicas in parallel. The parameters w and r only determine how many responses we wait for before considering the operation successful. Think of w and r as the minimum number of "votes" required.

Limitations of Quorum Consistency

Even with w + r > n, stale reads can still occur. The quorum guarantee is weaker than it appears:

Edge Cases Where Quorums Fail

  • Sloppy quorum: w writes land on different nodes than the r reads — no guaranteed overlap
  • Concurrent writes: No clear ordering; if LWW is used, writes can be lost due to clock skew
  • Concurrent read + write: The write may appear on only some replicas; read result is undefined
  • Partial write failure: Write succeeds on some nodes but fails overall (< w); not rolled back on successful nodes
  • Node restoration: A failed node restored from a stale replica drops below w copies of the new value
  • Timing edge cases: Linearizability violations even when everything is "working correctly"
"Dynamo-style databases are generally optimized for use cases that can tolerate eventual consistency. The parameters w and r allow you to adjust the probability of stale values being read, but it's wise to not take them as absolute guarantees."

Sloppy Quorum ≠ True Quorum

A sloppy quorum is only a durability assurance: data is stored on w nodes somewhere. There is no guarantee a read of r nodes will see it until hinted handoff completes. It is like knocking on your neighbor's door when you're locked out — you have a place to stay, but your mail still goes to your house.

Default Settings in Practice

  • Riak: Sloppy quorums enabled by default
  • Cassandra: Sloppy quorums disabled by default
  • Voldemort: Sloppy quorums disabled by default

Detecting Concurrent Writes

In Dynamo-style databases, multiple clients can write to the same key concurrently, even with strict quorums. Events may arrive in different orders at different nodes due to variable network delays. If each node simply overwrites on every write request, nodes become permanently inconsistent.

The "Happens-Before" Relationship and Concurrency

Two operations have three possible relationships:

A → B

A happens before B.
B knows about / depends on A.

B → A

B happens before A.
A knows about / depends on B.

A || B

A and B are concurrent.
Neither knows about the other.

Diagram 9: Happens-Before vs Concurrency
Happens-Before (Causal)
A: INSERT user
↓ B knows about A
B: UPDATE user.name
B depends on A. There is a clear order.
The later operation overwrites the earlier one.
Concurrent (No Causal Link)
A: SET x=1
No knowledge of B
||
B: SET x=2
No knowledge of A
Neither knows about the other.
This is a conflict that must be resolved.

Concurrency ≠ Simultaneity

Two operations are concurrent if they are both unaware of each other, regardless of physical clock time. Due to network delays, operations separated by seconds (or even minutes) can be concurrent if neither could observe the other.

Last Write Wins (LWW) and Its Dangers

The simplest convergence strategy: attach a timestamp to every write, pick the biggest timestamp as the "winner," discard all others. Used as the only conflict resolution in Cassandra and optionally in Riak.

Advantages

  • Simple to implement
  • Achieves eventual convergence
  • No extra application logic needed

Dangers

  • Silent data loss: Concurrent writes reported as successful, but only one survives
  • Clock skew: Can discard even non-concurrent (causally later) writes
  • Durability violation: Client receives "success" for a write that will be discarded

Safe Usage Pattern

The only safe way to use LWW: ensure a key is written once and treated as immutable. For example, Cassandra recommends using a UUID as the key so each write operation targets a unique key, avoiding concurrent updates entirely.

Capturing Happens-Before with Version Numbers

To detect concurrent writes without losing data, the server maintains a version number per key. Here is the step-by-step algorithm:

Diagram 10: Version Number Algorithm (Single Replica)
1
Server maintains a version number for every key. Increments on each write. Stores new version alongside the value.
2
Client reads a key: Server returns all values not yet overwritten plus the latest version number. Client must read before writing.
3
Client writes a key: Must include the version number from the prior read AND merge all values received in that read.
4
Server receives write with version V: Overwrites all values with version ≤ V (they've been merged). Keeps all values with version > V (those are concurrent).
5
No version number included? The write is concurrent with everything — it won't overwrite anything, just becomes another sibling.

Siblings and Tombstones

Concurrent values are called siblings (Riak's terminology). Merging siblings is the client's responsibility. For shopping carts, taking the union of sibling sets works well. But for deletions, you cannot simply remove an item — you must leave a tombstone marker with a version number, or removed items will reappear when sibling sets are unioned.

Version Vectors: Multi-Replica Concurrency Detection

The single version number approach works for one replica. With multiple replicas accepting writes, we need a version number per replica per key. The collection of all replica version numbers is called a version vector.

Diagram 11: Version Vectors for Multi-Replica Detection
Single Replica: One Version Number

↓ Generalize to multiple replicas ↓
Multiple Replicas: Version Vector
Replica A
Key X → vA=3
Replica B
Key X → vB=5
Replica C
Key X → vC=2
Version Vector: [A:3, B:5, C:2]
How Version Vectors Work:
  • Each replica increments its own version number on every write it processes
  • Each replica tracks version numbers from all other replicas
  • Sent to clients on reads; clients send it back on writes (Riak calls this the causal context)
  • Allows the database to distinguish overwrites (one vector dominates) from concurrent writes (neither dominates)
  • Safe to read from one replica and write to another — siblings may be created, but no data is lost

Version Vector vs Vector Clock

The terms are sometimes used interchangeably, but they are subtly different. A version vector tracks versions per replica, while a vector clock tracks events per process. For comparing replica state, version vectors are the correct data structure. Riak 2.0 uses a refinement called a dotted version vector.

Chapter 5 Summary: Three Replication Approaches Compared

Replication serves four key purposes: high availability (keep running despite failures), disconnected operation (tolerate network interruptions), latency (keep data close to users), and scalability (handle more reads via replicas).

Consistency Models Under Replication Lag

  • Read-after-write consistency: Users always see their own submitted data
  • Monotonic reads: Once you've seen data at time T, you never see older data
  • Consistent prefix reads: Causally related events appear in the correct order

Stronger guarantees (e.g., linearizability) require transactions or consensus, covered in Chapters 7 and 9.

First-Principles Questions

1Why does w + r > n guarantee that a read will see the latest write, and what are two real-world scenarios where this guarantee breaks down?
▼

The pigeonhole principle: if w nodes have the write and we read r nodes, and w + r > n, the read set and write set must overlap in at least one node that has the latest value. The client picks the highest-versioned response.

This breaks down when: (1) Sloppy quorums are used — the w writes may land on "neighbor" nodes outside the designated n, so the r read nodes have zero overlap with write nodes. (2) Concurrent writes with LWW — clock skew can cause a causally later write to receive a smaller timestamp, making it "lose" to an earlier write, effectively discarding data even though quorum was satisfied.

2Why is "concurrent" defined by mutual unawareness rather than physical clock time, and what consequence does this have for conflict detection?
▼

In a distributed system, clocks cannot be trusted to be perfectly synchronized (Chapter 8). Two operations separated by seconds on a wall clock can still be "concurrent" if network delays prevented either from observing the other. Conversely, two operations at the exact same millisecond are NOT concurrent if one causally depends on the other (e.g., a read followed by a dependent write).

The consequence: we cannot use timestamps for conflict detection. Instead, we use version numbers (single replica) or version vectors (multiple replicas) to track the causal "happens-before" relationship. Operations that are neither before nor after each other are concurrent and produce siblings that the client must merge.

3A sloppy quorum trades consistency for availability. Explain the exact mechanism: what is written where, and why does hinted handoff not immediately fix the problem?
▼

When a network partition prevents the client from reaching the designated "home" nodes for a key, a sloppy quorum allows the write to proceed on w reachable nodes that are NOT among the key's n home nodes. Each stand-in node stores the data along with a hint indicating which home node the data truly belongs to.

Once the partition heals, hinted handoff transfers the data from the stand-in nodes to the correct home nodes. However, until handoff completes, reads querying the home nodes will not see the write — even if w + r > n. The sloppy quorum only provides a durability guarantee (data is stored on w nodes somewhere), NOT a consistency guarantee. This window of inconsistency can be arbitrarily long if the partition persists.

4Why do version vectors need a version number per replica (not just one global version), and how does this enable safe cross-replica reads and writes?
▼

With multiple replicas accepting writes concurrently, a single global version number would require coordination (defeating the purpose of leaderless replication). Each replica independently increments its own counter, producing a vector like [A:3, B:5, C:2].

To compare two version vectors: if every component of vector V1 is ≥ the corresponding component in V2, then V1 dominates V2 (the V1 state includes everything in V2 — it can overwrite). If neither dominates, the operations are concurrent and must be kept as siblings.

This enables safe cross-replica operations: you can read from Replica A (receiving its version vector), then write to Replica B. B uses the version vector to determine what the write has "seen" and what is concurrent. Siblings may be created, but no data is lost as long as the application correctly merges siblings.
