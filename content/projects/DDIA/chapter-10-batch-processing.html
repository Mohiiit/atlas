DDIA — Part III: Derived Data

Chapter 10: Batch Processing

"A system cannot be successful if it is too strongly influenced by a single person. Once the initial design is complete and fairly robust, the real test begins as people with many different viewpoints undertake their own experiments." — Donald Knuth. This chapter explores how large-scale batch processing systems transform immutable inputs into derived outputs, drawing a line from Unix pipes to MapReduce to modern dataflow engines.

Three Types of Systems

Before diving into batch processing, the book distinguishes three fundamentally different styles of data processing systems, each optimized for a different use case:

System Classification by Processing Style
Services
Online Systems

Wait for client requests, handle them ASAP. Key metric: response time. Availability is critical. Examples: web servers, REST APIs, databases serving queries.

Batch Processing
Offline Systems

Takes a large bounded dataset, crunches it, produces output. No user waiting. Key metric: throughput (time to process N records). Runs periodically (e.g., daily). Examples: MapReduce, Spark jobs.

Stream Processing
Near-Real-Time

Between online and offline. Consumes unbounded input events shortly after they happen. Lower latency than batch but same produce-output pattern. Covered in Chapter 11.

Why Study Batch Processing?

Batch processing is an important building block for reliable, scalable, maintainable applications. MapReduce, published by Google in 2004, provided a clear model for why and how batch processing works at scale. Although its importance has declined in favor of newer dataflow engines, understanding it illuminates the design principles that carry forward.

Sorting vs. In-Memory Aggregation

In-Memory Hash Table (Ruby/Python)

  • Build a hash map: URL → count
  • Works if working set (distinct URLs) fits in memory
  • Simple, fast for small cardinality
  • Fails when distinct keys exceed available RAM

Sort-Based (Unix Pipeline)

  • Sort all URLs, then count adjacent duplicates
  • Scales to datasets larger than memory
  • Uses mergesort: sort chunks in RAM, write to disk, merge sorted segments
  • Sequential access patterns — great disk performance

The Unix Philosophy

The ease of composing Unix tools into powerful pipelines is no accident. Doug McIlroy, inventor of Unix pipes, described them in 1964: "We should have some ways of connecting programs like a garden hose."

Four Principles of Unix Philosophy (1978)

1. Do One Thing Well

Build afresh rather than complicating old programs with new "features". sort is a better sort than most standard libraries (handles disk spill, multi-thread).

2. Composability via stdout

Expect output of every program to become input to another, as yet unknown, program. Don't clutter output. Avoid interactive input.

3. Rapid Prototyping

Design and build software to be tried early, ideally within weeks. Don't hesitate to throw away clumsy parts and rebuild.

4. Tools Over Manual Work

Use tools to lighten a programming task, even if you have to build them first and throw some out after use.

What Makes Unix Composability Work?

Three Pillars of Unix Composability
Uniform Interface

Everything is a file (file descriptor): actual files, stdin/stdout, pipes, sockets, device drivers. All programs use the same interface. By convention: ASCII text, newline-separated records. This is what allows any program's output to connect to any other program's input.

Separation of Logic & Wiring

Programs use stdin/stdout without knowing where input comes from or output goes. The shell user wires things together. This is a form of loose coupling / inversion of control. The sort program doesn't care if it's talking to awk or your custom tool.

Transparency & Experimentation

Inputs are treated as immutable. You can rerun commands freely. You can pipe into less at any stage to inspect. You can save intermediate output to files. Easy to debug, easy to experiment.

The Biggest Limitation of Unix Tools

They run only on a single machine. And that's where tools like Hadoop come in — extending the Unix philosophy to distributed systems.

MapReduce and Distributed Filesystems

MapReduce is like Unix tools, but distributed across potentially thousands of machines. Like Unix tools, it is a brute-force but surprisingly effective tool. A single MapReduce job is comparable to a single Unix process: it takes inputs, produces outputs, does not modify the input, and has no side effects other than producing output.

"Where Unix tools use stdin and stdout, MapReduce reads and writes files on a distributed filesystem."

Key HDFS Properties

  • Shared-nothing: no special hardware, just commodity machines connected by datacenter network

  • Cost: much cheaper than equivalent dedicated storage appliance capacity

MapReduce Job Execution

The pattern of data processing in MapReduce mirrors the Unix log analysis example. There are four fundamental steps:

MapReduce: Four Steps
Step 1: Split

Read input files, break into records. (In the log example, each line is a record; \n is the separator.)

Step 2: Map

Call the mapper function to extract a key-value pair from each input record. (Like awk '{print $7}' extracting the URL.)

Step 3: Sort

Implicit in MapReduce — you don't write it. All key-value pairs are sorted by key before reaching the reducer. (Like the first sort command.)

Step 4: Reduce

Call the reducer to iterate over sorted key-value pairs. Multiple occurrences of the same key are adjacent → easy to combine without much state. (Like uniq -c.)

Mapper

Called once for every input record. Extracts key and value. May generate any number of key-value pairs (including none). Stateless — no state from one record to the next. Each record handled independently.

Reducer

Framework collects all values belonging to the same key and calls reducer with an iterator over that collection. Produces output records. The role of the mapper is to prepare data for sorting; the role of the reducer is to process sorted data.

Key Design Decision: Putting Computation Near the Data

The MapReduce scheduler tries to run each mapper on one of the machines that stores a replica of the input file block. This saves copying the input over the network. The framework also copies your application code (e.g., JAR files) to the machines where the map tasks will run.

Reduce-Side Joins and Grouping

In batch processing, a "join" means resolving all occurrences of some association within a dataset (not looking up one particular record). MapReduce has no indexes — it performs a full table scan. For analytic queries over large datasets, this is actually reasonable.

Sort-Merge Joins

The canonical example: joining a log of user activity events with a user profile database. One set of mappers extracts (user_id, activity_event), another extracts (user_id, date_of_birth). The framework brings all records with the same user ID together at the same reducer.

Why This Works So Well

  • The reducer processes all records for a user in one go — only needs one user record in memory
  • No network requests during reduce — all data pre-fetched by the shuffle
  • Secondary sort ensures user profile record comes before activity events
  • Mappers "send messages" to reducers: the key acts as a destination address
  • MapReduce separates physical network communication from application logic

GROUP BY

Grouping = Same Pattern as Joining

Set up mappers so key-value pairs use the desired grouping key. Partitioning and sorting bring all records with the same key to the same reducer. Common uses:

  • Counting records per group (COUNT(*))
  • Summing values (SUM(fieldname))
  • Top-k records by some ranking function
  • Sessionization: collating all activity events for a user session (using session cookie as grouping key)

Handling Skew (Hot Keys)

The "bring all same-key records together" pattern breaks down when a single key has disproportionately many records — linchpin objects or hot keys (e.g., a celebrity with millions of followers).

Strategies for Handling Hot Keys

Pig: Skewed Join

First runs a sampling job to detect hot keys. Then sends records for hot keys to random reducers (instead of one deterministic reducer). The other join input for hot keys is replicated to all those reducers.

Hive: Skewed Join Optimization

Hot keys specified explicitly in table metadata. Records for hot keys stored in separate files. Uses a map-side join for hot keys instead of reduce-side.

Two-Stage Aggregation

Stage 1: send records to random reducers, each produces partial aggregates. Stage 2: combine partial aggregates into final result. Distributes the load across reducers.

Map-Side Joins

Reduce-side joins are general (no assumptions about input data), but expensive due to sorting, shuffling, and merging. If you can make assumptions about your input, map-side joins are much faster: no reducers, no sorting. Each mapper reads one input file block and writes one output file.

Broadcast Hash Join

One input is small enough to fit in memory. Each mapper loads the small input into a hash table, then scans the large input, looking up each record. The small input is "broadcast" to all partitions of the large input.

Pig: "replicated join", Hive: "MapJoin"

Partitioned Hash Join

Both inputs partitioned the same way (same key, same hash function, same number of partitions). Each mapper loads only its partition of the small input. Much less data per mapper.

Hive: "bucketed map join"

Map-Side Merge Join

Both inputs partitioned and sorted by the same key. Mapper does incremental merge (like a reducer merge). Input doesn't need to fit in memory — reads both files in ascending key order.

Requires prior MR jobs to prepare sorted inputs

The Output of Batch Workflows

What is the result of all this processing? Batch process output is not a report for a human (that's analytics). It's typically a derived data structure:

Building Search Indexes

Google's original use of MapReduce. Mappers partition documents, each reducer builds the index (term dictionary + postings lists) for its partition. Index files are immutable once written. To update: rerun the entire indexing workflow and replace old index files wholesale.

Key-Value Stores as Batch Output

Machine learning models, recommendation systems, people-you-may-know. Build a brand-new database inside the batch job, write files to HDFS, bulk-load into read-only servers (Voldemort, Terrapin, ElephantDB, HBase). Server atomically switches from old to new files.

Why NOT Write Directly to a Database from MapReduce

  • Network request per record is orders of magnitude slower than batch throughput
  • All mappers/reducers writing concurrently can overwhelm the database
  • Breaks MapReduce's all-or-nothing guarantee: partial results become externally visible; you lose the ability to safely retry failed tasks

Philosophy of Batch Process Outputs

MapReduce Inherits the Unix Philosophy

Inputs Are Immutable

Never modify input data. If you introduce a bug, simply roll back code and rerun — output will be correct again. This is human fault tolerance.

No Side Effects

Output is the only externally visible result. Failed tasks are discarded. Framework can safely retry. Output is completely replaced each run.

Easy Rollback

Keep old output in a different directory, switch back if needed. Feature development proceeds faster when mistakes are reversible.

Separation of Concerns

Jobs separate logic from wiring (input/output directories). Same files can be used as input to many different jobs, including monitoring jobs.

Comparing Hadoop to Distributed Databases (MPP)

MapReduce was not new algorithmically — MPP databases (Teradata, Gamma, Tandem NonStop SQL) had implemented parallel join algorithms for over a decade. The biggest difference: MPP databases focus on parallel SQL; Hadoop + HDFS provides a general-purpose operating system that can run arbitrary programs.

Why MapReduce Writes to Disk So Eagerly

At Google, MapReduce tasks running at low priority have ~5% chance of being preempted per hour to make room for higher-priority processes. For a job with 100 tasks of 10 minutes each, there's a >50% chance at least one task is killed. MapReduce was designed to tolerate frequent termination — not because hardware is unreliable, but because preemption enables better resource utilization.

In open source clusters (YARN, Mesos, Kubernetes), preemption is less common, making MapReduce's design decisions less justified — hence the rise of dataflow engines.

Beyond MapReduce

MapReduce is robust but has fundamental performance problems. Implementing complex jobs with raw MapReduce APIs is laborious. Higher-level languages (Pig, Hive, Cascading, Crunch) helped, but the underlying execution model itself has issues.

Materialization of Intermediate State

In MapReduce workflows, the output of one job is written to HDFS, and the next job reads it. This process of writing intermediate state to files is called materialization. Compare with Unix pipes, which stream data incrementally using a small in-memory buffer.

Three Problems with Materializing Intermediate State

  • Straggler problem: A job can only start when all tasks in the preceding job complete. One slow task (straggler) delays the entire workflow. Unix pipes start all processes simultaneously.
  • Redundant mappers: Mappers in the next stage often just read back the file a reducer just wrote and re-partition it. The mapper code could have been part of the previous reducer — eliminating an entire read-write-read cycle.
  • Unnecessary replication: Intermediate state written to HDFS is replicated across several nodes — overkill for temporary data that only one downstream job needs.

Dataflow Engines: Spark, Tez, Flink

These engines fix MapReduce's problems by handling an entire workflow as one job, not independent sub-jobs. Functions are called operators and can be assembled in flexible ways (not just alternating map/reduce).

Advantages of Dataflow Engines

  • Sorting only where required, not between every map/reduce
  • No unnecessary mapper stages — mapper logic folded into preceding operator
  • Scheduler sees all data dependencies → can colocate producer and consumer on the same machine (shared memory buffer instead of network)
  • Intermediate state kept in memory or local disk, not replicated to HDFS
  • Pipelined execution — no waiting for entire preceding stage
  • JVM process reuse — less startup overhead
  • Same Pig/Hive/Cascading code can switch from MR to Tez/Spark with a config change

Fault Tolerance in Dataflow Engines

Spark: RDD Lineage

Tracks the resilient distributed dataset (RDD) ancestry: which input partitions were used, which operators applied. If a node fails, recompute lost partitions from lineage. Requires deterministic operators to avoid cascading recomputations.

Flink: Checkpointing

Periodically checkpoints operator state. On failure, resume from last checkpoint. Built around pipelined execution — incrementally passes output of one operator to the next without waiting for full input (like Unix pipes).

When to Materialize vs. Recompute

Recomputing is not always cheaper. If intermediate data is much smaller than source data, or computation is very CPU-intensive, it's probably cheaper to materialize to files than to recompute on failure. Deterministic operators reduce the blast radius of recomputation.

Graphs and Iterative Processing

Graph analysis in batch processing (e.g., PageRank, transitive closure) requires iterating until convergence — something plain MapReduce cannot express (single pass only). Naive approach: external scheduler loops over MR jobs, but MapReduce reads the entire graph every iteration even if only a small part changed.

The Pregel / BSP Model

The Bulk Synchronous Parallel (BSP) model, popularized by Google's Pregel paper, is an optimization for graph processing. Implemented in Apache Giraph, Spark GraphX, and Flink Gelly.

Pregel Computation Model

Core Idea: "Thinking Like a Vertex"

Each vertex can "send a message" to another vertex (typically along graph edges). In each iteration (superstep), a function is called for each vertex, passing all messages sent to it. Unlike MapReduce, vertices remember state between iterations. If no messages arrive, no work is done for that vertex.

→

or threshold met

Pregel vs. Actor Model

Similar: each vertex is like an actor communicating via messages. Different: Pregel has fixed-round timing (all messages from iteration N delivered in iteration N+1). Vertex state and messages are fault-tolerant and durable. Actors have no such timing guarantee.

The Partitioning Problem

Graph is partitioned across machines by vertex ID (often arbitrary). Related vertices may be on different machines → lots of cross-machine communication. Message overhead can exceed the graph size itself. If graph fits on one machine, a single-machine algorithm often outperforms the distributed version.

Fault Tolerance in Pregel

Periodically checkpoints all vertex state to durable storage at end of an iteration. On failure: roll back entire graph to last checkpoint and restart. If the algorithm is deterministic and messages are logged, selective partition recovery is possible.

High-Level APIs and Languages

As batch processing infrastructure matured, attention shifted to improving the programming model. Higher-level languages (Hive, Pig, Cascading, Crunch, Spark SQL) use relational-style building blocks: joins, grouping, filtering, aggregation.

The Move Toward Declarative Query Languages

  • Specifying joins as relational operators lets the framework automatically choose the best join algorithm (broadcast vs. partitioned vs. sort-merge)
  • Hive, Spark, and Flink have cost-based query optimizers that can even reorder joins to minimize intermediate state
  • Declarative simple operations (filter, select fields) enable column-oriented storage optimizations — read only required columns
  • Vectorized execution: tight inner loops friendly to CPU caches (Hive, Spark DataFrames, Impala)
  • Spark generates JVM bytecode; Impala uses LLVM for native code generation

Best of Both Worlds

By incorporating declarative aspects while retaining the ability to run arbitrary code, batch processing frameworks achieve MPP-like performance with Hadoop-like flexibility. They can read data in arbitrary formats, use any library (ML, NLP, image analysis), yet also benefit from query optimization when expressing joins and filters declaratively.

Specialization for different domains: Mahout (ML on MR/Spark/Flink), MADlib (ML inside MPP databases), spatial algorithms (k-nearest neighbors), genome analysis. Batch systems and MPP databases are converging.

Chapter Summary: Key Concepts

Two Main Problems Solved by Batch Processing Frameworks

Partitioning

In MapReduce, mappers partitioned by input file blocks, output repartitioned/sorted/merged into reducer partitions. Purpose: bring all records with the same key to the same place. Dataflow engines avoid sorting unless required.

Fault Tolerance

MapReduce: writes to disk eagerly, easy to retry individual tasks, slower when no faults. Dataflow engines: keep more in memory, recompute on failure (RDD lineage / checkpoints). Deterministic operators reduce recomputation scope.

1What are the three types of systems and how do they differ in their primary performance metric?▼
Services (online): primary metric is response time — handle requests as quickly as possible, availability is critical. Batch processing (offline): primary metric is throughput — time to crunch through a dataset of known, fixed size; no user is waiting. Stream processing (near-real-time): like batch but on unbounded input with lower latency; processes events shortly after they happen rather than on a fixed dataset.
2Why does the Unix sort-based approach scale better than an in-memory hash table for log analysis?▼
An in-memory hash table requires the entire working set (all distinct keys + counters) to fit in RAM. If the number of distinct keys exceeds available memory, it fails. The sort-based approach uses mergesort: sort chunks in memory, write sorted segments to disk, then merge them. It has sequential access patterns that perform well on disk. GNU sort automatically handles larger-than-memory datasets by spilling to disk and parallelizes across CPU cores. The sort approach scales to datasets far larger than available RAM.
3How does the Unix philosophy's principle of "separation of logic and wiring" apply to MapReduce?▼
In Unix, programs use stdin/stdout without knowing where input comes from or output goes — the shell user wires them together. MapReduce does the same: jobs are configured with input and output directories, and the job code doesn't know or care who produces its input or consumes its output. This separation enables reuse (one team writes the job, another decides where and when to run it) and loose coupling (jobs can be composed into workflows without changing their code). MapReduce also handles all network communication and fault tolerance transparently, so application code never touches the physical data movement layer.
4In a sort-merge join, why is "secondary sort" important and what problem does it solve?▼
In a sort-merge join (e.g., joining user activity events with user profiles), the reducer receives all records for a given user ID. With secondary sort, MapReduce arranges the records so the user profile record comes first, followed by activity events in timestamp order. This means the reducer can read the profile, store the date of birth in a local variable, then iterate over events — requiring only one user record in memory at a time. Without secondary sort, the reducer might see events before the profile and would need to buffer records, using more memory and complicating the logic.
5Why is writing directly to a production database from a MapReduce job a bad idea?▼
Three reasons: (1) Making a network request for every single record is orders of magnitude slower than batch throughput. (2) All mappers/reducers writing concurrently can overwhelm the database, degrading query performance for other users. (3) It breaks MapReduce's all-or-nothing guarantee: partial results from a failed job become externally visible, you lose the ability to safely retry or discard failed tasks, and you must worry about speculative execution producing duplicates. The better approach is to build a new database file within the job and bulk-load it atomically.
6What is the "hot key" problem in MapReduce joins, and how do Pig and Hive solve it differently?▼
A hot key (linchpin object) is a key with disproportionately many records (e.g., a celebrity with millions of followers). Since all records with the same key go to the same reducer, one reducer becomes a bottleneck, and the entire job waits for it. Pig's skewed join: runs a sampling job to detect hot keys, then sends hot-key records to random reducers (not one deterministic reducer), replicating the other join input to all those reducers. Hive's approach: hot keys are specified explicitly in table metadata, stored in separate files, and joined using a map-side join instead of reduce-side, avoiding the shuffle entirely for those keys.
7Under what conditions can you use a broadcast hash join vs. a partitioned hash join vs. a map-side merge join?▼
Broadcast hash join: one input is small enough to fit entirely in memory in each mapper. The small input is loaded into a hash table, the large input is scanned. Partitioned hash join: both inputs are partitioned the same way (same key, same hash function, same number of partitions), so each mapper only loads one partition of the small input — much less data per mapper, works even if the total small input doesn't fit in memory. Map-side merge join: both inputs are partitioned AND sorted by the same key. The mapper performs an incremental merge of both files in ascending key order, requiring no hash table at all — works regardless of input size.
8Why did Google design MapReduce to tolerate frequent task termination, and how does this differ from open-source Hadoop environments?▼
Google runs MapReduce at low priority in mixed-use datacenters. Higher-priority production services can preempt (kill) batch tasks to reclaim resources. A task running for an hour has ~5% chance of being terminated. For 100 tasks of 10 minutes each, there's a >50% chance at least one gets killed. So MapReduce writes eagerly to disk and supports task-level recovery — not because hardware is unreliable, but because preemption enables better resource utilization. In open-source environments (YARN, Mesos, Kubernetes), preemption is less common, making MapReduce's design choices less justified. This is one reason dataflow engines (Spark, Flink) that keep more data in memory perform better in these environments.
9What are the three key problems with materializing intermediate state in MapReduce, and how do dataflow engines solve them?▼
(1) Straggler problem: next job waits for ALL tasks in preceding job to finish. Dataflow engines pipeline execution — operators start as soon as their input is ready. (2) Redundant mappers: next-stage mappers just re-read and re-partition what a reducer just wrote. Dataflow engines fold mapper logic into the preceding operator, eliminating the read-write-read cycle. (3) Unnecessary replication: intermediate state on HDFS is replicated across nodes. Dataflow engines keep intermediate state in memory or local disk without replication. Additionally, dataflow engines sort only when needed (not between every stage), reuse JVM processes, and the scheduler can colocate operators that exchange data on the same machine.
10How does the Pregel (BSP) model differ from MapReduce for graph processing, and why is it more efficient?▼
MapReduce performs a single pass over data, so graph algorithms requiring iteration need an external scheduler to loop over MR jobs. Each iteration reads the entire graph even if only a small part changed. Pregel is fundamentally different: vertices remember state between iterations (supersteps) and communicate by sending messages along edges. In each superstep, only vertices that received messages do work. Messages from superstep N are delivered in superstep N+1 (fixed-round timing). This is more efficient because: unchanged parts of the graph do no work, vertex state persists in memory rather than being re-read from HDFS, and the programming model ("think like a vertex") naturally fits graph algorithms like PageRank.
11Why is determinism important for fault tolerance in dataflow engines, and what can accidentally introduce nondeterminism?▼
Dataflow engines recover from faults by recomputing lost data from still-available data (not by reading materialized intermediate state from HDFS). If a recomputed operator produces different output than the original lost data, downstream operators face contradictions between old and new data, requiring cascading restarts. Sources of accidental nondeterminism include: hash table iteration order (many languages don't guarantee order), probabilistic/statistical algorithms using random numbers, system clock usage, and external data source reads. To ensure reliable recovery, use fixed random seeds and avoid external state dependencies.
12How are batch processing frameworks and MPP databases converging, and what does each contribute?▼
They're converging from opposite directions. Batch processing frameworks (Hadoop/Spark/Flink) are adopting declarative, SQL-like APIs with cost-based query optimizers that automatically choose join algorithms, use column-oriented storage, and perform vectorized execution (Spark generates JVM bytecode, Impala generates native code via LLVM). MPP databases are becoming more programmable, supporting user-defined functions and broader data formats. Batch frameworks contribute flexibility: arbitrary code, arbitrary data formats, diverse processing models on the same cluster. MPP databases contribute query optimization and declarative semantics. The end result is systems that can run efficient SQL queries AND arbitrary ML/NLP/image processing code on the same data.
