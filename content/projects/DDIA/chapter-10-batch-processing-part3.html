Batch Processing Output Types

Batch jobs produce different kinds of output depending on the use case. The output type influences how the job is structured and what the consumer expects.

1. Building Search Indexes

MapReduce's original purpose at Google: building inverted indexes for web search. Crawled web pages in → inverted index out (term → list of doc IDs). The index is then served by a separate search engine.

Web crawl data (billions of pages)
↓
Mapper: tokenize → emit (word, doc_id)
↓ shuffle by word
Reducer: for each word → collect all doc_ids → write posting list
↓
Inverted index on HDFS (served by Lucene/Elasticsearch)

2. Key-Value Stores from Batch

Build a key-value store entirely from a batch job and load it into production. Eliminates the need for online writes. The batch job produces SST files (sorted string table files) that are directly importable into RocksDB, HBase, etc.

This is called "bulk loading." Advantages: no write amplification on the DB during import; the SST files are pre-sorted so compaction cost is minimal; the import is a simple file copy. Used by Facebook's data pipeline, LinkedIn, etc.

3. Machine Learning Models

Training: a batch job processes the entire training dataset, runs gradient descent iterations, outputs a trained model file. Serving: the model file is loaded by inference servers. The batch job is the training pipeline.

4. ETL and Data Warehouses

Extract-Transform-Load: batch jobs clean, normalize, and reshape raw data before loading into a data warehouse. The output is a set of relational tables in the warehouse, ready for analysts to query.

The "data lake" pattern: store raw data in HDFS/S3 in open formats. Run multiple different tools over it: SQL engines (Hive, Presto, BigQuery) for analytics; Spark for ML; MapReduce for batch ETL. The data is not locked into any proprietary format. This is the dominant enterprise data architecture today.

Graph Processing with Batch

Many problems (PageRank, social network analysis, shortest paths) are naturally graph algorithms. Running these iteratively in MapReduce is inefficient — each iteration is a separate MapReduce job.

Pregel Model (Bulk Synchronous Parallel)

Google's Pregel (and Apache Giraph, GraphX in Spark) uses the "think like a vertex" model:

Pregel BSP model:
1. Each vertex is a computing unit
2. Each superstep (iteration): every vertex receives messages from its neighbors, updates its state, sends messages to its neighbors
3. Repeat until no more messages are sent (convergence)
4. Vertex can "vote to halt" when done — woken up if it receives a message

PageRank in Pregel: each vertex (web page) sends its current rank / number of outgoing links to each neighbor. Neighbors sum up received ranks. Repeat until ranks stabilize.
Pregel is inefficient for graphs with high-degree vertices (e.g., social network celebrities with millions of followers). A single vertex receives millions of messages each superstep, creating a bottleneck. Partitioning schemes that keep high-degree vertices and their neighbors together help.

Declarative APIs: Pig, Hive, Spark SQL

Writing MapReduce jobs in Java is verbose. High-level declarative APIs compile to MapReduce or dataflow plans:

Apache Pig / PigLatin
Dataflow language. Write the steps as relations. Compiled to MapReduce jobs. Good for ETL-style pipelines.
Apache Hive / HiveQL
SQL-like language. Compiled to MapReduce or Tez. Enables SQL users to query HDFS data. Schema stored in Hive metastore.
Spark SQL / DataFrames
SQL or DataFrame API over Spark engine. Catalyst optimizer chooses join strategies, pushes filters, prunes columns. Very fast.

Chapter 10 Summary

Unix philosophy
Small composable tools, immutable input, text streams. The template for distributed batch processing. Scale = distribute the same ideas.
MapReduce
Map → shuffle → reduce. Fault tolerant via HDFS materialization. Simple model, high overhead per stage. Still used for massive-scale ETL.
Joins
Sort-merge join (both large), broadcast hash join (one small), partitioned hash join (pre-bucketed). Skew → salting. Combiner for aggregation.
Spark / Dataflow
Keep intermediates in memory. General DAG, not just map/reduce. Lineage for fault tolerance. 10–100× faster for iterative algorithms.
Output types
Search indexes, key-value stores (bulk load), ML models, ETL into warehouse. Batch is a pipeline: immutable input → transform → durable output.
Graph processing
Pregel BSP: "think like a vertex." Supersteps of message passing. GraphX, Giraph. Inefficient for high-degree nodes.

Review Q&A

1
What is the shuffle phase in MapReduce and why is it expensive?
›
The shuffle phase takes the output of all mappers (key-value pairs) and delivers to each reducer all pairs with the same key. This requires: (1) Each mapper writes its output to local disk, sorted by key; (2) Each reducer fetches ("pulls") its relevant key ranges from every mapper's local disk across the network; (3) The reducer merges the fetched data (sorted merge) to process keys in order. The shuffle is expensive because it involves: all-to-all network transfer (every mapper sends data to every reducer), disk I/O (mapper output is written to local disk before transfer), and memory for merge buffers. In a 1000-node cluster doing a join, the shuffle generates up to 1000×1000 = 1 million network connections. Combiners reduce shuffle volume by pre-aggregating on the mapper side. In Spark, the shuffle is still needed for groupBy/join operations but intermediates don't go through HDFS.
2
What is a combiner and when can you use one?
›
A combiner is a mini-reducer that runs locally on each mapper node after the map phase, before the shuffle. It applies the same aggregation function as the reducer to the mapper's local output, reducing the amount of data sent across the network during the shuffle. For word count: instead of sending (the, 1), (the, 1), (the, 1) across the network, the combiner reduces them to (the, 3) locally. Network traffic: 1 record instead of 3. Combiners can only be used when the aggregation function is: (1) Commutative (order of operands doesn't matter: a+b = b+a), (2) Associative (grouping doesn't matter: (a+b)+c = a+(b+c)). Valid: sum, count, max, min, set union. Invalid: average (you need the sum AND count separately; pre-averaging partial results and averaging those gives wrong answers). Combiners are optional optimizations — the correctness of the job doesn't depend on them running.
3
Compare sort-merge join, broadcast hash join, and partitioned hash join. When would you use each?
›
Sort-merge join (reduce-side): Both datasets are emitted with join key by mappers, shuffled together, and the reducer joins matching keys. Works for any size datasets. Cost: full shuffle of both datasets. Use when: both datasets are large (hundreds of GB to PB).

Broadcast hash join (map-side): Load the smaller dataset into a hash table in memory. Broadcast it to all mappers. Each mapper streams through the large dataset and probes the hash table. No shuffle needed. Cost: memory on every mapper for the small dataset. Use when: one dataset is small enough to fit in mapper RAM (typically <1GB). Much faster than sort-merge.

Partitioned hash join: Both datasets are pre-partitioned by the same hash of the join key into equal-sized buckets. Mapper for bucket K only needs to load bucket K of the small dataset — much less memory per mapper. No shuffle needed at join time. Cost: datasets must be pre-bucketed at write time. Use when: datasets are large but re-reading is expensive, and you can pay the cost of pre-bucketing once at write time.
4
Why is Spark significantly faster than MapReduce for iterative algorithms?
›
Iterative algorithms (machine learning, PageRank, graph algorithms) process the same dataset multiple times. In MapReduce: each iteration is a separate MapReduce job. Between jobs, the intermediate data is written to HDFS (disk) and read back. A 100-iteration training job does 200 HDFS reads/writes for the intermediate model state — plus the data reads each iteration. Disk I/O dominates. In Spark: the dataset (RDD) is loaded into memory once and cached (if it fits). Each iteration is a new set of transformations on the in-memory RDD. Intermediates stay in memory — no disk write between iterations. Only the final result is written out. Result: for iterative ML, Spark is typically 10–100× faster than MapReduce. For single-pass ETL jobs (where MapReduce excels), the speedup is more modest (2–5×) because the bottleneck is I/O from the input data, not intermediates. Spark's RDD lineage also provides fault tolerance: if a partition is lost, it's recomputed from its lineage, not from an HDFS checkpoint.
5
What is the "schema on read" approach and what are its trade-offs vs. schema on write?
›
Schema on write (traditional databases): The schema is defined before loading data. Data is validated and rejected if it doesn't conform at write time. The benefit: queries are faster (data is guaranteed to be well-formed); issues are caught early. The cost: schema changes require migrations; you need to know the schema before you have the data; if requirements change, old data may not fit the new schema.

Schema on read (Hadoop/data lakes): Data is stored in raw form (JSON, log files, Parquet with flexible schema). The schema is applied at query time — the query defines how to interpret the bytes. Benefits: store data now, figure out the schema later; different queries can impose different schemas on the same data; schema evolution is trivial (old and new schema can coexist in the same files). Costs: queries are slower (no optimization at write time); bad data is discovered only at query time; no guarantee of consistency across records. In practice: modern data lakes use semi-structured formats like Parquet with embedded schema metadata, getting benefits of both: efficient columnar storage with schema flexibility.
6
How does data locality work in MapReduce and why does it matter?
›
Data locality is the practice of scheduling computation close to where the data already lives, avoiding unnecessary network transfer. In HDFS, each block (128MB) is replicated across 3 DataNodes. The MapReduce scheduler knows the location of each block (from the NameNode) and the location of worker nodes. When scheduling a map task, the scheduler tries to run it on a DataNode that already holds the input block: (1) Same node: best — zero network transfer for input; (2) Same rack: good — only intra-rack transfer; (3) Different rack: worst — cross-rack transfer. Why it matters: in a 1000-node cluster processing 1PB, if every mapper had to fetch input over the network, the network bandwidth would be the bottleneck. By reading from local disk, each mapper processes at local disk speed (100–200MB/s), and only the smaller intermediate output (shuffled key-value pairs) traverses the network. This is why Google's insight "bring computation to data" (vs. "bring data to computation" in traditional RDBMS) was transformative for large-scale analytics.
