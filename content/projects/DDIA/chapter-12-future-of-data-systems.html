Data Integration

In complex applications, data is used in many different ways. There is unlikely to be one piece of software suitable for all the different access patterns, so you inevitably end up combining several specialized tools. The challenge is keeping them all consistent.

Combining Specialized Tools by Deriving Data

It is common to integrate an OLTP database with a full-text search index, analytics systems, caches, denormalized views, machine learning pipelines, and notification systems. As the number of representations increases, the integration problem becomes harder.

Reasoning About Dataflows

When copies of the same data must be maintained in several storage systems, you need to be clear about inputs and outputs: where is data written first, and which representations are derived from which sources?

The Single-Leader Principle

If you funnel all user input through a single system that decides on an ordering for all writes, it becomes much easier to derive other representations by processing writes in the same order. This is an application of state machine replication.

  • Use Change Data Capture (CDC) as the only way to update derived systems
  • The database is the only way of supplying new input -- derived views are entirely consistent with it
  • Avoid dual writes (writing directly to both DB and search index) -- leads to race conditions and permanent inconsistency
  • Updating derived data based on an event log can be made deterministic and idempotent

The Limits of Total Ordering

Constructing a totally ordered event log is feasible for small systems (single-leader replication), but limitations emerge at scale:

When Total Ordering Breaks Down

  • Throughput exceeds single node: Must partition the log. Order between partitions is ambiguous.
  • Multi-datacenter deployment: Separate leaders per datacenter. Cross-datacenter synchronous ordering is too slow.
  • Microservices: Each service has independent state. No defined order for events from different services.
  • Offline-capable clients: Client-side state updated immediately without server confirmation. Clients and servers see different event orders.

Total order broadcast is equivalent to consensus, and scaling consensus beyond one node's throughput remains an open research problem.

Ordering Events to Capture Causality

When there is no causal link between events, lack of total order is fine. The hard cases are subtle causal dependencies.

Causality Problem: The Social Network Example
Step 1: Alice unfriends Bob
→
Friendship service processes
Step 2: Alice posts about Bob
→
Message service processes
Bug: Notification service joins messages + friends
→
Sees message before unfriend → notifies Bob!

The causal dependency (unfriend happened before message) is lost because events are stored in different services.

Approaches to Capture Causality

  • Logical timestamps: Provide total ordering without coordination, but recipients must handle out-of-order events
  • Causal event references: Log the system state the user saw before their decision, and give it a unique ID. Later events reference that ID to record causal dependency.
  • Conflict resolution algorithms: Help with maintaining state, but don't help if actions have external side effects (like sending notifications)

Batch and Stream Processing

The goal of data integration is to make sure data ends up in the right form in all the right places. Batch and stream processors are the tools for achieving this -- consuming inputs, transforming, joining, filtering, aggregating, and writing to the appropriate outputs.

Maintaining Derived State

Functional Flavor of Data Pipelines

  • Batch processing encourages deterministic, pure functions -- output depends only on input, no side effects beyond explicit outputs, inputs are immutable, outputs are append-only
  • Stream processing extends this with managed, fault-tolerant state
  • Think in terms of data pipelines that derive one thing from another, pushing state changes through functional application code
  • Asynchrony makes systems robust: a fault in one part is contained locally, unlike distributed transactions which amplify failures
  • Secondary indexes that cross partition boundaries are most reliable when maintained asynchronously

Reprocessing Data for Application Evolution

Without reprocessing, schema evolution is limited to simple changes (adding optional fields). With reprocessing, you can restructure a dataset into a completely different model.

Problems with Lambda Architecture

  • Dual maintenance: Same logic must run in both batch and stream frameworks -- significant operational complexity
  • Output merging: Batch and stream produce separate outputs that must be merged. Easy for tumbling windows, hard for joins and sessionization
  • Incremental batch: Reprocessing entire history is expensive. Incremental batching adds complexity that defeats the purpose of keeping the batch layer simple

Unifying Batch and Stream Processing

Requirements for Unification

  • Replay historical events through the same engine that handles live streams (e.g., log-based message brokers can replay messages)
  • Exactly-once semantics: Ensure output is the same as if no faults occurred, even if they did (discard partial output of failed tasks)
  • Windowing by event time, not processing time -- processing time is meaningless when reprocessing historical events

Spark uses microbatching (stream on top of batch engine). Flink uses batch on top of stream engine. Apache Beam provides a unified API runnable on both Flink and Google Cloud Dataflow.

Unbundling Databases

At the most abstract level, databases, Hadoop, and operating systems all perform the same function: store data, process it, and query it. Unix and relational databases approached information management with very different philosophies -- and the tension has lasted for decades.

Unix Philosophy

Thin wrapper around hardware. Pipes and byte sequences. Small tools that do one thing well. Low-level, composable.

Relational DB Philosophy

High-level abstraction. SQL and transactions. Hides complexity of indexes, concurrency, crash recovery. Powerful but monolithic.

Composing Data Storage Technologies

A database internally provides secondary indexes, materialized views, replication logs, and full-text search. In the emerging architecture, these facilities are provided by different pieces of software, running on different machines, administered by different teams.

Creating an Index: The Replication Log Analogy

CREATE INDEX = Reprocessing

When you run CREATE INDEX, the database:

  • Scans a consistent snapshot of the table
  • Picks out field values, sorts them, writes out the index
  • Processes the backlog of writes since the snapshot
  • Continues keeping the index up to date on every write

This is remarkably similar to setting up a new follower replica and to bootstrapping CDC. The database is essentially reprocessing the existing dataset and deriving a new view.

Making Unbundling Work: Operators and Dataflow

Advantages of Log-Based Integration

  • System-level: Asynchronous event streams make the system robust to outages. The log buffers messages. Producer and other consumers run unaffected. Faulty consumer catches up when fixed.
  • Human-level: Different software components can be developed, improved, and maintained independently by different teams. Event logs provide an interface that is powerful enough for strong consistency but general enough for any kind of data.

What's Missing?

We don't yet have the unbundled-database equivalent of the Unix shell -- a high-level language for composing storage and processing systems declaratively. Imagine being able to write mysql | elasticsearch and have it continuously sync all changes, like an unbundled CREATE INDEX.

Designing Applications Around Dataflow

The "database inside-out" approach: compose specialized storage and processing systems with application code. Like spreadsheets -- when a cell changes, all formulas depending on it automatically recalculate. Data systems should do the same.

Write Path vs. Read Path
Write Path

Precomputed eagerly. Data flows through batch/stream processors to update derived datasets (indexes, caches, models). Like eager evaluation.

Read Path

Happens on demand when a user queries. Reads from derived datasets and constructs a response. Like lazy evaluation.

Derived Dataset
Where write path and read path meet -- the trade-off between write-time and read-time work

Stream Processors vs. Microservices

Both compose independent operators, but the communication mechanism differs. Example: currency conversion for purchases.

Microservices:

Query exchange-rate service synchronously at purchase time. Network request = latency + failure risk.

Dataflow:

Subscribe to exchange rate updates, store locally. At purchase time, query local DB only. Stream join. Faster and more robust.

"The fastest and most reliable network request is no network request at all!"

Designing for Correctness

Stateless services are easy to fix: restart and everything returns to normal. Stateful systems remember things forever, so if something goes wrong, the effects potentially last forever. We want to build applications whose semantics are well-defined even in the face of faults.

The End-to-End Argument for Databases

Just because your database provides serializable transactions does not mean your application is free from data loss or corruption. Application bugs can write incorrect data, and serializable transactions won't save you from that.

Exactly-Once Execution of an Operation

If processing fails, you can either drop the message (data loss) or retry (risk of duplicate execution). Processing twice is a form of data corruption -- billing a customer twice, incrementing a counter twice.

Exactly-once means arranging computation such that the final effect is the same as if no faults occurred, even if the operation was retried.

Duplicate Suppression

TCP suppresses duplicates within a single connection using sequence numbers. But this only works within that connection. When a TCP connection drops after COMMIT but before the client hears back, the client doesn't know if the transaction committed or not.

Operation Identifiers

End-to-End Idempotence via Unique IDs

Generate a unique operation ID (UUID) at the client and include it in the request (e.g., as a hidden form field). Pass it all the way to the database:

  • Insert the operation ID into a requests table with a UNIQUE constraint
  • If the insert fails (duplicate ID), the transaction aborts -- preventing double execution
  • Relational DBs maintain uniqueness correctly even at weak isolation levels
  • The requests table also acts as an event log (hinting toward event sourcing)

The balance updates don't even need to be in the same transaction -- they can be derived from the request event by a downstream consumer, as long as each event is processed exactly once.

"The function in question can completely and correctly be implemented only with the knowledge and help of the application standing at the endpoints of the communication system." -- Saltzer, Reed, and Clark, 1984 (The End-to-End Argument)

Uniqueness Constraints and the Log

Enforcing a uniqueness constraint requires consensus -- all nodes must agree on which of potentially conflicting operations goes first. The most common approach: route all events for a particular value through a single partition of the log, using the value as the partition key.

The Scaling Trade-off

  • If uniqueness needs to be enforced per-partition, it's straightforward -- a stream processor consumes events sequentially
  • If uniqueness needs to be enforced across partitions (e.g., unique usernames globally), you need cross-partition coordination -- expensive and limits throughput
  • Many real-world uniqueness constraints can be scoped to a single partition (e.g., per-user request dedup)

Timeliness and Integrity

Two properties often confused under the umbrella of "consistency" that are actually quite different:

Timeliness

  • Users see an up-to-date state of the system
  • Reading your own writes
  • Violation = seeing stale data
  • Typically annoying but not catastrophic
  • "Eventual consistency" is a timeliness guarantee (or lack thereof)

Integrity

  • No data corruption, no contradictions, no lost data
  • Every credit has a matching debit
  • Violation = permanent, catastrophic damage
  • Must not lose data, even in the face of faults
  • "Perpetual consistency" -- must hold at all times

The ACID-in-Dataflow Translation

Violations of timeliness are "eventual consistency." Violations of integrity are "perpetual inconsistency." You can often relax timeliness while maintaining strict integrity:

  • Integrity can be maintained via event-based dataflow without atomic commit, linearizability, or synchronous coordination
  • Many applications accept loose constraints on timeliness: "we sold more items than we have in stock? Apologize and compensate."
  • Coordination-avoiding data systems achieve better performance and fault tolerance while preserving integrity
  • The "cost of apology" is a business decision: fewer inconsistencies (more coordination) vs. fewer outages (less coordination)

Trust, but Verify

Our system models assume some things might fail while others won't -- processes crash but disks don't silently corrupt data. In reality, it is a question of probabilities.

Why Blind Trust Is Dangerous

  • Data can become corrupted on disk (bit rot) even while sitting untouched
  • Network corruption can sometimes evade TCP checksums
  • Rowhammer: Certain memory access patterns can flip bits in memory with no hardware fault
  • MySQL has had bugs failing to maintain uniqueness constraints. PostgreSQL's serializable isolation has exhibited write skew anomalies.
  • Application code has even more bugs than database code

A Culture of Verification

  • Auditing: Checking the integrity of data. Not just for finance -- every system should verify its own integrity.
  • HDFS and S3: Run background processes that continually read back files, compare to replicas, move files between disks to mitigate silent corruption
  • Restore from backups: Test regularly -- you may discover your backup is broken only when you need it most
  • Event sourcing + deterministic derivation: You can rerun batch/stream processors on the event log to verify you get the same result
  • Cryptographic auditing: Merkle trees (used in certificate transparency and blockchains) can efficiently prove data integrity
  • End-to-end integrity checks give confidence to move faster -- like automated testing for data systems

Doing the Right Thing: Ethics of Data Systems

Every system is built for a purpose; every action has both intended and unintended consequences. We, the engineers building these systems, have a responsibility to carefully consider those consequences.

"We talk about data as an abstract thing, but remember that many datasets are about people: their behavior, their interests, their identity. We must treat such data with humanity and respect. Users are humans too, and human dignity is paramount."

Predictive Analytics and Privacy

The "Algorithmic Prison"

  • Someone labeled "risky" by an algorithm may be systematically excluded from jobs, air travel, insurance, property rental, and financial services
  • Criminal justice presumes innocence until proven guilty, but automated systems can exclude without proof of guilt and with little chance of appeal
  • Algorithms learn from biased data and amplify that bias: "machine learning is like money laundering for bias"
  • Features correlated with protected traits (postal code → race) become proxies for discrimination
  • Predictive analytics extrapolate from the past -- if the past is discriminatory, they codify that discrimination

"Data and models should be our tools, not our masters."

1Why is dual writing (directly writing to both a database and a search index) dangerous?▼
When two clients send conflicting writes, the database and search index may process them in different orders, since neither is "in charge" of determining write order. This leads to permanent inconsistency between the two systems. The solution is to funnel all writes through a single system of record and use Change Data Capture (CDC) to propagate changes to derived systems in a consistent order.
2Why does the author argue that log-based derived data is superior to distributed transactions for data integration?▼
Distributed transactions (XA/2PC) have poor fault tolerance: if any participant fails, the entire transaction aborts, amplifying failures across the system. They also have poor performance due to blocking locks. Log-based derived data uses asynchronous event streams that buffer messages, contain faults locally, and allow each component to fail and recover independently. The log provides ordering guarantees while idempotent consumers provide exactly-once semantics. The trade-off is weaker timeliness (eventual consistency) but much better availability and fault tolerance.
3What are the fundamental limits of total ordering, and why can't we just use consensus for everything?▼
Total order broadcast is equivalent to consensus, and most consensus algorithms only work when a single node can process the entire stream. Four situations break this: (1) throughput exceeding a single machine requires partitioning the log, making cross-partition order ambiguous; (2) multi-datacenter deployments need separate leaders per DC due to network latency; (3) microservices have independent state with no defined cross-service ordering; (4) offline-capable clients see events in different orders than servers. Designing consensus that scales beyond one node remains an open research problem.
4What are the problems with the lambda architecture, and how does the unified approach solve them?▼
Lambda architecture runs batch (accurate, slow) and stream (approximate, fast) in parallel and merges outputs. Three problems: (1) same logic maintained twice in different frameworks -- significant operational complexity; (2) merging batch and stream outputs is hard beyond simple aggregations (joins, sessionization); (3) reprocessing entire history is expensive, and incremental batching adds complexity that defeats the batch layer's simplicity. The unified approach (e.g., Apache Flink, Beam) allows the same engine to replay historical events and process live streams, with exactly-once semantics and event-time windowing.
5What is the "meta-database of everything" concept, and how does it reframe batch/stream processing?▼
The dataflow across an entire organization looks like one huge database. Batch, stream, and ETL processes act like the database subsystem that keeps indexes or materialized views up to date -- they are elaborate implementations of triggers and stored procedures. Each derived data system (search index, cache, analytics warehouse, ML model) is like a different index type. Instead of these being features of a single integrated product, they are provided by different software running on different machines. CREATE INDEX in a single database is the same pattern as setting up CDC to populate an external search index.
6How does the end-to-end argument apply to database correctness, and why is TCP duplicate suppression insufficient?▼
TCP suppresses duplicates only within a single connection. If a client sends COMMIT but the connection drops before hearing back, it doesn't know if the transaction committed. Retrying opens a new TCP connection -- from the database's perspective, it's a new transaction, so TCP dedup doesn't help. Even worse, the user's browser may retry the HTTP POST entirely. The end-to-end argument states that duplicate suppression can only be correctly implemented with the help of the endpoints. Solution: generate a unique operation ID at the client, pass it all the way to the database, and use a uniqueness constraint to prevent double execution.
7What is the critical distinction between timeliness and integrity, and why does it matter for system design?▼
Timeliness = users see up-to-date state. Violation is annoying but usually not catastrophic (stale data). Integrity = no data corruption, no contradictions, no lost data. Violation is permanent and catastrophic. This distinction matters because you can often relax timeliness while maintaining strict integrity. Integrity can be maintained via event-based dataflow without atomic commit, linearizability, or synchronous coordination. Many applications accept temporary timeliness violations ("we oversold? apologize and compensate") as long as integrity is preserved, enabling coordination-avoiding systems with better performance and fault tolerance.
8How do coordination-avoiding data systems achieve both good performance and strong integrity?▼
Two key observations: (1) dataflow systems can maintain integrity on derived data without atomic commit or synchronous coordination, using deterministic derivation and idempotent operations; (2) many applications only need loose constraints that can be temporarily violated and fixed later (the "cost of apology" is a business decision). Together, these mean systems can operate multi-datacenter with async replication, with each datacenter independent -- weak timeliness (not linearizable) but strong integrity. Serializable transactions are still useful but scoped to small, local operations rather than heterogeneous distributed transactions.
9Why should we adopt a "trust, but verify" approach rather than blind trust in database guarantees?▼
Hardware isn't perfect: disks suffer bit rot, network corruption can evade TCP checksums, and rowhammer can flip memory bits without hardware faults. Software isn't perfect either: MySQL has failed to maintain uniqueness constraints, PostgreSQL's serializable isolation has exhibited write skew. Application code has even more bugs. The ACID culture led us to blindly trust technology and neglect auditability. With the shift to NoSQL and weaker guarantees, blind trust became more dangerous. Systems should continuously audit themselves: HDFS reads back files and compares replicas; you should test backups regularly; event sourcing enables rerunning derivations to verify results; Merkle trees enable cryptographic integrity proofs.
10What is the "write path vs. read path" framework, and how does it unify caches, indexes, and materialized views?▼
The write path is everything done eagerly when data arrives (precomputation), analogous to eager evaluation. The read path is everything done lazily when a user queries, analogous to lazy evaluation. Caches, indexes, and materialized views all serve the same purpose: they shift the boundary between write-path and read-path work. An index moves work from read time (full scan) to write time (index maintenance). A cache of common queries moves even more work to write time. No index at all means zero write-path work but expensive reads (grep). The optimal boundary depends on your workload. This is exactly the Twitter fan-out example from Chapter 1 -- full circle after 500 pages.
11How do predictive analytics create "algorithmic prisons" and self-reinforcing feedback loops?▼
When someone is labeled "risky" by an algorithm, they may be systematically excluded from jobs, insurance, loans, and housing -- an "algorithmic prison" with no proof of guilt and little appeal. This creates feedback loops: financial misfortune lowers credit score, which reduces employability, which worsens finances, which further lowers the score -- a downward spiral hidden behind mathematical rigor. Algorithms learn from biased historical data and amplify that bias. Features correlated with protected traits (zip code as proxy for race) become vehicles for discrimination. The system extrapolates the past rather than imagining a better future.
12What does the author mean by comparing data collection to surveillance, and why does this framing matter?▼
When a service stores data the user explicitly entered for processing, it's performing a service. When user activity is tracked as a side effect (especially in ad-funded businesses where advertisers are the real customers), the relationship becomes surveillance. The reframing matters because we have voluntarily built mass surveillance infrastructure that totalitarian regimes could only dream of -- microphones in every room via smartphones, smart TVs, and IoT devices. The "nothing to hide" argument ignores that not everyone is aligned with existing power structures. Combined with predictive analytics, tracking data determines insurance, employment, and freedom. Engineers bear ethical responsibility because "a technology is not good or bad in itself -- what matters is how it is used and how it affects people."
