DDIA -- Part II: Distributed Data

Chapter 7: Transactions

"Some authors have claimed that general two-phase commit is too expensive to support, because of the performance or availability problems that it brings. We believe it is better to have application programmers deal with performance problems due to overuse of transactions as bottlenecks arise, rather than always coding around the lack of transactions." -- Spanner paper (2012). Transactions group reads and writes into a logical unit, simplifying the programming model so applications don't have to worry about partial failure and concurrency.

The Slippery Concept of a Transaction

In the harsh reality of data systems, many things can go wrong: hardware/software faults mid-write, application crashes halfway through operations, network interruptions, concurrent writes overwriting each other, and race conditions causing surprising bugs. For decades, transactions have been the mechanism of choice for simplifying these issues.

What Is a Transaction?

A way for an application to group several reads and writes into a logical unit. Conceptually, all operations execute as one: either the entire transaction succeeds (commit) or it fails (abort/rollback). If it fails, the application can safely retry.

Transactions are not a law of nature. They were created to simplify the programming model. By using transactions, the application ignores certain error scenarios and concurrency issues because the database handles them instead (these are called safety guarantees).

Both extremes are hyperbole: "transactions are the antithesis of scalability" and "transactions are essential for serious applications." The truth is that transactions have advantages and limitations, and understanding the trade-offs requires examining the details.

The Meaning of ACID

The safety guarantees of transactions are described by the acronym ACID (coined in 1983 by Harder and Reuter). In practice, one database's ACID is not another's. ACID has unfortunately become mostly a marketing term. Systems that don't meet ACID criteria are sometimes called BASE (Basically Available, Soft state, Eventual consistency) -- which is even more vague.

Atomicity

In the context of ACID, atomicity is not about concurrency (that's isolation). Rather, atomicity describes what happens if a client wants to make several writes, but a fault occurs after some of the writes have been processed. If the writes are grouped into an atomic transaction, and the transaction cannot be completed (committed) due to a fault, then the transaction is aborted and the database must discard or undo any writes it has made so far. The ability to abort a transaction on error and have all writes from that transaction discarded is the defining feature of ACID atomicity. Perhaps abortability would have been a better term.

Without atomicity, if an error occurs after A is debited but before B is credited, the money vanishes. Atomicity ensures we never get stuck in a half-finished state.

Isolation

Most databases are accessed by several clients at the same time. If they are accessing the same database records, you can run into concurrency problems (race conditions). Isolation in the sense of ACID means that concurrently executing transactions are isolated from each other: they cannot step on each other's toes. The classic database textbook formalizes isolation as serializability: each transaction can pretend it is the only transaction running on the entire database, and the result is the same as if they had run serially. However, in practice serializable isolation is rarely used because of the performance penalty.

Durability

Durability is the promise that once a transaction has committed successfully, any data it has written will not be forgotten, even if there is a hardware fault or the database crashes. In a single-node database, this means write-ahead log or B-tree pages flushed to disk. In a replicated database, it means the data has been copied to some number of nodes. Perfect durability does not exist -- there is always a risk (all disks and all backups could be destroyed simultaneously). Replication + backups provide the strongest practical guarantees.

Single-Object and Multi-Object Operations

Atomicity and isolation apply when a single transaction makes multiple writes. Multi-object transactions are needed when you need to keep several pieces of data in sync: a foreign key reference, a denormalized document, or secondary indexes. Without them, error handling is much more complicated for the application developer.

Single-Object Writes

Storage engines almost universally aim to provide atomicity and isolation on the level of a single object (e.g., one row or one document). This is implemented using a write-ahead log for crash recovery, and a lock on each object for isolation. Some databases also provide an increment operation and compare-and-set, removing the need for read-modify-write cycles. These are not transactions in the usual sense of the word. A transaction is usually understood as a mechanism for grouping multiple operations on multiple objects into one unit of execution.

When Multi-Object Transactions Are Needed

  • Foreign keys: inserting into multiple tables with references between them
  • Denormalized data: updating a document and all its denormalized copies
  • Secondary indexes: every write to a record must also update its indexes

Without transactions, error handling becomes difficult. Without atomicity, you risk half-done state. Without isolation, you risk reading partial updates.

Handling Errors and Aborts

A key feature of a transaction is that it can be aborted and safely retried if an error occurred. Not all systems follow this philosophy -- leaderless replication datastores work on a "best effort" basis. Retrying an aborted transaction is simple but imperfect:

  • If the transaction actually succeeded but the network failed on the ACK, retrying causes duplication (need application-level dedup / idempotency)
  • If the error is due to overload, retrying makes it worse (use exponential backoff + jitter)
  • Retrying is pointless for permanent errors (constraint violation)
  • If the transaction has side effects outside the database (sending an email), those happen again on retry

Weak Isolation Levels

If two transactions don't touch the same data, they can safely run in parallel. Concurrency issues (race conditions) only come into play when one transaction reads data that is concurrently modified by another transaction, or when two transactions try to simultaneously modify the same data. Concurrency bugs are hard to reason about, hard to find by testing (only triggered by unlucky timing), and hard to reproduce. Serializable isolation means the database guarantees that transactions have the same effect as if they ran serially, but it has a performance cost. For that reason, it is common for systems to use weaker levels of isolation, which protect against some concurrency issues but not all.

Read Committed

The most basic level of transaction isolation. It makes two guarantees:

  1. No dirty reads: You only see data that has been committed. If another transaction has written data but not yet committed, you can't see it.
  2. No dirty writes: You only overwrite data that has been committed. If another transaction has written data but not yet committed, you wait or abort.

Snapshot Isolation and Repeatable Read

Read committed still has a problem: read skew (also called a non-repeatable read). You can see different parts of the database at different points in time during a single transaction. This is a problem for backups (which read the entire database) and for analytic queries and integrity checks.

Solution: Snapshot Isolation -- each transaction reads from a consistent snapshot of the database.

Snapshot isolation is the most common solution. Each transaction reads from a consistent snapshot of the database -- it sees all the data that was committed at the start of the transaction. Even if the data is subsequently changed by another transaction, each transaction sees only the old data from that particular point in time. This is implemented using multi-version concurrency control (MVCC).

Snapshot Isolation in Practice

  • Supported by PostgreSQL, MySQL/InnoDB, Oracle, SQL Server
  • PostgreSQL and MySQL call it "repeatable read" (though the SQL standard's definition of repeatable read is ambiguous)
  • Readers never block writers, writers never block readers
  • The database potentially keeps multiple different committed versions of an object: this technique is known as MVCC
  • Read committed can use a simplified version of MVCC: just keep two versions (committed and uncommitted). Snapshot isolation extends this to many versions.

Indexes and Snapshot Isolation

How do indexes work in a multi-version database? One option is to have the index simply point to all versions of an object and require an index query to filter out any versions not visible to the current transaction. PostgreSQL has optimizations to avoid index updates when different versions of the same object can fit on the same page. CouchDB, Datomic, and LMDB use append-only/copy-on-write B-trees that create a new copy of each modified page, with parent pages up to the root being copied and updated.

Preventing Lost Updates

The lost update problem can occur if an application reads some value from the database, modifies it, and writes back the modified value (a read-modify-write cycle). If two transactions do this concurrently, one of the modifications can be lost because the second write does not include the first modification. This is a very common problem.

Scenarios Where Lost Updates Occur

  • Incrementing a counter or updating an account balance
  • Making a local change to a complex value (e.g., adding an element to a list in a JSON document)
  • Two users editing a wiki page at the same time, each saving the whole page content

Stateless Approach

In replicated databases: Locks and CAS assume a single up-to-date copy of the data. Multi-leader or leaderless replication allow concurrent writes that create conflicting versions (siblings). Atomic operations work well if they are commutative (e.g., incrementing a counter). "Last write wins" (LWW) is the default in many replicated DBs but is prone to lost updates.

Write Skew and Phantoms

Write skew is a more subtle race condition than dirty writes or lost updates. It occurs when two transactions read the same set of objects, and then each updates different objects based on what they read. Neither is a dirty write (they update different objects), neither is a lost update (same reason), yet the outcome violates an application invariant.

More Examples of Write Skew

  • Meeting room booking: Two people book the same room for overlapping times (each checks no conflict, then inserts)
  • Multiplayer game: Two players move to the same position simultaneously
  • Claiming a username: Two users try to register the same username at the same time
  • Double-spending: Two transactions each check the account balance is sufficient, then both debit

Write skew is not automatically detected by most databases (except under true serializable isolation). Solutions: (1) true serializable isolation, (2) explicit locking of the rows that the transaction depends on using SELECT ... FOR UPDATE.

Serializability

Serializable isolation is regarded as the strongest isolation level. It guarantees that even though transactions may execute in parallel, the end result is the same as if they had executed one at a time, serially, without any concurrency. Thus, the database prevents all possible race conditions. Most databases that provide serializability today use one of three techniques: (1) literally executing transactions serially, (2) two-phase locking (2PL), or (3) serializable snapshot isolation (SSI).

Actual Serial Execution

The simplest way to avoid concurrency problems: remove the concurrency entirely. Execute only one transaction at a time, in serial order, on a single thread. This became feasible around 2007 (VoltDB, Redis, Datomic) due to: (1) RAM became cheap enough to keep the active dataset in memory, and (2) database designers realized that OLTP transactions are usually short and make only a small number of reads and writes.

Summary: Constraints of Serial Execution

  • Every transaction must be small and fast (no interactive multi-statement transactions)
  • The active dataset must fit in memory (accessing disk would stall the single thread)
  • Write throughput must be low enough to be handled by a single CPU core, or else data must be partitionable
  • Cross-partition transactions are possible but there is a hard limit on how many can be supported

Two-Phase Locking (2PL)

For about 30 years (approximately 1975-2005), there was only one widely used algorithm for serializability: two-phase locking (2PL). Note: 2PL is completely different from two-phase commit (2PC). In 2PL, several transactions are allowed to concurrently read the same object as long as nobody is writing to it. But as soon as anyone wants to write (modify or delete), exclusive access is required.

2PL Locking Rules

  • If Txn A has read an object and Txn B wants to write to it, B must wait until A commits or aborts (and vice versa)
  • If Txn A has written an object and Txn B wants to read it, B must wait until A commits or aborts
  • Key difference from snapshot isolation: Readers block writers AND writers block readers. In snapshot isolation, readers never block writers and writers never block readers.

Deadlocks can happen in 2PL, and databases automatically detect them. The performance cost of 2PL comes from both reduced concurrency (locks cause waiting) and from transaction aborts due to deadlocks.

The big downside of 2PL is performance. Transaction throughput and response times are significantly worse under 2PL than under weak isolation. This is partly due to the overhead of acquiring and releasing locks, but more importantly due to reduced concurrency: if two concurrent transactions try to do anything that may result in a conflict, one has to wait for the other. Unstable latencies and potential deadlocks make 2PL difficult for applications that require predictable response times.

Predicate Locks and Index-Range Locks

A predicate lock applies to all objects that match some search condition. If Txn A has read objects matching some condition (e.g., all bookings for room 123 between noon and 1 PM), then no other transaction can insert, update, or delete any object matching that condition until A commits. This prevents phantoms. However, checking for matching predicate locks is time-consuming.

Serializable Snapshot Isolation (SSI)

Serializable snapshot isolation (SSI) is a fairly new algorithm, first described in 2008 by Michael Cahill, Uwe Rohm, and Alan Fekete. It provides full serializability with only a small performance penalty compared to snapshot isolation. It is used in PostgreSQL (since 9.1) and FoundationDB.

SSI Core Idea

SSI is an optimistic concurrency control technique. Transactions proceed without blocking, as in snapshot isolation. When a transaction wants to commit, the database checks whether anything bad happened (i.e., whether isolation was violated). If so, the transaction is aborted and has to be retried. Only transactions that executed serializably are allowed to commit.

Diagram 19 -- SSI: Detecting Writes That Affect Prior Reads

SSI tracks two things: (1) uncommitted writes that were ignored by MVCC reads (stale reads), and (2) writes after reads to the same data.

Detection Method 1: Stale MVCC Read
1. Txn 43 reads from snapshot, ignoring Txn 42's uncommitted write
2. Txn 42 commits
3. When Txn 43 wants to commit, DB checks if any of the ignored writes have since been committed
4. If yes: Txn 43 must abort
Detection Method 2: Write After Read
1. DB uses index-level tracking: when Txn 42 reads on_call = true, the index entry is noted
2. When Txn 43 writes to on_call (modifying an indexed value), DB checks if any other txn recently read from that index entry
3. DB notifies Txn 42 that its read may be outdated
4. Whichever txn commits second is aborted
The granularity of tracking matters. Detailed tracking (individual records) = fewer false aborts but more bookkeeping overhead. Coarse tracking (table-level) = more false aborts but less overhead. SSI strikes a balance using index-range tracking.

SSI Performance

  • One trade-off: transactions that read and write a lot of data are more likely to conflict and be aborted
  • SSI is less sensitive to slow transactions than 2PL or serial execution (no waiting for locks, no single-thread bottleneck)
  • The rate of aborts significantly affects overall performance -- if contention is high, aborts waste work
  • SSI is probably the future of serializable isolation: it performs well and still provides the strongest guarantee

First-Principles Questions

1Why does the "C" in ACID not really belong? What is the fundamental difference between Consistency and the other three properties?
▼

Atomicity, Isolation, and Durability are properties of the database -- they are guarantees that the database provides regardless of what the application does. Consistency, however, is a property of the application. It means that certain invariants (e.g., credits equal debits, foreign keys are valid) are always maintained. The database cannot guarantee this on its own -- it depends on the application writing correct transaction code. The database can help (e.g., with foreign key constraints, uniqueness constraints), but the application defines what "consistent" means. This is why Kleppmann calls it "the odd one out."

2Why do databases implement read committed by remembering old values rather than using read locks? What is the fundamental trade-off?
▼

Using read locks would mean that every reader blocks every writer and vice versa. This is terrible for performance in a typical workload where reads vastly outnumber writes. A single long-running read (e.g., a backup or analytics query) could block all writes for its entire duration. Instead, databases keep the old committed value alongside the new uncommitted value. Readers see the old value, writers hold locks only against other writers. This means reads never block writes and writes never block reads -- a massive concurrency improvement at the cost of a small amount of extra memory.

3In MVCC, why is an UPDATE implemented as a DELETE + INSERT of a new version rather than an in-place modification? What deeper principle does this serve?
▼

If you modify a row in place, any transaction reading from an older snapshot would see the new data, breaking snapshot isolation. By treating an UPDATE as a soft-delete of the old version (marking it with deleted_by = current txn ID) and an INSERT of a new version (with created_by = current txn ID), the database can maintain multiple versions simultaneously. Each transaction's snapshot sees exactly the versions that were committed at its start time. This is the fundamental principle of MVCC: immutability over mutation. Old versions are garbage-collected only when no running transaction could need them.

4Why can't snapshot isolation prevent write skew, even though it prevents lost updates in some databases? What is the structural difference between the two anomalies?
▼

Lost updates involve two transactions writing to the same object -- the second write overwrites the first. Snapshot isolation databases can detect this by tracking concurrent writes to the same row. Write skew involves two transactions writing to different objects, where each transaction's write invalidates the premise of the other transaction's read. Since they modify different rows, there's no direct write-write conflict for the database to detect. The conflict is between one transaction's read and another's write -- a much subtler relationship that requires tracking read-write dependencies, which snapshot isolation doesn't do (but SSI does).

5Why did literal serial execution only become feasible around 2007? What changed in hardware and workload understanding?
▼

Two things changed: (1) RAM became cheap enough to keep entire datasets in memory. Serial execution on a single thread is only fast enough if every transaction can complete without hitting disk I/O. In-memory databases made this possible. (2) Database designers realized that OLTP transactions are short -- they read/write only a small number of records and complete in milliseconds. This means the single-threaded bottleneck actually has decent throughput for OLTP workloads. Long-running analytics queries (OLAP) were moved to separate systems via snapshot reads. The combination of in-memory data + short transactions + stored procedures (no network round trips) made single-threaded execution surprisingly performant.

6What is the fundamental problem with phantoms, and why can't row-level locks solve it? How do predicate locks and index-range locks address this?
▼

The phantom problem is that a transaction's decision is based on a search condition (e.g., "how many doctors are on call?"), and another transaction inserts or deletes a row that would match that condition. Row-level locks can't help because the row doesn't exist yet at the time of the first transaction's read -- there's nothing to lock. Predicate locks solve this by locking the condition itself: "no one can insert, update, or delete any row matching WHERE on_call = true AND shift_id = 1234." Index-range locks are a practical approximation: instead of locking the exact predicate, they lock a range in the index that covers the predicate. This is coarser (may block some non-conflicting transactions) but much faster to check.

7Why is SSI described as "optimistic" concurrency control? In what conditions does it perform well vs poorly?
▼

SSI is optimistic because it doesn't block transactions from proceeding -- it lets them execute on their snapshots without any locking, and only checks for conflicts at commit time. If a conflict is detected, the transaction is aborted. This works well when contention is low (most transactions touch different data), because transactions rarely conflict and the abort rate is low. It works poorly under high contention (many transactions touching the same hot data), because many transactions are aborted and retried, wasting work. This is the classic optimistic-vs-pessimistic trade-off: optimistic wins when conflicts are rare, pessimistic wins when conflicts are frequent.

8Why is "repeatable read" a confusing term? What does it mean in the SQL standard vs what PostgreSQL/MySQL actually implement?
▼

The SQL standard's definition of isolation levels is based on surface-level anomaly descriptions that are ambiguous and imprecise. "Repeatable read" in the standard was supposed to prevent non-repeatable reads but the standard didn't account for snapshot isolation (which didn't exist when the standard was written). PostgreSQL and MySQL/InnoDB both call their snapshot isolation "repeatable read," but the implementations differ. PostgreSQL's repeatable read automatically detects lost updates (because it's actually doing snapshot isolation with some conflict detection), while MySQL/InnoDB's repeatable read does NOT detect lost updates. The SQL standard doesn't distinguish between them because it was written before these implementation techniques existed. This is why Kleppmann advises not relying on isolation level names and instead understanding what each database actually does.

9What is the relationship between 2PL and deadlocks? Is it possible to have 2PL without deadlocks?
▼

Deadlocks are inherent to 2PL because transactions acquire locks incrementally during the "growing phase" without knowing what locks they'll need in the future. If Txn A holds lock on X and wants Y, while Txn B holds lock on Y and wants X, neither can proceed. You could theoretically avoid deadlocks by having all transactions acquire locks in the same global order, but this is impractical because the set of objects accessed often depends on runtime values. So deadlock detection becomes a necessary part of any 2PL implementation. The database maintains a wait-for graph and periodically checks for cycles. When a deadlock is detected, one transaction (the "victim") is aborted. This is an inherent cost of pessimistic concurrency control -- you trade potential deadlock aborts for the guarantee that no transaction's work is wasted by optimistic conflict detection.

10Why does Kleppmann say that retrying aborted transactions is "not perfect"? What are the edge cases?
▼

Five problems: (1) If the transaction actually committed but the network lost the acknowledgment, retrying creates duplicate effects. You need application-level deduplication (idempotency keys). (2) If the error is due to overload, retrying makes the overload worse -- need exponential backoff with jitter. (3) Retrying is pointless for permanent errors like constraint violations -- only transient errors should be retried. (4) If the transaction has side effects outside the database (sending an email, charging a credit card), those happen again on retry unless you use two-phase commit or compensating transactions. (5) If the client process fails while retrying, any data it was trying to write is lost.

11How does materializing conflicts work as a technique? Why does Kleppmann call it a "last resort"?
▼

Materializing conflicts turns a phantom (a write that affects a search condition where no lockable row exists) into a lock conflict on a concrete row. For the meeting room example, you pre-create rows for every possible (room, time_slot) combination. Now when two transactions try to book the same slot, they both try to lock the same pre-existing row using SELECT FOR UPDATE, and one blocks. The reason it's a "last resort" is that it leaks concurrency control concerns into the application data model. You're creating rows not because the application needs them, but solely to serve as lock targets. This makes the schema harder to understand, pollutes the data model, and is error-prone (what if you forget to create rows for a new room?). Serializable isolation handles all of this automatically without polluting the data model.

12If you were designing a new database today, which serializable isolation technique would you choose and why? Consider the trade-offs of all three approaches.
▼

The answer depends on the workload, but for most general-purpose databases, SSI is the most compelling choice today. Here's the reasoning:

  • Serial execution is the simplest but limits throughput to what a single CPU core can handle. It requires all data to fit in RAM and only works well with stored procedures. It can't handle workloads with cross-partition transactions efficiently. Good for specific use cases (VoltDB, Redis) but too restrictive for general use.
  • 2PL provides strong guarantees and doesn't waste work, but has the worst performance characteristics: readers block writers, writers block readers, deadlocks require detection and victim selection, and a single slow transaction can bottleneck the entire system. Its latency profile is unpredictable.
  • SSI builds on snapshot isolation (proven performant and well-understood), adds conflict detection at commit time, and allows readers and writers to never block each other. The cost is potential aborts under high contention, but for most real-world workloads contention is moderate. PostgreSQL's adoption of SSI since version 9.1 has proven it practical. The main caution: monitor abort rates and optimize hot paths if contention becomes an issue.

Chapter Summary

Key Takeaways

  • Transactions are an abstraction that allows an application to pretend that certain concurrency problems and certain kinds of hardware and software faults don't exist. A large class of errors is reduced to a simple transaction abort, and the application just needs to retry.
  • Isolation levels: Read committed prevents dirty reads/writes. Snapshot isolation (MVCC) prevents read skew. Neither prevents write skew or phantoms in read-write transactions. Only serializable isolation prevents all race conditions.
  • Three approaches to serializability: (1) Literally serial execution -- simple but limited throughput. (2) Two-phase locking -- the traditional approach, pessimistic, poor performance. (3) Serializable snapshot isolation (SSI) -- optimistic, good performance, the most promising approach.
  • The chapter's implicit message: Weak isolation levels are a source of subtle bugs. If you can afford serializable isolation, use it. If you can't, understand exactly which race conditions your chosen isolation level doesn't prevent, and code defensively.
