DDIA — Part II: Distributed Data

Chapter 7: Transactions

"Some authors have claimed that general two-phase commit is too expensive to support, because of the performance or availability problems that it brings. We believe it is better to have application programmers deal with performance problems due to overuse of transactions as bottlenecks arise, rather than always coding around the lack of transactions." — Spanner paper (2012). Transactions are a mechanism for grouping reads and writes into a logical unit, simplifying the programming model so applications don't have to worry about partial failure and concurrency.

The Slippery Concept of a Transaction

In the harsh reality of data systems, many things can go wrong: hardware/software faults mid-write, application crashes halfway through a series of operations, network interruptions, concurrent writes overwriting each other, and race conditions causing surprising bugs. For decades, transactions have been the mechanism of choice for simplifying these issues.

What Is a Transaction?

A way for an application to group several reads and writes into a logical unit. Conceptually, all operations execute as one: either the entire transaction succeeds (commit) or it fails (abort/rollback). If it fails, the application can safely retry.

Transactions are not a law of nature. They were created to simplify the programming model. By using transactions, the application ignores certain error scenarios and concurrency issues, because the database handles them instead (these are called safety guarantees).

Both extremes are hyperbole: "transactions are the antithesis of scalability" and "transactions are essential for serious applications." The truth is that transactions have advantages and limitations, and understanding the trade-offs requires examining the details.

The Meaning of ACID

The safety guarantees of transactions are described by the acronym ACID (coined in 1983 by Harder and Reuter). In practice, one database's ACID is not another's. ACID has unfortunately become mostly a marketing term. Systems that don't meet ACID criteria are sometimes called BASE (Basically Available, Soft state, Eventual consistency) — which is even more vague.

"Consistency" Is Terribly Overloaded

  • Replica consistency — eventual consistency in async replication (Ch. 5)
  • Consistent hashing — partitioning approach (Ch. 6)
  • CAP theorem — means linearizability (Ch. 9)
  • ACID consistency — application-specific invariants always hold

The C in ACID was "tossed in to make the acronym work" (Joe Hellerstein). A, I, and D are properties of the database. C is a property of the application.

Single-Object vs. Multi-Object Operations

Atomicity and isolation describe what happens when a client makes several writes within the same transaction. Multi-object transactions are needed when pieces of data must be kept in sync.

Why Multi-Object Transactions Are Needed
Foreign key refs

Inserting rows that reference each other — all foreign keys must be correct and up to date, or data becomes nonsensical.

Denormalized data

Document DBs encourage denormalization. When denormalized info is updated, multiple documents must change together or data goes out of sync.

Secondary indexes

Without transaction isolation, a record might appear in one index but not another because the second index update hasn't happened yet.

Single-Object Writes

Storage engines provide atomicity (crash-recovery log) and isolation (object lock) at the single-object level. Some provide increment and compare-and-set operations. These are useful but are not transactions in the full sense — "lightweight transactions" is misleading marketing.

Multi-Object Transactions

Group multiple operations on multiple objects into one unit. In relational DBs: everything between BEGIN TRANSACTION and COMMIT on a TCP connection. Many NoSQL DBs abandoned these (hard to implement across partitions) — but nothing fundamentally prevents it.

Handling Errors and Aborts

A key transaction feature: abort and safely retry on error. ACID databases would rather abandon a transaction entirely than leave it half-finished. But retrying is not perfect:

Pitfalls of Retrying Aborted Transactions

  • Network ACK lost: Transaction succeeded but client thinks it failed → retry causes duplicate execution unless you have application-level deduplication
  • Overload-caused error: Retrying makes the problem worse. Use exponential backoff, limit retries, distinguish overload errors from others
  • Transient vs. permanent errors: Only retry on transient errors (deadlock, network blip). Constraint violations are permanent — retry is pointless
  • Side effects outside DB: Sending an email on each retry? Side effects don't roll back with the transaction. Need two-phase commit across systems
  • Client crash during retry: Any data the client was trying to write is lost

Many ORMs (Rails ActiveRecord, Django) don't retry aborted transactions — the error just bubbles up. This wastes the whole point of aborts.

Weak Isolation Levels

Serializable isolation has a performance cost. Many databases use weaker levels that protect against some concurrency issues but not all. These bugs have caused real financial losses, data corruption, and audit investigations. The answer is not "just use ACID" — even popular relational DBs use weak isolation.

Hierarchy of Isolation Levels
Serializable — prevents ALL race conditions
Snapshot Isolation / Repeatable Read — prevents dirty reads, dirty writes, read skew, (some) lost updates
Read Committed — prevents dirty reads, dirty writes
Read Uncommitted — prevents dirty writes only

Read Committed

The most basic useful isolation level. Default in PostgreSQL, Oracle 11g, SQL Server 2012. Two guarantees:

No Dirty Reads

You only see data that has been committed. If a transaction has written but not yet committed, other transactions see the old value. Prevents seeing partially updated state and data that gets rolled back.

No Dirty Writes

You only overwrite data that has been committed. If another transaction hasn't committed its write yet, you wait. Prevents mixing up writes from different transactions (e.g., car sale going to Bob but invoice to Alice).

Read Committed — No Dirty Reads Implementation

Approach: For every object being written, the DB remembers both the old committed value and the new uncommitted value.

Tx1 writes x = 3 (uncommitted)

Tx2 reads x → gets old value (2)

Tx1 commits

Tx2 reads x → now gets new value (3)

Why not use read locks?

A long-running write transaction would force all read-only transactions to wait. One slow write → cascading slowdown across the entire application. Instead: DB just serves the old committed value to readers.

What Read Committed Does NOT Prevent

It does not prevent the lost update problem (two concurrent counter increments), read skew (seeing inconsistent snapshots), write skew, or phantoms. For those, you need stronger isolation.

Snapshot Isolation and Repeatable Read

Even with read committed, you can see read skew (nonrepeatable reads): Alice has $1,000 split across two accounts ($500 each). A transfer of $100 is in progress. She sees one account before the transfer ($500) and the other after ($400) — it looks like $100 vanished. This is acceptable under read committed but unacceptable for backups and analytics.

Snapshot Isolation: The Solution

Each transaction reads from a consistent snapshot of the database — all data that was committed at the start of the transaction. Even if data is subsequently changed by other transactions, each transaction sees only the old data from that point in time.

Key performance principle: readers never block writers, and writers never block readers. Supported by PostgreSQL, MySQL/InnoDB, Oracle, SQL Server.

MVCC (Multi-Version Concurrency Control)

To implement snapshot isolation, the database keeps multiple versions of each object. Each version is tagged with the transaction ID that created or deleted it.

MVCC Implementation — How PostgreSQL Does It

Every row has:

FieldPurpose
created_byTransaction ID that inserted this row
deleted_byTransaction ID that deleted this row (initially empty)

Key mechanics:

  • Deletes don't remove rows — they mark deleted_by
  • Updates = delete + create (old row marked deleted, new row created)
  • Garbage collection later removes rows invisible to all active transactions
  • Each transaction gets a unique, always-increasing txid

Example: Tx 13 transfers $100 from Account 2

Old row (marked deleted)

Account 2 | balance=$500 | created_by=1 | deleted_by=13

New row (created)

Account 2 | balance=$400 | created_by=13 | deleted_by=—

Tx 12 (started before Tx 13) still sees balance=$500 because Tx 13's changes are invisible to it.

Visibility Rules for Consistent Snapshots

When a transaction reads from the database, transaction IDs decide what's visible:

The Four Visibility Rules

  1. At the start of each transaction, the DB lists all in-progress transactions. Any writes by those transactions are ignored, even if they subsequently commit.
  2. Any writes made by aborted transactions are ignored.
  3. Any writes made by transactions with a later transaction ID (started after current) are ignored, regardless of commit status.
  4. All other writes are visible.

Simplified: An Object Is Visible If Both Are True

  • The transaction that created it had already committed when the reader started
  • The object is not marked for deletion, or if it is, the deleting transaction had not yet committed when the reader started

Repeatable Read and Naming Confusion

The SQL standard doesn't define snapshot isolation (it predates the concept). Oracle calls their snapshot isolation "serializable." PostgreSQL and MySQL call it "repeatable read." IBM DB2 uses "repeatable read" to mean serializability. Nobody really knows what "repeatable read" means.

Preventing Lost Updates

The lost update problem occurs during read-modify-write cycles: two transactions read a value, modify it, and write it back. The second write clobbers the first. Examples: incrementing counters, updating account balances, editing wiki pages, adding elements to a JSON list.

Lost Update: Two Concurrent Counter Increments

Transaction 1

Read counter = 42

Set counter = 42 + 1

Write counter = 43

Transaction 2

Read counter = 42

Set counter = 42 + 1

Write counter = 43

Result: 43
Expected: 44
Tx1's increment is lost!

Four Solutions to Lost Updates

1. Atomic Write Operations

UPDATE counters SET value = value + 1 WHERE key = 'foo';

Implemented via exclusive lock on the object (cursor stability) or single-threaded execution. Best choice when applicable. ORMs can accidentally bypass these!

2. Explicit Locking (SELECT FOR UPDATE)

SELECT * FROM figures WHERE name='robot' FOR UPDATE;

App locks rows, performs read-modify-write, then commits. Useful when logic is too complex for atomic ops (e.g., game rule validation). Easy to forget a lock → race condition.

3. Automatic Detection

Allow parallel execution. If the transaction manager detects a lost update, abort and force retry. Works in PostgreSQL, Oracle, SQL Server. MySQL/InnoDB does NOT detect lost updates. Great because it's automatic — less error-prone than manual locking.

4. Compare-and-Set (CAS)

UPDATE wiki_pages SET content='new' WHERE id=1234 AND content='old';

Update only if value hasn't changed since last read. Caveat: if the WHERE reads from an old snapshot, CAS may not be safe. Check your DB's implementation!

In Replicated Databases

Locks and CAS assume a single up-to-date copy. Multi-leader/leaderless replication allows concurrent writes on different nodes. Instead: allow conflicting versions (siblings) and merge after the fact. Commutative operations (increment, set-add) work well across replicas (Riak 2.0 datatypes). Last-write-wins (LWW) is prone to lost updates — unfortunately, it's the default in many replicated DBs.

Write Skew and Phantoms

Write skew is a generalization of lost updates: two transactions read the same data, then update different objects based on what they read. Neither dirty write nor lost update, but still a race condition.

Write Skew: The Doctor On-Call Example

Hospital rule: at least one doctor must be on call. Alice and Bob are both on call. Both feel unwell and click "go off call" at the same time.

Alice's Transaction

SELECT count(*) FROM doctors WHERE on_call=true AND shift_id=1234
→ Returns 2

2 >= 2, so safe to proceed

UPDATE doctors SET on_call=false WHERE name='Alice'

COMMIT

Bob's Transaction

SELECT count(*) FROM doctors WHERE on_call=true AND shift_id=1234
→ Returns 2

2 >= 2, so safe to proceed

UPDATE doctors SET on_call=false WHERE name='Bob'

COMMIT

Result: 0 doctors on call!
Both saw count=2 (snapshot isolation), both proceeded. Invariant violated.

More Write Skew Examples

  • Meeting room booking: Two users check that a room is free at noon, both insert a booking → double-booked
  • Multiplayer game: Two players move different pieces to the same position, each checking it's empty first
  • Username claiming: Two users concurrently register the same username (unique constraint solves this one)
  • Double-spending: Two spending items inserted concurrently, neither sees the other, balance goes negative

A phantom is when a write in one transaction changes the result of a search query in another transaction. When the write adds a row (not modifies an existing one), SELECT FOR UPDATE can't lock anything because the row doesn't exist yet.

Materializing Conflicts

If there's no object to lock (phantom), artificially create lock objects. Example: create a table of all room/time-slot combinations. Transactions lock (SELECT FOR UPDATE) the relevant slots before inserting bookings.

This is ugly (leaks concurrency control into the data model), hard to get right, and should be a last resort. Serializable isolation is much preferable.

Serializability

Serializable isolation guarantees that the result of concurrent transactions is the same as if they had executed one at a time, serially. It prevents all race conditions. Three approaches exist:

1. Actual Serial Execution

The simplest approach: remove concurrency entirely. Execute one transaction at a time on a single thread. This became feasible around 2007 due to two developments:

Why It Became Feasible

  • RAM got cheap: active dataset fits in memory → transactions execute fast (no disk I/O waits)
  • OLTP transactions are short: small number of reads and writes. Long analytics queries can run on snapshot isolation separately

Used by: VoltDB/H-Store, Redis, Datomic

Constraints

  • Every transaction must be small and fast (one slow tx stalls everything)
  • Active dataset must fit in memory
  • Write throughput must fit on a single CPU core
  • Cross-partition transactions are possible but very slow
Interactive Transactions vs. Stored Procedures

For single-threaded serial execution, interactive multi-statement transactions are not allowed. The entire transaction logic must be submitted as a stored procedure — no human in the loop.

Partitioning for Scale

If each transaction only touches one partition, each partition runs its own serial thread → throughput scales linearly with CPU cores. But cross-partition transactions require lock-step coordination across all partitions and are orders of magnitude slower (~1,000 cross-partition writes/sec in VoltDB, vs. much higher single-partition throughput).

2. Two-Phase Locking (2PL)

For ~30 years, the only widely used serializability algorithm. Not to be confused with two-phase commit (2PC) — completely different things.

The Key Difference from Snapshot Isolation

Snapshot isolation: readers never block writers, writers never block readers.
2PL: writers block readers AND readers block writers. If Tx A read an object and Tx B wants to write it, B waits. If Tx A wrote an object and Tx B wants to read it, B waits (no reading old versions).

2PL — Shared and Exclusive Locks
ActionLock Required
Read an objectShared lock — multiple readers OK
Write an objectExclusive lock — no other locks allowed
Read then writeShared → Upgrade to exclusive

Two Phases:

Phase 1
Acquire locks
(while executing)
→
Phase 2
Release ALL locks
(at commit/abort)

Deadlocks

With many locks, Tx A waits for B's lock, B waits for A's lock. The DB auto-detects deadlocks and aborts one transaction. Under 2PL, deadlocks occur much more frequently than under weaker isolation. Each aborted transaction must redo all its work.

Performance

Significantly worse throughput and response times than weak isolation. Reduced concurrency by design. One slow transaction can grind the whole system to a halt. Unstable latencies, very slow at high percentiles. This is why 2PL hasn't been universally adopted since the 1970s.

Predicate Locks and Index-Range Locks

To prevent phantoms under 2PL, we need locks that apply to objects that don't yet exist.

Predicate Locks vs. Index-Range Locks

Predicate Lock (Conceptual)

Locks all objects matching a search condition, including future ones. E.g., "lock all bookings for room 123 between noon and 1pm."

Problem: Checking for matching locks among many active transactions is very slow. Not practical.

Index-Range Lock (Practical Approximation)

Attach a shared lock to an index entry. Approximates the predicate by locking a larger range (e.g., all bookings for room 123 at any time, or all rooms between noon-1pm).

Trade-off: Locks more than strictly necessary, but much lower overhead. If no suitable index exists, falls back to a table-level lock.

3. Serializable Snapshot Isolation (SSI)

A fairly new algorithm (2008, Michael Cahill's PhD thesis) that provides full serializability with only a small performance penalty over snapshot isolation. Used in PostgreSQL 9.1+ and FoundationDB.

Pessimistic vs. Optimistic Concurrency Control
🔒

2PL (Pessimistic)

If anything might go wrong, wait until safe. Like mutual exclusion in multithreading.

⛔

Serial (Extreme Pessimistic)

Exclusive lock on the entire DB for each transaction. Compensate by making transactions very fast.

✅

SSI (Optimistic)

Let transactions proceed without blocking. At commit time, check for conflicts. Abort if not serializable.

SSI's Core Insight: Detect Outdated Premises

A transaction reads data, decides what to do based on the result (a premise), then writes. If the premise is no longer true at commit time, the transaction must be aborted. Two cases to detect:

SSI Detection Case 1: Stale MVCC Reads

A transaction reads from an MVCC snapshot, ignoring uncommitted writes. If those writes commit before this transaction commits, the read was stale.

Tx 42

Read: on_call doctors
(sees Alice=true)

Write: Alice.on_call = false

COMMIT OK (first)

Tx 43

Read: on_call doctors
(sees Alice=true — but Tx42 changed it!)

Write: Bob.on_call = false

COMMIT ABORT
Tx42's ignored write now committed → stale premise!

Why wait until commit? If Tx 43 were read-only, no need to abort. Also, Tx 42 might still abort. By waiting, SSI avoids unnecessary aborts and preserves snapshot isolation's support for long-running reads.

SSI Detection Case 2: Writes Affecting Prior Reads

When a transaction writes to the DB, it checks indexes for other transactions that recently read the affected data. Instead of blocking (like 2PL), it sets a tripwire notification.

Tx 42 and Tx 43 both read rows matching WHERE shift_id=1234
→
DB records: "Tx42 and Tx43 read index entry shift_id=1234"
→
Tx 42 writes to shift_id=1234 → notifies Tx 43: "your read may be stale"
→
Tx 42 commits first → OK
→
Tx 43 tries to commit → Tx 42's conflicting write already committed → ABORT

SSI Performance Advantages

  • No blocking: Writers don't block readers, readers don't block writers (like snapshot isolation)
  • Predictable latency: No lock waits → much more stable query times than 2PL
  • Scales beyond single CPU: FoundationDB distributes conflict detection across machines, allowing serializable isolation across partitions
  • Read-only queries run on snapshots without locks — great for read-heavy workloads

Trade-off: read-write transactions should be fairly short. High contention → many aborts → wasted work. But still less sensitive to slow transactions than 2PL or serial execution.

First-Principles Questions

Test your understanding. Click to reveal the answer.

1Why does the book say the "C" in ACID doesn't really belong?▼
Atomicity, Isolation, and Durability are properties of the database. But Consistency (in the ACID sense) means that application-defined invariants hold before and after every transaction. This is the application's responsibility — the database can't guarantee your invariants if you write transactions that violate them. The "C" was "tossed in to make the acronym work" (Joe Hellerstein). The database provides A, I, and D as tools that help the application maintain consistency, but consistency itself is not a database guarantee.
2What is the difference between "atomicity" in ACID vs. in multithreading?▼
In multithreading, "atomic" means no other thread can see a half-finished result — it's about concurrency. In ACID, atomicity is NOT about concurrency (that's isolation). ACID atomicity means if a fault occurs mid-transaction, all writes are discarded — it's about abortability on error. The defining feature is: if the transaction is aborted, nothing it wrote takes effect, so it's safe to retry. A better term would be "abortability" rather than atomicity.
3Why doesn't read committed isolation prevent the "read skew" problem (Alice seeing $900 total)?▼
Read committed only guarantees you see committed data. Alice reads Account 1 ($500) — that's the committed value at that moment. Then the transfer commits. Then Alice reads Account 2 ($400) — that's also a committed value. Each individual read saw committed data, satisfying read committed. The problem is that Alice's two reads saw the database at different points in time. Read committed makes no promise about consistency across multiple reads. That requires snapshot isolation, which gives each transaction a frozen view of the database at one point in time.
4Explain MVCC's visibility rules — when is an object visible to a transaction?▼
An object is visible to a transaction if both conditions hold: (1) the transaction that created the object had already committed when the reading transaction started, and (2) the object is not marked for deletion, or if it is, the transaction that requested deletion had not yet committed when the reading transaction started. Additionally, any writes by transactions that were in-progress when the reader started are ignored (even if they subsequently commit), and any writes by transactions with a later transaction ID are ignored regardless of their commit status. This creates a frozen snapshot of the database at the start of the transaction.
5Why is "repeatable read" such a confusing term across databases?▼
The SQL standard was written in 1975 based on System R's isolation levels, before snapshot isolation was invented. It defines "repeatable read" which looks superficially similar to snapshot isolation. PostgreSQL and MySQL implement snapshot isolation but call it "repeatable read" to claim standards compliance. Oracle calls their snapshot isolation "serializable." IBM DB2 uses "repeatable read" to mean actual serializability. The SQL standard's definition is ambiguous and imprecise, implementations vary widely despite the same name, and the formal academic definition doesn't match most implementations. Nobody really knows what "repeatable read" means in practice — you must check each database's actual behavior.
6What is write skew, and why can't it be prevented by atomic operations or lost-update detection?▼
Write skew occurs when two transactions read the same data, make a decision based on it, then write to different objects. The doctor on-call example: both see 2 doctors on call, both go off call, result is 0. It's not a lost update (they wrote to different objects), so atomic single-object operations don't help. Automatic lost-update detection doesn't help either — PostgreSQL, Oracle, MySQL, and SQL Server all fail to detect write skew under their snapshot isolation levels. The only solutions are: (1) use true serializable isolation, (2) explicitly lock all rows the decision depends on with SELECT FOR UPDATE, or (3) use database constraints (limited applicability). Write skew is a generalization of lost updates to multiple objects.
7What is a phantom, and why does SELECT FOR UPDATE fail to prevent it?▼
A phantom is when a write in one transaction changes the result of a search query in another transaction. In the meeting room booking example, two transactions check for existing bookings (none found), then both insert a booking — double-booked. SELECT FOR UPDATE fails because it locks rows returned by the query. If the query returns zero rows (checking for absence), there are no rows to lock! The write adds a row that would match the search condition, but that row didn't exist when the lock was attempted. Solutions: materializing conflicts (create lock objects in advance), predicate locks (conceptually lock the search condition), or index-range locks (practical approximation).
8Why did single-threaded serial execution become feasible only around 2007, not earlier?▼
Two key developments: (1) RAM became cheap enough that many workloads can keep the entire active dataset in memory. When data is in RAM, transactions execute in microseconds instead of waiting for disk I/O, making single-threaded execution viable. (2) Database designers realized that OLTP transactions are short — small numbers of reads and writes. Long-running analytics queries can run separately on consistent snapshots via snapshot isolation. With stored procedures (no network round-trips mid-transaction) and in-memory data, a single thread can process transactions fast enough to match or exceed the throughput of concurrent systems that spend most of their time on locking overhead.
9How do index-range locks approximate predicate locks, and what is the trade-off?▼
A predicate lock locks all objects matching a query condition (e.g., room 123 between noon-1pm). This is precise but slow — checking for matching predicates among many active transactions is expensive. Index-range locks simplify this by attaching a shared lock to an index entry that covers a superset of the predicate. For example, locking the index entry for room 123 (all times) or the index range noon-1pm (all rooms). This is safe because any write matching the original predicate will definitely match the broader approximation. The trade-off: more objects are locked than strictly necessary (reduced concurrency), but the overhead is much lower. If no suitable index exists, the fallback is a shared lock on the entire table — safe but bad for performance.
10How does SSI detect serialization conflicts, and why does it wait until commit time to abort?▼
SSI detects two types of conflicts: (1) Stale MVCC reads — a transaction ignored another's uncommitted write (per MVCC rules). If that write commits before this transaction commits, the read was stale. The DB tracks which writes were ignored. (2) Writes affecting prior reads — when a transaction writes, it checks indexes for other transactions that recently read the affected data. Instead of blocking them (like 2PL), it notifies them that their read may be outdated. Why wait until commit? Three reasons: the transaction might be read-only (no risk of write skew), the conflicting transaction might abort (making the read valid after all), and waiting avoids unnecessary aborts. This preserves snapshot isolation's benefit of supporting long-running read-only queries.
11Why is 2PL's performance so much worse than snapshot isolation?▼
Three compounding factors: (1) Lock overhead — acquiring and releasing many fine-grained locks has a direct CPU cost. (2) Reduced concurrency — by design, any potentially conflicting transactions must wait. Writers block readers and vice versa (unlike snapshot isolation where they don't). One slow transaction holding locks can cascade delays to many others. (3) Deadlocks — much more frequent under 2PL than weaker isolation. Each deadlock wastes work (one transaction is aborted and must redo everything). The result: unstable latencies, with very slow responses at high percentiles. A single transaction that accesses many objects and acquires many locks can grind the entire system to a halt.
12Under what conditions does optimistic concurrency control (SSI) perform poorly, and when does it shine?▼
SSI performs poorly under high contention — when many transactions access the same objects. This leads to a high proportion of transactions needing to abort and retry, wasting work. If the system is near maximum throughput, retried transactions add load that makes things worse (a vicious cycle). SSI performs well when: there is spare capacity, contention is low to moderate, and read-write transactions are short. Commutative operations (e.g., counter increments) can reduce apparent contention. SSI's big advantages are: no lock waits (predictable latency), readers never block writers, and conflict detection can be distributed across machines (FoundationDB). It is less sensitive to slow transactions than 2PL or serial execution, because it doesn't hold locks while waiting.
