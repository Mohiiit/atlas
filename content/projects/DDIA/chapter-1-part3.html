Maintainability

The majority of the cost of software is not in the initial development — it's in ongoing maintenance: fixing bugs, keeping systems operational, investigating failures, adapting to new platforms, modifying for new use cases, repaying technical debt.

Yet many engineers dislike working on legacy systems. The term "legacy system" often signals pain. Much of this pain is avoidable, not inevitable. Good design makes the difference. Three principles guide maintainability:

Diagram 1 — The Three Pillars of Maintainability

Operability: Making Life Easy for Operations

Good operations can often work around limitations of bad software, but good software cannot compensate for bad operations. Operations teams are responsible for keeping the system running — monitoring health, tracking down root causes of failures, keeping software up to date, doing capacity planning, managing configs across environments.

What Good Operability Looks Like

  • Visibility into runtime behavior: good monitoring, dashboards showing internal system state, metrics that tell you what the system is doing and why
  • Standard tools and automation: support for standard automation and configuration management tools. Don't require operators to invent bespoke workflows
  • Avoiding single-machine dependencies: designed to continue working even if individual machines fail. Maintenance can be performed on one machine at a time without downtime
  • Good documentation: when an operator asks "if I do X, what will happen?" — the answer is in the docs, not buried in the codebase
  • Predictable behavior: minimize surprises. Avoid "magic" that works in unknown ways under the hood
  • Self-healing where possible: systems that can automatically restart failed components, not require manual intervention for every hiccup

Simplicity: Managing Complexity

As systems grow, they often become very complex. Complexity slows down everyone working on the system — harder to understand, harder to reason about, harder to test, harder to change safely. There are two very different kinds of complexity:

Diagram 2 — Essential vs Accidental Complexity

Essential Complexity

Complexity inherent to the problem being solved. Can't be removed without changing what the system does. A distributed database coordinating 1000 nodes is fundamentally complex.

Irreducible Part of the domain

Accidental Complexity

Complexity introduced by the implementation — state space explosion, inconsistent naming, tight coupling, workarounds built on workarounds. It's not required by the problem.

Avoidable Caused by implementation choices

The Best Tool Against Accidental Complexity: Abstraction

A good abstraction hides implementation details behind a clean interface. High-level programming languages are an abstraction over machine code. SQL is an abstraction over B-tree / LSM-tree internals. A good abstraction can be reused across many applications and reduces cognitive load — you reason about the abstraction, not its internals. Bad abstractions leak their internals (leaky abstraction), adding accidental complexity back in.

Diagram 3 — Symptoms of Accidental Complexity
Explosion of state space

The number of possible states the system can be in grows exponentially. Hard to test all of them. Hard to reason about what "should" happen.

Tight coupling

A change in one module requires changes in many other modules. The modules are entangled instead of isolated. Refactoring one thing breaks ten others.

Tangled dependencies

Module A depends on B, B depends on C, C depends on A. Circular dependencies make the system impossible to understand in isolation.

Inconsistent naming

The same concept is called three different names in three different modules. Engineers have to build a mental translation table constantly.

Hacks and special cases

Workarounds built on top of workarounds. "This case is handled specially in the billing module because of the 2019 migration." Nobody knows why anymore.

Evolvability: Making Change Easy

Requirements change. The system you build today will need to do something different tomorrow. A new feature request arrives. Legal requirements change. A platform you depend on reaches end of life. The business pivots. Evolvability is the system's capacity to absorb these changes with ease — also called extensibility, modifiability, or plasticity.

Evolvability is not about a specific tool or framework — it's about the organizational and engineering process: agile methodologies, test-driven development, refactoring, good abstractions. At the data system level, it connects directly to Chapter 4's themes of schema evolution and backward/forward compatibility.

First-Principles Questions

Click to reveal. Try to answer first.

1What is the difference between a fault and a failure? Why does the distinction matter?▼
A fault is when one component deviates from its spec. A failure is when the whole system stops providing the required service. The distinction matters because fault-tolerant systems are designed to prevent faults from cascading into failures. Hardware redundancy (RAID, hot-swap) accepts that components will fault and keeps the system running anyway. You cannot prevent all faults — hardware wears out, bugs exist, humans make mistakes — but you can design so that faults don't become visible failures.
2Why does Netflix deliberately inject faults using Chaos Monkey? Isn't this counter-productive?▼
Because fault tolerance that isn't exercised tends to rot. If your system's fault tolerance code paths are never triggered in production, they may silently break (due to other changes) and you won't discover this until a real failure occurs — at the worst possible time. Chaos Monkey deliberately kills random VMs in production to continuously verify that the system actually tolerates those faults. It's counter-intuitive but sound engineering: the only way to be confident your fault tolerance works is to test it regularly in a real environment.
3Why are software errors more dangerous than hardware faults, even though hardware fails more frequently?▼
Hardware faults are uncorrelated: one disk failing doesn't make another more likely. Redundancy (RAID, replicas) directly addresses this. Software errors are correlated: a bug affects all nodes running that software simultaneously. More redundant nodes just means more nodes all fail at the same time. Redundancy doesn't help. A software bug that causes a CPU spin will hit every node that gets the triggering input. This is why the leap second bug in 2012 took down hundreds of servers simultaneously — they all ran the same buggy kernel code.
4Why is the median (p50) response time insufficient for SLA purposes? What should you use instead?▼
The median tells you that half of requests are faster — but it ignores the worst half. SLAs care about user experience, and the users who experience the worst latency are often your most valuable customers (they have the most data, the largest accounts, the heaviest sessions). Use high percentiles: p95 (95% of requests are faster), p99, or p999. For systems with fan-out (a page triggers many backend calls), tail latency is amplified: with 100 parallel calls each at p99 = 1s, the page is slow 63% of the time. The p50 of individual calls is irrelevant to the user experience of the page.
5Twitter considered two approaches for the home timeline. What was the critical load parameter that determined which approach to use?▼
The critical parameter is fan-out ratio — the average number of followers per user. With low fan-out, approach 2 (fan-out on write, pre-computed caches) works well: writing one tweet pushes to relatively few caches. But celebrities have millions of followers — fanning out to 30 million caches per tweet takes minutes and creates enormous write amplification. Twitter's hybrid solution — fan-out on write for regular users, fan-out on read for celebrities — reflects the shape of the load distribution, not just the average. The load parameter that matters is not just "average fan-out" but the entire distribution including the extreme tail.
6When is vertical scaling (scaling up) a better choice than horizontal scaling (scaling out)?▼
Vertical scaling is better when: (1) the system is stateful — distributing state across nodes introduces coordination, consistency, and replication complexity that often outweighs the benefits; (2) the load is predictable and slowly growing — the complexity of distributed systems isn't needed yet; (3) operational simplicity is valued — one beefy machine is easier to operate than 20 coordinated nodes; (4) you're early in development and want to avoid premature optimization. Vertical scaling has limits (a single machine can only be so large) but modern machines are surprisingly powerful, and horizontal scaling introduces distributed systems problems (see Chapters 5–9) that are genuinely hard.
7What is the difference between essential complexity and accidental complexity? Give an example of each.▼
Essential complexity is inherent to the problem. Example: a distributed database that replicates data across 5 datacenters must handle network partitions, replication lag, and consensus protocols — this complexity cannot be removed without changing the product. Accidental complexity is introduced by the implementation. Example: a monolith where every module directly imports every other module, creating circular dependencies and making it impossible to change one module without touching ten others. Or: the same concept called "user_id", "userId", "uid", and "member_id" in different files, forcing constant mental translation. Accidental complexity can be reduced through better design, refactoring, and abstraction — without changing what the system does.
8What is head-of-line blocking and why does it make server-side percentile measurements misleading?▼
Head-of-line blocking occurs when a slow request occupies a server worker, causing subsequent faster requests to queue behind it. Even if the queued requests would take 1ms to process, they experience 1 second of latency because they're waiting behind one slow request. Server-side measurements only count the actual processing time — they miss the time spent waiting in queue. Client-side measurements capture the full wall-clock time including queueing. This is why SLA measurements must be done from the client's perspective, and why p99 latency seen by clients often looks much worse than what the server reports for the same requests.
9What does "data-intensive" mean vs "compute-intensive"? Why does this distinction matter for architecture?▼
A compute-intensive application's bottleneck is raw CPU processing speed — scientific simulations, rendering, encryption. Scaling solution: faster CPUs or more of them. A data-intensive application's bottleneck is data volume, data complexity, and data change velocity — the amount of data it stores, the rate it's updated, the complexity of the queries. Most business applications are data-intensive. The architecture must address: storing data reliably (databases), retrieving it quickly (indexes, caches), routing updates (message queues), processing it in bulk (batch/stream processing). Getting each of these right requires different specialized tools — this is why a "data system" is a composition of components, not a single database.
10Why is there no single "scalable architecture"? What determines the right architecture for a given system?▼
Architecture is a direct function of the system's specific load parameters. A system handling 100k RPS of 1KB payloads has totally different bottlenecks from a system handling 3 RPS of 2GB video uploads, even at the same overall throughput. The right architecture depends on: the ratio of reads to writes, data volume and growth rate, query complexity, access patterns (random vs sequential), latency requirements, and the specific "shape" of extreme load (celebrity users, viral events). A social network optimizes for fan-out. A time-series database optimizes for sequential writes and range queries. A full-text search engine optimizes for inverted index traversal. These architectures cannot be exchanged. The failure mode of premature scaling is not just wasted complexity — it's architecture optimized for the wrong parameters.
