Consistency Guarantees

Eventual Consistency = Convergence

If you stop writing and wait long enough, all replicas will eventually return the same value. But "eventually" says nothing about when. Until convergence, reads could return anything. This is hard for developers because a database looks like a variable, but a variable in a single-threaded program never gives you stale data.

Bugs from eventual consistency are subtle: the application works fine most of the time. Edge cases only appear during faults or high concurrency.

Transaction isolation is about avoiding race conditions from concurrent transactions. Distributed consistency is about coordinating replica state in the face of delays and faults. Related but mostly independent concerns.

Linearizability

The idea: make the system appear as if there is only one copy of the data, and all operations on it are atomic. Also known as atomic consistency, strong consistency, or external consistency. It is a recency guarantee: once a write completes, all subsequent reads must see that value.

What Makes a System Linearizable?

Linearizability Timeline — Register x (initially 0)

Each bar represents the duration of a client operation (from request sent to response received). The value must "flip" atomically at one point in time.

Client A
read(x) → 0
read(x) → 1
Client B
read(x) → 0
read(x) → 1 ✓
Client C
write(x, 1) → ok

Key rule: Once any read returns the new value (1), all subsequent reads — on any client — must also return 1. The value cannot "flip back." There must exist a single point in time when x atomically changes from 0 to 1.

The Atomic Flip Point

  • Reads before the write completes may return old or new value (concurrent)
  • Once any read returns the new value, all subsequent reads must return it too
  • Operations can be placed on a timeline; the markers must always move forward, never backward
  • A cas(x, v_old, v_new) operation atomically sets x to v_new only if current value equals v_old

When Is Linearizability Needed?

Locking & Leader Election

Single-leader replication needs exactly one leader. A distributed lock must be linearizable: all nodes must agree who holds the lock. ZooKeeper and etcd provide this via consensus algorithms.

Uniqueness Constraints

Unique usernames, unique file paths, no double-booking a seat — these all require a single up-to-date value that all nodes agree on. Similar to acquiring a "lock" on the chosen username.

Cross-Channel Timing

When two communication channels exist (e.g., file storage + message queue), without linearizability the faster channel can "overtake" replication, causing the consumer to see stale data.

If file storage were linearizable, step 4 would always see the new image. The message queue is faster than internal replication, causing a race condition.

Quorums Are NOT Enough for Linearizability

Even with strict quorums (w + r > n), a Dynamo-style system can violate linearizability. Client A reads new value 1 from a quorum, then Client B (reading after A) gets old value 0 from a different quorum. Making quorums linearizable requires synchronous read repair (performance penalty) and writer reads latest state first. But linearizable CAS still requires consensus.

CAP Theorem — Better Stated

CAP is often stated as "pick 2 of 3: Consistency, Availability, Partition tolerance." This is misleading. Network partitions will happen — they're not a choice. The real trade-off:

  • When the network is working: you can have both consistency and availability
  • When a network partition occurs: choose either linearizability (some nodes become unavailable) or availability (accept non-linearizable behavior)

Better phrasing: "Consistent or Available when Partitioned." CAP as formally defined is very narrow (only linearizability, only partitions) and has been superseded by more precise results. It is of mostly historical interest.

The Real Reason Most Systems Drop Linearizability: Performance

Even modern multi-core CPUs are not linearizable — each core has its own cache, asynchronously synced to main memory. The reason is performance, not fault tolerance.

Attiya & Welch proved: if you want linearizability, response time is at least proportional to network delay uncertainty. A faster algorithm for linearizability does not exist. Weaker models can be much faster.

Ordering Guarantees

Ordering is a recurring theme throughout the book: leaders determine write order (Ch. 5), serializability means sequential order (Ch. 7), timestamps attempt to order events (Ch. 8). There are deep connections between ordering, linearizability, and consensus.

Ordering and Causality

Causality imposes an ordering: cause comes before effect. A message is sent before received. The question comes before the answer. If a system respects this ordering, it is causally consistent.

Linearizability Is Stronger Than Causal Consistency

Linearizability implies causality — a linearizable system automatically preserves causal ordering. But linearizability is expensive (network delays, CAP trade-off).

Good news: Causal consistency is the strongest consistency model that does not slow down due to network delays and remains available during network failures. Many systems that seem to need linearizability actually only need causal consistency.

Lamport Timestamps

How to create a total order consistent with causality without a single leader? Use Lamport timestamps: a pair of (counter, nodeID).

Lamport Timestamps — How They Work
Node 1
counter: 1 → 6
Node 2
counter: 1 → 5
Client A
max seen: 5

Key mechanism: Every node and client tracks the maximum counter seen so far. Each request includes this max. When a node receives a counter greater than its own, it jumps forward to that value. This ensures causal ordering: every causal dependency increases the timestamp.

Lamport Timestamps Are Not Sufficient

They define a total order after the fact, but cannot tell you right now whether your operation is safe. Example: two users try to register the same username concurrently. Lamport timestamps can pick a winner after collecting all operations, but a node cannot decide in real-time without checking every other node.

To know when the total order is finalized, you need total order broadcast.

Lamport Timestamps vs. Version Vectors

  • Lamport timestamps: enforce a total order, but cannot distinguish concurrent from causally ordered operations. More compact.
  • Version vectors: can distinguish concurrent operations from causally dependent ones. Used for conflict detection in leaderless databases.

Total Order Broadcast

A protocol for exchanging messages between nodes, requiring two safety properties:

Reliable Delivery

If a message is delivered to one node, it is delivered to all nodes. No messages are lost.

Totally Ordered Delivery

Messages are delivered to every node in the same order. Once delivered, the order is fixed — no retroactive insertion.

Uses of Total Order Broadcast

  • Database replication: Each message = a write. Same writes in same order = consistent replicas. This is state machine replication.
  • Serializable transactions: Each message = a deterministic stored procedure executed in order.
  • Append-only log: Delivering a message = appending to the log. All nodes see the same sequence.
  • Fencing tokens: Lock acquisition requests are totally ordered; sequence number serves as fencing token (ZooKeeper's zxid).

All three are equivalent: if you can solve one, you can transform it into a solution for the others. This is a profound result.

The FLP Impossibility Result

Fischer, Lynch, and Paterson (1985) proved that no algorithm can always reach consensus if there is a risk of a node crash — in the purely asynchronous model with deterministic algorithms. But if you allow timeouts, suspected-crash detection, or randomness, consensus becomes solvable. Distributed systems achieve consensus in practice.

The Two Points of No Return

  • Participant votes "yes": It promises it can definitely commit later (but coordinator may still abort)
  • Coordinator writes decision: The commit/abort decision is irrevocable

These two promises together ensure atomicity. Single-node atomic commit lumps them into one event (writing the commit record).

The stuck participant holds locks the entire time. If coordinator's disk is lost, those locks may be held forever. The only escape: wait for coordinator recovery, or an admin manually resolves with a "heuristic decision" (which probably breaks atomicity).

Why "In Doubt" Is So Dangerous

  • Database transactions hold row-level exclusive locks on modified rows
  • With 2PL, shared locks on read rows too
  • These locks cannot be released until the transaction commits or aborts
  • If coordinator is down for 20 minutes, locks are held for 20 minutes
  • If coordinator's log is permanently lost: locks held forever until manual intervention
  • Even rebooting the database doesn't help — correct 2PC must preserve in-doubt locks across restarts

Three-Phase Commit (3PC)

A Theoretical Alternative

3PC was proposed as a nonblocking alternative to 2PC. However, it assumes bounded network delay and bounded node response times. In real systems with unbounded delays and process pauses, 3PC cannot guarantee atomicity.

Nonblocking atomic commit requires a perfect failure detector — impossible with unbounded network delays (timeout != reliable crash detection). So 2PC continues to be used despite its coordinator failure problem.

Distributed Transactions in Practice

Two quite different types are often conflated:

Database-Internal

All participants run the same database software (e.g., VoltDB, MySQL Cluster NDB). Can use optimized internal protocols. Often works quite well.

Heterogeneous (XA)

Participants are different technologies (two databases from different vendors, or a database + message broker). Must use lowest-common-denominator protocol. Much more challenging.

XA Transactions

X/Open XA (eXtended Architecture): A C API standard (1991) for 2PC across heterogeneous systems. Supported by PostgreSQL, MySQL, DB2, SQL Server, Oracle, ActiveMQ, HornetQ, IBM MQ.

  • Coordinator is typically a library in the application process (not a separate service)
  • If the app crashes, the coordinator goes with it → participants stuck in doubt
  • Coordinator logs on local disk = application servers are no longer stateless
  • Cannot detect deadlocks across systems, doesn't work with SSI
  • MySQL distributed transactions reported to be 10x slower than single-node

Exactly-Once Message Processing

The killer use case for heterogeneous transactions: atomically commit a message acknowledgment and the database writes from processing that message in a single transaction. If either fails, both abort, and the message broker redelivers. This gives effectively exactly-once processing — but only if all systems support the same atomic commit protocol.

Fault-Tolerant Consensus

A consensus algorithm must satisfy four properties:

The Four Properties of Consensus

Uniform Agreement

No two nodes decide differently.

Integrity

No node decides twice.

Validity

If a node decides value v, then v was proposed by some node. (Rules out trivially deciding null.)

Termination

Every non-crashed node eventually decides. (This is the liveness property = fault tolerance. 2PC violates this.)

The first three are safety properties (always hold, even during total failure). Termination is a liveness property (requires a majority of nodes to be working). Safety is maintained even if all nodes crash.

Consensus Algorithms

The Four Major Fault-Tolerant Consensus Algorithms

Paxos (Lamport, 1998)

The foundational algorithm. Notoriously difficult to understand and implement. Multi-Paxos optimizes repeated rounds. Used in Google's Chubby.

Raft (Ongaro, 2014)

Designed for understandability. Equivalent to Multi-Paxos but structured more clearly. Used in etcd, CockroachDB, TiKV.

Zab (ZooKeeper)

ZooKeeper Atomic Broadcast. Primary-backup protocol. Implements total order broadcast directly. Used in ZooKeeper, which backs Kafka, HBase, Hadoop.

VSR (Viewstamped Replication)

One of the earliest consensus protocols (1988, Oki & Liskov). Uses "view numbers" for epoch numbering. Historically important.

Consensus = Total Order Broadcast

Most consensus algorithms decide on a sequence of values (not just one), making them total order broadcast algorithms. Each round of consensus = one message delivery:

  • Agreement → all nodes deliver same messages in same order
  • Integrity → messages not duplicated
  • Validity → messages not corrupted or fabricated
  • Termination → messages not lost

Raft, VSR, and Zab implement total order broadcast directly. For Paxos, this optimization is called Multi-Paxos.

Limitations of Consensus

The Price You Pay

  • Synchronous replication: Proposals require majority vote before being decided → performance cost
  • Minimum nodes: Need 3 nodes to tolerate 1 failure, 5 to tolerate 2
  • Network partitions: Only the majority side can make progress; minority is blocked
  • Fixed membership: Most algorithms assume a fixed set of voting nodes. Dynamic membership extensions exist but are less well understood.
  • Timeouts and false alarms: Highly variable network delays cause false leader-death suspicions → frequent elections → system spends more time electing leaders than doing work
  • Network sensitivity: Raft can get stuck bouncing leadership between two nodes if one network link is unreliable. Designing algorithms robust to unreliable networks is still open research.

Membership and Coordination Services

ZooKeeper, etcd, and Consul are not general-purpose databases. They hold small amounts of slow-changing data (fits in memory), replicated via consensus. They "outsource" the hard parts of distributed coordination.

What ZooKeeper Provides

Linearizable Atomic Ops

Atomic compare-and-set to implement distributed locks. Lock = lease with expiry (released if client fails).

Total Ordering

All operations get a monotonically increasing zxid (transaction ID) and cversion. Can serve as fencing tokens.

Failure Detection

Long-lived sessions with heartbeats. If heartbeats stop, session declared dead. Locks via ephemeral nodes auto-released on session timeout.

Change Notifications

Clients can watch for changes (new nodes joining, sessions expiring). Avoids polling. Event-driven coordination.

Who Uses ZooKeeper?

  • Apache Kafka — broker coordination, partition assignment (historically; moving to KRaft)
  • Apache HBase — region server coordination
  • Hadoop YARN — resource manager election
  • OpenStack Nova — service coordination

ZooKeeper runs on a small fixed number of nodes (usually 3 or 5), performing majority votes among them while supporting potentially thousands of clients. It "outsources" consensus, operation ordering, and failure detection to an external service.

Service Discovery vs. Consensus

ZooKeeper is often used for service discovery (finding IP addresses of services). But service discovery doesn't truly require consensus — DNS is not linearizable and works fine with caching. Leader election requires consensus. Since ZooKeeper already knows who the leader is, it makes sense to also serve that info to others, using read-only caching replicas for non-linearizable reads.

Membership Services

Coupling failure detection with consensus lets nodes agree on which nodes are alive. A node may be incorrectly declared dead, but at least there is agreement. This enables reliable leader election: choose the lowest-numbered live member — but only works if everyone agrees on the membership.

First-Principles Questions

Test your understanding. Click to reveal the answer.

1What exactly does linearizability guarantee, and why is it called a "recency guarantee"?▼
Linearizability makes the system behave as if there is only one copy of the data and all operations are atomic. It's a "recency guarantee" because once any write completes, all subsequent reads must see that value (or a later one). The system appears to have a single, atomic point in time where each operation takes effect. No stale reads are allowed once a new value has been observed by any client.
2Why can't Dynamo-style quorums (w + r > n) guarantee linearizability?▼
Even with strict quorums, race conditions occur due to variable network delays. Client A may read the new value from one quorum, while Client B (reading after A) gets the old value from a different quorum — violating linearizability. Making quorums linearizable requires synchronous read repair (waiting before returning results) and writers reading latest state first, both with performance penalties. A linearizable compare-and-set operation is impossible without full consensus.
3What is the real trade-off in the CAP theorem, and why is "pick 2 of 3" misleading?▼
"Pick 2 of 3" is misleading because network partitions are not a choice — they happen whether you like it or not. The real trade-off: when the network is working, you get both consistency and availability. When a partition occurs, you must choose: (1) linearizability — some replicas become unavailable because they can't verify recency; or (2) availability — all replicas process requests, but behavior is not linearizable. Better phrased: "Consistent or Available when Partitioned." In practice, most systems drop linearizability for performance reasons, not just fault tolerance.
4What is the difference between causal consistency and linearizability, and why does it matter?▼
Linearizability provides a total order: every operation is placed on a single timeline, no concurrency. Causal consistency provides a partial order: causally related operations are ordered (cause before effect), but concurrent operations are incomparable. Linearizability implies causality, but causality does not require linearizability. Why it matters: causal consistency is the strongest model that doesn't slow down due to network delays and remains available during partitions. Many systems that seem to require linearizability actually only need causal consistency.
5Why are Lamport timestamps insufficient for implementing a uniqueness constraint, even though they provide a total order consistent with causality?▼
Lamport timestamps define a total order after the fact — once you have collected all operations, you can compare their timestamps. But when a node receives a username registration right now, it doesn't know if another node is concurrently registering the same username with a lower timestamp. To find out, it would need to check every other node (and if any node is unreachable, the system grinds to a halt). You need to know when the total order is finalized, which requires total order broadcast, which is equivalent to consensus.
6Why is two-phase commit (2PC) not a good consensus algorithm, despite achieving atomic commit?▼
2PC violates the termination property: if the coordinator crashes after participants vote "yes" but before sending the commit/abort decision, participants are stuck "in doubt" forever — they cannot unilaterally commit or abort. They hold locks, blocking other transactions. The coordinator is a single point of failure with no automatic recovery mechanism. By contrast, true consensus algorithms (Raft, Paxos) elect new leaders automatically, require only a majority (not unanimity), and define recovery processes. 2PC is a blocking atomic commit protocol.
7What is the "in doubt" problem in 2PC, and why can't you just restart the database to fix it?▼
When a participant has voted "yes" in phase 1 but hasn't received the coordinator's decision in phase 2, it is "in doubt." It cannot abort (another participant might have committed) or commit (the coordinator might have decided to abort). The participant holds locks on all rows it modified. Restarting the database doesn't help because a correct 2PC implementation preserves in-doubt transaction locks across restarts — otherwise it would risk violating atomicity. The only resolution is coordinator recovery or manual "heuristic decisions" by an administrator (which may break atomicity).
8How do epoch numbers and quorums solve the "who is the leader?" problem in consensus algorithms?▼
Consensus protocols don't guarantee a unique leader forever — instead, they guarantee a unique leader within each epoch (ballot/term/view number). When a leader is suspected dead, nodes vote to elect a new leader with an incremented epoch number. Before a leader can make decisions, it must get approval from a quorum (majority). The key insight: the quorum for electing a leader and the quorum for approving proposals must overlap — at least one node participates in both. This ensures no conflicting decisions happen across epochs, even if the old leader is still alive.
9Why are linearizable compare-and-set, total order broadcast, and consensus all "equivalent" problems?▼
They are equivalent in the formal sense that if you can solve any one, you can transform it into a solution for the others. (1) Given total order broadcast, you can build a linearizable CAS register (append claims to the log, first one wins). (2) Given a linearizable register with increment-and-get, you can build total order broadcast (use it to assign gap-free sequence numbers). (3) Total order broadcast is equivalent to repeated rounds of consensus (each round decides the next message). This equivalence is a profound result: all these seemingly different problems reduce to the same fundamental challenge.
10What does ZooKeeper actually provide, and why don't applications use it as a general-purpose database?▼
ZooKeeper provides: (1) linearizable atomic operations (locks via CAS), (2) total ordering (zxid as fencing tokens), (3) failure detection (sessions with heartbeats, ephemeral nodes auto-released on timeout), and (4) change notifications (watch for changes without polling). It's not a general-purpose database because it's designed for small, slow-changing coordination data that fits in memory (e.g., "who is the leader for partition 7"). Consensus is expensive, so you only run it on 3-5 nodes for coordination metadata, not for your application's runtime data.
11What are the practical limitations that prevent consensus algorithms from being used everywhere?▼
Five key limitations: (1) Performance: synchronous replication for every proposal (votes before deciding). (2) Minimum node count: need 3 for 1-fault tolerance, 5 for 2-fault. (3) Partition sensitivity: only the majority partition can progress; minority is blocked. (4) Fixed membership: adding/removing nodes is complex and less well understood. (5) Network sensitivity: variable delays cause false leader-death suspicions, leading to frequent elections that waste more time than doing useful work. Raft specifically can oscillate leadership on a single unreliable link.
12Why does the book say "in order to elect a leader, we first need a leader" — and how is this paradox resolved?▼
Single-leader replication implements total order broadcast (writes go through the leader in order). Total order broadcast is equivalent to consensus. But choosing a leader requires consensus. So: to solve consensus, we need a leader; to get a leader, we need consensus. The paradox is resolved by epoch numbering: consensus algorithms don't permanently fix a leader. They use monotonically increasing epoch numbers, with a leader unique to each epoch. The protocol can bootstrap without a leader and converge to one through voting. Leadership can change, but within each epoch, decisions are consistent. The "chicken and egg" problem is solved by making leader election itself part of the consensus protocol.
