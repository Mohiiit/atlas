Part 2: B-Trees, Comparison with LSM-Trees, and Other Index Structures

B-Trees

B-trees are the dominant index structure in relational databases (PostgreSQL, MySQL, Oracle, SQL Server). They've been standard since the 1970s. Unlike LSM-trees which break data into variable-size segments and write sequentially, B-trees break data into fixed-size pages (typically 4KB) and read or write one page at a time.

Pages are organized in a tree structure. Each internal (non-leaf) page holds keys and pointers to child pages. Leaf pages hold the actual values (or pointers to values).

The Branching Factor

The number of child references per internal page is called the branching factor. In real databases it's typically several hundred (because pages are 4KB–16KB and each entry is just a key + pointer). The depth of the tree is therefore:

4-level tree
branching factor 500
500³ = 125 million keys
5-level tree
branching factor 500
500⁴ = 62.5 billion keys
Most databases fit into 3–4 levels → read requires only 3–4 page lookups (I/Os)

Updating a Value in a B-Tree

Find the leaf page containing the key, update the value, and write the page back to disk (in place). The page reference from the parent remains valid.

Adding a New Key

Find the leaf page where the new key belongs. If there's space, write it in and done. If the page is full:

Full leaf page
→
Split into two half-full pages
↓
Parent page updated with new separator key + pointer to new page
↓
If parent is also full → split parent → propagate up → may split root → tree grows taller

Making B-Trees Reliable: The Write-Ahead Log (WAL)

B-tree writes are in-place updates. A single insertion can modify multiple pages. If a crash happens mid-operation, the tree can be left in an inconsistent state.

Danger: If you split a leaf page and the database crashes before writing the updated parent, the parent still points to the full old page. Data corruption.

Solution: Write-Ahead Log (WAL) — append every intended modification to a log on disk before applying it to the tree. On crash recovery, replay the WAL to restore consistency.

Write WAL entry
(append to log file)
→
Modify B-tree pages
(in-place on disk)
→
If crash: replay WAL
(reapply changes)

Optimizations for B-Trees

  • Copy-on-write: Instead of WAL, write a new page and atomically update the parent pointer (used by LMDB). Safer for concurrency.
  • Abbreviate keys: Internal pages only need enough key bytes to route correctly. Saves space, increases branching factor.
  • Sequential leaf access: Some B-trees keep leaf pages in sorted order on disk, or add sibling pointers, so range scans don't jump around.
  • Fractal tree / B+ tree: Common variant where only leaf pages store values; internal pages are pure routing structure.

B-Trees vs. LSM-Trees

This is one of the most practically important tradeoffs in database storage design.

Aspect B-Trees LSM-Trees
Write performance Each write may cause multiple page writes (split + WAL). Random I/O. Better — sequential writes to memtable + WAL. Compaction I/O is amortized.
Read performance Better for point lookups — guaranteed log(n) traversal, one location per key. May need to check multiple SSTable levels. Bloom filters help but add complexity.
Write amplification Each page written twice (WAL + in-place). Moderate. Each byte written many times during compaction. Higher write amplification.
Space amplification ~50% utilization — pages are never fully packed after splits. Old values persist until compaction. Worse space, but compaction eventually reclaims it.
Range queries Good — leaves often sequential on disk. Good — keys sorted within SSTables.
Compaction interference None — no background compaction. Compaction competes with reads/writes for I/O bandwidth. Can cause latency spikes.
Transactions Easier — each key exists in exactly one place → simpler locking. Keys may exist in multiple SSTables → complex locking.
Used by PostgreSQL, MySQL, Oracle, SQL Server LevelDB, RocksDB, Cassandra, HBase, ScyllaDB
LSM-trees typically sustain higher write throughput because they convert random writes into sequential I/O. B-trees offer more predictable read latency and simpler transaction semantics. Real-world choice depends on your workload.

Other Indexing Structures

Secondary Indexes

A primary index maps primary keys to rows. A secondary index maps some other field (or combination of fields) to row locations. Both B-trees and LSM-trees can serve as secondary indexes.

Primary Index
user_id → row
1 → {Alice, London}
2 → {Bob, Paris}
3 → {Carol, London}
Secondary Index on city
city → [row_ids]
London → [1, 3]
Paris → [2]
Query "all users in London" → secondary index → fetch rows 1 and 3 from primary storage

Storing Values: Heap File vs. Clustered Index

When a secondary index points to a row, it can store:

Approach Index stores Trade-off
Heap file Pointer to actual row location (heap file) Updates that don't move the row are cheap (just update in place). But if the row grows and must move, all indexes must be updated.
Clustered index The full row data (directly within the index) Read is faster — one lookup gets the data. But storage is duplicated for each index. InnoDB's primary key is always a clustered index.
Covering index Some columns stored in index, not all "Index-only scan" — queries that only need those columns never touch the heap. Good for common query patterns.

Multi-Column Indexes

A concatenated index on (last_name, first_name) sorts first by last name, then by first name within each last name. Useful for queries that filter on last_name alone, or on (last_name, first_name) together. Useless for queries that filter on first_name alone.

Geospatial example: finding restaurants near a point
Single-axis index
Index on latitude only → scan all restaurants at right latitude, then filter longitude. Inefficient.
R-tree / Space-filling curve
Both dimensions indexed simultaneously. PostGIS uses R-trees. Query is a single spatial lookup.
The Hilbert space-filling curve maps 2D coordinates to a 1D value that preserves spatial locality — allowing standard B-tree indexes for geographic queries.

Full-Text Search and Fuzzy Indexes

Standard indexes require an exact match. Full-text search allows querying for similar or approximate keys:

  • Edit distance: Find all keys within k character edits of the query
  • Inverted index: Map each word to the set of documents containing it
  • Trie / SSTable variant: Lucene (used by Elasticsearch) uses an SSTable-like structure where the in-memory index is a finite-state automaton over keys, allowing efficient fuzzy matching
Lucene's segment structure is very similar to SSTables with leveled compaction. Elasticsearch is essentially an LSM-tree optimized for text search. Full-text indexes are typically maintained separately from the main data store.

In-Memory Databases

The whole point of SSTable/B-tree complexity is managing disk I/O. What if the entire dataset fits in RAM?

Redis / Memcached
Pure in-memory. Durability via snapshots or AOF log. Restart reloads from disk. Accept data loss or async replication.
VoltDB / MemSQL / H-Store
In-memory relational DBs with durability via write-ahead log and replication. Full ACID.
Oracle TimesTen / RAMCloud
Research systems exploring in-memory databases with disk-based durability.

In-memory databases are faster not because they avoid disk reads — the OS page cache already caches frequently-read data. The speed comes from avoiding the overhead of encoding data into a disk-friendly format. They can use richer data structures (sorted sets, priority queues) that can't be efficiently stored on disk.

Anti-caching: When an in-memory database grows beyond available RAM, it can evict least-recently-used records to disk and reload them on access — similar to virtual memory but at the database level, which is more efficient.

Bloom Filters

LSM-trees suffer from one key read problem: to look up a key that doesn't exist, you must check every SSTable level (and find nothing). A Bloom filter is a compact probabilistic data structure that can answer "is this key definitely NOT in this SSTable?" very efficiently.

Query: is "elephant" in this SSTable?
→
Check Bloom filter (in RAM)
→
"Definitely NO"
→ skip this SSTable entirely
"Probably YES"
→ do the actual disk read
False negative rate: 0% (it never says "no" when the key IS there). False positive rate: tunable (maybe 1% at 10 bits per key). Trade memory for fewer disk reads.

Summary: Choosing a Storage Engine

If your workload is... Consider... Why
Write-heavy, time-series, event logs LSM-tree (RocksDB, Cassandra) Sequential writes, high throughput
Mixed reads/writes, OLTP B-tree (PostgreSQL, InnoDB) Predictable latency, simple transactions
Key-value with limited keys Hash index (Bitcask/Riak) O(1) reads when all keys fit in RAM
Full-text search Lucene / Elasticsearch Inverted index, fuzzy matching
Caching, session state Redis / Memcached In-memory, sub-millisecond, rich data types
Small dataset, read-heavy In-memory database No disk encoding overhead
