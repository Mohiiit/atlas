DDIA — Part II: Distributed Data

Chapter 6: Partitioning

"We must break away from the sequential and not limit the computers." — Grace Murray Hopper (1962). When a dataset is too large for a single machine, you must split it across multiple nodes. This chapter explores how to do that well — and how hard it really is.

Why Partition? Scalability

The main reason for partitioning data is scalability. Different partitions can be placed on different nodes in a shared-nothing cluster. A large dataset is distributed across many disks, and query load is distributed across many processors.

The Goal

Spread data and query load evenly across nodes. If every node takes a fair share, then 10 nodes should handle 10x the data and 10x the throughput of a single node.

  • Each partition is effectively a small database of its own
  • Queries on a single partition can execute independently on one node
  • Complex queries can be parallelized across nodes (but this is harder)

Terminological Confusion

What this book calls a partition is called: a shard in MongoDB/Elasticsearch/SolrCloud, a region in HBase, a tablet in Bigtable, a vnode in Cassandra/Riak, and a vBucket in Couchbase. Same concept, different names.

Partitioning and Replication Combined

Partitioning is usually combined with replication so that copies of each partition are stored on multiple nodes for fault tolerance. Each record belongs to exactly one partition, but that partition may be replicated across several nodes.

Partitioning of Key-Value Data

How do you decide which records to store on which nodes? The goal is to spread data and queries evenly. Random assignment distributes evenly but makes reads impossible (you'd have to query all nodes). We need smarter strategies.

Strategy 1: Partitioning by Key Range

Assign a continuous range of keys to each partition, like volumes of a paper encyclopedia. Volume 1 gets A-B, Volume 2 gets C-D, etc. Ranges need not be evenly spaced — they adapt to data distribution.

Boundaries adapt to actual data distribution. Within each partition, keys are sorted → efficient range scans. Used by Bigtable, HBase, RethinkDB, MongoDB (<2.4).

Advantage: Range Queries

Keys are sorted within each partition. You can treat the key as a concatenated index and efficiently fetch related records. E.g., sensor data keyed by timestamp: fetch all readings from March easily.

Danger: Hot Spots

If the key is a timestamp, all writes go to the partition for "today" while others sit idle. Fix: prefix timestamp with sensor name → (sensor_name, timestamp) so writes spread across partitions.

Strategy 2: Partitioning by Hash of Key

Use a hash function to determine the partition for each key. A good hash function takes skewed data and makes it uniformly distributed. Assign each partition a range of hashes (not a range of keys).

Hash function need not be cryptographic: Cassandra/MongoDB use MD5, Voldemort uses Fowler-Noll-Vo. Warning: Java's Object.hashCode() is NOT suitable — different processes produce different hashes for the same key.

Advantage: Even Distribution

Similar keys get scattered across all partitions. No more hot spots from sequential keys like timestamps.

Disadvantage: No Range Queries

Adjacent keys are now scattered. In MongoDB with hash sharding, range queries hit all partitions. Riak, Couchbase, Voldemort don't support range queries on primary key at all.

Cassandra's Compromise: Compound Primary Keys

A table can have a compound primary key (partition_key, sort_key). Only the first part is hashed to determine the partition. The remaining columns are used as a concatenated index for sorting within that partition.

Example: (user_id, update_timestamp) → hash distributes users across partitions, but within each user, updates are sorted by time. Elegant one-to-many data model.

Skewed Workloads and Relieving Hot Spots

Hashing can reduce hot spots but cannot eliminate them entirely. If all reads and writes target the same key, all requests route to the same partition regardless of hashing.

The Celebrity Problem

On a social media site, a celebrity with millions of followers posts an update. The key is the celebrity's user ID or the action ID. All the resulting writes (comments, likes) hit the same partition. Twitter famously dedicated 3% of its servers to Justin Bieber.

  • Workaround: Append a random 2-digit number to the hot key → splits writes across 100 different keys → 100 different partitions
  • Trade-off: Reads must now query all 100 keys and merge results. Only worth it for the small number of known hot keys.
  • Today, most systems do NOT automatically detect/compensate for hot keys — it remains the application's responsibility

Partitioning and Secondary Indexes

Everything above assumes key-value access (look up by primary key). But secondary indexes are the bread and butter of relational databases and search engines. The problem: secondary indexes don't map neatly to partitions. Two approaches exist.

Approach 1: Document-Partitioned Index (Local Index)

Each partition maintains its own secondary index, covering only the documents in that partition. When you write a document, you only update the local partition's index.

Document-Partitioned (Local) Secondary Index
Partition 0 (IDs 0-499)

Documents: red car #11, blue car #23, red car #78

Local index:
color:red → [11, 78]
color:blue → [23]

Partition 1 (IDs 500-999)

Documents: red car #512, black car #700

Local index:
color:red → [512]
color:black → [700]

Problem: To find ALL red cars, you must query every partition and merge results. This is called scatter/gather and is prone to tail latency amplification.

Approach 2: Term-Partitioned Index (Global Index)

Construct a global index that covers data in all partitions. The global index itself is partitioned — but by the indexed term, not by the document's primary key.

Term-Partitioned (Global) Secondary Index
Index Partition 0 (terms a-r)

color:black → [700]
color:blue → [23]
color:red → [11, 78, 512]
(all red cars from ALL partitions in one place!)

Index Partition 1 (terms s-z)

color:silver → [44, 888]
color:white → [102]
color:yellow → [333]

Advantage: Reading is efficient — a lookup goes to a single index partition. Disadvantage: Writes are slower — a single document write may update multiple index partitions (each term might live on a different node).

Rebalancing Partitions

Over time, things change: query throughput grows, dataset grows, machines fail. Data and requests must be moved from one node to another. This process is called rebalancing.

Rebalancing Requirements

  • Fair share: After rebalancing, load should be shared fairly between nodes
  • Availability: Database must continue accepting reads and writes during rebalancing
  • Minimal movement: Move no more data than necessary to minimize network/disk I/O

Anti-Pattern: hash mod N

Why Not hash(key) mod N?

If the number of nodes N changes, most keys must be moved. Example: hash(key) = 123456.

  • 10 nodes: key goes to node 6 (123456 mod 10 = 6)
  • 11 nodes: key moves to node 3 (123456 mod 11 = 3)
  • 12 nodes: key moves to node 0 (123456 mod 12 = 0)

Adding one node shuffles almost every key. This makes rebalancing excessively expensive. We need strategies where only a small fraction of keys move when nodes change.

Rebalancing Strategies

1. Fixed Number of Partitions

Create many more partitions than nodes up front (e.g., 1000 partitions across 10 nodes = 100 per node). When a new node joins, it steals a few partitions from existing nodes. Only whole partitions move.

Used by: Riak, Elasticsearch, Couchbase, Voldemort

2. Dynamic Partitioning

Partitions split when they exceed a size threshold (HBase default: 10 GB) and merge when they shrink. Number of partitions adapts to data volume automatically.

Used by: HBase, RethinkDB, MongoDB (≥2.4)

3. Proportional to Nodes

Fixed number of partitions per node (Cassandra: 256/node). When a new node joins, it randomly splits existing partitions and takes half. Partition count grows with cluster size.

Used by: Cassandra, Ketama

Choosing the Right Number of Fixed Partitions

If the total data size is highly variable, fixed partitions create a dilemma:

  • Too few partitions: Each partition becomes huge → rebalancing and recovery are expensive
  • Too many partitions: Management overhead per partition adds up
  • The number of partitions is the maximum number of nodes you can have

The "right" number is hard to get right if you don't know how much the data will grow. Dynamic partitioning solves this but is more complex.

Dynamic Partitioning: Pre-Splitting Caveat

An empty database with dynamic partitioning starts with a single partition — all writes go to one node while others sit idle. HBase and MongoDB support pre-splitting: configuring an initial set of partition boundaries on an empty database. But you need to know the key distribution in advance.

Operations: Automatic vs. Manual Rebalancing

The Automation Spectrum
Fully Automatic

System decides when and where to move partitions. No admin interaction.

Semi-Automatic

System suggests a plan. Admin approves before execution. (Couchbase, Riak, Voldemort)

Fully Manual

Admin explicitly configures partition-to-node assignment.

The Danger of Full Automation

Automatic rebalancing combined with automatic failure detection can cause cascading failures:

  • One node is overloaded and responds slowly
  • Other nodes conclude it is dead
  • Automatic rebalancing moves load away from "dead" node → additional network/disk I/O
  • This overloads the already-struggling node even more, plus stresses others
  • Cascading failure spirals out of control

Having a human in the loop is slower but prevents operational surprises.

Request Routing

Once data is partitioned across nodes, how does a client know which node to talk to? As partitions are rebalanced, the assignment changes. This is an instance of the general service discovery problem.

Three Approaches to Request Routing
1. Contact Any Node

Client contacts any node via round-robin load balancer. If that node owns the partition, it handles the request directly. Otherwise, it forwards to the correct node, gets the reply, and passes it back.

2. Routing Tier

All requests go to a partition-aware load balancer (routing tier). It determines the correct node and forwards. The routing tier handles no data itself — it is only a router.

3. Client-Aware

Client is aware of the partitioning scheme and connects directly to the correct node. No intermediary needed. Requires the client library to stay up-to-date with partition assignments.

Service Discovery: How to Stay Up-to-Date

The key challenge: how does the routing component (whether it is a node, routing tier, or client) learn about changes in partition-to-node assignments?

Coordination Service (ZooKeeper)

A separate coordination service (ZooKeeper, etcd) maintains the authoritative mapping of partitions to nodes. Nodes register themselves. Routing tier or clients subscribe to changes and get notified when assignments change.

  • LinkedIn Espresso uses Helix (on ZooKeeper)
  • HBase, SolrCloud, Kafka use ZooKeeper directly
  • MongoDB uses its own config server + mongos routing daemons

Gossip Protocol

Nodes disseminate partition assignment changes among themselves using a gossip protocol. Any node can accept requests and forward to the correct node (Approach 1). No external dependency.

  • Cassandra and Riak use this approach
  • More complexity in database nodes
  • Avoids dependency on external coordination service

Parallel Query Execution

Everything above focused on simple queries: read/write a single key, or scatter/gather on secondary indexes. But massively parallel processing (MPP) relational databases used for analytics go much further.

MPP Query Execution

A typical data warehouse query involves joins, filtering, grouping, and aggregation. The MPP query optimizer breaks this into execution stages that can run in parallel on different nodes. Queries scanning large parts of the dataset benefit enormously.

This is a specialized topic covered in Chapter 10. Key examples: Teradata, Vertica, Amazon Redshift, Apache Impala.

First-Principles Questions

Test your understanding. Click to reveal the answer.

1Why can't you just assign records to nodes randomly?▼
Random assignment distributes data evenly, which is good. But when you want to read a particular record, you have no way of knowing which node it is on. You would have to query all nodes in parallel for every read, which is extremely wasteful. We need a deterministic mapping from key to partition so that reads can go directly to the right node.
2Why does key-range partitioning lead to hot spots with timestamp keys?▼
With key-range partitioning, each partition covers a contiguous range of keys. If the key is a timestamp, the partitions correspond to time ranges (e.g., one partition per day). Since all new data has the current timestamp, all writes go to the partition for "today" while all other partitions sit idle. The fix is to prefix the timestamp with something that distributes writes (e.g., sensor name), so the first element of the compound key scatters data across partitions.
3What does hashing the key buy you, and what do you lose?▼
Hashing buys you even distribution: even if keys are sequential or clustered (like timestamps or user IDs), their hashes are uniformly scattered across the hash space, eliminating most hot spots. You lose the ability to do efficient range queries, because keys that were adjacent in the original key space are now scattered across different partitions. MongoDB with hash sharding must send range queries to all partitions.
4How does Cassandra get the best of both key-range and hash partitioning?▼
Cassandra uses compound primary keys. The first column is hashed to determine the partition (giving even distribution). The remaining columns are used as a concatenated index within that partition, maintaining sort order. Example: (user_id, timestamp) — user_id is hashed for distribution, but within one user's partition, all updates are sorted by timestamp. You can do efficient range scans on the second column if you fix the first.
5Why is hash mod N a terrible rebalancing strategy?▼
If you assign keys using hash(key) mod N, then when N (the number of nodes) changes by even 1, the modular arithmetic changes for almost every key. For example, key with hash 123456 goes to node 6 when N=10, node 3 when N=11, and node 0 when N=12. This means adding or removing a single node requires migrating nearly all data across the cluster, making rebalancing prohibitively expensive.
6What is the trade-off between document-partitioned (local) and term-partitioned (global) secondary indexes?▼
Document-partitioned (local): Each partition maintains its own secondary index covering only its documents. Writes are fast (only update local partition), but reads require scatter/gather to all partitions — slow and prone to tail latency amplification. Term-partitioned (global): The index is partitioned by the indexed term. Reads are fast (query one index partition), but writes are slow because a single document write may need to update index entries on multiple partitions across different nodes, often done asynchronously.
7How can fully automatic rebalancing cause cascading failures?▼
Scenario: One node is overloaded and responds slowly. Automatic failure detection concludes it is "dead." Automatic rebalancing kicks in and moves partitions away from the "dead" node. This transfer generates heavy network and disk I/O, putting additional load on the already-overloaded node AND on the other nodes receiving the transferred partitions. The extra load makes things worse, potentially causing more nodes to appear "dead," triggering more rebalancing — a vicious cycle. Having a human approve rebalancing decisions prevents this.
8What are the three approaches to request routing, and which databases use each?▼
1. Any node: Client contacts any node; if it doesn't own the partition, it forwards the request. Used by Cassandra and Riak (gossip protocol). 2. Routing tier: A dedicated partition-aware load balancer forwards requests. Used by LinkedIn Espresso (Helix/ZooKeeper), MongoDB (mongos), HBase, Kafka. 3. Client-aware: Client knows the partitioning and connects directly to the right node. In all cases, the key challenge is keeping the routing information up to date as partitions are rebalanced.
9Why does ZooKeeper solve the request routing problem better than having each node figure it out?▼
The routing problem requires consensus — all participants must agree on which partition lives on which node. Achieving consensus in a distributed system is notoriously hard (see Chapter 9). ZooKeeper is a purpose-built coordination service that already solves consensus. Nodes register their partition assignments with ZooKeeper, and routing tiers or clients subscribe to changes. ZooKeeper guarantees that all subscribers see a consistent view. The alternative — gossip protocol (Cassandra, Riak) — avoids the external dependency but pushes the complexity into the database nodes themselves.
10Why can't hashing alone solve the celebrity hot-spot problem?▼
Hashing distributes different keys evenly. But if millions of reads and writes target the same key (e.g., a celebrity's user ID), the hash of that key is always the same value, routing everything to the same partition. Hashing only helps when the hot-spot problem is caused by many similar-but-different keys clustering together. For truly identical keys, the application must split the key itself (e.g., append random suffixes) and handle the fan-out on reads.
11What is scatter/gather and why is it problematic?▼
Scatter/gather is the pattern where a query on a document-partitioned secondary index must be sent to all partitions (scatter), and the results from all partitions are combined (gather). It is problematic because: (1) It is slow even with parallel execution — the response time is determined by the slowest partition (tail latency amplification). (2) It puts load on every node in the cluster for every query. (3) It gets worse as the number of partitions grows. This is why most database vendors recommend designing your partitioning scheme so secondary index queries can be served from a single partition when possible.
