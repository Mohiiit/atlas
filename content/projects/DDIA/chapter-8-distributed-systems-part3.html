Byzantine Faults

So far we've assumed nodes are honest: they may crash or be slow, but they never send deliberately incorrect data. Byzantine faults are a step beyond — nodes that lie, send contradictory messages to different nodes, or behave arbitrarily maliciously.

Crash-stop fault
Node stops working and stays silent. Other nodes see no response. Simple — either node is up or it's down.
✓ Assumed in most distributed systems
Byzantine fault
Node continues running but sends wrong, inconsistent, or contradictory data. May tell different nodes different things.
✗ Much harder to handle. Requires special algorithms.

When Do Byzantine Faults Actually Occur?

In most enterprise/web systems, Byzantine faults are not a practical concern. But they do matter in specific contexts:

Aerospace systems
Radiation can flip bits in memory (bit-rot). A sensor reading becomes incorrect data. Nodes may disagree on sensor values due to hardware errors.
Blockchain / P2P
Bitcoin, Ethereum: participants are anonymous, untrusted. Any peer may send bogus transactions. The protocol must reach consensus despite malicious actors.
Multi-org systems
Multiple competing companies sharing infrastructure. A participant may cheat to gain advantage. Byzantine fault tolerance (BFT) provides guarantees even when some organizations defect.

Byzantine Fault Tolerance (BFT)

A system is Byzantine fault tolerant if it continues to operate correctly even if some nodes are lying or malicious. The classic result:

Byzantine Generals Problem: With N nodes, you can tolerate at most f Byzantine nodes if N ≥ 3f + 1. To tolerate 1 traitor, you need at least 4 nodes. To tolerate 2, you need at least 7. The overhead is large — BFT algorithms are expensive and complex.

Most distributed databases (Zookeeper, etcd, Raft, Paxos) assume crash-stop faults only. They are NOT Byzantine fault tolerant. A single malicious node can corrupt the cluster.

For most systems (internal networks, trusted servers), Byzantine fault tolerance is overkill. The threat model is crashes, not adversaries. Use Byzantine fault tolerance when participants are untrusted (blockchains) or hardware errors are likely (aerospace).

Weak Forms of Byzantine Behavior in Practice

Even without explicit attackers, bugs and hardware issues can cause behaviors resembling Byzantine faults:

  • A firmware bug causes a NIC to occasionally corrupt packets (that pass the TCP checksum)
  • A memory bug causes a node to return incorrect data for a key
  • A software bug causes a node to send inconsistent responses to different callers

Defense: checksums at the application level, majority voting across redundant sensors, asserting invariants on received data. Not full BFT, but catches common hardware/software corruption.

System Model and Reality

Algorithms are designed against an abstract system model — a set of assumptions about what can and cannot happen. Choosing the right model matters: too weak and the algorithm doesn't work; too strong and no real system satisfies it.

Timing Models

ModelAssumptionReality?
Synchronous Network delay, process pauses, and clock drift all have known upper bounds. Messages arrive within some maximum time. Unrealistic for most distributed systems. Real networks have unbounded delay.
Partially synchronous System behaves synchronously most of the time, but can occasionally have periods of asynchrony (unbounded delays). The system eventually returns to synchrony. Most realistic model. Captures real-world behavior: networks are usually fast but occasionally hit delays.
Asynchronous No timing guarantees. No clocks. Messages can be delayed arbitrarily. Algorithm cannot rely on timeouts. Worst-case assumption. FLP impossibility applies: cannot solve consensus deterministically.

Node Failure Models

ModelWhat a node can doExamples
Crash-stop Node crashes and never comes back. Once it stops responding, it stays stopped. Simplest. Used by many algorithms. Process is killed (not rebooted).
Crash-recovery Node can crash but may restart and rejoin. It forgets in-memory state but retains durable (disk-persisted) state. Most realistic. Servers reboot. State on disk survives; RAM doesn't.
Byzantine (arbitrary) Node can do anything: crash, return wrong data, send different data to different peers, or lie about state. Worst case. Required for adversarial settings.
Most real distributed databases use the partially synchronous timing model + crash-recovery node model. Algorithms are designed to be correct under these assumptions — not the stronger synchronous model (which doesn't hold) and not the weaker Byzantine model (which is too expensive to handle).

Which System Model to Use?

Choosing the right model:
✓ Crash-recovery + partial synchrony → practical systems (ZooKeeper, Raft, Paxos, most databases)
⚠ Crash-stop + synchrony → simpler algorithms, but overly optimistic
✗ Asynchronous → FLP: cannot achieve consensus deterministically. Need randomization or failure detectors.
Byzantine → blockchains, aerospace, multi-org untrusted environments only

Correctness of Algorithms: Safety and Liveness

When we say an algorithm is correct, we need to define what "correct" means. Two fundamental properties:

Safety
"Nothing bad happens."
A safety property is violated at a specific point in time — once violated, it cannot be undone.

Examples:
• Uniqueness: no two nodes believe they hold the same exclusive lock
• Monotonic sequence: fencing tokens are always increasing
• No dirty reads: committed values only

If violated: the system made an incorrect decision that can't be fixed.
Liveness
"Something good eventually happens."
A liveness property says that eventually the system makes progress.

Examples:
• Availability: every request eventually gets a response
• Leader election: a new leader is eventually elected after failure
• Termination: the consensus algorithm eventually completes

Cannot be violated at a single point — system could still recover.

Why Both Properties Matter

An algorithm that does nothing trivially satisfies all safety properties (nothing bad happens, ever) but violates liveness (nothing good ever happens either). Useful systems need both.

In practice, distributed systems sometimes sacrifice liveness (availability) to preserve safety. A database that refuses writes during a partition is choosing safety over liveness. This is a deliberate design trade-off — not a bug.

Safety, Liveness, and CAP

How CAP relates to safety/liveness:
Consistency (CAP) = a safety property. Every read returns the most recent write.
Availability (CAP) = a liveness property. Every request eventually gets a non-error response.
Partition tolerance = not optional — partitions happen.
So CAP is really: during a network partition, choose to preserve safety (consistency) or liveness (availability). You can't have both simultaneously.

Chapter 8 Summary

Networks
Asynchronous, best-effort. Cannot distinguish crashed node from slow network. You only have timeouts — and timeouts don't tell you what happened.
Clocks
Time-of-day clocks are unreliable. Monotonic clocks only for durations. NTP can't give exact time. Google Spanner's TrueTime is the exception, not the rule.
Process pauses
GC, VM migration, I/O, OS scheduling. A thread can be paused at any point. When it wakes up, the world has moved on. Assume nothing about time.
Fencing tokens
The solution to stale lock holders. Monotonically increasing token issued with each lock grant. Storage rejects writes with old tokens. Correctness enforced at the resource.
Byzantine faults
For most systems: crash-stop is sufficient. Byzantine fault tolerance only for adversarial/untrusted environments. 3f+1 nodes to tolerate f traitors.
System models
Use: crash-recovery + partially synchronous. Design algorithms that are correct in this model. Verify: safety (nothing bad) + liveness (something good eventually).

Review Q&A

1
Why can't you tell the difference between a crashed node and a slow network?
›
In an asynchronous network, there is no upper bound on message delay. When you send a request and get no response within time T, there are multiple possible explanations: the request was lost in the network; the request arrived but the response was lost; the node crashed; the node is alive but slow due to GC, I/O, or load; the network is congested. None of these scenarios produces a distinct signal — they all look the same to the sender: silence. The only tool you have is a timeout, but a timeout just tells you "I haven't heard back in T time" — not why. This is a fundamental limitation of asynchronous networks, not a bug you can fix.
2
Why is it dangerous to use time-of-day clocks for ordering events in distributed systems?
›
Time-of-day clocks (e.g., System.currentTimeMillis()) are synchronized via NTP, but NTP has a precision of ~10ms on a well-run network and can be much worse. More critically, NTP can cause time-of-day clocks to jump backward when they drift ahead of the NTP server. If Node A has clock 10:00:00.010 and Node B has clock 10:00:00.000 (5ms behind), and B writes after A, B's timestamp appears earlier — B's write looks older and gets discarded by LWW (last-write-wins). The correct write is silently dropped. The fix: use logical clocks (Lamport timestamps, version vectors) for ordering, not wall clocks.
3
How does a GC pause break the assumption of distributed lock safety?
›
Consider: Client A acquires a distributed lock with 30-second TTL. Client A checks the time (12:00:00), sees the lease expires at 12:00:30, and decides to proceed. Then, between the time-check and the write, the JVM triggers a stop-the-world GC pause for 60 seconds. Client A is frozen. During the pause, the lock expires. Client B acquires the lock. Client B writes to the shared resource. Client A wakes up, doesn't re-check the time (or re-checks but is now preempted again between the check and the write), and also writes. Both clients wrote — the lock failed to provide mutual exclusion. The fundamental problem: you cannot safely act on a lease check, because you can be paused between the check and the action.
4
How do fencing tokens solve the stale lock holder problem?
›
The key insight: don't rely on clients to know whether their lock is still valid. Instead, make the resource (storage server) the authority. Every time a lock is granted, the lock server issues a monotonically increasing fencing token (e.g., ZooKeeper's zxid). The client includes this token in every write to the storage service. The storage service tracks the highest token it has processed. If a write arrives with a token lower than the last seen, it is rejected. So: Client A gets token 33, pauses. Client B gets token 34, writes (storage records: last=34). Client A wakes up, tries to write with token 33 → storage sees 33 < 34, rejects. Client A's stale write is blocked, even though Client A doesn't know it's stale.
5
What is the Byzantine Generals Problem and why does it require N ≥ 3f + 1 nodes?
›
The Byzantine Generals Problem asks: can N generals, some of whom may be traitors, reach agreement on a battle plan by exchanging messages? Traitors may send contradictory messages to different generals. The result (Lamport, Shostak, Pease 1982): you need at least 3f+1 nodes to tolerate f traitors. Intuition: with f traitors and 3f nodes (not enough), the traitors can create ambiguity such that honest nodes can't distinguish correct from incorrect information — there are too few honest nodes to outvote the liars. With 3f+1 nodes, an honest majority of 2f+1 can detect and override up to f Byzantine nodes. This is expensive: for f=1 (tolerating 1 traitor), you need 4 nodes; for f=2, you need 7.
6
What is the difference between the synchronous, partially synchronous, and asynchronous system models?
›
Synchronous model: All network delays, process pauses, and clock drifts have known finite upper bounds. Algorithms can use timeouts to detect failures. This model is unrealistic for real distributed systems — actual networks have unbounded delay. Partially synchronous model: The system behaves synchronously most of the time, but occasionally has asynchronous periods with unbounded delays. The system eventually returns to synchrony. This is the most realistic model — it captures the real world: networks are usually fast but sometimes slow. Most practical algorithms (Raft, Paxos, ZooKeeper) are designed for partial synchrony. Asynchronous model: No timing assumptions at all. No clocks. Messages can be delayed arbitrarily. The FLP impossibility result applies: you cannot deterministically solve consensus in a fully asynchronous system with even one crash fault.
7
What is the difference between a safety property and a liveness property? Give examples.
›
Safety: "Nothing bad happens." A safety property is violated at a specific moment — once violated, it cannot be undone. Examples: (1) Uniqueness — no two nodes simultaneously believe they hold the same exclusive lock; (2) Monotonicity — fencing tokens are always strictly increasing; (3) No dirty reads — a transaction only sees committed data. If a safety property is violated, the system has made a mistake that cannot be corrected. Liveness: "Something good eventually happens." A liveness property cannot be violated at a single point — it always leaves hope that the system will eventually catch up. Examples: (1) Availability — every request eventually gets a response; (2) Progress — a distributed transaction eventually commits or aborts; (3) Leader election — after a crash, the system eventually elects a new leader. A system that does nothing satisfies all safety properties trivially (nothing bad can happen) but violates all liveness properties. Both are needed for useful systems.
8
Why does Google Spanner use GPS receivers and atomic clocks, and what does TrueTime give them?
›
Spanner is a globally distributed database that provides external consistency (a stronger form of serializability across globally distributed transactions). To achieve this, Spanner needs to assign commit timestamps that respect real-time ordering: if transaction T1 commits before T2 starts, T1's commit timestamp must be smaller than T2's. This requires knowing the real time accurately. Regular NTP has ~10ms uncertainty — too large. Spanner's solution: GPS receivers and atomic clocks in every datacenter provide extremely accurate time. The TrueTime API returns an interval [earliest, latest] that is guaranteed to contain the true current time. The uncertainty interval is typically 1–7ms. Spanner waits for the uncertainty window to pass before committing a transaction — this ensures that even if two datacenters' clocks are at the extremes of their uncertainty windows, the commit timestamps are still correctly ordered. This "commit wait" is the cost of external consistency without distributed locking.
9
Why is the crash-recovery model more realistic than crash-stop, and what does it require of algorithms?
›
Crash-stop assumes that once a node crashes, it stays crashed forever. This is unrealistic — in real deployments, servers are restarted: either automatically by a watchdog, or manually by operators. A node may crash, lose its in-memory state, and rejoin the cluster with only its disk-persisted state intact. The crash-recovery model captures this reality. Algorithms designed for crash-recovery must: (1) Persist critical state to disk before taking any action (e.g., write-ahead log in 2PC, voted values in Paxos); (2) Re-read disk state on restart and continue from where they were; (3) Handle "in-doubt" transactions — where the node crashed after preparing but before learning the commit decision. This is why Paxos, Raft, and ZooKeeper all write to stable storage before responding to requests — to survive crashes and restarts correctly.
10
A node believes it is still the leader. Under what conditions is this belief dangerous, and how should systems handle it?
›
A node may believe it's the leader because it held that role — but the rest of the cluster may have elected a new leader (due to the old leader appearing unresponsive, e.g., during a GC pause). The old leader's belief is now stale. If it acts on this belief — accepting writes, responding to reads as authoritative — it becomes a split-brain: two nodes both behave as leader simultaneously. This can cause: (1) Two leaders accepting conflicting writes; (2) Stale reads returning outdated data; (3) Data corruption. Systems handle this via: (a) Fencing tokens: the lock server's token is incremented when a new leader is elected; the storage layer rejects writes with old tokens; (b) Epoch numbers: Raft's term numbers, ZooKeeper's epoch — any message from an old term/epoch is rejected; (c) Lease-based checks: before acting, the leader verifies its lease is still valid. The key principle: don't rely on the node's self-belief about being leader. Make the decision at the resource or via quorum confirmation.
