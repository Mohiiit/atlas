DDIA -- Part I: Foundations of Data Systems

Chapter 2: Data Models and Query Languages

"The limits of my language mean the limits of my world." -- Wittgenstein. Data models are perhaps the most important part of developing software, because they shape how we think about the problem. Each data model embodies assumptions about how it will be used -- some operations are easy, some are not supported; some are fast, some perform badly.

Data Models Are Layers of Abstraction

Most applications are built by layering one data model on top of another. For each layer, the key question is: how is it represented in terms of the next-lower layer? Each layer hides the complexity of the layers below it by providing a clean data model. These abstractions allow different groups of people -- for example, the engineers at the database vendor and the application developers using their database -- to work together effectively.

Diagram 1: Layers of Data Models (Abstraction Stack)

↓ represented as ↓
2General-Purpose Data Model -- JSON/XML documents, tables in a relational DB, vertices/edges in a graph. The choice made here profoundly affects what your app can and cannot do easily.

4Hardware -- Electrical currents, pulses of light, magnetic fields. Hardware engineers figured out how to represent bytes in these physical media.

In a complex application there may be more intermediary levels (APIs built upon APIs), but the basic idea remains: each layer hides the complexity below it.

Why This Matters

There are many different kinds of data models, and every data model embodies assumptions about how it is going to be used. Some operations are easy, some are not supported; some operations are fast, some perform badly. It can take a lot of effort to master just one data model. Choosing the right data model for your application is arguably the most important decision you make, because it determines what the software above it can and can't do.

The Relational Model

The relational model was proposed by Edgar Codd in 1970. Data is organized into relations (called tables in SQL), where each relation is an unordered collection of tuples (rows). The relational model was a theoretical proposal, and many people at the time doubted it could be implemented efficiently. But by the mid-1980s, RDBMSes and SQL had become the tools of choice for most people who needed to store and query data with some kind of regular structure.

The dominance of relational databases has lasted around 25-30 years -- an eternity in computing. Relational databases turned out to generalize very well beyond their original scope of business data processing (transaction processing and batch processing). Much of what you see on the web today is still powered by relational databases: online publishing, discussion forums, social networks, ecommerce, games, SaaS applications, and much more.

The Birth of NoSQL

The name "NoSQL" is unfortunate -- it was originally just a catchy Twitter hashtag for a meetup in 2009, and it has been retroactively reinterpreted as Not Only SQL. Several driving forces behind the adoption of NoSQL databases include:

  • Greater scalability than relational databases can easily achieve, including very large datasets or very high write throughput
  • A widespread preference for free and open source software over commercial database products
  • Specialized query operations not well supported by the relational model
  • Frustration with the restrictiveness of relational schemas, and a desire for a more dynamic and expressive data model

Polyglot persistence: different applications have different requirements, and the best choice of technology for one use case may well be different from the best choice for another. In the foreseeable future, relational databases will continue to be used alongside a broad variety of nonrelational datastores -- this is sometimes called polyglot persistence.

The Object-Relational Mismatch (Impedance Mismatch)

Most application development today is done in object-oriented programming languages, which leads to a common criticism of the SQL data model: if data is stored in relational tables, an awkward translation layer is required between the objects in the application code and the database model of tables, rows, and columns. The disconnect between the models is sometimes called an impedance mismatch.

Object-relational mapping (ORM) frameworks like ActiveRecord and Hibernate reduce the amount of boilerplate code required for this translation layer, but they can't completely hide the differences between the two models.

The Key Trade-Off: IDs vs. Human-Readable Strings

Why Use IDs (Normalization)?

When you store a text string like "Greater Seattle Area" directly, you're using human-meaningful information. If that information ever changes (e.g., a city is renamed), every record that duplicates it must be updated. With an ID (like region_id: 42), the human-meaningful info is stored in only one place (the regions table). The ID itself has no meaning to humans, so it never needs to change -- even if the information it identifies changes.

Removing such duplication is the key idea behind normalization in databases. Unfortunately, normalizing data requires many-to-one relationships (many users live in one region), which don't fit nicely into the document model.

Data Gets More Interconnected Over Time

Even if the initial version of an application fits nicely in a join-free document model, data has a tendency of becoming more interconnected as features are added:

  • Organizations and schools as entities: Instead of plain strings ("Microsoft"), they become first-class records with their own pages, logos, news feeds. Now many users reference the same organization → many-to-one.
  • Recommendations: "User A recommends User B" is a many-to-many relationship between users.
  • Shared references: A user's profile photo might appear in multiple activity feeds, search results, etc.

The more interconnected the data, the more you need joins and references -- the strengths of relational and graph models.

Document Databases Are Repeating History

The most popular database for business data processing in the 1970s was IBM's Information Management System (IMS), first developed for the Apollo space program's inventory tracking. IMS used the hierarchical model, which represented all data as a tree of records nested within records -- remarkably similar to the JSON model used by document databases today.

Like document databases, IMS worked well for one-to-many relationships, but it made many-to-many relationships difficult, and it didn't support joins. Developers had to decide whether to denormalize their data or to manually resolve references. These exact same problems resurface with document databases today. Two prominent solutions were proposed to solve the limitations of the hierarchical model:

Relational vs. Document Databases Today

The main arguments in favor of the document data model are schema flexibility, better performance due to locality, and that for some applications it is closer to the data structures used by the application. The relational model counters by providing better support for joins, and many-to-one and many-to-many relationships.

Which Data Model Leads to Simpler Application Code?

If your application data has a document-like structure (a tree of one-to-many relationships, where typically the entire tree is loaded at once), then it's probably a good idea to use a document model. But if your application does use many-to-many relationships, the document model becomes less appealing. It's possible to reduce the need for joins by denormalizing, but then the application code needs to do additional work to keep the denormalized data consistent.

For highly interconnected data, the document model is awkward, the relational model is acceptable, and graph models are the most natural.

Schema Flexibility: Schema-on-Read vs Schema-on-Write

Document databases are sometimes called schemaless, but that's misleading. The code that reads the data usually assumes some kind of structure -- there is an implicit schema, but it is not enforced by the database. A more accurate term is schema-on-read (the structure of the data is implicit, and only interpreted when the data is read), in contrast with schema-on-write (the traditional approach of relational databases, where the schema is explicit and the database ensures all written data conforms to it).

Diagram 9: Schema-on-Read vs Schema-on-Write

// App handles both formats gracefully

Best when: Items don't all have the same structure (heterogeneous data), structure determined by external systems you don't control, or schema changes frequently.

Schema-on-Write (Relational)

Like static type checking in programming languages. Schema enforced on every write.

↓ must match schema ↓

↓ guaranteed shape ↓

-- MySQL: copies entire table! Can be slow.
-- PostgreSQL: ADD COLUMN is fast (sets NULL default)

Best when: All records have the same structure, data integrity is paramount, schema is stable, or you want the database to catch errors.

"Schema-on-read is similar to dynamic (runtime) type checking in programming languages, whereas schema-on-write is similar to static (compile-time) type checking. Just as with static vs. dynamic type checking, there's no universally right answer -- the choice depends on your situation."

Data Locality for Queries

A document is usually stored as a single continuous string, encoded as JSON, XML, or a binary variant thereof (e.g., MongoDB's BSON). If your application often needs to access the entire document, there is a performance advantage to storage locality: all the data is in one place and can be fetched in a single disk seek.

Locality Trade-offs

✓ Advantage: If you frequently need the entire document, one fetch is faster than multiple joins across tables.

✗ Disadvantage: On updates, the entire document usually needs to be rewritten (unless the modification doesn't change the encoded size). For large documents, this is wasteful.

The locality advantage only applies if you need large parts of the document at the same time. The database typically loads the whole document even if you access only a small portion. On updates that increase document size, the document may need to be rewritten entirely. These limitations significantly reduce the usefulness of locality for documents that are frequently modified.

Note: Locality is not limited to the document model. Google's Spanner database offers locality in a relational data model by allowing tables to be interleaved (nested) within a parent table. Oracle has multi-table index cluster tables. The column-family concept in Bigtable/Cassandra/HBase also has a similar goal.

It seems that relational and document databases are becoming more similar over time, and that is a good thing. A hybrid of the two models is a good route for databases to take.

Query Languages for Data

When the relational model was introduced, it included a new way of querying data: SQL is a declarative query language, whereas IMS and CODASYL queried the database using imperative code. What does that difference mean?

Imperative (IMS, CODASYL, general code)

Tell the computer how to achieve a result, step by step.

function getSharks(animals) {
  var sharks = [];
  for (var i = 0; i < animals.length; i++) {
    if (animals[i].family === "Sharks") {
      sharks.push(animals[i]);
    }
  }
  return sharks;
}

✗ Hard to parallelize (shared mutable state)

✗ Implementation details exposed

✗ Cannot be automatically optimized

Declarative (SQL, CSS, Cypher, Datalog)

Tell the computer what result you want, not how to compute it.

SELECT * FROM animals
WHERE family = 'Sharks';

-- li.selected > p { color: blue; }
-- (The browser decides HOW to apply it)

✓ Concise and easier to work with

✓ Hides implementation details

✓ Database can optimize (query optimizer)

✓ Can parallelize across multiple machines

Declarative Queries on the Web (CSS Analogy)

The advantages of declarative query languages are not limited to databases. In a web browser, declarative CSS styling (li.selected > p { background: blue; }) is far better than manipulating styles imperatively in JavaScript. The declarative approach lets the browser optimize rendering, while imperative DOM manipulation is fragile and error-prone.

If there were no CSS, you'd have to imperatively set element.style.backgroundColor in loops, handle additions/removals, etc. This is exactly analogous to why SQL is better than imperative database queries.

MapReduce Querying

MapReduce is a programming model for processing large amounts of data in bulk across many machines, popularized by Google. A limited form of MapReduce is supported by some NoSQL datastores, including MongoDB and CouchDB, as a mechanism for performing read-only queries across many documents.

MapReduce is neither a declarative query language nor a fully imperative query API, but somewhere in between: the logic of the query is expressed with snippets of code, which are called repeatedly by the processing framework. It is based on the map (also known as collect) and reduce (also known as fold or inject) functions that exist in many functional programming languages.

Usability problem with MapReduce

Having to write two carefully coordinated JavaScript functions is harder than writing a single SQL query. Moreover, a declarative query language offers the query optimizer more opportunities for improving performance. For these reasons, MongoDB added support for a declarative query language called the aggregation pipeline, which is similar in expressiveness to a subset of SQL.

MapReduce (MongoDB syntax):

db.observations.mapReduce(
  function map() {
    var year = this.observationTimestamp.getFullYear();
    var month = this.observationTimestamp.getMonth() + 1;
    emit(year + "-" + month, this.numAnimals);
  },
  function reduce(key, values) {
    return Array.sum(values);
  },
  {
    query: { family: "Sharks" },
    out: "monthlySharkReport"
  }
);

Equivalent SQL:

SELECT
  date_trunc('month', observation_timestamp)
    AS observation_month,
  SUM(num_animals)
    AS total_animals
FROM observations
WHERE family = 'Sharks'
GROUP BY observation_month;

Graph-Like Data Models

We saw earlier that many-to-many relationships are an important distinguishing feature between different data models. If your application has mostly one-to-many relationships (tree-structured data) or no relationships between records, the document model is appropriate. But what if many-to-many relationships are very common in your data? The relational model can handle simple cases of many-to-many relationships, but as the connections within your data become more complex, it becomes more natural to start modeling your data as a graph.

A graph consists of two kinds of objects: vertices (also known as nodes or entities) and edges (also known as relationships or arcs). Many kinds of data can be modeled as a graph. Typical examples include social graphs, the web graph, road/rail networks. Graphs are not limited to homogeneous data: an equally powerful use of graphs is to provide a consistent way of storing completely different types of objects in a single datastore.

Key features of the property graph model:

  • Each vertex has: a unique identifier, a set of outgoing edges, a set of incoming edges, and a collection of properties (key-value pairs).
  • Each edge has: a unique identifier, the tail vertex (start), the head vertex (end), a label describing the relationship, and a collection of properties.
  • Any vertex can have an edge connecting it with any other vertex. There is no schema restricting which kinds of things can be associated.
  • Given any vertex, you can efficiently find both incoming and outgoing edges, and thus traverse the graph in either direction.
  • By using different labels for different kinds of relationships, you can store several different kinds of information in a single graph while still maintaining a clean data model.

Representing a Property Graph in SQL

You can represent a property graph using a relational schema with two tables:

CREATE TABLE vertices (
  vertex_id  INTEGER PRIMARY KEY,
  properties JSON
);

CREATE TABLE edges (
  edge_id     INTEGER PRIMARY KEY,
  tail_vertex INTEGER REFERENCES vertices(vertex_id),
  head_vertex INTEGER REFERENCES vertices(vertex_id),
  label       TEXT,
  properties  JSON
);

CREATE INDEX edges_tails ON edges (tail_vertex);
CREATE INDEX edges_heads ON edges (head_vertex);

Important aspects: Any vertex can connect to any other (no schema restricting edge types). Given any vertex, you can efficiently traverse in both directions. Using labels, you can store multiple kinds of relationships in one graph.

The Cypher Query Language

Cypher is a declarative query language for property graphs, created for the Neo4j graph database. (Named after a character in the movie The Matrix, not the cryptographic cipher.)

Insert data in Cypher:

CREATE
  (NAmerica:Location {name:'North America', type:'continent'}),
  (USA:Location      {name:'United States', type:'country'}),
  (Idaho:Location    {name:'Idaho',         type:'state'}),
  (Lucy:Person       {name:'Lucy'}),
  (Idaho)  -[:WITHIN]->  (USA)   -[:WITHIN]-> (NAmerica),
  (Lucy)   -[:BORN_IN]-> (Idaho)

Query: Find people who emigrated from US to Europe:

MATCH
  (person) -[:BORN_IN]->  () -[:WITHIN*0..]-> (us:Location {name:'United States'}),
  (person) -[:LIVES_IN]-> () -[:WITHIN*0..]-> (eu:Location {name:'Europe'})
RETURN person.name

-- [:WITHIN*0..] means "follow a chain of
-- WITHIN edges, zero or more times"
-- (i.e., transitive closure)

Graph Queries in SQL -- Possible but Painful

The same query can be expressed in SQL using recursive common table expressions (WITH RECURSIVE), but it's much more awkward. In Cypher, :WITHIN*0.. means "follow a chain of WITHIN edges, zero or more times" (transitive closure). In SQL, this requires variable-length traversal, which requires recursive CTEs that are syntactically cumbersome. The number of joins is not fixed in advance -- you need recursive traversal because the graph depth is unknown ahead of time.

Triple-Stores and SPARQL

The triple-store model is mostly equivalent to the property graph model, using different words to describe the same ideas. In a triple-store, all information is stored in the form of very simple three-part statements: (subject, predicate, object). For example, (Jim, likes, bananas).

Datalog: The Foundation

Datalog is a much older language than either SPARQL or Cypher, having been studied extensively by academics in the 1980s. It provides the foundation that later query languages build upon. It is used in a few data systems: for example, it is the query language of Datomic, and Cascalog is a Datalog implementation for querying large datasets in Hadoop.

Datalog's data model is similar to the triple-store model, generalized slightly. Instead of writing a triple as (subject, predicate, object), we write it as predicate(subject, object).

Datalog -- Rules Build Upon Rules

/* Facts (like triples): */
name(lucy, "Lucy").
born_in(lucy, idaho).
name(idaho, "Idaho").
type(idaho, state).
within(idaho, usa).
name(usa, "United States").

/* Define reusable rules: */
within_recursive(X, Y) :- within(X, Y).                          /* base case */
within_recursive(X, Y) :- within(X, Z), within_recursive(Z, Y).  /* recursive */

/* Now the emigration query is clean: */
migrated(X, F, T) :-
  name(X, N),
  born_in(X, BL), within_recursive(BL, F),
  lives_in(X, LL), within_recursive(LL, T).

?- migrated(X, usa, europe).  /* Who moved from USA to Europe? */

The key power of Datalog: rules can be combined and reused. within_recursive is defined once and used in different queries. You can build complex queries step by step by combining small rules. This is less convenient in Cypher or SPARQL, where you'd need to repeat the recursive pattern each time.

Chapter Summary

Data models are a huge subject, and in this chapter we took a quick look at a broad variety of different models. Historically, data started out being represented as one big tree (the hierarchical model), but that wasn't good for representing many-to-many relationships, so the relational model was invented to solve that problem. More recently, developers found that some applications don't fit well in the relational model either. New nonrelational "NoSQL" datastores have diverged in two main directions:

Document Databases

Target use cases where data comes in self-contained documents and relationships between one document and another are rare. Schema flexibility, better performance due to locality, and closer match to application data structures.

Graph Databases

Go in the opposite direction, targeting use cases where anything is potentially related to everything. Excellent for highly interconnected, heterogeneous data with complex traversal patterns.

Key Takeaways

  • All three models (document, relational, graph) are widely used today, and each is good in its respective domain.
  • One model can be emulated in terms of another (e.g., graph in relational), but the result is often awkward. That's why we have different systems for different purposes.
  • Document and relational databases are converging: relational DBs now support JSON, document DBs now support joins.
  • Each data model comes with its own query language or framework: SQL, MapReduce, MongoDB's aggregation pipeline, Cypher, SPARQL, Datalog.
  • There are also query languages for CSS and XSL/XPath that operate in web browsers -- CSS being declarative and XSL using XPath (also declarative).

First-Principles Questions

Test your deep understanding. Click to reveal each answer.

1Why do data models matter so much in software development -- more than most other technical decisions?▼
Data models have a profound effect not only on how the software is written, but also on how we think about the problem we are solving. Each data model embodies assumptions about how it is going to be used -- some usages are easy, some are not supported; some operations are fast, some perform badly. The model you choose determines what the software above it can and can't do. Since most applications layer one data model on top of another (application objects → general-purpose model → storage engine → hardware), the choice ripples up through every layer of your architecture.
2What is the "impedance mismatch" and why haven't ORMs solved it completely?▼
The impedance mismatch is the fundamental disconnect between object-oriented programming (objects with nested structures, inheritance, references) and the relational data model (flat tables, rows, columns, foreign keys). Application code uses objects; the database stores rows. An awkward translation layer is required between them. ORMs like ActiveRecord and Hibernate reduce the boilerplate code, but they cannot completely hide the differences because the data models have fundamentally different structures: OOP favors nesting and encapsulation, while relational favors flat normalization and explicit joins. The mismatch leaks through in performance characteristics, query patterns, and how relationships are navigated.
3Why does the book argue for using IDs instead of human-readable strings for references? What database principle does this embody?▼
An ID is meaningless to humans, so it never needs to change. Human-meaningful information (like "Greater Seattle Area") may change, and if it's duplicated across records, every copy must be updated -- risking write overhead and inconsistencies. By using an ID, human-meaningful info is stored in only one place. This is the core idea behind normalization: removing duplication. The trade-off is that normalization requires many-to-one relationships (many records referencing one canonical record), which requires joins -- the strength of relational databases and the weakness of document databases.
4Why did the relational model defeat the network model (CODASYL)? What was the key architectural innovation?▼
The network model required programmers to manually track access paths -- chains of pointers through nested records. This was like navigating an n-dimensional data space. Changing the data model meant rewriting all query code. The relational model's key innovation was the query optimizer: it laid all data in the open (flat tables and rows) and let the database system automatically decide execution order, which indexes to use, and how to perform joins. You build the optimizer once, and all applications benefit. When you add a new index, existing queries automatically use it without code changes. This is why "the general-purpose solution wins in the long run."
5Explain schema-on-read vs schema-on-write with their programming language analogy. When is each approach better?▼
Schema-on-read (document model): the structure is implicit; the data is only interpreted when read. This is analogous to dynamic (runtime) type checking. The database stores whatever you give it; the application handles different formats gracefully. Best when: items don't all have the same structure, data comes from external systems you don't control, or you need to migrate schema gradually. Schema-on-write (relational model): the schema is explicit and enforced on every write. This is analogous to static (compile-time) type checking. The database guarantees data conforms to the schema before accepting it. Best when: all records have the same structure, data integrity is critical, and you want the database to catch errors early. Neither approach is universally better -- it depends on the situation.
6Why is a declarative query language fundamentally better than an imperative one for databases? Give both the conceptual and practical reasons.▼
Conceptually: A declarative language (SQL, Cypher, CSS) specifies what results you want, not how to achieve them. This separates specification from implementation, following the principle of abstraction. Practically: (1) It's more concise and easier to work with. (2) It hides implementation details, so the query optimizer can choose the best execution strategy -- and improve it over time without changing queries. (3) It naturally lends itself to parallel execution, because there's no prescribed order of operations (no shared mutable state). (4) Adding new indexes or changing storage layouts doesn't require rewriting queries. Imperative code specifies operations in a particular order, ties you to implementation details, and makes optimization and parallelization much harder.
7When should you choose a graph model over relational or document? What makes graph queries difficult in SQL?▼
Choose a graph model when many-to-many relationships are the dominant feature of your data, connections between items are as important as the items themselves, and you have heterogeneous data types (people, locations, events, etc.) with complex interconnections. Graph queries are difficult in SQL because they often require traversals of unknown depth -- for example, "find all locations within Europe" requires following a chain of WITHIN edges of arbitrary length. In SQL, this requires recursive common table expressions (WITH RECURSIVE), which are syntactically cumbersome. In Cypher, the same is just [:WITHIN*0..]. The number of joins isn't fixed in advance, which is fundamentally at odds with SQL's fixed-structure queries.
8How are modern document databases different from the 1970s hierarchical model (IMS), and how are they similar?▼
Similar: Both store nested records within their parent record (one-to-many as a tree structure). IMS's hierarchical model is remarkably similar to the JSON model used by modern document databases. Both work well for data with tree-like structure. Different: For many-to-one and many-to-many relationships, modern document databases use document references (like foreign keys) that are resolved at read time by follow-up queries or joins. They have not followed CODASYL's approach of manual pointer-based access paths. Both relational and modern document databases use unique identifiers to reference related items -- the key lesson learned from the CODASYL era.
9What is "data locality" in document databases, and what are its often-overlooked downsides?▼
Data locality means the entire document is stored as one contiguous block (a single string of JSON/BSON). Advantage: If you frequently need the entire document, one fetch is faster than multi-table joins. Often-overlooked downsides: (1) The database typically loads the entire document even if you only need a small portion -- wasteful for large documents. (2) On updates that change document size, the entire document must be rewritten. (3) For this reason, most document databases recommend keeping documents small and avoiding writes that increase document size. (4) Locality is not unique to documents: Google Spanner offers locality via interleaved tables, Oracle has multi-table index clusters, and Bigtable/Cassandra/HBase's column families serve a similar purpose. The locality advantage only applies when you need large parts of the document at once.
10How are relational and document models converging? What might the future look like?▼
Relational → Document: Most relational databases now support JSON or XML documents (PostgreSQL JSONB, MySQL JSON type, IBM DB2), with the ability to query and index inside documents using SQL/JSON path queries. Document → Relational: Some document databases are adding join-like capabilities (RethinkDB supports joins, MongoDB's $lookup in the aggregation pipeline, Couchbase N1QL). A hybrid approach is likely the future: databases that handle document-like, relational, and graph queries, letting you use each model's strengths as needed. As the book says, "if a database is able to handle document-like data and also perform relational queries on it, applications can use the combination of features that best fits their needs."
11What makes MapReduce "neither fully declarative nor fully imperative"? Why did MongoDB move beyond it?▼
MapReduce sits between declarative and imperative: the framework controls when and where to run your code (declarative scheduling), but the logic is expressed as user-written map and reduce functions (imperative snippets). The map and reduce functions must be pure functions (no side effects, no database queries, no external state), which allows the framework to run them anywhere, in any order, and re-run them on failure. Why MongoDB moved beyond it: Writing two carefully coordinated JavaScript functions is harder than writing a single query. A declarative query language gives the query optimizer more opportunities for improving performance (choosing indexes, parallelization, etc.). So MongoDB added the aggregation pipeline -- a declarative JSON-based query language similar in expressiveness to a subset of SQL.
12What is Datalog's unique advantage over Cypher and SPARQL? Why does composability matter for complex queries?▼
Datalog's unique advantage is composable, reusable rules. You define small rules (like within_recursive) that can be combined and built upon to create complex queries incrementally. In Cypher or SPARQL, you'd have to repeat the recursive pattern each time it appears. Why composability matters: For complex queries with repeated subpatterns, Datalog lets you define the logic once and reuse it -- like functions in programming. You can build up layers of abstraction: basic facts → derived rules → higher-level rules → final queries. This makes complex queries more readable, more maintainable, and less error-prone. It's the database equivalent of the DRY principle (Don't Repeat Yourself). The trade-off is that Datalog is less well-known and has a steeper learning curve.
