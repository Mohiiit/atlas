Chapter 2: Data Models and Query Languages

"The limits of my language mean the limits of my world." — Wittgenstein. Data models are perhaps the most important part of developing software, because they shape how we think about the problem.

Data Models Are Layers of Abstraction

Each layer hides the complexity of the layers below by providing a clean data model:

Layers of Data Models
1Application — Real-world objects modeled as data structures & APIs (specific to your app)
2General-Purpose Model — JSON, XML, tables in relational DB, or graph model
3Storage Engine — Bytes in memory, on disk, or on a network (how to query, search, process)
4Hardware — Electrical currents, light pulses, magnetic fields

Relational Model vs. Document Model

The Relational Model (SQL)

Proposed by Edgar Codd in 1970. Data organized into relations (tables), each an unordered collection of tuples (rows). It hid implementation details behind a clean interface and dominated for 25-30 years.

The Birth of NoSQL

Originally just a catchy Twitter hashtag (2009). Retroactively: "Not Only SQL." Driving forces:

  • Need for greater scalability (very large datasets, very high write throughput)
  • Preference for free and open source software
  • Specialized query operations not well supported by relational model
  • Frustration with restrictiveness of relational schemas — desire for more dynamic, expressive data model

Polyglot persistence: relational databases used alongside nonrelational datastores, each for what it's best at.

The Object-Relational Mismatch (Impedance Mismatch)

OOP languages use objects; SQL uses tables/rows/columns. The disconnect requires an awkward translation layer. ORMs (ActiveRecord, Hibernate) reduce boilerplate but can't fully hide the difference.

The Key Trade-Off: IDs vs. Human-Readable Strings

Why Use IDs?

When you use an ID, human-meaningful information is stored in one place. The ID has no meaning to humans, so it never needs to change. If you store duplicate human-readable strings, every copy must be updated when the info changes → write overhead & risk of inconsistency. Removing duplication = normalization.

But normalization requires many-to-one relationships, which don't fit nicely into the document model (where joins are weak).

Data Gets More Interconnected Over Time

Even if the initial version fits a join-free document model, features like "organizations as entities" or "user recommendations" create many-to-many relationships, requiring references and joins.

Document DBs Are Repeating History

IBM's IMS (1968) used the hierarchical model — remarkably similar to JSON. Great for one-to-many, terrible for many-to-many. Two solutions emerged:

Network Model (CODASYL)

  • Records can have multiple parents
  • Links are like pointers, stored on disk
  • Access via access paths — navigating chains of pointers
  • Programmer must track all paths manually
  • Like navigating an n-dimensional data space
  • Changing data model = rewrite all query code

Relational Model (Won)

  • All data laid out in the open: tables + rows
  • No labyrinthine nested structures
  • Query optimizer automatically decides access paths
  • Add a new index → queries use it automatically
  • Build the optimizer once → all apps benefit
  • "General-purpose solution wins in the long run"

Modern document DBs: like hierarchical model for nested data, but use document references (like foreign keys) for many-to-one/many-to-many. They have not followed CODASYL's path.

Query Languages

Imperative (IMS, CODASYL)

Tell the computer how to do it: loop, filter, accumulate. Order matters. Hard to parallelize.

Declarative (SQL, CSS)

Tell the computer what you want. The optimizer decides how. Easier to parallelize. More concise.

MapReduce: Neither Fully Declarative nor Imperative

Logic expressed as map and reduce functions. Must be pure functions (no side effects). Can run anywhere, in any order. MongoDB supports it but also added a declarative aggregation pipeline.

Graph-Like Data Models

When many-to-many relationships are the dominant feature, graph models are natural. Vertices (nodes) connected by edges (relationships). Not limited to homogeneous data — can mix people, locations, events, etc.

Property Graph

Vertex: unique ID + properties (key-value) + set of incoming/outgoing edges. Edge: unique ID + tail vertex + head vertex + label + properties. Used by: Neo4j, Titan, InfiniteGraph.

Triple-Store

All info as (subject, predicate, object) triples. Subject = vertex. Object = value or another vertex. Used in semantic web, RDF. Query: SPARQL.

Cypher (Query Language)

Declarative query language for property graphs (Neo4j). Pattern matching: (person)-[:BORN_IN]->(loc)-[:WITHIN*0..]->(us {name:'US'})

Datalog: The Foundation

Older than SPARQL or Cypher. Data as predicate(subject, object) facts. Rules can be combined and reused. Queries build on rules incrementally — powerful for complex data.

First-Principles Questions

Test your understanding. Click to reveal the answer.

1Why do data models matter so much in software development?▼
Data models have a profound effect not only on how the software is written, but also on how we think about the problem we are solving. Each data model embodies assumptions about how it is going to be used — some usages are easy, some are not supported; some operations are fast, some perform badly. Choosing the right one is critical because it determines what the software above it can and can't do.
2What is the "impedance mismatch" and why does it arise?▼
The impedance mismatch is the disconnect between object-oriented programming (objects, classes) and the relational data model (tables, rows, columns). Application code uses objects; the database stores rows. An awkward translation layer is required between them. ORMs like ActiveRecord and Hibernate reduce boilerplate but can't completely hide the differences.
3Why does the book say "use an ID, not a human-readable string" for references? What principle does this relate to?▼
An ID has no meaning to humans, so it never needs to change. Human-meaningful info (like "Greater Seattle Area") may change, and if it's duplicated across records, all copies need updating — risking inconsistencies. Using IDs means human-meaningful info is stored in only one place. This is the key idea behind normalization.
4Why did the relational model win over the network model (CODASYL)?▼
The network model required programmers to manually track "access paths" — chains of pointers through records. Changing the data model meant rewriting all query code. The relational model laid all data in the open (tables and rows) and introduced the query optimizer which automatically decides access paths. You build the optimizer once, and all applications benefit. "The general-purpose solution wins in the long run."
5What is the difference between schema-on-read and schema-on-write?▼
Schema-on-read (document model): the structure is implicit; the data is interpreted when read. Like dynamic type checking — if data doesn't match expected format, you find out at runtime. Schema-on-write (relational model): the schema is explicit and enforced on every write. Like static type checking — the database guarantees data conforms to the schema. Schema-on-read is advantageous when items don't all have the same structure.
6Why is a declarative query language better than an imperative one for databases?▼
A declarative language (like SQL) specifies what results you want, not how to get them. This is more concise, easier to work with, and critically — it hides implementation details, allowing the database's query optimizer to choose the best execution strategy. It can also take advantage of parallel execution without you rewriting anything. Imperative code specifies operations in a particular order, making it harder for the system to optimize.
7When would you choose a graph model over relational or document?▼
When many-to-many relationships are the dominant feature of your data, and the connections between items are as important as the items themselves. Graph models excel when you have heterogeneous data (people, locations, events, etc.) with complex interconnections, and when you need to traverse relationships of unknown depth (like "find all people who live in a location within Europe").
8How are modern document databases different from the 1970s hierarchical model (IMS)?▼
They are similar in storing nested records within their parent (one-to-many as a tree). However, for many-to-one and many-to-many relationships, modern document databases use document references (like foreign keys) that are resolved at read time via follow-up queries. They have not followed CODASYL's approach of pointer-based access paths. Both relational and document databases use unique identifiers to reference related items.
9What does "data locality" mean for document databases, and when is it an advantage?▼
In a document database, the entire document (e.g. a JSON object) is stored as one contiguous block. If your application frequently needs the entire document, this provides better locality than a relational schema where data is split across multiple tables (requiring multiple queries or joins). However, the document is usually loaded entirely even if you only need a small part, and updates that change the document size may require rewriting the whole document.
10The book says relational and document models are converging. How?▼
Relational databases are adding support for JSON/XML documents (PostgreSQL, MySQL, IBM DB2) with the ability to query and index inside them. Document databases are adding relational-like features such as joins (RethinkDB supports join-like queries). The models are becoming more similar, and a hybrid approach may become the norm — each model learning from the other.
