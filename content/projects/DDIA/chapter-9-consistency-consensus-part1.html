Chapter 9 — Consistency and Consensus

Part 1: Linearizability — The Strongest Consistency Model

Distributed systems offer many consistency guarantees, from the weakest (eventual consistency) to the strongest (linearizability). Chapter 9 explores what these guarantees actually mean and how to achieve the strongest ones.

Consistency Guarantees

Most replicated databases offer eventual consistency: if you stop writing, all replicas will eventually converge to the same value. But "eventually" gives you no timing guarantees, and in the meantime, different replicas return different data.

Eventual consistency
All replicas converge if you stop writing. In the meantime: stale reads, write conflicts, no ordering guarantees.
Most NoSQL databases
Linearizability
The system appears as if there is a single copy of data. Every read sees the most recent write. All operations appear to happen at a single point in time.
ZooKeeper, etcd, Spanner

Linearizability

Linearizability (also called atomic consistency or strong consistency) is the strongest single-object consistency model. The core idea: make a distributed system appear as if it runs on a single machine with a single copy of data.

Formal definition: A system is linearizable if every operation appears to take effect atomically at some single point in time between the operation's start and end. Once a write completes, all subsequent reads must see the new value.

What Linearizability Looks Like

Timeline — two clients, one shared variable x
Time →
Client A: write(x=1) ——————————[completes]
Client B: ———————————————————read(x) → must return 1
Correct: B's read starts after A's write completes → B must see x=1
Client C: read(x) → might see 0 or 1 (overlaps with A's write)
C's read overlaps the write — could see either value, but once C sees 1, all subsequent reads must also return 1

Linearizability vs. Serializability

These two terms are often confused:

PropertyLinearizabilitySerializability
Applies to Individual read/write operations on a single object Transactions (multi-object, multi-step)
What it guarantees Operations appear to happen at a single point in real time Transactions appear to execute in some serial order
Timing Must respect real-time ordering Serial order needn't match real-time order
Combined Strict serializability = serializability + linearizability (most expensive, what Spanner provides)

Relying on Linearizability

Which use cases actually require linearizability?

1. Locking and Leader Election

A distributed lock must be linearizable: only one node can hold the lock at a time. Any linearizability violation means two nodes could both believe they hold the lock — split brain. ZooKeeper and etcd are used for this precisely because they provide linearizable reads and writes.

2. Constraints and Uniqueness Guarantees

Enforcing a unique username, unique flight seat booking, or unique account number requires linearizability. Two users registering the same username simultaneously: if the check-then-set is not linearizable, both may succeed.

User A: check(username "alice" available?) → yes
User B: check(username "alice" available?) → yes
User A: register("alice") → OK
User B: register("alice") → OK
Without linearizability: both succeed. Constraint violated.
With linearizability: one succeeds, the other fails (sees "alice" taken).

3. Cross-Channel Timing Dependencies

If a system uses two different communication channels (e.g., file storage + message queue), linearizability violations can cause races:

1. User uploads photo → stored in file store
2. Server sends message to image resizer via queue: "resize photo at /path"
3. Resizer reads /path → gets old version (replica hasn't caught up yet)
File storage isn't linearizable: resizer reads stale data, produces wrong thumbnail.

Implementing Linearizable Systems

How can different replication strategies achieve linearizability?

Replication styleLinearizable?Notes
Single-leader (read from leader) Potentially yes Leader has the latest data. But: if you can't identify the current leader, or the leader has stale data during failover, reads may be stale.
Single-leader (read from followers) No Followers may lag. You may read stale data from an async follower.
Multi-leader No Multiple nodes accept writes concurrently. Conflicts must be resolved. Inherently non-linearizable.
Leaderless (Dynamo-style) Probably not Quorum reads/writes don't guarantee linearizability due to clock skew, concurrent writes, and read-repair races.
Consensus algorithms (Raft, Paxos) Yes Designed to be linearizable. ZooKeeper, etcd are consensus-based. Reads from leader are linearizable; sync reads require round-trip.
Even single-leader setups can fail to be linearizable: if a node thinks it's the leader but has been replaced, it may serve stale reads. True linearizability requires confirming leadership on every read — a round-trip per read.

The Cost of Linearizability

Linearizability is expensive. The CAP theorem captures the core tension.

CAP Theorem

Consistency
Every read sees the most recent write (linearizability). Safety property.
Availability
Every request gets a response (not an error). Liveness property.
Partition Tolerance
System continues operating despite network partitions. Not optional — partitions happen.
During a network partition: you must choose C or A. You cannot have both.
CAP is often misunderstood. A better framing: when a network partition occurs, a system must choose between (a) remaining available and serving potentially stale data, or (b) becoming unavailable until the partition heals. "CP" systems choose (b); "AP" systems choose (a). Most real systems are somewhere in between.

Network Partition and Linearizability

Network partitioned: Node A and Node B cannot communicate.
Client 1 writes x=1 to Node A. Write completes.
Client 2 reads x from Node B.
Option 1 (CP): Node B refuses the read (returns error). Linearizable, but unavailable.
Option 2 (AP): Node B returns x=0 (old value). Available, but not linearizable.
No option satisfies both. This is the unavoidable CAP trade-off.

Linearizability and Performance

Even without partitions, linearizability is slow. Achieving it requires:

  • Reading from the leader (or doing a round-trip to confirm leadership)
  • Every read potentially waiting for in-progress writes to complete
  • Replication latency adds to read/write latency

This is why multi-datacenter systems (multi-leader or leaderless) often sacrifice linearizability for lower latency — the cross-DC round-trip needed for linearizable reads is too expensive.
