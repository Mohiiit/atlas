DDIA -- Part II: Distributed Data

Chapter 9, Part 1: Linearizability and Ordering

Consistency and Consensus are deeply linked. This first part explores linearizability -- the strongest single-object consistency guarantee -- and ordering guarantees including causal ordering, Lamport timestamps, and total order broadcast, which together build the bridge to consensus.

1. The Spectrum of Consistency Guarantees

Distributed databases offer a range of consistency models. Stronger models are easier to reason about but more expensive. Weaker models are performant and available but shift complexity to the application developer.

Eventual Consistency

If you stop writing, all replicas eventually converge. No timing guarantee. You may read stale data at any time.

Linearizability

System behaves as if there is a single copy of the data and all operations are atomic. A recency guarantee: reads always return the latest write.

"The idea behind linearizability is to make a system appear as if there were only one copy of the data, and all operations on it are atomic." -- Kleppmann

2. What Makes a System Linearizable?

Linearizability (also called atomic consistency, strong consistency, or external consistency) is a recency guarantee: once a write completes, all subsequent reads must return that value (or a later one). The key insight is that there must be a single point in time during each operation when it "takes effect" atomically, and these points must form a valid sequential order.

The Core Rules

Three Requirements for Linearizability

  • If op B starts after op A finishes -- B must see A's effect (or something later).
  • Once any read returns a new value -- all subsequent reads (from any client) must also return the new value. No "flipping back."
  • Each operation appears to take effect atomically at some instant between its invocation and completion.

The Atomic Flip: 3 Clients Timeline

Diagram 2 -- Linearizability Timeline (3 Clients, Atomic Flip)

Each operation takes effect at its linearization point (dot). The value of x flips atomically from 0 to 1 at the red dashed line. Once Client A's read returns 1, Client B's subsequent read must also return 1 -- it cannot flip back to 0.

3. Linearizability vs Serializability

These two terms sound similar but describe fundamentally different guarantees. They can be combined into strict serializability (strong-1SR).

Key Distinction

Serializable Snapshot Isolation (SSI) is serializable but not linearizable -- it reads from a consistent snapshot that intentionally excludes recent writes to avoid lock contention. 2PL and actual serial execution are typically both serializable and linearizable.

4. When Is Linearizability Needed?

Linearizability is not always required, but there are three important scenarios where it is essential for correctness.

Scenario 1: Locking and Leader Election

Diagram 4 -- Leader Election via Linearizable Lock

All nodes must agree on who holds the lock. If the lock is not linearizable, two nodes could both believe they are the leader (split brain), leading to data divergence and corruption.

Scenario 2: Uniqueness Constraints

Diagram 5 -- Concurrent Username Registration

An atomic compare-and-set on a linearizable register ensures exactly one user claims the username. The same principle applies to bank account balances, seat reservations, and stock levels.

Scenario 3: Cross-Channel Timing Dependencies

Diagram 6 -- Image Resizer Race Condition

Two communication channels (file storage + message queue) create a race. If the storage is not linearizable, the queue message may arrive before the image is visible to the resizer, producing a permanently inconsistent thumbnail.

5. Implementing Linearizable Systems

Can replication strategies provide linearizability? The answer depends on the approach.

Quorums Are Not Enough

Even with strict quorums (w+r > n), a Dynamo-style system can violate linearizability: Client A reads the new value from a quorum, but concurrently Client B reads the old value from a different quorum. B started after A finished, so this is a linearizability violation.

Making quorums linearizable requires synchronous read repair before returning results and writers reading latest state before writing. Even then, linearizable CAS requires a consensus algorithm.

6. The CAP Theorem

CAP is often misquoted as "pick 2 out of 3 (Consistency, Availability, Partition tolerance)." In reality, network partitions will happen whether you like it or not. The real question is: when a partition occurs, do you choose consistency or availability?

Diagram 8 -- Network Partition Scenario: Consistent or Available?

"Consistent or Available when Partitioned"

A better phrasing of CAP:

  • Choose Consistency (CP): Disconnected replicas refuse requests (return errors). System is unavailable but never returns stale data.
  • Choose Availability (AP): Every replica processes requests independently. System stays up but may return stale/conflicting data.

When the network is healthy, you get both C and A. The choice only matters during a partition.

Why CAP Has Limited Practical Value

  • It only considers one consistency model (linearizability) and one fault type (network partitions).
  • It says nothing about network delays, dead nodes, or other trade-offs.
  • The real reason most systems drop linearizability is performance, not fault tolerance -- even multi-core CPU RAM is not linearizable (each core has its own cache).
  • Linearizability is inherently slow: response time is proportional to network delay uncertainty (proven by Attiya and Welch).

7. Causal Ordering vs Total Ordering

Ordering is deeply connected to consistency. The relationship between causal consistency and linearizability is captured by two different kinds of ordering.

Diagram 9 -- Total Order vs Partial (Causal) Order

Linearizability = Total Order

Single copy illusion. Every operation is on one timeline. For any two operations, one happened before the other. No concurrency exists.

Causality = Partial Order

Operations are ordered only if causally related. Concurrent (independent) operations are incomparable -- like Git branches that haven't merged.

Linearizability Implies Causality (but not vice versa)

Linearizability is strictly stronger than causal consistency. Any linearizable system automatically preserves causality. But causal consistency can be achieved without the performance cost of linearizability.

Causal consistency is the strongest model that does not slow down due to network delays and remains available during network failures. The CAP theorem does not apply to causal consistency.

8. Lamport Timestamps

How do you create a total ordering that is consistent with causality in a system with no single leader? Lamport timestamps (Leslie Lamport, 1978) provide an elegant solution.

How They Work

Diagram 10 -- Lamport Timestamps in Action

Lamport Timestamp = (counter, nodeId)

  • Each node maintains a counter, incremented on every operation.
  • Every request/response carries the node's current maximum counter.
  • When a node receives a counter greater than its own, it immediately jumps to that value.
  • Ties in counter are broken by node ID -- guaranteeing a unique total order.
  • Consistent with causality: if A causally happened before B, then A's timestamp < B's timestamp.

Why Lamport Timestamps Are Insufficient

Diagram 11 -- The Username Problem: Timestamps Can't Decide in Real Time

Lamport Timestamps: After-the-Fact, Not Real-Time

  • They provide a total order consistent with causality -- but only after collecting all operations.
  • A node receiving a request right now cannot decide if it should succeed or fail -- another node might have a concurrent operation with a lower timestamp.
  • To be sure, you'd need to check every other node, and if one is down, you're stuck.
  • This is why we need total order broadcast: to know when the order is finalized.

Lamport Timestamps vs Version Vectors

  • Version vectors can distinguish concurrent operations from causally ordered ones.
  • Lamport timestamps always enforce a total order -- you cannot tell if two operations were concurrent or causal from timestamps alone.
  • Lamport timestamps are more compact (a single counter + nodeId vs a vector of counters).

9. Total Order Broadcast

Total order broadcast (also called atomic broadcast) solves the problem Lamport timestamps cannot: it provides a finalized total order that nodes can act on in real time.

Two Safety Properties

1. Reliable Delivery

No messages are lost. If a message is delivered to one node, it is delivered to all nodes.

2. Totally Ordered Delivery

Messages are delivered to every node in the same order. No node sees a different sequence.

Uses of Total Order Broadcast

Diagram 12 -- Total Order Broadcast Applications

Implementing Linearizable Storage from Total Order Broadcast

Linearizable CAS via Total Order Broadcast (Username Example)

Step 1: Append to log
Append a message: "I want to claim username 'jdoe'"
Step 2: Wait for delivery
Read the log and wait until your own message is delivered back to you.
Step 3: Check for conflicts
If the first message claiming "jdoe" is yours -- success! Commit. If someone else's message came first -- abort.

Because all nodes see messages in the same order, they all agree on who "won." This gives linearizable writes. For linearizable reads, you must also sequence reads through the log (or use sync replicas).

The Other Direction: Total Order Broadcast from Linearizable Storage

Using a Linearizable Integer Register

  • Assume a linearizable register with an atomic increment-and-get operation.
  • For every message, call increment-and-get to get the next sequence number.
  • Attach that number to the message; recipients deliver messages in sequence-number order.
  • Unlike Lamport timestamps, these numbers have no gaps -- if you have msg 4 and receive msg 6, you know msg 5 exists and must wait for it.

The Equivalence

Diagram 13 -- The Consensus Equivalence Triangle

This is a profound result: a linearizable compare-and-set register, total order broadcast, and consensus are all equivalent problems. If you can solve one, you can build the other two. This insight connects Part 1 (linearizability and ordering) to Part 2 (consensus algorithms like Raft, Paxos, Zab).

Review Questions

1A system uses strict quorums (w + r > n) with Dynamo-style replication. Is it linearizable? Why or why not? ▾

No, it is probably not linearizable. Even with strict quorums, race conditions are possible due to variable network delays. Client A can read the new value from its quorum while concurrently Client B reads the old value from a different quorum -- even though B's request started after A's completed. Making it linearizable requires synchronous read repair before returning results and writers reading latest state before writing, and even then a linearizable compare-and-set requires a consensus algorithm. Systems like Cassandra and Riak do not guarantee linearizability with quorums.

2An engineer says "Our system provides serializability, so it must also be linearizable." Is this correct? ▾

Not necessarily. Serializability and linearizability are different guarantees. Serializability is an isolation property of transactions (multi-object), while linearizability is a recency guarantee on individual objects. Serializable Snapshot Isolation (SSI) is serializable but explicitly not linearizable -- it reads from a consistent snapshot that excludes recent writes. However, 2PL and actual serial execution are typically both serializable and linearizable (strict serializability).

3Why can't Lamport timestamps alone solve the unique username problem in a distributed system? ▾

Lamport timestamps provide a total order consistent with causality, but this order only emerges after collecting all operations. When a node receives a username registration request right now, it cannot determine whether another node is concurrently processing a registration for the same username with a lower timestamp. To find out, it would have to check every node -- and if any node is unreachable, the system halts. The total order is an after-the-fact ordering, not a real-time decision mechanism. This is why we need total order broadcast, which tells you when the order is finalized.

4A startup deploys multi-leader replication across three datacenters. Can they offer linearizable reads to users? What trade-off are they making? ▾

No. Multi-leader replication is fundamentally not linearizable because writes are processed concurrently on multiple leaders and asynchronously replicated. There is no single copy of the data. Conflicting writes require resolution (last-write-wins, merge, etc.), which is an artifact of not having linearizability. The trade-off is: they gain availability during network partitions and lower write latency (writes are local to each datacenter), but lose the guarantee that all clients see the same, most recent data. This is the "Available when Partitioned" choice from CAP.

5Explain how total order broadcast can be used to implement a linearizable compare-and-set register. What additional step is needed for linearizable reads? ▾

Linearizable writes via total order broadcast: (1) Append a tentative claim message to the log. (2) Wait for your message to be delivered back. (3) Check if yours is the first message for that key -- if yes, commit; if no, abort. All nodes see the same order, so they agree on who wins.

For linearizable reads: The above only gives sequential consistency (writes are linearizable, but reads from an async replica may be stale). Three options: (a) sequence reads through the log too (append a read message, wait for delivery, then read); (b) fetch the latest log position in a linearizable way, wait for all entries up to that position, then read (ZooKeeper's sync()); (c) read from a synchronously updated replica (chain replication).

6Why is causal consistency considered the "sweet spot" for distributed systems? How does it relate to linearizability and the CAP theorem? ▾

Causal consistency is the strongest possible consistency model that does not slow down due to network delays and remains available during network failures. Unlike linearizability, it does not require coordinating through a single point (no single timeline), so the CAP theorem does not apply -- you can have causal consistency even during partitions.

Linearizability implies causality (any linearizable system is also causally consistent), but causal consistency is strictly weaker -- it only orders operations that are causally related and allows concurrent operations to be unordered. Many systems that seem to require linearizability actually only need causal consistency, which can be implemented more efficiently. The trade-off: causal consistency cannot enforce constraints like uniqueness (that requires knowing the finalized order), but it handles most read-after-write and ordering requirements correctly.
