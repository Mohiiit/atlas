Chapter 7 — Transactions

Part 1: The Concept of Transactions, ACID, and Read Committed Isolation

Transactions are one of the most important abstractions in databases. They take a bunch of reads and writes and wrap them up as a single logical unit — either everything commits, or it all rolls back. You don't have to worry about partial failures, concurrent access, or machine crashes mid-operation.

Not every application needs transactions. But understanding what transactions give you helps you make an informed decision about when to use them — and what you're giving up when you don't.

The Slippery Concept of a Transaction

Transactions were introduced by IBM System R in 1975. NoSQL databases (2000s) moved away from them for scale and flexibility. But the safety guarantees they provide are genuinely valuable, and many NoSQL databases have reintroduced transaction-like features.

Why we need transactions: things that can go wrong
Database crashes mid-write
Application crashes between two related writes
Network partition splits write across nodes
Two clients write concurrently, corrupting data
Client reads partially-updated data
Race condition causes incorrect balance
Without transactions, your application code must handle all of these. With transactions, the database handles them.

The Meaning of ACID

ACID is the set of properties that transactions provide. The term was coined by Theo Härder and Andreas Reuter in 1983.

A — Atomicity
All-or-nothing. If any part fails, the whole transaction is aborted. No partial updates visible.
C — Consistency
The database goes from one valid state to another. Application-defined invariants must hold (e.g., no negative balances).
I — Isolation
Concurrent transactions appear as if they ran serially. You don't see another transaction's intermediate state.
D — Durability
Once committed, data survives crashes. Written to disk + WAL. In distributed systems: replicated.
The "C" in ACID is a bit of a lie — it's really about application-level invariants, not something the database enforces automatically. The database ensures AID; consistency is your responsibility to define.

Atomicity: Abortability

If a transaction has made several writes and then crashes or is aborted, all partial writes are undone. This property is better called abortability.

BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE user_id = alice;
  UPDATE accounts SET balance = balance + 100 WHERE user_id = bob;
COMMIT;  -- both changes visible at once, or neither
Success path
Alice -100
Bob +100
COMMIT
Both visible atomically
Failure path
Alice -100
Crash before Bob +100
ROLLBACK
Alice gets her money back

Isolation: The Hard Part

Full isolation means serializable isolation — as if transactions ran one at a time, even if they're concurrent. But serializability has a performance cost. In practice, databases offer weaker isolation levels that prevent some but not all race conditions.

Durability

Committed data must be stored persistently. Typically: write-ahead log, fsync to disk before acknowledging commit. In distributed databases: data replicated to N nodes before ack. "Perfect" durability is impossible — it's a spectrum.

Single-Object and Multi-Object Operations

The classic transaction model (above) handles multi-object operations — multiple rows across multiple tables in one atomic operation. This requires the database to determine which writes belong to the same transaction (often via a session token or transaction ID).

Why multi-object transactions are necessary
Example 1: Foreign key
INSERT into orders (user_id, ...)
user_id references users table
→ both tables must be updated consistently
Example 2: Denormalized document
Update username in users table
AND update username in all their posts
→ must be atomic or data is inconsistent
Example 3: Counters
Read unread count from counters table
Update emails table (mark as read)
→ counter and data must stay in sync
Example 4: Transfer
Debit Alice's account
Credit Bob's account
→ must be atomic (no money appears/disappears)

Many distributed databases have dropped multi-object transactions because they're hard to implement across nodes. If you need them, you must handle them at the application level — or use a database that supports distributed transactions.

Weak Isolation Levels

Serializable isolation means no concurrency anomalies — but it's expensive. Database vendors offer weaker isolation levels that prevent only some anomalies. Understanding exactly what each level prevents (and what it doesn't) is critical.

Warning: Isolation level names are not standardized across databases. "Repeatable Read" in PostgreSQL is different from "Repeatable Read" in MySQL. Always check the specific database's documentation.

Read Committed

The weakest useful isolation level. Provides two guarantees:

  1. No dirty reads — You only see data that has been committed (not partially-written by an in-progress transaction)
  2. No dirty writes — You only overwrite data that has been committed (not overwrite another transaction's in-progress write)
Dirty Read — prevented by Read Committed
Time
Transaction 1
Transaction 2
t1
BEGIN; SET x = 3
t2
READ x → should NOT see 3!
t3
ROLLBACK
Too late — Tx2 already acted on the dirty value
With Read Committed: Tx2 at t2 still sees old value of x (before Tx1's uncommitted write)
Dirty Write — prevented by Read Committed
Time
Bob buys car
Alice buys car
t1
SET listing.buyer = Bob
t2
SET listing.buyer = Alice (overwrites Bob's uncommitted!)
t3
SET invoice.buyer = Bob
Result without protection: Alice gets the car, Bob gets the invoice. Neither has a complete transaction.
With dirty write prevention: Tx2 waits until Tx1 commits or aborts before writing.

Implementing Read Committed

How do databases implement "no dirty reads" without blocking all reads?

Row-level versioning approach
In-progress update
Database holds both:
old value (committed)
new value (uncommitted)
→
Other readers
See old committed value
until the write commits,
then see new value
Row-level locks for dirty write prevention: writer acquires lock on row, other writers must wait. Readers never blocked.

What Read Committed Does NOT Prevent

Read Committed prevents dirty reads and dirty writes. But it does NOT prevent all concurrency problems. The classic failure: non-repeatable reads (read skew).

Read skew — allowed by Read Committed
Time
Alice (checking accounts)
Transfer transaction
t1
READ account1 = $500
t2
Transfer $100: account1 → account2. COMMIT.
t3
READ account2 = $200 (new value)
Alice sees account1=$500 (before transfer) and account2=$200 (after transfer). She sees $700 total when the correct total is $600. Neither read was dirty, but the combined view is inconsistent.

This is a problem for:

  • Backups: A multi-hour backup process reads different parts of the database at different times. Backup reflects an inconsistent state that never actually existed.
  • Integrity checks: Scanning the whole database for consistency violations — reads at different times will see inconsistent combinations.
  • Long-running reports: The report computes totals across many tables; concurrent updates cause it to see some data before and some after an update.

The solution: Snapshot Isolation — covered in Part 2.
