2. Dataflow Through Databases

In a database, the process that writes encodes the data, and the process that reads decodes it. Often it's the same process at a different time — "sending a message to your future self." But during a rolling upgrade, different processes at different code versions access the same database simultaneously.

Solution: Application code must preserve unknown fields when reading and rewriting a record. Some encoding libraries (Avro) handle this automatically. Not all do.

"Data Outlives Code"

You can replace all running code in minutes via a rolling upgrade. But database contents stay in their original encoding unless explicitly migrated. Data written 5 years ago is still there in the old format. A single database may contain records encoded with many different historical schema versions.

Schema evolution allows the entire database to appear as though encoded with a single schema (the current one), even though the underlying records use many versions. Most relational databases allow simple additions (ALTER TABLE ADD COLUMN NULL) without rewriting existing rows — old rows simply return null for the new column.

3. Dataflow Through Services: REST and RPC

The most common network communication pattern: clients and servers. The server exposes an API; the client connects and makes requests. A server can itself be a client to another service (microservices architecture). The key constraint: services must be independently deployable and evolvable — old and new versions run simultaneously.

Why "Location Transparency" in RPC Is Fundamentally Flawed

RPC frameworks try to make a remote call look like a local function call. This is a leaky abstraction. A network request is fundamentally different from a local call:

  • Unpredictable network: requests and responses can be lost, delayed, or duplicated
  • Timeouts: a local call either returns or throws. A network call might just hang — you don't know if the request arrived
  • Retries may duplicate: if you retry after a timeout, the request may have already executed. Need idempotence
  • Wildly variable latency: local = nanoseconds; network = milliseconds to seconds
  • Parameters must be serialized: no pointers, no complex in-memory objects
  • Cross-language types: JavaScript number ≠ Java long ≠ Go int64

Modern RPC frameworks (gRPC, Finagle) acknowledge this by exposing futures/promises, streams, and explicit error types. They don't pretend remote = local.

Compatibility Direction for Services

In practice, servers are updated before clients. So: requests need backward compatibility (new server handles old client requests). Responses need forward compatibility (old clients handle new server responses with extra fields). API versioning (/api/v1/) is common for REST to signal which schema the client expects.

4. Async Message-Passing Dataflow

Between direct RPC and databases sits asynchronous message passing: a message broker (message queue) intermediates. The sender writes to a named queue or topic; the broker delivers to one or more consumers. The sender fires and forgets — no waiting for a response.

Message Broker Advantages Over Direct RPC

  • Buffer if recipient unavailable: messages queue until consumer is ready — improves resilience
  • Auto-redeliver on consumer crash: message isn't lost if consumer dies mid-processing
  • Sender doesn't need recipient's address: useful in cloud with dynamically created/destroyed VMs
  • Fan-out: one published message → multiple consumers without modifying the sender
  • Logical decoupling: add new consumers without touching producers

Message brokers don't enforce a data model — a message is just bytes with metadata. You can use any encoding. If the encoding is forward and backward compatible, publishers and consumers can be deployed independently in any order.

5. Distributed Actor Frameworks

The actor model is a concurrency model: logic is encapsulated in actors, each with local state, communicating by sending and receiving asynchronous messages. Unlike threads, actors never share state. Message delivery isn't guaranteed even within a single process — the model embraces this from the start.

Why Actors Handle Distribution Better Than RPC

The actor model already assumes messages may be lost within a single process. When extended across nodes, there's less fundamental mismatch — the model was designed for unreliable delivery from day one. RPC fails because it tries to hide network unreliability behind a synchronous function call interface. Actors embrace it. That said: rolling upgrades still require encoding compatibility, and cross-node latency is still higher than intra-node.

First-Principles Questions

Click to reveal. Try to answer before expanding.

1Why do we need BOTH backward and forward compatibility? Why isn't one sufficient?▼
During rolling upgrades, some nodes run new code while others still run old code. New code must read old data (backward compatibility) — easy, because you wrote the new code and know the old format. Old code must handle data from new code without breaking (forward compatibility) — trickier, because old code must gracefully ignore additions it doesn't know about. Without both, you cannot do rolling upgrades — you'd need to update all nodes simultaneously, which requires downtime. For client-side apps it's even worse: users may not upgrade for months, giving you many versions in the wild simultaneously.
2Why are language-specific formats (Java Serializable, Python pickle) dangerous for anything beyond transient purposes?▼
Four deep problems: (1) Language lock-in: encoding tied to one language; other languages can't read the data, committing you potentially for years. (2) Security: decoding instantiates arbitrary classes. An attacker who feeds your application a crafted byte sequence can trigger arbitrary class instantiation → remote code execution. (3) No versioning: forward and backward compatibility are afterthoughts — schema evolution is painful or impossible. (4) Performance: Java's built-in serialization is notorious for slow speed and bloated output size.
3Why is MessagePack only marginally smaller than JSON, while Protobuf is dramatically smaller?▼
MessagePack keeps the JSON data model: it still includes all field names ("userName", "favoriteNumber") as strings in every encoded record, because it has no schema. The field name strings dominate the payload. Protocol Buffers uses a schema with numbered field tags — field names never appear in the encoded data, only small tag numbers (1, 2, 3). Additionally, Protobuf uses variable-length integers so small numbers use fewer bytes. The biggest savings come from omitting field names, not from binary-encoding the values.
4How do Thrift/Protobuf achieve forward compatibility (old code reads data written by new code)?▼
Fields are identified by tag numbers and include a datatype annotation. When old code encounters a tag number it doesn't recognize (a new field added by newer code), the datatype annotation tells the parser exactly how many bytes to skip. The unknown field is ignored. This is why old code can safely read records written by new code — it skips what it doesn't understand. This mechanism also explains why new fields must be optional or have defaults: if new code reads old data, the new field is absent, and the code must handle that gracefully.
5Why can you never change a field's tag number, and why can you never reuse a removed field's tag?▼
The tag number is the identity of the field in encoded bytes. Changing it makes every existing encoded record unreadable — the parser maps tag → wrong field, causing data corruption. Reusing a removed tag is equally dangerous: old data in databases, message queues, and log files still contains records with that tag number. The parser would silently misinterpret the old field as the new field — subtle, hard-to-detect corruption. These rules are absolute because encoded data can persist for years, long after the original code is gone.
6How is Avro's schema evolution fundamentally different from Thrift/Protobuf, and what use case does this enable?▼
Avro has no tag numbers. Fields are matched by name between writer's and reader's schema. This enables dynamically generated schemas. Example: dumping a relational database to binary files. With Avro, you auto-generate a schema from the database schema (column names → field names). When the DB schema changes, regenerate the Avro schema automatically. With Thrift/Protobuf, you'd need to manually assign tag numbers every time a column is added or removed — one mistake and you corrupt data. Avro was designed specifically for this Hadoop use case.
7What does "data outlives code" mean, and what specific danger does it create?▼
You can replace all running code in minutes. But database contents stay in their original encoding until explicitly migrated — data written 5 years ago is still there. During a rolling upgrade, an old app instance may read a record written by new code (with a new "email" field), update some other field, and write the record back without the unknown "email" field — silently losing data. The solution: encoding libraries (or application code) must preserve unknown fields when reading and rewriting records. Not all libraries do this correctly.
8Why is "location transparency" in RPC fundamentally flawed?▼
Location transparency tries to hide the difference between local and remote calls. But six properties make them fundamentally different: (1) Unpredictable network — requests can be lost. (2) Timeouts — no way to know if request succeeded. (3) Retries may execute twice — idempotence required. (4) Variable latency — nanoseconds vs milliseconds. (5) Serialization required — no pointers or complex objects. (6) Cross-language types — JavaScript number ≠ Java long. Hiding all this behind a function call leads to brittle systems. Modern frameworks (gRPC) expose futures, streams, and errors explicitly.
9What are the five advantages of a message broker over direct RPC?▼
(1) Buffer if recipient unavailable: messages queue until consumer is ready — improves resilience. (2) Auto-redeliver on crash: consumer crash doesn't lose the message — broker redelivers to another worker. (3) Sender doesn't need recipient's address: useful in cloud where VMs come and go dynamically. (4) Fan-out: one published message delivered to multiple subscribers without modifying the sender. (5) Logical decoupling: add new consumers without touching producers. The fundamental trade-off: communication is one-way and asynchronous. If you need a response, you must implement a separate reply-queue pattern.
10Why does the actor model handle distribution better than traditional RPC?▼
The actor model already assumes messages may be lost, even within a single process. All communication is asynchronous and one-way — no waiting for responses. When extended across nodes, there's less fundamental mismatch because the model was designed for unreliable delivery from the start. RPC fails because it tries to hide network unreliability behind a synchronous interface, which is a lie — the network is unreliable. Actors embrace it. That said, rolling upgrades still require encoding compatibility, and Akka's default Java serialization is a known pain point.
11Why did Twitter have to include tweet IDs twice in their API responses?▼
Tweet IDs are 64-bit integers. JavaScript's Number type (IEEE 754 double-precision float) can only represent integers exactly up to 253 ≈ 9 quadrillion. Tweet IDs can exceed this. When a JavaScript app parses a large tweet ID as a JSON number, the last few digits are silently corrupted due to floating-point precision loss. Twitter's workaround: include each ID twice — once as a JSON number (for non-JS consumers) and once as a decimal string (for JavaScript consumers). This is a direct consequence of JSON having no integer/float distinction and no way to specify numeric precision.
12For service requests vs responses in microservices, which compatibility direction matters and why?▼
Servers are typically updated before clients. So: requests need backward compatibility — the new server must understand requests from old clients (which use the old request schema). Responses need forward compatibility — old clients must tolerate responses from the new server (which may include new fields they don't know about). Think of it directionally: the server is always "ahead" of the client. It receives old-format requests (backward compat) and sends new-format responses that old clients must silently tolerate (forward compat). API versioning (/api/v1/) lets the server know which schema the client expects, giving you an escape hatch when changes are too large.
