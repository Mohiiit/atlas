Stream Processing: Event Streams, Messaging, and Logs

From batch to real-time: how event streams replace fixed time slices, the landscape of messaging systems, and why log-based brokers like Kafka changed everything.

From Batch Processing to Streams

In batch processing you collect a day's (or hour's) worth of data and process it all at once. The fundamental limitation: changes in the input are only reflected in the output after the batch completes. To reduce that delay, you can run processing more frequently -- every second, or even continuously, processing every event as it happens. That is stream processing.

What is an Event?

A small, self-contained, immutable object containing the details of something that happened at some point in time. It usually contains a timestamp and can be encoded as JSON, text, or binary.

A producer (publisher/sender) generates an event once. Multiple consumers (subscribers/recipients) can then process it. Related events are grouped into a topic or stream. In principle, a file or database could connect producers to consumers, but polling becomes expensive at low latencies. Instead, specialized messaging systems push notifications to consumers.

Messaging Systems

A messaging system lets a producer send a message containing an event, which is then pushed to consumers. Two critical design questions differentiate all messaging systems:

Q1: What if producers outpace consumers?

  • Drop messages -- accept data loss
  • Buffer in a queue -- what if queue overflows? Memory? Disk?
  • Backpressure (flow control) -- block the producer (Unix pipes, TCP)

Q2: What if nodes crash?

  • Accept message loss -- higher throughput, lower latency
  • Write to disk -- durability at cost of speed
  • Replicate -- durability + availability

Direct Messaging: Producer to Consumer

Several systems skip intermediate brokers entirely, communicating directly between producers and consumers. Each has distinct tradeoffs:

Limitations of Direct Messaging

  • Application code must handle message loss explicitly
  • Generally assumes producers and consumers are constantly online
  • If consumer is offline, messages sent during that time are lost
  • If producer crashes, its retry buffer is lost
  • Limited fault tolerance compared to broker-based approaches

Message Brokers

A message broker (message queue) is a server that acts as an intermediary. Producers write messages to the broker; consumers read from it. The broker centralizes durability, buffering, and consumer management.

A key property: consumers are asynchronous. When a producer sends a message, it only waits for the broker to confirm buffering, not for consumer processing. Delivery happens at some undetermined future point -- often sub-second, but potentially much later under backlog.

By centralizing data in the broker, these systems can more easily tolerate clients that come and go (connect, disconnect, and crash), and the question of durability is moved to the broker instead.

  • Each message delivered to one consumer
  • Parallelizes expensive processing
  • Add consumers to scale throughput
  • JMS: shared subscription

Fan-Out (JMS: topic subscription)

  • Each message delivered to all consumers
  • Independent consumers "tune in" to broadcast
  • Streaming equivalent of multiple batch jobs reading same file
  • AMQP: exchange bindings

Acknowledgments and Redelivery

Consumers may crash at any time. To prevent message loss, brokers use acknowledgments: a consumer must explicitly tell the broker it has finished processing a message. If the connection closes or times out without an ack, the broker redelivers to another consumer.

Diagram 6: The Message Reordering Problem
Producer
Broker Queue
Consumer 1
Consumer 2
Step 1: Broker delivers m3 to Consumer 2, m4 to Consumer 1
Both consumers begin processing in parallel
Step 2: Consumer 2 crashes while processing m3
m3 is NOT acknowledged -- broker considers it unprocessed
Step 3: Consumer 1 finishes m4, acknowledges it
m4 is removed from the queue
Step 4: Broker redelivers m3 to Consumer 1
Consumer 1 now processes m5
Result: Consumer 1 processed m4, then m3, then m5
Order violated! m3 and m4 delivered out of original order.

Key Insight: Load Balancing + Redelivery = Reordering

Even if the broker tries to preserve message order (as JMS and AMQP standards require), the combination of load balancing with acknowledgment-based redelivery inevitably leads to messages being reordered.

Workarounds:

  • Use a separate queue per consumer (no load balancing)
  • Accept reordering if messages are completely independent
  • Use partitioned logs (Kafka approach) -- discussed next

Partitioned Logs

Traditional message brokers treat messaging as transient: messages are deleted after delivery. But databases treat data as durable: kept until explicit deletion. Can we combine durable storage with low-latency notification? That is the idea behind log-based message brokers.

Why can we not have a hybrid, combining the durable storage approach of databases with the low-latency notification facilities of messaging? This is the idea behind log-based message brokers.

Using Logs for Message Storage

A log is an append-only sequence of records on disk. A producer appends messages to the end; a consumer reads sequentially. When the consumer reaches the end, it waits for notification of new appends (like tail -f). To scale throughput, the log is partitioned across multiple machines.

Key Properties of Partitioned Logs

  • Monotonically increasing offset per partition -- makes ordering trivial within a partition
  • Append-only -- messages are immutable once written
  • Consumer reads sequentially -- single-threaded, straightforward
  • Partitions enable parallelism -- each partition is an independent log on potentially different machines
  • Fault tolerance via replication -- each partition is replicated across brokers
  • Implementations: Apache Kafka, Amazon Kinesis Streams, Twitter DistributedLog

Consumer Offsets

Because a consumer reads a partition sequentially, tracking progress is simple: all messages with an offset less than the consumer's current offset have been processed; all messages with a greater offset have not. The broker only needs to periodically record each consumer's offset -- not track acknowledgments per message.

Analogy: Database Replication
The consumer offset is very similar to a log sequence number (LSN) in database replication. The broker acts like a leader database; the consumer acts like a follower. If a consumer fails, another node in the group takes over its partitions and resumes from the last recorded offset.

Disk Space and Buffering

The log is divided into segments. Old segments are deleted or archived to reclaim space. This makes the log a bounded-size buffer (circular/ring buffer on disk). Back-of-envelope: a 6 TB drive at 150 MB/s sequential write = ~11 hours of buffer at max throughput. In practice, logs retain days or weeks of data.

If a consumer falls behind: Only that consumer is affected. It does not disrupt other consumers. You can even consume a production log for dev/testing without impacting production services.

Log Compaction

Instead of keeping all log history or taking periodic snapshots, log compaction retains only the most recent value for each key. The storage engine periodically scans for duplicate keys and discards older entries, keeping just the latest update. A special null value (tombstone) marks deletions.

When to Use Which?

Choose Traditional Broker When...

  • Messages are expensive to process individually
  • You need message-level parallelism (not partition-level)
  • Message ordering is not critical
  • You need complex routing patterns (exchanges, bindings)
  • Messages can be deleted after processing
  • You need protocol standards (AMQP, JMS compliance)

Choose Log-Based Broker When...

  • You need high throughput (millions of msgs/sec)
  • Each message is fast to process
  • Message ordering matters
  • You need to replay / reprocess messages
  • Multiple independent consumers need the same data
  • You want durable storage with notification
  • Building event sourcing / CDC pipelines

Replaying Old Messages

With AMQP/JMS brokers, processing and acknowledging is destructive -- messages are deleted. With log-based brokers, consuming is read-only -- the log is not modified. The consumer offset is under the consumer's control: you can reset it to yesterday's position, reprocess data, and write to a different output. This makes log-based messaging more like batch processing, where derived data is clearly separated from input data through a repeatable transformation.

Test Your Understanding

1Why does polling a database for new events become problematic at low latencies, and how do messaging systems solve this?
▼

When you poll a database frequently (e.g., every 100ms), most poll requests return no new events. The ratio of useful responses to total requests becomes very low, wasting network, CPU, and database resources. As polling frequency increases, overhead grows proportionally while useful work per poll decreases.

Messaging systems solve this with push-based notification: instead of consumers repeatedly asking "anything new?", the system proactively pushes events to consumers when they arrive. This eliminates wasted polls and reduces latency since consumers learn about events immediately rather than at the next poll interval. Databases traditionally lacked good notification mechanisms (triggers are limited), which is why specialized messaging tools were developed.

2Explain the message reordering problem with traditional brokers. Why does it not occur with log-based brokers?
▼

In traditional brokers using load balancing, messages m3 and m4 might be dispatched to consumers C2 and C1 respectively. If C2 crashes before acknowledging m3, the broker redelivers m3 to C1 -- which has already processed m4. So C1 sees m4 before m3, violating the original order.

Log-based brokers avoid this because they assign entire partitions (not individual messages) to consumers. Within a partition, one consumer reads all messages sequentially. There is no scenario where two consumers handle adjacent messages from the same partition. If a consumer fails, the replacement picks up from the last committed offset and processes messages in the same order. The tradeoff: you cannot have more consumers in a group than partitions.

3A slow consumer in a traditional broker can affect other consumers, but not in Kafka. Why?
▼

In traditional brokers, a slow consumer causes the queue to grow. Since brokers typically hold messages in memory, a growing queue consumes memory that all consumers share. When the broker runs out of memory and starts spilling to disk, throughput degrades for everyone, not just the slow consumer. Also, undeleted queues from crashed consumers continue accumulating messages.

In Kafka, messages are always written to disk regardless (append-only log). A slow consumer simply has a larger gap between its current offset and the log head. This does not consume additional memory or affect other consumers' read performance. Each consumer reads independently from its own position in the log. The worst case is the consumer falling so far behind that its offset points to a deleted segment -- but even then, only that consumer is impacted.

4How does log compaction allow a new consumer to get a full database snapshot without a separate snapshot process?
▼

Log compaction retains the most recent value for every key that has ever been written. Old updates for the same key are garbage collected, and tombstones mark deletions. The compacted log therefore contains exactly one entry per live key -- essentially a snapshot of the current database state.

A new consumer starts reading from offset 0 of the compacted topic and sequentially scans all entries. By the time it reaches the head of the log, it has seen the latest value for every key. This is equivalent to a full database copy. The disk space required depends only on the current number of distinct keys, not the total number of writes ever made. This eliminates the need for a separate snapshot process and simplifies bootstrapping new derived data systems (search indexes, caches, etc.).

5Why is log-based messaging described as "more like batch processing" than traditional messaging?
▼

Three properties make it batch-like:

1. Input is not destroyed by reading. In batch processing, input files are read-only. In log-based messaging, reading a message does not delete it from the log. In traditional messaging, acknowledgment deletes the message.

2. Repeatable processing. You can reset a consumer's offset to reprocess the same data with different processing logic, just like re-running a batch job on the same input file. Traditional brokers cannot do this since messages are gone after ack.

3. Derived data separation. The log (input) is clearly separated from the consumer's output. This allows experimentation, A/B testing of processing logic, and easier recovery from bugs -- the same advantages that made batch processing robust in Chapter 10.

6You have a topic with 3 partitions and a consumer group with 5 consumers. What happens, and how would you fix it?
▼

Only 3 of the 5 consumers will be assigned partitions. The remaining 2 consumers sit idle, consuming no messages. In log-based systems, the maximum useful parallelism within a consumer group equals the number of partitions. Each partition is assigned to exactly one consumer in the group.

Fixes:

  • Increase partitions: Create 5 or more partitions so all consumers get work. But note: increasing partitions is a one-way operation in Kafka and affects key-based ordering guarantees.
  • Reduce consumers: Use 3 consumers instead of 5. If processing is the bottleneck, optimize per-message processing time.
  • Rethink the architecture: If you need message-level parallelism beyond the partition count, consider a traditional broker for that specific workload, or have each consumer use internal thread pools for parallel processing within its assigned partitions.
