Chapter 11 — Stream Processing

Part 1: Events, Message Brokers, Delivery Semantics

Batch processing works on bounded datasets (a file, a day's worth of logs). But most real-world data is unbounded — it keeps arriving continuously. Stream processing is the paradigm for processing this continuous flow of events as they happen.

Events and Streams

A stream is an unbounded sequence of events over time. An event is a small, self-contained fact about something that happened: a user clicked a button, a sensor recorded a temperature, a payment was made.

Event properties
  • Immutable: happened in the past, can't be changed
  • Small: typically bytes to kilobytes
  • Time-stamped: when the event occurred
  • Self-contained: carries all relevant data
Event examples
{type:"click", user:123, url:"/home", ts:1712345678}
{type:"payment", amount:99.99, currency:"USD"}
{type:"login", user_id:"alice", ip:"1.2.3.4"}
{type:"sensor_reading", sensor_id:"T-42", temp:22.3}

Message Brokers

A message broker (or message queue) is a system that connects producers (who generate events) with consumers (who process events). It decouples producers from consumers: they don't need to know about each other or be running simultaneously.

Producers and consumers are decoupled. Multiple producers can write to the same topic. Multiple consumers can independently read.

Traditional Message Brokers (AMQP/JMS style)

Systems like RabbitMQ, ActiveMQ, and IBM MQ implement the AMQP or JMS protocol. They work as queues: messages are pushed to consumers, and once acknowledged, deleted from the queue.

Ordering Per-queue ordering, not per-topic across partitions Strict ordering per partition
Use case Task queues, RPC-style, short-lived jobs Event streams, audit log, data pipelines

Delivery Semantics

A critical design question: what happens if a consumer crashes after processing a message but before acknowledging it?

At-most-once
Send and forget. Consumer processes, doesn't acknowledge. If consumer crashes: message is lost. No duplicates, but possible data loss.
At-least-once
Retry until acknowledged. If consumer crashes after processing but before ACK: redelivered. Possible duplicates. Must be idempotent.
Exactly-once
Hardest. Requires distributed transaction or idempotent writes + deduplication. Kafka Streams and Flink support this.
In practice, "exactly-once" is achieved via: (1) idempotent producers (messages deduplicated by ID at the broker); (2) transactional writes (producer atomically writes to multiple partitions); (3) consumers tracking their progress and writing it atomically with their output. All three are needed for true end-to-end exactly-once.

Producer-Consumer Decoupling

The key benefit of message brokers: producers and consumers are decoupled in time, space, and behavior.

Time decoupling
Producer can generate events even when the consumer is down. Consumer processes when it comes back up. No need to be online simultaneously.
Rate decoupling
Producer can write faster than the consumer can process. Broker absorbs the spike. Consumer drains the backlog at its own pace.
Fan-out
One event is consumed by many consumers independently: analytics, fraud detection, audit log, notifications — all from the same stream.

Consumer Groups and Load Balancing

A single consumer may not be fast enough to keep up with the stream. Consumer groups allow horizontal scaling of consumers.

Traditional MQ: Competing consumers
Each message delivered to exactly one consumer in the group. Load balancing: work distributed across consumers. If consumer fails: unacknowledged messages redelivered to another consumer.
Trade-off: no ordering guarantee across consumers. Two messages may be processed in different order by different consumers.
In Kafka, consumer groups and partitions work differently: each partition is consumed by exactly one consumer in the group. This guarantees ordering within a partition, while distributing load across the group. The number of partitions is the maximum parallelism for a consumer group.

Message Broker Failure Modes

What happens when the broker itself fails?

  • In-memory queues: Messages lost if broker restarts. Acceptable only for transient data.
  • Durable queues: Messages written to disk. Survive broker restart. Slower (disk writes on every message).
  • Replicated brokers: Messages written to multiple brokers. Survive single broker failure. Kafka replicates each partition across N brokers (configurable).
Durability is a trade-off: writing every message to disk synchronously before acknowledging adds latency. Async writes (acknowledge immediately, flush to disk in background) improve throughput but risk losing messages if the broker crashes before flushing. Kafka's default: write to the OS page cache (fast), flush in background — a pragmatic balance.
