1. Schema Evolution: Thrift & Protocol Buffers

Schemas inevitably change. The question is: how can you change a schema while keeping encoded data readable by both old and new code? For Thrift and Protocol Buffers, the answer revolves around field tag numbers — these are the stable identity of a field across all schema versions.

Diagram 1 — Adding and Removing Fields (Tag Rules)

✓ Adding a New Field

Schema v1:

1: string userName
2: i64 favoriteNumber
▼ add field

Schema v2:

1: string userName
2: i64 favoriteNumber
3: string email (optional, NEW)
Forward compat: old code sees tag 3, doesn't know it → skips it. OK.
Backward compat: new code reads old data, tag 3 absent → uses default. OK only if field is optional.

✗ Removing a Field (dangers)

Schema v1:

1: string userName
2: i64 favoriteNumber
3: string email (optional)
▼ remove field

Schema v2:

1: string userName
2: i64 favoriteNumber
Tag 3: RETIRED — never reuse!
Can only remove optional fields (never required).
Must never reuse the tag number — old data still has tag 3; reuse would misinterpret it.

The Cardinal Rules:

Change field names freely (not in encoded data) NEVER change a field's tag number NEVER reuse a removed tag number New fields must be optional or have defaults Type changes: possible but risky (may truncate)

Why Tag Numbers Are Sacred

The tag number is the identity of the field in encoded bytes. Changing a tag turns every existing encoded record unreadable — the parser maps tag → wrong field. Reusing a removed tag is equally dangerous: old data in databases, message queues, and log files still contains records with that tag number. The parser silently misinterprets the old field as the new one. These rules are absolute because encoded data can persist for years.

Datatypes: changing a 32-bit int to a 64-bit int is possible — new code reads old values (fills missing bits with zeros). But old code reading new 64-bit values will truncate silently if the value exceeds 32 bits. Changing types is a compatibility hazard.

2. Avro: A Fundamentally Different Approach

Apache Avro was started in 2009 as a Hadoop subproject. Hadoop needed to dump relational database tables to binary files, and Thrift wasn't designed for this. Avro's key difference: there are no tag numbers in the schema or the encoding.

Avro IDL (Human-Editable)

record Person {
  string              userName;
  union {null, long}  favoriteNumber = null;
  array<string>       interests;
}

Avro JSON Schema

{
  "type": "record",
  "name": "Person",
  "fields": [
    {"name":"userName","type":"string"},
    {"name":"favoriteNumber",
     "type":["null","long"],
     "default":null},
    {"name":"interests",
     "type":{"type":"array",
             "items":"string"}}
  ]
}

The Avro binary encoding (32 bytes) is just values concatenated: length-prefixed strings, varint integers — no field tags, no type markers. The decoder reads fields in schema order and uses the schema to determine each field's datatype. This means: the binary can only be decoded correctly with the same (or a compatible) schema used to write it.

No 0B type bytes. No 00 01 tag bytes. Just values. The schema (stored separately) tells the decoder: "first field is a string, second is union{null,long}, third is array of strings."

Writer's Schema vs Reader's Schema

The key Avro insight: the writer's schema and reader's schema don't have to be identical — they just need to be compatible. When reading, the Avro library resolves differences by matching fields by name (not by tag number or field position).

Diagram 3 — Avro Schema Resolution: Writer v2 vs Reader v1

Writer's Schema (v2)

userName: string
favoriteNumber: union{null,long}
interests: array<string>
email: string (NEW field)
↔ ↔ ↔ ✗

Reader's Schema (v1)

userName: string
favoriteNumber: union{null,long}
interests: array<string>
(no email field)
MatchField in both schemas with same name → resolve, may type-convert
IgnoreField in writer but not reader (e.g., "email") → reader skips it
DefaultField in reader but not writer → filled with default from reader's schema
Diagram 4 — Avro Schema Evolution Rules

✓ Allowed

  • Add a field with a default value
  • Remove a field that has a default value
  • Change datatype (if Avro can convert it)
  • Change field name using an alias (backward compat only)
  • Reorder fields freely (matched by name, not position)

✗ Not Allowed

  • Add field without default → breaks backward compat
  • Remove field without default → breaks forward compat
  • Change field name without alias → breaks backward compat
  • Add union branch → backward compat only (not forward)

Avro's Null Handling (more explicit than Protobuf)

Avro has no optional/required. To allow null, declare a union type: union {null, long}. You can only use null as a default if null is the first union branch. This is more verbose, but forces you to be explicit about nullability — a source of many bugs in Protobuf/Thrift schemas.

A schema registry also functions as living documentation — the schema required for decoding is always up to date, unlike hand-written documentation that diverges from reality.

Avro's Killer Feature: Dynamically Generated Schemas

Why does it matter that Avro has no tag numbers? Consider dumping a relational database to Avro binary. With Avro, you auto-generate an Avro schema from the database schema: each column becomes a field, column names map directly to field names. When the database schema changes (column added or removed), just regenerate the Avro schema automatically.

With Thrift or Protocol Buffers, field tags must be assigned by hand. Every time the database schema changes, someone manually updates the tag mapping. One mistake and you silently corrupt data. Avro was specifically designed for this Hadoop use case; Thrift was not.

Thrift vs Protobuf vs Avro: When to Use Which

  • Thrift: older, two wire formats, good for polyglot microservices at Facebook-style scale. More expressive type system (nested lists).
  • Protocol Buffers (gRPC): dominant choice for new internal microservices. Strong tooling, gRPC HTTP/2 transport, proto3 removes required. Best for stable schemas.
  • Avro: best for dynamic schema generation (database → file pipelines, Kafka Schema Registry, Hadoop). Schema evolution by name matching is more flexible for generated schemas.

Why Schema-Driven Formats Win for Internal Use

  • Compact: omit field names from every encoded record (32 bytes vs 81 bytes)
  • Documentation that can't drift: the schema is required to decode data — it's always accurate, unlike README files
  • Deployment guardrails: a schema registry lets you check compatibility before deploying — catch breaking changes at CI time
  • Code generation: in statically typed languages (Java, Go, C++), generated classes give compile-time type checking and IDE autocomplete
  • Same flexibility as schemaless: Avro/Protobuf evolution gives you JSON's flexibility with better guarantees

Historical note: these ideas aren't new. ASN.1 (1984) uses tag numbers like Protobuf. Its binary encoding (DER) is still used today in SSL certificates (X.509).
