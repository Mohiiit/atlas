Implementation of Replication Logs

How does a leader actually communicate changes to followers? Several approaches:

MethodWhat's sentProsCons
Statement-based SQL statements: UPDATE users SET age=30 WHERE id=1 Compact, human-readable Non-deterministic functions (NOW(), RAND()) cause divergence. MySQL used this before 5.1.
WAL shipping Raw write-ahead log bytes (storage engine level) Exact bytes replicated, no divergence Couples replication to storage format. Can't replicate between different PostgreSQL versions.
Row-based (logical) Logical log: which rows changed, to what values Decoupled from storage engine. Works across versions. Easy to parse externally. Larger volume than statements for bulk writes
Trigger-based Application code fires on data changes Flexible — can replicate subset of tables High overhead, more error-prone. Escape hatch when you need custom logic.
PostgreSQL uses WAL shipping internally. Its logical replication (since PostgreSQL 10) uses row-based logical decoding. MySQL uses binlog in row-based format by default. Kafka Connect's CDC uses logical replication.

Problems with Replication Lag

In async replication, followers may be seconds or minutes behind the leader. If a user reads from a lagging follower, they see stale data. This is eventual consistency — followers will catch up eventually, but there's a window where they diverge.

"Eventual consistency" is deliberately vague. "How eventual?" matters enormously in practice. Typically milliseconds, but under load it can be minutes.

Three Important Consistency Guarantees

1. Read-your-own-writes (read-after-write consistency)

Problem: user edits their profile, then immediately reads it from a lagging follower and sees the old version
Without guarantee
User posts comment → reads from follower → comment not there → thinks their write was lost
Implementation
1. Read user's own data from leader (your profile → leader; others' → followers)
2. Track last-write timestamp; follower must be at least that current
3. Route all reads to leader for 1 min after any write

2. Monotonic reads

Problem: user makes two reads from two different followers with different lag — sees data "go back in time"
Without guarantee
Read from F1: sees comment X
Read from F2 (more lagged): comment X doesn't exist yet!
Time apparently went backwards.
Implementation
Route each user to the same follower consistently (e.g., hash(user_id) mod n_followers). User never reads from a more-lagged follower than their previous read.

3. Consistent prefix reads

Problem: observer reads replies before questions in a distributed database where writes happen out of order
Without guarantee
Alice: "How's the weather?"
Bob: "It's raining!"
Observer reads Bob's answer first (from fast follower), Alice's question later (from slow follower). Causality violated.
Implementation
Writes causally related to the same partition. Or vector clocks to track causality. Harder in sharded databases.
These three guarantees are part of a broader framework. Stronger guarantees (linearizability, sequential consistency) are covered in Chapter 9. Transactions provide many of these automatically — the complexity here arises only in non-transactional or eventually-consistent systems.

Multi-Leader Replication

Rather than one leader accepting all writes, allow multiple nodes to accept writes. Each node acts as a leader to the others' followers. Also called active-active or master-master replication.

Use Cases

Multi-datacenter
One leader per datacenter. Local writes don't cross WAN. Datacenters replicate to each other asynchronously. Tolerates datacenter outages.
Offline clients (mobile)
Each device is a "leader" of its local data. When online, syncs to server. CouchDB was designed for this. Google Docs / Notion use variants.
Collaborative editing
Multiple users edit a doc simultaneously. Each user's browser is a local leader. Changes are sent to other users. Requires conflict resolution.
Geo-distributed writes
Users in different regions write to local datacenters without WAN latency. Changes propagate asynchronously.

Handling Write Conflicts

The biggest complication of multi-leader replication: write conflicts. If two leaders accept conflicting writes to the same record, the database must decide what the final value is.

Write conflict: two users edit the same wiki page title simultaneously in different datacenters
Leader 1 (NYC)
User A sets title = "LeaderDB"
Leader 2 (LON)
User B sets title = "FollowerDB"
↘↙
CONFLICT: Both writes succeed locally. What is the "correct" title?

Conflict Resolution Strategies

StrategyHowProblem
Avoid conflicts Route all writes for a user/record to the same datacenter Loses multi-leader benefits. Fails if preferred datacenter is down.
Last write wins (LWW) Higher timestamp wins. Concurrent writes: use wall clock. Clocks can't be perfectly synchronized. One write is silently dropped.
Merge values Concatenate the conflicting values: "LeaderDB/FollowerDB" Makes sense for sets, not for general fields. Awkward.
Record + resolve later Save all conflicting versions; surface to application or user to resolve Complex for applications to implement. CouchDB does this.
CRDT Data structures that automatically merge: counters, sets, text (like Google Docs OT) Only works for supported data types. Complex to implement.
There is no universally correct approach. The right strategy depends on the application's semantics. "Last write wins" is convenient but silently loses data. Many databases (DynamoDB) default to LWW; developers are often unaware of the implications.

Multi-Leader Replication Topologies

How do writes propagate between multiple leaders?

Circular
A → B → C → A
Each node forwards writes to next.
If one node fails, chain breaks.
Star
Central node forwards all writes.
B, C, D all connect to center A.
Center is single point of failure.
All-to-all
Every leader replicates to every other.
Most fault-tolerant.
Ordering problems: writes may arrive out of order. Need vector clocks.
Multi-leader replication is often a retrofitted feature and considered "dangerous territory" by DDIA. The conflict handling is subtle and easy to get wrong. Unless you have a compelling need (multi-datacenter, offline clients), single-leader is simpler.
