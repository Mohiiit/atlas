Replication Lag & Multi-Leader

Eventual consistency pitfalls, read guarantees, multi-leader architectures, write conflict handling, and replication topologies

Problems with Replication Lag

Leader-based replication with asynchronous followers enables read-scaling: distribute reads across many followers while all writes go through the leader. But asynchronous replication introduces a fundamental problem — replication lag.

The Eventual Consistency Bargain

If you read from an asynchronous follower, you may see outdated information. The leader and follower may return different results for the same query at the same time. This inconsistency is temporary — followers will eventually catch up. Hence: eventual consistency. But "eventually" is deliberately vague — lag can range from fractions of a second to several minutes.

Three specific anomalies arise from replication lag, each with its own consistency guarantee to prevent it.

1. Reading Your Own Writes

A user submits data (goes to the leader) then immediately views it (routed to a follower). If the follower has not yet received the write, it looks like their data was lost. To prevent this, we need read-after-write consistency (also called read-your-writes consistency): a guarantee that users always see their own submitted data.

Diagram 2 — Read-After-Write Violation Timeline

t=0.5s — Follower returns OLD data
User sees old profile picture. Appears data was lost!
t=3s — Follower catches up
Replication completes. But user already left in frustration.

Solutions for Read-After-Write Consistency

  • Read from leader for user-modified data: e.g., always read a user's own profile from the leader, other profiles from followers
  • Time-based leader reads: For 1 minute after the last update, route all reads to the leader
  • Client-tracked timestamps: Client remembers timestamp of most recent write; system ensures the serving replica reflects updates at least up to that timestamp (using logical timestamps like log sequence numbers)
  • Cross-device consideration: When accessing from multiple devices, metadata must be centralized and requests from all devices may need to route to the same datacenter

2. Monotonic Reads

A user reads from one follower (fresh data), then reads from a different follower (stale data). They see data go backward in time — e.g., a comment appears and then disappears. Monotonic reads guarantees a user will never read older data after having previously read newer data.

Solution: Sticky Sessions

Ensure each user always reads from the same replica. Choose the replica based on a hash of the user ID (not randomly). Different users can read from different replicas. If that replica fails, reroute to another — but during normal operation, the user sees a consistent timeline. This is stronger than eventual consistency but weaker than strong consistency.

3. Consistent Prefix Reads

This anomaly violates causality. If writes happen in a certain order, readers should see them in the same order. In partitioned databases, different partitions may replicate at different speeds, causing an observer to see effects before their causes.

Consistent Prefix Reads Guarantee

If a sequence of writes happens in a certain order, anyone reading those writes will see them in the same order. Solutions:

  • Write causally related data to the same partition — ensures ordering within a single replication stream
  • Use algorithms that track causal dependencies — e.g., version vectors, happens-before tracking
  • This is primarily a problem in partitioned databases where different partitions operate independently with no global write ordering

The Case for Transactions

Dealing with replication lag in application code is complex and error-prone. Transactions exist so that databases can provide stronger guarantees and applications can be simpler. The claim that "transactions are too expensive for distributed systems" is overly simplistic. Chapters 7 and 9 will revisit this topic with more nuance.

Multi-Leader Replication

Single-leader replication has a major downside: all writes must go through one node. If you cannot connect to the leader (network interruption, datacenter failure), you cannot write. Multi-leader replication (also called master-master or active/active) allows more than one node to accept writes. Each leader forwards changes to all other leaders, and simultaneously acts as a follower to them.

When Multi-Leader Makes Sense

It rarely makes sense within a single datacenter (complexity outweighs benefits). The primary use cases are: (1) Multi-datacenter operation, (2) Clients with offline operation, and (3) Collaborative editing. The biggest trade-off: you must handle write conflicts.

Multi-Datacenter Operation

With replicas in several datacenters, single-leader forces all writes through one datacenter. Multi-leader places a leader in each datacenter. Within each datacenter, regular leader-follower replication is used. Between datacenters, leaders replicate asynchronously to each other.

Collaborative Editing

Real-time collaborative editing (Etherpad, Google Docs) is conceptually similar to multi-leader replication. Each user's browser/client has a local replica. Changes are instantly applied locally, then asynchronously replicated to the server and other users.

Lock-Based (Single-Leader)

  • Obtain lock on document before editing
  • Other users must wait
  • Equivalent to single-leader with transactions
  • No conflicts, but slow collaboration

Fine-Grained (Multi-Leader)

  • Unit of change = single keystroke
  • No locking — multiple users edit simultaneously
  • Requires conflict resolution
  • Fast collaboration, but complex

Handling Write Conflicts

The biggest problem with multi-leader replication: two users concurrently modify the same record on different leaders. Each write succeeds locally, but a conflict is detected when changes replicate asynchronously.

In single-leader, the second write would block or abort. In multi-leader, both succeed — conflict surfaces later during replication.

Conflict Detection

Synchronous vs Asynchronous Detection

Synchronous detection: Wait for all replicas to confirm before telling user the write succeeded. But this defeats the purpose of multi-leader — you lose independent writes. Might as well use single-leader.

Asynchronous detection: Conflict discovered later during replication. May be too late to ask the user. This is the reality of multi-leader systems.

Conflict Avoidance

The Simplest Strategy: Don't Have Conflicts

Route all writes for a particular record through the same leader. For example, assign each user a "home" datacenter based on geography. From that user's perspective, it behaves like single-leader.

Breaks down when: A datacenter fails and traffic must be rerouted, or a user moves to a different geographic region. Then you must handle concurrent writes on different leaders anyway.

Converging Toward a Consistent State

In multi-leader, there is no defined write ordering. If Leader 1 sees writes in order B then C, and Leader 2 sees C then B, they must still converge to the same final value. The database must resolve conflicts in a convergent way.

Conflict Resolution Strategies

Strategy How It Works Data Loss? Use Case
Last Write Wins (LWW) Each write gets a timestamp/UUID. Highest ID wins, others discarded. Yes — losing writes silently dropped Acceptable when overwrites are fine (e.g., cache updates)
Value Merging Merge conflicting values (e.g., concatenate: "B/C") No — but result may be unexpected Simple fields where concatenation is meaningful
CRDTs Conflict-free replicated data types: sets, maps, counters that auto-resolve. Two-way merge. No — mathematically guaranteed convergence Counters, sets, collaborative data (Riak 2.0)
Operational Transform (OT) Transform operations against concurrent operations to preserve intent. Designed for ordered lists (text). No — preserves all user intentions Collaborative text editing (Google Docs, Etherpad)
Mergeable Persistent DS Track history explicitly (like Git). Three-way merge function. No — full history preserved Complex data structures needing full audit trail
Explicit Conflict Storage Store all conflicting versions. Resolve on read (prompt user or auto-resolve). No — all versions preserved CouchDB-style apps, shopping carts

Custom Conflict Resolution Logic

On Write

Database calls a conflict handler as soon as it detects a conflict in the replication log. Runs in background, must execute quickly, cannot prompt user. Example: Bucardo (Perl snippets).

On Read

All conflicting versions are stored. Next read returns multiple versions. Application prompts user or auto-resolves, then writes result back. Example: CouchDB.

Amazon shopping cart: conflict resolution preserved items added to cart but not items removed. Customers saw items reappearing after deletion — a cautionary tale about custom conflict resolution handlers.

Beware: Poorly Implemented Conflict Detection

Many multi-leader systems have weak conflict detection. PostgreSQL BDR does not provide causal ordering. Tungsten Replicator for MySQL does not even try to detect conflicts. If using multi-leader replication, read the documentation carefully and test thoroughly.

First-Principles Questions

1 Why can't you simply use timestamps to resolve all write conflicts in a multi-leader system?
▼

Timestamps rely on synchronized clocks, which are fundamentally unreliable in distributed systems. Different machines have different clock skew (Chapter 8 covers "Unreliable Clocks" in detail). A write with a "later" timestamp on one machine may actually have happened earlier in real time. Using timestamps (Last Write Wins) is simple but dangerously prone to data loss — it silently drops the "losing" write. For causal ordering, you need logical clocks like version vectors that track actual happens-before relationships rather than depending on physical time.

2 How is an offline-capable mobile app architecturally identical to multi-datacenter replication?
▼

Each device acts as its own "datacenter" with a local leader that independently accepts reads and writes. The network between devices (cellular, WiFi, or none) is like the internet link between datacenters — except far more unreliable. When connectivity returns, the sync process is exactly multi-leader replication: each local leader sends its changes to a central server (or peers), which must merge them, detect conflicts, and converge to a consistent state. The replication lag isn't milliseconds but hours or days. This framing explains why calendar sync has such a "rich history" of bugs — it's one of the hardest multi-leader problems.

3 Why do circular and star topologies have worse fault tolerance than all-to-all, yet all-to-all has its own ordering problem?
▼

Circular/star fault tolerance: Every message must pass through intermediate nodes. If any node on the path fails, downstream nodes stop receiving updates. The topology must be manually reconfigured around the failure. It's a chain — one broken link severs communication.

All-to-all ordering problem: Every leader sends directly to every other leader, so no single point of failure. But different network links have different speeds. A causally-later write (UPDATE) can arrive at a node before the causally-earlier write (INSERT) it depends on. The node receives an update to a row that doesn't exist yet. This is the same causality problem as consistent prefix reads, but at the inter-leader level. Version vectors solve this by encoding causal dependencies, allowing a node to wait for prerequisite writes.

4 Why does Kleppmann argue that transactions are a better solution for replication lag than application-level workarounds?
▼

Application-level solutions (read-from-leader for recent writes, timestamp tracking, sticky sessions) are complex, ad-hoc, and easy to get wrong. Each consistency guarantee must be hand-implemented per feature. You're essentially reimplementing database guarantees in application code — a leaky abstraction that creates subtle bugs.

Transactions push this complexity into the database layer, providing declarative guarantees the application can simply rely on. The claim that "transactions are too expensive for distributed systems" is overly simplistic. While there are real trade-offs (performance, availability), Chapters 7 and 9 develop a more nuanced view, showing that stronger guarantees are possible without sacrificing everything. The alternative — "pretending that replication is synchronous when it is asynchronous" — is a recipe for problems.
