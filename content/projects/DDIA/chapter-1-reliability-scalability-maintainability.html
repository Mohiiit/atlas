Thinking About Data Systems

A data-intensive application is typically built from standard building blocks that provide commonly needed functionality: databases (store data), caches (remember expensive results), search indexes (search by keyword), stream processing (send messages asynchronously), and batch processing (crunch accumulated data periodically).

We typically think of databases, queues, and caches as very different categories. But the boundaries are blurring: Redis is a datastore used as a message queue; Apache Kafka is a message queue with database-like durability guarantees. A single tool can no longer satisfy all data processing and storage needs. Work is broken down into tasks that can be performed efficiently on a single tool, and those tools are stitched together using application code.

Why "Data System" as an Umbrella Term?

  • New tools don't fit neatly into traditional categories (e.g., Redis is both a cache and a queue)
  • Many apps combine several tools stitched together by app code
  • When you combine tools, your application code becomes a data system — hiding internal complexity behind an API
  • You are now both an application developer and a data system designer

When you create such a composite data system, three questions arise about the service you're providing:

Reliability

The system works correctly even when things go wrong (hardware faults, software bugs, human errors).

Scalability

As load grows, there are reasonable ways to cope — in data volume, traffic, or complexity.

Maintainability

Over time, many people work on the system. They should all be able to work on it productively.

Reliability

"Continuing to work correctly, even when things go wrong." Typical expectations for software: it performs the function the user expected, it tolerates the user making mistakes or using it in unexpected ways, its performance is good enough for the required use case, and it prevents unauthorized access and abuse.

The Goal of Fault-Tolerance
Design mechanisms that prevent faults from causing failures. We cannot reduce faults to zero, so we build reliable systems from unreliable parts.
"Counterintuitively, in fault-tolerant systems, it can make sense to increase the rate of faults by triggering them deliberately — for example, by randomly killing individual processes without warning." This is the Netflix Chaos Monkey approach. Many critical bugs are due to poor error handling; by deliberately inducing faults, you ensure the fault-tolerance machinery is continually exercised and tested.

1. Hardware Faults

Hard disks have a mean time to failure (MTTF) of about 10 to 50 years. On a storage cluster with 10,000 disks, you should expect on average one disk to die per day. Our first response is to add redundancy to individual hardware components: RAID for disks, dual power supplies, hot-swappable CPUs, batteries and diesel generators for backup power. When one component fails, the redundant component takes over while the broken one is replaced.

As data volumes and computing demands have increased, more applications use larger numbers of machines, proportionally increasing the rate of hardware faults. There is a move toward systems that tolerate the loss of entire machines by using software fault-tolerance techniques in addition to or instead of hardware redundancy. A system that can tolerate machine failure can be patched one node at a time with a rolling upgrade, without downtime of the entire system.

2. Software Errors

Hardware faults are usually random and independent — one machine's disk failing doesn't imply another machine's disk is about to fail. Software errors are a different class: they are systematic and tend to cause far more system failures because they are correlated across nodes. They lie dormant for a long time until they are triggered by an unusual set of circumstances, when an assumption in the software stops being true.

Examples of Systematic Software Faults

  • A software bug that causes every instance of an application server to crash when given a particular bad input (e.g., the Linux kernel leap-second bug on June 30, 2012)
  • A runaway process that uses up shared resource — CPU time, memory, disk space, network bandwidth
  • A service that the system depends on slows down, becomes unresponsive, or returns corrupted responses
  • Cascading failures, where a small fault in one component triggers a fault in another, which in turn triggers further faults

Software Fault Prevention Strategies

  • Carefully think about assumptions and interactions: The bug often lies in an assumption that is usually true but stops being true in certain edge cases
  • Thorough testing: Including integration tests and stress tests, not just unit tests
  • Process isolation: One process crashing shouldn't take down unrelated services
  • Crash-and-restart: Design processes to recover gracefully from crashes
  • Measuring, monitoring, and alerting: Instrument the system to spot anomalies early
  • Self-checking: e.g., verifying that the number of messages sent equals the number received

3. Human Errors

Humans are known to be unreliable. A study of large internet services found that configuration errors by operators were the leading cause of outages, whereas hardware faults (servers or network) played a role in only 10-25% of outages. How do we make systems reliable in spite of unreliable humans?

How Important Is Reliability?

Bugs in business applications cause lost productivity (and legal risk if figures are reported incorrectly). Outages of ecommerce sites can have huge costs in revenue and reputation. Even in "noncritical" applications, we have a responsibility to our users. There are situations where we may choose to sacrifice reliability to reduce development cost (e.g., prototyping) — but we should be very conscious when cutting corners.

Scalability

Scalability is not a one-dimensional label — it is not meaningful to say "X is scalable" or "Y doesn't scale." Rather, scalability means: "If the system grows in a particular way, what are our options for coping with the growth?" and "How can we add computing resources to handle the additional load?"

Describing Load

Before discussing coping strategies, we must succinctly describe the current load on the system. Load can be described with a few numbers called load parameters. The best choice of parameters depends on the architecture: requests per second to a web server, the ratio of reads to writes in a database, the number of simultaneously active users in a chat room, the hit rate on a cache, or something else.

Describing Performance

Once you have described the load, you can investigate what happens when the load increases. Two ways to look at it:

Option A: Fixed Resources

When you increase a load parameter and keep system resources unchanged, how is the performance affected?

Option B: Fixed Performance

When you increase a load parameter, how much do you need to increase resources to keep performance unchanged?

For batch processing systems (like Hadoop), we usually care about throughput — the number of records we can process per second, or the total time to run a job on a dataset of a certain size. For online systems, we care about response time — the time between a client sending a request and receiving a response.

Latency vs. Response Time — They Are NOT Synonyms

Response time = what the client sees: service time + network delays + queueing delays.
Latency = the duration a request is latent, awaiting service — it is waiting to be handled. Latency is one component of response time.

Even if you make the same request over and over, you'll get a slightly different response time each time. Response time is not a single number but a distribution of values. Most requests are reasonably fast, but there are occasional outliers that are much slower. Perhaps the slow requests are intrinsically more expensive (e.g., they process more data). But often the slowness is due to additional latency introduced by context switches, loss of a CPU cache, network packet loss, TCP retransmission, garbage collection pause, page fault forcing a disk read, mechanical vibrations in the server rack, etc.

Percentiles: The Right Way to Measure Performance

Don't use the mean (arithmetic average) — it doesn't tell you how many users actually experienced a particular delay. Use percentiles instead. Sort your response times from fastest to slowest: the median (p50) is the halfway point. If p50 = 200ms, that means half of requests are served in less than 200ms, and half take longer. The median is a good metric for what the "typical" user experiences.

To learn how bad outliers are, look at higher percentiles: p95, p99, p999 (the 95th, 99th, 99.9th percentiles). If p95 = 1.5s, that means 95 out of 100 requests take less than 1.5 seconds. These high percentiles are known as tail latencies.

Why Tail Latency Matters (Amazon's Reasoning)

Amazon describes response time requirements for internal services in terms of the 99.9th percentile, even though it only affects 1 in 1,000 requests. This is because the customers with the slowest requests are often those who have the most data on their accounts — because they have made many purchases. They are the most valuable customers. Optimizing for the 99.9th percentile was found to have enough benefit to the bottom line. On the other hand, optimizing the 99.99th percentile was deemed too expensive (the slowest requests are often caused by events beyond your control, like random I/O stalls).

Amazon internal observation: every 100ms increase in response time reduces sales by 1%.

Percentiles are often used in service level objectives (SLOs) and service level agreements (SLAs). For example, an SLA may state that the service is considered "up" if p50 < 200ms and p99 < 1s, and the service must be "up" at least 99.9% of the time. These metrics set expectations for clients and allow customers to demand a refund if the SLA is broken.

Measuring Percentiles in Practice

If you want to add response time percentiles to the monitoring dashboards for your services, you need to efficiently calculate them on an ongoing basis. For example, you may want to keep a rolling window of response times of requests in the last 10 minutes. Naive sorting is too expensive. Algorithms like forward decay, t-digest, or HdrHistogram are good approximate algorithms that compute percentiles with minimal CPU and memory cost. Beware: averaging percentiles is mathematically meaningless — percentiles of the aggregated data must be computed from the raw data.

Elastic vs. Manual Scaling

Elastic systems automatically add computing resources when a load increase is detected. This is useful when load is highly unpredictable. Manual scaling means a human analyzes capacity and decides to add more machines. This is simpler and has fewer operational surprises.

In practice, good architectures involve a pragmatic mixture: keep stateless services on multiple machines, but keep your database on a single node until scaling cost or high-availability needs forced you to distribute. Distributing stateful data systems introduces significant additional complexity. The common wisdom until recently was to keep the database on a single node — but as tools and abstractions improve, this is changing.

"There Is No Such Thing as Magic Scaling Sauce"

An architecture that is appropriate for one level of load is unlikely to cope with 10 times that load. A system designed for 100,000 requests/sec at 1 kB each looks very different from one designed for 3 requests/min at 2 GB each — even though the data throughput is the same in both cases.

Scalable architectures are specific to the application — there is no generic, one-size-fits-all architecture. The architecture is built around assumptions about which operations will be common and which will be rare — the load parameters. If those assumptions are wrong, the engineering effort is at best wasted and at worst counterproductive.

Maintainability

It is well known that the majority of the cost of software is not in its initial development, but in its ongoing maintenance — fixing bugs, keeping its systems operational, investigating failures, adapting it to new platforms, modifying it for new use cases, repaying technical debt, and adding new features. Yet unfortunately, many people working on software systems dislike maintenance of so-called "legacy" systems.

We should design software in such a way that it will hopefully minimize pain during maintenance, and thus avoid creating legacy software ourselves. We pay attention to three design principles:

"Good operations can often work around the limitations of bad (or incomplete) software, but good software cannot run reliably with bad operations." While some aspects of operations can be automated, it is still up to humans to set up that automation and make sure it is working correctly.

Simplicity: Managing Complexity

As projects grow large, they often become very complex and difficult to understand. This complexity slows down everyone who needs to work on the system, further increasing the cost of maintenance. A software project mired in complexity is sometimes described as a big ball of mud.

Complexity has many possible symptoms: explosion of the state space, tight coupling of modules, tangled dependencies, inconsistent naming and terminology, hacks aimed at solving performance problems, special-casing to work around issues elsewhere, and many more. There is an important distinction:

Evolvability: Making Change Easy

It is extremely unlikely that your system's requirements will remain unchanged forever. They are much more likely to be in constant flux: you learn new facts, previously unanticipated use cases emerge, business priorities change, users request new features, new platforms replace old platforms, legal or regulatory requirements change, growth of the system forces architectural changes, etc.

In terms of organizational processes, Agile working patterns provide a framework for adapting to change. The Agile community has also developed technical tools and patterns that are helpful when developing software in a frequently changing environment, such as test-driven development (TDD) and refactoring.

Diagram: Patterns for Evolvability
Simple & Well-Abstracted Systems
Systems that are simpler and have good abstractions are much easier to modify than complex ones. Simplicity is a prerequisite for evolvability.
Test-Driven Development (TDD)
Write tests before code. Tests give you confidence to make changes — if a refactor breaks something, the tests will catch it immediately.
Refactoring
Restructure existing code without changing its behavior. Keep the codebase clean and adapted to current understanding of the problem.
Agile at Data Systems Level
Apply agile principles not just to code but to data system architecture. Example: Twitter evolved from Approach 1 to Approach 2 to Hybrid as they learned more about their load parameters.
Modular Design
Design modules with clear boundaries. When requirements change, ideally only one module needs to change, not the entire system.

Evolvability = Agile for Data Systems

The ease with which you can modify a data system, adapting it to changing requirements, is closely linked to its simplicity and its abstractions. Simple and easy-to-understand systems are usually easier to modify than complex ones. This is why the book uses the term evolvability to refer to agility on a data system level.

Key Takeaways

  • An application has to meet various requirements: functional (what it should do) and nonfunctional (general properties like reliability, scalability, compliance, maintainability, compatibility, etc.).
  • Reliability means making systems work correctly, even when faults occur. Faults can be in hardware (usually random and uncorrelated), software (usually systematic bugs), and humans (who inevitably make mistakes). Fault-tolerance techniques can hide certain types of faults from the end user.
  • Scalability means having strategies for keeping performance good, even when load increases. You must first describe load (load parameters), then measure performance (percentiles are the right tool), then choose scaling approaches appropriate to your architecture.
  • Maintainability is about making life better for the engineering and operations teams. Good abstractions reduce complexity. Good operability means good visibility. Good evolvability comes from simplicity and modularity.

1What is the difference between a fault and a failure? Why does the distinction matter?▼
A fault is one component of the system deviating from its spec. A failure is when the system as a whole stops providing the required service to the user. The distinction matters because the goal of fault-tolerance design is to build systems that prevent faults from causing failures. You cannot reduce the probability of a fault to zero, so instead you design mechanisms to prevent faults from escalating. We build reliable systems from unreliable parts.
2Why would you deliberately trigger faults in a production system? Isn't that counterproductive?▼
Many critical bugs are due to poor error handling. By deliberately inducing faults (like Netflix's Chaos Monkey, which randomly kills individual processes without warning), you ensure the fault-tolerance machinery is continually exercised and tested. This increases confidence that faults will be handled correctly when they occur naturally. If you only test happy paths, you won't discover that your failover mechanism is broken until a real outage happens.
3Why are software faults harder to deal with than hardware faults?▼
Hardware faults are typically random and independent — one disk failing doesn't imply another will fail. Software faults are systematic and correlated across nodes. A software bug that crashes one instance will likely crash all instances running the same code. They tend to cause far more system failures. Moreover, they lie dormant for long periods, only triggered when an unusual set of circumstances arises (when an assumption the software was making ceases to be true).
4Why is the mean (average) a poor metric for response times? What should you use instead?▼
The mean doesn't tell you how many users actually experienced a particular delay — it is a bad metric because it doesn't reflect what users "typically" see. Use percentiles: the median (p50) tells you the typical experience (half of requests are faster, half slower). p95, p99, p999 reveal how bad the outliers are. Amazon uses p999 because the customers with the slowest requests are often those with the most data on their accounts — they have made many purchases and are the most valuable customers.
5What was Twitter's key load parameter, and why did they end up with a hybrid approach?▼
The key load parameter was the distribution of followers per user (weighted by tweet frequency). Approach 1 (query-time join) couldn't keep up with 300k reads/sec needing heavy JOINs. Approach 2 (write-time fan-out into per-user caches) was fast for reads but a celebrity with 30M followers would trigger 30M writes per single tweet. The hybrid: use fan-out at write time for normal users, but for celebrities (users with millions of followers), fetch their tweets at read time and merge them into the timeline. This way, the average user gets fast cached reads, and the system isn't overwhelmed by celebrity write storms.
6What is "tail latency amplification" and why does it get worse with more backend services?▼
When a single end-user request requires multiple backend calls (even in parallel), it must wait for the slowest one to complete. Even if only 1 in 100 backend calls is slow (p99), the more backend calls you make in parallel, the higher the probability that at least one of them will be slow. With 5 parallel calls where each has 1% chance of being slow, the end-user has a 1 - 0.99^5 = ~5% chance of a slow experience. This is why even a small percentage of slow backend calls can degrade a significant percentage of end-user requests.
7What is "accidental complexity" and what is the best tool for removing it?▼
Accidental complexity is complexity that is not inherent in the problem the software solves, but arises only from the implementation. Moseley and Marks (in their paper "Out of the Tar Pit") suggest this complexity should be removed. The best tool for removing it is abstraction — hiding implementation detail behind a clean, simple-to-understand facade. Examples: SQL hides on-disk and in-memory data structures, indexing, and query processing; high-level languages hide machine code, registers, and syscalls. Finding good abstractions is extremely hard, but when found, they can benefit all applications that use them.
8Why is distributing a stateful data system fundamentally harder than distributing stateless services?▼
Distributing stateless services is straightforward — any machine can handle any request, no coordination needed. But taking a stateful data system from a single node to distributed introduces immense additional complexity: you must handle data consistency, replication lag, conflict resolution, and distributed coordination. This is why the conventional wisdom until recently was to keep your database on a single node until scaling cost or high-availability requirements forced distribution. The landscape is evolving as tools and abstractions for distributed systems become more mature.
9What is the relationship between latency and response time?▼
They are not synonyms, even though they are often confused. Response time is what the client sees: the total time between sending a request and receiving a response, including service time + network delays + queueing delays. Latency is specifically the duration that a request is latent — waiting to be handled, awaiting service. Latency is one component of response time. When people casually say "latency," they often mean response time, but the book makes the distinction precise.
10What does "there is no such thing as magic scaling sauce" mean?▼
An architecture that scales well is built around assumptions about which operations will be common and which will be rare (the load parameters). A system for 100,000 requests/sec at 1 kB each looks very different from 3 requests/min at 2 GB each — even though data throughput is the same. Scalable architectures are specific to the application, not generic or one-size-fits-all. They are built from general-purpose building blocks, arranged in familiar patterns — but the specific arrangement depends on your specific load assumptions. If those assumptions turn out wrong, the engineering effort is wasted or counterproductive.
11What is head-of-line blocking and why does it affect percentile measurement?▼
Head-of-line blocking occurs when a small number of slow requests hold up the processing of subsequent requests in the queue. Even though the queued requests might be fast to serve, the client waiting for them sees a high response time because of queueing delay (waiting behind the slow request). This is why it's important to measure response times on the client side. It also means that when artificially generating test load, the load-testing client must keep sending requests independently of the response time — otherwise it artificially keeps queues shorter, skewing measurements to look better than reality.
12Why does the book argue operability is at least as important as good software design?▼
The book states: "Good operations can often work around the limitations of bad (or incomplete) software, but good software cannot run reliably with bad operations." Good operations means: providing good monitoring, providing good support for automation, avoiding dependency on individual machines, maintaining good documentation, providing good default behavior but allowing overrides, self-healing but also providing manual control, and exhibiting predictable behavior. Even the best-designed software will fail in production without good ops practices around it.
