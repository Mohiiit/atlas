DDIA — Part II: Distributed Data

Chapter 5: Replication

"The major difference between a thing that might go wrong and a thing that cannot possibly go wrong is that when a thing that cannot possibly go wrong goes wrong it usually turns out to be impossible to get at or repair." — Douglas Adams. Replication means keeping a copy of the same data on multiple machines. All difficulty lies in handling changes to replicated data.

Why Replicate Data?

Replication serves several critical purposes. Every distributed database must replicate for at least one of these reasons:

Low Latency

Keep data geographically close to users. A user in London shouldn't wait for a round-trip to a datacenter in Virginia for every read.

High Availability

If one machine (or an entire datacenter) goes down, the system continues working from other replicas. No single point of failure.

Read Throughput

Scale out reads across many machines. If one node can serve 10k reads/sec, three replicas can serve 30k reads/sec.

The assumption for this chapter: your dataset is small enough that each machine can hold a complete copy. Splitting data across machines (partitioning/sharding) is covered in Chapter 6.

Three Replication Approaches

Almost all distributed databases use one of these three algorithms for replicating changes:

Replication Architecture Overview
Single-Leader
One leader, many followers

All writes go to one node (the leader). Leader streams changes to followers. Reads from any replica. Simple, no write conflicts. Most common.

Multi-Leader
Multiple leaders accept writes

Multiple nodes accept writes. Leaders replicate to each other. Handles multi-datacenter writes, but introduces write conflicts. Complex.

Leaderless
Any node accepts writes

Client writes to several nodes directly. Reads from several nodes in parallel. Uses quorums. Dynamo-style (Riak, Cassandra, Voldemort).

Leaders and Followers (Single-Leader Replication)

The most common approach. One replica is the leader (master/primary). When clients write, they send to the leader. The leader writes to its local storage, then sends a replication log (change stream) to all followers (read replicas/slaves/secondaries). Each follower applies writes in the same order.

Synchronous vs. Asynchronous Replication

The critical design choice: does the leader wait for followers to confirm before reporting success to the client?

Sync vs Async Replication Timeline
Synchronous

Leader waits for follower to confirm write → then responds to client. Guarantee: follower has up-to-date copy. Risk: if follower is down/slow, leader is blocked. One slow node blocks all writes.

Asynchronous

Leader sends to follower but doesn't wait → responds to client immediately. Fast, doesn't block on slow followers. Risk: if leader fails before replication, writes are lost (even if confirmed to client!).

Semi-synchronous

One follower is synchronous, the rest are asynchronous. Guarantees at least two nodes have the data. If the sync follower becomes slow, another one is promoted to synchronous. This is what "synchronous" usually means in practice.

The Durability Trade-off

Fully asynchronous replication is widely used (especially with many followers), but it means if the leader fails and is not recoverable, any writes not yet replicated are lost — even though they were confirmed to the client. This is a violation of the durability promise. But fully synchronous replication is impractical because a single node outage blocks all writes.

Setting Up New Followers

You can't just copy data files from the leader — the data is constantly changing. Locking the database would violate high-availability goals. The process:

Adding a New Follower — Step by Step
Step 1: Take a consistent snapshot of the leader's database (without locking). Most databases have this built in.
↓
Step 2: Copy the snapshot to the new follower node.
↓
Step 3: Follower connects to leader and requests all changes since the snapshot. Requires the snapshot's exact position in the replication log (PostgreSQL: log sequence number; MySQL: binlog coordinates).
↓
Step 4: Follower processes the backlog of changes. Once caught up, it can continue processing new changes in real time.

Handling Node Outages

Any node can go down (crash, reboot, maintenance). The system must keep running with minimal disruption.

Follower Failure: Catch-Up Recovery

Each follower keeps a log of data changes from the leader. After a crash, the follower knows the last transaction it processed. It reconnects to the leader and requests all changes since that point. Once caught up, it continues receiving the live stream. Straightforward.

Leader Failure: Failover

Much harder. A follower must be promoted to be the new leader. Clients must be reconfigured. Other followers must start consuming from the new leader. This is called failover. Can be manual or automatic.

Automatic Failover Steps

  • Detect leader failure: Usually via timeout (e.g., 30 seconds with no response). Hard to distinguish crash from network issue.
  • Choose a new leader: Election process or appointed by a controller node. Best candidate = the replica with the most up-to-date data (smallest replication lag).
  • Reconfigure the system: Clients send writes to the new leader. Old leader must become a follower if it comes back.

What Can Go Wrong With Failover

  • Lost writes: If asynchronous replication, the new leader may not have all the old leader's writes. When the old leader rejoins, what happens to those writes? Usually discarded — violating durability.
  • Split brain: Two nodes both believe they are leader. Both accept writes → data diverges → data loss/corruption. Must use fencing (STONITH: "Shoot The Other Node In The Head") to shut down one.
  • Stale data leaks: GitHub incident — MySQL follower promoted, used an out-of-date auto-increment counter, assigned already-used primary keys → leaked private data to wrong users.
  • Timeout too short: Unnecessary failovers during transient load spikes, making the situation worse.

Implementation of Replication Logs

How does the leader send data changes to followers? Several approaches:

Why Logical Logs Win

The trend is toward logical (row-based) replication logs. MySQL's binlog (row-based mode), PostgreSQL's logical decoding, and change data capture (CDC) tools all use this approach. It decouples replication from storage internals, allowing rolling upgrades, heterogeneous replicas, and external consumers (search indexes, data warehouses).

Problems with Replication Lag

With read-scaling architectures, reads go to followers while writes go to the leader. This only works with asynchronous replication (otherwise one slow follower blocks everything). But async replication means followers may be behind the leader. If you read from a lagging follower, you may see stale data.

If an application reads from an asynchronous follower, it may see outdated information. This inconsistency is temporary — if you stop writing and wait, followers will eventually catch up. This is called eventual consistency. The "eventually" is deliberately vague — under normal conditions it's fractions of a second, but under load or with network problems, it could be seconds or even minutes.

Three Consistency Guarantees

1. Read-After-Write Consistency (Read-Your-Writes)

Users should always see data they submitted themselves. Example: you update your profile, refresh the page, and it shows the old profile — unacceptable.

  • Technique 1: Read user's own data from the leader, other users' data from followers. E.g., always read your own profile from the leader.
  • Technique 2: Track update time — for 1 minute after any write, read from the leader. After that, followers have probably caught up.
  • Technique 3: Client remembers the timestamp of its most recent write. Ensure the replica serving the read has caught up to at least that timestamp. If not, route to another replica or wait.
  • Cross-device complication: If user updates on laptop, views on phone — the phone's request might go to a different datacenter. Need centralized metadata about update timestamps.

2. Monotonic Reads

Users should not see time go backward. If they saw data at time T, they should not later see data from before T. This happens when successive reads hit different followers with different lag.

Example: User refreshes a comment section. First request hits an up-to-date follower (sees 10 comments). Second request hits a lagging follower (sees only 8 comments). Comments seem to "disappear."

Solution: Each user always reads from the same replica (e.g., hash the user ID to choose a replica). If that replica fails, reroute to another.

3. Consistent Prefix Reads

Users should see data in causal order. If write A causally precedes write B, every reader should see A before B.

Example in a chat: "How far can you throw?" → "About 2 meters." If these writes land on different partitions with different lag, a third-party observer might see the answer before the question — confusing.

Solution: Ensure causally related writes go to the same partition. Or use algorithms to explicitly track causal dependencies.

Multi-Leader Replication

Single-leader replication has a major downside: all writes must go through one node. If that node is far away or unavailable, writes suffer. Multi-leader (master-master / active-active) replication allows multiple nodes to accept writes.

Use Cases for Multi-Leader Replication

Multi-Datacenter Operation

One leader per datacenter. Each datacenter's leader replicates to leaders in other datacenters. Writes go to the local leader → low latency. Inter-datacenter replication is async.

Offline Clients

Calendar apps (Google Calendar, etc.) need to work offline. Each device is a "leader" that accepts writes locally. When back online, sync with other devices. CouchDB designed for this.

Collaborative Editing

Google Docs, Etherpad. Each user's edits are immediately applied locally (like a "leader"), then asynchronously replicated. Smaller unit of change (single keystroke) than a DB transaction.

Handling Write Conflicts

The biggest problem with multi-leader replication: two leaders may concurrently modify the same data → write conflict. This doesn't happen in single-leader (the leader orders all writes sequentially).

Write Conflict Example
User 1 @ Leader A
Changes title to "B"
User 2 @ Leader B
Changes title to "C"
↓ Both succeed locally, then replicate ↓
CONFLICT! Leader A has "B", Leader B has "C". Both are "correct." What should the final value be?

Conflict Avoidance

The simplest strategy: avoid conflicts entirely. Route all writes for a particular record through the same leader. E.g., all writes for user X always go to datacenter Y's leader. This breaks down if you need to reroute (datacenter failure, user moves).

Conflict Resolution Strategies

StrategyHow It WorksTrade-off
Last Write Wins (LWW) Attach timestamp to each write, keep the one with the highest timestamp, discard others Prone to data loss. "Winning" write is arbitrary. Achieves convergence but silently drops writes. Only safe if keys are immutable after creation (e.g., use UUIDs).
Merge values Concatenate or union the conflicting values (e.g., "B/C" instead of choosing one) Preserves data but may produce weird results. App must handle merge logic.
Custom conflict handler Application code resolves conflicts (on write or on read) Most flexible but error-prone. CouchDB stores all versions, presents to app code on read. Bucardo calls handler in Perl.
CRDTs Data structures that are automatically mergeable (counters, sets, maps, registers designed for concurrency) Elegant but limited to specific data types. Used by Riak 2.0. Active area of research.
Operational Transformation Used by Google Docs. Transforms concurrent operations against each other Complex algorithm, designed for ordered lists of items (text editing).

All-to-All Causality Problem

With all-to-all topology, some network links may be faster than others, causing writes to overtake each other. A causally later write may arrive before the write it depends on. Example: an INSERT arrives at node C before the UPDATE that modifies the inserted row. To fix this, use version vectors to order events correctly. But many systems implement this poorly.

Leaderless Replication (Dynamo-Style)

Some databases take a radically different approach: abandon the concept of a leader and allow any replica to directly accept writes from clients. This was popularized by Amazon's Dynamo system (the internal one, not DynamoDB). Open-source implementations: Riak, Cassandra, Voldemort.

How It Works

The client sends writes to several replicas in parallel (or a coordinator node does it on the client's behalf, but unlike a leader, this coordinator doesn't enforce ordering). For reads, the client also reads from several nodes in parallel and uses version numbers to determine which value is newest.

Leaderless Read Repair — How Stale Data Gets Fixed

Node 2

Node 3
Value: v7 (current)
↓ Client reads from all 3 ↓
Client: Sees v7 from nodes 1&3, v6 from node 2. Uses v7 (highest version). Writes v7 back to node 2 → read repair.

Additionally, a background anti-entropy process constantly scans for inconsistencies between replicas.

Quorums for Reading and Writing

If there are n replicas, every write must be confirmed by w nodes, and every read must query r nodes. As long as w + r > n, you are guaranteed to read the latest value (at least one node in the read set has the latest write).

Common configurations:

• n=3, w=2, r=2 — typical. Tolerates 1 unavailable node.

• n=5, w=3, r=3 — tolerates 2 unavailable nodes.

• Fewer w and more r = faster writes, slower reads (and vice versa).

• w=n, r=1 — fast reads but just one unavailable node makes writes impossible.

Limitations of Quorum Consistency

Even with w + r > n, edge cases can cause stale reads:

  • Sloppy quorum: Writes may land on different nodes than reads (see below).
  • Concurrent writes: If two writes happen concurrently, it's unclear which one "won." LWW may drop data.
  • Write concurrent with read: The write may be reflected on only some replicas. Unclear if read returns old or new value.
  • Failed write rollback: If a write succeeded on some replicas but failed on others (not reaching w), the successful ones don't roll back. Subsequent reads may or may not see the value.
  • New node restoring: If a node carrying a new value fails and is restored from a stale replica, the write quorum may be broken retroactively.

Dynamo-style databases are generally optimized for use cases that can tolerate eventual consistency.

Sloppy Quorums and Hinted Handoff

What happens when network interruptions cut off a client from many of the database nodes? The client can still reach some nodes, but not the designated quorum nodes for a particular key.

Strict Quorum

If the client can't reach w of the n designated nodes, the write fails. Correct but reduces availability.

Sloppy Quorum

Accept writes on nodes that aren't among the designated n nodes for this key. Write goes to w nodes, but they might be "wrong" nodes — ones that don't normally hold this data. Increases write availability.

Hinted Handoff

When the network interruption is resolved, the temporary nodes forward the writes they accepted back to the correct "home" nodes. This is called hinted handoff.

Analogy: You lock yourself out of your house. Your neighbor holds your mail. When you get back, the neighbor gives you the accumulated mail. The mail was temporarily at the "wrong" house, but nothing was lost.

Important: A sloppy quorum is not a true quorum. Even if w writes succeed, there is no guarantee that a read of r nodes will see the latest value, because the writes may be on nodes outside the r set. Sloppy quorums are useful for increasing write availability and durability but do not guarantee reading the latest value.

Detecting Concurrent Writes

In multi-leader and leaderless systems, multiple clients can concurrently write to the same key. The database must handle this or data will be silently lost.

Last Write Wins (LWW)

Dangerous Simplicity

Attach a timestamp to each write. Keep the "biggest" timestamp, discard the rest. Achieves convergence (all replicas end up with the same value) but at the cost of durability: if two concurrent writes are both reported as successful, only one survives.

Even non-concurrent writes can be dropped if clock skew causes a causally later write to get a lower timestamp. Cassandra uses LWW as the only option; Riak as an optional feature.

Safe usage: Only write each key once (use UUIDs as keys), then treat as immutable. Otherwise, you will lose data.

The "Happens-Before" Relationship and Concurrency

Defining Concurrency

Two operations are concurrent if neither knows about the other. It doesn't matter if they literally happened at the same clock time (clocks are unreliable in distributed systems). What matters is causal dependency.

  • Operation A happens before B if B knows about A, depends on A, or builds upon A.
  • Two operations are concurrent if neither happens before the other.
  • Three possibilities for any two operations: A happened before B, B happened before A, or they are concurrent.
Concurrency vs. Happens-Before
Happens-Before

Client A: INSERT milk

↓ A's result is visible

Client B: UPDATE milk → 2

B depends on A's write. B happened after A. Later write wins.

Concurrent

Client A: SET x = "foo"

Client B: SET x = "bar"

Neither knew about the other. No causal dependency. This is a conflict that must be resolved.

Capturing Happens-Before with Version Numbers

The server maintains a version number per key, incremented on every write. The algorithm:

Version Number Algorithm
1. Server keeps a version number for every key, increments it on each write.
↓
2. Client reads a key: server returns all non-overwritten values + the latest version number. Client must read before writing.
↓
3. Client writes a key: must include the version number from its prior read + merge all values it received.
↓
4. Server receives write: overwrites all values with that version or below (they've been merged). Keeps values with higher versions (they're concurrent). Assigns new version number.

If a write arrives without a version number, it is concurrent with everything and won't overwrite any existing values.

Merging Sibling Values

When concurrent writes produce multiple values (called siblings in Riak), clients must merge them. For a shopping cart, take the union. But if items can be removed, simple union causes deleted items to reappear. Solution: use tombstones (deletion markers) to track removals during merge.

This merging is error-prone, which is why CRDTs (Conflict-free Replicated Data Types) are valuable — they handle merging automatically for supported data structures.

Version Vectors

The version-number algorithm above works for a single replica. With multiple replicas (leaderless), we need a version vector: a version number per replica per key.

Version Vector — Tracking Causality Across Replicas
Replica A
Version: {A:3, B:2, C:1}
Replica B
Version: {A:2, B:4, C:1}
Replica C
Version: {A:3, B:3, C:2}

Each replica increments its own version number when processing a write and tracks versions it has seen from other replicas.

To compare two version vectors: if every component in vector X is ≥ the corresponding component in vector Y, then X dominates (happened after) Y. Otherwise they are concurrent.

Version vectors are sent from database to clients on read, and back on write (Riak calls this the causal context).

Version Vectors vs. Vector Clocks

These terms are sometimes used interchangeably, but they are subtly different. Version vectors track the version of data per replica; vector clocks track logical time per process. For comparing replica state and detecting concurrency in databases, version vectors (specifically dotted version vectors as used in Riak 2.0) are the correct tool.

Chapter Summary

Replication — Complete Decision Tree
Single-Leader: Simple, no write conflicts. All writes → one leader. Followers may lag (eventual consistency). Used by most relational DBs, MongoDB, Kafka.
Multi-Leader: Multiple datacenters, offline clients, collaborative editing. Must handle write conflicts (LWW, merge, CRDTs). Complex topologies (circular, star, all-to-all).
Leaderless: Dynamo-style. Client writes/reads multiple nodes. Quorum: w+r>n. Sloppy quorums for availability. Version vectors for concurrency detection. Riak, Cassandra.

Replication Purposes

  • High availability — survive node/datacenter failures
  • Disconnected operation — work during network interruptions
  • Latency — data geographically near users
  • Scalability — handle read volume beyond one machine

Consistency Models Under Lag

  • Read-after-write — see your own writes
  • Monotonic reads — don't see time go backward
  • Consistent prefix reads — see causal order preserved
  • Eventual consistency — replicas converge if writes stop

First-Principles Questions

Test your understanding. Click to reveal the answer.

1Why is fully synchronous replication impractical, and why is fully asynchronous replication dangerous?▼
Synchronous: The leader must wait for every follower to confirm before acknowledging to the client. If any single follower is down, slow, or has a network issue, all writes are blocked. One bad node freezes the entire system. Asynchronous: The leader acknowledges immediately without waiting. If the leader then crashes before replication completes, those confirmed writes are permanently lost — a durability violation. The compromise is semi-synchronous: one follower is synchronous (guaranteeing at least two copies), the rest are asynchronous.
2Why can't you just copy the data files from the leader to set up a new follower?▼
The leader is constantly processing writes. Simply copying files would result in an inconsistent snapshot — some files contain data from one point in time, others from a different point. You'd need to lock the database to get a consistent copy, but that violates the goal of high availability. Instead, you take a consistent snapshot without locking (most databases support this), copy it to the new follower, then have the follower request all changes since that snapshot's position in the replication log to catch up.
3What makes leader failover so dangerous, and what is "split brain"?▼
Failover is dangerous because: (1) With async replication, the new leader may be missing writes from the old leader — these are typically discarded, violating durability. (2) If the old leader comes back, it still thinks it's the leader → split brain: two nodes both accept writes, causing data divergence and corruption. You must use fencing (forcibly shutting down the old leader, a.k.a. STONITH) to prevent this. (3) Choosing the right timeout is hard — too short causes unnecessary failovers, too long means a longer outage. GitHub had a real incident where a promoted MySQL follower reused primary keys, leaking private data to wrong users.
4Why is WAL (Write-Ahead Log) shipping a problematic replication method, and what replaced it?▼
WAL shipping sends the leader's low-level write-ahead log (byte-level disk block changes) to followers. The problem: the WAL is tightly coupled to the storage engine's internals (which disk blocks, which byte offsets). If the leader and follower run different versions of the database software, the WAL format may be incompatible. This means you cannot do zero-downtime upgrades — you can't run a newer version on followers first. Logical (row-based) logs replaced this: they describe changes at the row level (INSERT: all column values; UPDATE: changed columns) and are decoupled from storage internals, enabling version-independent replication and change data capture (CDC).
5Explain the three replication lag problems. Why does each occur, and how is each solved?▼
Read-after-write: User writes data, reads from a lagging follower, doesn't see their own write. Solution: read your own data from the leader, or track the write timestamp and ensure the serving replica has caught up. Monotonic reads: User's successive reads hit different followers with different lag, causing data to seem to "disappear" or go backward. Solution: pin each user to the same replica (e.g., hash user ID). Consistent prefix reads: Causally dependent writes land on different partitions with different lag. An observer sees the effect before the cause (e.g., an answer before its question). Solution: route causally related writes to the same partition, or use explicit causal dependency tracking.
6Why does multi-leader replication introduce write conflicts, and why can't you just use locking to prevent them?▼
In single-leader replication, all writes are serialized through one node, so conflicts are detected immediately (the second write blocks or fails). In multi-leader, two leaders can independently accept conflicting writes at the same time, each succeeding locally. The conflict is only detected later during asynchronous replication between leaders — by which time both writes have been confirmed to their clients. You can't use locking because the leaders are in different datacenters (that's the whole point) — acquiring a cross-datacenter lock for every write would defeat the latency and availability benefits of multi-leader replication.
7Why is Last Write Wins (LWW) a dangerous conflict resolution strategy, and when is it safe?▼
LWW achieves convergence by attaching timestamps to writes and discarding all but the "latest." The danger: concurrent writes that were all reported as successful are silently dropped. Worse, with clock skew, even non-concurrent writes can be dropped if a causally later write happens to get a lower timestamp. LWW is only safe when each key is written exactly once and treated as immutable afterward (e.g., use a UUID as the key for each write). Cassandra uses LWW as the default; if you update the same key multiple times, you will lose data.
8What is the quorum condition w + r > n, and why doesn't it guarantee strong consistency in practice?▼
With n replicas, writing to w nodes and reading from r nodes where w + r > n guarantees that at least one node in the read set has the latest write (by the pigeonhole principle). However, in practice, several edge cases break this: (1) Sloppy quorums: writes may go to nodes outside the normal "home" set, so the overlap guarantee doesn't hold. (2) Concurrent writes: which one "won" is ambiguous. (3) Concurrent write and read: the write may only be on some replicas during the read. (4) Partial write failures: successful replicas don't roll back. (5) Node restoration from stale backup: can retroactively break the quorum count. Dynamo-style databases are optimized for eventual consistency, not linearizability.
9What is a sloppy quorum, and how does hinted handoff work? Why doesn't a sloppy quorum provide the same guarantees as a strict quorum?▼
A sloppy quorum accepts writes on nodes that aren't among the designated n "home" nodes for a key, because the home nodes are temporarily unreachable. The write still goes to w nodes, but some may be "wrong" nodes. Hinted handoff: when the network heals, those temporary nodes forward the writes back to the correct home nodes. The reason it doesn't give strict quorum guarantees: the w nodes that received the write and the r nodes you read from may have no overlap — the writes are on temporary nodes outside the read set. So you may not see the latest value on read, even with w + r > n. Sloppy quorums increase write availability at the cost of read consistency.
10How does the version number algorithm detect concurrent writes, and why do clients need to merge sibling values?▼
The server maintains a version number per key. Each write includes the version number from the client's prior read. The server can determine: values at or below that version have been seen by the client and can be overwritten. Values at higher versions are concurrent (the client didn't know about them) and must be kept. When multiple concurrent values coexist, the server returns all of them as siblings. The client must merge them (e.g., union for a shopping cart) and write the merged result back. Without client-side merging, concurrent values accumulate forever. CRDTs automate this merging for supported data types.
11What is a version vector, and why is a single version number insufficient for leaderless replication?▼
A single version number works for one replica, but with multiple replicas accepting writes concurrently, a single counter can't capture which replica has seen which writes. A version vector maintains a version number per replica (and per key). Each replica increments its own counter on writes and tracks other replicas' versions. To compare: if every component in vector X ≥ the corresponding component in vector Y, X dominates (happened after) Y. Otherwise they are concurrent. This allows the system to correctly distinguish between overwrites (one causally follows the other) and conflicts (neither dominates). Riak sends the version vector to clients as a "causal context" which must be included in subsequent writes.
12Why does the book say concurrency is not about "happening at the same time"? How does this relate to the special theory of relativity?▼
In distributed systems, clocks are unreliable and cannot be used to determine ordering. Two operations are concurrent if neither knows about the other — regardless of physical time. Two writes could happen minutes apart and still be concurrent if a network partition prevented one from learning about the other. This mirrors Einstein's relativity: two events that occur some distance apart cannot affect each other if the time between them is less than the time light takes to travel that distance. In distributed systems, we replace "speed of light" with "speed of information propagation through the network." The key insight: causality (what information was available), not wall-clock time, defines ordering.
