Part 2: Secondary Indexes in Partitioned Databases, Rebalancing Strategies

Partitioning and Secondary Indexes

Primary key partitioning routes reads to a single partition. But what about queries on non-primary-key fields? Secondary indexes in a partitioned database are tricky — the index must also be partitioned.

Approach 1: Document-Partitioned Index (Local Index)

Each partition maintains its own secondary index covering only the data in that partition.

Used cars database: partitioned by car_id. Secondary index on color.
Partition 0 (car_id 0–499)
Data: [id:51, red], [id:120, green], [id:300, red]...

Local index:
red → [51, 300, ...]
green → [120, ...]
Partition 1 (car_id 500–999)
Data: [id:532, red], [id:710, blue]...

Local index:
red → [532, ...]
blue → [710, ...]
Query "all red cars": must query ALL partitions (scatter-gather). Wait for all to respond, merge results.
Write: only updates the partition containing that car_id. Fast writes.
MongoDB, Riak, Cassandra, Elasticsearch, SolrCloud, VoltDB all use local (document-partitioned) secondary indexes. The scatter-gather read is a known limitation.

Approach 2: Term-Partitioned Index (Global Index)

The secondary index itself is partitioned, by the indexed term. A single global index covers all partitions' data, but the index is itself spread across nodes.

Global index: partitioned by the indexed term (e.g., color range)
Index Partition A (a–r)
blue → [car_id:710, car_id:882, ...]
green → [car_id:120, ...]
red → [car_id:51, car_id:300, car_id:532, ...]
Index Partition B (s–z)
silver → [car_id:44, ...]
white → [car_id:191, car_id:440, ...]
Query "all red cars": hits Index Partition A only → gets all red car IDs. Fast reads!
Write "add new red car id:777": must update Index Partition A (red) AND Data Partition for car_id:777. Two separate nodes. Cross-partition write → requires distributed transaction or async update.
DynamoDB uses global secondary indexes. They're updated asynchronously — reads are eventually consistent. AWS documentation calls this out explicitly. For consistent global index reads: route to the index partition; data may be slightly stale.
Index typeReadWriteUsed by
Local (document-partitioned) Scatter-gather: query all partitions Only one partition (same as data) MongoDB, Cassandra, Elasticsearch
Global (term-partitioned) One index partition (fast) Multiple partitions (data + index) DynamoDB GSI, AWS Aurora

Rebalancing Partitions

Over time, things change: more data, more nodes, node failures. We need to move partitions between nodes — rebalancing. Requirements:

  • Load must be fairly distributed after rebalancing
  • Database must remain available during rebalancing (no downtime)
  • Move only the minimum data necessary (not everything)

Anti-Pattern: hash mod N

The naive approach: partition = hash(key) % number_of_nodes. When you add node N+1, most keys map to different partitions. Everything must be moved. This makes rebalancing extremely expensive.

Never use hash mod N for partitioning. Adding or removing a node triggers remapping of ~N/(N+1) of all keys. The entire dataset migrates.

Rebalancing Strategy 1: Fixed Number of Partitions

Create many more partitions than nodes. Move whole partitions (not data).
Before (3 nodes)
Node 1: P1,P2,P3,P4
Node 2: P5,P6,P7,P8
Node 3: P9,P10,P11,P12
Add Node 4
Node 1: P1,P2,P3
Node 2: P5,P6,P7
Node 3: P9,P10,P11
Node 4: P4,P8,P12 ← moved
Total partitions fixed at 12 (or 1000 in practice). Adding a node steals a few partitions from each existing node. Only those partitions' data moves. Used by: Elasticsearch, Riak, Couchbase, Voldemort.
Choose the number of fixed partitions carefully. Too few: partitions become large and rebalancing/recovery is slow. Too many: too much overhead per partition (metadata, coordination). Elasticsearch defaults to 5 shards per index.

Rebalancing Strategy 2: Dynamic Partitioning

Start: 1 partition
grows as data is written
→
Exceeds threshold (~10GB)
partition split in two
→
Two partitions
each can be moved to different nodes
Partitions merge when data shrinks below threshold. HBase and RethinkDB use this. Pro: adapts to data size automatically. Con: all data starts on 1 partition (new database is bottlenecked). Fix: pre-splitting (configure initial number of partitions).

Rebalancing Strategy 3: Partitioning Proportional to Nodes

Keep a fixed number of partitions per node (e.g., 256 per node). Add a node → add 256 new partitions; randomly steal half of existing partitions to fill them. Used by Cassandra and Ketama.

Operations: Automatic vs. Manual Rebalancing

ModeProsCons
Fully automatic Less operational work. Rebalances happen without human intervention. Can be unpredictable. A node under heavy load may appear slow → automatic rebalancing moves partitions away → more load on other nodes → cascade. Couchbase example.
Human-in-the-loop System suggests rebalancing, operator approves. Prevents cascades. More control. Requires operators to act. Slower to respond to failures.
Riak, Couchbase, Voldemort generate rebalancing suggestions but require operator approval. ZooKeeper maintains partition ownership, preventing conflicting moves.
