Chapter 6 -- Part 2

Partitioning: Rebalancing & Request Routing

How partitions migrate between nodes, and how clients find the right node for a given key. Covers rebalancing strategies, automatic vs. manual trade-offs, service discovery with ZooKeeper and gossip protocols, and massively parallel query execution.

Why Rebalancing Is Needed

Over time, the conditions of a running cluster change: query throughput grows, dataset sizes increase, and machines fail. Rebalancing is the process of moving data and requests from one node to another so the cluster can adapt to these changes.

Rebalancing Requirements

  • Fair load distribution -- after rebalancing, data and request load should be shared evenly across all nodes
  • Availability during rebalancing -- the database must continue serving reads and writes while data is being moved
  • Minimal data movement -- only move what is necessary to keep rebalancing fast and reduce network/disk I/O

How NOT To Do It: hash mod N

A naive approach is to assign keys using hash(key) mod N, where N is the number of nodes. This seems elegant at first: if you have 10 nodes, hash(key) mod 10 gives you a node number 0-9. But the moment N changes, nearly every key must move.

Diagram 1: Why hash mod N Breaks on Cluster Resize

Adding just 1 node causes ~90% of keys to be reassigned. This is catastrophic for large datasets.

The Core Problem

If N changes from 10 to 11, a key with hash 123456 moves from node 6 to node 3. When N goes to 12, it moves again to node 0. For a cluster with millions of keys, this means most data must be shuffled on every topology change -- exactly what we want to avoid.

Strategy 1: Fixed Number of Partitions

The solution is to decouple partitions from nodes. Create many more partitions than nodes (e.g., 1000 partitions for 10 nodes) and assign ~100 partitions to each node. When a node is added or removed, only entire partitions move -- the mapping of keys to partitions never changes.

Advantages

  • Key-to-partition mapping never changes
  • Only whole partitions move between nodes
  • Can assign more partitions to more powerful nodes
  • Operationally simple -- used by Riak, Elasticsearch, Couchbase, Voldemort

Trade-offs

  • Must choose partition count upfront -- hard to change later
  • Too many partitions = management overhead
  • Too few = limits future scaling
  • If data grows unpredictably, individual partitions may become too large or too small

Strategy 2: Dynamic Partitioning

With dynamic partitioning, partition boundaries are not fixed. When a partition exceeds a configured size threshold (e.g., 10 GB in HBase), it splits into two. When a partition shrinks below a threshold due to deletions, it merges with an adjacent partition. This adapts naturally to data volume.

Pre-splitting Caveat

An empty database starts with a single partition, so all writes initially go to one node while others sit idle. HBase and MongoDB allow pre-splitting -- configuring an initial set of partition boundaries -- to distribute the write load from the start. This requires some knowledge of the expected key distribution.

Dynamic partitioning works with both key-range and hash-partitioned data. MongoDB since version 2.4 supports dynamic splitting for both schemes. Used by HBase, RethinkDB, and MongoDB.

Strategy 3: Partitioning Proportional to Nodes

A third approach, used by Cassandra and Ketama, makes the number of partitions proportional to the number of nodes rather than data size. Each node has a fixed number of partitions (Cassandra defaults to 256 per node).

There is a spectrum between fully automatic and fully manual rebalancing. Each point on the spectrum has different operational trade-offs.

Danger: Automatic Rebalancing + Automatic Failure Detection

If a node is temporarily overloaded and slow to respond, other nodes may conclude it is dead. Automatic rebalancing then moves load away from it -- but this transfer creates additional load on the already-stressed node, the remaining nodes, and the network. This can trigger a cascading failure that takes down the entire cluster. Having a human in the loop prevents this scenario.

Couchbase takes yet another path: it does not rebalance automatically. A routing tier called moxi learns about routing changes directly from cluster nodes. For finding IP addresses to connect to in the first place (which change less frequently than partition assignments), DNS is often sufficient.

Parallel Query Execution (MPP)

Most NoSQL distributed datastores support simple single-key queries (or scatter/gather for document-partitioned secondary indexes). But massively parallel processing (MPP) relational database products, typically used for analytics and data warehousing, support much more sophisticated queries involving joins, filtering, grouping, and aggregation.

MPP vs. NoSQL Query Models

Simple key-value access and scatter/gather are the typical level of query sophistication in NoSQL stores. MPP databases (used in data warehousing and analytics) go far beyond this with multi-stage parallel execution plans. Techniques for parallel query execution are discussed further in Chapter 10 (Batch Processing).

Chapter 6 Summary

Partitioning Approaches

  • Key range partitioning -- keys sorted, partition owns a min-max range. Enables efficient range queries but risks hot spots on sequential keys. Rebalanced dynamically by splitting ranges.
  • Hash partitioning -- hash function applied to each key, partition owns a hash range. Destroys key ordering (no range queries) but distributes load more evenly. Often uses fixed partition count.
  • Hybrid (compound key) -- one part of the key identifies the partition, another part determines sort order within the partition.

Secondary Index Partitioning

  • Document-partitioned (local) indexes -- secondary index stored in same partition as primary data. Single partition updated on write, but reads require scatter/gather across all partitions.
  • Term-partitioned (global) indexes -- secondary index partitioned by indexed value. Reads served from a single partition, but writes may touch multiple index partitions.

Rebalancing Strategies

  • DO NOT use hash mod N -- almost all keys move when N changes
  • Fixed partition count -- create many partitions upfront, move whole partitions between nodes (Riak, Elasticsearch)
  • Dynamic partitioning -- split/merge partitions as data grows/shrinks (HBase, MongoDB)
  • Proportional to nodes -- fixed partitions per node, new nodes split existing partitions (Cassandra)

Request Routing

  • Three approaches: any-node forwarding, dedicated routing tier, smart client-side routing
  • ZooKeeper provides centralized coordination for many systems (HBase, Kafka, SolrCloud)
  • Gossip protocol provides decentralized alternative (Cassandra, Riak)
  • MPP databases support parallel multi-stage query execution for analytics
"By design, every partition operates mostly independently -- that's what allows a partitioned database to scale to multiple machines. However, operations that need to write to several partitions can be difficult to reason about."

Review Questions

1Why is hash(key) mod N a bad strategy for assigning keys to nodes? ▼

When the number of nodes N changes (even by just one), the modulo operation produces a different result for most keys. For example, a key with hash 123456 maps to node 6 with 10 nodes (mod 10 = 6), but moves to node 3 with 11 nodes (mod 11 = 3). This means roughly (N-1)/N of all keys -- around 90% or more -- must be moved between nodes on every topology change. This excessive data movement makes rebalancing prohibitively expensive for large datasets.

2How does the fixed partition count strategy solve the rebalancing problem? ▼

By creating many more partitions than nodes (e.g., 1000 partitions for 10 nodes), the mapping of keys to partitions never changes -- only the mapping of partitions to nodes changes. When a node is added, it "steals" some whole partitions from existing nodes. When a node is removed, its partitions are distributed to remaining nodes. Only the data in moved partitions needs to be transferred, and the number of partitions stays constant. The old assignment remains valid during transfer for serving reads and writes.

3What is the "cold start" problem with dynamic partitioning, and how is it mitigated? ▼

An empty database with dynamic partitioning starts with a single partition, since there is no prior knowledge of where to draw partition boundaries. This means all writes initially go to a single node while other nodes sit idle, creating a bottleneck. The mitigation is pre-splitting: HBase and MongoDB allow configuring an initial set of partition boundaries on an empty database. For key-range partitioning, this requires some advance knowledge of the expected key distribution.

4Why can fully automatic rebalancing be dangerous when combined with automatic failure detection? ▼

Consider a node that is temporarily overloaded and responding slowly. Automatic failure detection may conclude this node is dead. Automatic rebalancing then kicks in and starts moving partitions away from the "dead" node. But this transfer process puts additional load on: (1) the already-overloaded node (which must send data), (2) the other nodes (which must receive and serve the extra partitions), and (3) the network. This additional load can push more nodes into overload, triggering more rebalancing -- a cascading failure that can take down the entire cluster. A human in the loop can recognize this pattern and prevent it.

5Compare ZooKeeper-based service discovery with gossip-based approaches. What are the trade-offs? ▼

ZooKeeper-based (HBase, Kafka, SolrCloud, Espresso): A centralized coordination service maintains the authoritative partition-to-node mapping. Nodes register with ZooKeeper; routing tiers and clients subscribe for updates. The advantage is a single source of truth, but it introduces an external dependency -- the ZooKeeper cluster itself must be highly available and managed.

Gossip-based (Cassandra, Riak): Nodes communicate cluster state changes directly to each other via a gossip protocol. Any node can receive any request and forward it. This eliminates the external dependency but puts more complexity into the database nodes themselves. There is also a propagation delay as state changes ripple through the gossip network, so routing information may be briefly stale.

6How does MPP query execution differ from the typical query model of NoSQL distributed datastores? ▼

Most NoSQL distributed datastores support simple queries: single-key reads/writes, or scatter/gather across partitions for secondary index lookups. MPP (Massively Parallel Processing) databases, used primarily for analytics and data warehousing, support complex SQL queries with joins, filtering, grouping, and aggregation. The MPP query optimizer decomposes a complex query into multiple execution stages, many of which can run in parallel across different nodes. Each node processes its partition of the data for each stage, and partial results are merged. This is especially beneficial for queries that scan large portions of the dataset, providing orders-of-magnitude speedups over sequential execution.
