Chapter 8 — The Trouble with Distributed Systems

Part 1: Partial Failures, Unreliable Networks, Timeouts

A single computer operates in a clean model: if a computation fails, it either crashes (total failure) or gives a wrong answer. In a distributed system, components can fail partially — some work, some don't — in ways that are almost impossible to reason about.

Faults and Partial Failures

A single computer's hardware either works or doesn't. The OS provides clean abstractions. Programs behave deterministically. If something goes wrong, the whole computer halts.

Single computer
Deterministic. Either works or crashes. No partial failure. Reboot and it's back.
Distributed system
Non-deterministic. Components fail independently. Network delays are unbounded. The system may be partially operational in ways you can't detect.
If we want to build reliable systems from unreliable components, we need to understand what can go wrong — and design systems that tolerate those failures. The first step is accepting that partial failures will happen.

Cloud Computing vs. Supercomputing

AspectHPC / SupercomputerCloud / Internet-scale
Hardware Specialized, highly reliable, expensive Commodity hardware, failures expected
Network High-bandwidth low-latency (InfiniBand, shared memory) Ethernet, variable latency, packet loss
On failure Stop job, fix hardware, restart Continue operating despite partial failures
Goal Run one big computation reliably Always-on, serve millions of users

Internet services can't afford to stop and restart when a node fails. They must continue operating — degraded if necessary — while repairs happen. This requires treating failures as normal operating conditions.

Unreliable Networks

The internet (and most datacenter networks) uses asynchronous packet networks: best-effort delivery, no guaranteed timing, no guaranteed ordering.

When you send a request and get no response, you can't tell why:
Request lost
Your message never arrived. Network dropped it.
Request queued
Message is waiting in a queue (network or remote OS) and will arrive later.
Remote node failed
The node crashed, isn't running anymore.
Response lost
Request processed OK, but reply got lost on the way back.
Response delayed
Reply is queued somewhere and will arrive eventually.
You CANNOT distinguish these cases from the outside. All you know is: you sent a request and didn't hear back within some time.

Network Faults in Practice

Network faults are common, even in well-maintained datacenters:

  • Amazon study: median network partition lasted about 10 minutes
  • Shark Tank cable bites, accidental fiber cuts, misconfigured switches
  • A rack can become partitioned from the rest of the datacenter
  • An EC2 instance can lose its network for an indefinite time while still running
Your system must handle network faults even if you don't handle them by recovering — you need at least a defined error response. Untested failure paths are often broken. Netflix's Chaos Monkey deliberately kills production instances to ensure teams test their failure handling.

Detecting Faults

How do you detect if a node is down? You can't know for certain. You can only get probabilistic evidence:

Positive signals (node is up)
  • TCP SYN-ACK (OS is running)
  • Explicit "I'm alive" heartbeat
  • Successful response to request
  • Process responds to signals
No signal
  • TCP RST → process crashed (OS still up)
  • ICMP unreachable → network issue
  • Timeout → could be anything
  • Silence → unknown state
Heartbeat timeouts are the only general mechanism. HBase: node is considered dead after 3 missed heartbeats.

Timeouts and Unbounded Delays

How long should you wait before declaring a node dead? There's no right answer — it depends on your latency distribution and what happens when you're wrong.

Too short timeout
Declare nodes dead that are just slow (high GC pause, network congestion). Trigger unnecessary failovers, adding load at exactly the wrong time.
Too long timeout
Long wait before detecting failures. Users wait a long time for timeouts. Meanwhile the failed node may have committed half a transaction.

Network Congestion and Queueing

Variable network latency comes from many queueing points:

Message leaves sender
↓
Network switch queue (full switch → drop packet, TCP retry)
↓
Receiver OS network queue (if receiver is busy)
↓
Virtualization hypervisor queue (if CPU is allocated to another VM)
↓
Application-level queue (if application is handling another request)
↓
Message processed by application
Each queue adds variable delay. In cloud environments, other VMs compete for the same physical hardware. A 99th percentile latency can be 100× the median.

Synchronous vs. Asynchronous Networks

The telephone network is synchronous: when you make a call, a circuit is established with guaranteed bandwidth reserved end-to-end. Latency is bounded. No queueing.

The internet is asynchronous: no circuits, no reservations. Packets compete. Queuing delays are unbounded. This is why TCP timeouts are necessary.

Why not build data center networks like telephone circuits? TCP connections do have a concept of "establishing a connection" — but they use statistical multiplexing (many flows share links) which gives much higher utilization. A circuit reservation for every database query would be extremely wasteful. The cost is variable latency — a trade-off that made the internet work at scale.
