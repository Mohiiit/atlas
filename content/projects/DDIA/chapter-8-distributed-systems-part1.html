Faults and Partial Failures

On a single computer, things are deterministic: the same operation always produces the same result. Hardware faults cause total system failure (kernel panic, blue screen). Computers are designed to hide the fuzzy physical reality and present an idealized model of mathematical perfection.

In a distributed system, we are no longer in this idealized world. A partial failure means some parts of the system are broken in unpredictable ways, while other parts work fine. Partial failures are nondeterministic -- you may not even know whether something succeeded or not.

And I'm not even an ops guy." -- Coda Hale

Building Reliable Systems from Unreliable Components

This is not as paradoxical as it sounds. Just like TCP provides reliable delivery over unreliable IP, and error-correcting codes provide accurate data over noisy channels, we can build a system more reliable than its parts. But there are always limits -- TCP cannot hide network delays, and ECC cannot handle total signal interference.

Unreliable Networks

The systems in this book are shared-nothing systems: machines connected only by a network. The internet and most datacenter networks are asynchronous packet networks -- they give no guarantees about when (or whether) a message arrives.

Detecting Faults

Many systems need to automatically detect faulty nodes (load balancers removing dead nodes, databases promoting followers). The network's uncertainty makes this hard, but some feedback mechanisms exist:

Explicit Signals (Sometimes Available)

  • TCP RST/FIN: OS refuses connection if no process is listening
  • Process crash notification: Script notifies peers (e.g., HBase)
  • Switch management interface: Query hardware for link failures
  • ICMP Unreachable: Router reports destination unreachable

Why You Can't Rely on Them

  • Node may crash after TCP ACK but before processing request
  • No access to switches in shared/cloud datacenters
  • Routers have the same limitations as other network participants
  • Asymmetric failures: one direction works, other doesn't

In general, you must assume you will get no response at all. Retry a few times, wait for a timeout, then eventually declare the node dead.

Timeouts and Unbounded Delays

If timeouts are the only sure way to detect faults, how long should they be?

The Timeout Tradeoff

✗ Risk of false positives
✗ Cascading failures from premature declarations

In a fictitious world with maximum network delay d and maximum processing time r, the ideal timeout would be 2d + r. But in asynchronous networks, neither d nor r has an upper bound.

Cascading Failure Risk

If a node is not dead but merely slow due to overload, prematurely declaring it dead transfers its load to other nodes, which can cause them to become overloaded and also be declared dead. In the extreme case, all nodes declare each other dead and everything stops working.

Network Congestion and Queueing

Just like road traffic, the variability of packet delays is most often due to queueing. A packet can be delayed at any of several stages on its journey.

5-Stage Queueing -- Where Packets Get Delayed
1 Network Switch Queue
Multiple nodes send to the same destination simultaneously. Switch queues packets and feeds them one by one. If the queue fills up, packets are dropped.
2 OS Incoming Queue
Packet arrives at destination, but all CPU cores are busy. OS queues the request until the application is ready. Wait time can be arbitrary.
3 VM Monitor Buffer (Virtualized Environments)
VM is paused for tens of ms while another VM uses the CPU. Incoming data is buffered by the hypervisor, adding more delay variability.
4 TCP Flow Control (Sender-Side)
TCP limits sending rate to avoid overloading network or receiver (congestion avoidance / backpressure). Data queues at the sender before entering the network.
5 TCP Retransmission
If a packet is not ACKed within a timeout, TCP retransmits it. The application sees the delay (waiting for timeout + retransmission + new ACK) but not the loss itself.

TCP vs. UDP: A Latency Tradeoff

Latency-sensitive applications (VoIP, video calls) use UDP to avoid flow control and retransmission delays. If a packet is lost, the application fills the gap with silence rather than waiting. "Could you repeat that please? The sound just cut out for a moment." -- the retry happens at the human layer.

TCP hides packet loss but adds variable delays. UDP exposes packet loss but keeps latency predictable.

Adaptive Timeouts

Rather than hard-coded constant timeouts, systems can continually measure response times and jitter, automatically adjusting timeouts based on the observed distribution. The Phi Accrual failure detector (used in Akka and Cassandra) does exactly this. TCP retransmission timeouts work similarly.

Why packet switching wins for data: Internet traffic is bursty. A circuit wastes capacity when idle. If you guess bandwidth too low, the transfer is unnecessarily slow. Too high, and the circuit cannot be established. TCP dynamically adapts to available capacity, making packet switching far more efficient for data transfer.

Latency and Resource Utilization

Variable delays are a consequence of dynamic resource partitioning. A telephone wire with 10,000 circuit slots allocates them statically -- even if 9,999 are unused, your call gets the same fixed bandwidth. The internet shares bandwidth dynamically: senders jostle for access, causing queueing but maximizing utilization.

The fundamental tradeoff: Static partitioning (circuits) = predictable latency but waste. Dynamic partitioning (packets) = efficient utilization but variable delays. Variable delays in networks are not a law of nature, but the result of a cost/benefit tradeoff.

Unreliable Clocks

Clocks answer two different kinds of questions: durations (how long did something take?) and points in time (when did something happen?). In a distributed system, both are tricky because communication is not instantaneous and each machine has its own clock -- a quartz crystal oscillator that drifts independently.

Clocks can be synchronized via NTP (Network Time Protocol), which adjusts the computer clock according to a group of time servers. But NTP itself depends on the network, which, as we just discussed, is unreliable.

Clock Synchronization and Accuracy

Monotonic clocks don't need synchronization. But time-of-day clocks must be set via NTP or other external source -- and this process is surprisingly unreliable.

Sources of Clock Error in NTP Synchronization
1
Quartz Crystal Drift
Google assumes 200 ppm drift = 6 ms per 30s sync interval, or 17 seconds per day without sync. Temperature-dependent.
2
Forced Clock Reset
If clock differs too much from NTP, it may jump forward or backward. Applications see time go backward.
3
Firewall Blocking NTP
Node accidentally firewalled from NTP servers. The misconfiguration can go unnoticed for a long time.
4
Network Delay
NTP accuracy is limited by network round-trip time. Minimum 35 ms error over internet. Spikes up to 1 second.
5
Wrong/Misconfigured NTP Servers
Some NTP servers report time off by hours. Clients query multiple servers and ignore outliers, but it's still risky.
6
Leap Seconds
A minute with 59 or 61 seconds. Has crashed many large systems. Best mitigation: "smearing" the adjustment over a day.
7
Virtualized Hardware Clocks
VM paused while other VMs run. Clock appears to jump forward by tens of ms when VM resumes.
8
Untrusted User Devices
Users can deliberately set wrong time (e.g., to cheat games). Clock may be wildly in the past or future.

Timestamps for Ordering Events

It is tempting but dangerous to use time-of-day clocks to order events across nodes. The classic failure case involves Last Write Wins (LWW).

Node 2 sees x=1 with timestamp 42.004 and x=2 with timestamp 42.003. Since 42.004 > 42.003, LWW keeps x=1. But x=2 was causally later (Client B read x=1 before incrementing). The 3ms clock skew between Node 1 and Node 3 reversed the causal order.

Three Problems with LWW

  • Silent data loss: A node with a lagging clock cannot overwrite values from a node with a fast clock until the skew has elapsed. Arbitrary amounts of data can be silently dropped.
  • Cannot distinguish sequential from concurrent: LWW does not know if two writes were causally related or truly concurrent. Version vectors are needed for causality tracking.
  • Timestamp ties: Two nodes can generate the same millisecond timestamp. Tiebreakers (random numbers) can also violate causality.

Alternative: Logical clocks (incrementing counters) are a safer alternative. They don't measure wall-clock time -- only the relative ordering of events.

Clock Readings Have a Confidence Interval

Even if your clock reads with microsecond resolution, that doesn't mean the value is accurate to that precision. A clock reading is not a point in time -- it's a range of times within a confidence interval. If you only know the time +/- 100 ms, the microsecond digits are meaningless.

Key Takeaways for Part 1

  • Partial failures are nondeterministic -- you may not know if something succeeded or not
  • Networks are unreliable: packets can be lost, delayed, duplicated, or reordered. The only option is timeouts, and there is no "correct" timeout value.
  • Queueing causes variable delays at 5 different stages (switch, OS, VM, TCP sender, TCP retransmission).
  • Packet switching trades latency guarantees for efficiency -- variable delays are a cost/benefit tradeoff, not a law of nature.
  • Time-of-day clocks can jump, making them unsafe for measuring durations. Monotonic clocks are safe for durations but not comparable across nodes.
  • NTP has at least 8 sources of error. Clock readings are confidence intervals, not points in time.
  • LWW with wall-clock timestamps causes silent data loss due to clock skew. Use logical clocks or version vectors instead.
  • Google's TrueTime/Spanner solves ordering by exposing clock uncertainty and waiting for intervals to separate.

Test Your Understanding

1Why is a distributed system fundamentally different from a single-node system in terms of failure modes?
▼

A single computer is deterministic: it either works correctly or crashes completely. Hardware faults lead to total system failure (kernel panic). A distributed system experiences partial failures that are nondeterministic -- some parts may be broken in unpredictable ways while other parts work fine. You may not even know whether an operation succeeded or not, because the network can lose both the request and the response. This nondeterminism makes distributed systems fundamentally harder to reason about.

2If you send a request over the network and don't get a response, what are the possible explanations, and why can't you distinguish between them?
▼

Six things could have happened: (1) the request was lost, (2) the request is queued, (3) the remote node is dead, (4) the remote node is temporarily paused (e.g., GC), (5) the server processed the request but the response was lost, (6) the response is delayed. From the sender's perspective, all six look identical: you simply don't have a response. In an asynchronous network, the only information you have is the absence of a response. The only recourse is a timeout, but even after a timeout, you still don't know which scenario occurred -- and if the request is queued somewhere, it may still be delivered later.

3Why can setting timeouts too short be worse than setting them too long? Describe the cascading failure scenario.
▼

A short timeout detects faults faster but risks false positives -- declaring a node dead when it's merely slow. If a node is just overloaded (not dead), declaring it dead triggers transferring its responsibilities to other nodes, putting more load on them. Those nodes may then become slow, get declared dead themselves, and their load shifts again. In the extreme case, all nodes declare each other dead and the entire system stops working. Additionally, if a "dead" node was actually processing a request (e.g., sending an email), another node may take over and perform the action a second time. A long timeout causes users to wait but avoids this cascading collapse.

4Why do computer networks use packet switching instead of circuit switching, even though circuit switching provides bounded delays?
▼

Internet traffic is bursty: a web page request, email, or file transfer needs high bandwidth briefly, then nothing. Circuit switching allocates a fixed bandwidth slot that stays reserved even when idle -- wasteful for bursty traffic. If you guess the allocation too low, the transfer is unnecessarily slow; too high, and the circuit cannot be established. Packet switching shares bandwidth dynamically: TCP adapts the data rate to available capacity, maximizing utilization. The tradeoff is queueing (unbounded delays) in exchange for much better utilization of the wire. Variable delays are not a law of nature, but the result of choosing efficiency over predictability.

5Why is Last Write Wins (LWW) dangerous when using wall-clock timestamps, even with "good" NTP synchronization? What should you use instead?
▼

Even with NTP synchronization within a few ms, clock skew between nodes can reverse causal ordering. In the textbook example, Client B reads x=1 and increments to x=2 (causally after the write of x=1), but because Node 3's clock is ~3ms behind Node 1, the x=2 write gets a lower timestamp than x=1. Node 2, using LWW, keeps x=1 and silently drops x=2. This is silent data loss with no error reported. The problems are: (a) nodes with lagging clocks cannot overwrite fast-clock values, (b) LWW cannot distinguish sequential from concurrent writes, (c) timestamp ties need arbitrary tiebreakers that can also violate causality. Instead of physical timestamps, use logical clocks (incrementing counters like Lamport timestamps or vector clocks) that track only the relative ordering of events, not wall-clock time.

6How does Google Spanner's TrueTime API solve the problem of ordering distributed transactions using clocks? What is the key insight?
▼

Most systems report a single timestamp, hiding the uncertainty. TrueTime reports a confidence interval [earliest, latest] representing the range within which the actual time lies. The key insight: if two transactions' intervals do not overlap (Alatest < Bearliest), then B definitely happened after A. If they overlap, the order is uncertain. To guarantee correct ordering, Spanner deliberately waits for the length of the confidence interval before committing a read-write transaction. This ensures any subsequent transaction's interval will start after the current one ends, making the causal order unambiguous. To minimize wait time, Google deploys GPS receivers or atomic clocks in each datacenter, keeping uncertainty to ~7 ms. This is an active area of research but not yet implemented in mainstream databases outside Google.
