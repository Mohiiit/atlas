Stream Processing

A stream processor takes one or more input streams, applies continuous computations, and produces output streams (or writes to a database/output). Unlike batch jobs, stream processors run continuously — no defined start/end.

Uses of Stream Processing

Complex event processing (CEP)
Detect patterns across a stream: "alert if 5 failed logins from the same IP within 10 minutes."
Stream analytics
Rolling aggregations: windowed counts, averages, histograms. Dashboard metrics updated in near-real-time.
Materialized view maintenance
Keep a cache or read model updated as the underlying data changes (CDC → stream processor → cache).
Search on streams
Stored queries matched against incoming events: "notify me of any new article mentioning keyword X."

Time in Stream Processing

Time is the hardest problem in stream processing. There are two notions of time:

Event time
When the event actually occurred (recorded in the event's timestamp). This is what you usually care about.
Problem: events arrive out of order and late (mobile app offline for 2 hours, then sends a batch of events with old timestamps).
Processing time
When the event arrives at the stream processor. Easy to measure (wall clock). Not meaningful for analytics — a late event's processing time has nothing to do with when it happened.

Windows

Windows group events by time for aggregation. Three main types:

Tumbling window
Fixed-size, non-overlapping.
Window [10:00–10:01], [10:01–10:02]...
Each event in exactly one window.
Sliding window
Fixed-size, overlapping.
"Events in the last 5 minutes" evaluated at every second.
Each event may be in multiple windows.
Session window
Activity-based. Windows end after a gap of inactivity (e.g., 30 min no events).
Variable size. Used for user sessions.

Watermarks and Late Events

How long should a window wait for late events before emitting its result? The stream processor needs to know when it's "safe" to close a window.

Watermark definition:
A watermark W(t) is an assertion: "all events with event time ≤ t have been received."
When the watermark passes t, the window [_, t] can be closed and its result emitted.

Heuristic watermark: estimate based on observed event timestamps (max seen - some lag)

Events after watermark: "late" events. Options: (1) ignore; (2) emit a correction; (3) reprocess the window (expensive)
Google Dataflow and Apache Flink pioneered the event-time + watermarks model. The key insight: accuracy vs. latency trade-off. A generous watermark (wait longer) gives more accurate results. A tight watermark (emit quickly) gives lower latency but more late events. The watermark lag is a system parameter you tune per use case.

Stream Joins

Joining streams is harder than joining batches — you never have the full dataset.

Stream-Stream Join (Windowed Join)

Match events from two streams within a time window.
Example: join search queries with ad clicks within 1 hour (to measure ad conversion).
Implementation: buffer each stream's events for the window duration. When an event arrives, look for matches in the other stream's buffer. Emit matches.
State: buffer must be maintained in memory (or spilled to disk). Window must be bounded or state grows unbounded.

Stream-Table Join (Enrichment)

Enrich stream events with current state from a database or lookup table.
Example: add user demographics to click stream events based on user_id.
Implementation: stream processor loads the table into local state (or queries remotely per event).
Table updates: use CDC to keep the local copy updated as the table changes.
This pattern is very common: event stream enriched with reference data from a slowly-changing dimension table.

Table-Table Join (Materialized View)

Both inputs are tables (change streams via CDC). Output is a materialized view of the join. When either table changes, the join output updates. This is how stream processors maintain derived read models.

Fault Tolerance in Stream Processing

A stream processor must handle failures without losing or duplicating events. Two main approaches:

ApproachHow it worksTrade-off
Microbatching Collect events into small batches (e.g., 1 second). Process each batch atomically. If it fails, retry the batch. Spark Streaming uses this. Simpler implementation. Minimum latency = batch interval (1s). Not true continuous processing.
Checkpointing Periodically snapshot the entire processing state to durable storage. On failure, recover from the last checkpoint and reprocess events from that point. Flink and Kafka Streams use this. Lower latency (true streaming). Recovery cost = state size + events since last checkpoint.
Idempotent writes + at-least-once Reprocess events on failure. Ensure output writes are idempotent (same event processed twice → same result). Simpler than exactly-once. Depends on idempotency of output. Not always achievable (e.g., incrementing a counter is not idempotent).

Chapter 11 Summary

Events
Immutable facts. Small, time-stamped. Unbounded stream. Batch = bounded subset of a stream.
Message brokers
Traditional (RabbitMQ): push, delete on ack, competing consumers. Log-based (Kafka): pull, retain, replay, fan-out.
Kafka
Topics → partitions → offsets. Consumer groups. Ordering per partition. Key-based routing. Compacted topics for state.
CDC
DB WAL → change events → derived systems. DB is source of truth. No dual-write problem.
Time
Event time vs. processing time. Windows: tumbling, sliding, session. Watermarks for late events.
Fault tolerance
Microbatching (Spark), checkpointing (Flink), idempotent writes. Exactly-once via transactional APIs.

Review Q&A

1
What are the key differences between a log-based message broker (Kafka) and a traditional message queue (RabbitMQ)?
›
Retention: Traditional MQs delete messages after acknowledgment. Kafka retains messages for a configurable period (or forever for compacted topics). Consumer model: Traditional MQs push messages to consumers and use acknowledgments to track delivery. Kafka consumers pull at their own pace and track progress via offsets. Fan-out: In a traditional MQ, a message in a queue is delivered to one consumer (competing consumers). In Kafka, each consumer group independently reads all messages in a topic. Multiple groups can process the same events independently. Replay: Traditional MQ: once acknowledged, gone. Kafka: any consumer can rewind to an earlier offset and reprocess. Great for bug fixes, new consumer deployments, auditing. Ordering: Kafka guarantees ordering within a partition. Traditional MQs often don't guarantee order across concurrent consumers. Use case: Traditional MQ: task queues, RPC-style messaging, short-lived jobs. Kafka: event streams, audit logs, data pipelines, stream processing.
2
What is at-least-once delivery and when is it a problem?
›
At-least-once delivery means: a message will be delivered to the consumer at least once, but possibly more times. This happens when: a consumer processes a message and crashes before acknowledging → the broker redelivers it to another consumer (or the same consumer on restart) → the message is processed twice. At-least-once is a problem when the consumer's operation is not idempotent — applying the same operation twice gives different results than once. Examples where it's a problem: (1) Incrementing a counter (count ends up too high); (2) Sending an email notification (user gets two emails); (3) Charging a payment (user charged twice). Solutions: (1) Make the operation idempotent (use a unique event ID and check if it's already been processed); (2) Use exactly-once semantics (Kafka's transactional API + idempotent producers); (3) Use a deduplication layer (track processed message IDs in a database). Many systems accept at-least-once for analytics (slightly wrong counts) but require exactly-once for financial or operational use cases.
3
How does Kafka use partitions to achieve both parallelism and ordering?
›
Kafka topics are divided into partitions. Each partition is an ordered, append-only log. Within a partition, messages have monotonically increasing offsets and are consumed in order. Parallelism: each consumer in a consumer group is assigned one or more partitions. Multiple consumers can process different partitions simultaneously — the degree of parallelism equals the number of partitions. Ordering: messages within a partition are strictly ordered. A producer assigns messages to partitions by key (hash(key) % num_partitions). All messages with the same key go to the same partition → same consumer → processed in order. Example: all events for user_id=42 go to partition 7 → one consumer processes all user 42's events in order. Cross-partition: no ordering guarantee. If user A's events go to partition 0 and user B's to partition 1, the relative order of A's and B's events across partitions is undefined. This is the fundamental design trade-off: ordering is guaranteed only within a partition. To get total ordering across all events, you'd need a single partition — which eliminates parallelism.
4
What is Change Data Capture (CDC) and what problem does it solve?
›
CDC captures every change (insert, update, delete) to a database and emits them as a stream of events. It solves the dual-write problem: if an application writes to both a database and an Elasticsearch index (or cache), the two writes are not atomic — a crash between them leaves the systems inconsistent. With CDC: the application only writes to the primary database. CDC reads the database's write-ahead log (WAL / binlog / oplog) and publishes change events to Kafka. All derived systems (Elasticsearch, cache, data warehouse) consume the change events and apply updates. The database is the single source of truth; all other systems are derived from it via CDC. Tools like Debezium implement CDC for PostgreSQL (WAL), MySQL (binlog), MongoDB (oplog), etc. The CDC pipeline preserves the exact order of changes as they occurred in the database. This enables reliable replication to derived systems without any application-level coordination.
5
What is the difference between event time and processing time, and why does it matter for windowed aggregations?
›
Event time: when the event actually occurred (the timestamp in the event data). Processing time: when the event arrives at and is processed by the stream processor. The two differ because of: network latency, consumer lag, offline devices sending batched events, out-of-order delivery. Why it matters for windows: if you're computing "clicks per minute" and you use processing time windows, a mobile user who was offline for 2 hours and reconnects will dump 120 minutes of events in 1 processing-time minute — badly distorting the count for that minute. Using event time, those clicks are attributed to their actual minutes. The challenge with event-time windows: you don't know when all events for a given event-time window have arrived. You need watermarks — heuristic assertions that "all events before time T have been received" — to decide when to close a window. A loose watermark (wait a long time) → more accurate results but higher latency. A tight watermark → lower latency but more late events (missed from the window). Late events can trigger window corrections in systems like Flink.
6
How does Flink achieve fault tolerance while maintaining low latency?
›
Flink uses distributed snapshots (Chandy-Lamport algorithm) to achieve fault tolerance without microbatching. Periodically, the job coordinator inserts barrier messages into all input streams. Barriers flow through the DAG of operators like regular events. When an operator receives barriers from all its input partitions, it: (1) snapshots its local state to durable storage (typically S3 or HDFS); (2) forwards the barrier to its downstream operators. Once all operators have checkpointed, the checkpoint is complete. On failure: recover the latest checkpoint (all operators restart from their checkpointed state) and replay input events from the position recorded at checkpoint time (Kafka offsets are part of the checkpoint). This is true event-at-a-time processing between checkpoints — no batching, so latency is milliseconds. The checkpoint cost is the state size (written to durable storage). The recovery cost is: time to restore state + time to reprocess events since last checkpoint. Checkpoint interval controls the trade-off: frequent checkpoints → less reprocessing on failure but more overhead during normal operation.
