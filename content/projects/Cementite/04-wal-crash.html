Section 01

Why WAL?

You have data in memory (MemTable). Writing it to disk is slow — and if the process crashes halfway, you get corrupted data. The WAL solves this by writing intent first, then acting second.

The Core Problem
Writing to a complex data structure on disk (SSTable, B-Tree page) takes time. If your process crashes HALFWAY through, you have corrupted or partially-written data. The WAL solves this by writing intent first, then acting second.

Without WAL: Total Data Loss

// What can go wrong during a MemTable flush (no WAL)
Timeline:
  T=0:     MemTable: { alice→30, bob→25, carol→28, ... 10,000 keys }
  T=1ms:   Begin writing SSTable to disk...
           [DATA BLOCK 0: alice→30, bob→25] ← written
           [DATA BLOCK 1: carol→28, dave→31] ← written
           [DATA BLOCK 2: ... writing ...
  T=2ms:   ⚡ POWER CUT / SEGFAULT / OOM KILLER ⚡

  RESULT ON DISK:
  ┌──────────────────────────────────────────────────┐
  │ SSTable (PARTIAL/CORRUPT FILE)                    │
  │ [DATA BLOCK 0] ← valid                           │
  │ [DATA BLOCK 1] ← valid                           │
  │ [DATA BLOCK 2] ← TRUNCATED halfway through       │
  │ [no index block, no bloom filter, no footer]     │
  └──────────────────────────────────────────────────┘

  On restart:
  - The SSTable file is corrupted and unreadable
  - The MemTable is GONE (it was in RAM)
  - 10,000 user writes are LOST FOREVER
  - You cannot even tell WHICH writes succeeded vs failed

With WAL: Atomic + Recoverable

// WAL makes individual writes atomic and recoverable
Fundamental principle: APPEND to a sequential log BEFORE touching any data structure.

Modified timeline with WAL:
  T=0:    User: set("alice", "30")

  T=1μs:  Step 1: WRITE to WAL (append-only, sequential, fast)
          wal.log  ←  [SET alice 30 seq=1042 crc=A3F2]  ← appended

  T=2μs:  Step 2: fsync() WAL file  ← DATA IS NOW PHYSICALLY ON DISK
          (Typically 0.1ms on SSD, 10ms on HDD)

  T=3μs:  Step 3: Acknowledge to user ✓  "write succeeded"

  T=4μs:  Step 4: Update MemTable in-memory (no disk I/O)

  ⚡ CRASH AT ANY POINT AFTER T=2μs ⚡

  On restart: Open WAL file. Replay all valid records. Stop at bad CRC.
  Result: ALL fsync'd writes recovered. Partial records discarded safely.

KEY INSIGHT: WAL is sequential (fast appends). ONE fsync per write (or per batch).
SSTable flush can be lazy WITHOUT fsync — always recoverable from WAL.

WAL Segment Lifecycle

// When can we delete a WAL segment?
A WAL entry is only needed until its data is safely in an SSTable on disk.

  WAL segment A:  [seq 1–5000 writes]
  WAL segment B:  [seq 5001–10000 writes]  ← active

  MemTable (in RAM): contains seq 1–10000

  Checkpoint event: MemTable flushed → SSTable written + fsync'd
  → WAL segment A is now REDUNDANT. Can be deleted.

  If crash NOW:
    - SSTable has seq 1–5000 safely
    - WAL segment B has seq 5001–10000
    → Replay WAL B → recover seq 5001–10000 → fully recovered!

  [CHECKPOINT: sstable_id=42 contains seq 1–5000. WAL segments 1–A can be deleted]

  Bitcask: the datafile IS the WAL. No separation needed.
  LSM:     WAL for durability, SSTable for queryable organized storage.
Section 02

WAL Format

Every WAL record has a fixed binary structure. Understanding it tells you exactly how crash detection works at the byte level.

Binary Record Format

// Individual WAL record — binary, variable length
┌────────────────────────────────────────────────────────────────────────────┐
│                         WAL Record Format                                    │
├──────────────┬────────────┬───────────┬──────────────┬──────────────────────┤
│ Record Length│  CRC32C    │   Type    │ Sequence No. │  Payload             │
│   4 bytes    │  4 bytes   │  1 byte   │   8 bytes    │  (variable)          │
├──────────────┴────────────┴───────────┴──────────────┴──────────────────────┤
│ Type (1 byte):                                                                │
│   0x05 = SET        simple key-value write                                  │
│   0x06 = DELETE     tombstone/deletion                                       │
│   0x07 = BATCH      grouped batch of writes (WriteBatch)                    │
│   0x08 = CHECKPOINT MemTable has been safely flushed to SSTable             │
│   0x01 = FULL       complete record fits in one WAL block                   │
│   0x02 = FIRST      first fragment of a large record                        │
│   0x04 = LAST       last fragment                                            │
│ CRC32C: checksum of [Type + SequenceNo + Payload]                            │
│         detects: truncated write, bit flip, partial record                  │
│ Sequence No: monotonically increasing, globally unique, used for MVCC       │
└───────────────────────────────────────────────────────────────────────────────┘

Example bytes for set("alice", "30"):
  [00 00 00 21]  ← record length = 33 bytes
  [A3 F2 01 CC]  ← CRC32C of [type + seq + payload]
  [05]           ← type: SET
  [00 00 00 00 00 00 04 12]  ← sequence number = 1042
  [00 00 00 05]  ← key_length = 5
  [00 00 00 02]  ← val_length = 2
  [61 6C 69 63 65]  ← "alice"
  [33 30]           ← "30"

Block-Based File Structure

// RocksDB-style 32KB block alignment
WAL File:
┌───────────────────────────────────────────────────────────────┐
│ BLOCK 0 (32768 bytes):                                         │
│  [ Record 1: FULL seq=1042 ] [ Record 2: FULL seq=1043 ]      │
│  [ Record 3: FIRST fragment seq=1044, too big for block... ]  │
├───────────────────────────────────────────────────────────────┤
│ BLOCK 1 (32768 bytes):                                         │
│  [ Record 3: LAST fragment ]  [ Record 4: FULL seq=1045 ]     │
└───────────────────────────────────────────────────────────────┘

Why 32KB blocks?
  → Aligned to disk sector size (512 or 4096 bytes)
  → A corrupt block can be isolated during recovery
  → Block boundaries are natural "restart points" during replay

Simpler alternative: record-per-record, no block alignment.
  Easier to implement. Start here for Phase 6.

WriteBatch: Atomic Multi-Key Ops

// Transfer 100 from alice to bob — must be atomic
WriteBatch WAL payload:
┌──────────────────────────────────────────────────────────────────────┐
│  BATCH_HEADER: sequence_number: u64  count: u32                       │
│  OPERATION 0: SET  key="account:alice"  val="balance=900"             │
│  OPERATION 1: SET  key="account:bob"    val="balance=1100"            │
│  [CRC32C of entire batch]                                              │
└──────────────────────────────────────────────────────────────────────┘

On recovery:
  CRC valid AND all ops decoded → apply all operations.
  CRC invalid OR truncated      → discard entire batch.
  → "transfer 100 from alice to bob" is atomic. Can't partially apply.
Section 03

Durability Ladder

When you return Ok(()) from a write, have you actually guaranteed the data is safe? This is one of the most important architectural decisions — it directly determines your throughput ceiling and crash safety guarantees.

Level 0 · No fsync (Fire and Forget)
Speed: ★★★★★ (1M+ writes/sec)
Safety: ☆☆☆☆☆ (data loss on crash)
write() copies data into kernel page cache. OS decides when to flush (up to 30 seconds). Power cut = everything in cache is gone. Why Redis has periodic snapshots.
rust
file.write_all(&buf)?;
// NO sync — OS may buffer for 30s before touching disk
Used by: Redis (default), time-series caches, non-critical logs
Level 1 · fdatasync per write
Speed: ★★☆☆☆ (~5-10K/sec SSD, ~100/sec HDD)
Safety: ★★★★★ (no data loss)
fdatasync() blocks until device confirms data is on stable storage. Skips metadata (mtime) — slightly faster than fsync. NVMe: ~0.05–0.1ms per call. HDD: ~10ms (one rotation). Classic throughput bottleneck.
rust
file.write_all(&buf)?;
file.sync_data()?;  // fdatasync — skips metadata update
Used by: PostgreSQL (synchronous_commit=on), etcd, ZooKeeper
Level 2 · fsync per write
Speed: ★★☆☆☆ (same as fdatasync, slightly slower)
Safety: ★★★★★ (data + metadata durable)
fsync() also flushes inode metadata (file size, mtime). For databases, file size matters — a partial write could leave unexpected file size. But for WAL records with CRC, fdatasync is usually sufficient.
rust
file.write_all(&buf)?;
file.sync_all()?;  // fsync — includes inode metadata
Used by: SQLite WAL mode, careful embedded databases
Level 3 · Group Commit ★ Best Balance
Speed: ★★★★☆ (100K–500K writes/sec NVMe)
Safety: ★★★★★ (configurable window)
Collect N writes → ONE fsync for all N → acknowledge all N callers. This is the key insight behind most production databases. Amortizes the fsync cost across many writers. MySQL calls this "group commit". Typical window: 1–10ms.
rust // group commit
let batch: Vec<(Waker, WALRecord)> = drain_queue();
for record in &batch { wal.append(record)?; }
wal.file.sync_data()?;        // ONE fsync for the entire batch
for (waker, _) in batch { waker.wake(); }  // notify all writers
Used by: MySQL InnoDB, PostgreSQL, RocksDB, TiKV — most production databases
Level 4 · O_DIRECT + O_SYNC
Speed: ★★★☆☆ (raw device speed)
Safety: ★★★★★ (strongest guarantee)
O_DIRECT bypasses OS page cache — buffers DMA'd directly to storage controller. Predictable latency (no cache eviction surprises) but you must manage your own read caching. Must use 512-byte-aligned buffers.
rust // Linux only
use std::os::unix::fs::OpenOptionsExt;
let file = OpenOptions::new()
    .write(true).create(true)
    .custom_flags(libc::O_DIRECT | libc::O_SYNC)
    .open("wal.log")?;
// Buffers must be 512-byte aligned (use posix_memalign)
Used by: Oracle, specialized high-performance databases

Group Commit Impact

// Same hardware, same durability, 50× throughput
Storage Type          │ fsync latency  │ Throughput (per-write fsync)
──────────────────────┼────────────────┼──────────────────────────────
HDD (7200 RPM)        │ ~8–12ms        │ ~80–125 writes/sec
SATA SSD (consumer)   │ ~0.5–2ms       │ ~500–2,000 writes/sec
NVMe SSD (consumer)   │ ~0.05–0.2ms   │ ~5,000–20,000 writes/sec
NVMe (datacenter)     │ ~0.01–0.05ms  │ ~20,000–100,000 writes/sec

With GROUP COMMIT (batch 50 writes per fsync):
NVMe consumer   → ~250K–1M writes/sec
NVMe datacenter → ~1M–5M writes/sec

Without group commit: NVMe gives you 20K writes/sec
With 50-wide group commit: same NVMe gives you 1M writes/sec
Same hardware. Same durability guarantees. 50× throughput.
Section 04

Crash Recovery

Understanding what types of crashes exist and what the WAL can and cannot protect against is essential before implementing recovery.

Taxonomy of Crashes

// What can go wrong, and what the WAL handles
TYPE 1: Process crash (OOM killer, SIGSEGV, panic)
  State: kernel + disk are fine. Only process memory is lost.
  Recovery: Easy — re-open WAL, replay from last checkpoint.
  Risk: None if WAL was fsync'd before acknowledging writes.

TYPE 2: OS crash (kernel panic, reboot)
  State: OS page cache is lost. Unflushed writes may be gone.
  Recovery: WAL replay, but only fsync'd records survive.
  Risk: Writes acknowledged after write() but before fsync() LOST.

TYPE 3: Power failure (hardware)
  State: Page cache lost. In-flight disk writes may be corrupt.
  Risk: "Torn write" — record partially written at device level.
  Recovery: WAL records have CRC32C. Bad CRC = discard, stop replay.

TYPE 4: Storage media failure (bad sectors, SSD wear-out)
  Recovery: CRC32C detects corruption. Return errors for affected keys.
  Production: checksums at every layer (WAL, SSTable, data block)

TYPE 5: Filesystem bugs / bit rot
  Recovery: CRC32C detects silent corruption (bit flips in storage)
  Some DBs use SHA-256 instead of CRC32 for stronger guarantees

Recovery Algorithm Step by Step

// On startup: how to bring the database back to a consistent state
DATABASE STARTUP / RECOVERY PROCEDURE

Step 1: Find the last valid checkpoint
  Read MANIFEST file:
  "SSTables {sst-42, sst-43} contain all data up to seq 50000"
  These SSTables are on disk and fsync'd. Trusted.

Step 2: Open WAL files written after the checkpoint
  Find WAL files with sequence numbers > 50000

Step 3: Replay WAL records one by one

  for record in wal.records_from(seq=50001) {
      match verify_crc(&record) {
          Ok(()) => {
              match record.record_type {
                  RecordType::Set    => memtable.insert(key, value, seq),
                  RecordType::Delete => memtable.delete(key, seq),
                  RecordType::Batch  => { for op in batch_ops { apply(op); } }
              }
          }
          Err(CrcMismatch) => {
              // TORN WRITE — everything after this is also invalid
              eprintln!("[WAL] Torn write at seq={}, stopping replay", seq);
              break;  // ← STOP. We've reached the recovery frontier.
          }
      }
  }

Step 4: MemTable + SSTables = consistent state
  ALL writes that received Ok() are present (they were fsync'd).
  Writes in progress during crash are lost — that's the durability contract.

Step 5: Write new checkpoint, truncate old WAL

Torn Write Detection with CRC32C

// Why every WAL record MUST have a checksum
TORN WRITE SCENARIO:

  Writing a 200-byte WAL record. Storage device writes in 512-byte sectors.

  Before crash: record being DMA'd to disk:
    Sector N:   [length=200, CRC=0xABCD, type=SET, seq=1042, key_len=5]
    ← POWER CUT HERE, mid-sector write ←
    Sector N+1: [NOT WRITTEN — key and value are missing]

  What's on disk: [length=200, CRC=0xABCD, type=SET, seq=1042, 00 00 00 00...]
  Looks like a complete record, but key+value are all zeros.

  WITHOUT CRC: Read 200 bytes → key = "\x00\x00\x00\x00\x00" (garbage)
    → Corrupt data inserted into MemTable! SILENT CORRUPTION!

  WITH CRC32C:
    Read header: CRC=0xABCD
    Read payload: [all zeros...]
    Compute CRC of payload: 0x0000
    Compare: 0xABCD ≠ 0x0000 → CRC MISMATCH
    → Stop replay. Don't apply garbage data.
    → Database is consistent up to the previous valid record. ✓

  CRC32C has hardware acceleration:
    x86: CRCQ instruction (1 cycle per 8 bytes)
    ARM: CRC32CB/CRC32CW instructions
    Rust: crc32c crate uses these intrinsics automatically
Section 05

Rust Implementation

Complete WalWriter (append path) and WalReader (iterator for crash recovery). The WalReader stops cleanly at torn writes.

rust // src/wal/mod.rs
use std::fs::{File, OpenOptions};
use std::io::{BufReader, BufWriter, Read, Write};
use std::path::Path;
use crc32fast::Hasher;

#[repr(u8)]
#[derive(Debug, Clone, Copy, PartialEq)]
pub enum RecordType {
    Set = 0x01, Delete = 0x02, Batch = 0x03, Checkpoint = 0x04,
}

#[derive(Debug)]
pub struct WalRecord {
    pub record_type: RecordType,
    pub sequence:    u64,
    pub key:         Vec<u8>,
    pub value:       Option<Vec<u8>>,  // None for Delete
}

#[derive(Clone, Copy)]
pub enum SyncMode {
    None,        // OS decides (risky, fast)
    PerWrite,    // fdatasync after every write (safe, slow)
    GroupCommit, // caller calls flush_and_sync() explicitly
}

pub struct WalWriter {
    writer:    BufWriter<File>,
    sequence:  u64,
    sync_mode: SyncMode,
}

impl WalWriter {
    pub fn open(path: impl AsRef<Path>, mode: SyncMode) -> std::io::Result<Self> {
        let file = OpenOptions::new().create(true).append(true).open(path.as_ref())?;
        let start_seq = scan_last_sequence(path.as_ref()).unwrap_or(0);
        Ok(Self {
            writer: BufWriter::with_capacity(64 * 1024, file), // 64KB write buffer
            sequence: start_seq + 1,
            sync_mode: mode,
        })
    }

    pub fn append_set(&mut self, key: &[u8], value: &[u8]) -> std::io::Result<u64> {
        let seq = self.next_seq();
        self.write_record(RecordType::Set, seq, key, Some(value))?;
        Ok(seq)
    }

    pub fn append_delete(&mut self, key: &[u8]) -> std::io::Result<u64> {
        let seq = self.next_seq();
        self.write_record(RecordType::Delete, seq, key, None)?;
        Ok(seq)
    }

    fn write_record(&mut self, rtype: RecordType, seq: u64,
                    key: &[u8], value: Option<&[u8]>) -> std::io::Result<()> {
        // Build payload: [type(1)|seq(8)|key_len(4)|val_len(4)|key|val]
        let val_len = value.map(|v| v.len()).unwrap_or(0) as u32;
        let mut payload = Vec::with_capacity(17 + key.len() + val_len as usize);
        payload.push(rtype as u8);
        payload.extend_from_slice(&seq.to_le_bytes());
        payload.extend_from_slice(&(key.len() as u32).to_le_bytes());
        payload.extend_from_slice(&val_len.to_le_bytes());
        payload.extend_from_slice(key);
        if let Some(v) = value { payload.extend_from_slice(v); }

        // CRC32C of payload
        let mut h = Hasher::new();
        h.update(&payload);
        let crc = h.finalize();

        // Write: [record_len(4) | crc(4) | payload]
        let record_len = (8 + payload.len()) as u32;
        self.writer.write_all(&record_len.to_le_bytes())?;
        self.writer.write_all(&crc.to_le_bytes())?;
        self.writer.write_all(&payload)?;

        if let SyncMode::PerWrite = self.sync_mode {
            self.writer.flush()?;
            self.writer.get_ref().sync_data()?; // fdatasync
        }
        Ok(())
    }

    pub fn flush_and_sync(&mut self) -> std::io::Result<()> {
        self.writer.flush()?;
        self.writer.get_ref().sync_data()
    }

    fn next_seq(&mut self) -> u64 { let s = self.sequence; self.sequence += 1; s }
}

// ─── Read / Recovery Path ──────────────────────────────────────────────────
pub struct WalReader { reader: BufReader<File> }

impl WalReader {
    pub fn open(path: impl AsRef<Path>) -> std::io::Result<Self> {
        Ok(Self { reader: BufReader::new(File::open(path)?) })
    }
}

impl Iterator for WalReader {
    type Item = std::io::Result<WalRecord>;

    fn next(&mut self) -> Option<Self::Item> {
        let mut len_buf = [0u8; 4];
        match self.reader.read_exact(&mut len_buf) {
            Ok(()) => {}
            Err(e) if e.kind() == std::io::ErrorKind::UnexpectedEof => return None,
            Err(e) => return Some(Err(e)),
        }
        let record_len = u32::from_le_bytes(len_buf) as usize;

        let mut buf = vec![0u8; record_len];
        if let Err(e) = self.reader.read_exact(&mut buf) {
            return Some(Err(e)); // truncated — likely torn write
        }

        let crc_actual = u32::from_le_bytes(buf[..4].try_into().unwrap());
        let payload = &buf[4..];

        let mut h = Hasher::new();
        h.update(payload);
        if crc_actual != h.finalize() {
            return Some(Err(std::io::Error::new(
                std::io::ErrorKind::InvalidData, "WAL CRC mismatch: torn write detected",
            )));
        }

        let seq     = u64::from_le_bytes(payload[1..9].try_into().unwrap());
        let key_len = u32::from_le_bytes(payload[9..13].try_into().unwrap()) as usize;
        let val_len = u32::from_le_bytes(payload[13..17].try_into().unwrap()) as usize;
        let key     = payload[17..17+key_len].to_vec();
        let value   = if val_len > 0 { Some(payload[17+key_len..17+key_len+val_len].to_vec()) } else { None };

        Some(Ok(WalRecord {
            record_type: if payload[0] == 0x01 { RecordType::Set } else { RecordType::Delete },
            sequence: seq, key, value,
        }))
    }
}
rust // src/db.rs — recovery function called on startup
pub fn recover_from_wal(wal_path: &Path, memtable: &mut MemTable) -> std::io::Result<u64> {
    let reader = WalReader::open(wal_path)?;
    let mut last_seq = 0u64;
    let mut count = 0usize;

    for result in reader {
        match result {
            Ok(record) => {
                match record.record_type {
                    RecordType::Set    => memtable.insert(record.key, record.value.unwrap(), record.sequence),
                    RecordType::Delete => memtable.delete(record.key, record.sequence),
                    _ => {}
                }
                last_seq = record.sequence;
                count += 1;
            }
            Err(e) if e.kind() == std::io::ErrorKind::InvalidData => {
                // Torn write — everything before this is safe, stop here
                eprintln!("[WAL] Torn write detected. {} records replayed.", count);
                break;
            }
            Err(e) => return Err(e),
        }
    }

    eprintln!("[WAL] Recovery complete. {} records replayed. Last seq: {}", count, last_seq);
    Ok(last_seq)
}
