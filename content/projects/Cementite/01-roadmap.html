Phase 01
In-Memory HashMap with Persistence
The Simplest Possible DB

Phase 02
Append-Only Log (Bitcask Phase 1)
Your first durable storage
2–3 days Model: Log-structured Hash Table (Bitcask)
Goal

Make data survive crashes. Understand sequential writes and why they're fast.

Key Insight

Sequential writes are 10–100× faster than random writes on both HDDs and SSDs. By ONLY appending, you get maximum write throughput. But you pay in space (old values stay on disk).

What You Build
  • An append-only file: writes go to the END of a file, never overwrite
  • Each entry: [CRC(4B) | timestamp(8B) | key_size(4B) | val_size(4B) | key | value]
  • In-memory KeyDir: HashMap<String, (file_id, offset, val_size)>
  • On startup: scan log file to rebuild KeyDir
  • Tombstone records for deletions
Real DBs
Bitcask (Riak) Early Redis AOF Apache Kafka (conceptually)
Pros
  • Survives crashes
  • Sequential writes = fast
  • Simple recovery (just re-scan)
Cons
  • Space grows unbounded
  • Startup slow if log is huge
  • All keys must fit in RAM
Amplification
Read
1 disk seek O(1)
Write
1 disk write
Space
Grows until compaction
Rust Notes
Use std::fs::OpenOptions with .append(true)
BufWriter<File> for write batching
Seek to offset with file.seek(SeekFrom::Start(offset))
crc32fast crate for checksums
Phase 03
Compaction + Hint Files (Bitcask Phase 2)
Space reclamation & fast startup
3–4 days Model: Log-structured Hash Table (Bitcask complete)
Goal

Reclaim disk space by merging old log segments. Speed up startup with hint files.

Key Insight

This is the first time you see the fundamental tension in all DBs: you trade CPU/IO work (compaction) for space efficiency. The merge process is essentially a MapReduce over your log.

What You Build
  • Segment rotation: when active file > 64MB, close it (immutable), open new one
  • Merge process: scan all immutable segments, keep only latest value per key
  • Write merged result as new clean segment files
  • Hint files: compact index-only files (no values) for fast KeyDir rebuild on startup
  • Background compaction thread
Real DBs
Bitcask (complete) Riak KV
Pros
  • Bounded disk usage
  • Fast startup via hint files
  • Background compaction doesn't block reads
Cons
  • Compaction uses temporary 2× space
  • Still all keys in RAM
  • Not great for range scans
Amplification
Read
1 disk seek
Write
~1–2× (compaction rewrites)
Space
~1–2× during merge
Rust Notes
Use std::sync::Arc<RwLock<KeyDir>> for concurrent reads during compaction
Spawn compaction as tokio::task or std::thread
Hint file format: [key_size(4B) | file_id(4B) | offset(8B) | key]
Phase 04
LSM-Tree: MemTable + SSTable
How RocksDB & Cassandra work
1–2 weeks Model: Log-Structured Merge Tree (LSM)
Goal

Understand the most dominant storage model in modern databases. Keys no longer need to fit in RAM.

Key Insight

Writes are ALWAYS sequential (MemTable → WAL → SSTable). Reads are slower because you might need to check multiple SSTables. You've traded read performance for write performance. This is the RUM conjecture in action.

What You Build
  • MemTable: in-memory BTreeMap (sorted!), absorbs all writes
  • WAL: Write-Ahead Log, every write goes here first for durability
  • SSTable flush: when MemTable hits ~4MB, sort & flush to disk as immutable SSTable file
  • SSTable format: [data blocks | index block | bloom filter | footer]
  • Read path: MemTable → L0 SSTables (newest→oldest) → L1 → ...
  • Simple single-level compaction (merge sort two SSTables)
Real DBs
RocksDB (Facebook) LevelDB (Google) Cassandra (Apache) TiKV (PingCAP) ScyllaDB
Pros
  • Keys don't need to fit in RAM
  • Excellent write throughput
  • Range scans work well
Cons
  • Reads may touch multiple files (read amplification)
  • Background compaction I/O
  • Complexity jump
Amplification
Read
Up to O(L) files
Write
~10–30× leveled
Space
~1.1× leveled
Rust Notes
MemTable: std::collections::BTreeMap<Vec<u8>, Entry>
Entry enum: Value(Vec<u8>) | Tombstone
SSTable: write sorted key-value pairs with a block index at the end
Use memmap2 crate for mmap-based SSTable reads
Phase 05
Bloom Filters + Multi-Level Compaction
Making reads fast again
1 week Model: LSM-Tree (production-grade)
Goal

Add Bloom filters to avoid unnecessary disk reads. Implement proper leveled compaction.

Key Insight

Bloom filters are the single biggest read optimization in LSM systems. A 10-bit Bloom filter has only ~1% false positive rate, meaning 99% of 'key not found' lookups avoid ANY disk I/O. Leveled compaction ensures at most 1 SSTable per key range per level (except L0).

What You Build
  • Bloom filter per SSTable: O(1) probabilistic check — 'is this key definitely NOT here?'
  • Leveled compaction: L0 (overlap OK), L1–L6 (no key overlap within level)
  • L0→L1: when L0 has 4+ files, merge into L1
  • Each level 10× larger than previous: L1=10MB, L2=100MB, L3=1GB…
  • Compaction picks SSTable from Ln, merges with overlapping SSTables in Ln+1
  • Manifest file: tracks which SSTables exist at which level
Real DBs
LevelDB (leveled) RocksDB (leveled + tiered) Cassandra (tiered)
Pros
  • Near-zero read amplification for point lookups
  • Predictable space usage
  • Good write throughput
Cons
  • Leveled compaction = higher write amplification than tiered
  • Complex compaction scheduling
Amplification
Read
~1–2 disk reads w/ Bloom
Write
10–30× (leveled)
Space
~1.1×
Rust Notes
Use bloomfilter crate or implement your own (2 hash functions + bit array)
Store serialized Bloom filter in SSTable footer
Manifest: append-only log of SSTable add/remove events
Phase 06
WAL + Crash Recovery
Production-grade durability
3–5 days Model: WAL + LSM
Goal

Handle crashes safely. Understand the difference between fsync, fdatasync, and buffered writes.

Key Insight

There are only 3 durability guarantees you can offer: (1) data persisted on write (fsync every write — slow), (2) data persisted per batch (group commit — good balance), (3) data possibly lost on crash (async — fast but dangerous). Every real DB makes one of these choices.

What You Build
  • WAL file: append-only log of all operations before they touch MemTable
  • WAL record: [length(4B) | type(1B) | key | value | CRC(4B)]
  • On crash: replay WAL to reconstruct in-flight MemTable
  • fsync vs fdatasync: understand when data actually hits disk
  • WAL rotation: archive WAL when MemTable is fully flushed to SSTable
  • Checkpoints: record 'all data up to sequence N is durable in SSTables'
Real DBs
PostgreSQL WAL RocksDB WAL etcd (Raft log)
Pros
  • Crash safety
  • Tunable durability (sync every write vs batch)
  • Point-in-time recovery possible
Cons
  • fsync is slow (~1–10ms on SSD)
  • WAL takes extra disk space
  • Recovery time proportional to WAL size
Amplification
Read
No impact
Write
+1 sequential write per op
Space
Temporary WAL files
Rust Notes
file.sync_data() = fdatasync (faster, skips metadata)
file.sync_all() = fsync (syncs metadata too)
Group commit: collect N writes, fsync once, acknowledge all
Phase 07
Concurrency + MVCC
Multiple readers, safe writers
1–2 weeks Model: MVCC + LSM
Goal

Allow concurrent reads without blocking writes. Understand MVCC from first principles.

Key Insight

MVCC is the reason modern DBs can have 'readers don't block writers and writers don't block readers.' Each reader has a consistent point-in-time view of the data. The cost: you store multiple versions of each key, needing periodic GC.

What You Build
  • Sequence numbers: every write gets a monotonically increasing seq_no
  • Keys internally become (user_key, seq_no) tuples
  • Snapshots: a reader takes a snapshot = captures current seq_no
  • Reader only sees keys with seq_no ≤ snapshot's seq_no
  • Writer appends new (key, new_seq_no, value) without blocking readers
  • Garbage collection: remove old versions when no snapshot needs them
Real DBs
RocksDB (sequence numbers) TiKV (MVCC) FoundationDB
Pros
  • Non-blocking reads
  • Consistent snapshots
  • Foundation for transactions
Cons
  • Version accumulation needs GC
  • More complex read path
  • Higher memory usage
Amplification
Read
Same + version filtering
Write
Same + seq_no overhead
Space
~2–3× until GC
Rust Notes
Use AtomicU64 for sequence number generation
Arc<SnapManager> to track live snapshots
parking_lot::RwLock for MemTable (many readers, one writer)
crossbeam::epoch for lock-free reads if pushing for performance
Phase 08
Benchmarking Infrastructure
Measure everything
Ongoing Model: All of the above
Goal

Build a rigorous benchmark harness. Know exactly how fast your DB is and WHY.

Key Insight

A benchmark without a controlled workload is meaningless. Real systems use Zipfian key distributions (80% of requests hit 20% of keys) not uniform random. p99 latency matters more than average — your worst 1% of requests determine user experience.

What You Build
  • Workload generator: YCSB-style (100% read, 100% write, 50/50, Zipfian distribution)
  • Metrics: throughput (ops/sec), p50/p95/p99/p999 latency
  • Write amplification counter: bytes written to disk / bytes written by user
  • Read amplification counter: disk reads per user read
  • Space amplification: actual disk usage / logical data size
  • Comparison suite: your DB vs sled vs RocksDB (via rust-rocksdb crate)
Real DBs / Benchmarks
YCSB benchmarks RocksDB db_bench LevelDB benchmarks
Pros
  • Know where time is spent
  • Compare against real DBs
  • Catch regressions
Cons
  • Easy to benchmark the wrong thing
  • Microbenchmarks can mislead
