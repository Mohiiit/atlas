Model 01
Bitcask
Log-structured Hash Table
What it is
Bitcask is a log-structured hash table designed by the Riak team. ALL writes go sequentially to an append-only file. An in-memory hash table maps every key to its exact byte offset on disk. Guarantees exactly 1 disk seek per read.
Record Format (Byte Level)
BITCASK RECORD FORMAT
+------------------------------------------------------------------+
|                    BITCASK RECORD FORMAT                          |
+----------+-----------+----------+-----------+--------+-----------+
|  CRC32   | Timestamp | Key Size | Val Size  |  Key   |   Value   |
|  4 bytes |  8 bytes  |  4 bytes |  4 bytes  |  var.  |   var.    |
+----------+-----------+----------+-----------+--------+-----------+
| Total header = 20 bytes                                           |
| CRC32: checksum of [timestamp + key_sz + val_sz + key + value]   |
| Timestamp: Unix nanoseconds, used for conflict resolution         |
| Val Size = 0xFFFFFFFF -> TOMBSTONE (key deleted)                 |
|                                                                   |
| Example: set("user:42", "alice") at offset 0                     |
|   [A3 F2 01 CC]              <- CRC32                            |
|   [00 00 01 93 A4 5E 20 00]  <- timestamp (nanoseconds)         |
|   [00 00 00 07]              <- key_size = 7                     |
|   [00 00 00 05]              <- val_size = 5                     |
|   [75 73 65 72 3A 34 32]     <- "user:42"                       |
|   [61 6C 69 63 65]           <- "alice"                          |
|   Total = 20 + 7 + 5 = 32 bytes                                  |
+------------------------------------------------------------------+
KeyDir (In-Memory Index)
KeyDir HashMap Structure
KeyDir: HashMap<String, KeyDirEntry>
+-----------------+----------+------------+-----------+------------+
|      Key        |  file_id | val_offset |  val_size |  timestamp |
+-----------------+----------+------------+-----------+------------+
| "user:42"       |    3     |   0x00A4   |     5     | 1704067200 |
| "user:99"       |    3     |   0x00C8   |    12     | 1704067201 |
| "config:theme"  |    1     |   0x1F40   |     4     | 1704060000 |
+-----------------+----------+------------+-----------+------------+

Rust struct:
  struct KeyDirEntry {
      file_id:    u32,   // which segment file contains the value
      val_offset: u64,   // byte offset to seek to in that file
      val_size:   u32,   // bytes to read starting at val_offset
      timestamp:  u64,   // for compaction conflict resolution
  }
Read & Write Paths
Read Path
get("user:42"):
  1. Lookup "user:42" in KeyDir -> (file_id=3, offset=0x00A4, size=5)
  2. Open segment file 3 (or use cached file handle)
  3. Seek to byte offset 0x00A4
  4. Read 5 bytes -> "alice"
  5. Verify CRC32 (detect corruption)
  6. Return "alice"

ONE disk seek guaranteed. Always. Regardless of dataset size.
Write Path
set("user:42", "bob"):
  1. Build record: [CRC32 | ts | 7 | 3 | "user:42" | "bob"]
  2. Append to active segment file (sequential write = fast)
  3. Update KeyDir: "user:42" -> (file_id=3, new_offset, size=3)
  4. Return OK

Old value "alice" is still on disk at old offset.
KeyDir now points to new offset. Old entry is orphaned garbage.
Compaction will clean it up later.
Compaction & Hint Files
Compaction Process
Compaction trigger: active segment > 64MB, OR total disk usage > threshold

Process:
  1. Collect all immutable segment files (not the active one)
  2. For each key in KeyDir that points to an immutable segment:
       Read current value from disk
  3. Write to new merged segment file (only newest value per key)
  4. Write hint file alongside: [key_size(4B) | file_id(4B) | offset(8B) | key]
     (hint files contain NO values -- index only)
  5. Atomically update KeyDir to point to new segment
  6. Delete old segment files

Hint file benefit:
  Without hints: startup scans all data files to rebuild KeyDir
    -> 1M keys = 10+ seconds
  With hints:    read compact index-only hint files
    -> 1M keys = ~100ms  (100x faster startup)
Limitations

ALL keys must fit in RAM — the entire KeyDir lives in memory. No range scans (hash map has no ordering, you can't do "give me all keys from 'a' to 'b'"). Not suitable when keyspace exceeds available RAM. Not suitable for sorted iteration or prefix queries.

Model 02
LSM-Tree
Log-Structured Merge Tree
What it is
Never update data in place. Buffer all writes in a sorted in-memory structure (MemTable), flush to immutable sorted disk files (SSTables) when full, merge them in the background. Converts random writes into sequential writes. Keys do NOT need to fit in RAM.
MemTable + WAL Relationship
Write Flow: MemTable + WAL
Write: set("alice", "30")
  [1] WAL (Write-Ahead Log)                [2] MemTable (RAM)
  +----------------------------+           +---------------------+
  | SET alice 30 seq=1042      | <---+---> | BTreeMap (sorted):  |
  | SET bob   25 seq=1041      |     |     |  "alice" -> (30,    |
  | DEL carol    seq=1040      |     |     |             seq=1042)|
  +----------------------------+     |     |  "bob"   -> (25,    |
  | fsync() here = crash safe  |     |     |             seq=1041)|
  +----------------------------+     |     +---------------------+
                                     |     BTreeMap keeps keys sorted.
  User write MUST be in WAL          |     Sorted = fast range scans.
  before updating MemTable.          |     No random I/O on disk yet.

[Background] MemTable hits ~4MB:
  -> Freeze MemTable (immutable, keep accepting writes to new MemTable)
  -> Flush immutable MemTable -> SSTable file on disk (sorted)
  -> Advance WAL checkpoint (old WAL segment safe to delete)
  -> Add new SSTable to L0 in the manifest file
SSTable Format (Detailed)
SSTable Binary Layout
An SSTable is an IMMUTABLE, SORTED file. Written once, never modified.

+-----------------------------------------------------+
|  BLOCK 0 (4096 bytes)                               |
|  Entries sorted by key:                             |
|    [key_len:2 | val_len:4 | key_bytes | val_bytes]  |
|    "alice" -> "30"                                  |
|    "bob"   -> "25"                                  |
|    "carol" -> "28"                                  |
+-----------------------------------------------------+
|  BLOCK 1 (4096 bytes)                               |
|    "dave"  -> "31"                                  |
|    "eve"   -> "22"                                  |
+-----------------------------------------------------+
|  ...more data blocks...                             |
+-----------------------------------------------------+
|  INDEX BLOCK                                        |
|  First key of each data block -> block offset       |
|    "alice" -> offset 0                              |
|    "dave"  -> offset 4096                           |
+-----------------------------------------------------+
|  BLOOM FILTER                                       |
|  10 bits per key, ~1% false positive rate           |
|  Serialized bit array                               |
+-----------------------------------------------------+
|  FOOTER (fixed size = 48 bytes)                     |
|  index_block_offset:  u64                           |
|  bloom_filter_offset: u64                           |
|  num_entries:         u64                           |
|  min_key:             [16 bytes, truncated]         |
|  max_key:             [16 bytes, truncated]         |
|  magic_number:        0xCAFEBABEDEADBEEF            |
+-----------------------------------------------------+

To find key "carol":
  1. Read footer (last 48 bytes)
  2. Load index block -> "alice" block at offset 0
  3. Read 4096 byte data block
  4. Scan block for "carol"
  Total: 3 disk reads (or 2 if index is cached)
Level Structure
LSM Level Hierarchy
L0: up to 4 SSTables (direct MemTable flushes, keys CAN overlap)
    |
    v  [trigger: L0 has 4 files]
L1: <= 10MB total, ZERO key overlap within level
    |
    v  [trigger: L1 exceeds 10MB]
L2: <= 100MB
    |
    v
L3: <= 1GB
    |
    v
L4: <= 10GB
    |
    v
L5: <= 100GB
    |
    v
L6: <= 1TB (usually the bottom "stable" level)

L0 is special:
  - Written directly from MemTable flushes (no compaction step)
  - Multiple L0 files CAN overlap in key range
  - READ must check ALL L0 files (can't binary search by range)

L1 and beyond:
  - Compaction enforces: within any level, no two files share a key
  - READ: binary search manifest -> exactly 1 candidate file per level
  - Much faster reads than L0

Each level is 10x larger than the previous.
A key is rewritten once per level crossing (source of write amplification).
Read Amplification: With vs Without Bloom Filters
Read Path Comparison
WITHOUT Bloom filters, get("nonexistent"):
  L0-0: load index -> binary search -> load data block -> not found (2 reads)
  L0-1: load index -> binary search -> load data block -> not found (2 reads)
  L0-2: load index -> binary search -> load data block -> not found (2 reads)
  L0-3: load index -> binary search -> load data block -> not found (2 reads)
  L1:   load index -> binary search -> load data block -> not found (2 reads)
  L2:   load index -> binary search -> load data block -> not found (2 reads)
  ...
  Total: ~30+ disk reads to confirm 1 key doesn't exist!

WITH Bloom filters, get("nonexistent"):
  L0-0: read footer -> Bloom says NOPE!  (1 read, saved 1)
  L0-1: read footer -> Bloom says NOPE!  (1 read, saved 1)
  L0-2: read footer -> Bloom says NOPE!  (1 read, saved 1)
  L0-3: read footer -> Bloom says maybe -> check -> not found (3 reads)
  L1:   read footer -> Bloom says NOPE!  (1 read, saved 1)
  L2-L6: all NOPE                        (5 reads, saved 5)
  Total: ~9 reads (mostly footer reads, cached after first access)

In practice with caching: ~1-2 actual disk reads for most misses.
Model 03
B-Tree
Balanced Tree, In-Place Updates
What it is
A balanced tree where every node is a disk page (4096 bytes). Leaf nodes hold sorted key-value pairs. Internal nodes hold keys and child pointers. Updates happen IN-PLACE on disk. The OS page cache acts as a buffer, making reads from hot data extremely fast.
Node Structure
B-Tree Node Layout
Internal Node (4096 bytes, stored as one disk page):
  +----------------------------------------------------------+
  | Header: [node_type=1 | num_keys | free_space_offset]    |
  +----------------------------------------------------------+
  | Key pointers: ["bob", ptr_left] ["frank", ptr_right]    |
  |   ptr_left  -> child page with keys < "bob"             |
  |   ptr_right -> child page with "bob" <= keys < "frank"  |
  |   implicit  -> child page with keys >= "frank"          |
  +----------------------------------------------------------+
  Fanout (keys per node): ~100-1000 (depends on key size)

Leaf Node (4096 bytes):
  +----------------------------------------------------------+
  | Header: [node_type=0 | num_keys | next_leaf_ptr]        |
  +----------------------------------------------------------+
  | Sorted KV entries: [key_len:2 | val_len:4 | key | val]  |
  |   "alice" -> "30"                                       |
  |   "carol" -> "28"                                       |
  +----------------------------------------------------------+
  | Leaf chain: next_leaf_ptr points to next leaf page      |
  |   -> enables efficient range scans (walk the leaf chain)|
  +----------------------------------------------------------+

Tree height for 1 billion records, fanout=100: ~5 levels
Tree height for 1 billion records, fanout=500: ~4 levels
Tree Traversal Example
Read: Finding "dave"
READ get("dave"), tree height = 3:

Level 0 (Root, 1 disk read):
  Internal node: ["bob", "frank"]
  "dave" is between "bob" and "frank" -> follow middle child pointer

Level 1 (1 disk read):
  Internal node: ["carol", "eve"]
  "dave" is between "carol" and "eve" -> follow middle child pointer

Level 2 (Leaf, 1 disk read):
  Leaf node: [("carol","28"), ("dave","31"), ("eve","22")]
  Binary search within leaf -> found "dave" = "31"

Total: 3 disk reads. Predictable. Always O(log n).
With hot OS page cache: root and upper levels stay in RAM -> ~1 disk read.
Page Splits & Write Amplification
Write Path and Page Split
UPDATE set("dave", "35"):
  1. Traverse to leaf (3-4 reads, same as read)
  2. Modify leaf page IN-PLACE on disk: 1 random write
  If leaf is full (no room for new key):
    3. SPLIT leaf: allocate new page, move half keys to new page
    4. Update parent: add new key + pointer -> 1 random write to parent
    If parent is also full:
    5. SPLIT parent -> update grandparent... (cascade up to root)
  Write amp: 2-5x on average, spikes during cascading splits

Random I/O cost:
  HDD spinning disk: 5-10ms per random seek = terrible
  At 200 writes/sec sustained, disk is at ~100% IOPS utilization
  SSD (NVMe): ~100us per random write = much better
  But: SSDs wear out faster from random writes vs sequential
  LSM wins on SSDs too (sequential = fewer write cycles)

Copy-on-Write variant (LMDB, used by Lightning MDB):
  Instead of modifying a page: copy it, modify the copy, update parent
  Old page version stays valid -> readers see consistent state without locks
  Writer creates a new tree root atomically
  Readers never blocked. But: writes use ~2x disk space temporarily.
When B-Tree Wins
B-Tree Strengths
Predictable read latency:
  O(log n) reads, always. No compaction pauses, no stall.
  p99 latency is close to p50. Great for OLTP with SLAs.

Working-set-fits-in-cache scenario:
  Hot pages (root + upper levels) stay in OS page cache.
  Effective read latency approaches RAM speed (~us, not ms).
  RocksDB can't do this as well -- SSTables change frequently.

ACID transactions:
  B-Tree page locking is well-studied.
  Row-level locking, MVCC on top of B-Tree pages (PostgreSQL does this).
  Rollback: restore original page (undo log).
  LSM + MVCC is more complex to implement correctly.

SQLite, PostgreSQL, MySQL InnoDB, LMDB: all use B-Tree variants.
These are decades of production hardening.
Your Cementite Phase 4-7 is already a more modern design than SQLite!

Range scans None (hash = no order) Good (sorted SSTables) Good (leaf chain)

Space amplification ~2× during compact ~1.1× (leveled) ~1.3× (fragmentation)
Crash recovery Scan datafile → rebuild KeyDir Replay WAL Undo log or WAL

Predictable latency High Medium (compaction spikes) High

Best workload Small keyspace, high R+W Large data, write-heavy Read-heavy, OLTP
When to choose
Bitcask
  • Small, bounded keyspace
  • All keys fit in RAM
  • High throughput R+W mixed
  • No need for range scans
  • Simplest possible durability
riak, embedded KV
LSM-Tree
  • Large dataset, exceeds RAM
  • Write-heavy or mixed workload
  • Need range scans + point reads
  • Can tolerate compaction pauses
  • Modern cloud-native systems
rocksdb, cassandra, tikv
B-Tree
  • Read-heavy workloads
  • Need predictable latency
  • ACID transactions required
  • Working set fits in cache
  • General OLTP databases
postgresql, mysql, sqlite
