Section 01

Why Bloom Filters?

In an LSM-Tree, a key might exist in ANY SSTable across ANY level. Without a way to skip files quickly, every read checks many files. Bloom filters are the primary mechanism to avoid unnecessary disk I/O for non-existent keys.

The Read Amplification Problem
A 10-bit-per-key Bloom filter eliminates ~99% of unnecessary disk reads for missing keys. The RAM cost is tiny — 1.25 MB for an entire level of 1M keys. It's one of the highest-ROI optimizations in any LSM implementation.

Without Bloom Filters

// get('zara') — key does not exist anywhere
Database: 7 levels, 1M keys, none of them is "zara"

get("zara"):
  1. MemTable (RAM): not found
  2. Immutable MemTables (RAM): not found

  3. L0 has 4 SSTables (keys may OVERLAP across L0 files):
     → L0-0: read footer + index + maybe data block → NOT FOUND (2-3 disk reads)
     → L0-1: same → NOT FOUND (2-3 disk reads)
     → L0-2: same → NOT FOUND (2-3 disk reads)
     → L0-3: same → NOT FOUND (2-3 disk reads)

  4. L1 (no overlap, 1 file owns each key range):
     Manifest lookup → 1 candidate file
     → Open it: footer + index + data block → NOT FOUND (2-3 disk reads)

  5. Repeat for L2, L3, L4, L5, L6...

  TOTAL: ~30 disk reads just to confirm "zara" does not exist!

  At 10ms/read on HDD: 300ms to confirm one key miss
  At 0.1ms/read on NVMe: 3ms — still terrible at scale

With Bloom Filters

// Same query with Bloom filters loaded in RAM
Each SSTable stores a Bloom filter at its footer.
When an SSTable is opened, the Bloom filter is loaded into RAM.
Cost: 10 bits/key × 1M keys = 1.25 MB for an entire level.

get("zara"):
  1. MemTable: not found

  2. L0 (4 files):
     L0-0 Bloom (RAM): "DEFINITELY NOT here" → skip!  (0 disk reads)
     L0-1 Bloom (RAM): "DEFINITELY NOT here" → skip!  (0 disk reads)
     L0-2 Bloom (RAM): "DEFINITELY NOT here" → skip!  (0 disk reads)
     L0-3 Bloom (RAM): "MAYBE here" (false positive!)
       → do disk read → NOT FOUND  (2 reads, unavoidable for FP)

  3. L1 Bloom: "DEFINITELY NOT" → skip all files   (0 reads)
  4. L2-L6: all "DEFINITELY NOT" → skip            (0 reads)

  TOTAL: 2 disk reads (was 30) — 15× improvement!

KEY PROPERTY (guaranteed):
  "DEFINITELY NOT HERE" → always correct. Zero disk I/O. No false negatives.
  "MAYBE HERE"          → must check disk. Can be a false positive (~1% at 10 bits/key).
  Never claims "DEFINITELY HERE" — can only disprove, never prove presence.
Section 02

Bloom Math & Tuning

A Bloom filter's false positive rate is mathematically determined by the number of bits per key (b) and hash functions (k). Understanding the math lets you tune it precisely for your use case.

How a Bloom Filter Works

// Bit array + k hash functions — insertion and querying
A Bloom filter is:
  - A bit array of m bits  (e.g. m=10,000 for 1,000 keys at 10 bits/key)
  - k hash functions each mapping a key to a position in [0, m)

INSERTION: add("alice")
  hash1("alice") % 10000 = 2347 → bit[2347] = 1
  hash2("alice") % 10000 = 5821 → bit[5821] = 1
  hash3("alice") % 10000 = 8103 → bit[8103] = 1
  hash4("alice") % 10000 = 0441 → bit[0441] = 1

INSERTION: add("bob")
  hash1("bob") % 10000 = 1234 → bit[1234] = 1
  hash2("bob") % 10000 = 5821 → already 1 (hash collision, OK)
  hash3("bob") % 10000 = 7777 → bit[7777] = 1

QUERY: contains("carol")  [never inserted]
  hash1("carol") % 10000 = 2347 → 1  (alice's bit)
  hash2("carol") % 10000 = 9999 → 0  ← STOP
  → "DEFINITELY NOT PRESENT"  (bit 9999 was never set)
  → Skip this SSTable.  Zero disk I/O!

QUERY: contains("dave")  [never inserted, but FALSE POSITIVE]
  hash1("dave") % 10000 = 2347 → 1  (alice)
  hash2("dave") % 10000 = 5821 → 1  (alice+bob collision)
  hash3("dave") % 10000 = 7777 → 1  (bob)
  hash4("dave") % 10000 = 8103 → 1  (alice)
  → All k bits are 1 → "MAYBE PRESENT" ← false positive
  → We check disk → not found → 1 wasted read

NO FALSE NEGATIVES: if a key was inserted, all its k bits were set.
"DEFINITELY NOT PRESENT" is always correct.

Mathematical Tuning

// Optimal k and false positive rate by bits-per-key
FORMULAS:
  bits_per_key (b) = m / n
  Optimal k = b × ln(2)  ≈ 0.693 × b
  False positive rate p = (1 - e^(-k/b))^k

LOOKUP TABLE:

  b (bits/key)   k (optimal)   FP rate     RAM / 1M keys
  ────────────   ───────────   ───────     ─────────────
       4               3        14.7%        500 KB
       6               4         5.6%        750 KB
       8               6         2.1%      1,000 KB
      10               7         0.82%     1,250 KB  ← RocksDB default
      12               8         0.30%     1,500 KB
      14              10         0.11%     1,750 KB
      16              11         0.042%    2,000 KB
      20              14         0.006%    2,500 KB

SIZING EXAMPLE (SSTable with 100,000 keys at 10 bits/key):
  m = 100,000 × 10 = 1,000,000 bits = 125,000 bytes = ~122 KB
  k = 7 hash functions
  At 1M lookups/sec: 8,200 wasted disk reads/sec
  Raising to 12 bits/key: 3,000 wasted reads/sec (3× better, costs 50 KB more)

Two-Hash Trick (Kirsch-Mitzenmacher)
You do not need k truly independent hash functions. Compute just 2 hashes (h1, h2), then derive all k positions as g_i = (h1 + i × h2) mod m. Only 2 hash computations regardless of k. Used by RocksDB and most modern implementations.
Section 03

Bloom Filter in Rust

A complete, production-quality BloomFilter using double-hashing (Kirsch-Mitzenmacher). Includes serialization for SSTable footers and unit tests that verify the no-false-negatives guarantee.

rust // src/bloom.rs
use std::hash::{Hash, Hasher};
use std::collections::hash_map::DefaultHasher;

/// Bloom filter for a single SSTable.
/// Uses double-hashing (Kirsch-Mitzenmacher) — only 2 hash computations for k positions.
pub struct BloomFilter {
    bits:       Vec<u8>,  // bit array packed as bytes
    num_bits:   usize,    // m: total number of bits
    num_hashes: usize,    // k: hash positions per key
}

impl BloomFilter {
    /// Create for expected_keys at bits_per_key bits each.
    /// Optimal: k = bits_per_key × ln(2) ≈ 0.693 × bits_per_key
    pub fn new(expected_keys: usize, bits_per_key: usize) -> Self {
        let num_bits    = (expected_keys * bits_per_key).max(64).next_power_of_two();
        let num_hashes  = ((bits_per_key as f64 * std::f64::consts::LN_2) as usize).clamp(1, 30);
        Self { bits: vec![0u8; num_bits / 8 + 1], num_bits, num_hashes }
    }

    /// Reconstruct from bytes stored in SSTable footer.
    pub fn from_bytes(data: &[u8], num_hashes: usize) -> Self {
        Self { num_bits: data.len() * 8, bits: data.to_vec(), num_hashes }
    }

    pub fn insert(&mut self, key: &[u8]) {
        let (h1, h2) = double_hash(key);
        for i in 0..self.num_hashes {
            let pos = h1.wrapping_add((i as u64).wrapping_mul(h2)) as usize % self.num_bits;
            self.bits[pos / 8] |= 1 << (pos % 8);
        }
    }

    /// false → DEFINITELY NOT present (never wrong, no false negatives)
    /// true  → MAYBE present (check the actual SSTable on disk)
    pub fn may_contain(&self, key: &[u8]) -> bool {
        let (h1, h2) = double_hash(key);
        for i in 0..self.num_hashes {
            let pos = h1.wrapping_add((i as u64).wrapping_mul(h2)) as usize % self.num_bits;
            if self.bits[pos / 8] & (1 << (pos % 8)) == 0 {
                return false; // bit is 0 → key definitely not inserted
            }
        }
        true // all bits set → probably inserted (or false positive)
    }

    pub fn to_bytes(&self)   -> &[u8] { &self.bits }
    pub fn num_hashes(&self) -> usize  { self.num_hashes }
}

fn double_hash(key: &[u8]) -> (u64, u64) {
    let h1 = { let mut h = DefaultHasher::new(); key.hash(&mut h); 0u64.hash(&mut h); h.finish() };
    let h2 = { let mut h = DefaultHasher::new(); key.hash(&mut h); 1u64.hash(&mut h); h.finish() };
    (h1, h2 | 1)  // h2 must be odd so all positions are reachable
}

        let keys: Vec<Vec<u8>> = (0..1000u32).map(|i| i.to_le_bytes().to_vec()).collect();
        for k in &keys { bloom.insert(k); }

Section 04

Why MVCC?

When multiple threads access your database simultaneously, naive locking kills throughput. MVCC (Multi-Version Concurrency Control) keeps multiple versions of each key so readers and writers never block each other.

The Concurrency Problem
Naive locking means a long-running read (backup, analytics) blocks ALL writers. MVCC is the production solution: PostgreSQL, MySQL InnoDB, RocksDB, TiKV all use it. Readers NEVER block writers. Writers NEVER block readers.

Why Naive Locking Fails

// Three approaches that all break under realistic concurrent workloads
APPROACH 1: One global RwLock<HashMap>

  Thread A (reading 1000 keys):  db.read()  → holds lock for 5ms
  Thread B (writing):            db.write() → BLOCKED until A releases
  Thread D (writing):            db.write() → BLOCKED behind A and C

  Problem: one long-running read (backup, analytics) blocks ALL writers.

APPROACH 2: Fine-grained per-key locks

  Thread A: lock("alice"), lock("bob"), lock("carol")
  Thread B: lock("carol"), lock("alice") → DEADLOCK (different acquisition order)

  Prevention needs a global lock ordering protocol. Complex and error-prone.

APPROACH 3: Optimistic concurrency (compare-and-swap)

  read("alice") → value=30, version=5
  new_value = 30 + 10 = 40
  CAS("alice", expected_version=5, new_value=40)
    if version still 5: success
    if version changed: RETRY ← starvation under contention

  Works for single-key updates, breaks for multi-key atomic operations.

MVCC: The Core Insight

// Append new versions instead of updating in-place
Instead of overwriting a key, APPEND a new version with a higher sequence number.
Each reader takes a snapshot (current global seq) and only sees versions ≤ that seq.

WITHOUT MVCC (in-place):
  DB: { "alice" → 30 }
  Writer sets alice = 35 → { "alice" → 35 }  (30 is GONE)
  Concurrent reader (started before write) → reads 35 — INCONSISTENT!

WITH MVCC:
  DB: { ("alice", seq=5) → 30,  ("alice", seq=8) → 35 }

  Reader snapshot = 6:
    seq=5 ≤ 6 → VISIBLE  → reads alice = 30  (correct, consistent)
    seq=8 > 6  → INVISIBLE (too new for this snapshot)

  Reader snapshot = 9:
    seq=8 ≤ 9 → VISIBLE  → reads alice = 35  (correct, consistent)

  Writer adds ("alice", seq=11) → 42
    → does not block any reader
    → old versions stay until GC removes them

Sequence numbers are the logical clock.
Every write increments the global counter atomically (fetch_add, SeqCst).
Every reader captures the counter value at start time as their snapshot.
Section 05

MVCC Internals

The InternalKey format, sort order, and garbage collection algorithm are what make MVCC actually work in an LSM-Tree. The sort order — newest version first — is the critical design decision.

InternalKey Format and Sort Order

// How keys are encoded on disk and in MemTable with MVCC
Every stored key is an INTERNAL KEY:
  InternalKey = (user_key, seq_no, record_type)

On-disk encoding:
  [ user_key bytes ] [ seq_and_type: u64 big-endian ]
  seq_and_type = (seq_no << 8) | record_type
  record_type: 0x1 = Value,  0x0 = Tombstone

SORT ORDER (critical):
  Primary:   user_key ASCENDING   (lexicographic)
  Secondary: seq_no   DESCENDING  ← newest version comes FIRST

Why descending? On a read we seek to (user_key, MAX_SEQ) and scan forward.
The first result we encounter is the newest version.

MEMTABLE CONTENTS (sorted):
  ("alice", seq=1042, Value)     → "35"   ← newest, found first on seek
  ("alice", seq=0503, Value)     → "30"
  ("alice", seq=0200, Value)     → "28"   ← oldest
  ("bob",   seq=0800, Tombstone) → ""     ← bob deleted (newest entry)
  ("bob",   seq=0100, Value)     → "25"   ← pre-deletion value
  ("carol", seq=0900, Value)     → "28"

READ: get("alice") with snapshot seq=600
  Seek to ("alice", seq=MAX)
  ("alice", seq=1042): 1042 > 600 → INVISIBLE (too new)
  ("alice", seq=0503): 503  ≤ 600 → VISIBLE  → type=Value → return "30"

  alice="35" is invisible to this snapshot — consistent read guaranteed.

Snapshot Lifecycle and GC

// How snapshots are tracked and old versions garbage-collected
SNAPSHOT = a single u64 (the seq_no at snapshot creation time)

  acquire: snap_seq = current_global_seq(); register(snap_seq); return snap_seq
  use:     only read entries where entry.seq_no ≤ snap_seq
  release: unregister(snap_seq)

SNAPSHOT TRACKER:
  struct SnapManager { live: BTreeSet<u64> }  // sorted set of active snap seqs
  oldest_snapshot() → self.live.iter().next().copied()  // min active seq

VERSION GC LOGIC:

  State: alice → [v@1000=35, v@500=30, v@100=28]
  Active snapshots: {400, 750}   oldest = 400

  v@1000: keep (newest version, always kept)
  v@500:  is any snapshot in range (500, 1000)? → yes, 750 → KEEP
  v@100:  is any snapshot in range (100, 500)?  → yes, 400 → KEEP

  After snapshots 400 and 750 are released:
  oldest_snapshot = None

  v@500:  no snapshot needs it → GC deletes
  v@100:  no snapshot needs it → GC deletes
  → Only v@1000 remains. Space reclaimed.

TOMBSTONE CLEANUP:
  State: bob → [TOMBSTONE@800, Value@100]
  oldest_snapshot = 1100  (all snapshots > 800 have released)
  No snapshot can see bob@100 (100 < 800 < 1100)
  → GC removes BOTH tombstone AND old value
  → bob is completely erased from disk. This is how deletes truly commit.

GC happens during compaction:
  RocksDB: compaction filter callbacks
  TiKV:    GC worker thread with "GC safepoint" watermark
  Postgres: autovacuum daemon (same concept, different storage engine)
Section 06

MVCC in Rust

Three files: the global sequence counter + snapshot manager, the InternalKey with correct sort order, and the MVCC-aware MemTable that uses snapshot-isolated reads.

rust // src/mvcc/seq.rs — global sequence counter + snapshot manager
use std::sync::atomic::{AtomicU64, Ordering};
use std::sync::{Arc, Mutex};
use std::collections::BTreeSet;

static GLOBAL_SEQ: AtomicU64 = AtomicU64::new(1);

pub fn next_seq()    -> u64 { GLOBAL_SEQ.fetch_add(1, Ordering::SeqCst) }
pub fn current_seq() -> u64 { GLOBAL_SEQ.load(Ordering::SeqCst) }

// ─── Snapshot ────────────────────────────────────────────────────────────────

#[derive(Debug, Clone)]
pub struct Snapshot {
    pub seq: u64,
    mgr:     Arc<SnapManager>,
}

impl Drop for Snapshot {
    fn drop(&mut self) { self.mgr.release(self.seq); }  // auto-release on drop
}

pub struct SnapManager {
    live: Mutex<BTreeSet<u64>>,
}

impl SnapManager {
    pub fn new() -> Arc<Self> {
        Arc::new(Self { live: Mutex::new(BTreeSet::new()) })
    }

    pub fn acquire(self: &Arc<Self>) -> Snapshot {
        let seq = current_seq();
        self.live.lock().unwrap().insert(seq);
        Snapshot { seq, mgr: Arc::clone(self) }
    }

    fn release(&self, seq: u64) {
        self.live.lock().unwrap().remove(&seq);
    }

    /// Oldest still-live snapshot. GC may remove versions older than this.
    pub fn oldest(&self) -> Option<u64> {
        self.live.lock().unwrap().iter().next().copied()
    }
}
rust // src/mvcc/key.rs — InternalKey with correct sort order
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct InternalKey {
    pub user_key: Vec<u8>,
    pub seq_no:   u64,
    pub ktype:    KeyType,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum KeyType { Value, Tombstone }

impl Ord for InternalKey {
    fn cmp(&self, other: &Self) -> std::cmp::Ordering {
        // user_key ascending, then seq_no DESCENDING (newest first)
        match self.user_key.cmp(&other.user_key) {
            std::cmp::Ordering::Equal => other.seq_no.cmp(&self.seq_no),
            ord => ord,
        }
    }
}
impl PartialOrd for InternalKey {
    fn partial_cmp(&self, other: &Self) -> Option<std::cmp::Ordering> {
        Some(self.cmp(other))
    }
}

impl InternalKey {
    /// Seek key to find newest version: sets seq_no to MAX so we land before all versions.
    pub fn seek(user_key: &[u8]) -> Self {
        Self { user_key: user_key.to_vec(), seq_no: u64::MAX, ktype: KeyType::Value }
    }

    /// On-disk encoding: user_key || (seq_no << 8 | type_byte) as u64 big-endian
    pub fn encode(&self) -> Vec<u8> {
        let tb: u64 = match self.ktype { KeyType::Value => 1, KeyType::Tombstone => 0 };
        let packed = (self.seq_no << 8) | tb;
        let mut out = self.user_key.clone();
        out.extend_from_slice(&packed.to_be_bytes());
        out
    }
}
rust // src/mvcc/memtable.rs — MVCC-aware MemTable
use std::collections::BTreeMap;
use super::{InternalKey, KeyType, Snapshot, next_seq};

pub struct MVCCMemTable {
    data:  BTreeMap<InternalKey, Vec<u8>>,
    bytes: usize,
}

impl MVCCMemTable {
    pub fn new() -> Self { Self { data: BTreeMap::new(), bytes: 0 } }

    pub fn write(&mut self, user_key: Vec<u8>, value: Vec<u8>) -> u64 {
        let seq = next_seq();
        self.bytes += user_key.len() + value.len() + 24;
        self.data.insert(InternalKey { user_key, seq_no: seq, ktype: KeyType::Value }, value);
        seq
    }

    pub fn delete(&mut self, user_key: Vec<u8>) -> u64 {
        let seq = next_seq();
        self.bytes += user_key.len() + 24;
        self.data.insert(InternalKey { user_key, seq_no: seq, ktype: KeyType::Tombstone }, vec![]);
        seq
    }

    /// Snapshot-isolated point read.
    pub fn get<'a>(&'a self, user_key: &[u8], snap: &Snapshot) -> Option<&'a [u8]> {
        // Seek to (user_key, MAX_SEQ) — our Ord puts newest first
        let seek = InternalKey::seek(user_key);

        for (k, v) in self.data.range(seek..) {
            if k.user_key.as_slice() != user_key { break; }  // past this key
            if k.seq_no > snap.seq               { continue; }  // too new for snapshot
            return match k.ktype {
                KeyType::Value     => Some(v.as_slice()),
                KeyType::Tombstone => None,  // deleted
            };
        }
        None
    }

    pub fn size_bytes(&self) -> usize { self.bytes }

    /// Sorted iterator for SSTable flush — BTreeMap guarantees InternalKey sort order.
    pub fn iter(&self) -> impl Iterator<Item = (&InternalKey, &Vec<u8>)> {
        self.data.iter()
    }
}
Pitfall: MVCC Bloat from Long-Running Snapshots
A snapshot held open for a long time keeps every MVCC version it can see alive. GC cannot remove those versions until the snapshot is released, causing unbounded disk growth.

PostgreSQL calls this "table bloat" — fixed by autovacuum. TiKV uses a "GC safepoint" watermark. RocksDB has a snapshot_checker callback.

In your implementation: warn if any snapshot is held longer than 30 seconds, and provide a forced-expiry timeout for read-only analytical queries.
