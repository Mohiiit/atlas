Section 01

What to Measure

Most published benchmarks are wrong. Either the workload doesn't match reality, the measurement introduces overhead, the storage isn't warmed up, or the wrong metric is being optimized. This section teaches you how to build an honest, reproducible benchmark.

Why Benchmarking Is Hard
Most published benchmarks are wrong. Either the workload doesn't match reality, the measurement itself introduces overhead, the storage isn't warmed up, or the wrong metric is being optimized. You need an honest, reproducible benchmark that tells you where your DB actually spends its time.

The 5 Metrics That Matter

METRIC 1: THROUGHPUT (ops/sec or MB/sec)
How many operations complete per second under sustained load. Run 1M+ ops to amortize warmup. Measure separately: sequential write, random write, sequential read, random read. Never measure burst — measure sustained (60+ seconds).
total_ops = 1_000_000;
let start = Instant::now();
for _ in 0..total_ops { db.set(rand_key(), rand_value()); }
let throughput = total_ops as f64 / start.elapsed().as_secs_f64();
// ops/sec
METRIC 2: LATENCY PERCENTILES (p50, p95, p99, p99.9)
NEVER just measure average — it hides tail latency. A DB with avg=0.5ms but p99=200ms has a serious compaction stall problem. The average lies. The p99 reveals the truth. p99 is usually the SLA boundary for production systems.
let mut latencies: Vec<u64> = Vec::with_capacity(100_000);
for _ in 0..100_000 {
    let t = Instant::now();
    db.get(&key);
    latencies.push(t.elapsed().as_micros() as u64);
}
latencies.sort_unstable();
let p50  = latencies[latencies.len() / 2];
let p99  = latencies[latencies.len() * 99 / 100];
let p999 = latencies[(latencies.len() * 999 / 1000).min(n - 1)];
METRIC 3: WRITE AMPLIFICATION FACTOR (WAF)
Bytes actually written to storage / bytes sent by user. If user writes 1 GB but the storage device sees 10 GB written: WAF = 10×. Typical for leveled LSM compaction. Measure using /proc/self/io on Linux.
METRIC 4: READ AMPLIFICATION FACTOR (RAF)
Bytes read from storage / bytes returned to user. For a 100-byte value, if you had to read 4 × 4KB blocks to find it: RAF = (4 × 4096) / 100 = 163×. For Bloom-optimized LSM: typically 1-2 block reads per found key → RAF ~40-80×.
METRIC 5: SPACE AMPLIFICATION FACTOR (SAF)
Actual disk bytes used / logical data size. If you stored 1 GB of unique data but the DB uses 1.5 GB on disk: SAF = 1.5×. SAF during compaction can temporarily spike to 2× (old + new files coexist). Typical for LSM with leveled compaction.
Section 02

YCSB Workloads

A DB that does 2M ops/sec on sequential writes might do 100K on Zipfian random reads. Real applications have specific access patterns. YCSB (Yahoo Cloud Serving Benchmark) defines standard workloads so you can compare apples to apples.

Key Distributions

// Uniform vs Zipfian — drastically different behavior
UNIFORM DISTRIBUTION (unrealistic):
  Every key has equal probability of being accessed.
  With N=1M keys: each key accessed ~1 time on average.
  Cache hit rate: near 0% if data > RAM. Every read is a cache miss.
  This is EASY for all DBs and doesn't reflect reality.

ZIPFIAN DISTRIBUTION (realistic — "80/20 rule"):
  Named after linguist George Zipf.
  Rank 1 key (most popular): accessed N × (1/1^s) times  ← a LOT
  Rank k key:                accessed N × (1/k^s) times  (s ≈ 0.99)

  Practical effect with 1M keys:
    Top 100 keys  → ~30% of all requests
    Top 10,000    → ~60% of all requests
    Bottom 900K   → ~40% spread thinly

  Top keys fit in memory → very high cache hit rate for hot data.
  But cold data accesses (bottom 900K keys) are still cache misses.
  This is what your block cache, MemTable, and OS page cache optimize for.

LATEST DISTRIBUTION (for time-series, logs):
  Most recently inserted K keys are accessed most often.
  Naturally benefits append-only systems like Bitcask and LSM.

HOTSPOT DISTRIBUTION (for conflict testing):
  80% of ops target 20% of keys, uniformly within that 20%.
  Tests lock contention, MemTable hot path, per-key locking issues.

YCSB Workload Profiles

// Standard YCSB workloads — start with A and C for your DB
Workload A: 50% reads / 50% updates  (Zipfian)  ← "session store"
Workload B: 95% reads / 5% updates   (Zipfian)  ← "photo tagging"
Workload C: 100% reads               (Zipfian)  ← "user profile cache"
Workload D: 95% reads / 5% inserts   (Latest)   ← "activity log, read latest"
Workload E: 95% scans / 5% inserts   (Zipfian)  ← "thread conversations"
Workload F: 50% reads / 50% RMW      (Zipfian)  ← "user database, read-modify-write"

For YOUR KV store: start with Workload A and C.
  A tests write throughput under concurrent reads.
  C tests read-only performance (your best case and your ceiling).

Benchmark Phases for Your DB

// The correct sequence of benchmarks to run at each implementation phase
PHASE 1: BASELINE — Sequential write throughput
  Write 1M keys sequentially (key_0000001, key_0000002, ...) with 100-byte values.
  Measure: ops/sec, total MB/sec, p99 latency.
  This is your DB's PEAK WRITE performance (sequential = optimal).

PHASE 2: RANDOM WRITE THROUGHPUT
  Write 1M keys in random order (from a pre-shuffled array — avoid rand overhead in loop!).
  Bitcask: random ≈ sequential (append-only, order doesn't matter!)
  B-Tree:  random < sequential (causes page fragmentation and random I/O)
  LSM:     random ≈ sequential (MemTable absorbs disorder, SSTable sorted at flush)

PHASE 3: READ THROUGHPUT (after writing 1M keys)
  Test A — Sequential reads (keys in sorted order):
    Bitcask: TERRIBLE (no ordering, each read is a random seek)
    LSM:     GOOD (SSTables sorted, sequential reads benefit from prefetch)
    B-Tree:  GOOD (leaf pages linked list, prefetching works)

  Test B — Random reads (Zipfian distribution):
    Repeat 1M reads with Zipfian keys from the 1M-key dataset.
    Measure: p50, p99 latency; ops/sec; cache hit rate.

PHASE 4: MIXED WORKLOAD (Workload A)
  50% reads + 50% writes simultaneously from multiple threads.
  Most important benchmark — reveals lock contention, write-stall, compaction interference.
  Run at: 1, 2, 4, 8, 16, 32 concurrent threads.
  If throughput drops after 4 threads → you have lock contention.

PHASE 5: COMPACTION IMPACT (LSM only)
  Write enough data to trigger multiple compactions.
  Plot throughput over TIME (not just the average!).
  You should see: normal → write stall during compaction → recovery.

PHASE 6: WORST CASE (100% random read misses)
  Write 1M keys, then read 1M DIFFERENT keys (all will be misses).
    Bitcask: RAM lookup → not found → instant!
    LSM + Bloom: Bloom says no → 0 disk reads → fast!
    LSM without Bloom: must check every SSTable → SLOW
    B-Tree: log(N) reads to confirm miss → medium
Section 03

Measuring Amplification

Linux's /proc/self/io gives you the actual bytes read from and written to the storage device — not just what went into the OS page cache. This is the ground truth for amplification measurement.

/proc/self/io Fields

// Reading /proc/self/io to measure actual disk writes
$ cat /proc/self/io
rchar:  152370     ← bytes read (including from page cache)
wchar:  4096       ← bytes written (including buffered)
syscr:  20         ← number of read() syscalls
syscw:  1          ← number of write() syscalls
read_bytes:  0     ← bytes actually read from STORAGE DEVICE (after page cache)
write_bytes: 4096  ← bytes actually written to STORAGE DEVICE (flushed through cache)
cancelled_write_bytes: 0

KEY FIELDS:
  write_bytes: bytes that landed on actual storage device
    → This is what we want for write amplification
    → Unlike wchar, this doesn't count bytes still in page cache

  read_bytes: bytes that caused actual device reads
    → Use this for read amplification
    → Reads served from page cache are NOT counted here

CAVEAT: write_bytes is only accurate AFTER fsync / storage flush.
  If you don't fsync, the OS batches writes.
  ALWAYS call fsync() or db.flush() before reading post-benchmark stats.
rust // src/bench/amplification.rs
#[derive(Debug, Default, Clone)]
pub struct IoStats {
    pub read_bytes:  u64,   // actual storage reads
    pub write_bytes: u64,   // actual storage writes
}

impl IoStats {
    pub fn snapshot() -> std::io::Result<Self> {
        let content = std::fs::read_to_string("/proc/self/io")?;
        let mut stats = IoStats::default();
        for line in content.lines() {
            let mut parts = line.splitn(2, ": ");
            let key = parts.next().unwrap_or("");
            let val: u64 = parts.next().and_then(|v| v.trim().parse().ok()).unwrap_or(0);
            match key {
                "read_bytes"  => stats.read_bytes  = val,
                "write_bytes" => stats.write_bytes = val,
                _ => {}
            }
        }
        Ok(stats)
    }
}

pub struct AmplificationMeasurement {
    before: IoStats,
    user_bytes_written: u64,
    user_bytes_read:    u64,
}

impl AmplificationMeasurement {
    pub fn start() -> std::io::Result<Self> {
        Ok(Self { before: IoStats::snapshot()?, user_bytes_written: 0, user_bytes_read: 0 })
    }

    pub fn record_write(&mut self, key_len: usize, val_len: usize) {
        self.user_bytes_written += (key_len + val_len) as u64;
    }

    /// Call AFTER db.flush() and waiting for compaction to complete.
    pub fn finish(self) -> std::io::Result<AmplificationReport> {
        let after = IoStats::snapshot()?;
        let disk_writes = after.write_bytes.saturating_sub(self.before.write_bytes);
        let disk_reads  = after.read_bytes.saturating_sub(self.before.read_bytes);
        Ok(AmplificationReport {
            write_amplification: disk_writes as f64 / self.user_bytes_written.max(1) as f64,
            read_amplification:  disk_reads  as f64 / self.user_bytes_read.max(1) as f64,
            disk_bytes_written:  disk_writes,
            disk_bytes_read:     disk_reads,
        })
    }
}

pub struct AmplificationReport {
    pub write_amplification: f64,
    pub read_amplification:  f64,
    pub disk_bytes_written:  u64,
    pub disk_bytes_read:     u64,
}

Space Amplification

rust // Walk data directory to measure actual disk usage
pub fn space_amplification(data_dir: &Path, logical_bytes: u64) -> std::io::Result<f64> {
    let actual: u64 = std::fs::read_dir(data_dir)?
        .filter_map(|e| e.ok())
        .filter_map(|e| e.metadata().ok())
        .filter(|m| m.is_file())
        .map(|m| m.len())
        .sum();
    Ok(actual as f64 / logical_bytes as f64)
}

// Usage: call AFTER all compaction completes
let logical = num_keys * (key_size + value_size) as u64;
let saf = space_amplification(Path::new("/tmp/mydb"), logical)?;
println!("Space amplification: {:.2}×", saf);
// Expected: ~1.5× for leveled LSM, ~2-5× for tiered (size-tiered) compaction
Section 04

Rust Benchmark Harness

A complete YCSB-style benchmark harness with a ZipfianGenerator, latency recording, and a trait that lets you plug in any KV store implementation.

rust // benches/ycsb.rs — complete YCSB harness
use std::time::{Duration, Instant};
use rand::prelude::*;
use rand_distr::Zipf;

/// Zipfian key generator — simulates real workload access patterns
pub struct ZipfianGenerator {
    dist:    Zipf<f64>,
    rng:     StdRng,
    max_key: usize,
}

impl ZipfianGenerator {
    /// s ≈ 0.99 for typical web workloads, s=1.0 for strict Zipf's law
    pub fn new(max_key: usize, s: f64) -> Self {
        Self {
            dist:    Zipf::new(max_key as u64, s).unwrap(),
            rng:     StdRng::from_entropy(),
            max_key,
        }
    }

    pub fn next_key(&mut self) -> usize {
        (self.dist.sample(&mut self.rng) as usize).saturating_sub(1)
    }
}

/// Format a key index as a fixed-width byte string (sortable)
pub fn format_key(i: usize) -> [u8; 16] {
    let mut key = [0u8; 16];
    let s = format!("{:016}", i);
    key.copy_from_slice(s.as_bytes());
    key
}

#[derive(Debug)]
pub struct BenchResult {
    pub name:        String,
    pub ops:         u64,
    pub duration:    Duration,
    pub ops_per_sec: f64,
    pub p50_us:      u64,
    pub p95_us:      u64,
    pub p99_us:      u64,
    pub p999_us:     u64,
    pub mb_per_sec:  f64,
}

impl BenchResult {
    pub fn print(&self) {
        println!("  [{:}]", self.name);
        println!("  ops/sec: {:10.0}  MB/sec: {:10.2}", self.ops_per_sec, self.mb_per_sec);
        println!("  p50: {}μs  p95: {}μs  p99: {}μs  p99.9: {}μs",
            self.p50_us, self.p95_us, self.p99_us, self.p999_us);
    }
}

/// Run a benchmark, collecting per-operation latencies
pub fn run_benchmark<F>(name: &str, num_ops: u64, mut op: F) -> BenchResult
where F: FnMut(u64) -> usize  // takes op index, returns bytes touched
{
    let mut latencies_us = Vec::with_capacity(num_ops as usize);
    let mut total_bytes = 0u64;
    let start = Instant::now();

    for i in 0..num_ops {
        let op_start = Instant::now();
        let bytes = op(i);
        latencies_us.push(op_start.elapsed().as_micros() as u64);
        total_bytes += bytes as u64;
    }

    let duration = start.elapsed();
    latencies_us.sort_unstable();
    let n = latencies_us.len();

    BenchResult {
        name:        name.to_string(),
        ops:         num_ops,
        duration,
        ops_per_sec: num_ops as f64 / duration.as_secs_f64(),
        p50_us:      latencies_us[n * 50 / 100],
        p95_us:      latencies_us[n * 95 / 100],
        p99_us:      latencies_us[n * 99 / 100],
        p999_us:     latencies_us[(n * 999 / 1000).min(n - 1)],
        mb_per_sec:  total_bytes as f64 / duration.as_secs_f64() / 1_048_576.0,
    }
}

/// YCSB Workload A: 50% reads, 50% updates, Zipfian
pub fn workload_a<DB: KVStore>(db: &mut DB, num_keys: usize, num_ops: u64) -> BenchResult {
    let mut zipf = ZipfianGenerator::new(num_keys, 0.99);
    let mut rng = StdRng::from_entropy();
    let value = vec![b'x'; 100];

    run_benchmark("YCSB-A: 50% read / 50% write (Zipfian)", num_ops, |_| {
        let key = format_key(zipf.next_key());
        if rng.gen_bool(0.5) {
            db.set(&key, &value).unwrap();
            key.len() + value.len()
        } else {
            let v = db.get(&key).unwrap();
            key.len() + v.map_or(0, |v| v.len())
        }
    })
}

/// YCSB Workload C: 100% reads, Zipfian
pub fn workload_c<DB: KVStore>(db: &mut DB, num_keys: usize, num_ops: u64) -> BenchResult {
    let mut zipf = ZipfianGenerator::new(num_keys, 0.99);
    run_benchmark("YCSB-C: 100% read (Zipfian)", num_ops, |_| {
        let key = format_key(zipf.next_key());
        db.get(&key).unwrap().map_or(0, |v| v.len())
    })
}

/// Sequential write — best case, establishes ceiling
pub fn bench_sequential_write<DB: KVStore>(db: &mut DB, num_ops: u64) -> BenchResult {
    let value = vec![b'v'; 100];
    run_benchmark("Sequential Write (100B values)", num_ops, |i| {
        let key = format_key(i as usize);
        db.set(&key, &value).unwrap();
        key.len() + value.len()
    })
}

/// Random write — more realistic
pub fn bench_random_write<DB: KVStore>(db: &mut DB, num_ops: u64) -> BenchResult {
    let mut rng = StdRng::from_entropy();
    let value = vec![b'v'; 100];
    // Pre-generate to avoid rand overhead in hot path
    let key_indices: Vec<usize> = (0..num_ops as usize)
        .map(|_| rng.gen_range(0..num_ops as usize))
        .collect();
    run_benchmark("Random Write (100B values, uniform)", num_ops, |i| {
        let key = format_key(key_indices[i as usize]);
        db.set(&key, &value).unwrap();
        key.len() + value.len()
    })
}

// ─── Trait for your DB to implement ─────────────────────────────────────────
pub trait KVStore: Send + Sync {
    fn set(&mut self, key: &[u8], value: &[u8]) -> Result<(), Box<dyn std::error::Error>>;
    fn get(&self, key: &[u8]) -> Result<Option<Vec<u8>>, Box<dyn std::error::Error>>;
    fn delete(&mut self, key: &[u8]) -> Result<(), Box<dyn std::error::Error>>;
    fn flush(&mut self) -> Result<(), Box<dyn std::error::Error>> { Ok(()) }
}

Comparing Real Databases

How to wire your DB into the benchmark harness alongside sled and RocksDB, and what numbers to expect at each implementation phase.

rust // benches/comparison.rs — your DB vs sled vs rocksdb
fn bench_all_engines(num_ops: u64) {
    // ── Your Bitcask Implementation ───────────────────────────────────────────
    {
        let mut db = your_kv::BitcaskDB::open("/tmp/bench_yours").unwrap();
        println!("[ Your Bitcask Implementation ]");
        bench_sequential_write(&mut db, num_ops).print();
        bench_random_write(&mut db, num_ops).print();
        workload_c(&mut db, num_ops as usize, num_ops).print();
        workload_a(&mut db, num_ops as usize, num_ops).print();
    }
    std::fs::remove_dir_all("/tmp/bench_yours").ok();

    // ── Sled (pure Rust, BTree-like) ──────────────────────────────────────────
    {
        let db = sled::Config::new().path("/tmp/bench_sled").open().unwrap();
        let mut w = SledWrapper(db);
        println!("\n[ Sled (pure Rust) ]");
        bench_sequential_write(&mut w, num_ops).print();
        bench_random_write(&mut w, num_ops).print();
        workload_c(&mut w, num_ops as usize, num_ops).print();
    }
    std::fs::remove_dir_all("/tmp/bench_sled").ok();

    // ── RocksDB ───────────────────────────────────────────────────────────────
    {
        let mut opts = rocksdb::Options::default();
        opts.create_if_missing(true);
        opts.set_write_buffer_size(64 * 1024 * 1024);   // 64MB MemTable
        opts.set_max_write_buffer_number(4);
        opts.set_level_compaction_dynamic_level_bytes(true);
        let db = rocksdb::DB::open(&opts, "/tmp/bench_rocksdb").unwrap();
        let mut w = RocksWrapper(db);
        println!("\n[ RocksDB (Facebook LSM) ]");
        bench_sequential_write(&mut w, num_ops).print();
        bench_random_write(&mut w, num_ops).print();
        workload_c(&mut w, num_ops as usize, num_ops).print();
        workload_a(&mut w, num_ops as usize, num_ops).print();
    }
    std::fs::remove_dir_all("/tmp/bench_rocksdb").ok();
}

struct SledWrapper(sled::Db);
impl KVStore for SledWrapper {
    fn set(&mut self, k: &[u8], v: &[u8]) -> Result<(), Box<dyn std::error::Error>> { self.0.insert(k, v)?; Ok(()) }
    fn get(&self, k: &[u8]) -> Result<Option<Vec<u8>>, Box<dyn std::error::Error>> { Ok(self.0.get(k)?.map(|v| v.to_vec())) }
    fn delete(&mut self, k: &[u8]) -> Result<(), Box<dyn std::error::Error>> { self.0.remove(k)?; Ok(()) }
}

struct RocksWrapper(rocksdb::DB);
impl KVStore for RocksWrapper {
    fn set(&mut self, k: &[u8], v: &[u8]) -> Result<(), Box<dyn std::error::Error>> { Ok(self.0.put(k, v)?) }
    fn get(&self, k: &[u8]) -> Result<Option<Vec<u8>>, Box<dyn std::error::Error>> { Ok(self.0.get(k)?) }
    fn delete(&mut self, k: &[u8]) -> Result<(), Box<dyn std::error::Error>> { Ok(self.0.delete(k)?) }
}

Expected Results by Phase

// What numbers to expect at each build phase (NVMe SSD, 100B values)
PHASE 1 (HashMap + bincode persistence — no WAL):
  Sequential write:  ~2-5M ops/sec  (pure RAM, no disk during run)
  Random read:       ~1-3M ops/sec  (HashMap lookup)
  p99 write:         < 5μs
  Crash safe?        NO

PHASE 2 (Bitcask: append-only log, no compaction):
  Sequential write:  ~400-800K ops/sec  (bounded by sequential NVMe ~200MB/s)
  Random write:      ~400-800K ops/sec  (append-only = always sequential)
  Random read:       ~100-300K ops/sec  (1 seek + read per lookup)
  p99 write:         ~10-50μs  (dominated by fsync if using PerWrite mode)
  Crash safe?        YES (with fsync)

PHASE 3 (Bitcask + compaction + hint files):
  Same throughput as Phase 2, but:
  Disk usage:        Bounded (not growing forever)
  Startup time:      ~100ms for 1M keys (vs 5-10s without hint files)
  Write amp:         ~1.5-2× during active compaction

PHASE 4 (LSM: MemTable + SSTable, no Bloom, basic compaction):
  Sequential write:  ~600K-1M ops/sec  (MemTable in RAM, flush async)
  Random read:       ~50-150K ops/sec  (must check multiple SSTables, no Bloom)
  Read miss:         SLOW (30+ disk reads to confirm a key doesn't exist)
  Write amp:         ~10-15×

PHASE 5 (LSM + Bloom filters):
  Random read:       ~200-500K ops/sec (Bloom eliminates most disk reads)
  Read miss:         ~100-300K ops/sec (most misses = 0 disk reads via Bloom)
  Write amp:         same ~10-15×

PHASE 6 (LSM + WAL + crash recovery):
  Throughput:        ~80% of Phase 5 (WAL write overhead)
  p99 write:         5-50μs (group commit), 100-500μs (per-write fsync)
  Crash safe?        YES

COMPARISON — same hardware, same workload A:
  Your Phase 5 DB:  ~300K ops/sec
  Sled (v0.34):     ~200-400K ops/sec
  RocksDB:          ~500K-1M ops/sec  (10 years of engineering)

  Your DB within 3-5× of RocksDB on simple workloads is EXCELLENT
  for a learning implementation.
