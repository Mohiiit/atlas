Concept 01
RUM Conjecture
The fundamental tradeoff in ALL storage systems
TL;DR

You can optimize any 2 of: Read overhead, Update overhead, Memory overhead — but the 3rd will suffer. This is why no single DB is best at everything.

What it says
RUM Tradeoff Space
R = Read Amplification
  How many bytes does the disk read per byte of actual data returned?
  B-Tree: ~1-2x   |  LSM: 5-10x (without Bloom)  |  Bitcask: 1x (one seek)

U = Update (Write) Amplification
  How many bytes are written to disk per byte of user data?
  B-Tree: ~2-5x   |  LSM (leveled): ~10-30x  |  Bitcask: ~1-2x

M = Memory Amplification
  How much memory is needed beyond the actual data?
  Bitcask: ALL keys must be in RAM (bad)
  LSM: just MemTable + Bloom filters (good)
  B-Tree: OS page cache (moderate)

The conjecture: R + U + M >= constant. Minimize 2, the 3rd grows.
How real DBs choose
Storage Model Choices
Bitcask:     min(R, U) -> pays M (all keys in RAM)
LSM-Tree:    min(U, M) -> pays R (check many SSTables)
B-Tree:      min(R, M) -> pays U (in-place writes, 2-5x write amp)
FractalTree: attempts to minimize all 3 (Tokutek/Percona -- complex)
Applying RUM to your build

Bloom filter: improves R (99% of misses skip disk entirely), costs M (10 bits/key held in RAM). A direct R vs M tradeoff — you're buying read speed with memory.

WAL: improves durability (crash safety), costs U (+1 sequential write per operation). You're paying write amplification for the guarantee that data survives a crash.

Leveled compaction: improves R and M (fewer files to check, bounded disk space), costs U (each key is rewritten 10–30× across level transitions). You're spending write I/O to keep read paths clean.

The deeper lesson

Every time you add a feature to Cementite, ask: which of R, U, or M am I improving? What am I sacrificing? If you can't answer that, you don't understand the feature yet.

Concept 02
Write-Ahead Log (WAL)
How DBs survive crashes without losing data
TL;DR

Before writing data anywhere, write what you're ABOUT to do to a log. If you crash, replay the log. This is how every serious database achieves durability.

The core problem WAL solves
Without WAL — crash scenario
Step 1: MemTable updated in RAM       (data exists in memory only)
Step 2: SSTable write in progress     (disk write started, not finished)
Step 3: CRASH!
Result: MemTable is gone, SSTable is partial/corrupt -> DATA LOST

With WAL:
Step 1: Append to WAL + fsync()       (data is durable on disk NOW)
Step 2: Acknowledge user: "write OK"  (safe to return success)
Step 3: Update MemTable in RAM        (can do this async)

On restart:
  Open WAL file -> scan records -> replay each op -> MemTable rebuilt
  Data is exactly as if the crash never happened.
WAL Record Format
WAL Record Binary Layout
+----------------------------------------------------------+
|               WAL Record Binary Format                    |
+-------------+--------+--------+-----------+--------------+
| Record Len  | CRC32C |  Type  |  Seq No.  |   Payload    |
|   4 bytes   | 4 bytes| 1 byte |  8 bytes  |  variable    |
+-------------+--------+--------+-----------+--------------+

Types:
  0x01 = FULL     (complete record in one fragment)
  0x02 = FIRST    (first fragment of a large record)
  0x03 = MIDDLE   (middle fragment)
  0x04 = LAST     (final fragment)
  0x05 = SET      (key-value write)
  0x06 = DELETE   (tombstone)
  0x07 = BATCH    (atomic batch of operations)
  0x08 = CHECKPOINT (MemTable flushed, WAL can be truncated)

CRC32C checksum covers [Type + SeqNo + Payload].
A bad CRC = torn write (crash mid-record). Stop replay here.
Database is consistent up to the last complete record.
Durability ladder
Durability vs Performance Tradeoffs
Level 0 -- No fsync:        ***** speed  *     safety  (Redis default)
  write() -> kernel buffer -> disk (eventually)
  Throughput: 1-5M ops/sec
  Risk: last ~100ms of writes lost on crash/power loss

Level 1 -- fsync per write:  **    speed  ***** safety  (PostgreSQL sync)
  write() -> fsync() -> ack
  Throughput: 5-20K ops/sec on SSD (1 fsync = ~0.1ms)

Level 2 -- Group commit:     ****  speed  ****  safety  (RocksDB, MySQL)
  Collect 100 writes -> 1 fsync -> ack all 100
  Throughput: on NVMe 20K ops/sec -> 1M ops/sec (same hardware!)
  Risk: last group lost on crash (~1-5ms window)

Group commit is almost always the right answer for production.
Concept 03
LSM-Tree Architecture
The most dominant storage model today
TL;DR

Never update data in place. Write to memory first, batch to sorted disk files, merge them in the background. Converts random writes into sequential writes.

Write Path
LSM Write Path
User: set("alice", "30")
  1. WAL append: [SET alice 30 seq=1042] + fsync()   <- crash-safe now
  2. MemTable insert: BTreeMap { "alice" -> (30, seq=1042) }
  3. Acknowledge user: OK

  [Background] MemTable reaches ~4MB threshold:
     -> Freeze MemTable (becomes immutable)
     -> New empty MemTable for incoming writes
     -> Flush immutable MemTable to SSTable file (sorted on disk)
     -> Delete WAL segment (data is safe in SSTable now)
     -> Add SSTable to L0 in the manifest
Read Path
LSM Read Path (with Bloom filters)
User: get("alice")
  1. Active MemTable         -> not found
  2. Immutable MemTable(s)   -> not found
  3. L0 SSTables (check ALL, keys may overlap between L0 files):
       L0-0: Bloom filter -> NOPE! (0 disk reads saved)
       L0-1: Bloom filter -> maybe -> binary search -> not found
       L0-2: Bloom filter -> maybe -> binary search -> FOUND! alice=30
       Done.
  4. If not in L0: check L1 (binary search manifest -> 1 file, no overlap)
  5. Continue L2, L3... until found or all levels exhausted

Why L0 is special:
  Writes flush directly from MemTable -> L0.
  Multiple L0 files can contain overlapping key ranges.
  L1 and beyond: compaction enforces zero overlap per level.
SSTable Internal Format
SSTable File Layout
+----------------------------------------------------+
|  Data Block 0 (4KB): sorted key-value pairs        |
|    alice -> 30                                     |
|    bob   -> 25                                     |
|    carol -> 28                                     |
+----------------------------------------------------+
|  Data Block 1 (4KB):                               |
|    dave  -> 31                                     |
|    eve   -> 22                                     |
+----------------------------------------------------+
|  Index Block: first key of each block -> offset    |
|    "alice" -> offset 0                             |
|    "dave"  -> offset 4096                          |
+----------------------------------------------------+
|  Bloom Filter (10 bits/key, ~1% false positive)    |
+----------------------------------------------------+
|  Footer: index_offset, bloom_offset, magic_number  |
+----------------------------------------------------+

To find "carol":
  load footer -> read index -> "alice" block at offset 0
  -> read 4KB data block -> scan for "carol"
  Total: 2-3 disk reads (footer + index + 1 data block)
Concept 04
Compaction Strategies
Leveled vs Tiered vs FIFO
TL;DR

Compaction is background work that merges SSTables, removes stale versions, reclaims space. Your compaction strategy determines the RUM tradeoff.

Leveled Compaction (LevelDB, RocksDB default)
Leveled Compaction Structure
L0: 4 files max (direct MemTable flushes, overlap OK)
L1: 10MB total, NO key overlap within level
L2: 100MB    L3: 1GB    L4: 10GB    L5: 100GB    L6: 1TB

Trigger: L0 has 4+ files
  -> Pick 1 L0 file + all overlapping L1 files
  -> Merge-sort them together
  -> Produce new L1 SSTables (no key overlap)
  -> Delete input files

Write amp: HIGH (~10-30x) -- keys rewritten at each level crossing
Read amp:  LOW  -- at most 1 file per key range per level (L1+)
Space amp: LOW  (~1.1x)   -- clean non-overlapping invariant

Best for: read-heavy and mixed workloads. RocksDB default.
Tiered (Size-Tiered) Compaction
Tiered Compaction Structure
4 similar-sized SSTables at a tier -> merge into 1 larger SSTable

4 x 10MB  -> merge -> 1 x 40MB
4 x 40MB  -> merge -> 1 x 160MB
4 x 160MB -> merge -> 1 x 640MB

Write amp: LOW  (~5-10x) -- keys rewritten less often
Read amp:  HIGH          -- multiple overlapping runs per tier
Space amp: HIGH (~2x)   -- input + output files coexist during merge

Best for: write-heavy workloads (time-series, logging, events)
Cassandra default. Apache HBase also uses this by default.
FIFO Compaction
FIFO Compaction (TTL-Based)
When total size exceeds configured limit:
  -> Delete the OLDEST SSTables.
  -> No merging. No rewriting. Just deletion.

Write amp: 1x (no compaction work at all!)
Read amp: CAN be high (many files accumulate)
Space: Fixed (bounded by your configured max_size limit)

Only suitable for:
  - Time-series data (old data irrelevant after a window)
  - Monitoring / metrics (keep last 7 days, delete the rest)
  - Logs (rolling log retention)

NOT for general-purpose KV storage.
NOT when you need consistent point-in-time reads of old data.
Concept 05
Bloom Filters
The magic "definitely not here" checker
TL;DR

A Bloom filter can say "this key is DEFINITELY NOT in this SSTable" in O(1) with zero disk I/O, using only ~10 bits per key. This is the single biggest read optimization in LSM systems.

How it works
Bloom Filter Mechanics
A Bloom filter: bit array of m bits + k hash functions

INSERT "alice" (k=4 hash functions):
  hash1("alice") % 1000 = 42  -> bit[42]  = 1
  hash2("alice") % 1000 = 317 -> bit[317] = 1
  hash3("alice") % 1000 = 891 -> bit[891] = 1
  hash4("alice") % 1000 = 441 -> bit[441] = 1

QUERY "bob" (never inserted):
  hash1("bob") % 1000 = 42  -> bit[42]  = 1 (check)
  hash2("bob") % 1000 = 500 -> bit[500] = 0 <- STOP!
  -> "bob is DEFINITELY NOT HERE" -> Skip SSTable. Zero disk reads!

QUERY "carol" (false positive -- all 4 bits happen to be set):
  All 4 bits = 1 -> "maybe here" -> check disk -> not found
  This costs 1 disk read, but happens only ~1% of the time at 10 bits/key.

KEY PROPERTY: No false negatives (if key exists, filter always says "maybe").
              False positives possible but low probability.
Impact on read performance
Read Performance: With vs Without Bloom Filters
Without Bloom filters in 7-level LSM:
  get("nonexistent"): 30+ disk reads to confirm key doesn't exist!

With Bloom filters (10 bits/key, 1% false positive rate):
  get("nonexistent"):
    L0-0: Bloom -> NOPE (0 disk reads)
    L0-1: Bloom -> NOPE (0 disk reads)
    L0-2: Bloom -> NOPE (0 disk reads)
    L0-3: Bloom -> maybe -> check disk (2 reads) -> not found
    L1-L6: all Bloom -> NOPE (0 disk reads)
  Total: ~2 reads vs 30+   <- ~15x improvement for misses

Tuning table (bits per key):
  4 bits:  14.7% FP rate,  500 KB/1M keys
  8 bits:   2.1% FP rate,  1.0 MB/1M keys
  10 bits:  0.82% FP rate, 1.25 MB/1M keys  <- RocksDB default
  12 bits:  0.30% FP rate, 1.5 MB/1M keys
  16 bits:  0.04% FP rate, 2.0 MB/1M keys
Implementation in Rust

The Kirsch-Mitzenmacher trick: generate k hash functions using only 2 real hashes. g_i(key) = h1(key) + i * h2(key). This avoids computing k independent hashes.

rust
pub struct BloomFilter {
    bits: Vec,  // bit array, packed into u64s
    m: usize,        // total bits (e.g. num_keys * 10)
    k: usize,        // number of hash functions (e.g. 7)
}

impl BloomFilter {
    pub fn insert(&mut self, key: &[u8]) {
        let (h1, h2) = Self::double_hash(key);
        for i in 0..self.k {
            let bit = (h1.wrapping_add(i as u64 * h2)) % self.m as u64;
            let word = (bit / 64) as usize;
            self.bits[word] |= 1u64 << (bit % 64);
        }
    }

    pub fn may_contain(&self, key: &[u8]) -> bool {
        let (h1, h2) = Self::double_hash(key);
        for i in 0..self.k {
            let bit = (h1.wrapping_add(i as u64 * h2)) % self.m as u64;
            let word = (bit / 64) as usize;
            if self.bits[word] & (1u64 << (bit % 64)) == 0 {
                return false;  // DEFINITELY NOT HERE
            }
        }
        true  // maybe here (could be false positive)
    }

    fn double_hash(key: &[u8]) -> (u64, u64) {
        // Use two independent hash seeds (e.g. xxhash or siphash)
        let h1 = xxhash(key, 0);
        let h2 = xxhash(key, 1);
        (h1, h2)
    }
}
Concept 06
MVCC
Multi-Version Concurrency Control
TL;DR

Instead of locking data for reads, keep multiple versions. Readers see a consistent snapshot, writers add new versions. No reader ever blocks a writer.

The problem MVCC solves
Naive RwLock vs MVCC
Without MVCC (naive RwLock):
  Thread A: get("balance") -> holds read lock for 5ms
  Thread B: set("balance", 200) -> BLOCKED (waiting for read lock)
  Thread C: get("balance") -> BLOCKED (writer is queued, starves readers)
  -> One long read or one pending write stalls everything.

With MVCC:
  Thread A: takes snapshot at seq=1000 -> reads only seq<=1000 versions
  Thread B: set("balance", 200) -> writes seq=1001 -> NOT BLOCKED AT ALL
  Thread A: still reads balance at seq=503 (its snapshot is consistent!)
  Thread C: new read takes snapshot seq=1001 -> sees Thread B's write

Result: perfect isolation without any blocking between readers and writers.
Sequence numbers and internal key format
MVCC Internal Key Layout
Internal key format: (user_key, seq_no, type)
Sort order: user_key ASC, seq_no DESC  <- newest version comes first

MemTable state:
  ("alice", seq=1042, Value)     -> "35"   <- newest write, found first
  ("alice", seq=0503, Value)     -> "30"
  ("alice", seq=0200, Value)     -> "28"   <- oldest write
  ("bob",   seq=0800, Tombstone) -> ""     <- deleted (most recent op)
  ("bob",   seq=0100, Value)     -> "25"   <- pre-deletion value

READ: get("alice") with snapshot seq=600
  Seek to ("alice", MAX_SEQ) in BTreeMap
  Scan forward:
    seq=1042: 1042 > 600 -> SKIP (too new for this snapshot)
    seq=0503: 503 <= 600 -> VISIBLE, type=Value -> return "30"

Alice="35" is invisible to this reader. Perfectly consistent snapshot.
GC and snapshot bloat

The garbage collector tracks the oldest_snapshot watermark: the minimum seq_no held by any active reader. Versions below this watermark are invisible to all current and future readers.

MVCC Garbage Collection
Active snapshots: [seq=800, seq=950, seq=1001]
oldest_snapshot = 800

GC can safely remove:
  ("alice", seq=0200, Value)  <- seq 200 < 800, seq 503 is still needed
  ("alice", seq=0503, Value)  <- wait, seq 503 < 800 too, but seq 1042
                                 is above. Keep seq 503 as the "oldest
                                 visible" for snapshots at or above 503.

Rule: for each key, keep the newest version <= oldest_snapshot.
      Delete all older versions.

Tombstone cleanup:
  Can delete BOTH the tombstone AND all old values for a key
  ONLY when no snapshot can see the deleted key's old value.
  Safe when: tombstone seq < oldest_snapshot

PITFALL: Long-running snapshots (analytics queries, backups) prevent GC.
  PostgreSQL has the same problem with autovacuum and long transactions.
  Solution: warn users when snapshots are held too long. Force close if needed.
Concept 07
B-Tree vs LSM-Tree
When to use which storage model
TL;DR

B-Trees update data in-place (good for reads, bad for writes). LSM-Trees always append (great for writes, need compaction for reads). Your workload determines the winner.

B-Tree internals
B-Tree Structure and Operations
A B-Tree: balanced tree where every node is a disk page (4096 bytes)
Leaf nodes:     contain actual key-value pairs (sorted)
Internal nodes: contain keys + child pointers (fanout ~100-1000)

Tree height for 1 billion records: ~4 levels (log_100(1B) ~ 4.5)

READ "dave":
  Root: ["bob","frank"] -> "dave" is between -> middle child
  Middle: ["carol","eve"] -> "dave" is between -> leaf
  Leaf: scan for "dave" -> value
  Total: 3 disk reads. Predictable. Always O(log n).

UPDATE "dave" (change value):
  Find leaf: 3-4 reads (same as read)
  Modify the 4KB page IN-PLACE on disk: 1 random write
  If page is full: SPLIT -> update parent -> maybe split parent...
  Write amp: 2-5x on average, CAN spike on page splits

Random I/O on HDD: 5-10ms per seek = terrible write throughput
SSD: much better, but still slower than LSM's sequential writes
Copy-on-write B-Trees (LMDB): old pages stay valid -> readers are
  lock-free. Writers create new pages and update the root atomically.

Choosing the right model
Decision Guide
Choose LSM-Tree if:
  + Writes >= Reads (or mixed 50/50)
  + Sequential writes (logs, events, time-series)
  + Dataset >> Available RAM
  + Can tolerate occasional compaction pauses
  + Need keys to outlive RAM size
  -> RocksDB, Cassandra, TiKV, ScyllaDB, LevelDB

Choose B-Tree if:
  + Reads >> Writes
  + Need predictable latency (no compaction surprises)
  + Working set fits in OS page cache
  + Need ACID transactions (B-Tree locking is more mature)
  -> PostgreSQL, MySQL InnoDB, SQLite, LMDB

The Cementite path:
  Phases 1-3: Bitcask (simplest, teaches log + compaction fundamentals)
  Phases 4-7: LSM-Tree (teaches the dominant modern storage model)
  Phase 9+:   B-Tree is a great optional extension to compare against
Concept 08
Benchmarking Strategy
How to measure your DB honestly
TL;DR

A bad benchmark lies. Use YCSB workloads, measure p99 not average, and always compare against a known reference like sled or RocksDB.

The 3 numbers that matter
Core Benchmark Metrics
1. THROUGHPUT (ops/sec)
   -> Measure SUSTAINED (60+ seconds), not burst (1 second is a lie)
   -> Test: sequential write, random write, Zipfian read, mixed 50/50
   -> At: 1, 4, 16, 64 concurrent threads (to find contention cliffs)

2. LATENCY PERCENTILES (NEVER report just average!)
   p50   = median -- half of all requests are faster than this
   p99   = 1 in 100 requests -- this is your SLA boundary
   p99.9 = 1 in 1000 requests -- the worst day experience

   Example disaster: avg=0.5ms, p99=200ms -> you have a compaction stall!
   The average lies. The p99 reveals the truth.

3. AMPLIFICATION FACTORS
   Write amp = disk_bytes_written / user_bytes_written
   Read amp  = disk_bytes_read / user_bytes_read
   Space amp = actual_disk_size / logical_data_size

   Measure via /proc/self/io (Linux): read_bytes + write_bytes fields
   MUST call fsync() + db.flush() before reading post-write stats.
YCSB Workload Profiles
YCSB Standard Workloads
Key distributions:
  Uniform:  all keys equally likely -> unrealistic, every read = cache miss
  Zipfian:  top 100 keys get ~30% of requests -> realistic hot keys
  Latest:   reads skewed toward recently written keys

YCSB standard workloads:
  A: 50% read  / 50% update  (Zipfian) -> "balanced production"
  B: 95% read  /  5% update  (Zipfian) -> "read-mostly social app"
  C: 100% read               (Zipfian) -> "pure read ceiling"
  D: 95% read  /  5% insert  (Latest)  -> "read recent logs"
  F: 50% read  / 50% RMW     (Zipfian) -> "read-modify-write"

Cementite benchmark sequence:
  1. Sequential write 1M keys (peak write performance)
  2. Random write 1M keys (realistic write performance)
  3. Random read Zipfian keys (hit rate reveals cache effectiveness)
  4. Workload A at 1/4/16/32 threads (finds lock contention)
  5. Plot throughput over time (compaction stalls show up as dips)
  6. 100% read misses (measures Bloom filter effectiveness)
Expected results per phase
Phase-by-Phase Expected Performance
Phase 1 (HashMap):    2-5M ops/sec write  |  1-3M read  |  p99 < 5us
Phase 2 (Bitcask):    400-800K write       |  100-300K read  |  NOT crash-safe yet
Phase 3 (+compact):   same throughput      |  startup 100ms vs 10s w/o hints
Phase 4 (LSM basic):  600K-1M write        |  50-150K read (no Bloom = slow!)
Phase 5 (+Bloom):     same write           |  200-500K read  |  misses = 0 disk reads
Phase 6 (+WAL):       ~80% of Phase 5      |  p99 write 5-50us with group commit
Phase 7 (+MVCC):      slight overhead      |  reads non-blocking (key advantage)

vs production systems (Workload A, same hardware):
  Your Phase 5:   ~300K ops/sec
  sled (v0.34):   ~200-400K ops/sec
  RocksDB:        ~500K-1M ops/sec (years of production engineering)

Within 3-5x of RocksDB is EXCELLENT for a learning build.
Matching sled means you've built something genuinely useful.
