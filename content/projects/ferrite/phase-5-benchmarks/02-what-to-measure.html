Phase 5 · Benchmarking

What to Measure

Four benchmarks that cover the entire write and read path. Plus how to interpret what you find.

The 4 Core Benchmarks

1

Sequential Insert

Insert N rows with monotonically increasing keys (0, 1, 2, ...). New rows always append to the rightmost leaf. Minimal B-tree splits. Measures the raw pager + WAL write path.

O(n log n)
2

Random Insert

Insert N rows with random keys. Keys land in arbitrary positions, causing splits throughout the tree. Every insert is a potential O(log n) rebalance + possible cascade. Tests split correctness under load.

O(n log n)
3

Point Lookup

Find one row by primary key with a WHERE clause. Traverses from root to leaf: O(log n) comparisons. With a warm page cache (pages in memory), this should be extremely fast — no disk I/O.

O(log n)
4

Full Table Scan

SELECT * with no WHERE. Traverse all leaf pages from leftmost to rightmost, reading every row. Measures sequential leaf traversal efficiency and page cache behavior across all N rows.

O(n)

The WAL advantage is entirely on writes. Reads are identical because both modes serve reads from the page cache once warmed up. The fsync-per-commit cost dominates write throughput.

Insert Throughput: No WAL vs WAL Mode

The throughput difference is stark. WAL's sequential writes + single fsync give ~10× the inserts per second.

The "batched" row represents grouping multiple INSERTs into a single transaction. Each transaction still costs one fsync, so N inserts in one transaction costs the same as a single insert. This is the most impactful optimization available.

Profiling with cargo flamegraph

A flamegraph shows where your program spends time. The X-axis is time (wider = more time). The Y-axis is the call stack (higher = deeper into the call chain).

bench_insert_1000 (100%)

B-tree operations — 35% of time
WAL writes (pwrite + fsync) — 22%
SQL parsing — 15%
Misc / overhead — 10%

This flamegraph is expected. B-tree and WAL dominate. If SQL parsing takes >30% of total time, your parser is too slow — worth optimizing. If fsync alone is >40%, you are calling it too frequently.

Getting a Baseline: SQLite speedtest1

SQLite ships a reference benchmark called speedtest1. Compile it against the SQLite amalgamation to get a hardware-calibrated baseline for your specific machine.

# Download SQLite amalgamation (single-file distribution)
curl -O https://sqlite.org/2024/sqlite-amalgamation-3450000.zip
unzip sqlite-amalgamation-3450000.zip
cd sqlite-amalgamation-3450000

# Compile speedtest1 and SQLite together
gcc -O2 -o speedtest1 speedtest1.c sqlite3.c -lpthread -ldl

# Run the benchmark
./speedtest1 --size 5 --wal   # --wal enables WAL mode
./speedtest1 --size 5          # without WAL (rollback journal)

The --size N flag scales the number of rows. --size 5 is a quick run (~30 seconds). --size 100 gives more statistical confidence. Record the "total" line at the end — that's your hardware's baseline.

Tip: Your toy DB achieving 10–20% of SQLite's throughput is excellent. 5% is acceptable. Below 1% means something is fundamentally wrong (likely calling fsync way too often or O(n²) somewhere).

Interpreting Results: What's OK vs What Needs Work

Symptom Likely Cause Fix
O(n²) inserts Linear scan instead of binary search on leaf pages during insert. Or re-reading all pages from disk each insert. Binary search on leaf. Check page cache is hit, not cold-read every time.
Reads slower than expected despite warm cache Page cache too small — evicting pages between reads. Or not caching at all. Increase cache size. Profile with flamegraph — look for repeated pread calls.
WAL barely faster than no WAL Calling fsync too frequently — perhaps once per page write instead of once per transaction commit. Audit WAL write path. fsync only on commit frame, not on every frame append.
Point lookup O(n) instead of O(log n) Full table scan instead of B-tree traversal. WHERE clause not reaching the B-tree search. Verify query executor calls btree_search(key) not btree_scan_all().
High std deviation (±40%) OS scheduling noise. Or non-deterministic code path (hash map with random iteration, etc.). Pin to one core (taskset). Increase Criterion sample size. Check for rand usage.

Summary: The Measurement Checklist

Do These

  • ✓ Run all 4 benchmarks on every significant change
  • ✓ Compare WAL vs no-WAL throughput numbers
  • ✓ Verify O(log n) scaling with BenchmarkId
  • ✓ Generate flamegraph before optimizing
  • ✓ Calibrate against SQLite speedtest1

Avoid These

  • ✗ Optimizing before profiling (guessing)
  • ✗ Measuring debug builds (cargo bench = release)
  • ✗ Single-run wall-clock comparisons
  • ✗ Forgetting black_box() in benchmarks
  • ✗ Reporting numbers without confidence intervals

Realistic target: Toy DB with WAL: ~5,000–20,000 inserts/sec and >100,000 point lookups/sec. Below this? Check fsync frequency and page cache hit rate first.
