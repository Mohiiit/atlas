Benchmarking with Criterion

Statistical rigor over wall-clock guesswork. Criterion gives you confidence intervals, outlier detection, and HTML reports.

Why Criterion Over std::time

The naive approach — wrap code in Instant::now(), print the elapsed time, repeat a few times — is unreliable. Modern CPUs have branch predictors, instruction caches, frequency scaling, and OS scheduling noise. A single measurement tells you almost nothing.

std::time::Instant (naive)

Single measurement. No warmup. Compiler may optimize away the whole computation. OS scheduling noise ±20%. No baseline comparison. Numbers change between runs.

Criterion

Thousands of iterations. Warmup phase for CPU caches. Statistical outlier detection. Confidence intervals. HTML charts. Regression detection — alerts when code gets slower.

Key stat: Criterion runs your benchmark enough times that the mean has a 95% confidence interval. It then reports: "this function takes 1.23 µs ± 12 ns". That ± 12 ns is a real bound, not noise.

Setup: Cargo.toml

Add Criterion as a dev-dependency and declare the benchmark binary with harness = false (disables the default test harness so Criterion can run its own).

# Cargo.toml
[dev-dependencies]
criterion = { version = "0.5", features = ["html_reports"] }

[[bench]]
name = "db_benchmarks"    # → benches/db_benchmarks.rs
harness = false           # required for criterion

Create the file at benches/db_benchmarks.rs. Cargo auto-discovers files in benches/.

Writing Your First Benchmark

use criterion::{criterion_group, criterion_main, Criterion, BenchmarkId, black_box};
use ferrite::Database;

fn bench_insert(c: &mut Criterion) {
    c.bench_function("insert_1000_rows", |b| {
        b.iter(|| {
            let mut db = Database::open_temp(); // fresh in-memory or tmpfile
            for i in 0..1000 {
                db.execute(&format("INSERT INTO t VALUES ({}, 'user{}')", i, i))
                  .unwrap();
            }
            black_box(db) // prevent optimizer from discarding the result
        })
    });
}

criterion_group!(benches, bench_insert);
criterion_main!(benches);

The closure passed to b.iter() is what gets measured. Criterion calls it thousands of times, discards outliers, and computes statistics across the rest.

The black_box() Function

The Rust compiler is aggressive. If the result of a computation is never used, LLVM may optimize away the entire benchmark body — making it run in 0 nanoseconds and measuring nothing.

// BAD: compiler sees db is unused, may delete all 1000 inserts
b.iter(|| {
    let mut db = Database::open_temp();
    for i in 0..1000 { db.execute(...).unwrap(); }
    // db dropped here, no side effect observed
});

// GOOD: black_box creates an opaque "use" the compiler cannot reason through
b.iter(|| {
    let mut db = Database::open_temp();
    for i in 0..1000 { db.execute(...).unwrap(); }
    black_box(db)
});

Rule: Always wrap the final result (or any value you want to "observe") in black_box(). For benchmarks that return nothing, wrap any key intermediate value instead.

Parameterized Benchmarks with BenchmarkId

Use BenchmarkId to run the same benchmark with different input sizes. Criterion groups them together in the HTML report with a line chart showing how performance scales.

fn bench_insert_scaling(c: &mut Criterion) {
    let row_counts = [100, 1_000, 10_000, 100_000];
    let mut group = c.benchmark_group("sequential_insert");

    for &n in &row_counts {
        group.bench_with_input(BenchmarkId::from_parameter(n), &n, |b, &n| {
            b.iter(|| {
                let mut db = Database::open_temp();
                for i in 0..n {
                    db.execute(&format("INSERT INTO t VALUES ({}, 'v')", i))
                      .unwrap();
                }
                black_box(db)
            })
        });
    }
    group.finish();
}
