Pages and the Pager

The fundamental unit of database storage, and the abstraction layer that makes disk I/O feel like array access.

The Building Block

What is a page?

A page is a fixed-size chunk of bytes — SQLite defaults to 4096 bytes (4 KB). The entire database file is nothing more than a contiguous sequence of these same-size blocks numbered from zero. There are no variable-length sections, no gaps, no headers between pages. Just a clean array.

⚡
O(1) seek
Page N always starts at byte offset N × 4096. No scanning needed.
🧱
Matches the OS
4096 bytes is the kernel virtual memory page size. Reads align perfectly — no wasted I/O.
♻️
Simple recycling
Free pages go on a freelist. Any page can fill any slot — uniform size means no fragmentation math.

A database file is just an array of 4 KB pages on disk. Page 0 is special — it contains the database header (file format version, page size, root page numbers). Everything else is B-tree nodes or overflow data.

The Abstraction

What the Pager does

The B-tree layer thinks in pages, not files. It says "give me page 7" and gets a pointer to 4096 bytes. It never calls read() or write() directly. That's the Pager's job — it hides disk I/O behind a dead-simple interface.

↓
Read: get_page(page_num)
Check the in-memory cache. Cache hit? Return the cached pointer instantly — no I/O. Cache miss? Compute the file offset as page_num × PAGE_SIZE, seek to that position, read exactly 4096 bytes into a cache slot, return it.
◉
Cache: keep hot pages in memory
The cache is an array of up to 100 page slots. Once loaded, a page stays in memory until evicted or the file is closed. Subsequent reads of the same page are free — a pointer dereference, not a syscall.
↑
Write: flush dirty pages on close / commit
When the B-tree modifies a page, it's marked dirty in the cache — but not immediately written to disk. On flush() or db_close(), the Pager iterates dirty slots, seeks to the correct offset, and calls write(). This batching strategy is critical for performance.

Page Cache

Why we don't read from disk on every access

Disk I/O is ~1,000× slower than RAM access. A typical B-tree lookup traverses 3–4 levels. Without caching, each page access would be a separate pread() syscall. With a cache, traversing a warm tree costs only pointer arithmetic.
