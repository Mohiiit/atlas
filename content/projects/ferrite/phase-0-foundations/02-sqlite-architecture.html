SQLite's Architecture

Eight layers that take a SQL string and turn it into bytes on disk — and which ones we're actually building.

The Stack

8 layers, top to bottom

Every SQL statement travels through every layer in sequence. The front-end converts text to instructions; the back-end executes them against real storage.

01
Tokenizer
tokenize.c
Splits the raw SQL string into a flat stream of tokens — keywords, identifiers, literals, punctuation.
02
Parser
parse.y → lemon
Consumes tokens and builds an Abstract Syntax Tree (AST) using a generated Lemon parser. Catches syntax errors here.
03
Code Generator
select.c, insert.c…
Walks the AST and emits VDBE bytecode instructions. Also runs the query planner to pick optimal algorithms.
04
VDBE skip
vdbe.c
Virtual Database Engine — a register-based bytecode interpreter. We skip this; our engine interprets the AST directly.
05
B-Tree we build
btree.c
Organises rows into sorted tree nodes stored across pages. One B-tree per table, one per index.
06
Pager we build
pager.c
Manages fixed-size 4 KB pages: cache in memory, write dirty pages to disk, handle rollback and WAL.
07
VFS simplified
os_unix.c
Virtual File System abstraction — provides open, read, write, fsync across platforms. We use Rust's std::fs directly.
08
OS / Kernel
kernel syscalls
The actual file I/O via kernel syscalls. Below our abstraction boundary.

Why we skip the VDBE

SQLite's VDBE is a full register-based virtual machine with ~150 opcodes. Implementing it would be a curriculum unto itself. Instead, our Rust engine will walk the AST produced by the parser and dispatch directly to B-tree operations. This is a valid architecture — many embedded databases (and the original SQLite before version 2) work exactly this way. You lose the query caching benefit of pre-compiled bytecode, but you gain simplicity and a much shorter path to a working database.

Worked Example

Tracing INSERT INTO users VALUES (1, 'alice')

Follow one statement through every layer. Each arrow is a real function boundary in SQLite's source.

1
Tokenizer
Scans the string character by character. Emits tokens: TK_INSERT TK_INTO TK_ID("users") TK_VALUES TK_INTEGER(1) TK_STRING("alice"). Pushes each token to the parser immediately — no buffering.
2
Parser
The Lemon-generated parser matches the token stream against the SQL grammar. Constructs an AST node: Insert { table: "users", values: [1, "alice"] }. Rejects here if syntax is wrong.
3
Code Generator
Looks up "users" in the schema, confirms column types match, emits VDBE instructions like OpenWrite NewRowid MakeRecord Insert Halt.
4
VDBE we skip
Executes each opcode in sequence, maintaining registers and a program counter. Calls into the B-tree layer when it needs to actually store data.
5
B-Tree
Finds the correct leaf node for rowid=1, serialises the row into bytes, inserts into the page. If the page is full, splits the node (this is the complex part we'll implement carefully).
6
Pager
The B-tree requested page N. Pager checks its cache — cache miss, so reads from disk. Marks the page as dirty after the write. On COMMIT, flushes dirty pages to disk in the correct order.
7
VFS → OS
Pager calls sqlite3OsWrite() which maps to pwrite(fd, buf, 4096, offset). Then fsync() to flush kernel buffers to physical storage — this is what makes Durability real.
