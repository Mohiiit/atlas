Why Databases Exist

From chaotic flat files to guaranteed data integrity — and where SQLite fits in the landscape.

The Problem

Why not just use a CSV file?

Flat files seem simple enough — rows of text, one record per line. But the moment two processes touch the same file, or the program crashes mid-write, things unravel fast. There's no concept of a transaction, no protection against partial writes, and no way to query efficiently beyond reading every line.

CSV / Flat File
id,name,balance
1,alice,1000
2,bob,500
2,bob,1500 ← crash here
1,alice,0 ← already written

⚠ bob lost $500, alice lost $1000
Database
BEGIN;
  UPDATE accounts
    SET balance = 0
    WHERE id = 1;
  UPDATE accounts
    SET balance = 1500
    WHERE id = 2;
COMMIT; ← or ROLLBACK
Core Guarantees

ACID — what a database actually promises you

ACID is the contract a database makes with your application. Violate any of these and you're back to CSV territory.

A
Atomicity
A transaction is all-or-nothing. Either every operation in the batch succeeds, or none of them do — no partial states.
Transfer $500: alice → bob
debit(alice, 500) ✓
credit(bob, 500) ✗ crash
→ both rolled back
C
Consistency
A transaction moves the database from one valid state to another. Rules like "balance ≥ 0" or unique constraints are never violated.
Constraint: balance >= 0
alice.balance = 100
debit(alice, 200)
→ rejected, rule broken
I
Isolation
Concurrent transactions behave as if they ran one at a time. You never see another transaction's half-finished work.
Two transfers at once:
T1: reads alice = 1000
T2: reads alice = 1000
T1 writes 500, T2 writes 500
→ isolated: one waits
D
Durability
Once a transaction commits, it stays committed — even if the power goes out a millisecond later. Data is on disk.
After COMMIT:
write to WAL log ✓
fsync() to disk ✓
power loss ...
→ data survives reboot
Two Worlds

OLTP vs OLAP — the great divide

Not all databases are built for the same job. The workload shape determines everything: how data is laid out on disk, what indexes matter, and what "fast" even means.

OLTP

Online Transaction Processing

Many short, targeted reads and writes. "Get me user 42's balance." "Insert this order." Row-oriented storage. SQLite lives here.

row-store low latency high concurrency SQLite · PostgreSQL · MySQL
OLAP

Online Analytical Processing

Few large queries scanning millions of rows. "Sum all sales in Q3 by region." Column-oriented storage for compression and vectorized scans. DuckDB lives here.

column-store high throughput batch reads DuckDB · Snowflake · ClickHouse

This curriculum is OLTP territory. We're building a row-store B-tree engine — exactly what SQLite is.

Our Target

Where SQLite fits

SQLite is the most widely deployed database engine in existence — more than MySQL, PostgreSQL, and Oracle combined. Its design choices are radically different from server databases, and they're exactly what make it a perfect teaching subject.
