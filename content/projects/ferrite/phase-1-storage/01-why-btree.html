Why B-Trees?

Every database picks a core data structure. Understanding what goes wrong with simpler choices explains exactly why B-Trees dominate disk-based storage.

The Alternatives and Their Problems
📋

Sorted Array

Binary search gives fast lookups, but every insert must shift all elements after the insertion point. Deletes are equally brutal.

Insert: O(n) shifts
#

Hash Map

O(1) lookups are great, but hash maps are fundamentally unordered. Range queries like WHERE id BETWEEN 100 AND 200 require a full scan.

Range query: O(n)
🌲

Binary Search Tree

Ordered and fast in theory. But sequential inserts produce a degenerate chain — every node has one child and height becomes O(n).

Worst case: O(n) depth

The Core Insight
📀

One Node = One Page

Each B-Tree node maps exactly to a 4096-byte disk page. A single disk read fetches an entire node. No partial reads, no wasted I/O.

⚖️

Self-Balancing

The split algorithm guarantees every leaf stays at the same depth. Height is always O(logb n) where b is the branching factor — typically 100–500.

🔗

Leaf Chaining

In a B+ Tree, leaves are linked in sorted order. Range scans become sequential reads — the most efficient access pattern possible on spinning disk and SSD alike.
