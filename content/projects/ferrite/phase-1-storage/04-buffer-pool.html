Buffer Pool Manager

Disk I/O is the bottleneck in every database. The buffer pool is the layer that makes disk-based data structures feel fast by keeping hot pages in memory.

The Latency Cliff Between Memory and Disk
L1 cache
~1ns
L3 cache
~10ns
DRAM (memory)
~100ns
NVMe SSD
~100µs
HDD (spinning)

The rule: A single disk read costs as much as ~100,000 memory reads. Every extra page fault is catastrophic. The buffer pool's job is to make most reads a cache hit.

Key invariants: (1) A page with pin_count > 0 can never be evicted — a thread is actively reading/writing it. (2) A dirty page must be flushed to disk before its frame is reused — otherwise we lose committed writes.
Page Replacement: LRU and the Clock Algorithm

When pool is full, evict the tail (least recently accessed). On every access, move that frame to the head. Uses a doubly-linked list + hash map. O(1) access and eviction.

Clock Algorithm

A cheaper LRU approximation using a circular buffer and a single "reference bit" per frame. A clock hand sweeps around:

  1. 1.If ref_bit = 1: clear it, advance hand (give a second chance)
  2. 2.If ref_bit = 0: evict this frame (not recently used)
  3. 3.On page access: set ref_bit = 1
Tradeoff: No linked list, no hash map updates on every access — just a single bit flip. Used by PostgreSQL's buffer manager.

Our Toy Pager in Rust

For Phase 1 we keep it simple: a fixed array of Option<Box<[u8; PAGE_SIZE]>>. No LRU yet — eviction is deferred until we need it. Pages are flushed on .exit.

const PAGE_SIZE:    usize = 4096;
const TABLE_MAX_PAGES: usize = 100;

struct Pager {
    file:       File,
    file_len:   u64,
    num_pages:  u32,
    pages:      [Option<Box<[u8; PAGE_SIZE]>>; TABLE_MAX_PAGES],
}

impl Pager {
    fn get_page(&mut self, page_num: usize) -> &mut [u8; PAGE_SIZE] {
        if self.pages[page_num].is_none() {
            // Cache miss: allocate and load from disk
            let mut page = Box::new([0u8; PAGE_SIZE]);
            let offset = (page_num * PAGE_SIZE) as u64;
            if offset < self.file_len {
                self.file.seek(SeekFrom::Start(offset)).unwrap();
                self.file.read_exact(page.as_mut()).unwrap();
            }
            self.pages[page_num] = Some(page);
        }
        self.pages[page_num].as_mut().unwrap()
    }

    fn flush(&mut self, page_num: usize) {
        if let Some(page) = &self.pages[page_num] {
            let offset = (page_num * PAGE_SIZE) as u64;
            self.file.seek(SeekFrom::Start(offset)).unwrap();
            self.file.write_all(page.as_ref()).unwrap();
        }
    }
}
What we skip for now: dirty tracking, LRU eviction, pin counts, and concurrent access. Each of these becomes necessary as the database grows. Real buffer pool managers (PostgreSQL, InnoDB) have thousands of lines dedicated to these concerns.
Frame Metadata Summary

Dirty Bit

Set when a write modifies the in-memory frame. Cleared after a successful flush. An eviction of a dirty page without flushing = data loss.

Pin Count

Incremented when a thread acquires the frame, decremented on release. Only frames with pin_count = 0 are eviction candidates.

Page Table

Maps logical page_id → physical frame_id. On a cache hit, this is the only lookup needed. On miss, the table is updated after loading.

Free List

Tracks which frames are currently unpopulated. On cache miss, check the free list first before invoking the eviction policy.
