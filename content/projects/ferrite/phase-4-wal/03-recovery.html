Crash Recovery with WAL

On startup, scan the WAL, find the last valid commit, replay committed frames, discard the rest.

Recovery on Startup: Scan the WAL

When your database opens, it checks if a WAL file exists. If it does, a crash may have occurred. The recovery algorithm scans the WAL frame by frame from the beginning, validating each frame's checksum. Frames with valid checksums up to the last commit frame are replayed into memory. Everything after the last valid commit is discarded.

How to Detect a Valid Frame

Each WAL frame carries a cumulative checksum. The checksum of frame N covers all bytes in frames 1 through N. This means a frame can only have a valid checksum if all previous frames were also written correctly.

  1. Read the WAL header — extract salt-1 and salt-2. These are random values written when the WAL was initialized. They change at each checkpoint, invalidating old WAL frames from a previous generation.

  2. For each frame: verify the frame's salt values match the WAL header salts. Mismatch → this is a stale frame from a previous WAL generation. Stop.

  3. Compute the running checksum over the frame header + page data. Compare to the checksum stored in the frame header.

  4. Mismatch → frame is corrupt or incomplete. Stop scanning. All frames from here to end of file are discarded.

  5. If the frame has db_size != 0, it is a commit frame. Record this as the latest valid commit point.

Why cumulative checksums? If a frame in the middle is corrupted, all subsequent checksums will also fail. You can't accidentally "skip" a corrupt frame and replay later frames — the chain of checksums enforces linear validity.

The ARIES Protocol (Simplified)

ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) is the industrial-strength recovery algorithm used by IBM DB2, SQL Server, and others. It operates in three phases:

1

Analysis

Scan the log forward from the last checkpoint. Identify which pages were dirty (modified but not yet flushed to disk) at crash time. Identify which transactions were still active (uncommitted) at crash time.

2

Redo

Replay ALL logged operations from the earliest dirty page — including operations from uncommitted transactions. This restores the exact state the database was in at the moment of crash, including in-flight work.

3

Undo

Roll back all transactions that were still active at crash time (they were never committed). Apply their inverse operations in reverse order. Write compensation log records (CLRs) so that undo itself survives crashes.

Analysis Phase — forward scan

Build the "dirty page table" and "active transaction table" from the checkpoint log record forward to the end of the log.

Redo Phase — forward scan from earliest dirty page

For each log record: apply the operation to the page if the page's LSN is older than the log record's LSN. Redo is idempotent — safe to re-apply.

Undo Phase — backward scan through active transactions

For each active transaction: process its log records in reverse order, apply the inverse operation. Write a Compensation Log Record (CLR) that "undoes the undo" if we crash again during recovery.

For Our Toy DB: Simplified Recovery

Full ARIES is complex. For a toy database with WAL and no concurrent transactions, a much simpler approach works correctly:

  1. Open the WAL file. If it doesn't exist, no recovery needed — open DB normally.

  2. Scan WAL frames in order. Validate each frame's salt and checksum. Stop at first invalid frame.

  3. Find the last commit frame (highest index frame with db_size != 0 and valid checksum).

  4. Replay all frames from 1 to (inclusive) the last commit frame into the page cache. These are all committed data.

  5. Discard all frames after the last commit frame — these were in-flight transactions that never committed.

  6. Proceed with normal operation. The first checkpoint will flush the replayed pages to the DB file.

Why this is safe: We only replay committed frames. The database never sees partial transactions. The WAL's checksum chain guarantees we stop at exactly the right frame. No undo phase needed because we never apply uncommitted data.

Checkpointing Strategy

Without checkpointing, the WAL grows forever and read performance degrades (every page lookup scans more WAL frames). The checkpoint policy controls this tradeoff.

SQLite Default

Checkpoint when WAL exceeds 1000 frames (~4 MB). Automatic, triggered on any write commit that pushes WAL past the threshold.

Toy DB Recommendation

Checkpoint every 100 frames to keep WAL small and reads fast. Tune upward once the write path is proven correct.
