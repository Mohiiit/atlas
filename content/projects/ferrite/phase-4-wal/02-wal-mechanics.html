How Write-Ahead Logging Works

Sequential appends, deferred DB writes, and the one rule that makes it all safe.

"Write to the log before writing to the database."

WAL File Structure

The WAL file is a simple append-only sequence of frames. Each frame contains a header describing which database page it holds, followed by a full copy of that page.

A frame is a commit frame when db_size != 0. This marks the end of a committed transaction. Frames after the last valid commit checksum are ignored on recovery.

How a Write Works

The original database file is never touched during a transaction. All modifications go to the WAL first.

  1. Accumulate modified pages in memory (the write buffer).

  2. Append each modified page as a new frame to the WAL file. Use O_APPEND — sequential writes only.

  3. Write a commit frame with db_size set. This frame's checksum covers all previous frames in the transaction.

  4. fsync(wal_fd) — one single fsync makes the entire transaction durable. If crash happens before this: transaction never happened (WAL frames are incomplete). If crash after: frames are valid and will be replayed.

  5. The original DB file is unchanged. Readers still see the pre-transaction state from the DB file OR can read new data from the WAL.

Key insight: One fsync per commit. Rollback journal needs two. This is why WAL mode achieves ~10× the insert throughput in benchmarks.

How a Read Works

Reads must check the WAL first. A reader always sees a consistent snapshot by setting an "end mark" at the last committed frame at the time the read begins.

  1. Record the current WAL end mark: the index of the last committed frame visible to this reader.

  2. For each page requested: scan the WAL index (in memory) for the latest frame for that page number, up to the end mark.

  3. If found in WAL: read that frame's page data. This is the current version.

  4. If not found in WAL: read from the DB file. This is older but still valid (WAL hasn't checkpointed yet).

The WAL index (a memory-mapped .shm file) maps page numbers to their most recent WAL frame offset. Without it, every read would scan the entire WAL linearly — O(n) per lookup.

Checkpointing: Copying WAL Back to DB

The WAL grows with every write. Eventually it must be transferred back to the main database file and truncated. This is a checkpoint.

  1. Wait until all readers have end marks past the frames being checkpointed (no reader is still reading those frames).

  2. Copy WAL frames to their corresponding page offsets in the DB file (random writes, but that's OK — this is a background operation).

  3. fsync the DB file.

  4. Truncate the WAL file back to zero (or reuse it — WAL reuse avoids filesystem metadata operations).

SQLite default: checkpoint automatically when WAL grows past 1000 frames (~4 MB). For a toy DB, checkpoint after every 100 frames to keep things simple.

Reader/Writer Concurrency

This is WAL's killer feature over rollback journal. Readers and writers can operate simultaneously with zero blocking.

Multiple Readers

Each reader sets its own end mark at the last committed frame when it starts. Readers see a stable snapshot. A writer appending new frames doesn't affect any existing reader's end mark. Zero locks needed for reads.

Single Writer

Only one writer can append to the WAL at a time — a simple mutex. The writer never touches the DB file during the transaction. It doesn't block readers reading old data from the DB file.

Rollback Journal Concurrency

Writer takes an exclusive lock on the entire DB. All readers block until commit. Serialized access. OLTP killer.

WAL Concurrency

Readers and writers never block each other. Multiple concurrent SELECT queries while INSERT is running. OLTP friendly.

Why WAL Is Faster

Random DB File Writes

Rollback journal writes modified pages back to their positions in the DB file. Position = page_num × 4096. Random seek + write for each modified page. SSD handles this better than HDD, but it's still scattered.

Sequential WAL Appends

WAL writes are always appended to the end of the WAL file. No seeking. Sequential I/O is the fastest pattern for both HDD and SSD. The OS can also batch consecutive small writes into fewer I/O operations.

Typical numbers: Rollback journal — ~5,000 inserts/sec. WAL mode — ~50,000–80,000 inserts/sec on the same hardware. The 10× gap comes from sequential vs random writes and 1× vs 2× fsync.
