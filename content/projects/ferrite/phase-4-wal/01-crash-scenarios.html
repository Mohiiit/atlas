Phase 4 · Crash Safety

What Goes Wrong Without Crash Safety

A single power cut can leave your database permanently corrupted. Here's exactly how it happens.

The Core Problem: Writes Are Not Atomic

A single logical operation — like inserting one row — may require updating multiple pages on disk. If the process dies between any two of those writes, the database ends up in an inconsistent state that no code can automatically repair.

This isn't a rare edge case. Crashes happen: power failures, OOM kills, OS panics, runaway signals. Your database must survive all of them.

Three Concrete Crash Scenarios

Scenario 1 — Partial Multi-Page Update

Inserting a row into a full leaf requires updates to: the leaf page, the parent page, and possibly the root. Three disk writes. One crash window between each.

Scenario 2 — Leaf Split Orphan

A B-tree leaf split allocates a new page and writes it, then must update the parent to add the new separator key. Crash between those two steps creates an orphaned page.

Page 12 exists on disk but no parent points to it. It's lost space. Worse: the split key range is unreachable — rows in that range silently disappear.

Scenario 3 — The Torn Write

A 4 KB page write is not atomic at the OS level. The OS writes sectors (512 B or 4096 B each). If power fails mid-page, only the first N sectors land on disk. You get half the old page and half the new page — a "torn write".

Checksums in the page header will fail, but only if you compute them. Without checksums, this silent corruption goes undetected for months.

fsync: The Brutal Hammer

Without fsync(), writes sit in the OS page cache — fast but volatile. A crash before the OS flushes loses them. fsync() blocks until the storage controller confirms the data is on physical media.

Without fsync

Write goes to page cache. Returns in ~1 µs. Crash → data gone. Fast but unsafe.

With fsync

OS flushes to disk. Blocks for 1–10 ms on SSD, 5–20 ms on HDD. Safe but expensive.

Implication: You cannot call fsync after every single write operation — that limits you to ~100 writes/sec on spinning disk. The whole art of durability engineering is minimizing fsync calls while preserving correctness.

Two Classic Approaches

ApproachStrategyDurabilityWrite Pattern
Rollback Journal Save original pages to journal BEFORE overwriting. On crash, restore from journal. Strong (2× fsync per txn) Random writes to DB file
Write-Ahead Log Append modified pages to WAL BEFORE updating DB. DB file unchanged until checkpoint. Strong (1× fsync per txn) Sequential writes to WAL

Rollback Journal — How It Works

1
Copy original page(s) into db.journal file
2
fsync the journal — it is now durable
3
Overwrite pages in the main DB file
4
fsync the DB file — transaction committed
5
Delete journal file (commit point)
↩
If journal exists at startup → crash recovery: restore pages from journal, delete journal

The Tradeoff: Rollback journal requires 2× fsync per commit and random writes to the DB file. WAL requires only 1× fsync and uses sequential writes — much better for throughput. But WAL needs a checkpoint process and has more complex reader/writer coordination.
