The Query Execution Pipeline

From SQL text to result rows — understanding the 4-stage pipeline and the Volcano iterator model.

The Four Stages

01
Parse
SQL text → AST. Tokenize, validate syntax.
02
Plan
AST → Execution plan. Decide how to retrieve rows.
03
Execute
Walk the plan tree, call storage APIs, yield rows.
04
Results
Collected rows formatted and returned to caller.

Our Trivial Planner

In Phase 3, the planner is intentionally naive — its entire job is to map statement type to an operator tree:

  • SELECT → always a full table scan (no index optimization yet)
  • INSERT → B-tree insert at the correct leaf position
  • No join planning, no cost estimation, no index selection

Real databases (like toydb) add constant folding, filter pushdown, and index scan replacement. We'll get there — but first, correctness over cleverness.

The Executor Trait

/// Every operator in the plan implements this trait.
/// Pull one row at a time — None signals end of stream.
trait Executor {
    fn next(&mut self) -> Option<Row>;
}

/// Collecting all results is just looping next()
fn collect_all(exec: &mut dyn Executor) -> Vec<Row> {
    let mut rows = Vec::new();
    while let Some(row) = exec.next() {
        rows.push(row);
    }
    rows
}

The Volcano / Iterator Model

Each operator is an iterator. Operators nest inside each other — the parent calls next() on its child, which calls next() on its child, and so on down to the storage layer. No intermediate result buffers needed.

Projection outermost
Receives full rows, strips to only requested columns. Calls Filter.next()
Filter
Calls Scan.next() in a loop, skips rows that fail the predicate, returns matching rows
Scan innermost
Holds a cursor into the B-tree, advances one cell per next() call, reads raw bytes

Composition in Practice

/// SELECT id FROM users WHERE age > 18
/// builds this operator tree at plan time:

let plan = Project {
    columns: vec!["id"],
    source: Box::new(Filter {
        predicate: Predicate::Gt("age".into(), Value::Int(18)),
        source: Box::new(Scan {
            table: "users".into(),
            cursor: None, // initialized on first next()
        }),
    }),
};

// Execution: Project.next() → Filter.next() → Scan.next() → B-tree

Why this model is elegant

  • Operators are composable — any operator can wrap any other
  • Lazy evaluation — rows flow one at a time, no large intermediate buffers
  • Early termination — LIMIT 10 just stops calling next() after 10 rows
  • Adding a new operator (Sort, Join, Limit) means implementing one trait method
