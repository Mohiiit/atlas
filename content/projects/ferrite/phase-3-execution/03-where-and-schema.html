WHERE Clause & Table Schema

How predicate trees are evaluated against rows, and where schema definitions live in the database file.

Table Schema in Rust

/// One column definition inside a table
struct ColumnDef {
    name:     String,
    col_type: ColType,
}

/// The three primitive types we support in Phase 3
enum ColType { Integer, Text, Boolean }

/// Full schema: table name + ordered list of columns
struct TableSchema {
    name:    String,
    columns: Vec<ColumnDef>,
}
/// Column index = position in Vec → byte offset in row

Example schema for a users table:

ColumnTypeDescriptionVec index
idIntegerPrimary key, monotonically increasing0
nameTextFixed-length padded string (32 bytes)1
ageInteger4-byte signed integer2
emailTextFixed-length padded string (64 bytes)3

Schema Storage: The Catalog Page

The schema is not hardcoded — it lives in the database file itself. Page 0 is reserved as the catalog page (similar to SQLite's sqlite_schema table). It stores serialized TableSchema entries for every table in the database.

  • On CREATE TABLE, serialize the schema and write it to the catalog page
  • On database open, read page 0 first to load all schema definitions into memory
  • The planner uses in-memory schema to resolve column names → column indexes before execution
  • Phase 3 simplification: we may keep schema in a static HashMap for now

The Predicate Type

/// A predicate tree — mirrors the WHERE clause AST
enum Predicate {
    Eq(String, Value),                          // col = value
    Gt(String, Value),                          // col > value
    Lt(String, Value),                          // col < value
    And(Box<Predicate>, Box<Predicate>),      // left AND right
    Or(Box<Predicate>, Box<Predicate>),       // left OR right
}

enum Value { Int(i64), Text(String), Bool(bool) }

Evaluating a Predicate

fn eval(pred: &Predicate, row: &Row, schema: &TableSchema) -> bool {
    match pred {
        Predicate::Eq(col, val) => {
            let idx = schema.col_index(col);   // "age" → 2
            row.get(idx) == val               // compare stored value
        }
        Predicate::Gt(col, val) => {
            let idx = schema.col_index(col);
            row.get(idx) > val
        }
        Predicate::And(left, right) =>
            // short-circuit: skip right if left is false
            eval(left, row, schema) && eval(right, row, schema),
        Predicate::Or(left, right) =>
            eval(left, row, schema) || eval(right, row, schema),
        _ => unimplemented!()
    }
}

Schema Lookup: Name → Byte Offset

To evaluate age > 18, the executor must know where age lives in the raw row bytes. The schema provides the answer in two steps:

Column name
"age"
→
Vec index
2
→
Byte offset in row
sum of preceding\ncolumn sizes

For a row layout of [id:4][name:32][age:4][email:64], the byte offset of age = 4 + 32 = 36. The schema makes this calculation automatic — no hardcoded offsets in executor code.

The Filter Operator in Context

struct Filter {
    predicate: Predicate,
    schema:    TableSchema,
    source:    Box<dyn Executor>,
}

impl Executor for Filter {
    fn next(&mut self) -> Option<Row> {
        loop {
            let row = self.source.next()?;  // None propagates EOF
            if eval(&self.predicate, &row, &self.schema) {
                return Some(row);           // passes filter → yield it
            }
            // else: loop, pull next row from source
        }
    }
}

What we are NOT building yet

  • Query optimization: predicates are always evaluated post-scan; no filter pushdown into the B-tree
  • Index selection: the planner never picks an index scan based on WHERE column — always full scan
  • Statistics: no row count estimates, no cost-based decisions
  • These are left for later phases — correctness before performance
