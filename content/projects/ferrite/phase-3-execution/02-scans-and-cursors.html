Scans, Cursors & Projections

How a cursor abstracts B-tree position, and how full scans differ from key lookups.

The Cursor Abstraction

A cursor marks a position inside a table's B-tree. All execution operators talk to cursors, never directly to pages or cells. This is what lets the same execute_select() logic work whether the storage is a flat array or a multi-level B-tree.

table
*mut Table
Raw pointer back to the owning table — lets cursor functions navigate pages
page_num
usize
Which page (node) the cursor is currently positioned on
cell_num
usize
Which cell (key-value slot) within that page the cursor points to
end_of_table
bool
True when the cursor has moved past the last row — signals EOF to the executor
struct Cursor {
    table:        *mut Table,
    page_num:     usize,
    cell_num:     usize,
    end_of_table: bool,
}

impl Cursor {
    /// Read the row bytes at the current position
    fn value(&self) -> &[u8] { /* → cell payload bytes */ }

    /// Move to the next cell; set end_of_table when past last leaf cell
    fn advance(&mut self) { /* → increment cell_num, carry to next page */ }

    /// Position at leftmost leaf, cell 0 — ready for full scan
    fn table_start(table: *mut Table) -> Self { /* → walk left-spine from root */ }

    /// Position one past the last row — used by INSERT
    fn table_end(table: *mut Table) -> Self { /* → rightmost leaf, last cell + 1 */ }
}

  • Full scan is O(n) — visits every leaf cell. Used for all non-PK WHERE clauses in Phase 3.
  • Point lookup is O(log n) — follows the B-tree path from root to one specific leaf cell.
  • The same Cursor struct is used for both — only how it is initialized differs.

Index Scan (Preview)

A secondary index is a separate B-tree keyed on a non-primary column. An index scan on WHERE email = 'x' works in two hops:

  • Lookup email = 'x' in the index B-tree → get the primary key (e.g., id = 42)
  • Use that primary key to do a point lookup in the main table B-tree → get the full row
  • This avoids scanning every row, but requires maintaining two B-trees on write
  • We don't build secondary indexes in Phase 3 — this is context for later phases

Column Projection

After fetching a full row from the cursor, the Projection operator strips it down to only the requested columns. This happens at execution time, not storage time — the B-tree always stores complete rows.

Full row from cursor
id: 42
name: "alice"
age: 28
email: "a@b.c"
bio: "..."
→
SELECT id, name
id: 42
name: "alice"
impl Executor for Project {
    fn next(&mut self) -> Option<Row> {
        let full_row = self.source.next()?;
        // SELECT *  → return all columns unchanged
        // SELECT id → return only the column at schema index 0
        Some(self.project_columns(full_row))
    }
}
