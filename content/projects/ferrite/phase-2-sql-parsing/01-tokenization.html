SQL Tokenization

How a raw SQL string becomes a structured stream of tokens — the first stage of every SQL parser.

The Lexer Pipeline

From Characters to Tokens

A lexer (or tokenizer) is the front door of any compiler or parser. It reads a raw string character by character and groups those characters into meaningful units called tokens. The parser never sees raw text — it only works with the token stream the lexer produces.

The lexer handles whitespace, recognises reserved keywords, distinguishes identifiers from literals, and flags unknown input early — before the parser ever runs.

Rust Implementation

The Token Enum

In Rust, every token variant is represented as an enum. Variants that carry data — identifiers, integers, string literals — hold their value inline:

/// Every lexeme the lexer can produce
enum Token {
    // Keywords
    Select, From, Where,
    Insert, Into, Values,
    Create, Table,

    // Identifiers & literals
    Ident(String),
    Integer(i64),
    StringLit(String),

    // Symbols
    Star, Comma, Eq,
    LParen, RParen, Semicolon,

    Eof, // signals end of input
}

Keywords are unit variants — they carry no extra data because the variant name already conveys the meaning. Ident and StringLit wrap a String; Integer wraps an i64.

Implementation Sketch

A Simple Rust Lexer

The core loop: consume characters, classify them, emit tokens. Keywords are detected after scanning an identifier by checking against a lookup table.

struct Lexer {
    input: Vec<char>,
    pos: usize,
}

impl Lexer {
    fn next_token(&mut self) -> Token {
        self.skip_whitespace();
        match self.peek() {
            None       => Token::Eof,
            Some('*') => { self.advance(); Token::Star },
            Some(',') => { self.advance(); Token::Comma },
            Some('=') => { self.advance(); Token::Eq },
            Some('(') => { self.advance(); Token::LParen },
            Some(')') => { self.advance(); Token::RParen },
            Some(';') => { self.advance(); Token::Semicolon },
            Some('\'') => self.scan_string(),
            Some(c) if c.is_ascii_digit() => self.scan_integer(),
            Some(c) if c.is_alphabetic() || *c == '_' => self.scan_word(),
            _ => panic!("unexpected character"),
        }
    }

    fn scan_word(&mut self) -> Token {
        let word = self.take_while(|c| c.is_alphanumeric() || c == '_');
        match word.to_uppercase().as_str() {
            "SELECT"  => Token::Select,
            "FROM"    => Token::From,
            "WHERE"   => Token::Where,
            "INSERT"  => Token::Insert,
            "INTO"    => Token::Into,
            "VALUES"  => Token::Values,
            "CREATE"  => Token::Create,
            "TABLE"   => Token::Table,
            _         => Token::Ident(word),
        }
    }
}

The keyword check happens after scanning the full word — so selects is correctly classified as an identifier, not the keyword SELECT.

Key Insight

Why Lex Before Parsing?

Separating lexing from parsing is the canonical two-stage compiler design. It keeps each stage simple and focused. The lexer deals with characters and whitespace; the parser deals only with typed tokens and grammar rules.

In SQLite's source, this is described as the "front-end SQL compiler" that transforms raw text into an internal representation before the virtual machine ever sees it. Reducing complexity in each component also makes errors easier to locate and report.
