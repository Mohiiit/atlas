Madara Storage Proofs
Snapshots + Trie Logs Visual Deep Dive

content-rich explainer: RocksDB internals, Bonsai path mechanics, Starknet proof retrieval

This page focuses on one hard operational question: how does a Madara node prove historical state at block B while the live database keeps moving forward to newer blocks? The answer is a coordinated system: RocksDB point-in-time snapshots provide a stable base, Bonsai trie logs provide reversible deltas, and the RPC method composes proof nodes into a witness that a verifier can recompute against the declared roots.

1 — Why this matters

  • Historical integrity: Light clients, bridges, indexers, and auditors need cryptographic answers for old blocks, not only latest.
  • Operational survivability: If proof generation for older blocks is too expensive, a public node becomes vulnerable to resource exhaustion.
  • Correctness boundary: A proof is only valid if it corresponds to the exact roots associated with the requested block hash.
  • Tunable tradeoff: Snapshot frequency and trie-log retention directly control proof latency, storage footprint, and historical window length.

In practical terms, proofs are not a feature you can bolt on after sync. The data-retention policy must be designed from the start. Madara exposes this through --db-max-saved-trie-logs, --db-max-kept-snapshots, --db-snapshot-interval, and --rpc-storage-proof-max-distance. Each parameter shifts the boundary between speed, cost, and exposure.

2 — First principles

Principle A: RocksDB stores bytes, not semantics.

RocksDB does not know what a contract nonce or class hash is. It only stores key/value byte slices and maintains version visibility through sequence numbers. Application semantics are encoded by Madara/Bonsai key construction, then interpreted above the storage layer.

This explains why the same RocksDB engine can serve wildly different applications. Meaning is in key schema and access patterns, not in the engine itself.

Principle B: Historical proofs need historical views.

A storage proof is valid only if each node hash recomputes to the root associated with block B. Using current head state to answer old requests is wrong unless head and B are identical. Therefore, the node must reconstruct a point-in-time trie view.

Madara obtains this with a nearest snapshot plus trie-log transitions, then runs multiproof extraction from that reconstructed state.

3 — Mechanism walkthrough

  1. Request gate: RPC receives block_id plus any combination of class hashes, contract addresses, and per-contract storage keys. It checks distance, key count, and used-tries limits.
  2. Transactional state creation: For each relevant trie family, Madara asks Bonsai for get_transactional_state(block_n). Bonsai asks backend for closest snapshot at-or-after requested block.
  3. Delta replay: Trie logs between snapshot id and requested id are replayed into a transactional overlay. This reconstructs a deterministic point-in-time trie view.
  4. Proof extraction: Multiproof generation traverses requested paths and returns hash->node mappings (binary + edge nodes) for the union of paths.
  5. Leaf metadata assembly: Contract leaves data is returned with nonce/class_hash/storage_root so verifiers can rebuild contract-leaf hash inputs.
  6. Response binding: Output includes global_roots plus associated block_hash, so verifiers know exactly which header commitment this witness targets.

Compaction interaction is subtle but important: old key versions remain queryable while live snapshots need them. Once no active snapshot references older versions, compaction can drop them. This is why snapshot lifecycle directly affects historical visibility guarantees.

4 — Worked examples

Simpler example (toy trie): use 6-bit paths so we can see every bit. This is only for readability; real Starknet paths are 251 bits, but the logic is identical.

  • k1 = 000010 -> value 7
  • k2 = 000110 -> value 9
  • k3 = 000011 -> not inserted (used for non-membership proof)
  • Shared prefix of inserted keys: 000 (then split: k1=010, k2=110)

  B0 -->|1| E2["Edge E2: path=10, len=2"]
  E1 --> L1["Leaf: k1 -> 7"]

What exact proof nodes are sent for k1?

Membership proof for k1=000010 needs just these nodes: E0, B0, E1.

  • E0 consumes 000.
  • B0 consumes split bit 0.
  • E1 consumes 10.
{
  "proof_nodes": [
    { "id": "E0", "type": "edge", "path": "000", "length": 3, "child": "B0" },
    { "id": "B0", "type": "binary", "left": "E1", "right": "E2" },
    { "id": "E1", "type": "edge", "path": "10", "length": 2, "child": "Leaf(k1=7)" }
  ]
}

Non-membership for k3=000011:

  • After E0, k3 also takes branch 0 at B0.
  • Remaining bits for k3 are 11, but stored edge E1 is 10.
  • That mismatch is the non-membership proof.

In real RPC, node IDs are hashes (node_hash felt) instead of labels like E0, but verification is the same.

Commitment chain is still: key witness -> storage_root(contract) -> contract_leaf -> contracts_root -> global_state_root.

Sparse proof view: ignore non-path nodes

Add extra branches to the same toy trie. For proof(k1), we include only the queried path preimages and keep irrelevant branches as sibling hashes.

  • Included preimages: Broot, E0, B0, E1.
  • Omitted preimages: X subtree and E2 subtree.
  • Omitted branches are still cryptographically bound through their hashes inside parent binary nodes.

This makes the storage-proof path explicit: you do not send full sibling subtrees, only the sibling hashes needed to recompute ancestor hashes.

Quick checklist: membership

  • Have root hash at target block.
  • Have path bits for queried key.
  • Replay proof nodes and recompute hashes.
  • End state matches expected path outcome.

Quick checklist: non-membership

  • Replay path from root exactly like membership.
  • Detect branch/edge mismatch for requested key bits.
  • Recomputed chain still anchored at committed root.
  • Mismatching continuation proves absence.

5 — Failure modes

  • Historical window too short: low trie-log retention or strict distance gate causes deterministic rejection for older proofs.
  • Sparse snapshots: high snapshot interval with low snapshot count increases replay distance and proof latency spikes.
  • Configuration mismatch: tries disabled/special sync mode prevents full per-block proofability.
  • Resource abuse: oversized key/tries requests trigger hard limits; without limits, node is exposed to expensive proof workloads.
  • Verifier mismatch: wrong path conversion (not dropping top 5 bits) or wrong hash function per trie invalidates proofs.

6 — What to monitor

Signal Why it matters Healthy trend Status model
proof_request_latency_p95 Primary UX + RPC SLO metric for historical proof callers. Stable under mixed key cardinalities. track continuously
proof_old_block_reject_rate Shows if distance gate blocks real user workloads. Low and expected relative to policy. review weekly
snapshot_count_kept Confirms retention policy is actually active. Near configured cap, no oscillation bugs. policy aligned
trie_log_missing_errors Direct indicator that historical reconstruction can fail. Zero for supported block distance. page immediately
rocksdb_compaction_backlog Backlog influences read amplification and latency variance. No sustained upward drift. capacity tune
proof_key_count_distribution Validates current max key limit against real demand. Most requests below enforced cap. inform limit

A practical runbook pattern is to evaluate these metrics together: rising compaction backlog plus rising proof latency usually means the node is IO-bound, not necessarily proof-logic-bound. Rising old-block rejection with stable latency usually means policy mismatch, not infrastructure saturation.
