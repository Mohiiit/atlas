Primer · POC-A / POC-B / POC-C

Understanding the Three POCs
from First Principles

Start with the simplest question: how can a smart contract believe that a real device just performed a biometric-gated action, without learning more than it needs?

1. Why This Exists

A dApp may want to ask: "Is this action backed by a fresh human-presence signal?" In this POC, the human-presence signal is not a face image or fingerprint. It is a cryptographic signature produced by a Secure Enclave key that only signs after Face ID or Touch ID succeeds.

Non-crypto analogy: imagine a nightclub wristband. The door staff check your ID once, then put a wristband on you. Later, the bartender does not re-check your full ID; they only check whether the wristband is valid. Our POCs explore three ways to build that wristband for on-chain systems.

The core object

The raw biometric never leaves the device. The thing that leaves the device is a signature. The signature says: "the biometric-gated key inside this device signed this challenge."

3. Running Example

We will use the same character throughout: Bob has an iPad. The app creates a Secure Enclave key. Bob taps "Run Live Demo." Face ID succeeds. The iPad signs a relay-issued challenge:

Private values

Bob's private Secure Enclave key, the signature witness, and in ZK paths the hidden key coordinates.

Public values

The challenge hash, transaction data, and whatever the contract stores or emits.

4. The Three POCs

Each POC answers the same question differently: what should be visible to the chain?

POC-A: show everything

Send the public key and signature directly to the contract. The chain checks the math itself. Fastest and easiest, but public key privacy is zero.

POC-B: hide key and signature

Put the P-256 verification inside a ZK proof. The chain sees proof + message hash, not (qx, qy, r, s). Wallet is still visible.

POC-C: hide which registered key

Add a Merkle membership proof and a nullifier. The chain sees that one registered key acted, but does not receive a wallet parameter.

5. Mechanism Walkthrough

POC-A: Direct Verification

1

Bob signs

The iPad creates signature = ECDSA_P256_sign(challenge).

2

Contract receives raw data

Madara receives msg_hash, r, s, qx, qy in calldata.

3

Contract verifies

Cairo's secp256r1 syscall checks whether the signature is valid for the public key.

4

Credential recorded

If valid, the contract writes expires_at[wallet] = now + 24h.

POC-B: ZK-Wrapped P-256 Verification

1

Bob signs exactly as before

The device proof is the same kind of P-256 signature.

2

Operator builds witness

The witness contains hidden qx, qy, r, s. The public input is only msg_hash.

3

Noir proves the statement

The proof says: "there exists a valid P-256 signature for this hash" without revealing which key signed.

4

Contract verifies proof

The Garaga verifier checks the HONK proof and records humanness for the wallet.

POC-C: ZK + Set Membership + Nullifier

1

Build a registered-key set

The server builds a Merkle tree whose leaves are hashes of registered public keys.

2

Bob proves membership privately

The circuit proves Bob's hidden key is one leaf in the tree, without revealing which leaf.

3

Bob proves signature validity

The same hidden key must also verify the biometric P-256 signature.

4

Contract consumes nullifier

The nullifier prevents replay. In the current POC it is derived from the hidden leaf and message hash.

Common misconception

POC-B is not the same as v1 Option B. Current POC-B proves the raw biometric P-256 signature in ZK. v1 Option B would prove a backend issuer signature over a validated bundle.

7. Worked Examples

Example 1: Bob acts twice in POC-B

Monday proof: wallet 0xBob, hidden key, message hash 0xaaa. Tuesday proof: same wallet 0xBob, hidden key, message hash 0xbbb. Observer cannot see the key, but the same wallet links both actions.

Example 2: Bob acts twice in POC-C

Monday proof: root R, nullifier N1. Tuesday proof: root R, nullifier N2. No wallet parameter appears. A chain observer sees two registered-user actions, not Bob's wallet.

Failure scenario

If Bob reuses the same POC-C message, the circuit derives the same nullifier. The registry already consumed it, so the second transaction reverts with "Nullifier already used."

8. Failure Modes

POC-A failure

Wrong calldata ordering or high/low u256 split makes a valid signature fail on-chain. Privacy failure is inherent: key and signature are public.

POC-B failure

If the s-value is not canonicalized to low-s, the Noir signature check may fail. If the wallet is reused, the privacy story is only pseudonymous.

POC-C failure

Wrong Merkle path, stale root, or reused nullifier causes rejection. If leaves are publicly mapped to users, anonymity weakens.

9. Check Your Understanding

Q1. What is the minimum thing the iPad must produce?

A biometric-gated P-256 signature over a fresh challenge.

Q2. Why is POC-A easiest to debug?

Because the chain receives the raw public key and signature, so there is no ZK witness/proof layer hiding what went wrong.

Q3. Why is POC-B private but not anonymous?

It hides the device key and signature, but the wallet address remains visible and linkable.

Q4. What does POC-C add on top of POC-B?

Merkle set membership and a nullifier, so the contract can accept "someone in this registered set acted once" without a wallet parameter.

Q5. Why is v1 Option B cheaper than current POC-B?

Because the circuit would verify a backend issuer signature, not the raw Apple P-256 biometric signature.
