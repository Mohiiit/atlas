Decision Map · Option A/B/C vs Actual POC-A/B/C

Where the New v1 Options
Match the Existing POCs

The confusing part is that "Option B" and "POC-B" sound related, but they are different layers. Options A/B/C describe the on-chain claim boundary. POCs A/B/C describe the things we already built.

Short answer

Current POC-B is not Option B. Current POC-B proves the raw biometric P-256 signature inside ZK, so it is closest to Option A. v1 Option B would instead prove that our backend signed a validated credential bundle.

The Core Difference

The clean way to separate them is this:

Option A / Current POC-B

The proof says: "a real P-256 biometric signature exists." The biometric signature itself is inside the ZK witness.

Option B

The proof says: "Pinocchio backend signed this wallet as fresh." The backend already checked the biometric proof before signing.

Option C / Partial POC-C overlap

The proof says: "this item is in a Merkle set." In Option C the set is fresh wallets; in POC-C the set is registered biometric keys.

Option A POC-B Bind the signed message/challenge to wallet address and freshness epoch. Keep App Attest validation as the registry gate. If the phrase "client-side proof" is interpreted strictly, move Noir/BB proving from operator machine to the iPad. From "there exists a valid biometric signature for this hash" to "there exists a valid biometric signature for this wallet and epoch." Best if we want the strongest trust claim and can tolerate expensive proving.
Option B Relay + POC-B pipeline Add backend issuer key. After validating App Attest and P-256, sign a canonical bundle such as bundleVersion, walletAddress, freshnessEpoch. Replace the POC-B raw P-256 circuit with an issuer-signature circuit and pin BACKEND_ISSUER_PK in the verifier path. From "ZK checked the biometric signature" to "ZK checked Pinocchio signed a credential after validating the biometric/device proof." Best v1 path: cheaper circuit, cleaner demo, explicit operator trust.
Option C POC-C Change the Merkle tree from "registered biometric public keys" to "fresh wallet addresses for this epoch." Add root publication/authorization. Remove raw P-256 from the circuit if following pure Option C; biometric validation happens before wallet insertion into the daily set. From "this hidden registered key signed this message" to "this wallet is included in the current fresh-human set." Best for batching and scale, but not the shortest path for the next live demo.

Practical recommendation

If the goal is the cleanest v1 demo, evolve the current relay + POC-B flow into Option B. It keeps the iPad biometric/App Attest story, makes backend trust explicit, and avoids paying for raw P-256 verification inside the circuit.

Why Option B Feels Like POC-B

They feel similar because both use the same outer flow:

Shared outer flow

iPad creates biometric proof → relay validates it → operator/prover creates calldata → Madara transaction updates state.

But the cryptographic claim is different:

QuestionCurrent POC-Bv1 Option B
What does ZK verify?Raw P-256 biometric signature.Backend issuer signature over wallet + epoch.
Who is trusted for biometric validation?Circuit verifies the biometric signature; backend also checks it as defense-in-depth.Backend is trusted to validate biometric evidence correctly before signing.
What is the expensive part?P-256 verification inside the proof.Mostly just issuer signature verification.
What is the demo story?"We proved the biometric signature in ZK.""We proved Pinocchio's trusted issuer attested this wallet as fresh."

One-sentence correction

Option B is operationally similar to POC-B, but cryptographically different. POC-B trusts the ZK circuit to verify P-256. Option B trusts the backend issuer key to summarize that validation.
