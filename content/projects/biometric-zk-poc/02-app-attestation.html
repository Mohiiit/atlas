Chapter 02

Apple App Attest

How the relay knows the proof came from a genuine, unmodified app running on a real Apple device — not an emulator, modified binary, or script — and how refreshes stay bound to the same App Attest key.

The Problem App Attest Solves

Even if a user has a valid P-256 key and produces a correct ECDSA signature, how does the relay know the signing app is the real Pinocchio app and not a tampered version? A modified app could:

Without App Attest

An attacker could extract the protocol and build a custom app that signs without biometric checks, bypasses security controls, or runs on an emulator with fabricated keys.

With App Attest

Apple cryptographically binds the attestation to the app's code signature, the specific device, and the development team. The relay verifies this certificate chain before accepting any proof.

Certificate Chain

App Attest produces a certificate chain rooted at Apple's CA. The relay validates this chain to establish trust.

Attestation Flow: Step by Step

Relay-Side Validation Details

1

Parse Attestation Object (CBOR)

The attestation object is CBOR-encoded (Concise Binary Object Representation). It contains:

FieldContents
fmt"apple-appattest" — format identifier
attStmt.x5cCertificate chain: [leaf_cert, intermediate_cert]
authDataAuthenticator data with RP ID hash, flags, counter, credential ID

File: app_attest.py:133-170

2

Certificate Chain Verification

The relay verifies the X.509 certificate chain back to Apple's root CA:

# 1. Load Apple App Attestation Root CA (hardcoded PEM) root_cert = load_pem_x509_certificate(APPLE_ROOT_CA_PEM) # 2. Verify intermediate cert is signed by root intermediate_cert.verify(root_cert.public_key()) # 3. Verify leaf cert is signed by intermediate leaf_cert.verify(intermediate_cert.public_key()) # If any verification fails -> reject the attestation

This establishes that the leaf certificate was issued by Apple, not forged.

File: app_attest.py:315-340

3

Nonce Verification

The leaf certificate contains a custom extension with a nonce. The relay recomputes the expected nonce and checks it matches:

# Expected nonce = SHA-256(authData || clientDataHash) # where clientDataHash = SHA-256(challenge_hash || pub_key_x || pub_key_y) expected_nonce = SHA256(auth_data_bytes + client_data_hash) actual_nonce = leaf_cert.extensions["1.2.840.113635.100.8.2"] assert expected_nonce == actual_nonce # Binds attestation to this specific proof

Why the nonce matters

The nonce binds the attestation to the specific proof data. An attacker cannot reuse an attestation from one proof with different signature data — the nonce would mismatch.

4

RP ID Hash & AAGUID Checks

Two additional checks ensure the attestation belongs to the right app and environment:

RP ID Hash

The first 32 bytes of authData contain SHA-256(app_id) where app_id = teamID + "." + bundleID. This confirms the attestation was generated by our specific app.

AAGUID

The Authenticator Attestation GUID distinguishes environment:
appattestdevelop = development
appattest\x00\x00\x00\x00\x00\x00\x00 = production
Prevents using dev attestations in production.

# AuthenticatorData layout (app_attest.py:45-90): # Bytes 0-31: RP ID hash (SHA-256 of app identifier) # Byte 32: Flags (bit 0=UP, bit 2=UV, bit 6=AT, bit 7=ED) # Bytes 33-36: Sign counter (big-endian u32) # Bytes 37-52: AAGUID (16 bytes) # Bytes 53-54: Credential ID length (big-endian u16) # Bytes 55+: Credential ID (== keyID from generateKey)

What App Attest Proves

Proven

The request originated from a genuine Apple device running an unmodified copy of the specific app, built by the registered development team.

NOT Proven

App Attest does NOT prove the user is human or that biometrics were used. That's handled separately by the Secure Enclave's biometry-gated key. App Attest only proves the app is genuine.

Attestation vs Assertion

Attestation happens at enrollment: Apple proves the app/device context and key ID. Assertion happens on each refresh: the same App Attest key signs proof-bound data, and the relay verifies the signature plus an increasing counter.

Enrollment Attestation

Validates the certificate chain, nonce, RP ID hash, AAGUID, and key ID. This records the original device/app binding.

Refresh Assertion

Validates the assertion signature, RP ID hash, proof-bound client data hash, and monotonic counter. This prevents replaying an old App Attest response.

Simulator Caveat

The iOS Simulator does not support App Attest. When running on simulator, the app sets appAttestSupported: false and the relay skips attestation validation. In production, this check would be mandatory.
