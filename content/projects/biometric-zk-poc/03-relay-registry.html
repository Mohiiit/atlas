Chapter 03

Relay & Registry Backend

The Python server that validates App Attest, verifies biometric signatures, manages credential state, and orchestrates the proving pipeline.

Role in the System

The relay is the bridge between the mobile app and the blockchain. It performs three critical functions:

Current POC vs Preferred v1 Boundary

Today, the relay validates App Attest and the biometric P-256 signature, then the operator still proves the raw P-256 signature in ZK. In the preferred v1 Option B, the relay would instead sign a small validated credential bundle, and the ZK circuit would verify that backend issuer signature. That is cheaper, but it makes the backend issuer key part of the trust model.

API Endpoints

GET /challenge?purpose=demo

Issues a one-time relay challenge with a challengeId, purpose, issue time, and expiration time. The iOS app signs this challenge with the biometric-gated Secure Enclave key.

POST /demo/start

The main entry point for the live demo flow. Accepts a ProofPackage JSON payload, assigns a run_id, and launches the full pipeline in a background thread. Returns the run_id immediately for polling.

This is the endpoint the iOS app calls after creating a proof.

GET /demo/run/{run_id}

Returns the current status of a demo run, including all completed steps with their titles, details, and technical data. The iOS app polls this every second to update the UI progress.

POST /enroll

Registers a new device key. Validates App Attest, stores the public key, and associates it with a wallet address. Used for the first-time enrollment of a device.

POST /refresh

Refreshes an existing credential by verifying a fresh P-256 signature. Checks that the signature was made with the enrolled key and updates the expiration timestamp.

GET /wallet/{address}

Checks the current credential status for a wallet address. Returns enrollment state, last refresh time, and expiration.

Demo Execution Pipeline

When /demo/start is called, the relay executes these steps sequentially. Each step is tracked with status updates that the app polls.

Step 1: Proof Received + Challenge Consumed

ProofPackage JSON is parsed and validated. The relay checks the backend-issued challengeId, rejects expired or reused challenges, and marks the challenge consumed.

server.py:380-400

Step 2: Credential Assignment

The relay assigns a pre-funded Madara devnet account to this demo run. Account #0 for POC-A, #1 for POC-B, #2 for POC-C.

server.py:402-420

Step 3: App Attest Validation

Calls store.enroll(proof) and refresh assertion checks. Enrollment validates the Apple attestation object; refresh validates the App Attest assertion, RP ID hash, proof binding, and counter. On simulator builds, this step is skipped.

server.py:422-440 → app_attest.py:315-385

Step 4: Biometric Signature Verification

Calls store.refresh(proof) which reconstructs the P-256 public key from (qx, qy), then verifies the ECDSA signature over the fresh backend challenge. Uses the cryptography Python library.

server.py:121-133 — verify_p256_signature()

Step 5: ZK Proof Generation + On-Chain Submission

Calls run_operator() which spawns the operator shell script as a subprocess. The operator runs nargo, bb, garaga, and sncast to generate the proof and submit it to Madara. This step takes 20-60 seconds.

server.py:459 → v1/operator/submit_poc_b_latest.sh

Step 6: On-Chain State Query

After the transaction is confirmed, the relay queries the protocol contract's is_human(wallet) to verify the credential was successfully recorded. Returns the result to the polling app.

server.py:480-505

P-256 Signature Verification (Python)

The relay independently verifies the biometric signature before forwarding to the proving pipeline. In the current POC, this is a defense-in-depth check before the raw P-256 proof is generated. In preferred v1 Option B, this check becomes the basis for minting the backend-signed credential bundle.

def verify_p256_signature(qx_hex, qy_hex, challenge, der_hex=None, r_hex=None, s_hex=None): """Verify P-256 ECDSA signature over a challenge message.""" # Reconstruct the public key from coordinates pub_numbers = ec.EllipticCurvePublicNumbers( x=int(qx_hex, 16), y=int(qy_hex, 16), curve=ec.SECP256R1() ) pub_key = pub_numbers.public_key() # Use DER signature directly if available, otherwise reconstruct from (r, s) if der_hex: signature = bytes.fromhex(der_hex) else: signature = encode_dss_signature(int(r_hex, 16), int(s_hex, 16)) # Verify: raises InvalidSignature on failure pub_key.verify(signature, challenge.encode(), ec.ECDSA(hashes.SHA256()))

RegistryStore: Credential State Machine

The RegistryStore class maintains credential state in a JSON file. States: enrolled (key registered, App Attest validated), active (fresh signature verified, expires in 24h), expired (TTL elapsed), revoked (manually invalidated). The 24-hour freshness window matches the on-chain TTL.
