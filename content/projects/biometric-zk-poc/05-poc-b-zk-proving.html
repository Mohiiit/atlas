Naming Note

This page describes the already-built POC-B: raw P-256 biometric signature verification inside a ZK proof. That is closest to the v1 interface doc's Option A claim shape. It is not the preferred v1 Option B, where the circuit verifies a backend-signed credential bundle.

The Key Insight

In POC-A, the public key (qx, qy) is visible on-chain. In POC-B, we wrap the signature verification inside a zero-knowledge proof. The proof demonstrates:

ZK Statement

"I know a P-256 public key (qx, qy) and a valid ECDSA signature (r, s) such that ECDSA_verify(msg_hash, r, s, qx, qy) = true" — without revealing qx, qy, r, or s.

Public Inputs (visible on-chain)

Only the hashed_message — the SHA-256 of the challenge. This is 32 bytes.

Private Witnesses (hidden in proof)

pub_key_x (32B), pub_key_y (32B), signature (64B) — total 128 bytes never revealed.

Stage 1: Witness Preparation

The Python script reads the ProofPackage and writes a Prover.toml file that Noir can consume.

# prover_toml_from_app_proof.py - Key transformation # Read ProofPackage JSON proof = json.load(open("latest-proof-package.json")) # Extract and format values as byte arrays msg_hash = bytes.fromhex(proof["challengeHashHex"]) # 32 bytes qx = bytes.fromhex(proof["publicKeyXHex"]) # 32 bytes qy = bytes.fromhex(proof["publicKeyYHex"]) # 32 bytes r = bytes.fromhex(proof["signatureRHex"]) # 32 bytes s = bytes.fromhex(proof["signatureSHex"]) # 32 bytes # CRITICAL: Canonicalize s-value for the Noir circuit P256_ORDER = 0xFFFFFFFF00000000FFFFFFFFFFFFFFFFBCE6FAADA7179E84F3B9CAC2FC632551 s_int = int.from_bytes(s, 'big') if s_int > P256_ORDER // 2: s_int = P256_ORDER - s_int # Low-s canonical form s = s_int.to_bytes(32, 'big') # Combine r + s into 64-byte signature array signature = r + s # [r0, r1, ..., r31, s0, s1, ..., s31] # Write Prover.toml # hashed_message = [0xca, 0x97, 0x81, ...] (pub input) # pub_key_x = [0x3a, 0xb2, ...] (private witness) # pub_key_y = [0x7f, 0x1c, ...] (private witness) # signature = [0xd4, 0x2e, ...] (private witness)

Why canonicalize s?

ECDSA signatures have a malleability property: if (r, s) is valid, so is (r, ORDER - s). The Noir circuit's verify_signature function expects the "low-s" canonical form. Without this step, valid signatures from the Secure Enclave may fail verification in the circuit.

Stage 2: The Noir Circuit

The core circuit is surprisingly concise — Noir's standard library handles P-256 ECDSA via Barretenberg's custom gates.

// poc-b/biometric_circuit/src/main.nr fn main( // Public input: the challenge message hash (visible on-chain) hashed_message: pub [u8; 32], // Private witnesses: the device key and signature (hidden in proof) pub_key_x: [u8; 32], pub_key_y: [u8; 32], signature: [u8; 64], // r (32 bytes) || s (32 bytes) ) { // Verify the P-256 ECDSA signature. // std::ecdsa_secp256r1 is a Barretenberg black-box function - // it uses custom gates, not field-emulated arithmetic, // so it is fast even for a foreign field like P-256. let valid = std::ecdsa_secp256r1::verify_signature( pub_key_x, pub_key_y, signature, hashed_message, ); // This assertion is the heart of the ZK proof: // the proof is only valid if the signature is valid. assert(valid, "P256 signature verification failed"); }

Custom Gates for P-256

Noir doesn't emulate P-256 arithmetic in its native field (BN254). Instead, Barretenberg provides dedicated custom gates for secp256r1 operations. This makes the circuit much smaller and faster to prove than a naive field-emulation approach.

Stage 4: Garaga Calldata Generation

Garaga is the bridge from Barretenberg's proof format to Cairo's calldata format. It generates a complete Cairo verifier project from the verification key.

# Generate the Cairo verifier project from the VK garaga gen \ --system ultra_keccak_zk_honk \ --vk biometric_circuit/target/vk \ --project-name biometric_honk_verifier # This creates: # biometric_honk_verifier/ # src/honk_verifier.cairo - The verifier contract # src/honk_verifier_circuits.cairo - Circuit-specific constants # src/honk_verifier_constants.cairo - VK constants baked in # Scarb.toml # Generate calldata for a specific proof garaga calldata \ --system ultra_keccak_zk_honk \ --vk target/vk \ --proof target/proof \ --public-inputs target/public_inputs # Output: felt252 array ready for sncast invoke

Stage 5: On-Chain Verification

The protocol contract wraps the Garaga-generated Honk verifier and records the credential.

// poc-b/protocol/src/lib.cairo fn submit_biometric_proof( ref self: ContractState, wallet: ContractAddress, full_proof_with_hints: Span<felt252>, // Garaga calldata ) -> u256 { // 1. Call the Honk verifier (Garaga-generated contract) let verifier = IHonkVerifierDispatcher { contract_address: self.verifier_address.read() }; // verify_ultra_keccak_zk_honk_proof returns Option<Span<u256>> // The Span contains the public inputs if verification succeeds let result = verifier.verify_ultra_keccak_zk_honk_proof( full_proof_with_hints ); assert(result.is_some(), 'ZK proof verification failed'); // 2. Extract the message hash from public inputs let public_inputs = result.unwrap(); let message_hash = *public_inputs.at(0); // First 256 bits // 3. Record the credential with 24-hour TTL let now = get_block_timestamp(); self.last_verified_at.write(wallet, now); self.expires_at.write(wallet, now + 86400); // 24 hours self.emit(HumannessRefreshed { wallet, message_hash, ... }); message_hash }

What Changed vs POC-A

POC-A (Native)

Contract receives (msg_hash, r, s, qx, qy) in plaintext. Does ECDSA verification directly. Public key is fully exposed on-chain.

POC-B (ZK)

Contract receives a ZK proof blob. Garaga verifier checks the proof. Only msg_hash is visible. The public key and signature exist only inside the proof — never on-chain.

How This Relates to the New v1 Option B

The current POC proves the Apple biometric signature directly. The preferred v1 architecture moves the claim boundary: the backend validates the Apple evidence first, then signs a compact credential bundle that the circuit verifies cheaply.

Current POC-B

ZK proves: ECDSA_P256_verify(msg_hash, r, s, qx, qy) == true. Stronger claim, heavier proof. Backend validates too, but the circuit still proves the raw biometric signature.

Preferred v1 Option B

ZK proves: Verify(BACKEND_ISSUER_PK, H(wallet, epoch), issuerSignature) == true. Cheaper proof, clearer demo path, but the backend issuer key becomes trusted.

Practical takeaway

We already proved the harder raw-signature path. Moving to preferred v1 Option B is mostly a trust-boundary change: replace the P-256 witness circuit with a backend-issuer-signature circuit and make the signed bundle the artifact that crosses into the prover.

Deep Dive: What Exactly Does POC-B Leak?

POC-B hides the device key, but privacy is not just about key hiding. Let's walk through exactly what an on-chain observer sees and why POC-B is pseudonymous, not anonymous.

Scenario: Alice Uses POC-B Twice

Alice has an iPhone with a biometric key. She submits two proofs on different days.

Why Does the Wallet Address Leak?

The ZK proof hides the cryptographic identity (device key), but the proof is submitted as a Starknet transaction. Every transaction on Starknet has:

// What's in a Starknet transaction: Transaction { sender: 0xAlice_Wallet_123, // <-- THE LEAK: wallet is always visible function: "submit_biometric_proof", calldata: [ wallet_address, // <-- Explicitly passed as argument too ...proof_blob... // ZK proof (opaque blob, hides key + sig) ], signature: [...], // Starknet account signature (NOT the P-256 sig) nonce: 42, max_fee: ... } // Even though the P-256 key is hidden inside the ZK proof, // the WALLET ADDRESS ties all actions together.

The Core Problem

POC-B proves: "Some valid biometric key signed this message for wallet 0xAlice..."
An observer knows the wallet identity but not the device identity. If Alice uses the same wallet for other on-chain actions (DeFi, NFTs, etc.), her biometric proofs can be correlated with all of those.

What If Alice Uses a Fresh Wallet?

Alice could create a new wallet for each proof. But then:

Problem 1: Gas Funding

The fresh wallet needs gas to submit the transaction. If Alice funds it from her main wallet, the transfer creates an on-chain link between the fresh wallet and her identity. Even using a mixer or bridge leaves metadata traces.

Problem 2: Relay Knows

The relay server receives the full ProofPackage (including the public key) before generating the ZK proof. The relay itself can link the wallet to the device. You're trusting the relay not to log or leak this mapping.

This is precisely why POC-C exists. It eliminates both problems by removing the wallet from the equation entirely.

POC-B Privacy Scorecard

Tough Questions You Might Get Asked

Q

"Can someone brute-force the public key from the proof?"

No. The ZK proof's zero-knowledge property mathematically guarantees that no information about the witness (public key, signature) leaks from the proof. This isn't security-by-obscurity — it's a cryptographic guarantee of the HONK proof system. A computationally bounded attacker cannot extract the private witness from the proof.

The proof only proves existence: "there exists a valid key and signature satisfying the circuit constraints." It reveals nothing about which key or signature.

Q

"If I have a list of known public keys, can I figure out which one made this proof?"

No. Even if you know all possible public keys in the world, you cannot determine which one was used to generate a POC-B proof. The proof does not leak any partial information (like a hash of the key) that would allow elimination.

This is the whole point of zero-knowledge: the verifier learns nothing beyond the truth of the statement.

Q

"Then why isn't POC-B fully private?"

Because privacy isn't just about the cryptographic proof. The transaction itself reveals the wallet address, and the proof is bound to that wallet. The ZK proof is perfectly zero-knowledge, but the infrastructure around it (Starknet accounts, gas payments, relay communication) creates linkability.

Think of it like wearing a mask (ZK proof hides your face) but carrying your name badge (wallet address). The mask is perfect, but the badge gives you away.
