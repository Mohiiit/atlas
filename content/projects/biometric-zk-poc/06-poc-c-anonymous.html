The Three Simultaneous Proofs

POC-C's Noir circuit proves three things at once — and this combination is what makes it anonymous to on-chain observers:

1

Signature Validity

"I know a valid P-256 ECDSA signature over this message, made with some public key PK."

2

Set Membership

"PK is a member of the registered key set (Merkle root R)." Proved via authentication path without revealing PK.

3

Nullifier

"This proof is bound to this specific message and cannot be replayed." In the current POC, it is computed as BLAKE2s(leaf, message_hash).

Privacy Guarantee

Two proofs from the same user, for different actions, cannot be linked to each other on-chain. The nullifier ties a proof to a specific action, but the nullifier itself does not reveal the user's identity.

Current POC Caveat

The current relay still receives the full ProofPackage before generating the proof. Also, the current nullifier is derived from the hidden leaf and message hash, not from a separate private nullifier secret. So the anonymity described here is the on-chain property of this demo. A production anonymous design should use a commitment/nullifier-secret model or device-side proving.

Merkle Tree Construction

The anonymity set is a Merkle tree of public keys. Each leaf is a hash of a registered user's P-256 public key.

# build_merkle.py / build_witness_from_app_proof.py TREE_DEPTH = 2 # 4 leaves = 2^2 users PROVER_INDEX = 1 # Our prover is at index 1 # Hash function: BLAKE2s (matching Noir's std::hash::blake2s) def hash_pair(a: int, b: int) -> int: combined = a.to_bytes(32, 'big') + b.to_bytes(32, 'big') return int.from_bytes(hashlib.blake2s(combined).digest(), 'big') def pubkey_to_leaf(qx: int, qy: int) -> int: return hash_pair(qx, qy) # Build the tree bottom-up leaves = [pubkey_to_leaf(k["qx"], k["qy"]) for k in keys] # Level 0: [leaf0, leaf1, leaf2, leaf3] # Level 1: [H(leaf0, leaf1), H(leaf2, leaf3)] # Level 2: [H(level1[0], level1[1])] = ROOT # Extract authentication path for leaf at index 1 # path = [leaf0 (sibling), H(leaf2,leaf3) (sibling)] # indices = [1, 0] -- leaf1 is RIGHT child, then LEFT child

The Circuit: Three Proofs in One

// poc-c/biometric_anon_circuit/src/main.nr fn main( // --- Public inputs (visible on-chain) --- hashed_message: pub [u8; 32], // SHA-256 of challenge merkle_root: pub Field, // 31-byte truncation of BLAKE2s Merkle root nullifier: pub Field, // 31-byte truncation of BLAKE2s(leaf, message) // --- Private witnesses (hidden) --- pub_key_x: [u8; 32], // Device P-256 public key X pub_key_y: [u8; 32], // Device P-256 public key Y signature: [u8; 64], // ECDSA (r||low-s) merkle_path: [[u8; 32]; DEPTH], // Sibling hashes path_indices: [u8; DEPTH], // 0=left, 1=right at each level ) { // PROOF 1: Verify the P-256 ECDSA signature let sig_valid = std::ecdsa_secp256r1::verify_signature( pub_key_x, pub_key_y, signature, hashed_message, ); assert(sig_valid, "P256 signature verification failed"); // PROOF 2: Verify Merkle set membership let leaf = hash_pair(pub_key_x, pub_key_y); let computed_root = merkle_root_from_path(leaf, merkle_path, path_indices); assert(hash_to_field(computed_root) == merkle_root, "Merkle root mismatch"); // PROOF 3: Verify nullifier prevents replay for this message let computed_nullifier = hash_pair(leaf, hashed_message); assert(hash_to_field(computed_nullifier) == nullifier, "Nullifier mismatch"); } fn hash_pair(left: [u8; 32], right: [u8; 32]) -> [u8; 32] { let mut input: [u8; 64] = [0; 64]; for i in 0..32 { input[i] = left[i]; input[i + 32] = right[i]; } std::hash::blake2s(input) } fn merkle_root_from_path( leaf: [u8; 32], path: [[u8; 32]; DEPTH], indices: [u8; DEPTH], ) -> [u8; 32] { let mut current = leaf; for level in 0..DEPTH { let sibling = path[level]; if indices[level] == 1 { current = hash_pair(sibling, current); } else { current = hash_pair(current, sibling); } } current }

Nullifier: Preventing Double-Spend

The nullifier is the key to preventing replay attacks while maintaining anonymity.

How the current POC works

The current POC computes nullifier = H(leaf, hashed_message), where leaf = H(qx, qy). The leaf is hidden inside the proof witness, so chain observers do not see which registered key produced the nullifier.

On-chain tracking

The AnonRegistry contract stores used nullifiers. Before accepting a proof, it checks is_nullifier_used(nullifier). If already used, the transaction reverts. This prevents replay without exposing which leaf produced the original proof.

Production note

If the full leaf set is public, observers could test known leaves against public message hashes. A stronger production design should derive nullifiers from a private per-user secret or commitment, not only from the public-key leaf.

Similar Patterns in Production

This POC uses the same broad family of ideas as Semaphore and Tornado Cash: Merkle membership plus nullifier-based replay prevention. The production versions of those systems use stronger commitment/secret material than this POC's public-key leaf design.

Deep Dive: How Exactly Does POC-C Achieve Anonymity?

Let's walk through a concrete scenario with real (simplified) numbers to make the privacy properties crystal clear.

Setup: Four Users Register

Four people enroll their devices. The server builds a Merkle tree from their public keys:

Action: Bob Submits an Anonymous Proof

Bob wants to prove he's a registered user without revealing that he's Bob. Here's what happens:

1

Inside the ZK Circuit (Private)

The Noir circuit receives Bob's private witness and verifies three things in zero-knowledge:

// INSIDE THE ZK CIRCUIT (none of this is visible to anyone) // Check 1: "I have a valid signature" ECDSA_verify(msg_hash, r_bob, s_bob, qx_bob, qy_bob) == true ✓ // Check 2: "My key is in the registered set" leaf = H(qx_bob, qy_bob) = 0x2e8c... step1 = H(leaf0, leaf) = H(0x7a3f, 0x2e8c) = 0x55bb... = node_L ✓ step2 = H(node_L, node_R) = H(0x55bb, 0xcc11) = 0xdead... = ROOT ✓ // Check 3: "This nullifier is correctly computed" nullifier = H(leaf_bob, msg_hash) = 0x4444... ✓

The circuit outputs only three values: msg_hash, merkle_root, and nullifier. Everything else stays hidden.

2

What Appears On-Chain (Public)

// THE ENTIRE ON-CHAIN FOOTPRINT: Transaction { // Note: this can be submitted by ANY account (operator, relayer) // It does NOT need to be Bob's wallet sender: 0xOperator_Address, // The relay/operator, NOT Bob function: "submit_anon_biometric_proof", calldata: [ ...proof_blob..., // Opaque ZK proof (~3KB) ] } // Public inputs extracted by the verifier contract: msg_hash: 0xa7f3... // Which challenge was signed merkle_root: 0xdead... // Which user set (all 4 users share this) nullifier: 0x4444... // Unique to this hidden leaf + message hash // THAT'S IT. No wallet. No key. No signature. No identity.

Key Difference from POC-B

In POC-B, the transaction included wallet: 0xAlice_Wallet_123 as a parameter. In POC-C, there is no wallet parameter. The proof is submitted by the operator on behalf of someone in the set. The contract doesn't know or care who.

3

What an Observer Sees

An on-chain observer (blockchain explorer, analytics firm, etc.) sees:

All they know

"One of the 4 registered users signed a message with a valid biometric key. The action is identified by nullifier 0x4444... and cannot be repeated."

What they don't know

"Was it Alice, Bob, Carol, or Dave? Which device key was used? What was the signature? Where in the tree is this user?"

The Unlinkability Test: Bob Acts Twice

Now Bob performs a second action (different action_id). Let's see what changes:

Same hidden leaf, same message = same nullifier (prevents replay)

If Bob tries to submit the same message again, the circuit would produce the same nullifier 0x4444... because H(leaf_bob, msg_hash_vote_7) is deterministic. The AnonRegistry contract would reject it: "Nullifier already used." This prevents replay while preserving on-chain anonymity in the demo.

Tough Questions You Might Get Asked

Q

"With only 4 users, can't you just try all 4 keys and see which one makes a valid proof?"

In the current POC, this depends on whether the leaf set is public. The circuit hides Bob's public key and Merkle path, but the nullifier is derived from H(leaf, message_hash). If an observer already knows every leaf, they can test leaves against public message hashes.

They still cannot forge a valid proof without the private key, because the ZK proof also verifies a real ECDSA signature. But for production-grade anonymity, the leaf/nullifier design should include a separate private secret or commitment.

The anonymity set size (4 users here) does affect the practical anonymity — a larger set makes statistical analysis harder. In production, this would be thousands or millions of keys.

Q

"How does the contract know the prover is actually in the set, not just fabricating a Merkle proof?"

The circuit enforces this cryptographically:

1. The prover must provide a valid ECDSA signature (proving they own a real P-256 key, not a random number)

2. The same key that signed must hash to a leaf that's in the Merkle tree (the circuit computes the leaf from the key coords, not from user input)

3. The authentication path must reconstruct to the on-chain root (which is stored in the AnonRegistry contract)

Fabricating any of these would mean either breaking ECDSA, finding a SHA-256 collision, or forging a Merkle proof — all computationally infeasible.

Q

"The relay sees the proof before it goes on-chain. Doesn't the relay know who Bob is?"

This is a valid concern. In the current POC, the relay receives the full ProofPackage (including public key) and generates the ZK proof server-side. So yes, the relay knows the mapping.

In production, the proving would happen on the device itself (using Mopro/Swoir for mobile ZK proving). The device would send only the finished ZK proof to the relay — the relay would never see the public key, signature, or Merkle path. The relay would only see the opaque proof blob + public inputs.

Client-side proving is flagged as a stretch goal in the POC docs. The current architecture is relay-proven for simplicity, with the understanding that device-side proving is the production path.

Q

"What happens when a new user joins? Does the Merkle root change?"

Yes. When a new user enrolls, their public key becomes a new leaf, the tree is rebuilt, and a new root is posted to the AnonRegistry contract on-chain. All existing users' authentication paths change (they get new siblings at some levels).

The contract stores the current root. Proofs generated with an old root will fail verification after the root is updated. In production, you'd either batch enrollments or support multiple valid roots with a transition window.

Q

"How is this different from Tornado Cash?"

Same cryptographic pattern, different domain:

Tornado Cash: Merkle tree of deposit commitments. Proves "I deposited funds" without revealing which deposit. Nullifier prevents double-withdrawal.

POC-C: Merkle tree of biometric public keys. Proves "I'm a registered human with a valid biometric" without revealing which human. Nullifier prevents double-action.

The math is identical. The leaves are different (deposit hashes vs public key hashes), but the Merkle proof + nullifier mechanism is the same proven construction.
