What POC-A Does

This is the simplest path: the raw signature components (r, s) and the device public key (qx, qy) are submitted directly to the smart contract on Madara. Cairo's built-in secp256r1 syscall verifies the ECDSA math on-chain.

What is proven on-chain

A valid ECDSA P-256 signature exists over a specific message hash.

What is visible on-chain

Everything: the public key (qx, qy), the message hash, and the signature (r, s). Anyone can see which device key signed.

Why include a non-ZK path?

POC-A exists to validate the Secure Enclave → Madara pipeline before adding ZK complexity. If signatures from the device aren't verifying on-chain, it's much easier to debug without ZK layers in between.

Architecture

The Cairo Contract

The contract is remarkably simple — it's a thin wrapper around Cairo's native secp256r1 operations.

// poc-a/biometric_verifier/src/lib.cairo fn verify_biometric_signature( ref self: ContractState, msg_hash: u256, // SHA-256 of the challenge r: u256, // ECDSA signature component r s: u256, // ECDSA signature component s qx: u256, // Device public key X coordinate qy: u256, // Device public key Y coordinate ) -> bool { // 1. Reconstruct the P-256 public key point from coordinates // This calls Madara's native secp256r1 syscall path let public_key: Option<Secp256r1Point> = Secp256Trait::secp256_ec_new_syscall(qx, qy).unwrap_syscall(); assert(public_key.is_some(), 'Invalid public key point'); // 2. Verify the ECDSA signature // Cairo natively supports P-256 (secp256r1) signature verification let valid = is_valid_signature::<Secp256r1Point>(msg_hash, r, s, pk); assert(valid, 'Biometric signature invalid'); // 3. Record the verification self.last_verified.write(get_caller_address()); self.verification_count.write(self.verification_count.read() + 1); self.emit(BiometricVerified { caller, msg_hash, qx, qy, count }); true }

Calldata Conversion

On Starknet, u256 values are passed as two felt252 values: low 128 bits then high 128 bits. The Python script convert_calldata.py handles this:

# convert_calldata.py def u256_to_calldata(hex_str): """Convert a 256-bit hex value to two 128-bit felt252 values.""" n = int(hex_str.lstrip('0x'), 16) low = n & ((1 << 128) - 1) # Lower 128 bits high = (n >> 128) & ((1 << 128) - 1) # Upper 128 bits return f"0x{low:032x}", f"0x{high:032x}" # Each of msg_hash, r, s, qx, qy becomes two felts # Total calldata: 10 felt252 values for 5 u256 parameters

Privacy Analysis

No Privacy for the Public Key

In POC-A, anyone reading the chain can see (qx, qy) — the device's public key. This means actions from the same device can be linked. The only privacy comes from the hardware: the raw biometric (face scan, fingerprint) never leaves the Secure Enclave. But the public key identity is fully exposed.

This is why POC-B and POC-C exist: they hide the public key inside a ZK proof, giving progressively stronger privacy guarantees.
