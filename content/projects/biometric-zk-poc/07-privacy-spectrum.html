POC Names vs v1 Claim Options

The POC letters and the new interface-doc options are related, but they are not the same naming system. The important question is: what does the on-chain proof actually verify?

Path What the chain verifies New-doc parallel Status
Current POC-A Raw P-256 signature and public key in plaintext via Cairo secp256r1. Option A shape, without ZK privacy. Built
Current POC-B Raw P-256 signature inside a Noir/HONK ZK proof. Closest to Option A: strongest claim, heavier proof. Built
Preferred v1 Option B Backend issuer signature over a validated credential bundle. Option B: cheaper proof, stronger backend trust assumption. Not built yet
Current POC-C Merkle membership + nullifier + raw P-256 signature inside ZK. Related to Option C, but not the daily signed fresh-set root design. Built as concept demo

The naming trap

Current POC-B is not the same thing as the v1 interface doc's Option B. Current POC-B proves the biometric P-256 signature directly. Preferred v1 Option B proves that the backend signed a validated bundle after checking the biometric evidence. See the Options vs POCs map for the full gap diagram.

Attack Scenarios

For each POC, what can different adversaries learn?

POC-A Attacker

Chain observer

Reads (qx, qy) from the transaction. Searches all historical transactions for the same (qx, qy). Instantly links every action this device ever performed. If the wallet is connected to an ENS name, exchange KYC, or social account — they know who owns the device.

Effort: Trivial. One block explorer query.

POC-B Attacker

Chain observer

Cannot see the key. But sees the wallet address in every transaction. Searches all transactions from 0xBob_Wallet. Links the biometric proof to Bob's DeFi trades, NFT purchases, and any other on-chain activity.

Effort: Trivial. The wallet address is the link.

POC-C Attacker

Chain observer

Sees the proof was from "one of N registered users" (Merkle root). Two proofs have different nullifiers and no wallet parameter. If the key leaves are not publicly mapped, there is no obvious on-chain link to the user.

Caveat: the current POC nullifier is derived from the hidden leaf and message hash; production should use a private secret/commitment design.

The Relay Trust Question

One nuance that cuts across all three POCs: the relay server currently sees the full proof data because proving happens server-side.

Current POC Device-Side Proving Goal
Where ZK proof is generated Relay/operator server On the device (Mopro/Swoir)
What relay sees Full ProofPackage (key + sig) Only the ZK proof + public inputs
Relay can deanonymize? Yes (it sees the key) No (never sees the key)
Trust assumption Relay is honest Relay cannot inspect the witness

Client-side proving removes the relay's visibility into the witness. If v1 chooses backend-signed bundles instead, the relay remains trusted but the circuit becomes much cheaper.

Summary: The Privacy Ladder

A
Transparent
Everyone sees your device key and every action you take. Privacy comes only from the Secure Enclave protecting the raw biometric. Fast and cheap, good for debugging the pipeline.
B
Pseudonymous
Your device key is hidden in a ZK proof — nobody knows which device signed. But your wallet address still ties all your actions together. Like wearing a mask but carrying your name tag. Good when device privacy matters but wallet identity is acceptable.
C
Anonymous
No device key visible, no wallet linkage, and no direct action correlation on-chain. Two proofs from the same person can look like proofs by two different people if the leaf set is not publicly mapped. Nullifiers prevent replay; production anonymity should add a private commitment/nullifier secret.

Future Direction: Custom STARK AIR

Path D (Research)

A custom Stwo Circle STARK AIR that natively arithmetizes P-256 operations over M31 — the theoretical ideal: transparent, post-quantum, STARK-native, no SNARK wrap. However, no such AIR exists today, and building one is a significant research effort. This is flagged as future work, not a current POC.

When to Use Which

Use POC-A when...

You need the fastest, cheapest verification. Privacy of the public key doesn't matter. Good for: internal systems, compliance checks where the verifier is trusted, or when the identity being proven is already public (e.g., "this known device signed this message").

Use POC-B when...

You want to hide the device key but actions can still be tied to a wallet. Good for: proving humanness for a specific wallet without revealing which device was used. The wallet is pseudonymous but consistent.

Use POC-C when...

On-chain anonymity is required and the system can keep leaf-to-user mappings private. Good for: anonymous voting, private attestations, or any protocol where even pseudonymous wallet identity is too revealing. Production should strengthen the nullifier secret model.
