Chapter 01

Device Proof Creation

How the iOS app captures a biometric signature using the Secure Enclave and packages it for relay submission.

Step-by-Step: How a Biometric Proof is Created

1

Key Enrollment

BiometricProofService.swift:534-586

When the user taps "Enroll", the app creates a new P-256 elliptic curve keypair inside the Secure Enclave. This is the same curve used by Apple Pay, FIDO2, and WebAuthn.

// Key creation with biometric constraint let access = SecAccessControlCreateWithFlags( kCFAllocatorDefault, kSecAttrAccessibleWhenUnlockedThisDeviceOnly, [.privateKeyUsage, .biometryCurrentSet], // Requires Face ID / Touch ID &error ) // Store in Secure Enclave with tag let attrs: [String: Any] = [ kSecAttrKeyType: kSecAttrKeyTypeECSECPrimeRandom, // P-256 kSecAttrKeySizeInBits: 256, kSecAttrTokenID: kSecAttrTokenIDSecureEnclave, // Hardware-backed kSecAttrApplicationTag: "com.karnot.pinocchiopoc.biometric-key.v1" ]

Security Property

The .biometryCurrentSet flag means the key is invalidated if biometric enrollment changes (e.g., new fingerprint added). This prevents another person from using a device after adding their own biometric.

On Real Device

Key lives in the Secure Enclave Processor (SEP) — a dedicated hardware chip. The private key material is never exposed to the application processor or OS kernel.

On Simulator

Falls back to an ephemeral in-memory P-256 key using CryptoKit.P256.Signing.PrivateKey(). No biometric gating. Tagged as simulator-ephemeral-p256.

2

Backend-Issued Challenge

BiometricProofService.swift:90-95

For the live demo, the app first asks the relay for a one-time challenge. This prevents replay attacks because the relay accepts each challenge only once. A local timestamp + UUID challenge remains available as an offline/simulator fallback.

// Live demo path: fetch a one-time challenge from the relay let issuedChallenge = try await fetchBackendChallenge( relayBaseURL: relayBaseURL, purpose: "demo" ) let challenge = issuedChallenge.challenge // Fallback path: local challenge for simulator/offline proof creation let challenge = "pinocchio-device-proof:local:\(Int(Date().timeIntervalSince1970)):\(UUID().uuidString)" // SHA-256 hash of the challenge (this becomes the message signed by the key) let challengeHash = SHA256.hash(data: challenge.data(using: .utf8)!)

Why SHA-256 the challenge?

ECDSA signs a fixed-length hash, not arbitrary data. The 32-byte SHA-256 digest becomes the hashed_message — the only public input visible on-chain in POC-B and POC-C.

3

Biometric-Gated Signing

BiometricProofService.swift:96-100

Loading the signing key triggers the biometric prompt. The user must authenticate with Face ID or Touch ID before the Secure Enclave will perform any operation with the key.

// This triggers Face ID / Touch ID let key = try await loadSigningKey() // Sign the challenge hash with ECDSA-P256-SHA256 let signature = try sign(data: challengeData, with: key) // The signature is DER-encoded by the Secure Enclave: // SEQUENCE { INTEGER r, INTEGER s } // Example: 30 44 02 20 [32 bytes r] 02 20 [32 bytes s]

The signing happens entirely within the Secure Enclave. The application processor sends the hash in and gets the signature out — it never sees the private key.

4

DER Signature Parsing

BiometricProofService.swift:666-701

The Secure Enclave outputs an ASN.1 DER-encoded signature. The app parses this to extract the raw (r, s) components needed by the ZK circuit and on-chain verifier.

// DER format: 30 <total_len> 02 <r_len> <r_bytes> 02 <s_len> <s_bytes> // // Example breakdown: // 30 44 - SEQUENCE, 68 bytes // 02 20 - INTEGER, 32 bytes // [r: 32 bytes] // 02 20 - INTEGER, 32 bytes // [s: 32 bytes] // // Gotcha: DER integers may have a leading 0x00 byte // if the high bit is set (to avoid being interpreted as negative). // normalizeInteger() strips/pads to exactly 32 bytes. func parseDERSignature(_ der: Data) -> (r: Data, s: Data) { var i = 2 // Skip 0x30 and total length assert(der[i] == 0x02) // INTEGER tag for r let rLen = Int(der[i+1]); i += 2 let r = normalizeInteger(der[i..<i+rLen]); i += rLen assert(der[i] == 0x02) // INTEGER tag for s let sLen = Int(der[i+1]); i += 2 let s = normalizeInteger(der[i..<i+sLen]) return (r, s) // Each exactly 32 bytes }

S-Value Canonicalization

For ZK circuits (POC-B, POC-C), the operator must canonicalize the s-value: if s > P256_ORDER/2, use P256_ORDER - s. This is done in the Python witness generation script, not in Swift. The Noir circuit's verify_signature expects the low-s form.

What Makes This Secure?

Hardware Isolation

The private key exists only inside the Secure Enclave Processor — a dedicated chip with its own encrypted memory. Even a jailbroken device cannot extract it.

Biometric Binding

The .biometryCurrentSet flag ties the key to the current biometric enrollment. If a new face or fingerprint is enrolled, the key is automatically invalidated.

Freshness

Live proofs are bound to a relay-issued one-time challenge. The relay rejects unknown, expired, or already consumed challenge IDs, then enforces the credential freshness window.

App Attest

Apple's App Attest service cryptographically proves the request came from a genuine, unmodified build of the app running on a real Apple device. Enrollment uses an attestation object; refreshes use App Attest assertions.
